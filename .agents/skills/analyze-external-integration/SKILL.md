# /analyze-external-integration 외부 연동 설계 분석

목표: **외부 시스템은 느려질 수 있고, 실패할 수 있고, 중복 실행될 수 있으며, 성공했지만 응답이 유실될 수 있다는 전제로 설계를 검토한다.**

이 스킬은 새 설계를 대신 만들어내기보다, 이미 있는 설계나 코드를 읽고 리스크와 선택지를 드러내는 데 사용한다.

## 1. 호출 흐름보다 불확실성을 먼저 본다

단순히 "A가 B를 호출한다"로 요약하지 않는다.

먼저 다음 질문으로 문제를 재해석한다.

- 외부 시스템이 지연되면 사용자 요청은 얼마나 기다리는가?
- 외부 호출이 실패하면 내부 상태는 어떻게 남는가?
- 외부 호출은 성공했지만 응답이 유실되면 어떻게 판단하는가?
- 같은 요청이 재시도되어 두 번 실행되면 어떤 데이터가 중복되는가?
- 콜백, 메시지, 이벤트가 늦게 오거나 중복으로 오면 상태 전이가 안전한가?

## 2. 트랜잭션 경계를 검증한다

외부 호출이 DB 트랜잭션 내부에 있는지 확인한다.

- 트랜잭션 시작 지점과 종료 지점을 찾는다.
- 트랜잭션 안에 HTTP 호출, 메시지 발행, 콜백 전송, Redis/Kafka 등 외부 I/O가 있는지 확인한다.
- 내부 DB 상태와 외부 시스템 상태가 하나의 트랜잭션처럼 취급되고 있는지 판단한다.

반드시 답한다.

- 외부 호출 실패 시 내부 DB 변경은 commit 되는가 rollback 되는가?
- 내부 commit 이후 외부 호출이 실패하면 재시도 또는 복구 가능한가?
- 외부 성공 이후 내부 commit 이 실패하면 정합성을 어떻게 회복하는가?
- 외부 호출이 오래 걸릴 때 트랜잭션, lock, connection 을 얼마나 오래 점유하는가?

## 3. 상태 전이를 중심으로 다시 그린다

호출 순서가 아니라 상태를 기준으로 설명한다.

다음처럼 내부 상태와 외부 상태를 분리한다.

```text
내부 주문 상태: CREATED -> PAYMENT_REQUESTED -> PAID / PAYMENT_FAILED
외부 PG 상태:   없음 -> PENDING -> APPROVED / REJECTED
```

그리고 두 상태가 어긋날 수 있는 지점을 표시한다.

- 내부는 결제 요청됨인데 외부 거래가 생성되지 않은 경우
- 외부는 승인됐지만 내부 주문이 아직 결제 완료가 아닌 경우
- 내부는 실패 처리했지만 외부 콜백이 나중에 성공으로 도착하는 경우
- 같은 외부 거래 결과가 중복 도착하는 경우

## 4. 멱등성과 중복 실행을 확인한다

네트워크 재시도와 사용자 재시도를 기본 시나리오로 둔다.

- 요청마다 idempotency key, orderId, transactionKey 같은 중복 방지 기준이 있는가?
- 같은 요청이 두 번 들어오면 같은 결과를 반환하는가, 새 거래를 만드는가?
- 외부 콜백이 두 번 오면 상태가 한 번만 전이되는가?
- 이미 최종 상태인 데이터를 다시 처리하면 예외가 나는가, 무시되는가?
- unique constraint 만으로 충분한가, 애플리케이션 레벨 응답 재사용이 필요한가?

## 5. 실패 시나리오를 최소 3개 이상 만든다

정상 흐름보다 실패 흐름을 우선한다.

각 시나리오마다 다음을 짧게 분석한다.

- 데이터 정합성
- 사용자에게 보이는 상태
- 재시도 가능성
- 자동 복구 가능성
- 운영자가 수동 보정해야 하는지

예시:

```text
1. 외부 PG 요청 timeout
2. PG 승인 성공 후 내부 DB commit 실패
3. 내부 commit 성공 후 callback 전송 실패
4. 같은 결제 요청 재시도
5. 같은 callback 중복 수신
```

## 6. 대안은 선택지로 제시한다

정답처럼 단정하지 않는다.

현재 구조의 장점과 리스크를 먼저 분리한다.

그 다음 선택지를 비교한다.

| 선택지 | 장점 | 리스크 | 적합한 조건 |
|---|---|---|---|
| 동기 호출 유지 | 구현 단순, 응답 즉시성 | timeout/lock 점유, 실패 복구 부담 | 트래픽 낮고 외부 안정성이 높을 때 |
| 상태 기반 단계 분리 | 내부 상태 추적 쉬움 | 상태/보상 로직 증가 | 외부 성공 여부가 늦게 확정될 때 |
| 비동기 이벤트/outbox | commit 이후 전송 보장 강화 | 운영 복잡도 증가 | 메시지 유실과 재시도가 중요한 경우 |
| polling/reconciliation | 최종 정합성 회복 가능 | 지연과 별도 job 필요 | callback 유실 가능성이 있는 경우 |

## 8. 톤

- 구현을 비난하지 않는다. 리스크를 관찰한 사실로 설명한다.
- 코드 수정안을 바로 쓰기보다 책임, 경계, 상태 일관성을 먼저 말한다.
- "가능성이 있다"로 흐리지 말고 어떤 조건에서 문제가 되는지 구체화한다.
- 외부 시스템은 신뢰할 수 없다는 전제를 유지한다.

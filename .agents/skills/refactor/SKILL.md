# /refactor — 동작 보존 리팩토링

핵심 명제: **리팩토링은 외부 동작을 바꾸지 않고 내부 구조를 더 읽기 좋게 만든다.**

## 1. 먼저 분류한다

작업을 시작하기 전에 성격을 구분한다.

| 성격 | 예시 | 사용할 기준 |
|---|---|---|
| 리팩토링 | 이름 변경, 의미 있는 메서드 추출, 책임 위치 정리, 중복 축소 | 이 skill |
| 기능 추가 | 새 API, 새 상태, 새 예외, 새 분기 | `/tdd` |
| 설계 변경 | 도메인 경계, Facade/Service 트랜잭션, 패키지 재배치 | `/design` + 필요 시 `/tdd` |
| 버그 수정 | 기존 동작 오류 수정 | `/tdd` |

리팩토링 중 새 행위가 필요해지면 멈추고 기능/버그 작업으로 분리한다.

## 2. 범위를 작게 잡는다

- 요청받은 흐름과 직접 관련된 파일만 수정한다.
- 본인 변경으로 생긴 unused import, 죽은 코드만 정리한다.
- 기존 dead code, 오래된 스타일, 다른 도메인의 문제는 언급만 하고 건드리지 않는다.
- 포맷팅만 대량으로 바꾸지 않는다.
- 한 커밋에는 한 종류의 변경만 담는다. 기능/버그/리팩토링/포맷팅은 분리한다.

변경된 모든 라인은 사용자 요청 또는 현재 리팩토링 목적과 연결되어야 한다.

## 3. 안전망을 먼저 확인한다

리팩토링 전후 테스트 결과는 같아야 한다.

```text
1. 현재 보호 테스트 확인
2. 테스트가 있으면 먼저 실행해 baseline 확인
3. 테스트가 부족하면 핵심 흐름 characterization test 를 먼저 추가
4. 작은 리팩토링 1 단계 수행
5. 관련 테스트 재실행
6. 실패하면 즉시 원인 확인, 되돌리거나 범위 축소
```

테스트가 없는 핵심 로직을 바로 옮기지 않는다. 먼저 현재 동작을 고정한다.

## 4. 리스크 단계로 움직인다

| 리스크 | 예시 | 기준 |
|---|---|---|
| 낮음 | 지역 변수명 변경, private 메서드 추출, 중복 검증식 정리 | 관련 단위 테스트로 충분 |
| 중간 | 도메인 메서드 이동, Service 흐름 재배치, DTO 매핑 정리 | 단위 + 통합/E2E 일부 확인 |
| 높음 | 패키지 이동, Repository 경계 변경, 트랜잭션 위치 변경, 엔티티 관계 변경 | 사용자에게 방향 공유 후 진행 |

높은 리스크 작업은 "리팩토링" 이라는 이름으로 조용히 진행하지 않는다.

## 5. 이름과 추출은 의미가 있을 때만

줄 수를 줄이려고 메서드를 나누지 않는다. 이름을 붙였을 때 도메인 의미가 선명해야 한다.

```java
// Good
validateDuplicateProducts();
calculateTotalPrice();
deductStock();

// Bad
processData();
handleLogic();
doSomething();
execute();
```

이름이 내부 구현을 그대로 반복하면 추출하지 않는다.

## 6. 중복 제거보다 가독성

공통화는 세 가지를 모두 만족할 때만 한다.

1. 같은 이유로 변경되는 코드인가?
2. 공통화한 이름이 도메인 의미를 드러내는가?
3. 공통화 후 호출부가 더 읽기 쉬운가?

하나라도 애매하면 중복을 조금 허용한다.

```java
// 괜찮은 중복
validateDuplicateProducts();
validateDuplicateOrderItems();

// 의미가 흐려지는 공통화
validateByType(type, items, options, mapper, checker);
```

## 7. 책임 이동 기준

리팩토링 중 책임 위치가 어색하면 아래 기준을 따른다.

- 상태와 불변식을 가진 객체가 판단을 소유한다.
- Service 는 조회, 트랜잭션, 도메인 행위 호출 흐름을 조율한다.
- Facade 는 여러 도메인 Service 를 조합한다.
- Controller 는 요청 파싱과 응답 래핑만 한다.

책임 이동으로 동작이나 트랜잭션 경계가 바뀔 수 있으면 `/design` 판단으로 전환한다.

## 8. 레거시는 점진적으로

- 새 코드부터 현재 컨벤션을 적용한다.
- 기존 코드 전체 리라이트는 별도 작업으로 분리한다.
- 큰 파일을 만질 때도 현재 변경 흐름 주변만 개선한다.
- "겸사겸사" 정리는 하지 않는다.

## 보고 형식

긴 리팩토링에서는 의미 있는 체크포인트만 공유한다.

```text
[REFACTOR] <대상>: <구조 변경 요약> — 동작 보존, <검증 명령/결과>
```

행위 변경이 의심되면 멈추고 사용자에게 확인한다.

## 체크

- [ ] 외부 동작이 바뀌지 않았는가?
- [ ] 테스트 baseline 과 리팩토링 후 결과를 비교했는가?
- [ ] 변경 라인이 모두 리팩토링 목적과 연결되는가?
- [ ] 추출한 이름이 도메인 의미를 드러내는가?
- [ ] 공통화가 변경 이유, 이름, 가독성 기준을 모두 만족하는가?
- [ ] 트랜잭션/패키지/Repository 경계 변경은 사용자와 합의했는가?

# Spec: BRD-4 브랜드 등록

**소스**: `docs/volume-2/01-requirements.md` — BRD-4
**작성일**: 2026-05-25
**상태**: Draft

## 시나리오 요약

관리자가 새로운 브랜드를 등록한다. 관리자 인증을 통과한 요청만 허용하며, 브랜드 이름(1~50자)과 선택적 설명을 받아, 삭제되지 않은 기존 브랜드와 이름이 중복되지 않으면 새 브랜드를 생성하고 생성된 식별자를 반환한다. Brand 도메인의 첫 시나리오이자 모든 admin 시나리오의 첫 시나리오라, Brand aggregate 골격과 관리자 인증 토대를 함께 세운다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 정보를 보유하고 동일 이름의 활성 브랜드가 없을 때, **When** 이름과 설명으로 등록을 요청하면, **Then** 새 브랜드가 생성되고 생성된 브랜드 식별자가 반환된다(201).
2. **Given** 관리자 인증 정보를 보유하고 설명을 생략했을 때, **When** 이름만으로 등록을 요청하면, **Then** 설명 없이(null) 브랜드가 정상 생성된다.
3. **Given** 동일 이름의 브랜드가 이미 **삭제된** 상태로만 존재할 때, **When** 같은 이름으로 등록을 요청하면, **Then** 새 브랜드가 정상 생성된다(삭제된 이름은 중복 대상 아님).

### Exception Flow
1. **Given** 관리자 인증 정보가 없거나 `X-Loopers-Ldap` 값이 올바르지 않을 때, **When** 등록을 요청하면, **Then** 인증 실패로 응답한다(403 FORBIDDEN).
2. **Given** 관리자 인증을 통과했을 때, **When** 이름이 빈 문자열이거나 50자를 초과해 요청하면, **Then** 입력 검증 실패로 응답한다(400 BAD_REQUEST).
3. **Given** 동일 이름의 **삭제되지 않은** 브랜드가 이미 존재할 때, **When** 같은 이름으로 등록을 요청하면, **Then** 자원 충돌로 응답한다(409 CONFLICT).

### 비즈니스 규칙
- 브랜드 이름은 1자 이상 50자 이하 문자열이다.
- 설명은 선택 입력이다. 미입력 시 null로 저장하고, 입력된 값은 정규화 없이 그대로 저장한다. (결정 B)
- 이름 중복 검사 대상은 삭제되지 않은(활성) 브랜드뿐이다. 삭제된 브랜드 이름은 제외한다. (결정 7)
- 관리자 인증은 `X-Loopers-Ldap: loopers.admin` 헤더로 식별한다.

## 엣지 케이스

- 이름 경계값: 0자(빈 문자열, 실패) / 1자(통과) / 50자(통과) / 51자(실패).
- 설명: 미입력(null 저장) / 빈 문자열(그대로 저장) / 일반 문자열(그대로 저장).
- 삭제된 동일 이름 브랜드만 존재 → 등록 허용. 활성 동일 이름 존재 → 충돌.
- 권한 경계: 회원 인증 헤더(`X-Loopers-LoginId/Pw`)만 있고 admin 헤더가 없으면 인증 실패(403).
- 동시 등록 race로 인한 활성 이름 중복 → **범위 밖** (결정 C). 애플리케이션 선검사만 수행.

## 기능 요구사항

- **FR-001**: 시스템은 관리자 인증(`X-Loopers-Ldap: loopers.admin`)을 통과한 요청만 브랜드 등록을 허용해야 한다. 실패 시 403 FORBIDDEN으로 응답한다.
- **FR-002**: 시스템은 브랜드 이름이 1~50자 범위인지 검증해야 한다. 벗어나면 400으로 응답한다.
- **FR-003**: 시스템은 등록 이름이 삭제되지 않은 기존 브랜드와 중복되지 않는지 검사해야 한다(삭제된 브랜드 제외). 중복이면 409로 응답한다.
- **FR-004**: 시스템은 설명을 선택 입력으로 받아 미입력 시 null, 입력 시 값 그대로 저장해야 한다.
- **FR-005**: 시스템은 생성된 브랜드의 식별자를 반환해야 한다.

## 관련 엔티티

- **Brand** (신규 aggregate): 이름·설명 보유. 자기 생성·soft delete 책임. 다른 브랜드와의 중복 검사는 응용 계층 책임.
- **Name** (신규 VO, 브랜드 이름): 1~50자 검증을 생성 시점에 단일화.
- **설명(description)**: 검증·행위가 없어 VO 없이 `String` 필드로 직접 보유(선택).
- **BrandRepository** (신규): 저장, 활성 브랜드 이름 존재 검사, 저장.
- **신규 aggregate 골격 필요**: Brand(Model) / Name(VO) / BrandRepository.
- **신규 횡단 토대**: 관리자 인증 식별(`X-Loopers-Ldap` 헤더, 실패 시 FORBIDDEN) + `ErrorType.FORBIDDEN(403)` 추가. 이후 모든 admin 시나리오가 재사용.

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| VO/Model 단위 | Name | 1자·50자 통과, 빈 문자열·51자 예외(errorType BAD_REQUEST) |
| VO/Model 단위 | Brand | 생성 시 이름·설명(String) 보유, null 설명 허용 |
| Service/Facade 단위 | 브랜드 등록 유스케이스 | 활성 동일 이름 존재 시 CONFLICT, 삭제된 동일 이름은 통과, 정상 시 저장 후 식별자 반환 |
| Integration | BrandRepository | 저장·조회, 활성 이름 존재 검사가 삭제 행을 제외하는지 |
| E2E | `POST /api-admin/v1/brands` | 201 + meta.result SUCCESS + 응답에 식별자 / admin 인증 실패 403 / 이름 범위 위반 400 / 활성 이름 중복 409 (meta.result + errorCode까지, 메시지 문구는 단언 안 함) |

## 관련 결정

- **결정 7 (도메인별 soft delete)**: 브랜드는 soft delete. 삭제된 브랜드 이름은 중복 검사에서 제외된다.
- **본 cycle 신규 결정 (admin 인증 응답)**: 관리자 인증 실패는 403 FORBIDDEN으로 통일. `ErrorType.FORBIDDEN` 추가. 이후 모든 admin 시나리오에 적용.
- **본 cycle 신규 결정 (설명 입력)**: 설명은 선택, 값 정규화 없이 그대로 저장.
- **본 cycle 신규 결정 (중복 동시성)**: 애플리케이션 선검사만. 동시 등록 race 방어는 범위 밖.

## 성공 기준 / 범위 밖

- **성공**: 위 모든 수용 시나리오·테스트 계획이 green. `POST /api-admin/v1/brands`가 인증·검증·중복·생성 분기를 명세대로 처리.
- **범위 밖**: 브랜드 조회·목록·수정·삭제(다른 cycle), 동시 등록 race 방어, 브랜드 이름 DB UNIQUE 제약.

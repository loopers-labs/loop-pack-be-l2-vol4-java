# Spec: LIK-2 좋아요 취소

**소스**: `docs/volume-2/01-requirements.md` — LIK-2
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

로그인한 회원이 특정 상품에 눌렀던 좋아요를 취소한다. 회원 인증을 통과한 요청만 허용하며, 대상 상품이 존재하고 삭제되지 않았는지 확인한 뒤, 해당 회원·상품 조합의 좋아요 기록이 있으면 삭제하고 없으면 별도 동작 없이 정상 응답으로 마무리한다(멱등 취소). 좋아요는 hard delete(행 제거)다(결정 7). LIK-1에서 세운 Like aggregate에 회원·상품 조합 삭제 기능을 더한다. 회원 인증·상품 활성 검증은 기존 것을 재사용한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 로그인한 회원과 활성 상품이 있고 해당 상품을 좋아요한 상태일 때, **When** 좋아요 취소를 요청하면, **Then** 좋아요 행이 삭제(hard delete)되고 취소 완료가 정상 응답된다(200 SUCCESS).
2. **Given** 로그인한 회원과 활성 상품이 있으나 좋아요한 적이 없을 때, **When** 취소를 요청하면, **Then** 별도 동작 없이 정상 응답으로 마무리한다(200 SUCCESS, 멱등 취소).

### Alternate Flow
1. **Given** 동일 회원·동일 상품에 짧은 시간 안에 취소 요청이 여러 건 들어올 때, **When** 처리되면, **Then** 결과 상태는 한 번 취소한 것과 동일하다(좋아요 행 없음).

### Exception Flow
1. **Given** 회원 인증 정보가 없거나 `X-Loopers-LoginId/LoginPw`가 올바르지 않을 때, **When** 취소를 요청하면, **Then** 인증 실패로 응답한다(401 UNAUTHENTICATED).
2. **Given** 대상 상품이 존재하지 않거나 이미 삭제된 상태일 때, **When** 취소를 요청하면, **Then** 자원을 찾을 수 없다고 응답한다(404 NOT_FOUND).

### 비즈니스 규칙
- 좋아요 취소 동작은 진입 경로(상품 목록/상세/내 좋아요 목록)와 무관하게 같다.
- 좋아요 기록이 없는 상품에 취소 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 취소).
- 삭제된 상품에 대한 좋아요 동작은 등록·취소 모두 일관되게 차단한다(상품 부재/삭제 → 404). LIK-1 등록 차단과 대칭.

## 엣지 케이스

- 대상 상품 상태: 활성(취소 처리) / 미존재(404) / 삭제됨(404).
- 좋아요 상태: 좋아요함(삭제) / 좋아요 안 함(no-op, 멱등).
- 멱등성: 같은 상품에 취소를 두 번 요청해도 결과 동일(행 없음), 두 응답 모두 정상.
- 권한 경계: 회원 인증 헤더 누락·불일치 → 401.
- 다른 회원의 같은 상품 좋아요 행은 영향받지 않는다(취소는 인증된 회원 본인 행만).

## 기능 요구사항

- **FR-001**: 시스템은 회원 인증(`X-Loopers-LoginId/LoginPw`)을 통과한 요청만 좋아요 취소를 허용해야 한다. 실패 시 401 UNAUTHENTICATED로 응답한다.
- **FR-002**: 시스템은 대상 상품이 존재하고 삭제되지 않았는지 검증해야 한다. 아니면 404 NOT_FOUND로 응답한다.
- **FR-003**: 시스템은 인증된 회원·상품 조합의 좋아요 기록이 있으면 hard delete 하고, 없으면 별도 동작 없이 정상 응답(200 SUCCESS)으로 마무리해야 한다(멱등 취소).
- **FR-004**: 취소는 인증된 회원 본인의 좋아요 행만 대상으로 하며, 다른 회원의 행은 영향받지 않아야 한다.

## 관련 엔티티

- **Like** (재사용): LIK-1에서 세운 aggregate. 취소는 도메인 행위가 아니라 저장소 hard delete(행 제거).
- **LikeRepository** (편집): 회원·상품 조합 좋아요를 hard delete 하는 기능을 추가(없으면 no-op).
- **재사용**: `ProductRepository`(상품 활성 존재 검증 — 없으면 NOT_FOUND), `@LoginUser AuthenticatedUser`(회원 인증), `ErrorType`(NOT_FOUND·UNAUTHENTICATED).

## 관련 결정

- **결정 6 (DELETE 멱등 처리 — LIK-2는 C 패턴)**: 부모(상품)의 부재는 자원 부재(404), 삭제 대상 자신(좋아요 행)의 부재는 멱등(no-op). 좋아요는 부수 자원이라 상품 부재는 404로, 좋아요 행 부재는 멱등으로 통일. (BRD-6·PRD-7의 완전 멱등 B와 의도된 차이.)
- **결정 7 (소프트 삭제 C — 좋아요 hard delete)**: 취소는 좋아요 행을 실제로 제거(soft delete 아님). hard delete라 재등록 시 UNIQUE 충돌 없음.
- **LIK-3 전제**: 좋아요한 상품 목록에서 삭제 상품은 노출되지 않아 취소 진입 자체가 닫혀 있고, 잘못된 식별자 직접 호출은 404로 응답하는 일관성. (LIK-3은 본 cycle 범위 밖.)

## 성공 기준 / 범위 밖

- **성공**: 위 모든 수용 시나리오·테스트 계획이 green. `DELETE /api/v1/products/{productId}/likes`가 인증·상품 검증·멱등 취소(hard delete) 분기를 명세대로 처리.
- **범위 밖**: 좋아요 등록(LIK-1), 좋아요한 상품 목록(LIK-3), 좋아요 수 집계, 좋아요 이력 보존(hard delete라 이력 없음).

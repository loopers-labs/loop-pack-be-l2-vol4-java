## 시나리오 요약

관리자가 페이지 정보를 포함해 등록된 브랜드 목록을 조회한다. 관리자 인증을 통과한 요청만 허용하며, 삭제되지 않은 브랜드 한 페이지 분량을 등록 시각 내림차순으로, 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)와 함께 반환한다. 각 항목은 식별자·이름·설명·등록 시각·갱신 시각을 포함한다. Brand 도메인 첫 페이지네이션 시나리오이다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 + 활성 브랜드들이 존재, **When** page·size로 목록 요청, **Then** 해당 페이지 항목들과 페이지 메타가 등록 시각 내림차순으로 반환된다(200). 각 항목은 식별자·이름·설명·등록 시각·갱신 시각.
2. **Given** 관리자 인증 + 활성 브랜드 없음, **When** 목록 요청, **Then** 빈 목록과 페이지 메타가 반환된다(200).

### Exception Flow
1. **Given** 관리자 인증 실패, **When** 목록 요청, **Then** 403 FORBIDDEN.

### 비즈니스 규칙
- 삭제된 브랜드는 응답에서 제외.
- 페이지 크기 미지정 시 20건.
- 결과는 등록 시각 내림차순 정렬.

## 엣지 케이스
- 빈 결과(빈 목록 + 메타) / size 미지정(20 적용).
- 마지막 페이지 부분 채움.

## 기능 요구사항
- **FR-001**: 시스템은 관리자 인증 통과 요청만 허용한다(실패 403).
- **FR-002**: 시스템은 page·size로 페이징하며 기본값은 page=0, size=20이다.
- **FR-003**: 시스템은 삭제되지 않은 브랜드만 등록 시각 내림차순으로 페이징한다.
- **FR-004**: 시스템은 페이지 메타(현재 페이지·크기·총 페이지·총 항목)를 포함한다.
- **FR-005**: 각 항목에 식별자·이름·설명·등록 시각·갱신 시각을 포함한다.

## 관련 엔티티
- **BrandModel**: 읽기.
- **BrandRepository**: 활성 브랜드 페이지 조회(정렬·페이징) 신규.
- **신규 토대(plan에서 결정)**: 페이지네이션 표현·전달 방식(목록 응답 메타 구조). 이후 상품·주문 목록과 공유.

## 테스트 계획
| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Integration | BrandRepository | 활성만·등록 시각 내림차순·페이징(크기/오프셋)·총 개수 |
| E2E | `GET /api-admin/v1/brands?page&size` | 200 + 메타 키(페이지·크기·총 페이지·총 항목) + 항목 키(id·name·description·시각) / 빈 결과 200 / 403 |

## 관련 결정
- 결정 7(soft delete): 삭제 브랜드 제외.
- admin 인증 403.
- 페이지네이션 접근(Spring Data 노출 vs 도메인 PageResult)은 **plan에서 결정**.

## 성공 기준 / 범위 밖
- 성공: 위 수용 시나리오·테스트 green. `GET /api-admin/v1/brands`가 인증·필터·정렬·페이징·메타를 명세대로 처리.
- 범위 밖: 정렬 기준 다양화, 검색 필터.

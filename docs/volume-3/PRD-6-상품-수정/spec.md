# Spec: PRD-6 상품 수정

**소스**: `docs/volume-2/01-requirements.md` — PRD-6
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

관리자가 대상 상품과 새 이름·설명·가격·재고 수량을 포함해 상품 정보를 수정한다. 관리자 인증을 통과한 요청만 허용하며, 대상 상품이 존재하고 삭제되지 않았는지 확인한 뒤, 이름(1~100자)·가격(0 이상 정수)·재고(0 이상 정수) 검증을 통과하면 상품 정보를 갱신하고 갱신된 식별자를 반환한다. 소속 브랜드는 수정 대상이 아니다(브랜드를 바꾸려면 삭제 후 재등록). PRD-5에서 세운 Product aggregate에 `update` 행위와 활성 단건 조회를 추가한다. 관리자 인증 토대는 BRD-4에서 만든 것을 재사용한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 정보를 보유하고 대상 상품이 활성 상태일 때, **When** 유효한 이름·설명·가격·재고로 수정을 요청하면, **Then** 상품 정보가 갱신되고 갱신된 상품 식별자가 반환된다(200).
2. **Given** 관리자 인증 정보를 보유하고 설명을 생략했을 때, **When** 이름·가격·재고만으로 수정을 요청하면, **Then** 설명이 null로 갱신된다(PUT 전체 치환 의미 — BRD-5 수정 선례).
3. **Given** 가격 0·재고 0(경계 최소값)으로 요청할 때, **When** 수정을 요청하면, **Then** 정상 갱신된다.

### Exception Flow
1. **Given** 관리자 인증 정보가 없거나 `X-Loopers-Ldap` 값이 올바르지 않을 때, **When** 수정을 요청하면, **Then** 인증 실패로 응답한다(403 FORBIDDEN).
2. **Given** 대상 상품이 존재하지 않거나 이미 삭제된 상태일 때, **When** 수정을 요청하면, **Then** 자원을 찾을 수 없다고 응답한다(404 NOT_FOUND).
3. **Given** 관리자 인증을 통과했을 때, **When** 이름이 빈 문자열이거나 100자를 초과해 요청하면, **Then** 입력 검증 실패로 응답한다(400 BAD_REQUEST).
4. **Given** 관리자 인증을 통과했을 때, **When** 가격이 0 미만이거나 재고가 0 미만으로 요청하면, **Then** 입력 검증 실패로 응답한다(400 BAD_REQUEST).

### 비즈니스 규칙
- 소속 브랜드는 수정할 수 없다. 수정 요청에 브랜드 식별자를 받지 않으며, 브랜드를 바꾸려면 상품을 삭제하고 새로 등록한다.
- 재고 수량의 임의 갱신은 관리자 권한 안에서만 가능하다. 회원의 주문 흐름은 차감 방향으로만 동작한다. (결정 2 참조)
- 이름·가격·재고는 등록과 동일한 정책(1~100자 / 0 이상 / 0 이상)으로 재검증한다.
- 설명은 선택 입력이며, 미입력 시 null로 치환하고 입력된 값은 정규화 없이 그대로 저장한다.

## 엣지 케이스

- 이름 경계값: 0자(빈 문자열, 실패) / 1자(통과) / 100자(통과) / 101자(실패).
- 가격 경계값: -1(실패) / 0(통과).
- 재고 경계값: -1(실패) / 0(통과).
- 설명: 미입력(null 치환) / 빈 문자열(그대로 저장) / 일반 문자열(그대로 저장).
- 대상 상태: 활성(통과) / 미존재(404) / 삭제됨(404).
- 권한 경계: 회원 인증 헤더(`X-Loopers-LoginId/Pw`)만 있고 admin 헤더가 없으면 인증 실패(403).
- 동일 값으로 수정 요청해도 정상 처리되며 갱신 시각만 새로 찍힌다(BRD-5 수정 선례).

## 기능 요구사항

- **FR-001**: 시스템은 관리자 인증(`X-Loopers-Ldap: loopers.admin`)을 통과한 요청만 상품 수정을 허용해야 한다. 실패 시 403 FORBIDDEN으로 응답한다.
- **FR-002**: 시스템은 대상 상품이 존재하고 삭제되지 않았는지 검증해야 한다. 아니면 404 NOT_FOUND로 응답한다.
- **FR-003**: 시스템은 상품 이름이 1~100자 범위인지 검증해야 한다. 벗어나면 400으로 응답한다.
- **FR-004**: 시스템은 가격이 0 이상의 정수인지 검증해야 한다. 미만이면 400으로 응답한다.
- **FR-005**: 시스템은 재고가 0 이상의 정수인지 검증해야 한다. 미만이면 400으로 응답한다.
- **FR-006**: 시스템은 소속 브랜드를 수정 대상에서 제외해야 한다(요청에 브랜드 식별자를 받지 않음).
- **FR-007**: 시스템은 설명을 선택 입력으로 받아 미입력 시 null, 입력 시 값 그대로 갱신해야 한다.
- **FR-008**: 시스템은 갱신된 상품의 식별자를 반환해야 한다.

## 관련 엔티티

- **ProductModel** (편집): 이름·설명·가격·재고를 갱신하는 `update` 행위를 추가. 갱신 시 `Name`·`Price`·`Stock` VO로 재검증한다. 브랜드 식별자는 갱신 대상이 아니다.
- **Name / Price / Stock** (재사용): 수정 시점에도 동일 검증 규칙을 적용. (PRD-5에서 도입)
- **ProductRepository** (편집): 활성 단건 조회(없거나 삭제됐으면 NOT_FOUND)를 추가. (BRD-5 `getActiveById` 선례 — find/get 컨벤션)
- **재사용**: `AdminAuthInterceptor`(admin 인증), `ErrorType`(BAD_REQUEST·NOT_FOUND·FORBIDDEN).

| VO/Model 단위 | ProductModel.update | 이름·설명·가격·재고 갱신 / 새 이름 위반 BAD_REQUEST / 새 가격 음수 BAD_REQUEST / 새 재고 음수 BAD_REQUEST |

| E2E | `PUT /api-admin/v1/products/{productId}` | 200 + meta.result SUCCESS + productId & 값 갱신 / admin 인증 실패 403 / 대상 미존재 404 / 이름 101자 400 / 가격 -1 400 (statusCode + meta.result + errorCode까지, 메시지 문구는 단언 안 함) |

## 관련 결정

- **결정 2 (주문 시점 재고 차감 A)**: 재고 임의 갱신은 관리자 권한 안에서만. 회원 주문 흐름은 차감 방향으로만 동작 → 수정 API는 절대값 갱신, 주문은 별도 차감 경로(후속 cycle).
- **본 cycle 신규 결정 (브랜드 변경 불가)**: 수정 요청 DTO에 브랜드 식별자를 받지 않는다. 브랜드 변경은 삭제 후 재등록으로 처리(요구사항 명시).
- **재사용**: 관리자 인증(BRD-4 `AdminAuthInterceptor`), 입력 검증 단일화(`Name`/`Price`/`Stock` VO — PRD-5).
- **PUT 치환 의미 (BRD-5 선례)**: 설명 미입력은 null 치환. 부분 수정(PATCH)이 아닌 전체 치환.

## 성공 기준 / 범위 밖

- **성공**: 위 모든 수용 시나리오·테스트 계획이 green. `PUT /api-admin/v1/products/{productId}`가 인증·대상 검증·입력 검증·갱신 분기를 명세대로 처리.
- **범위 밖**: 상품 삭제(PRD-7), 브랜드 변경, 재고 차감(주문), 상품 조회·목록, 부분 수정(PATCH).

# Spec: PRD-2 상품 상세 (public, 좋아요 수)

**소스**: `docs/volume-2/01-requirements.md` — PRD-2
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

회원·비회원 누구나 로그인 없이 특정 상품의 상세 정보를 조회한다. 시스템은 대상 상품이 삭제되지 않고 존재하는지 확인한 뒤, 식별자·이름·설명·소속 브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 반환한다. 관리자 전용 정보(정확한 재고 수량·등록/갱신 시각·운영 메모)는 노출하지 않는다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 활성 브랜드 소속의 활성 상품 + 좋아요 일부 존재, **When** productId로 상세 요청, **Then** 200과 함께 식별자·이름·설명·브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수가 반환된다.
2. **Given** 좋아요가 한 건도 없는 활성 상품, **When** 상세 요청, **Then** 좋아요 수 0으로 반환된다.

### Exception Flow
1. **Given** 존재하지 않는 productId, **When** 상세 요청, **Then** 404 NOT_FOUND.
2. **Given** 삭제된 상품(브랜드 cascade 삭제 포함), **When** 상세 요청, **Then** 404 NOT_FOUND.

### 비즈니스 규칙
- 관리자 전용 정보(정확한 재고 수량·등록/갱신 시각·운영 메모)는 응답에 포함하지 않는다.
- 재고는 정확한 수량 대신 가용 여부(가용 여부 = 재고 1개 이상)만 노출.
- 좋아요 수는 좋아요 테이블에서 매 조회마다 집계 (결정 1).
- 응답에 노출되는 상품·소속 브랜드는 모두 삭제되지 않은 상태여야 한다. 브랜드 삭제 시 소속 상품 cascade 삭제(BRD-6)로 보장된다.

## 엣지 케이스
- 좋아요 0건 상품 → 좋아요 수 0.
- 재고 0 상품 → 가용 여부 false, 그래도 상세는 조회됨.
- 미존재·삭제 상품 → 404(동일 응답으로 존재 여부 비노출).

## 기능 요구사항

- **FR-001**: 시스템은 로그인 없이 상세 조회를 허용한다.
- **FR-002**: 시스템은 대상 상품이 삭제되지 않고 존재할 때만 응답하며, 미존재·삭제 시 404로 응답한다.
- **FR-003**: 시스템은 식별자·이름·설명·브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 반환한다.
- **FR-004**: 시스템은 정확한 재고 수량·등록/갱신 시각 등 관리자 전용 정보를 노출하지 않는다.
- **FR-005**: 좋아요 수는 매 조회마다 좋아요 테이블에서 집계한 값을 사용한다.

## 관련 엔티티
- **ProductModel**: 읽기(식별자·이름·설명·brandId·가격·재고).
- **BrandModel**: 브랜드 식별자·이름 표시를 위한 조인 대상.
- **LikeModel**: 해당 상품의 좋아요 수 집계 원천.
- **신규 토대(plan에서 결정)**: 상품 상세 projection(PRD-1 목록 항목 + 설명). 좋아요 수 집계 + 브랜드 조인 단건 조회.

## 테스트 계획
| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Integration | ProductRepository(신규 단건 조회) | 활성 상품 + 브랜드명 조인 / 좋아요 수 집계 정확(0건 포함) / 삭제·미존재 → 부재 |
| E2E | `GET /api/v1/products/{productId}` | 200 + 응답 키(식별자·이름·설명·브랜드·가격·재고 가용 여부·좋아요 수) / 관리자 전용 필드(정확한 재고·시각) 미포함 / 미존재 404 / 삭제 상품 404 |

## 관련 결정
- 결정 1(좋아요 누적 카운트): 매 조회 집계.
- 결정 7(soft delete): 삭제 상품 404.
- BRD-6 cascade: 브랜드 삭제 시 상품 cascade 삭제로 "삭제 브랜드 소속 상품 비노출" 보장. 단건 조회 시 brand.deletedAt 방어 필터 적용 여부는 **plan에서 결정**.

## 성공 기준 / 범위 밖
- 성공: 위 수용 시나리오·테스트 green. `GET /api/v1/products/{productId}`가 활성 판정·브랜드명·재고 가용 여부·좋아요 수를 명세대로 처리.
- 범위 밖: 관리자 전용 상세(PRD-4), 좋아요 수 캐싱/컬럼화.

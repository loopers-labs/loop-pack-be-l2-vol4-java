# Spec: ORD-6 관리자 주문 상세

**소스**: `docs/volume-2/01-requirements.md` — ORD-6
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

관리자가 특정 주문 식별자로 주문 상세를 조회한다. 관리자 인증을 통과한 요청만 허용하며, 주문이 존재하면 주문의 식별자·주문한 회원의 식별자·상태·주문 시각·총 결제 금액·주문 항목 전체(상품 식별자·상품명·브랜드명·단가·수량 스냅샷)를 반환한다. 본인 소유 검증이 없다는 점에서 ORD-4(본인 상세)와 다르며, 응답에 주문한 회원의 식별자가 포함된다는 점에서 더 넓다. 주문이 존재하지 않으면 자원 부재(404)로 응답한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 + 존재하는 주문, **When** 그 주문 식별자로 상세를 요청하면, **Then** 식별자·회원 식별자·상태·주문 시각·총 결제 금액·주문 항목 전체(상품 식별자·상품명·브랜드명·단가·수량)가 반환된다(200).

### Exception Flow
1. **Given** 관리자 인증 실패, **When** 상세를 요청하면, **Then** 403 FORBIDDEN.
2. **Given** 대상 주문이 존재하지 않을 때, **When** 상세를 요청하면, **Then** 404 NOT_FOUND.

### 비즈니스 규칙
- 주문 항목 정보는 주문 시점 스냅샷을 그대로 표시한다. (결정 5)
- 소유자 검증 없이 어떤 주문이든 조회 가능(관리자 권한). 응답에 주문한 회원 식별자를 포함한다.

## 엣지 케이스
- 존재 주문(200) / 미존재 주문(404).
- 주문 항목 여러 개 → 항목 전체 반환.
- 권한 경계: 회원 인증 헤더만 있고 admin 헤더 없으면 403.
- 스냅샷 보존: 주문 후 상품·브랜드 변경·삭제되어도 항목 스냅샷 유지.

## 기능 요구사항

- **FR-001**: 시스템은 관리자 인증(`X-Loopers-Ldap: loopers.admin`)을 통과한 요청만 허용해야 한다. 실패 시 403.
- **FR-002**: 시스템은 주문이 존재하지 않으면 404로 응답해야 한다.
- **FR-003**: 시스템은 주문의 식별자·주문한 회원의 식별자·상태·주문 시각·총 결제 금액과 주문 항목 전체(상품 식별자·상품명·브랜드명·단가·수량 스냅샷)를 반환해야 한다.

## 관련 엔티티

- **OrderModel**: 회원 식별자·상태·주문 시각·총액 조회 원천(소유 검증 없음).
- **OrderItemModel**: 항목 스냅샷(상품 식별자·상품명·브랜드명·단가·수량) 표시 원천.
- **재사용**: `AdminAuthInterceptor`(관리자 인증), `OrderRepository`(주문+항목 조회), `ErrorType`(NOT_FOUND·FORBIDDEN).
- **조회 형태**: 관리자 상세는 회원 식별자를 포함하므로 ORD-4 본인 상세와 응답 필드가 다르다. 조회 메서드·projection 공유 여부는 plan에서 결정.

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Service/Facade 단위 | 관리자 주문 상세 유스케이스 | 미존재 → NOT_FOUND, 존재 → 회원 식별자 포함 상세 반환(항목 포함) |
| Integration | OrderRepository | 주문+항목 함께 조회(소유자 필터 없음) |
| E2E | `GET /api-admin/v1/orders/{orderId}` | 200 + 응답 키(식별자·회원 식별자·상태·주문 시각·총액·항목 전체) / admin 인증 실패 403 / 미존재 404 (statusCode + meta.result + errorCode) |

## 관련 결정

- **결정 5 (스냅샷)**: 항목은 주문 시점 스냅샷 그대로 표시.
- **ORD-4 대비**: 소유 검증 없음(관리자), 회원 식별자 노출. 타인 주문을 404로 숨기는 ORD-4와 달리 관리자는 모든 주문을 본다.

## 성공 기준 / 범위 밖

- **성공**: 위 수용 시나리오·테스트가 green. `GET /api-admin/v1/orders/{orderId}`가 관리자 인증·미존재 404·회원 식별자 포함 항목 상세를 명세대로 처리.
- **범위 밖**: 본인 상세(ORD-4), 주문 목록(ORD-5), 주문 취소/수정, 소프트 삭제된 주문 노출 정책(본 라운드 삭제 API 없음).

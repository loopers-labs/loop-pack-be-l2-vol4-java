# Spec: PRD-1 상품 목록 (public, 좋아요 수)

**소스**: `docs/volume-2/01-requirements.md` — PRD-1
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

회원·비회원 누구나 로그인 없이 상품 목록을 조회한다. 브랜드 필터·정렬 기준·페이지 정보를 선택적으로 지정하며, 삭제되지 않고 소속 브랜드도 삭제되지 않은 상품 한 페이지 분량을 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)와 함께 반환한다. 각 항목은 식별자·이름·소속 브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 포함한다. Product 도메인의 첫 public 조회이자, 다중 정렬·브랜드 필터·좋아요 집계를 결합한 첫 시나리오이다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 활성 상품들이 활성 브랜드 소속으로 존재, **When** page·size로 목록 요청, **Then** 해당 페이지 항목들과 페이지 메타가 반환된다(200). 각 항목은 식별자·이름·브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수.
2. **Given** sort 미지정, **When** 목록 요청, **Then** 최신 등록 순(등록 시각 내림차순)으로 반환된다.
3. **Given** sort=가격 오름차순, **When** 목록 요청, **Then** 가격 오름차순으로 반환된다.
4. **Given** 상품마다 좋아요 수가 다름 + sort=좋아요 많은 순, **When** 목록 요청, **Then** 좋아요 수 내림차순으로 반환된다.
5. **Given** 특정 브랜드의 활성 상품 + 다른 브랜드 상품, **When** brandId 필터로 요청, **Then** 해당 브랜드 활성 상품만 반환된다.
6. **Given** 조건에 맞는 활성 상품 없음, **When** 목록 요청, **Then** 빈 목록과 페이지 메타가 반환된다(200).

### Exception Flow
1. **Given** 허용되지 않는 정렬 값, **When** 목록 요청, **Then** 400 BAD_REQUEST.

### 비즈니스 규칙
- 정렬 기준은 최신 등록 순(기본)·가격 오름차순·좋아요 많은 순 셋 중 하나.
- 페이지 크기 미지정 시 20건.
- 삭제된 상품과 소속 브랜드가 삭제된 상품은 응답에서 제외.
- 재고는 정확한 수량 대신 가용 여부(가용 여부 = 재고 1개 이상)만 노출.
- brandId로 지정한 브랜드가 존재하지 않거나 삭제됐으면 빈 목록.
- 좋아요 수는 좋아요 테이블에서 매 조회마다 집계 (결정 1).

## 엣지 케이스
- 빈 결과(빈 목록 + 메타) / size 미지정(20 적용) / 마지막 페이지 부분 채움.
- 좋아요 0건 상품도 좋아요 수 0으로 노출되고 정렬에 참여.
- 재고 0 상품은 가용 여부 false로 노출되되 목록에는 포함.
- brandId가 미존재·삭제 브랜드 → 빈 목록(에러 아님).

## 기능 요구사항

- **FR-001**: 시스템은 로그인 없이 목록 조회를 허용한다.
- **FR-002**: 시스템은 brandId(선택)·sort(최신·가격 오름차순·좋아요 많은 순, 기본 최신)·page(기본 0)·size(기본 20)를 수신하고, sort가 허용 값 밖이면 400으로 거절한다.
- **FR-003**: 시스템은 삭제되지 않고 소속 브랜드도 삭제되지 않은 상품만 조회한다.
- **FR-004**: brandId 지정 시 해당 브랜드 활성 상품만 반환하고, 브랜드가 미존재·삭제 상태면 빈 목록을 반환한다.
- **FR-005**: 시스템은 정렬 기준대로(최신=등록 시각 내림차순, 가격 오름차순, 좋아요 많은 순=좋아요 수 내림차순) 페이징한다.
- **FR-006**: 각 항목에 식별자·이름·브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 포함한다.
- **FR-007**: 좋아요 수는 매 조회마다 좋아요 테이블에서 집계한 값을 사용한다.
- **FR-008**: 시스템은 페이지 메타(현재 페이지·크기·총 페이지·총 항목)를 포함한다.

## 관련 엔티티
- **ProductModel**: 읽기(식별자·이름·brandId·가격·재고).
- **BrandModel**: 브랜드 식별자·이름 표시를 위한 조인 대상, 삭제 브랜드 제외 판정.
- **LikeModel**: 상품별 좋아요 수 집계 원천.
- **신규 토대(plan에서 결정)**: 상품 목록 projection(식별자·이름·브랜드 식별자·브랜드명·가격·재고 가용 여부·좋아요 수). PRD-2·LIK-3와 공유. 정렬·필터·페이징·좋아요 집계·브랜드 조인을 수행하는 조회 메서드.

## 관련 결정
- 결정 1(좋아요 누적 카운트): 매 조회 집계 채택.
- 결정 7(soft delete): 삭제 상품 제외.
- BRD-6 cascade: 브랜드 삭제 시 소속 상품이 cascade soft delete되므로 "소속 브랜드 삭제 상품 제외"는 상품 자신의 삭제 여부로 보장된다. 다만 목록 쿼리가 브랜드명 표시를 위해 brands를 조인하므로, brand.deletedAt 필터를 방어적으로 함께 적용할지는 **plan에서 결정**.
- 페이지네이션 접근(Spring Data `Pageable`/`Page` + 자체 `PageResponse` DTO)은 BRD-2 패턴을 따르되 **plan에서 확정**.

## 성공 기준 / 범위 밖
- 성공: 위 수용 시나리오·테스트 green. `GET /api/v1/products`가 필터·정렬·페이징·집계·메타를 명세대로 처리.
- 범위 밖: 검색어·카테고리 필터, 위 3종 외 정렬 옵션, 좋아요 수 캐싱/컬럼화(결정 1의 B·C안).

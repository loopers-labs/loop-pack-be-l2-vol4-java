# Spec: BRD-3 브랜드 상세 (admin)

**소스**: `docs/volume-2/01-requirements.md` — BRD-3
**작성일**: 2026-05-25
**상태**: Draft

## 시나리오 요약

관리자가 특정 브랜드의 상세 정보를 조회한다. 관리자 인증을 통과한 요청만 허용하며, 대상 브랜드가 활성 상태로 존재하면 식별자·이름·설명·등록 시각·갱신 시각을 반환한다. (대고객 BRD-1과 달리 등록·갱신 시각을 포함한다.)

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 + 활성 브랜드 존재, **When** 식별자로 상세 요청, **Then** 식별자·이름·설명·등록 시각·갱신 시각이 반환된다(200).

### Exception Flow
1. **Given** 관리자 인증 실패, **When** 상세 요청, **Then** 403 FORBIDDEN.
2. **Given** 대상 브랜드가 부재거나 이미 삭제됨, **When** 상세 요청, **Then** 404 NOT_FOUND.

### 비즈니스 규칙
- 활성 브랜드만 노출.
- 관리자 응답은 등록·갱신 시각을 포함(대고객 BRD-1과 대비).

## 엣지 케이스
- 삭제된 브랜드(404) / 존재하지 않는 id(404).
- 설명이 null인 브랜드(설명 null로 반환).

## 기능 요구사항
- **FR-001**: 시스템은 관리자 인증 통과 요청만 허용한다(실패 403).
- **FR-002**: 시스템은 활성 브랜드만 노출한다(부재/삭제 404).
- **FR-003**: 시스템은 식별자·이름·설명·등록 시각·갱신 시각을 반환한다.

## 관련 엔티티
- **BrandModel**: 읽기.
- **BrandRepository**: 활성 브랜드 단건 조회 — BRD-1·BRD-5와 공유.
- **표현 계층**: 기존 `BrandAdminV1Controller`에 GET 상세 추가, admin 상세 응답 DTO 신규.

## 테스트 계획
| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Service/Facade 단위 | 상세 조회 유스케이스 | 활성 존재 시 정보 반환(시각 포함), 부재/삭제 NOT_FOUND |
| E2E | `GET /api-admin/v1/brands/{brandId}` | 200 + 응답 키(id·name·description·등록/갱신 시각) / 403 / 404 |

(활성 단건 조회 Integration은 BRD-1과 공유 — 중복 작성하지 않음)

## 관련 결정
- 결정 7(soft delete): 삭제 브랜드 제외.
- admin 인증 403. 대고객/관리자 응답 shape 분리(시각 포함 여부).

## 성공 기준 / 범위 밖
- 성공: 위 수용 시나리오·테스트 green. `GET /api-admin/v1/brands/{brandId}`가 인증·존재·필드(시각 포함)를 명세대로 처리.
- 범위 밖: 운영 메모 등 추가 관리자 필드.

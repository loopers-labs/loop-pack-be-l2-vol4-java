# Spec: BRD-6 브랜드 삭제 (+상품 cascade)

**소스**: `docs/volume-2/01-requirements.md` — BRD-6, `02-sequence-diagrams.md` — BRD-6
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

관리자가 대상 브랜드를 삭제하면, 브랜드 자신과 소속된 삭제되지 않은 상품들이 **같은 트랜잭션 안에서** 함께 soft delete 처리된다. 데이터베이스 차원의 연쇄 제약이 아니라 애플리케이션이 같은 거래 단위로 cascade를 처리한다(둘 다 성공하거나 둘 다 실패). 관리자 인증을 통과한 요청만 허용하며, 존재하지 않거나 이미 삭제된 브랜드에 삭제 요청이 들어와도 별도 동작 없이 정상 응답으로 마무리한다(멱등 삭제, 결정 6). 삭제는 soft delete(결정 7)다. 브랜드 삭제 유스케이스가 상품 도메인의 일괄 삭제를 호출하므로, 응용 계층이 Brand·Product 두 저장소를 한 트랜잭션으로 조율한다(ProductFacade가 BrandRepository를 참조한 선례와 동형). 관리자 인증 토대는 BRD-4에서 만든 것을 재사용한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 정보를 보유하고 대상 브랜드가 활성 상태이며 소속 활성 상품이 있을 때, **When** 브랜드 삭제를 요청하면, **Then** 브랜드와 소속 활성 상품이 모두 soft delete 처리되고 삭제 완료가 정상 응답된다(200 SUCCESS).
2. **Given** 관리자 인증 정보를 보유하고 대상 브랜드가 활성이며 소속 상품이 없을 때, **When** 삭제를 요청하면, **Then** 브랜드만 soft delete 처리되고 정상 응답된다.
3. **Given** 관리자 인증 정보를 보유하고 대상 브랜드가 존재하지 않거나 이미 삭제된 상태일 때, **When** 삭제를 요청하면, **Then** 별도 동작 없이 정상 응답으로 마무리한다(200 SUCCESS, 멱등). 상품 삭제도 일어나지 않는다.

### Exception Flow
1. **Given** 관리자 인증 정보가 없거나 `X-Loopers-Ldap` 값이 올바르지 않을 때, **When** 삭제를 요청하면, **Then** 인증 실패로 응답한다(403 FORBIDDEN).

### 비즈니스 규칙
- 브랜드 삭제 시 해당 브랜드의 삭제되지 않은 상품도 모두 함께 삭제된다. DB 연쇄 제약이 아니라 애플리케이션이 같은 트랜잭션 안에서 처리한다.
- 삭제는 soft delete 방식으로, 행은 보존하되 외부 노출에서는 존재하지 않는 것으로 처리한다. (결정 7 참조)
- 존재하지 않거나 이미 삭제된 브랜드에 삭제 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 삭제). (결정 6 참조)
- 다른 브랜드의 상품과 이미 삭제된 상품은 영향받지 않는다.

### 비기능 요구사항
- 브랜드와 소속 상품의 삭제는 같은 트랜잭션 안에서 처리하여, 일부만 삭제된 상태로 남는 비정합 상태가 외부에 노출되지 않아야 한다.

## 엣지 케이스

- 대상 브랜드 상태: 활성(삭제) / 미존재(no-op) / 이미 삭제됨(no-op).
- cascade 범위: 소속 활성 상품 N개 → 모두 삭제 / 소속 상품 0개 → 브랜드만 삭제 / 소속에 이미 삭제된 상품 → 그 행은 변화 없음(멱등) / 다른 브랜드 상품 → 미영향.
- 멱등성: 동일 브랜드에 삭제를 두 번 요청해도 결과가 같고 두 응답 모두 정상.
- 트랜잭션: 상품 일괄 삭제 중 실패하면 브랜드 삭제도 롤백되어 부분 삭제 상태가 남지 않는다.
- 권한 경계: 회원 인증 헤더만 있고 admin 헤더가 없으면 인증 실패(403).

## 기능 요구사항

- **FR-001**: 시스템은 관리자 인증(`X-Loopers-Ldap: loopers.admin`)을 통과한 요청만 브랜드 삭제를 허용해야 한다. 실패 시 403 FORBIDDEN으로 응답한다.
- **FR-002**: 시스템은 대상 브랜드가 활성 상태이면 브랜드를 soft delete 처리해야 한다.
- **FR-003**: 시스템은 대상 브랜드의 소속 활성 상품을 같은 트랜잭션 안에서 모두 soft delete 처리해야 한다(cascade).
- **FR-004**: 시스템은 대상 브랜드가 존재하지 않거나 이미 삭제된 경우 별도 동작 없이 정상 응답(200 SUCCESS)으로 마무리해야 한다(멱등, 상품 삭제도 없음).
- **FR-005**: 브랜드 삭제와 상품 cascade 삭제는 같은 트랜잭션으로 묶여, 부분 삭제 상태가 외부에 노출되지 않아야 한다.

## 관련 엔티티

- **BrandModel** (재사용): `BaseEntity.delete()`(멱등 soft delete)를 그대로 사용. 도메인 신규 행위 없음.
- **BrandRepository** (편집): 멱등 처리를 위한 활성 단건 조회를 추가(부재·삭제 시 예외 없이 빈 결과 — no-op 분기용).
- **ProductRepository** (편집): 특정 브랜드의 소속 활성 상품을 조회하는 기능을 추가. 일괄 삭제 자체는 Facade가 도메인 `delete()`로 조율(메커니즘은 plan). 이미 삭제된 상품·다른 브랜드 상품은 조회에서 제외된다.
- **BrandFacade** (편집): 브랜드 삭제 유스케이스가 Brand·Product 두 저장소를 한 트랜잭션으로 조율하며, 브랜드·상품 모두 도메인 `delete()`를 Facade에서 호출(cross-domain 조율 — ProductFacade의 BrandRepository 참조 선례와 동형).
- **재사용**: `AdminAuthInterceptor`(admin 인증), `ErrorType`(FORBIDDEN), `ApiResponse.success()`(데이터 없는 200).

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Service/Facade 단위 | 브랜드 삭제 유스케이스 | 활성 브랜드 → 브랜드 + 소속 활성 상품 삭제 호출 / 미존재·이미 삭제 → no-op(상품 삭제 호출 없음) |
| Integration | BrandRepository | 활성 단건 조회: 활성 present / 삭제·부재 empty |
| E2E | `DELETE /api-admin/v1/brands/{brandId}` | 200 + meta.result SUCCESS & 브랜드·소속 상품이 활성 조회에서 제외 / 다른 브랜드 상품 잔존 / 미존재 200 / 동일 요청 반복 200 / admin 인증 실패 403 (statusCode + meta.result + errorCode까지, 메시지 문구는 단언 안 함). cascade 일괄 삭제(대상 브랜드 활성 상품만 `deletedAt` 기록 / 다른 브랜드·이미 삭제 미영향)는 실제 Facade 트랜잭션을 거치는 본 E2E가 검증 — `ProductRepository.deleteActiveByBrandId` 레포지토리 단독 통합 테스트는 review 결정으로 제외 |

## 관련 결정

- **결정 6 (DELETE 멱등 처리 B)**: 부재·이미 삭제 모두 정상 응답(완전 멱등). PRD-7과 동일 적용.
- **결정 7 (소프트 삭제 C — 브랜드·상품)**: 브랜드·상품 모두 soft delete. `BaseEntity.deletedAt` 사용.
- **결정 5 (주문 항목 스냅샷 B)**: 진행 중 주문이 있어도 항목 스냅샷이 보존되어 과거 주문 표시는 영향 없음. Order 미구현이라 본 cycle 검증 범위 밖.
- **시퀀스 다이어그램(BRD-6)**: 브랜드 자신 삭제 → 상품 도메인에 "소속 활성 상품 일괄 삭제" 요청 → 같은 거래로 묶음. 본 코드베이스에선 응용 계층(BrandFacade)이 조율.

## 성공 기준 / 범위 밖

- **성공**: 위 모든 수용 시나리오·테스트 계획이 green. `DELETE /api-admin/v1/brands/{brandId}`가 인증·멱등·cascade·트랜잭션 분기를 명세대로 처리.
- **범위 밖**: 주문 스냅샷 검증(Order 미구현), 좋아요 cascade(Like 미구현), hard delete, 삭제된 브랜드·상품 복원 API, 상품 단건 삭제(PRD-7).

**소스**: `docs/volume-2/01-requirements.md` — ORD-2 (시퀀스: `02-sequence-diagrams.md` ORD-2)
**작성일**: 2026-05-27

## 시나리오 요약

로그인한 회원이 상품 식별자와 수량의 쌍을 1개 이상 묶어 한 번의 주문으로 요청한다. ORD-1과 동일한 `POST /api/v1/orders` API를 공유하며, ORD-1이 항목 수 1인 케이스라면 ORD-2는 항목 수 1 이상인 일반화 케이스다. 모든 항목의 상품이 존재하고 삭제되지 않았는지 점검하고, 항목들을 상품 식별자 기준 일정한 순서로 정렬한 뒤 정렬된 순서로 각 상품의 재고를 요청 수량만큼 차감한다. 어느 한 항목이라도 실패(상품 부재·삭제·재고 부족)하면 이미 차감된 다른 항목의 재고까지 모두 되돌려 주문 전체를 없던 일로 만든다(전부 성공 또는 전부 실패). 모든 항목이 성공하면 항목별 스냅샷(상품명·브랜드명·단가·수량)으로 기록해 주문을 저장하고, 주문 식별자·상태·주문 시각·총 결제 금액(항목별 단가×수량의 합)·주문 항목 스냅샷 목록을 반환한다. 같은 상품 식별자가 둘 이상 포함된 요청은 거부한다(수량 합산 책임은 클라이언트).

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 회원 인증 + 재고가 충분한 활성 상품 여러 종, **When** 서로 다른 상품 식별자·수량 쌍을 묶어 주문을 요청하면, **Then** 각 상품 재고가 수량만큼 차감되고 주문이 생성되며 주문 식별자·상태·주문 시각·총 결제 금액(항목 합)·주문 항목 스냅샷 목록이 반환된다(201).
2. **Given** 회원 인증 + 항목 1개, **When** 항목 1개로 요청하면, **Then** ORD-1과 동일하게 정상 생성된다(단일은 다중의 특수 케이스).

### Exception Flow
1. **Given** 회원 인증 실패, **When** 주문을 요청하면, **Then** 401 UNAUTHENTICATED.
2. **Given** 항목이 비어 있을 때, **When** 주문을 요청하면, **Then** 400 BAD_REQUEST.
3. **Given** 어느 한 항목의 수량이 1 미만일 때, **When** 주문을 요청하면, **Then** 400 BAD_REQUEST.
4. **Given** 같은 상품 식별자가 둘 이상 포함되었을 때, **When** 주문을 요청하면, **Then** 400 BAD_REQUEST.
5. **Given** 어느 한 항목의 상품이 존재하지 않거나 삭제되었을 때, **When** 주문을 요청하면, **Then** 404 NOT_FOUND이며 이미 차감된 다른 항목의 재고는 원복된다.
6. **Given** 어느 한 항목의 재고가 요청 수량에 미치지 못할 때, **When** 주문을 요청하면, **Then** 409 CONFLICT이며 이미 차감된 다른 항목의 재고는 원복된다.

### 비즈니스 규칙
- 항목 간 정합성은 전부 성공 또는 전부 실패. 한 항목이라도 실패하면 다른 항목 처리도 되돌린다.
- 같은 상품 식별자가 둘 이상 포함된 요청은 거부한다. 수량 합산 책임은 클라이언트.
- 총 결제 금액은 항목들의 (`단가 × 수량`) 합계로 계산한다.
- 주문 항목은 항목별로 상품명·브랜드명·단가·수량을 스냅샷으로 저장한다. (결정 5)
- 항목들을 상품 식별자 순으로 정렬해 차감 잠금 획득 순서를 일관되게 유지(교착 회피). (결정 4)

## 엣지 케이스
- 항목 수: 0개(빈 목록, 400) / 1개(통과, ORD-1 동일) / N개(통과).
- 중복: 같은 productId 2회 이상(400).
- 수량: 어느 한 항목 0·음수(400).
- 부분 실패 원복: N개 중 1개가 404 또는 409 → 나머지 차감분 전부 원복, 주문 미생성.
- 동시성: 같은 상품 묶음에 동시 주문 → 교착 없이(정렬된 차감 순서) 재고 음수 불가. (결정 4)
- 실패 우선순위: 입력 검증(빈/수량/중복, 400)이 자원·재고 검사보다 먼저.

## 기능 요구사항

- **FR-001**: 시스템은 회원 인증을 통과한 요청만 허용해야 한다. 실패 시 401.
- **FR-002**: 시스템은 항목이 비어 있거나, 어느 항목의 수량이 1 미만이거나, 같은 상품 식별자가 둘 이상이면 400으로 거절해야 한다.
- **FR-003**: 시스템은 모든 항목의 상품이 존재하고 삭제되지 않았는지 검증해야 한다. 아니면 404.
- **FR-004**: 시스템은 항목을 상품 식별자 순으로 정렬한 뒤 정렬된 순서로 각 상품 재고를 요청 수량만큼 원자적으로 차감해야 한다.
- **FR-005**: 시스템은 어느 한 항목의 차감 실패(404·409) 시 이미 차감된 다른 항목의 재고까지 모두 원복해야 한다(전부 성공 또는 전부 실패).
- **FR-006**: 시스템은 항목별 스냅샷(상품명·브랜드명·단가·수량)을 기록하고 총 결제 금액을 항목 합으로 계산해야 한다.
- **FR-007**: 시스템은 생성된 주문의 식별자·상태·주문 시각·총 결제 금액·주문 항목 스냅샷 목록을 반환해야 한다.

## 관련 엔티티

- **OrderModel / OrderItemModel / Quantity / OrderStatus / OrderRepository**: ORD-1에서 세운 골격을 그대로 사용. 다중 항목은 `OrderModel`이 이미 항목 리스트를 받게 설계되므로 모델 변경 최소. (ORD-1 plan에서 리스트형으로 확정될 경우 모델은 그대로 재사용)
- **Product 재고 차감**: ORD-1의 원자적 조건부 차감을 항목 수만큼 반복 호출. 같은 트랜잭션 안에서 정렬 순서대로.
- **재사용**: ORD-1과 동일(`@LoginUser`, `ProductRepository`, `BrandRepository`, `ErrorType`).

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Service/Facade 단위 | 다중 주문 유스케이스 | 빈 목록/수량<1/중복 productId → BAD_REQUEST, 한 항목 상품 미존재 → NOT_FOUND(원복), 한 항목 재고 부족 → CONFLICT(원복), 정상 시 정렬 차감+항목별 스냅샷+총액 합 |
| Integration | OrderRepository / 재고 차감 | 다중 항목 주문·항목 영속화, 부분 실패 시 전체 롤백(차감분 원복) |
| Integration(동시성) | 같은 상품 묶음 동시 주문 | 교착 없이 처리, 재고 음수 불가 |
| E2E | `POST /api/v1/orders` (다중) | 201 + 항목 목록 응답 / 빈 목록·수량0·중복 → 400 / 한 항목 상품 미존재 → 404 / 한 항목 재고 초과 → 409 (statusCode + meta.result + errorCode) |

## 관련 결정

- **결정 2·5**: ORD-1과 동일(즉시 차감, 항목별 스냅샷).
- **결정 4 (동시성·교착 회피)**: 원자적 조건부 차감을 항목마다 적용하되, 항목을 상품 식별자 순으로 정렬해 차감(잠금 획득) 순서를 일관되게 유지 — 두 회원이 같은 두 상품을 반대 순서로 차감해 서로를 기다리는 교착을 끊는다.
- **전부 성공 또는 전부 실패**: 부분 성공("재고 있는 항목만 처리")은 정책 아님. 같은 트랜잭션 롤백으로 보장.
- **같은 상품 중복 거부**: 수량 합산 책임을 클라이언트로 미루고 서버는 단일 항목 단일 줄을 강제.
- **ORD-1과의 관계**: 같은 API·같은 유스케이스의 일반화. ORD-1 단건이 항목 수 1인 호출로 그대로 커버되면 ORD-2는 다중 처리 규칙(정렬·중복 거부·부분 실패 원복)만 추가. (구체 구현 분기는 plan)

## 성공 기준 / 범위 밖

- **성공**: 위 수용 시나리오·테스트가 green. `POST /api/v1/orders`가 1개 이상 항목에 대해 인증·입력 검증(빈/수량/중복)·전 항목 활성 검증·정렬 차감·부분 실패 원복·항목별 스냅샷·총액 합·트랜잭션 저장을 명세대로 처리.
- **범위 밖**: 주문 조회/내역/상세(ORD-3·4·5·6), 결제·2단계 차감, 같은 상품 수량 자동 합산(클라이언트 책임), 부분 성공 허용.

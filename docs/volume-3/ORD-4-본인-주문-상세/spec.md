# Spec: ORD-4 본인 주문 상세

**소스**: `docs/volume-2/01-requirements.md` — ORD-4
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

로그인한 회원이 특정 주문 식별자로 자신의 주문 상세를 조회한다. 회원 인증을 통과한 요청만 허용하며, 해당 주문이 존재하고 인증된 회원의 주문일 때만 주문의 식별자·상태·주문 시각·총 결제 금액·주문 항목 전체(상품 식별자·상품명·브랜드명·단가·수량 스냅샷)를 반환한다. 주문이 존재하지 않거나 존재하더라도 본인의 주문이 아니면 권한 거부가 아니라 자원 부재(404)로 응답해, 다른 회원의 주문 식별자 존재 여부가 응답으로부터 추론되지 않게 한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 회원 인증 + 본인이 생성한 주문, **When** 그 주문 식별자로 상세를 요청하면, **Then** 식별자·상태·주문 시각·총 결제 금액·주문 항목 전체(상품 식별자·상품명·브랜드명·단가·수량)가 반환된다(200).

### Exception Flow
1. **Given** 회원 인증 실패, **When** 상세를 요청하면, **Then** 401 UNAUTHENTICATED.
2. **Given** 대상 주문이 존재하지 않을 때, **When** 상세를 요청하면, **Then** 404 NOT_FOUND.
3. **Given** 대상 주문이 존재하지만 타인의 주문일 때, **When** 상세를 요청하면, **Then** 404 NOT_FOUND(권한 거부 아님).

### 비즈니스 규칙
- 주문 항목 정보는 주문 시점의 스냅샷을 그대로 표시한다. 이후 상품·브랜드 정보가 변경·삭제되어도 본 표시는 유지된다. (결정 5)
- 타인의 주문 접근은 403이 아니라 404로 응답한다(주문 식별자 enumeration 방지).

## 엣지 케이스
- 본인 주문(200) / 미존재 주문(404) / 타인 주문(404, 403 아님).
- 주문 항목이 여러 개인 주문 → 항목 전체 반환.
- 스냅샷 보존: 주문 후 상품·브랜드가 변경·삭제되어도 항목 스냅샷은 주문 시점 값 유지.

## 기능 요구사항

- **FR-001**: 시스템은 회원 인증을 통과한 요청만 허용해야 한다. 실패 시 401.
- **FR-002**: 시스템은 주문이 존재하고 인증된 회원 소유인지 검증해야 한다. 미존재 또는 타인 소유면 모두 404로 응답한다.
- **FR-003**: 시스템은 주문의 식별자·상태·주문 시각·총 결제 금액과 주문 항목 전체(상품 식별자·상품명·브랜드명·단가·수량 스냅샷)를 반환해야 한다.

## 관련 엔티티

- **OrderModel**: 헤더 필드(상태·시각·총액) 조회 원천. 소유 검증은 `OrderRepository.getActiveByIdAndUserId` 쿼리 필터로 처리(없으면 NOT_FOUND, 도메인 `isOwnedBy` 미도입 — analyze 결정). 항목은 `findActiveItemsByOrderId`로 별도 조회.
- **OrderItemModel**: 항목 스냅샷(상품 식별자·상품명·브랜드명·단가·수량) 표시 원천.
- **재사용**: `@LoginUser AuthenticatedUser`(회원 인증), `OrderRepository`(주문+항목 조회), `ErrorType`(NOT_FOUND·UNAUTHENTICATED).
- **조회 방식**: 본인 주문 상세 조회를 위한 projection 또는 엔티티 조회 — 미존재·타 소유 모두 404로 통일하는 조회 메서드 형태는 plan에서 결정.

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Service/Facade 단위 | 본인 주문 상세 유스케이스 | 미존재 → NOT_FOUND, 타인 소유 → NOT_FOUND, 본인 → 상세 반환(항목 포함) |
| Integration | OrderRepository | 주문+항목 함께 조회, 소유자 필터 |
| E2E | `GET /api/v1/orders/{orderId}` | 200 + 응답 키(식별자·상태·주문 시각·총액·항목 전체) / 인증 실패 401 / 미존재 404 / 타인 주문 404 (statusCode + meta.result + errorCode) |

## 관련 결정

- **결정 5 (스냅샷)**: 항목은 주문 시점 스냅샷을 그대로 표시.
- **보안(비기능)**: 타인 주문 접근을 404로 응답해 enumeration 방지. (LIK-3의 타인 식별자 비노출 정책과 같은 결)

## 성공 기준 / 범위 밖

- **성공**: 위 수용 시나리오·테스트가 green. `GET /api/v1/orders/{orderId}`가 인증·소유 검증·항목 포함 상세 반환을 명세대로 처리하고 타인·미존재를 404로 통일.
- **범위 밖**: 주문 목록·내역(ORD-3·5), 관리자 상세(ORD-6), 주문 취소/수정.

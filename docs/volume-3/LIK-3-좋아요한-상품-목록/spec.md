# Spec: LIK-3 좋아요한 상품 목록

**소스**: `docs/volume-2/01-requirements.md` — LIK-3
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

회원이 자신이 좋아요를 누른 상품 목록을 페이지 정보와 함께 조회한다. 회원 인증을 통과해야 하며, 요청 경로의 회원 식별자가 인증된 회원과 일치할 때만 인증된 회원이 좋아요한(삭제되지 않은) 상품 한 페이지 분량을 최신 좋아요 순으로 반환한다. 식별자가 일치하지 않거나 존재하지 않는 회원이어도 빈 목록(정상 응답)을 반환해, 다른 회원의 식별자 존재 여부·활동이 응답으로부터 추론되지 않게 한다. 각 항목은 PRD-1 목록 항목과 동일하게 식별자·이름·소속 브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 포함한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 회원 인증 + 본인이 활성 상품 여러 개에 좋아요, **When** 본인 userId·page·size로 요청, **Then** 좋아요한 활성 상품들이 최신 좋아요 순으로 페이지 메타와 함께 반환된다(200). 각 항목은 식별자·이름·브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수.
2. **Given** 회원 인증 + 좋아요 기록 없음, **When** 본인 userId로 요청, **Then** 빈 목록과 페이지 메타가 반환된다(200).

### Exception Flow
1. **Given** 회원 인증 실패, **When** 요청, **Then** 401 UNAUTHENTICATED.

### 비즈니스 규칙
- 정렬 기준은 최신 좋아요 순 하나로 고정.
- 좋아요한 시점 이후 상품이 삭제되었거나 소속 브랜드가 삭제된 경우 해당 항목은 응답에서 제외(LIK-2가 이 정책을 전제로 취소 진입을 차단).
- 페이지 크기 미지정 시 20건(기본값 page=0, size=20).
- 좋아요 수는 좋아요 테이블에서 매 조회마다 집계 (결정 1).
- 경로 userId가 인증 회원과 불일치하거나 미존재 회원이면 빈 목록(정상 응답)을 반환한다 — 타인 식별자 존재·활동 비노출.

## 엣지 케이스
- 빈 결과(좋아요 없음 또는 userId 불일치) → 빈 목록 + 메타(200).
- 좋아요 후 삭제된 상품·브랜드 cascade 삭제 상품 → 목록에서 제외.
- size 미지정(기본 20) / 마지막 페이지 부분 채움.
- 타인·미존재 userId → 401 아님, 200 빈 목록(인증 자체는 통과한 상태).

## 기능 요구사항

- **FR-001**: 시스템은 회원 인증 통과 요청만 허용한다(실패 401).
- **FR-002**: 시스템은 page(기본 0)·size(기본 20)를 수신해 페이징한다(범위 검증 없음, 클라 신뢰).
- **FR-003**: 경로 userId가 인증 회원과 일치하면 인증 회원이 좋아요한 삭제되지 않은 상품을, 불일치·미존재면 빈 목록을 반환한다.
- **FR-004**: 시스템은 좋아요한 상품 중 삭제되지 않고 소속 브랜드도 삭제되지 않은 상품만 최신 좋아요 순으로 페이징한다.
- **FR-005**: 각 항목에 식별자·이름·브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 포함한다(PRD-1 항목과 동일 집합).
- **FR-006**: 좋아요 수는 매 조회마다 좋아요 테이블에서 집계한 값을 사용한다.
- **FR-007**: 시스템은 페이지 메타(현재 페이지·크기·총 페이지·총 항목)를 포함한다.

## 관련 엔티티
- **LikeModel**: 인증 회원의 좋아요 기록(상품 식별자·좋아요 시각) 조회 원천, 최신 좋아요 순 정렬 기준.
- **ProductModel**: 좋아요한 상품의 활성 판정 및 표시 필드.
- **BrandModel**: 브랜드 식별자·이름 조인 대상.
- **재사용 토대**: PRD-1의 상품 목록 projection(식별자·이름·브랜드 식별자·브랜드명·가격·재고 가용 여부·좋아요 수)을 그대로 재사용. 조회 진입점은 "회원의 좋아요"로 다름.

## 테스트 계획
| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Integration | LikeRepository/ProductRepository(신규 조회) | 인증 회원의 좋아요한 활성 상품만 / 최신 좋아요 순 / 삭제 상품·브랜드 제외 / 페이징·총 개수 / 좋아요 수 집계 / 타 회원 좋아요 미포함 |
| E2E | `GET /api/v1/users/{userId}/likes` | 200 + 메타 키 + 항목 키(식별자·이름·브랜드·가격·재고 가용 여부·좋아요 수) / 빈 결과 200 / 타인·미존재 userId 200 빈 목록 / 인증 실패 401 |

## 관련 결정
- 결정 1(좋아요 누적 카운트): 매 조회 집계.
- 결정 7(soft delete): 삭제 상품·브랜드 cascade 삭제 상품 제외.
- LIK-2 대칭: 삭제 상품은 좋아요 목록에 노출되지 않아 취소 진입 자체가 닫힘.
- 보안(비기능): 타인·미존재 userId도 200 빈 목록 — enumeration 방지.
- 엔드포인트 위치(`/api/v1/users/{userId}/likes` 신규 컨트롤러 vs 기존 LikeV1Controller 확장)와 PRD-1 projection 재사용 구조는 **plan에서 결정**.

## 성공 기준 / 범위 밖
- 성공: 위 수용 시나리오·테스트 green. `GET /api/v1/users/{userId}/likes`가 인증·본인 일치·활성 필터·최신 좋아요 순·집계·메타를 명세대로 처리.
- 범위 밖: 좋아요 외 정렬 옵션, 좋아요 이력(취소 포함) 노출, 좋아요 수 캐싱/컬럼화.

# Spec: ORD-3 본인 주문 내역 (날짜 범위)

**소스**: `docs/volume-2/01-requirements.md` — ORD-3
**작성일**: 2026-05-27
**상태**: Draft

## 시나리오 요약

로그인한 회원이 선택적으로 시작일·종료일·페이지 정보를 담아 자신의 주문 내역을 조회한다. 회원 인증을 통과한 요청만 허용하며, 시작일·종료일이 누락되면 기본값(시작일은 오늘 기준 한 달 전, 종료일은 오늘)을 적용한다. 인증된 회원의 주문 중 주문 일자가 지정 범위(양 끝 포함) 안에 들어오는 한 페이지 분량을 주문 시각 내림차순으로 반환한다. 각 주문에는 식별자·상태·주문 시각·총 결제 금액·주문 항목 요약(상품명·브랜드명·단가·수량 스냅샷)과 페이지 메타(현재 페이지·크기·총 페이지·총 항목)가 포함된다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 회원 인증 + 범위 안 주문 여러 건, **When** 시작일·종료일·page·size로 요청하면, **Then** 범위 안 본인 주문이 주문 시각 내림차순으로 페이지 메타와 함께 반환된다(200). 각 항목은 식별자·상태·주문 시각·총액·주문 항목 요약(상품명·브랜드명·단가·수량).
2. **Given** 회원 인증 + 시작일·종료일 미지정, **When** page·size만으로 요청하면, **Then** 기본 범위(오늘 기준 한 달 전 ~ 오늘)가 적용되어 그 안의 본인 주문이 반환된다.
3. **Given** 회원 인증 + 범위 안 주문 없음, **When** 요청하면, **Then** 빈 목록과 페이지 메타가 반환된다(200).

### Exception Flow
1. **Given** 회원 인증 실패, **When** 요청하면, **Then** 401 UNAUTHENTICATED.
2. **Given** 시작일이 종료일보다 늦을 때, **When** 요청하면, **Then** 400 BAD_REQUEST.
3. **Given** 시작일이 오늘 이후일 때, **When** 요청하면, **Then** 400 BAD_REQUEST.

### 비즈니스 규칙
- 조회 범위는 주문 일자 기준이며, 시작일·종료일 모두 해당 일을 포함한다.
- 범위 기본값은 오늘 기준 한 달 전부터 오늘까지로 적용한다.
- 결과는 주문 시각 내림차순으로 정렬한다.
- 페이지 크기 미지정 시 20건.
- 주문 항목 정보는 주문 시점 스냅샷을 그대로 표시한다. (결정 5)

## 엣지 케이스
- 범위 경계: 시작일·종료일 당일 주문 포함(양끝 inclusive).
- 기본 범위: 시작일·종료일 둘 다 미지정 / 한쪽만 지정(나머지만 기본 적용 여부는 plan 결정).
- 잘못된 범위: 시작일>종료일(400) / 시작일이 오늘 이후(400).
- 페이지: size 미지정(20) / 마지막 페이지 부분 채움.
- 빈 결과: 범위 안 주문 없음 → 200 빈 목록 + 메타.
- 본인 격리: 타 회원 주문은 결과에 포함되지 않음.

## 기능 요구사항

- **FR-001**: 시스템은 회원 인증을 통과한 요청만 허용해야 한다. 실패 시 401.
- **FR-002**: 시스템은 시작일·종료일 누락 시 기본 범위(오늘 기준 한 달 전 ~ 오늘)를 적용해야 한다.
- **FR-003**: 시스템은 시작일이 종료일보다 늦거나 시작일이 오늘 이후이면 400으로 거절해야 한다.
- **FR-004**: 시스템은 page(기본 0)·size(기본 20)를 수신한다.
- **FR-005**: 시스템은 인증된 회원의 주문 중 주문 일자가 범위(양끝 포함) 안인 것만 주문 시각 내림차순으로 페이징해야 한다.
- **FR-006**: 각 주문에 식별자·상태·주문 시각·총 결제 금액·주문 항목 요약(상품명·브랜드명·단가·수량)을 포함해야 한다.
- **FR-007**: 시스템은 페이지 메타(현재 페이지·크기·총 페이지·총 항목)를 포함해야 한다.

## 관련 엔티티

- **OrderModel**: 인증 회원(userId)·주문 일자(orderedAt) 필터, 주문 시각 내림차순 정렬, 헤더 필드(상태·총액) 원천.
- **OrderItemModel**: 항목 요약(상품명·브랜드명·단가·수량) 표시 원천 — 주문당 N개 항목 포함.
- **재사용**: `@LoginUser AuthenticatedUser`(회원 인증), `OrderRepository`(범위·페이징 조회), `ErrorType`(BAD_REQUEST·UNAUTHENTICATED).
- **조회 형태**: 주문+항목 요약을 함께 담는 projection 또는 엔티티+항목 로딩 방식은 plan에서 결정(ORD-4 상세와 항목 포함 방식 공유 가능).

## 관련 결정

- **결정 5 (스냅샷)**: 항목 요약은 주문 시점 스냅샷 그대로 표시.
- **날짜 경계·타임존 (plan 확정 필요)**: ERD상 `ordered_at`은 UTC wall-clock DATETIME 저장. "오늘"·"한 달 전"의 기준 타임존과, 시작일·종료일을 `ordered_at` 비교용 시각 경계(예: 시작일 00:00 ~ 종료일 24:00)로 변환하는 방식은 **plan에서 확정**. (테스트 프로파일은 `Asia/Seoul`)
- **한쪽만 지정 시 처리 (plan 확정 필요)**: 시작일만 또는 종료일만 지정된 경우 나머지에 기본값을 채울지 — plan에서 확정. 소스는 "누락된 경우 기본값"이라 한쪽 누락도 그 한쪽만 기본 적용으로 해석(권장).

## 성공 기준 / 범위 밖

- **성공**: 위 수용 시나리오·테스트가 green. `GET /api/v1/orders`가 인증·기본 범위·범위 검증·페이징·본인 격리·정렬·항목 요약을 명세대로 처리.
- **범위 밖**: 주문 상세(ORD-4), 관리자 조회(ORD-5·6), 상태 필터(본 라운드 CREATED만), 정렬 옵션 선택(주문 시각 내림차순 고정).

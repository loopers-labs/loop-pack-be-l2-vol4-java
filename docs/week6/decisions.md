요구사항을 분석하고 기존 설계의 변경 지점을 짚어보겠습니다. 비동기 PG가 도입되면서 시스템의 가장 핵심적인 트랜잭션
설계가 완전히 뒤바뀌어야 하는 상황입니다.

### 1️⃣ 문제 상황 다시 정의하기 (요구사항의 재해석)
• [사용자 관점] 결제 요청 후 결과가 나오기까지 1~5초가 걸릴 수 있으며, 중간에 통신이 끊겼을 때 내 결제가 성공했는지
실패했는지 직관적으로 알기 어렵다.
• [비즈니스 관점] PG사 연동 과정에서 성공 70%, 실패 30%(한도초과/잘못된 카드 등)가 발생한다. 외부 시스템의 실패로
인해 우리 시스템의 재고나 쿠폰이 영구적으로 묶이는 일(가선점 상태 고착)이 발생해서는 안 되며, 반대로 결제가 안
되었는데 주문이 성공 처리되는 일도 없어야 한다.
• [시스템 관점] 가장 큰 문제입니다. 기존 설계는 **"단일 트랜잭션 내에서 동기식으로 PG를 호출"**하는 구조였습니다.
처리가 최대 5초 지연되는 PG를 트랜잭션 안에서 호출하게 되면, 단 몇십 건의 주문만으로도 DB 커넥션 풀이 고갈되고
비관적 락이 장기간 점유되어 시스템 전체가 마비(장애 확산)됩니다.

### 2️⃣ 애매한 요구사항 드러내기 (질문 및 확인 필요 사항)
코드를 작성하기 전, 다음 정책들에 대한 명확한 결정이 필요합니다.
1. [경계 질문] 주문과 결제의 분리
   • 제공해주신 결제 API( POST /api/v1/payments ) 스펙을 보면  orderId 를 입력받습니다. 이는 과거의 한 번에
   처리하던  /checkout  API와 달리, **"1. 주문 생성(재고 차감)"**과 "2. 결제 요청" API가 물리적으로 분리되어야
   함을 의미합니다. 이렇게 두 단계를 나누는 것이 맞을까요?
2. [정책 질문] PG사 콜백 누락(유실) 시 보정 정책
   • PG사에서 성공/실패 처리를 했지만, 우리 서버로 콜백( callbackUrl )을 보내주지 않거나 네트워크 단절로 유실될 수
   있습니다. 이때 어떤 방식으로 결제 상태를 복구할 것인지 정해야 합니다.
3. [정책 질문] 결제 "요청" 타임아웃 시의 처리
   • 콜백을 기다리기도 전에, 최초 결제 요청( POST /api/v1/payments ) 자체가 타임아웃(100~500ms 지연)으로 끊어질 수
   있습니다. 이때 즉시 우리 시스템의 주문을 취소할 것인지, 아니면 PG사에 요청이 도달했을 가능성을 고려해 대기할
   것인지 결정해야 합니다.

• Q2. 콜백 누락 보정 정책

• 선택지 B: 수동 단건 동기화 API 제공
• 사용자가 "결제 상태 새로고침" 버튼을 누르거나, 어드민이 수동으로 갱신합니다.
• 영향도: 구현은 단순하지만 사용자 경험이 떨어지고 붕 뜬 데이터가 남을 수 있습니다.
• Q3. 요청 타임아웃 시 처리 정책
• 선택지 A (Recommended): 타임아웃 시 즉시 취소하지 않고, 확정 후 보상 트랜잭션
• 타임아웃이 발생해도 PG쪽엔 요청이 접수되어 나중에 콜백으로 성공이 날아올 수 있습니다. 즉시 취소하지 않고
주문을  PENDING 으로 유지한 뒤, 콜백이나 폴링(배치)을 통해 상태를 확정받습니다.
• 영향도: 고객이 재결제를 시도할 때 다소 혼란이 있을 수 있으나 결제 사고(돈은 나갔는데 주문은 취소됨)를
원천 차단합니다.

### ⚠️ 설계의 잠재 리스크 (Risk & Trade-off)

이 설계를 도입할 때 피할 수 없는 리스크를 분명히 짚고 넘어가야 합니다.

1. 데이터 정합성 깨짐의 위험 (보상 트랜잭션 실패)
   • 비동기 콜백을 받아 결제 실패 처리를 할 때, DB 트랜잭션 예외나 서버 다운으로 "보상 트랜잭션(재고 복구)"이
   실패하면 영구적으로 재고가 증발합니다. 이벤트 기반 아키텍처(Spring Event + Outbox Pattern 등)로 재시도를
   보장할지, 아니면 단순  try-catch  후 실패 시 알럿(Alert)을 띄우고 수동으로 복구할지 수준(Trade-off)을 결정해야
   합니다.
1. 결제 요청 중복 (멱등성 문제):
   • 클라이언트가 결제 요청을 더블 클릭하거나 재시도할 때, 서버는 재고만 2번 깎거나 외부 PG에 요청을 2번 보낼
   위험이 있습니다.  Idempotency Key (멱등성 키)나 주문 상태( PENDING 인지 검사)를 기반으로 방어 로직을 촘촘히
   세워야 합니다.
2. 콜백의 동시성 경합 (Race Condition):
   • PG사에서 Callback을 재시도하여 동시에 2건이 들어오거나, 사용자가 '결제 확인' API를 누르는 시점과 Callback이
   들어오는 시점이 겹칠 수 있습니다. 이를 막기 위해 주문 정보를 갱신할 때 반드시 상태 기반 Optimistic Lock 혹은
   주문 ID 기준 DB 락이 필요합니다.
   위 내용에 대해 어떻게 생각하시나요? 트랜잭션 분리와 콜백/폴링 혼합 모델로 문서를 수정하는 방향으로 진행해도
   괜찮을지, 선택지 중에서 선호하시는 방향이 있다면 알려주시기 바랍니다!

### 1. 주문과 결제 API 분리 여부 (트랜잭션 경계 설계)

[문제 상황 및 분석]
현재  /docs/week2 의 요구사항과 시퀀스 다이어그램은 주문 생성, 재고 비관적 락 차감, 그리고 PG사 연동 처리가 단일
API ( /api/v1/orders/checkout ) 및 거대한 단일 DB 트랜잭션 내에서 한 번에 이루어지도록 설계되어 있습니다.
하지만 연동해야 할 PG 시스템은 최대 1초~5초의 처리 지연이 발생하는 비동기 환경입니다. 이 구조를 유지할 경우, DB
커넥션을 5초 동안 물고 있게 되어 단 몇십 건의 결제 요청만으로도 커넥션 풀이 고갈되고 재고 비관적 락으로 인해 시스템
전체에 병목이 발생합니다.

[선택지]

• 선택지 A (추천): 주문 API와 결제 API의 물리적 분리
• 기존 단일 설계를 폐기하고, 두 단계로 나눕니다.
• 1단계: 주문 생성 API ( POST /api/v1/orders ) - 재고를 차감하고 주문을  PENDING  상태로 DB에 짧은 트랜잭션으로
저장합니다.
• 2단계: 결제 요청 API ( POST /api/v1/payments ) - 분리된 트랜잭션에서 PG사로 결제를 요청하고, 우리 DB에는 결제
이력을  READY  상태로만 남깁니다. PG와의 통신 중에는 DB 락이 없습니다.
• 선택지 B: 기존 단일 API 유지 (내부 복잡도 증가)
• 사용자 입장에서는 기존처럼 한 번의 API 호출만 하도록 둡니다. 대신 백엔드 내부에서 Spring Event나 메시지 큐
등을 활용하여 주문 생성 트랜잭션과 외부 결제 호출 트랜잭션을 강제로 쪼갭니다.
• 영향: 클라이언트 코드는 변하지 않지만 백엔드 구현 복잡도가 급격히 상승합니다.

### 2. 콜백 유실(누락) 보정 정책

[문제 상황 및 분석]
PG사의 결제 처리가 완료되면 우리 서버의 Callback API ( POST /api/v1/examples/callback  등)를 호출하여 최종
성공/실패 여부를 알려주게 됩니다. 하지만 PG사 오류나 일시적인 네트워크 장애로 인해 우리 서버가 이 콜백을 영영
수신하지 못하는 상황이 발생할 수 있습니다.
만약 콜백이 누락되면 우리 시스템의 데이터는 영원히  READY(결제대기)  /  PENDING(주문대기)  상태에 갇히게 되고,
선점된 재고도 복구되지 못한 채 묶여버립니다.

[선택지]

• 선택지 A (추천): 스케줄러(Batch)를 통한 자동 폴링 동기화
• 일정 시간(예: 3~5분)이 지나도 상태가  READY 인 결제 건들을 주기적으로 수집하는 백그라운드 스케줄러를
구현합니다.
• 스케줄러가 알아서 PG사의 "단건 결제 상태 확인 API ( GET /api/v1/payments/... )"를 호출하여 유실된 성공/실패
상태를 가져와 우리 DB에 자동 반영(재고 복구 등)합니다.
• 장점: 사용자의 개입 없이 시스템이 스스로 일관성을 보장하며 재고 고착을 막습니다.
• 선택지 B: 사용자/어드민 수동 트리거 위주 (자동화 없음)
• 스케줄러를 개발하지 않고, 사용자가 화면에서 "결제 상태 새로고침" 버튼을 누르거나 어드민이 백오피스에서
수동으로 동기화 버튼을 누를 때만 PG 상태를 확인합니다.
• 장점: 구현은 단순합니다.
• 단점: 사용자가 앱을 그냥 종료해버리거나 어드민이 확인하지 않으면 재고가 무한정 묶이는 치명적인 문제가
있습니다.
### 추가 선택지 C: 지연 메시지 큐(Delayed Queue)를 활용한 이벤트 기반 보정
[작동 방식]

### 💡 Redis TTL(만료 시간)을 활용한 보정 방식
1. 결제 요청 시 Redis에 5분짜리 임시 키 생성
   • 결제가  READY  상태로 접수되면, DB에 저장함과 동시에 Redis에  payment_check:12345  (결제ID) 라는 키를 만들고
   TTL(수명)을 5분으로 설정합니다.
2. 5분 뒤 Redis에서 키가 자동으로 삭제(만료)됨
   • 시간이 흘러 정확히 5분이 지나면 Redis 시스템이 해당 키를 지워버립니다.
3. Spring Boot가 "키 삭제 알림"을 수신 (Listener)
   • Redis는 키가 만료되어 삭제될 때 Spring Boot로 "방금  payment_check:12345  키가 만료되었습니다" 라고 알림을
   보내줄 수 있습니다. (Spring Data Redis의  KeyExpirationEventMessageListener  사용)
4. 보정 로직 실행
   • Spring Boot는 이 알림을 받아 결제 ID(12345)를 추출한 뒤, DB를 조회해봅니다.
   • 만약 그 사이 콜백이 와서 결제가  COMPLETED 로 확정되었다면 그냥 무시합니다.
   • 만약 여전히  READY  상태라면 (콜백이 유실되었다면), 그때 PG사의 결제 상태 조회 API를 호출하여 상태를 맞춥니다.

### 3. 결제 타임아웃(Read Timeout) 발생 시 처리 정책

[문제 상황 및 분석]
여기서 가장 주의할 점은, 우리가 타임아웃을 겪었다고 해서 PG사의 결제가 실패한 것은 아닐 수 있다는 점입니다. PG사는
우리 서버에서 PG사로 결제 요청( POST )을 보냈는데, 지정한 타임아웃 시간(예: 500ms) 안에 응답이 오지 않아 우리
서버에서 예외가 발생한 상황입니다.
여기서 가장 주의할 점은, 우리가 타임아웃을 겪었다고 해서 PG사의 결제가 실패한 것은 아닐 수 있다는 점입니다. PG사는
이미 사용자의 카드를 정상적으로 결제(승인)했는데 우리에게 응답을 주는 네트워크 과정에서만 지연이 생겼을 가능성이
큽니다.
이때 만약 우리가 타임아웃이 났다는 이유만으로 섣불리 주문을 취소해버리면, "고객의 돈은 빠져나갔는데 주문은 취소되어
버린" 최악의 결제 사고가 발생합니다.

[선택지]

• 선택지 A (추천): 최종 확정 지연 및 결과 확인 후 보상 트랜잭션 수행
• 타임아웃 에러가 떨어지더라도 절대 즉시 주문을 취소하거나 재고를 복구하지 않습니다. 주문을  PENDING(대기)
상태로 둡니다.
• 고객에게는 "결제 진행 상태를 확인 중입니다"라고 안내합니다.
• 이후 도착하는 PG사의 비동기 콜백이나 스케줄러(배치)를 통해 PG사가 해당 결제를 정말로 성공했는지, 아니면
접수조차 안된 실패인지 명확한 사실을 확인한 뒤에야 주문을  COMPLETED 로 확정하거나, 실패로 확인됐을 때만 보상
트랜잭션(재고 복구 등)을 실행하여  CANCELED  처리합니다.
• 선택지 B: 즉시 자체 취소 및 무조건 망취소(Cancel) 비동기 전송
• 타임아웃이 떨어지면 즉시 우리 서버의 주문을 취소하고 재고를 원상복구시킵니다.
• 동시에 PG사로 **"방금 보낸 요청이 성공했든 실패했든 무조건 취소(망취소) 해달라"**는 취소 API를 비동기로
던집니다.
• 고객에게는 "네트워크 문제로 취소되었습니다. 다시 시도해주세요."라고 안내합니다. (단, 망취소 통신마저 실패하면
환불이 붕 뜨게 되어 어드민의 수동 대사가 필요합니다.)

### 4. 중복 결제 및 이중 주문 생성 방지 (멱등성 보장)

[문제 상황 및 분석]
사용자가 결제 버튼을 빠르게 더블 클릭하거나, 모바일 앱에서 네트워크 지연으로 인해 자동으로 API를 재요청(Retry)하는
상황은 흔하게 발생합니다.
만약 동일한  orderId 로 결제 요청( /payments )이 동시에 2번 접수되면, PG사로도 요청이 2번 나가게 되어 고객의 돈이
이중으로 출금되는 치명적인 결함이 발생할 수 있습니다. 동일한 장바구니로 주문( /orders ) 자체가 2번 생성되는 것도
막아야 합니다.

[선택지]

• 선택지 A (추천): DB 제약 조건을 통한 원천 차단 + 멱등성 키 활용
• 결제 중복 방어: 가장 확실한 방법으로,  PAYMENTS  테이블의  order_id  컬럼에 **Unique Index (고유 인덱스)**를
설정합니다. 이렇게 하면 찰나의 동시 요청(Race condition)이 발생해도 DB 구조 자체에서 1주문 당 1결제
강제하므로 이중 결제가 물리적으로 불가능합니다. 요청 진입 시 주문 상태가  PENDING 인지도 함께 검증합니다.
• 주문 중복 방어: 클라이언트가 API를 호출할 때 헤더에 멱등성 키( Idempotency-Key , 예: UUID)를 실어 보내도록
하고, 서버는 이를 짧은 시간(예: 10분) 동안 Redis나 DB에 저장하여 같은 키로 들어오는 재요청은 무시하고 이전
결과를 그대로 반환합니다.
• 선택지 B: 애플리케이션 상태(Select) 검증으로만 방어
• DB에 Unique Index를 걸거나 클라이언트로부터 멱등성 키를 받지 않습니다.
• 오직 결제 요청이 들어올 때 애플리케이션 코드 레벨에서  SELECT  쿼리를 날려 "이 주문으로 진행 중인 결제가
있는지" 확인 후 막아냅니다.
• 단점: 0.001초 차이로 거의 동시에 요청이 들어올 경우 SELECT 쿼리를 동시에 통과해버려서 이중 결제가 발생할
위험이 있습니다. 이를 막으려면 추가적으로 테이블 락(비관적 락)을 걸어야 해서 병목이 생깁니다.

### 1️⃣ 기능이 아니라 "불확실성" 관점의 재해석

현재 설계는 외부 PG 시스템의 동작을 전혀 신뢰하지 않는다는 훌륭한 전제를 깔고 있습니다.
• 지연될 수 있다: PG 호출(1~5초)을 DB 트랜잭션과 분리하여, 지연이 발생해도 내부 시스템의 커넥션 고갈을 막고
있습니다.
• 실패할 수 있다 (타임아웃): 타임아웃 발생 시 섣불리 실패 처리하지 않고 상태를  READY 로 대기시켜, 실제 PG의 성공
여부와 시스템 상태가 어긋나는 것을 방어하고 있습니다.
• 중복 실행될 수 있다:  PAYMENTS  테이블의  order_id  Unique Index와 멱등성 키를 통해 방어하고 있습니다.
• 성공했지만 응답이 유실될 수 있다: 비동기 콜백 누락을 대비해 Redis TTL( KeyExpirationEvent )을 안전망으로 두고
있습니다.

### 2️⃣ 트랜잭션 경계 검증

현재 설계의 트랜잭션 경계는 3개(주문 생성 / 결제 요청 / 콜백-보상)로 잘게 쪼개져 있습니다. 외부 호출은 트랜잭션
외부에 존재하여 아키텍처적으로 안전합니다.

• 외부 호출 실패 시 내부 상태는 어떻게 되는가?
• 타임아웃 등 에러 발생 시 시스템 상태는  PENDING(주문) / READY(결제) 를 유지합니다. 즉시 취소되지 않습니다.
• 내부 커밋 이후 외부 호출 실패 시 복구 가능한가?
• 내부 커밋(주문 생성) 이후 결제 자체가 아예 안 날아간 경우, Redis TTL 만료 이벤트가 5분 뒤 트리거되어 단건
조회를 해보고 내역이 없으면 보상 트랜잭션(재고 복구)을 실행하므로 복구 가능합니다.
• 외부 성공 후 내부 실패 시 상태는 어떻게 정합성을 유지하는가?
• PG는 성공해서 콜백을 줬는데, 우리 서버 내부에서 DB 장애 등으로 콜백 처리 트랜잭션이 롤백될 수 있습니다. 이
경우 상태가 일치하지 않게 됩니다.

### 3️⃣ 상태 기반 구조 전이

호출 흐름을 걷어내고 순수하게 **상태(State)**의 전이만 놓고 보겠습니다.
• [내부 상태]  PENDING / READY  ───(대기 구간)───▶  COMPLETED / APPROVED  (성공 시)
• [외부 상태] ───────────(승인 처리)───────────▶  SUCCESS
• ⚠️ 상태가 어긋나는 지점 (Risk Point):
• 대기 구간( READY  상태)에서는 내부 시스템은 결제가 완료되었는지 절대 알 수 없습니다.
• 이 상태에서 클라이언트(사용자)가 탭을 닫아버리거나 재시도를 누를 때의 UX 처리가 매우 중요해집니다.

### 4️⃣ 중복 요청 및 재시도 가능성 분석
네트워크 지연으로 클라이언트가 결제 버튼을 두 번 눌렀다고 가정해보겠습니다.
• 현재 설계의 방어:  PAYMENTS.order_id 에 Unique Index가 걸려 있으므로, 2번째 요청은 DB Insert 과정에서
DataIntegrityViolationException 을 발생시켜 완벽하게 이중 결제를 차단합니다.
• 잠재적 문제 (동시성 멱등성):
만약 "PG사 콜백 수신"과 "Redis TTL 만료 이벤트"가 1ms 차이로 동시에 발생한다면 어떻게 될까요?
두 개의 스레드가 동시에  READY  상태를 읽고, 각각 DB 업데이트 및 확정 처리를 시도하는 Race Condition이 발생할 수
있습니다.
──────
### 5️⃣ 장애 시나리오 분석

정상 흐름은 작동할 것입니다. 하지만 다음과 같은 실패 흐름(장애 시나리오)에 직면했을 때 시스템의 반응을
살펴보겠습니다.
1. 시나리오 A: 보상 트랜잭션 처리 중 내부 DB 장애 (매우 치명적)
   • 상황: PG사 잔액 부족으로 결제 실패 콜백을 받았습니다. 시스템이 재고를 복구(보상 트랜잭션)하려는데 DB
   데드락이나 장애로 인해 복구가 실패(Rollback)합니다.
   • 결과: 결제는 실패했는데, 재고는 영구히 차감된 상태로 남게 됩니다 (정합성 파괴).
   • 복구 가능성: 현재 아키텍처에서는 보상 트랜잭션 실패에 대한 '재시도(Retry)' 로직이 보이지 않아 수동 복구가
   필요합니다.
2. 시나리오 B: Redis 시스템의 일시적 다운
   • 상황: 결제는 들어왔지만 콜백이 유실되었는데, 마침 캐시용으로 쓰던 Redis 서버가 잠시 다운되거나
   재시작되었습니다.
   • 결과: 5분 TTL 알람( KeyExpiredEvent )이 영구히 소실됩니다.
   • 복구 가능성: 알람 시계가 고장 났으므로 해당 결제 건은 영원히  READY  상태에 갇히게 됩니다.
3. 시나리오 C: 콜백 상태 확정 중 동시성 충돌
   • 상황: 관리자가 수동으로 결제 상태 확인 버튼을 누르는 순간과, 비동기 콜백이 들어오는 순간이 겹칩니다.
   • 결과: 주문 상태 업데이트 과정에서 낙관적 락/비관적 락 처리가 없다면 덮어쓰기(Lost Update)가 발생할 수
   있습니다.

### 6 잠재 리스크 및 대안 구조 선택지

현재 설계는 "외부 시스템 지연 격리"라는 목적은 훌륭하게 달성했지만, 보상 트랜잭션 실패나 보조 시스템(Redis) 유실 시
데이터가 붕 뜨는 리스크를 내포하고 있습니다. 이를 보완하기 위한 대안 설계 선택지입니다.

• 선택지 1: 보상 로직의 단순화 유지 (현재 상태)
• 리스크를 수용합니다. 보상 트랜잭션이 실패하거나 Redis가 다운되는 일은 극히 드물다고 가정하고, 에러 로그에
알럿(Slack 등)을 걸어 어드민이 수동 대사(Manual Reconciliation)로 풀어냅니다.
• 영향도: 개발 복잡도가 가장 낮고 실용적입니다.
• 선택지 2: Redis 알람 + DB 스케줄러 이중화
• Redis TTL 이벤트를 주력으로 쓰되, 새벽 시간에 한 번씩 도는 DB 스케줄러(예: 2시간 이상 READY인 건 색출)를
최후의 안전망으로 하나 더 둡니다.
• 영향도: Redis 이벤트 유실이라는 치명적 단점을 가볍게 커버할 수 있는 훌륭한 타협안입니다.
• 선택지 3: 비동기 메시지 큐 기반의 완벽한 재시도 (Outbox Pattern)
• 보상 트랜잭션 요청 자체를  Kafka 나  RabbitMQ 의 실패 재시도 큐에 담아, 성공할 때까지 무한 재시도하게
만듭니다.
• 영향도: 데이터 정합성을 기계적으로 보장할 수 있으나, 시스템 인프라 및 운영 복잡도가 기하급수적으로 상승합니다.


현재 설계의 경계와 상태 분리는 매우 우수합니다. 발생 가능한 장애 시나리오(특히 보상 트랜잭션 실패)를 염두에 두시고,
비즈니스 요건에 맞�춰 리스크 수용 수�준을 결하시면 더욱 견고�한 시스템이 될 것�입니다.

두 방식을 비교해보면 무조건 "전자가 더 좋다"라고 말하기는 어렵습니다. 서비스의 트래픽 규모와 아키텍처의 복잡도를
어디까지 허용할 것인가에 따라 정답이 달라집니다.
실무적인 관점에서 두 방식의 장단점을 명확히 비교해 드리겠습니다.

### 1️⃣ RDB 스케줄링만 사용하는 방식 (단일화)
DB에 일정 시간(예: 5분) 이상  READY  상태로 머물러 있는 결제 건을  SELECT  해서 보정하는 단순한 방식입니다.
• 👍 장점 (Simplicity & Reliability):
• 압도적인 단순함: 백그라운드 스케줄러( @Scheduled ) 하나만 있으면 됩니다. 아키텍처가 단순해서 유지보수와
디버깅이 매우 쉽습니다.
• 절대 유실되지 않음: DB 자체가 Source of Truth(단일 진실 공급원)이므로, 서버가 꺼졌다 켜져도 데이터만
남아있으면 언젠가는 무조건 복구해 냅니다.
• 👎 단점:
• DB 부하 (Polling Overhead): 1분마다  SELECT * FROM payments WHERE status = 'READY'  쿼리를 날려야 합니다.
인덱스( status, created_at )를 잘 태우면 큰 문제는 없지만, 트래픽이 엄청나게 많은 대용량 서비스에서는 이
주기적인 폴링 자체가 DB에 지속적인 피로도를 줍니다.


### 2️⃣ Redis 만료 이벤트 + RDB 스케줄링 (이중화)

Redis TTL을 주력으로 쓰되, 혹시 모를 Redis 유실(Pub/Sub은 휘발성)을 대비해 새벽 시간에 RDB 스케줄러를 한 번씩
돌리는 방식입니다.

• 👍 장점 (Performance):
• DB 피로도 제로: DB를 주기적으로 긁지 않고, 정확히 5분이 지난 결제 건에 대해서만 알람(이벤트)이 울려
핀포인트로 처리하므로 성능상 매우 우아합니다.
• 👎 단점 (Complexity):
• 관리 포인트 증가: "결제 복구"라는 하나의 목적을 위해 Redis Listener 로직과 RDB Scheduler 로직 두 가지를 모두
유지보수해야 합니다.
• Redis Pub/Sub은 본질적으로 "Fire and Forget(쏘고 잊기)" 방식이라 알람이 울릴 때 서버가 배포 중이면 이벤트가
영구 증발합니다. 그래서 결국 RDB 스케줄러를 또 만들어야 하는 모순이 생깁니다.

### 💡 결론 및 실무 추천

• 스타트업 ~ 중대형 서비스: RDB 스케줄링만 사용하는 것을 강력히 추천합니다.
대부분의 서비스 규모에서는 1분에 한 번씩 인덱스를 타는 상태 체크 쿼리를 날리는 것이 DB에 전혀 무리가 되지 않습니다.
오히려 관리 포인트를 하나로 줄여 시스템 복잡도를 낮추는 것이 훨씬 이득입니다.
• 초대형 플랫폼 (네이버페이, 배달의민족 급): Redis(또는 Kafka) + RDB 스케줄러를 혼합합니다.
분당 수만 건이 쏟아져 DB 폴링 자체가 병목이 되는 수준이 오면, 이벤트 기반으로 부하를 분산하고 스케줄러는 하루에 한
번만 돌려 최후의 정산 용도로만 사용합니다

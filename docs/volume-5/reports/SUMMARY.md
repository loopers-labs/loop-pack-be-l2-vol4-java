# Volume 5 — 단계별 성능 누적 비교 (증거 척추)

> **PM·비개발자가 보는 문서.** 단계가 진행될수록 응답시간이 어떻게 줄어드는지 한 표로 본다.
> "왜 빨라졌나"의 기술 근거는 각 `0N-*.md` 보고서. 시나리오 정의는 `00-setup.md` §6.
> 모든 숫자는 **동시 50명** 측정. (※ 측정 전까지 빈칸 — 숫자를 지어내지 않는다)

## 응답시간 p95 (동시 50명) — 단계가 진행될수록 줄어든다 (낮을수록 좋음)

| 시나리오 | ① 베이스라인 | ② 비정규화 | ③ 인덱스 | ④ 캐시 |
|---|---|---|---|---|
| S1 좋아요·전역 | DNF (100% 에러) | 2,790ms | **260ms** | |
| S2 좋아요·브랜드 | DNF (100% 에러) | 560ms | **69ms** | |
| S3 최신·전역 | DNF (100% 에러) | 2,860ms | **247ms** | |
| S4 상세·단건 | 707ms | 19ms | **23ms** | |

> 베이스라인 목록(S1·S2·S3)은 동시 50명에서 7초~DNF 쿼리가 커넥션 풀을 고갈 → **100% 에러(3초 타임아웃)**. 단건 서비스 시간은 S3 ~7s·S1/S2 DNF(>30s). S4만 정상(p95 707ms).
> **비정규화**: 좋아요 COUNT 서브쿼리를 `like_count` 컬럼으로 대체 → 목록 3종 **DNF/100% 에러가 전부 0% 에러로 회복**(S4는 707→19ms). 다만 전역 목록(S1·S3)은 풀스캔+filesort가 동시 50명에서 직렬 병목이라 p95 2.8s대 — 인덱스 단계의 동기.
> **인덱스**: 복합·단일 인덱스로 정렬 filesort 제거 + **전역 정렬의 조인 함정**(brand 복합 인덱스가 brands-first 오판 유발)을 `b.deleted_at IS NULL` 중복 필터 제거로 해소 + count 쿼리의 불필요한 brands JOIN 제거 → 전역 목록 **2.8s → ~250ms(약 11×)**, 0% 에러 유지. 남은 병목은 전역 count 10만 풀스캔(~21ms) — 캐시 단계의 동기.

## 실행계획 핵심 변화 — S1(좋아요·전역) 기준

| 단계 | 접근방식 | 정렬 | 좋아요 집계 |
|---|---|---|---|
| ① 베이스라인 | 풀스캔 | filesort | 매 조회 COUNT 서브쿼리 |
| ② 비정규화 | 풀스캔 | filesort | like_count 컬럼 직접 |
| ③ 인덱스 | like_count 인덱스 역방향 20행 | 인덱스 정렬(filesort 제거) | like_count 컬럼 직접 |
| ④ 캐시 | | | |

## 한 줄 서사 (PM용)

- **① → ② 비정규화**: 좋아요 수를 미리 계산해 둬서 "매번 세기"를 없앰 → 응답조차 못 주던 목록 3종이 정상화(100% 에러 → 0% 에러)
- **② → ③ 인덱스**: 정렬·필터를 인덱스가 대신해 풀스캔·filesort가 사라짐 + 전역 목록의 숨은 함정(조인이 정렬 인덱스를 못 쓰게 막던 것, 총개수 세기에 딸린 불필요한 조인)을 걷어냄 → 전역 목록 2.8초 → 약 0.25초
- **③ → ④ 캐시**: (자주 같은 요청은 DB까지 안 가고 캐시에서 즉답)

> "왜 빨라졌나"의 기술 근거는 각 `0N-*.md` 보고서. 시나리오 정의는 `00-setup.md` §6.

## 📊 응답시간 p95 (단위 ms · 낮을수록 좋음)

| 시나리오 | ① 베이스라인 | ② 비정규화 | ③ 인덱스 | ④ 캐시 |
|---|---:|---:|---:|---:|
| **S1** 좋아요·전역 | DNF | 2,790 | 260 | **22** |
| **S2** 좋아요·브랜드 | DNF | 560 | 69 | 49 |
| **S3** 최신·전역 | DNF | 2,860 | 247 | 197 |
| **S4** 상세·단건 | 707 | 19 | 23 | **18** |

- `DNF` = 동시 50명에서 7초~30초+ 쿼리가 커넥션 풀을 고갈시켜 **100% 에러**.
- **굵게** = ④에서 캐시를 적용한 시나리오(S1·S4). S2·S3는 캐시 **우회**(핫셋 아님)라 ③과 동급.
- ②부터 전 구간 **에러율 0%**.

**한 줄 결론:** 응답조차 못 주던 인기순 목록(S1)이 **22ms**로, 상세(S4)는 **707 → 18ms**로. 비정규화로 살리고, 인덱스로 정렬을 없애고, 캐시로 DB를 비켜 갔다.

## 단계별 요약

### ② 비정규화 — "좋아요를 매번 세지 말고 미리 저장"
- **바꾼 것**: 좋아요 `COUNT(*)` 서브쿼리 → `products.like_count` 컬럼 직접
- **결과**: 목록 3종 **100% 에러 → 0% 에러** · 상세 707 → 19ms
- **남은 숙제**: 전역 목록은 풀스캔 + filesort로 p95 **2.8초** → ③ 인덱스로

### ③ 인덱스 — "정렬·필터를 인덱스가 대신"
- **바꾼 것**: 복합·단일 인덱스로 filesort 제거 + **전역 정렬의 조인 함정** 해소(`b.deleted_at IS NULL` 중복 필터 제거로 옵티마이저의 brands-first 오판 차단, count 쿼리의 불필요한 brands JOIN 제거)
- **결과**: 전역 목록 **2.8초 → ~0.25초 (약 11×)** · 에러율 0% 유지
- **남은 숙제**: 전역 count 10만 풀스캔(~21ms) → ④ 캐시로

### ④ 캐시 — "자주 같은 요청은 DB 대신 Redis가 즉답"
- **바꾼 것**: 상세 + 목록 **핫셋(필터 없음·좋아요순·1페이지)만** Redis 캐싱. 그 외(브랜드 필터·최신순)는 **우회**해 키 폭발 억제
- **결과**:

  | 시나리오 | ③ 인덱스 | ④ 캐시 | 효과 |
  |---|---:|---:|---|
  | S1 목록 인기순 | 260ms | **22ms** | 약 12× — DB 미접근 |
  | S4 상세 | 23ms | **18ms** | 지연은 소폭, 대신 **DB 호출 99.96% 제거**(replica 적중 113,060 / 미스 40) |
  | S2·S3 | 69·247ms | 49·197ms | 비캐싱 → ③ 동급(의도된 경계) |

- **트레이드오프(정합성↔속도)**: `likeCount`는 staleness 허용(근사 표시값), **TTL이 정합성 노브**(상세 60s·목록 30s). 상품 **본문 변경만** 커밋 직후(AFTER_COMMIT) 무효화하고, 좋아요 증감은 TTL이 흡수.
- **장애 안전망**: Redis 예외는 미스로 폴백 → Redis가 죽어도 DB로 정상 동작(Fail-Safe).
- 상세 근거: `04-cache.md`.

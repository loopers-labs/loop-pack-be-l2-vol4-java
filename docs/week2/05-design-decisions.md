# 05. 설계 의사결정 기록

> 설계 과정에서 고민하고 결정한 사항들을 기록합니다.  
> "왜 이렇게 했는가"와 "왜 다른 선택지를 버렸는가"를 함께 남깁니다.

## DD-001. Product / Stock 분리 (1:1)

**고민**

재고(`stock`)를 `Product` 테이블 컬럼으로 두면,
재고 차감(주문마다 발생)과 상품 수정(어드민이 가끔 수행)이 동일한 행에 락을 걸게 된다.

```
주문 처리: UPDATE products SET stock = stock - N WHERE id = ?  → products 행 락
상품 수정: UPDATE products SET name = ?, price = ? WHERE id = ? → 동일 행 락 경합
```

두 연산은 목적도, 빈도도, 담당자도 다른데 같은 행을 공유하는 구조다.

**제외한 선택지 — SKU/옵션 단위 재고**

실제 이커머스에서 재고는 상품이 아닌 구매 가능한 최소 단위(SKU/옵션)에 붙는다.

```
Product: 나이키 에어포스 1
  └── Option: 흰색/250mm → stock: 5
  └── Option: 흰색/255mm → stock: 0  ← 품절
```

도메인 정확도는 더 높지만, 현재 주문 API 계약이 `{ productId, quantity }` 구조여서
`optionId` 없이는 어떤 재고를 차감할지 알 수 없다. API 계약 변경 없이는 도입 불가.
→ 요구사항이 옵션을 명시하는 시점에 재도입.

**결정**

`stocks` 테이블을 `products`와 1:1로 분리.
두 연산이 서로 다른 행을 잠가 락 경합 해소.

```
Product { id, name, price, deleted_at }   ← 카탈로그
Stock   { id, product_id, quantity }      ← 재고 (UNIQUE on product_id)
```

**향후 고려**

상품 옵션이 도입되면 `stocks.product_id` → `stocks.option_id`로 자연스럽게 확장된다.

## DD-002. OrderItem 필드 명명 — `snapshot_` 제거

**고민**

주문 시점 상품 정보를 저장할 때, 초기 설계는 `snapshot_price`, `snapshot_product_name`처럼
`snapshot_` 접두사를 붙여 "상품 데이터를 복사했다"는 의미를 표현했다.

**결정**

`unitPrice`, `productName`, `brandName`으로 변경.

**이유**

주문은 구매자·판매자 간 계약이다.
계약이 체결되는 순간, 합의된 가격은 더 이상 상품 도메인의 현재 가격이 아니라
**주문 도메인이 자체적으로 소유하는 값**이 된다.

`snapshot_`은 "원본은 상품 도메인에 있고, 우리는 복사본을 갖고 있다"는
종속 관계를 암시한다. `unitPrice`는 OrderItem이 자신의 언어로 소유하는 값임을 명확히 한다.

**결과**

- `Product.price` 변경이 기존 주문에 영향 없음
- Order Aggregate가 상품 도메인에 의존하지 않고 자립적으로 완결됨
- 주문 생성 시 Product → OrderItem으로 명시적 값 복사 로직 필요 (트레이드오프)

## DD-003. `like_count` 역정규화 — 제거

**고민**

상품 목록에서 `likes_desc` 정렬을 지원하려면 `COUNT(likes)` 집계가 필요하다.
성능을 위해 `products.like_count` 컬럼을 두고 좋아요 등록·취소 시 함께 갱신하는
역정규화를 초기에 설계했다.

**제거한 이유**

1. `likes_desc` 정렬은 시나리오 문서에서 **선택 구현**으로 명시됨
2. 선택 기능을 위해 역정규화를 도입하면 동기화 정합성 요구사항이 추가로 생긴다
3. 실제 성능 문제가 발생했을 때 도입해도 늦지 않음 (YAGNI)

**결정**

`like_count` 컬럼 없이 필요 시 `COUNT(likes)`로 집계.
`likes_desc` 정렬 성능 이슈가 실제로 관찰되면 그때 역정규화 또는 캐시를 도입.

## DD-004. `subtotal_amount` 저장 — 제거

**고민**

`OrderItem`의 소계(`unit_price × quantity`)를 컬럼으로 저장할지, 조회 시 계산할지.

**제거한 이유**

`unit_price`와 `quantity`가 이미 저장되어 있으므로 소계는 순수 파생값이다.
저장하면 쓰기 시 항상 계산해서 넣어야 하는 부담이 생기고, 불일치 가능성도 생긴다.
조회 시 계산하는 게 더 단순하고 정합성 위험이 없다.

**결정**

저장하지 않고 조회 시 `unit_price * quantity`로 계산.

## DD-005. `OrderStatus` — ENUM 선택

**고민**

주문 상태를 `VARCHAR + CHECK`로 둘지, DB ENUM으로 둘지.

**선택지 비교**

| | VARCHAR + CHECK | ENUM |
|---|---|---|
| 상태 추가 시 | 코드만 변경 | ALTER TABLE 필요 |
| 잘못된 값 차단 | CHECK 제약으로 가능 | DB 레벨에서 차단 |
| 현재 상태 수 | 3개 (PENDING, CONFIRMED, CANCELLED) | 동일 |

**결정**

ENUM 선택.

현재 상태가 명확히 3개이고, 추가 상태가 생긴다면 어차피 도메인 로직도 함께 변경된다.
확장성을 이유로 VARCHAR를 선택하는 것은 지금 단계에서 이른 최적화다.
DB 레벨에서 허용 값을 명시적으로 제한하는 게 더 안전하다.

## DD-006. 소프트 딜리트 — `deleted_at` 방식

**고민**

브랜드·상품 삭제 시 물리 삭제(DELETE)할지, 소프트 딜리트(`deleted_at` 설정)할지.

**결정**

소프트 딜리트 선택.

**이유**

물리 삭제 시 `order_items.product_id`가 참조하는 행이 사라져 FK 정합성이 깨진다.
소프트 딜리트는 행을 유지하면서 고객 API 응답에서만 제외하므로,
주문 이력의 스냅샷 정합성을 유지할 수 있다.

**브랜드 삭제 연쇄 처리**

브랜드 삭제 시 소속 상품도 연쇄 소프트 딜리트.
DB CASCADE 대신 애플리케이션 레벨에서 처리해 의도를 명확히 한다.

## DD-007. 좋아요 멱등성 — 응답 코드 선택

**고민**

이미 좋아요한 상품에 재요청, 또는 이미 취소된 상품에 재취소 시 어떤 응답을 반환할지.
`409 Conflict`로 에러를 반환할지, `200 OK`로 정상 처리할지.

**결정**

`200 OK` 반환 (멱등 처리).

**이유**

클라이언트 입장에서 "좋아요 등록" 요청의 의도는 "이 상품이 좋아요 상태가 되길 바란다"는 것이다.
이미 그 상태라면 요청이 성공한 것이므로 에러를 반환할 이유가 없다.
`409`는 클라이언트가 요청 전에 현재 상태를 알아야 하는 부담을 만든다.

## DD-008. 타인 리소스 접근 — `403` 대신 `404` 반환

**고민**

타인의 주문·좋아요 목록을 조회할 때 `403 Forbidden`과 `404 Not Found` 중 선택.

**결정**

`404 Not Found` 반환.

**이유**

`403`은 "자원은 존재하지만 권한이 없다"는 사실을 노출한다.
`404`는 자원 존재 여부 자체를 숨겨 정보 노출을 최소화한다.
보안 관점에서 리소스 존재 여부를 숨기는 것이 일반적인 관행이다.

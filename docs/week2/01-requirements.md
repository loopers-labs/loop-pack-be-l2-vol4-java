# 01. 요구사항 명세서

> 루프팩 감성 이커머스 — 2주차 설계 문서
> 작성 범위: 상품/브랜드 조회, 좋아요(멱등), 주문 생성 및 결제 흐름
> 제외 범위: 회원가입, 내 정보 조회, 포인트(요구사항 삭제됨)

---

## 1. 제품 개요

### 배경
사용자가 여러 브랜드의 상품을 둘러보고, 마음에 드는 상품에 좋아요를 누르고, 여러 상품을 한 번에 주문·결제하는 감성 이커머스 서비스다. 사용자 행동(좋아요)은 이후 상품 랭킹과 추천으로 확장될 수 있는 기반 데이터가 된다.

### 대상 사용자
- **일반 사용자**: 상품을 탐색하고 좋아요를 누르며 주문·결제하는 고객
- **어드민**: 브랜드와 상품을 등록/수정/삭제하고 주문 현황을 조회하는 운영자

### 핵심 문제
- 여러 브랜드 상품을 한 화면에서 탐색하고 정렬/필터링할 수 있어야 한다.
- 좋아요는 여러 번 눌러도 결과가 일관되어야 한다(멱등).
- 여러 상품을 한 번에 주문할 때 재고와 금액이 정확히 맞아야 하며, 결제가 실패하면 깔끔하게 원상복구되어야 한다.

### 용어 정리 (비개발자용)
- **멱등(idempotent)**: 같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 같은 성질
- **스냅샷(snapshot)**: 주문 시점의 상품 정보를 사진처럼 그대로 복사해 보관하는 것
- **PG (Payment Gateway)**: 카드 결제 등을 대신 처리해 주는 외부 결제 시스템
- **Soft delete**: 데이터를 실제로 지우지 않고 "삭제됨" 표시만 하여 숨기는 방식

## 3. 유저 시나리오

### 시나리오 A: 상품 탐색 → 좋아요 (US-01, US-02, US-04)

**정상 흐름 (Happy Path)**
1. 사용자가 상품 목록 화면에 진입한다 → "전체 상품"이 최신순으로 표시된다.
2. 특정 브랜드로 필터링하고 가격 낮은 순으로 정렬한다 → 조건에 맞는 목록이 갱신된다.
3. 상품을 눌러 상세 화면에 진입한다 → 가격, 재고 여부, 브랜드명, 좋아요 수가 표시된다.
4. 좋아요 버튼을 누른다 → "좋아요 완료" 상태로 바뀌고 좋아요 수가 +1 된다.

**예외 흐름 (Exception Path)**
- E-A1: 존재하지 않는 상품 상세 진입 → "상품을 찾을 수 없습니다"(404).
- E-A2: 이미 좋아요한 상품에 다시 좋아요 → 화면 변화 없이 정상 응답(멱등, 2xx).
- E-A3: 삭제된(soft delete) 상품 → 목록·상세에서 노출되지 않음(404).

### 시나리오 B: 주문 및 결제 (US-06)

**정상 흐름 (Happy Path)**
1. 사용자가 상품 2개를 담아 주문을 요청한다(`POST /api/v1/orders`).
2. 시스템이 각 상품의 재고를 확인하고 차감한다 → 주문이 `PENDING` 상태로 생성된다.
3. 주문 시점의 상품 정보(가격·이름)가 스냅샷으로 저장된다.
4. 외부 결제 시스템(PG)에 결제를 요청한다 → 결제 성공.
5. 주문 상태가 `PAID`로 변경된다 → "결제가 완료되었습니다" 화면 표시.

**예외 흐름 (Exception Path)**
- E-B1: 주문 항목 중 하나라도 재고 부족 → 주문 전체 실패, 어떤 재고도 차감되지 않음, "재고가 부족합니다"(409).
- E-B2: PG 결제 실패(카드 거절 등) → 차감했던 재고를 모두 복구, 주문 상태 `FAILED`, "결제에 실패했습니다".
- E-B3: PG 응답 지연/타임아웃 → 결제 미확정으로 간주, 재고 복구, 주문 `FAILED` 처리(재시도는 신규 주문으로).
- E-B4: 삭제된 상품을 주문에 포함 → 주문 거부(400/404).

### 시나리오 C: 어드민 상품 관리 (US-08)

**정상 흐름**
1. 어드민이 브랜드를 등록한다 → 브랜드 생성.
2. 해당 브랜드로 상품을 등록한다 → 상품 생성(브랜드는 이미 등록된 것이어야 함).
3. 브랜드를 삭제한다 → 해당 브랜드의 상품도 함께 삭제(soft delete)된다.

**예외 흐름**
- E-C1: 등록되지 않은 브랜드로 상품 등록 시도 → 거부(400).
- E-C2: 상품 수정 시 브랜드 변경 시도 → 거부(브랜드는 수정 불가).

## 6. 인수 조건 (Acceptance Criteria)

| AC ID | 연결 FR | Given / When / Then |
|-------|---------|---------------------|
| AC-01 | FR-01 | given 삭제되지 않은 상품들이 존재하고 / when `sort=price_asc&brandId=1`로 목록을 조회하면 / then 브랜드 1의 상품만 가격 오름차순으로 페이징되어 반환된다 |
| AC-02 | FR-02 | given 삭제된 상품 ID가 주어지고 / when 상품 상세를 조회하면 / then 404가 반환된다 |
| AC-03 | FR-04 | given 사용자가 이미 상품 A를 좋아요했고 / when 같은 상품 A에 좋아요를 다시 요청하면 / then 2xx가 반환되고 좋아요 수는 증가하지 않는다 |
| AC-04 | FR-05 | given 사용자가 상품 B를 좋아요한 적이 없고 / when 상품 B 좋아요 취소를 요청하면 / then 2xx가 반환되고 상태 변화는 없다 |
| AC-05 | FR-07, FR-09 | given 주문 항목 중 한 상품의 재고가 요청 수량보다 적고 / when 주문을 생성하면 / then 주문은 실패하고 어떤 상품의 재고도 차감되지 않는다 |
| AC-06 | FR-07 | given 정상 재고의 상품들로 / when 주문을 생성하면 / then 재고가 차감되고 주문이 `PENDING`으로 생성되며 상품 정보가 스냅샷으로 저장된다 |
| AC-07 | FR-08 | given `PENDING` 주문이 있고 / when PG 결제가 성공하면 / then 주문 상태가 `PAID`로 변경된다 |
| AC-08 | FR-08 | given `PENDING` 주문이 있고 / when PG 결제가 실패하면 / then 차감된 재고가 복구되고 주문 상태가 `FAILED`로 변경된다 |
| AC-09 | FR-10 | given 사용자 X의 주문이 있고 / when 사용자 Y가 그 주문 상세를 조회하면 / then 접근이 거부된다 |
| AC-10 | FR-11 | given 브랜드와 그 소속 상품이 있고 / when 어드민이 브랜드를 삭제하면 / then 해당 브랜드의 상품도 함께 삭제 처리된다 |
| AC-11 | FR-12 | given 등록되지 않은 브랜드 ID로 / when 어드민이 상품을 등록하면 / then 요청이 거부된다 |

# 01. 요구사항 분석 및 기능 명세

> 이 문서는 **개발자만 보는 문서가 아닙니다.** 기획자, 디자이너, 프론트엔드 개발자, QA가
> 같은 용어로 같은 기능을 이야기할 수 있도록 만든 **협업의 기준 문서**입니다.
> 기술 용어는 최소화하고, 꼭 필요한 경우 풀어서 설명합니다.

---

## 0. 한눈에 보기

**"좋아요 누르고, 상품을 담아 한 번에 주문하고 결제하는 감성 이커머스"** 입니다.

```
브랜드를 둘러보고  →  마음에 드는 상품에 좋아요  →  여러 상품을 한 번에 주문  →  결제
```

이번 설계의 목표는 **개발에 들어갈 수 있는 수준으로 기능과 데이터 구조를 정리**하는 것입니다.
복잡한 대규모 트래픽 대응(동시성·멱등성·일관성·느린조회·동시주문 등)은 **이번 범위가 아니며**, 기본 기능이 모두 동작한 뒤 별도로 다룹니다.

---

## 1. 설계 범위

### 포함 (설계 대상)
| 영역 | 내용 |
|---|---|
| 상품·브랜드 조회 | 상품 목록 / 상품 상세 / 브랜드 조회 (비로그인도 가능) |
| 좋아요 | 상품 좋아요 등록 / 취소 (여러 번 눌러도 결과가 같은 **멱등** 동작) |
| 주문·결제 | 주문 생성 → 재고 차감 → 포인트 차감 → 외부 결제 시스템 연동 |
| 어드민 | 브랜드·상품 등록/수정/삭제 (브랜드 삭제 시 소속 상품도 함께 삭제) |

### 제외
- **회원가입 / 내 정보 조회** - 완료
- **인증/인가** - 시나리오상 주요 범위가 아님. 헤더(`X-Loopers-LoginId`)로 사용자를 **식별만** 함
- **쿠폰** - 시나리오 배경에만 언급, API 명세에 없으므로 이번 범위 외

## 2. 문제 상황 재해석

> 요구사항을 "무엇을 만들까"가 아니라 **"지금 어떤 문제를 풀려는가"** 로 다시 봅니다.
> 같은 기능이 라도 사용자/비즈니스/시스템 입장에서 풀려는 문제가 다릅니다.

### 2.1 상품·브랜드 조회 (비로그인 허용)
- **사용자 관점**: 가입하지 않은 사람도 상품을 둘러볼 수 있어야 한다. "로그인부터 하세요"라는 마찰을 없앤다.
- **비즈니스 관점**: 둘러보다가 좋아요·주문 단계에서 자연스럽게 로그인을 유도한다.
- **시스템 관점**: 로그인 여부에 따라 "내가 좋아요 한 상품인지" 같은 개인화 정보만 분기해서 보여주면 된다.

### 2.2 좋아요 (멱등)
- **사용자 관점**: 버튼을 두 번 눌러도, 네트워크가 끊겨 재시도돼도 결과가 흔들리면 안 된다.
- **비즈니스 관점**: 좋아요 수는 인기 정렬의 기준이 된다. 중복으로 부풀려지면 정렬이 신뢰를 잃는다.
- **시스템 관점**: 한 사용자가 한 상품에 누른 좋아요는 **항상 1건**으로 수렴해야 한다.
  > "좋아요를 등록한다"가 아니라 → **"같은 사용자·상품 조합은 결과가 하나로만 정해지게 한다"** 는 문제.

### 2.3 주문 & 결제
- **사용자 관점**: "결제는 실패했는데 재고만 빠졌다" 같은 어긋남을 겪으면 안 된다.
- **비즈니스 관점**: 재고는 한정 자원이고 포인트는 금전적 가치다. 둘 다 잘못 차감되면 곧바로 손해다.
- **시스템 관점**: 재고 차감·포인트 차감·외부 결제 결과가 **서로 어긋나지 않게** 맞추는 것이 핵심.
  > "주문 실패 시 결제를 취소한다" → **"결제 성공/실패와 주문 상태가 어긋나지 않게 일관성을 유지한다"** 는 문제.

### 2.4 주문 스냅샷
- **사용자 관점**: 주문 후 상품 가격·이름이 바뀌어도, 내 주문 내역엔 **결제 당시 정보**가 그대로 보여야 한다.
- **비즈니스 관점**: 환불·정산의 기준은 "거래한 그 순간"의 상품 정보다.
- **시스템 관점**: 주문 내역은 상품 원본을 따라가지 않고, **주문 시점 정보를 복사해 보관**한다.

### 2.5 브랜드 삭제
- **사용자/운영 관점**: 브랜드를 내리면 그 브랜드 상품도 더는 노출되지 않아야 한다.
- **비즈니스 관점**: 단, 이미 일어난 과거 주문은 사라지면 안 된다(스냅샷으로 보존).
- **시스템 관점**: 데이터를 실제로 지우면 과거 주문이 깨진다 → **"지운 표시"만 남기는 방식(soft delete)** 으로 통일.

## 3. 유비쿼터스 언어 (용어 사전)

> 같은 개념을 부르는 말이 사람마다 다르면 설계가 어긋납니다.
> 아래 단어는 **기획서·이 문서·코드·DB에서 모두 동일하게** 사용합니다.

| 한글 | 코드/영문 | 정의 |
|---|---|---|
| 회원 | `User` | 로그인 헤더로 식별되는 사용자 (가입은 1주차 완료) |
| 브랜드 | `Brand` | 상품을 공급하는 주체 |
| 상품 | `Product` | 브랜드에 속한 판매 단위. 이름·설명·가격·재고를 가짐 |
| 재고 | `stock` | 상품이 팔 수 있는 수량 (상품이 직접 보유) |
| 좋아요 | `Like` | (회원, 상품) 한 쌍당 최대 1건 |
| 포인트 | `Point` | 회원이 보유한 잔액. 결제 시 차감되는 수단 |
| 주문 | `Order` | 한 회원이 한 번에 묶어 주문한 거래 단위 |
| 주문 항목 | `OrderItem` | 주문에 담긴 개별 상품 줄. 주문 시점 정보를 스냅샷으로 보관 |
| 스냅샷 | snapshot | 주문 시점의 상품명·단가를 복사해 둔 값 |
| 결제 | `Payment` | 외부 결제 시스템 호출로 주문 금액을 처리하는 행위 |
| 결제 게이트웨이 | `PaymentGateway` | 외부 결제 시스템. |


### 3.1 주문 상태 `OrderStatus`
| 값 | 의미 |
|---|---|
| `CREATED` | 주문 생성됨, 결제 대기 |
| `PAID` | 결제 성공 (완료) |
| `FAILED` | 결제 실패 → 차감했던 재고·포인트 원복 |

## 4. 액터와 시스템

- **액터(사람)**: 비로그인 방문자, 로그인 회원, 어드민(운영자)
- **핵심 도메인**: 브랜드 / 상품 / 좋아요 / 주문 / 결제
- **외부 시스템**: 결제 게이트웨이(PG)

```
[유저 스토리]
- 방문자는 로그인 없이 브랜드와 상품 목록을 둘러볼 수 있다.

```

## 6. 기능 명세

> 각 기능을 **정상 흐름(Main) / 예외(Exception)** 로 나눠 정리합니다.
> "로그인 여부", "없을 때 어떻게", "중복일 때 어떻게" 같은 **조건과 예외**를 빠뜨리지 않는 것이 핵심입니다.

### F1. 상품 목록 조회 - `GET /api/v1/products`
- **사용 가능**: 누구나 (로그인 불필요)
- **요청 파라미터**: `brandId`(선택), `sort`(latest / price_asc / likes_desc), `page`(기본 0), `size`(기본 20)
- **Main**
  1. 정렬 기준에 따라 상품 목록을 페이지 단위로 조회
  2. 삭제된(soft delete) 브랜드·상품은 제외
  3. 로그인한 경우에만 각 상품에 "내가 좋아요 했는지" 표시
- **Exception**: 잘못된 `sort` 값 → 400
- **참고**: `latest`가 기본, `price_asc`·`likes_desc`는 선택 구현

### F2. 상품 상세 조회 - `GET /api/v1/products/{productId}`
- **Main**: 상품 정보 + 브랜드 요약 + 좋아요 수 (+ 로그인 시 내 좋아요 여부) 응답
- **Exception**: 없거나 삭제된 상품 → 404

### F3. 브랜드 조회 - `GET /api/v1/brands/{brandId}`
- **Main**: 브랜드 정보 응답
- **Exception**: 없거나 삭제된 브랜드 → 404

### F4. 좋아요 등록 - `POST /api/v1/products/{productId}/likes`
- **사용 가능**: 로그인 회원
- **Main**
  1. (회원, 상품) 좋아요가 이미 있는지 확인
  2. 없으면 저장, **이미 있으면 아무 일도 하지 않음**
  3. 결과는 항상 "좋아요 됨" → 200 OK
- **Exception**: 없거나 삭제된 상품 → 404
- **불변식**: (회원, 상품) 좋아요는 **최대 1건** (멱등)

### F5. 좋아요 취소 - `DELETE /api/v1/products/{productId}/likes`
- **Main**
  1. (회원, 상품) 좋아요가 있으면 삭제, **없으면 아무 일도 하지 않음**
  2. 결과는 항상 "좋아요 아님" → 200 OK
- **Exception**: 없거나 삭제된 상품 → 404

### F6. 내가 좋아요 한 상품 목록 - `GET /api/v1/users/{userId}/likes`
- **Main**: 본인이 좋아요 한 상품 목록 응답
- **Exception**: 타인 목록 조회 → 403

### F7. 주문 생성 & 결제 - `POST /api/v1/orders`
- **사용 가능**: 로그인 회원
- **요청 예시**
  ```json
  { "items": [ { "productId": 1, "quantity": 2 }, { "productId": 3, "quantity": 1 } ] }
  ```
- **Main** (하나의 흐름으로 처리, 중간 실패 시 전체 원복)
  1. 각 상품의 재고가 주문 수량 이상인지 확인
  2. 주문 시점의 상품명·단가를 **스냅샷으로 복사**하며 주문 항목 생성, 총액 계산
  3. 회원 포인트가 충분한지 확인하고 차감
  4. 외부 결제 게이트웨이에 결제 요청
  5. 성공 → 재고 차감 확정, 주문 상태 `PAID`
  6. 응답으로 주문 정보 반환
- **Exception**
  - 재고 부족 → 400, 아무것도 차감하지 않음
  - 포인트 부족 → 400, 재고도 차감하지 않음
  - 외부 결제 실패/타임아웃 → 차감했던 재고·포인트 원복, 주문 상태 `FAILED`
  - 없거나 삭제된 상품 포함 → 404
- **불변식**: 재고와 포인트는 **음수가 될 수 없다.**

### F8. 주문 조회 - `GET /api/v1/orders[?startAt=&endAt=]`, `GET /api/v1/orders/{orderId}`
- **Main**: 본인 주문만 조회. 기간 필터 적용 가능
- **Exception**: 타인 주문 조회 → 403, 없는 주문 → 404

### F9. (어드민) 브랜드·상품 관리 - `/api-admin/v1/...`
- **사용 가능**: `X-Loopers-Ldap: loopers.admin` 헤더를 가진 어드민
- **Main**: 브랜드/상품 등록·수정·삭제, 목록·상세 조회
  - 상품 등록 시 브랜드는 **이미 등록된 브랜드**여야 함
  - 상품 수정 시 **브랜드는 변경 불가**
  - **브랜드 삭제 시 소속 상품도 함께 삭제** (모두 soft delete)
- **Exception**: Ldap 헤더 없음/불일치 → 403
- **주의**: 진행 중인 과거 주문은 보존됨 (스냅샷 덕분에 그대로 표시 가능)

## 7. 비기능 요구사항 / 제약

- **식별**: 로그인 헤더(`X-Loopers-LoginId`)로 사용자를 식별. 인증/암호화는 이번 범위 아님.
- **삭제 정책**: 모든 도메인은 실제 삭제 대신 **soft delete**(`deleted_at` 표시)를 사용 → 과거 주문 무결성 보존.
- **데이터 정합성**: 같은 (회원, 상품) 좋아요는 1건만 존재(유니크 제약). 주문 항목은 스냅샷으로 상품 변경과 격리.
- **멱등성**: 좋아요 등록/취소는 여러 번 호출해도 결과가 동일.

## 8. 아직 결정되지 않은 것

> 추측해서 결정하지 않고, 합의가 필요한 지점을 드러냅니다.

| # | 구분 | 질문 | 선택지 / 영향 |
|---|---|---|---|
| Q1 | 정책 | 포인트로 **전액**만 결제 가능한가, 부족분은 외부 PG가 부담하는가? | A. 전액 포인트(단순) / B. 포인트+PG 분할(현실적, 계산 복잡) |
| Q2 | 정책 | 좋아요 수는 매번 집계하는가, 별도 컬럼에 누적하는가? | A. 매번 COUNT(단순·정확, 느릴 수 있음) / B. 누적 컬럼(빠름, 정합성 관리 필요) |
| Q3 | 경계 | 주문 생성과 결제는 한 흐름인가, 분리(주문 먼저 → 결제 별도)인가? | A. 단일 흐름(이번 채택, 구현 단순) / B. 분리(확장 유리, 복잡) - 시나리오상 결제는 추후 확장 |
| Q4 | 확장 | 좋아요·주문 트래픽이 커지면? | 이번 범위 밖. "나아가며"에서 캐시·비동기 등으로 별도 대응 |

## 9. 향후 과제

시나리오 "나아가며"에 따라, **기본 기능이 모두 동작한 뒤** 다음을 별도로 다룹니다.
- 동시성 (동시 주문 시 재고 정확성)
- 멱등성 강화 / 일관성
- 느린 조회 최적화 (인기순 정렬 등)

이번 설계는 **단순하고 정확하게 동작하는 것**을 우선하며, 위 항목은 의도적으로 범위에서 제외합니다.

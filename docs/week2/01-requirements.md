# 01. 요구사항 명세

> **설계 범위**: 상품 목록/상세/브랜드 조회 · 상품 좋아요 등록·취소 · 주문 생성 및 결제 흐름  
> **제외 범위**: 회원가입, 내 정보 조회  
> **기준 문서**: `docs/dev-guide/ecommerce-scenario.md`

---

## 1. 서비스 개요

루프팩 이커머스는 사용자가 브랜드별 상품을 탐색하고, 좋아요를 누르고, 여러 상품을 한 번에 주문·결제할 수 있는 감성 커머스 서비스입니다.  
유저의 행동 데이터는 향후 랭킹·추천 기능으로 확장될 수 있도록 설계합니다.

---

## 2. 인증 / 인가 정책

| 구분 | 방식 | 헤더 |
|------|------|------|
| 일반 사용자 | 헤더 기반 식별 (구현 범위 외) | `X-Loopers-LoginId`, `X-Loopers-LoginPw` |
| 어드민 | LDAP 식별 (구현 범위 외) | `X-Loopers-Ldap: loopers.admin` |

- 인증·인가 로직 자체는 구현하지 않으나, **헤더를 통해 사용자를 식별**하는 규약은 준수합니다.
- 유저는 타 유저의 정보에 직접 접근할 수 없습니다.

---

## 3. 유저 시나리오

```
1. 사용자가 상품 목록 페이지에 진입한다.
   → 브랜드 필터, 정렬(최신순·가격순·좋아요순)을 적용해 상품을 탐색한다.

2. 사용자가 특정 상품의 상세 페이지로 이동한다.
   → 상품명, 가격, 재고, 브랜드 정보를 확인한다.

3. 사용자가 마음에 드는 상품에 좋아요를 누른다.
   → 이미 좋아요가 되어 있으면 무시(멱등). 취소도 동일하게 멱등 동작.

4. 사용자가 여러 상품을 선택해 주문을 요청한다.
   → 재고가 충분하면 차감 후 주문을 생성한다.
   → 주문 시점의 상품 정보(가격, 명칭 등)를 스냅샷으로 저장한다.
   → 재고 부족 시 주문 전체를 실패 처리한다.

5. 사용자가 자신의 주문 내역을 조회한다.
   → 기간 필터(startAt, endAt)를 통해 특정 기간의 주문만 조회한다.
```

## 4. 기능 요구사항

### 4-1. 브랜드 (Brands)

#### 사용자 API

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| GET | `/api/v1/brands/{brandId}` | 불필요 | 브랜드 정보 조회 |

**FR-B-01. 브랜드 정보 조회**
- 브랜드 ID로 단일 브랜드 정보를 조회한다.
- 존재하지 않는 브랜드 ID 요청 시 `404 Not Found`를 반환한다.
- 응답에는 브랜드명, 설명 등 고객에게 공개 가능한 정보만 포함한다.

#### 어드민 API

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| GET | `/api-admin/v1/brands` | 필요 | 브랜드 목록 조회 (페이지네이션) |
| GET | `/api-admin/v1/brands/{brandId}` | 필요 | 브랜드 상세 조회 |
| POST | `/api-admin/v1/brands` | 필요 | 브랜드 등록 |
| PUT | `/api-admin/v1/brands/{brandId}` | 필요 | 브랜드 정보 수정 |
| DELETE | `/api-admin/v1/brands/{brandId}` | 필요 | 브랜드 삭제 |

**FR-BA-01. 브랜드 목록 조회 (어드민)**
- 기본값: `page=0`, `size=20`
- 전체 브랜드를 페이지네이션으로 조회한다.

**FR-BA-02. 브랜드 등록 (어드민)**
- 브랜드명은 필수값이며 중복을 허용하지 않는다.

**FR-BA-03. 브랜드 수정 (어드민)**
- 존재하지 않는 브랜드 수정 시 `404 Not Found`를 반환한다.

**FR-BA-04. 브랜드 삭제 (어드민)**
- 브랜드 삭제 시 해당 브랜드에 속한 **상품도 함께 삭제**되어야 한다.
- 삭제된 브랜드에 연결된 상품 조회 시 `404 Not Found`를 반환한다.

### 4-2. 상품 (Products)

#### 사용자 API

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| GET | `/api/v1/products` | 불필요 | 상품 목록 조회 |
| GET | `/api/v1/products/{productId}` | 불필요 | 상품 상세 조회 |

**FR-P-01. 상품 목록 조회**

| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|----------|------|------|--------|------|
| `brandId` | Long | 선택 | - | 특정 브랜드 필터링 |
| `sort` | String | 선택 | `latest` | `latest` / `price_asc` / `likes_desc` |
| `page` | int | 선택 | `0` | 페이지 번호 |
| `size` | int | 선택 | `20` | 페이지당 상품 수 |

- 정렬 기준 `latest`는 **필수 구현**, `price_asc` / `likes_desc`는 선택 구현.
- 삭제된 상품 및 브랜드에 속한 상품은 목록에 노출되지 않는다.
- 응답에는 상품명, 가격, 브랜드명, 좋아요 수 등 고객용 정보만 포함한다.

**FR-P-02. 상품 상세 조회**
- 상품 ID로 단일 상품의 상세 정보를 조회한다.
- 존재하지 않는 상품 ID 요청 시 `404 Not Found`를 반환한다.
- 응답에 재고 수량을 포함할지는 UX 판단에 따른다 (재고 0 표기 혹은 품절 여부만 표기).

#### 어드민 API

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| GET | `/api-admin/v1/products` | 필요 | 상품 목록 조회 (페이지네이션, 브랜드 필터) |
| GET | `/api-admin/v1/products/{productId}` | 필요 | 상품 상세 조회 |
| POST | `/api-admin/v1/products` | 필요 | 상품 등록 |
| PUT | `/api-admin/v1/products/{productId}` | 필요 | 상품 정보 수정 |
| DELETE | `/api-admin/v1/products/{productId}` | 필요 | 상품 삭제 |

**FR-PA-01. 상품 등록 (어드민)**
- 상품의 브랜드는 이미 등록된 브랜드여야 한다. 미등록 브랜드 지정 시 `400 Bad Request`.
- 상품명, 가격, 재고, 브랜드 ID는 필수값이다.
- 가격은 0보다 커야 하고, 재고는 0 이상이어야 한다.

**FR-PA-02. 상품 수정 (어드민)**
- 상품의 **브랜드는 수정할 수 없다**.
- 존재하지 않는 상품 수정 시 `404 Not Found`를 반환한다.

**FR-PA-03. 상품 삭제 (어드민)**
- 소프트 딜리트(soft delete) 방식으로 처리하여 주문 이력의 스냅샷 정합성을 유지한다.

---

### 4-3. 좋아요 (Likes)

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| POST | `/api/v1/products/{productId}/likes` | 필요 | 상품 좋아요 등록 |
| DELETE | `/api/v1/products/{productId}/likes` | 필요 | 상품 좋아요 취소 |
| GET | `/api/v1/users/{userId}/likes` | 필요 | 내가 좋아요 한 상품 목록 조회 |

**FR-L-01. 좋아요 등록 (멱등)**
- 이미 좋아요한 상품에 다시 요청해도 에러 없이 `200 OK`(또는 `204 No Content`)를 반환한다.
- 존재하지 않는 상품에 좋아요 시 `404 Not Found`를 반환한다.
- 삭제된 상품에는 좋아요를 등록할 수 없다.

**FR-L-02. 좋아요 취소 (멱등)**
- 이미 취소된 상태에서 다시 취소 요청해도 에러 없이 처리한다.

**FR-L-03. 좋아요 목록 조회**
- 본인의 좋아요 목록만 조회 가능하다 (타 유저 접근 차단).
- 삭제된 상품의 좋아요 항목은 목록에서 제외한다.

### 4-4. 주문 (Orders)

#### 사용자 API

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| POST | `/api/v1/orders` | 필요 | 주문 요청 |
| GET | `/api/v1/orders` | 필요 | 유저의 주문 목록 조회 |
| GET | `/api/v1/orders/{orderId}` | 필요 | 단일 주문 상세 조회 |

**FR-O-01. 주문 생성**

요청 예시:
```json
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ]
}
```

- `items`는 1개 이상이어야 하며, 각 항목의 `quantity`는 1 이상이어야 한다.
- **재고 확인**: 요청한 수량이 현재 재고보다 많으면 `400 Bad Request` 또는 `409 Conflict`를 반환한다.
- **재고 차감**: 재고 확인 후 즉시 차감한다. 동시 주문 시 재고 정합성이 보장되어야 한다.
- **스냅샷 저장**: 주문 시점의 상품명, 가격, 브랜드명 등을 주문 항목에 스냅샷으로 저장한다.
  - 이후 상품 정보가 변경되어도 주문 이력에는 주문 당시 정보가 유지된다.
- 존재하지 않는 상품을 주문하면 `404 Not Found`를 반환한다.
- 삭제된 상품은 주문할 수 없다.

**FR-O-02. 주문 목록 조회**

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|------|------|
| `startAt` | date (yyyy-MM-dd) | 선택 | 조회 시작일 |
| `endAt` | date (yyyy-MM-dd) | 선택 | 조회 종료일 |

- 본인의 주문만 조회 가능하다.
- `startAt` > `endAt`인 경우 `400 Bad Request`를 반환한다.

**FR-O-03. 단일 주문 상세 조회**
- 본인의 주문만 조회 가능하다.
- 존재하지 않거나 타인의 주문 조회 시 `404 Not Found`를 반환한다.

#### 어드민 API

| METHOD | URI | 인증 | 설명 |
|--------|-----|------|------|
| GET | `/api-admin/v1/orders` | 필요 | 전체 주문 목록 조회 (페이지네이션) |
| GET | `/api-admin/v1/orders/{orderId}` | 필요 | 단일 주문 상세 조회 |

## 5. 비기능 요구사항

### 5-1. 데이터 정합성

| 요구사항 | 설명 |
|----------|------|
| **주문 스냅샷** | 주문 시 상품명·가격·브랜드명을 `order_items` 테이블에 복사 저장 |
| **재고 원자성** | 재고 차감은 트랜잭션 내에서 원자적으로 처리 |
| **소프트 딜리트** | 상품·브랜드 삭제는 물리 삭제가 아닌 `deleted_at` 컬럼으로 관리 |

### 5-2. 멱등성

| 요구사항 | 설명 |
|----------|------|
| **좋아요 등록** | 이미 존재하는 좋아요 재요청 시 정상 응답 반환 |
| **좋아요 취소** | 이미 취소된 상태에서 재요청 시 정상 응답 반환 |

### 5-3. 응답 정책

- 공통 에러 응답 형식을 통일한다 (`code`, `message` 포함).
- 고객용 API와 어드민용 API는 응답 필드를 별도 DTO로 분리하여, 고객에게 불필요한 내부 정보가 노출되지 않도록 한다.

### 5-4. 페이지네이션

- 목록 조회는 기본 `page=0`, `size=20` 오프셋 페이지네이션을 사용한다.
- 응답에 `totalElements`, `totalPages`, `page`, `size`를 포함한다.

## 6. 도메인 요약

```
Brand (브랜드)
  ├── 브랜드명, 설명, 등록일, 삭제 여부
  └── 1:N → Product

Product (상품)
  ├── 상품명, 가격, 재고, 삭제 여부
  └── N:1 → Brand

Like (좋아요)
  ├── 유저 ID, 상품 ID, 등록일
  └── Unique(유저 ID, 상품 ID)

Order (주문)
  ├── 유저 ID, 주문 상태, 주문 총금액, 주문일
  └── 1:N → OrderItem

OrderItem (주문 항목 — 스냅샷)
  └── 상품 ID(참조), 주문 시 상품명, 주문 시 가격, 수량
```

## 7. 경계 조건 및 예외 케이스 요약

| 상황 | 처리 |
|------|------|
| 존재하지 않는 리소스 조회 | `404 Not Found` |
| 재고 부족 주문 | `400 Bad Request` 또는 `409 Conflict` |
| 삭제된 상품 주문·좋아요 | `400 Bad Request` |
| 미등록 브랜드 상품 등록 | `400 Bad Request` |
| 좋아요 중복 등록/취소 | `200 OK` (멱등 처리) |
| 타인의 주문·좋아요 조회 | `404 Not Found` (노출 차단) |
| 상품 브랜드 수정 시도 | `400 Bad Request` |
| 잘못된 날짜 범위 (startAt > endAt) | `400 Bad Request` |

# 01. 요구사항 명세

## 서비스 개요

좋아요를 누르고, 여러 브랜드 상품을 한 번에 주문하는 감성 이커머스 서비스.
유저 행동 데이터는 랭킹과 추천으로 확장될 수 있다.

---

## 액터 (Actor)

| 액터 | 설명 |
|---|---|
| 회원 (Member) | 상품 탐색, 좋아요, 주문 가능한 일반 유저 |
| 어드민 (Admin) | 브랜드/상품 CRUD, 주문 조회 가능한 운영자 |
| 결제 PG사 | 미래 확장 예정 (현재 설계 범위 밖) |

**인증 방식**
- 회원: `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더로 식별 (인증/인가 구현 없음)
- 어드민: `X-Loopers-Ldap: loopers.admin` 헤더로 식별

---

## 설계 범위

| 도메인 | 포함 여부 | 비고 |
|---|---|---|
| 브랜드 조회 | ✅ | 대고객 + 어드민 |
| 상품 목록/상세 | ✅ | 대고객 + 어드민 |
| 좋아요 | ✅ | 대고객 |
| 주문 | ✅ | 대고객 + 어드민 |
| 회원가입 / 내 정보 조회 | ❌ | 설계 범위 제외 |
| 결제 | ❌ (트랜잭션 경계는 설계에 포함) | 추후 개발 |

## 유스케이스 흐름

---

### UC-01. 브랜드 조회

**유저 스토리**
- 사용자는 특정 브랜드의 정보를 조회할 수 있다.

**기능 흐름**

```
[Main Flow]
1. 사용자가 브랜드 ID로 브랜드 정보를 요청한다.
2. 시스템은 해당 브랜드의 공개 정보를 반환한다.

[Exception Flow]
- 존재하지 않는 브랜드 ID → 404 Not Found
- 소프트 딜리트된 브랜드 ID → 404 Not Found (삭제된 브랜드는 노출하지 않음)
```

---

### UC-02. 상품 목록 조회

**유저 스토리**
- 사용자는 전체 상품 목록을 조회할 수 있다.
- 특정 브랜드로 필터링하거나 원하는 기준으로 정렬할 수 있다.

**기능 흐름**

```
[Main Flow]
1. 사용자가 상품 목록을 요청한다. (정렬 기본값: latest)
2. 시스템은 삭제되지 않은 상품 목록을 페이지네이션으로 반환한다.

[Alternate Flow]
- brandId 파라미터가 있으면 해당 브랜드 상품만 필터링
- sort=price_asc → 가격 오름차순 정렬
- sort=likes_desc → like_count 내림차순 정렬

[Exception Flow]
- 존재하지 않는 brandId로 필터링 → 빈 목록 반환 (404 아님)
- 지원하지 않는 sort 값 → 400 Bad Request
```

**페이지네이션**: `page` 기본값 0, `size` 기본값 20

### UC-04. 좋아요 등록

**유저 스토리**
- 사용자는 마음에 드는 상품에 좋아요를 등록할 수 있다.
- 이미 좋아요한 상품을 다시 눌러도 에러 없이 처리된다. (멱등 처리)

**기능 흐름**

```
[Main Flow]
1. 로그인 사용자가 특정 상품에 좋아요를 요청한다.
2. 시스템은 해당 (userId, productId) 쌍의 좋아요가 존재하는지 확인한다.
3. 존재하지 않으면 좋아요를 저장한다.
4. 해당 상품의 like_count를 1 증가시킨다.
5. 200 OK를 반환한다.

[Alternate Flow]
- 이미 좋아요가 존재하면 저장 없이 바로 200 OK 반환 (like_count 변경 없음)

[Exception Flow]
- 존재하지 않는 상품 ID → 404 Not Found
- 소프트 딜리트된 상품 → 404 Not Found
- 인증 헤더 없음 → 401 Unauthorized
```

```
[Alternate Flow]
```

**유저 스토리**

**기능 흐름**

```
- 인증 헤더 없음 → 401 Unauthorized
```

### UC-07. 주문 생성

**유저 스토리**
- 사용자는 여러 상품을 한 번에 주문할 수 있다.
- 주문 항목 중 하나라도 재고가 부족하면 주문 전체가 실패한다. (올-오어-낫싱)
- 주문 시점의 상품 정보는 변경되지 않도록 스냅샷으로 저장된다.

**기능 흐름**

```
[Main Flow]
1. 로그인 사용자가 상품 목록과 수량을 담아 주문을 요청한다.
2. 시스템은 각 상품의 존재 여부와 재고를 확인한다.
3. 모든 항목의 재고가 충분하면 재고를 차감하고 주문을 생성한다.
4. 주문 시점의 상품명·가격·브랜드명을 order_item_snapshots에 저장한다.
5. 주문 결과를 반환한다.

※ 2~4번은 단일 트랜잭션으로 처리된다.

[Exception Flow]
- 하나라도 재고 부족 → 주문 전체 실패 (400 Bad Request, 품절 상품 식별 정보 포함)
- 존재하지 않는 상품 ID 포함 → 400 Bad Request
- 소프트 딜리트된 상품 포함 → 400 Bad Request
- 인증 헤더 없음 → 401 Unauthorized
```

**정책**
- 재고 차감과 주문 생성은 동일 트랜잭션으로 처리
- 결제는 별도 트랜잭션. 결제 실패 시 보상 트랜잭션으로 재고 롤백
- 주문 후 상품 가격/이름 변경은 주문 내역에 영향 없음 (스냅샷 보장)

**OrderItem 상태값**

| 상태 | 의미 |
|---|---|
| `ORDERED` | 정상 주문됨 |
| `CANCELLED` | 주문 후 취소됨 (추후 확장) |

> **설계 결정 기록 — 부분 주문 채택하지 않은 이유**  
> 초기 설계에서는 재고 부족 항목을 SKIPPED 처리하고 나머지만 주문하는 부분 주문 방식을 검토했다.  
> 그러나 이 서비스는 B2C이므로 사용자가 명시적으로 선택한 상품·수량·금액이 동의 없이 바뀌어서는 안 된다.  
> 결제 금액이 사용자 의도와 달라지는 UX는 전자상거래법 관점에서도 문제가 될 수 있다.  
> 부분 주문은 B2B 발주나 정기 구독 배송처럼 "있는 것만 보내줘"가 자연스러운 맥락에서 적합하다.  
> 따라서 하나라도 재고 부족이면 전체 실패 후 품절 상품을 명시적으로 안내하는 방식을 채택한다.

### UC-08. 주문 목록 조회

**유저 스토리**
- 사용자는 자신의 주문 목록을 날짜 범위로 조회할 수 있다.

**기능 흐름**

```
[Main Flow]
1. 로그인 사용자가 startAt, endAt을 포함하여 주문 목록을 요청한다.
2. 시스템은 해당 기간 내 유저의 주문 목록을 반환한다.

[Exception Flow]
- startAt > endAt → 400 Bad Request
- 인증 헤더 없음 → 401 Unauthorized
```

**유저 스토리**
- 사용자는 특정 주문의 상세 정보를 조회할 수 있다.
- 다른 사용자의 주문은 조회할 수 없다.

**기능 흐름**

```
[Main Flow]

2. 시스템은 주문 항목(스냅샷 포함)과 상태를 반환한다.

[Exception Flow]

- 요청한 주문이 다른 유저의 주문 → 403 Forbidden
- 인증 헤더 없음 → 401 Unauthorized
```

### UC-10. 어드민 브랜드 관리

**유저 스토리**
- 어드민은 브랜드를 등록, 수정, 삭제할 수 있다.
- 브랜드 삭제 시 해당 브랜드의 모든 상품도 함께 삭제된다.

**기능 흐름**

```
[브랜드 등록 Main Flow]
1. 어드민이 브랜드 정보를 입력하여 등록을 요청한다.
2. 시스템은 브랜드를 저장하고 생성된 브랜드 정보를 반환한다.

[브랜드 수정 Main Flow]
1. 어드민이 브랜드 ID와 수정할 정보를 요청한다.
2. 시스템은 브랜드 정보를 수정하고 반환한다.

[브랜드 삭제 Main Flow]
1. 어드민이 브랜드 ID로 삭제를 요청한다.
2. 시스템은 해당 브랜드를 소프트 딜리트 처리한다. (deleted_at 설정)
3. 해당 브랜드의 모든 상품도 소프트 딜리트 처리한다.

[Exception Flow]
- 존재하지 않는 브랜드 ID → 404 Not Found
- 어드민 헤더 없음 → 401 Unauthorized
```

---

### UC-11. 어드민 상품 관리

**유저 스토리**
- 어드민은 상품을 등록, 수정, 삭제할 수 있다.
- 상품 등록 시 이미 등록된 브랜드여야 한다.
- 상품의 브랜드는 수정할 수 없다.

**기능 흐름**

```
[상품 등록 Main Flow]
1. 어드민이 상품 정보와 브랜드 ID를 입력하여 등록을 요청한다.
2. 시스템은 해당 브랜드가 존재하고 삭제되지 않았는지 확인한다.
3. 상품과 초기 재고를 저장하고 반환한다.

[상품 수정 Main Flow]
1. 어드민이 상품 ID와 수정할 정보를 요청한다. (브랜드 ID 제외)
2. 시스템은 상품 정보를 수정하고 반환한다.

[상품 삭제 Main Flow]
1. 어드민이 상품 ID로 삭제를 요청한다.
2. 시스템은 해당 상품을 소프트 딜리트 처리한다.

[Exception Flow]
- 존재하지 않는 브랜드 ID로 상품 등록 → 400 Bad Request
- 소프트 딜리트된 브랜드에 상품 등록 → 400 Bad Request
- 상품 수정 요청에 브랜드 ID 포함 → 400 Bad Request
- 존재하지 않는 상품 ID → 404 Not Found
- 어드민 헤더 없음 → 401 Unauthorized
```

**유저 스토리**
- 어드민은 전체 주문 현황을 조회할 수 있다.

**기능 흐름**

```
[Main Flow]
1. 어드민이 주문 목록 또는 특정 주문 상세를 요청한다.
2. 시스템은 전체 유저의 주문 데이터를 반환한다.

[Exception Flow]
- 존재하지 않는 주문 ID → 404 Not Found
- 어드민 헤더 없음 → 401 Unauthorized
```

## 비기능 요구사항 (설계 결정사항)

| 항목 | 결정 | 이유 |
|---|---|---|
| 재고 락 | `stocks` 별도 테이블 분리 후 행 단위 락 | 상품 정보 수정과 재고 차감이 서로 블로킹되지 않도록 |
| 스냅샷 저장 | `order_item_snapshots` 별도 테이블 | 주문 이력과 스냅샷 관심사 분리 |
| 소프트 딜리트 | `deleted_at` 컬럼으로 처리 | 이미 주문된 상품/브랜드의 FK 정합성 유지 |
| 좋아요 멱등성 | `(user_id, product_id)` DB 복합 유니크 제약 | 중복 좋아요 방지를 DB 레벨에서 보장 |
| 주문 실패 정책 | 하나라도 재고 부족 시 전체 주문 실패 (올-오어-낫싱) | B2C에서 사용자 동의 없이 결제 금액이 바뀌어서는 안 됨 |

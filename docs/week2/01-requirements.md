# 01. Requirements

## 목적

이 문서는 루퍼스의 "우리가 함께 만들어갈 단 하나의 감성 이커머스" 시나리오를 기준으로 2주차 설계 대상 요구사항을 정리한다.
요구사항 문서에서 사용하는 용어는 `docs/common/ubiquitous-language.md`를 기준으로 한다.
요구사항 분석은 `.claude/skills/requirements-analysis/SKILL.md`의 흐름을 따른다.

## 범위

2주차 설계 대상은 다음 기능이다.

- 상품 목록 조회
- 상품 상세 조회
- 브랜드 조회
- 상품 좋아요 등록/취소
- 주문 생성 및 결제 흐름

회원가입, 내 정보 조회, 비밀번호 변경은 2주차 설계 대상에서 제외한다.

## 전제

- 요구사항의 기준 문서는 `obsidian-notes/round-docs/우리가 함께 만들어갈 단 하나의 감성 이커머스 3625f76d26548054b26bde8e3fe2448c.md`이다.
- 명시적인 보완 사유가 없는 한 API URI, 기능 범위, 도메인 경계는 메인 시나리오 문서를 따른다.
- 대고객 API는 `/api/v1` prefix를 사용한다.
- 로그인 사용자가 필요한 기능은 요청 헤더의 `X-Loopers-LoginId`, `X-Loopers-LoginPw`를 통해 사용자를 식별한다.
- 인증과 인가는 주요 구현 범위가 아니므로, 이 문서에서는 사용자 식별이 완료된 상태를 전제로 요구사항을 정리한다.
- 사용자는 타 사용자의 정보에 직접 접근할 수 없다.
- 어드민 API는 `/api-admin/v1` prefix를 사용하지만, 2주차 설계 문서는 대고객 기능을 우선으로 다룬다.

## 문제 상황 재정의

이번 요구사항은 단순히 상품, 좋아요, 주문 API를 만드는 문제가 아니다.
사용자가 상품을 탐색하고 선호를 표현한 뒤 여러 상품을 한 번에 주문하는 과정에서, 시스템이 상품 상태와 재고, 주문 이력, 결제 결과를 일관되게 유지해야 하는 문제다.

### 사용자 관점

- 사용자는 브랜드와 상품을 탐색하고, 관심 있는 상품을 좋아요로 저장하고 싶다.
- 사용자는 여러 상품을 한 번에 주문했을 때 주문 당시의 상품 정보와 주문 결과가 안정적으로 남기를 기대한다.
- 사용자는 같은 좋아요 요청이나 취소 요청을 반복해도 결과가 이상하게 중복되거나 실패하지 않기를 기대한다.

### 비즈니스 관점

- 좋아요 데이터는 상품 선호도, 랭킹, 추천으로 확장될 수 있는 행동 데이터다.
- 주문 데이터는 결제, 재고, 매출, 정산으로 이어지는 핵심 거래 데이터다.
- 주문 이력은 상품명이나 가격이 나중에 바뀌어도 주문 당시 기준으로 보존되어야 한다.

### 시스템 관점

- 상품 조회는 브랜드, 상품 상태, 정렬 기준에 따라 일관된 결과를 제공해야 한다.
- 좋아요는 사용자와 상품의 관계를 중복 없이 관리해야 한다.
- 주문 생성은 재고 확인, 재고 차감, 주문 항목 스냅샷 생성, 결제 연동 경계를 함께 고려해야 한다.
- 결제 실패나 재고 부족 같은 예외 상황에서 주문 상태와 재고가 어긋나지 않아야 한다.

## 유저 시나리오 기반 기능 정의

### 시나리오 1. 사용자는 브랜드와 상품을 탐색한다

- 사용자 스토리: 사용자는 여러 브랜드의 상품을 둘러보고, 관심 있는 상품의 상세 정보를 확인할 수 있다.
- 주요 흐름:
  1. 사용자는 브랜드 ID로 브랜드 정보를 조회한다.
  2. 사용자는 상품 목록을 조회한다.
  3. 사용자는 필요하면 브랜드 ID로 상품 목록을 필터링한다.
  4. 사용자는 상품 상세 정보를 조회한다.
- 예외 흐름:
  - 존재하지 않는 브랜드를 조회하면 실패 응답을 반환한다.
  - 존재하지 않는 상품을 조회하면 실패 응답을 반환한다.
  - 사용자에게 노출되지 않아야 하는 상품은 목록과 상세에서 제외한다.
- 관련 기능: 브랜드 조회, 상품 목록 조회, 상품 상세 조회

### 시나리오 2. 사용자는 관심 있는 상품에 좋아요를 남긴다

- 사용자 스토리: 로그인 사용자는 관심 있는 상품에 좋아요를 등록하거나 취소할 수 있다.
- 주요 흐름:
  1. 로그인 사용자는 상품 상세 또는 목록에서 좋아요 등록을 요청한다.
  2. 시스템은 사용자와 상품의 좋아요 관계가 이미 존재하는지 확인한다.
  3. 좋아요가 없으면 좋아요를 등록한다.
  4. 사용자는 좋아요한 상품 목록을 조회한다.
- 예외 흐름:
  - 이미 좋아요한 상품에 다시 좋아요 등록 요청을 보내도 중복 좋아요를 만들지 않는다.
  - 이미 좋아요가 없는 상품에 좋아요 취소 요청을 보내도 시스템 상태를 변경하지 않는다.
  - 존재하지 않는 상품에 대한 좋아요 요청은 실패 응답을 반환한다.
- 관련 기능: 좋아요 등록, 좋아요 취소, 좋아요한 상품 목록 조회

### 시나리오 3. 사용자는 여러 상품을 한 번에 주문한다

- 사용자 스토리: 로그인 사용자는 여러 상품과 수량을 선택해 한 번에 주문할 수 있다.
- 주요 흐름:
  1. 사용자는 주문할 상품 ID와 수량 목록을 전달한다.
  2. 시스템은 각 상품의 존재 여부, 판매 가능 여부, 재고를 확인한다.
  3. 시스템은 주문 가능한 상품 항목만 주문 대상으로 확정한다.
  4. 시스템은 주문 가능한 상품의 주문 수량만큼 재고를 차감한다.
  5. 시스템은 주문과 주문 항목을 생성한다.
  6. 주문 항목에는 주문 당시의 상품명, 가격, 브랜드명 등 상품 정보 스냅샷을 저장한다.
- 예외 흐름:
  - 주문 항목 중 존재하지 않는 상품이 있으면 주문을 실패 처리한다.
  - 주문 항목 중 재고가 부족한 상품은 주문 실패 항목으로 처리하고, 재고가 충분한 상품은 정상 주문 대상으로 처리한다.
  - 수량이 1 미만인 주문 항목은 실패 응답을 반환한다.
- 관련 기능: 주문 생성

### 시나리오 4. 주문 생성 이후 결제 결과를 반영한다

- 사용자 스토리: 사용자는 주문 금액에 대한 결제 결과가 주문 상태에 일관되게 반영되기를 기대한다.
- 주요 흐름:
  1. 시스템은 생성된 주문의 총액을 계산한다.
  2. 시스템은 외부 결제 시스템에 결제를 요청한다.
  3. 결제가 성공하면 주문을 결제 완료 상태로 변경한다.
  4. 결제가 실패하면 주문을 실패 상태로 변경하거나 보상 처리를 수행한다.
- 예외 흐름:
  - 외부 결제 시스템이 실패를 반환하면 주문 상태와 재고 차감 결과가 어긋나지 않도록 처리한다.
  - 외부 결제 시스템 응답 지연 또는 장애가 발생하면 주문/결제 상태를 기록하고 보상 처리 대상으로 둔다.
- 관련 기능: 주문 생성 및 결제 흐름

## 요구사항 명세

### 브랜드 조회

- 사용자는 브랜드 ID로 브랜드 정보를 조회할 수 있다.
- 브랜드 정보에는 브랜드명과 브랜드 설명 등 사용자에게 노출할 정보가 포함된다.
- 존재하지 않는 브랜드를 조회하면 실패 응답을 반환한다.
- 소프트 삭제된 브랜드는 사용자 조회 대상에서 제외한다.

### 상품 목록 조회

- 사용자는 상품 목록을 페이지 단위로 조회할 수 있다.
- 사용자는 `brandId`로 특정 브랜드의 상품만 조회할 수 있다.
- 기본 정렬 기준은 `latest`다.
- 메인 시나리오의 선택 정렬 기준인 `price_asc`, `likes_desc`를 설계에 반영한다.
- `likes_desc` 정렬은 상품에 저장된 `Product.likeCount`를 기준으로 한다.
- 소프트 삭제된 상품은 목록에서 제외한다.

### 상품 상세 조회

- 사용자는 상품 ID로 상품 상세 정보를 조회할 수 있다.
- 상품 상세에는 상품명, 가격, 브랜드 정보, 재고 또는 판매 가능 상태, 좋아요 수가 포함될 수 있다.
- 존재하지 않는 상품을 조회하면 실패 응답을 반환한다.
- 소프트 삭제되거나 숨김 처리된 상품은 사용자에게 노출하지 않는다.

### 좋아요 등록

- 로그인 사용자는 상품에 좋아요를 등록할 수 있다.
- 같은 사용자는 같은 상품에 좋아요를 하나만 가질 수 있다.
- 이미 좋아요한 상품에 다시 좋아요 등록 요청을 보내도 중복 데이터가 생기면 안 된다.
- `POST /api/v1/products/{productId}/likes`는 멱등하게 동작해야 한다.
- 좋아요 등록 시 `Like`를 원본 데이터로 생성하고, `Product.likeCount`를 증가시킨다.
- 존재하지 않거나 좋아요할 수 없는 상품에 대한 요청은 실패 응답을 반환한다.

### 좋아요 취소

- 로그인 사용자는 상품 좋아요를 취소할 수 있다.
- 이미 좋아요가 없는 상품에 좋아요 취소 요청을 보내도 시스템 상태가 깨지면 안 된다.
- `DELETE /api/v1/products/{productId}/likes`는 멱등하게 동작해야 한다.
- 좋아요 취소 시 `Like`를 제거하고, `Product.likeCount`를 감소시킨다.
- 존재하지 않는 상품에 대한 요청은 실패 응답을 반환한다.

### 좋아요한 상품 목록 조회

- 로그인 사용자는 자신이 좋아요한 상품 목록을 조회할 수 있다.
- 메인 시나리오 문서를 기준으로 `GET /api/v1/users/{userId}/likes`를 사용한다.
- 사용자는 타 사용자의 정보에 직접 접근할 수 없으므로 요청 경로의 `userId`와 로그인 사용자가 일치해야 한다.

### 주문 생성

- 로그인 사용자는 여러 상품을 한 번에 주문할 수 있다.
- 주문 요청 항목은 `productId`, `quantity`를 가진다.
- 주문 생성 시 상품 존재 여부, 판매 가능 여부, 재고 충분 여부를 검증한다.
- 주문 가능한 상품은 주문 수량만큼 재고가 차감되어야 한다.
- 재고가 부족한 상품은 주문 실패 항목으로 처리하고, 재고가 충분한 상품은 정상 주문 대상으로 처리한다.
- 주문 응답에는 정상 주문 항목과 실패 항목이 구분되어야 한다.
- 주문 정보에는 주문 당시 상품명, 가격, 브랜드명 등 상품 정보가 스냅샷으로 저장되어야 한다.
- 결제 금액은 정상 주문 대상으로 확정된 항목 기준으로 계산한다.

### 결제 흐름

- 주문 생성과 결제 승인은 별도 단계로 분리한다.
- 주문 생성 이후 주문 금액에 대한 결제 요청이 발생한다.
- 결제는 외부 결제 시스템과의 연동으로 표현한다.
- 2주차 설계에서는 실제 결제 구현보다 주문과 외부 결제 시스템의 책임 경계를 명확히 표현한다.
- 결제 성공 시 주문은 결제 완료 상태가 된다.
- 결제 실패 또는 결제 기한 만료 시 주문은 실패 또는 취소 상태로 기록하고, 주문 생성 시 확보한 재고는 보상 처리 정책에 따라 해제한다.
- 결제 완료 제한 시간은 후속 상세 정책에서 결정한다.

## 정책 및 제약

- 좋아요는 사용자와 상품의 조합 기준으로 유일해야 한다.
- 좋아요 등록과 좋아요 취소는 모두 멱등하게 동작해야 한다.
- 상품은 반드시 하나의 브랜드에 속한다.
- 주문은 하나 이상의 주문 항목을 가진다.
- 주문 항목의 수량은 1 이상이어야 한다.
- 주문 생성은 재고 확인과 재고 차감을 함께 보장해야 한다.
- 주문 항목은 상품 정보 변경의 영향을 받지 않도록 주문 시점의 상품 정보를 스냅샷으로 가진다.
- 좋아요 관계의 원본 데이터는 `Like`이며, `Product.likeCount`는 상품 목록 조회와 `likes_desc` 정렬을 위한 파생 값으로 둔다.
- 상품과 브랜드 삭제는 소프트 삭제로 처리한다.
- 결제 시스템은 내부 도메인이 아니라 외부 시스템으로 본다.

## 결정 근거 및 리스크

- 주문 생성과 결제 승인을 분리한다. 주문과 외부 결제를 한 흐름으로 묶으면 트랜잭션이 비대해지고, 결제 실패가 사용자 재고 확보 실패로 이어질 수 있다.
- 결제 실패와 재고 해제는 보상 처리 대상으로 둔다. 실패 원인이 외부 장애, 사용자 이탈, 승인 지연 중 무엇인지에 따라 비즈니스 처리가 달라질 수 있다.
- 좋아요 수는 `Product.likeCount`로 캐시한다. 상품 목록 정렬에는 유리하지만, 좋아요 등록/취소와 카운트 갱신의 정합성을 보장해야 한다.
- 상품과 브랜드는 소프트 삭제한다. 주문 이력과 감사에는 유리하지만, 사용자 조회에서 삭제 데이터를 제외하는 조건이 필요하다.
- 재고 부족 항목만 실패 처리한다. 사용자 구매 기회를 보존할 수 있지만, 주문 응답에서 성공 항목과 실패 항목을 명확히 구분해야 한다.
- 현재 인증/인가는 주요 범위가 아니지만, 타 사용자 좋아요/주문 정보 접근을 막는 식별 경계는 유지해야 한다.

# 요구사항 명세

> 본 문서는 `requirements-analysis` 스킬의 흐름(문제 재해석 → 애매한 부분 드러내기 → 정책 결정 → 요구사항 정리 → 리스크 명시)에 따라 작성되었다.

## 1. 도메인 범위

| 도메인 | 주요 기능 |
|---|---|
| 회원 (User) | 회원가입, 내 정보 조회, 비밀번호 변경 |
| 브랜드 (Brand) | 브랜드 조회 (사용자), CRUD (어드민) |
| 상품 (Product) | 상품 목록/상세 조회 (사용자), CRUD (어드민) |
| 재고 (Stock) | 상품과 1:1, 차감/복구 |
| 좋아요 (Like) | 등록/취소 (멱등), 내 좋아요 목록 |
| 주문 (Order) | 주문 생성, 주문 조회 (사용자/어드민) |
| 결제 (Payment) | 외부 PG 연동, 주문과 단일 트랜잭션 |

산출물: 요구사항 명세, 시퀀스 다이어그램, 클래스 다이어그램, ERD.

## 2. 문제 정의

요구사항을 기능 목록으로만 보지 않고, **어떤 문제를 해결하려는지** 세 관점으로 재해석한다.

| 관점 | 문제 상황 |
|---|---|
| 사용자 | 상품을 탐색하고, 관심 상품을 표시하고, 구매까지 이어지는 흐름이 끊김 없이 이어져야 한다. 같은 행동을 여러 번 했을 때(좋아요 중복 클릭 등) 사용자가 당황하지 않아야 한다. |
| 비즈니스 | 상품/브랜드 정책 변경이 과거 주문 데이터를 훼손해서는 안 된다. 재고 정합성과 결제 결과가 어긋난 채로 노출되어선 안 된다. 희소성을 보여주되 전체 재고 전략을 외부에 노출하지 않는다. |
| 시스템 | 외부 결제 시스템(PG) 호출이 실패하거나 지연될 수 있고, 동시 주문 시 재고가 음수가 되어선 안 된다. 좋아요처럼 빈번한 요청은 중복 호출에 안전(idempotent)해야 한다. |

## 3. 개념 모델

코드/기술 얘기에 들어가기 전에, 누가 어떤 도메인과 외부 시스템에 작용하는지 먼저 합의한다.

**액터**
- 일반 사용자 (인증 / 비인증 둘 다)
- 어드민
- 외부 결제 시스템 (PG)

**핵심 도메인**
- 회원 (User)
- 브랜드 (Brand)
- 상품 (Product)
- 재고 (Stock) — 상품과 1:1로 분리된 별도 애그리거트
- 좋아요 (Like)
- 주문 (Order) + 주문 항목 (OrderItem)
- 결제 (Payment)

**보조/외부 시스템**
- 외부 PG (Payment Gateway) — 동기 호출 가정

## 4. 유비쿼터스 언어

| 한글 | 영문 | 설명 |
|---|---|---|
| 브랜드 | Brand | 상품을 판매하는 브랜드 |
| 상품 | Product | 브랜드가 등록한 판매 상품 |
| 재고 | Stock | 상품의 구매 가능 수량 |
| 좋아요 | Like | 사용자가 상품에 표시하는 관심 |
| 주문 | Order | 사용자가 상품을 구매하는 행위 |
| 주문 항목 | OrderItem | 주문 내 개별 상품 + 수량 + 당시 가격 스냅샷 |
| 결제 | Payment | 외부 PG를 통한 대금 처리 |
| 재고 차감 | StockDeduct | 주문 시 재고를 줄이는 동작 |
| 논리 삭제 | Soft Delete | `deleted_at` 컬럼으로 삭제 처리 (상품은 `status = DELETED`도 동기화) |
| 멱등 | Idempotent | 같은 요청을 N번 보내도 1번 보낸 것과 동일한 최종 상태가 되는 성질 |

## 5. 명확화된 정책 결정

요구사항만으로는 모호한 항목들을 드러내고, 검토한 대안과 함께 결정 내용을 기록한다.

| No | 항목 | 결정 | 검토한 대안 / 결정 이유 |
|---|---|---|---|
| P-1 | 좋아요 등록 멱등성 | 이미 좋아요한 상태로 다시 요청해도 **200 OK** 응답 (DB 변화 없음) | 대안: 409 Conflict. 좋아요는 토글성 액션이라 사용자가 더블클릭/네트워크 재시도 시에도 같은 결과를 받는 게 자연스럽다. |
| P-2 | 좋아요 취소 멱등성 | 좋아요가 없는 상태에서 취소 요청도 **200 OK** 응답 | 대안: 404 Not Found. 멱등성 일관 유지. |
| P-3 | 삭제된 상품 좋아요 등록 | **404 Not Found** | 멱등성과 별개 — 그 상품 자체가 존재하지 않는 케이스이므로 멱등 대상이 아니다. |
| P-4 | 재고 차감 시점 | **주문 생성 즉시 차감** | 대안: 결제 성공 시 차감 → 결제 지연 동안 다른 사용자가 품절 상품을 주문할 수 있어 사기 위험. |
| P-5 | 주문-결제 일관성 | 주문 + 재고 차감 + 결제 요청을 **단일 트랜잭션**으로 처리. 결제 실패 시 주문 `CANCELLED` + 재고 복구. | 대안: SAGA / 비동기 보상 → 인프라 복잡도 과함. 학습 범위에선 동기 보상으로 충분. |
| P-6 | 외부 결제 시스템 호출 방식 | **동기 호출**. 타임아웃 시 결제 실패로 간주. 재시도 X. | 대안: 비동기 콜백 → 별도 메시징/콜백 엔드포인트 필요. 추후 확장 여지. |
| P-7 | 주문 내역 상품 정보 | `order_items`에 **스냅샷 저장** (productName, productPrice 복사) | 대안: 매번 product 테이블 조인 → 상품 변경/삭제 시 과거 주문이 왜곡됨. |
| P-8 | 브랜드/상품 삭제 방식 | **Soft delete** (`deleted_at` + 상품은 `status = DELETED`) | 주문 내역(`order_items`)은 FK 없이 스냅샷만 들고 있어 hard delete해도 주문 자체가 깨지진 않지만, ① 주문 목록에서 "상품 다시 보기" 같이 `product_id`로 재조회하는 흐름이 필요할 수 있고, ② 스냅샷에 없는 부가 정보(이미지·상세 설명·옵션 등)는 원본이 살아 있어야 조회 가능하며, ③ 어드민의 감사/실수 복원 측면에서도 원본 보존이 유리하다. 그래서 hard delete 대신 soft delete를 택한다. |
| P-9 | 좋아요 수 집계 | `likes` 테이블 **COUNT 집계** | 대안: 캐시 컬럼 → 동기화 복잡. 성능 이슈 발생 시 도입 검토. |
| P-10 | 재고 노출 정책 | 사용자: **10개 이하 시 수량 노출**, 초과 시 유무만 / 어드민: 실제 수량 | 쿠팡·무신사 방식. 희소성 신호를 주되 전체 재고 전략은 숨김. |
| P-11 | 본인 외 주문 접근 | **404 Not Found** (403 Forbidden 아님) | 주문 존재 여부 자체를 노출하지 않기 위함. |
| P-12 | 동시 주문 시 재고 충돌 | 낙관적 락 (`@Version`) 또는 비관적 락 — 구현 단계에서 결정 | 대안: 단순 쓰기 → race condition 위험. |

## 6. 유저 시나리오

전형적인 사용자 여정은 다음과 같다.

1. 사용자는 상품 **목록**을 브랜드 필터 / 정렬(최신/가격/좋아요) / 페이징 조건으로 탐색한다.
2. 마음에 드는 상품의 **상세** 화면에서 브랜드명, 가격, 재고 정보(잔여 수량 또는 유무)를 확인한다.
3. 관심 표시를 위해 **좋아요**를 누른다. 의도치 않게 두 번 눌러도 정상 응답이 와야 한다.
4. 본인이 좋아요한 상품 목록을 다시 모아보고, 그중 하나 또는 여러 상품을 골라 **주문**한다.
5. 주문 시 재고가 차감되고, **외부 결제 시스템(PG)** 으로 결제 요청이 전달된다.
6. 결제 성공 → 주문 `COMPLETED`. 결제 실패 → 주문 `CANCELLED` + 차감된 재고 복구.
7. 이후 본인의 주문 내역을 날짜 범위로 조회할 수 있다.

어드민은 별도 화면에서 브랜드/상품을 등록·수정·삭제하고, 전체 주문 현황을 모니터링한다.

## 7. 기능 요구사항

| No | 구분 | 기능명 | 요구사항 상세 설명 | 비고 |
|---|---|---|---|---|
| F-01 | 회원 | 회원가입 | 신규 사용자는 `loginId`, `password`, `name`, `birthDate`, `email`을 입력해 가입할 수 있다. 유효성 위반 시 400, `loginId` 중복 시 409. | 인증 X |
| F-02 | 회원 | 내 정보 조회 | 로그인 사용자는 본인의 `loginId`, `name`(마스킹), `birthDate`, `email`을 조회할 수 있다. `password`는 응답에 포함되지 않는다. | 인증 O |
| F-03 | 회원 | 비밀번호 변경 | 로그인 사용자는 현재 비밀번호와 새 비밀번호를 입력해 변경할 수 있다. 현재 비밀번호 불일치 시 400. | 인증 O |
| F-04 | 브랜드 | 브랜드 상세 조회 | 사용자는 특정 브랜드 ID로 브랜드 정보(이름, 설명)를 조회할 수 있다. | 인증 X |
| F-05 | 상품 | 상품 목록 조회 | 사용자는 상품 목록을 브랜드 필터, 정렬(`latest`/`price_asc`/`likes_desc`), 페이징(기본 20개)으로 조회할 수 있다. 삭제된 상품은 노출되지 않는다. | 인증 X |
| F-06 | 상품 | 상품 상세 조회 | 사용자는 상품 상세 정보(브랜드명 포함, 재고 유무, 잔여 수량)를 조회할 수 있다. 재고가 10개 이하일 때만 잔여 수량이 노출된다. | 인증 X |
| F-07 | 좋아요 | 좋아요 등록 | 로그인 사용자는 특정 상품에 좋아요를 등록할 수 있다. **이미 등록된 경우에도 동일하게 성공 응답**(멱등). 삭제된 상품에는 등록 불가(404). | 인증 O. 멱등 (P-1) |
| F-08 | 좋아요 | 좋아요 취소 | 로그인 사용자는 좋아요를 취소할 수 있다. **좋아요가 없는 상태에서도 동일하게 성공 응답**(멱등). | 인증 O. 멱등 (P-2) |
| F-09 | 좋아요 | 내 좋아요 목록 | 로그인 사용자는 본인이 좋아요한 상품 목록을 조회할 수 있다. | 인증 O |
| F-10 | 주문 | 주문 생성 | 로그인 사용자는 여러 상품(items: productId + quantity)을 한 번에 주문할 수 있다. 재고 차감 + 주문 저장이 단일 트랜잭션으로 처리되며, 재고 부족이 한 건이라도 있으면 전체 실패한다. `order_items`에 당시 상품명·가격이 스냅샷으로 저장된다. | 인증 O. 단일 트랜잭션 (P-4, P-7) |
| F-11 | 결제 | 외부 결제 요청 | 주문 생성 트랜잭션 내에서 외부 PG로 결제 요청을 동기 호출한다. 성공 시 주문 `COMPLETED`. 실패/타임아웃 시 주문 `CANCELLED` + 재고 복구. | 외부 연동 (P-5, P-6) |
| F-12 | 주문 | 주문 목록 조회 | 로그인 사용자는 본인 주문 목록을 날짜 범위로 조회할 수 있다. 기본 범위는 최근 30일. | 인증 O |
| F-13 | 주문 | 주문 상세 조회 | 로그인 사용자는 본인의 단일 주문 상세를 조회할 수 있다. 타인 주문 접근 시 404로 응답한다. | 인증 O (P-11) |
| F-14 | 어드민 | 브랜드 CRUD | 어드민은 브랜드를 등록/수정/삭제/조회(목록·상세)할 수 있다. 삭제 시 소속 상품도 함께 soft delete 처리된다(동일 트랜잭션). | 어드민 인증 (P-8) |
| F-15 | 어드민 | 상품 CRUD | 어드민은 상품을 등록/수정/삭제/조회할 수 있다. 등록 시 초기 재고도 함께 설정된다. 수정 시 브랜드는 변경 불가. 어드민 화면에선 재고 실수량이 노출된다. | 어드민 인증 (P-10) |
| F-16 | 어드민 | 주문 모니터링 | 어드민은 전체 사용자의 주문 목록(페이징) 및 단일 주문 상세를 조회할 수 있다. | 어드민 인증 |

## 8. 응답 데이터 (주요 화면)

각 화면 응답에 어떤 필드가 포함되는지 명시한다. (DTO 단위로 상세 필드는 별도 클래스 다이어그램 참조)

| 화면 | 주요 응답 항목 |
|---|---|
| 회원가입 / 내 정보 | `loginId`, `name`(마지막 글자 `*` 마스킹), `birthDate`(yyyyMMdd), `email` |
| 브랜드 상세 | `id`, `name`, `description` |
| 상품 목록 (사용자) | `id`, `brandId`, `name`, `description`, `price`, `inStock`(boolean), `remainingStock`(10개 이하일 때만) |
| 상품 상세 (사용자) | 목록 항목 + `brandName` |
| 상품 목록/상세 (어드민) | `id`, `brandId`, `name`, `description`, `price`, `stock`(실수량) |
| 주문 상세/목록 | `id`, `status`, `totalPrice`, `orderedAt`, `items[{id, productId, productName, productPrice, quantity, subtotal}]` |
| 어드민 주문 | 위 + `userId` |

## 9. 잠재 리스크 및 선택지

설계 결정 P-5(주문+결제 단일 트랜잭션)에서 발생할 수 있는 위험을 숨기지 않고 적어둔다.

| 리스크 | 영향 | 대응 선택지 |
|---|---|---|
| 외부 PG 호출이 트랜잭션 내부에 있으면 DB 커넥션이 외부 호출 시간만큼 점유된다 | 동시 처리량 저하, 커넥션 풀 고갈 가능 | (A) 현재처럼 단일 트랜잭션 유지 (학습 단순화)<br>(B) 결제만 별도 트랜잭션으로 분리 + 보상 로직<br>(C) 결제 비동기화 (SAGA) |
| 결제 응답이 "성공도 실패도 아닌" 모호한 상태(타임아웃 등) | 재고 차감은 됐는데 결제 미확정 | (A) 일단 실패로 간주하고 재고 복구 (현재 결정)<br>(B) 별도 결제 상태 조회/재확인 잡 |
| 좋아요 동시 클릭으로 동시 INSERT 시 UK 위반 예외 | 사용자 입장에선 멱등인데 내부에선 예외 발생 | UK 위반을 잡아서 정상 응답으로 변환 (이미 좋아요한 상태로 간주) |
| 동시 주문 시 재고 race condition | 재고 음수 가능 | (A) `@Version` 낙관 락 (재시도 필요)<br>(B) `SELECT ... FOR UPDATE` 비관 락 |
| 상품 soft delete 시 검색 쿼리에 `deleted_at IS NULL` 누락 가능 | 삭제된 상품이 노출됨 | Repository 메서드명에 명시 + 통합 테스트로 검증 |
| `order_items` 스냅샷은 보존되지만 `productId` FK가 깨져 보일 수 있음 | 조인 쿼리에서 누락 | FK 설정하지 않고 Long ID로만 보유 (이미 결정) |

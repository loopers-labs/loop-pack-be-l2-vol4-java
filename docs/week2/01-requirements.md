# Requirements

이 문서는 도메인별 요구사항과 주요 정책 결정을 누적한다. HTTP 상태 코드나 구현 세부사항보다 사용자가 수행해야 하는 행동, 도메인이 지켜야 하는 규칙, 현재 범위에서 제외한 결정을 중심으로 기록한다.

## 공통 정책

- 삭제 기능이 있는 리소스는 soft delete를 기본으로 한다.
- 모든 엔티티는 `BaseEntity`를 상속하며 `created_at`, `updated_at`, `deleted_at`을 공유한다. 기본 키는 `BIGINT IDENTITY`를 사용한다.
- 금액은 KRW 원 단위 정수(`BIGINT`)로 저장한다.
- 관리자 전용 기능은 사용자 기능과 별도 엔드포인트(`/api/v1/admin/...`)로 제공하고, application service도 `XxxAdminService`로 분리한다. 도메인 모델과 Repository(port)는 사용자/관리자가 공유하며 중복하지 않는다.
- 관리자 인증은 LDAP 등 별도 인증 시스템에서 처리된 것으로 가정하며, 사용자 인증(`X-Loopers-LoginId`/`X-Loopers-LoginPw`)과는 별도 헤더로 구분한다.

## Users

### 요구사항

#### 사용자

1. 사용자는 회원가입을 할 수 있다.
2. 사용자는 내 정보 조회를 할 수 있다.
3. 사용자는 비밀번호를 변경할 수 있다.

#### 관리자

1. 관리자는 별도 인증 시스템(LDAP)을 통해 접근하는 주체이므로, 일반 사용자 도메인에 관리자 권한을 별도로 두지 않는다.

### 제공 기능

| 기능 | 인증 | 설명 |
| --- | --- | --- |
| 회원가입 | 불필요 | 신규 사용자를 등록한다. |
| 내 정보 조회 | USER 필요 | 인증된 사용자의 정보를 조회한다. |
| 비밀번호 변경 | USER 필요 | 인증된 사용자의 비밀번호를 변경한다. |

### 도메인 규칙

- `loginId`는 사용자를 식별하는 값이며 중복될 수 없다.
- `email`은 중복될 수 없다.
- 비밀번호에는 사용자의 생년월일이 포함될 수 없다.
- 비밀번호 변경 시 현재 비밀번호 확인이 필요하다.
- 새 비밀번호는 현재 비밀번호와 달라야 한다.
- 내 정보 조회 시 응답에서는 이름 마지막 글자를 마스킹해서 반환한다.

### 인증과 권한

- 일반 사용자 인증은 헤더 기반 인증(`X-Loopers-LoginId`, `X-Loopers-LoginPw`)으로 처리한다.
- 일반 사용자 인증에 성공하면 `userId`를 인증 주체로 사용한다.
- 일반 사용자와 관리자는 Security 권한으로 구분한다.
- 관리자 인증은 LDAP 등 별도 인증 시스템에서 처리된 것으로 가정한다.
- 관리자 권한은 `User` 도메인의 속성이나 `users` 테이블의 컬럼으로 관리하지 않는다.

### 설계 메모

- `User` 도메인은 일반 회원만 저장한다.
- 비밀번호 검증에는 인코딩 검증 같은 애플리케이션 관심사가 포함되므로, 순수 도메인 객체와 분리해서 다룬다.
- 비밀번호 정책처럼 단일 규칙은 `UserPasswordPolicy` 같은 도메인 정책 클래스로 분리한다.
- 반면 가입/비밀번호 변경 흐름 전체를 묶는 별도 validator 객체는 현재 기능 규모에서는 두지 않고, 유스케이스 흐름 안에서 검증한다.
- 검증 로직이 재사용되거나 복잡해지면 별도 객체로 분리할 수 있다.

## Brands

### 요구사항

#### 사용자

1. 사용자는 브랜드 정보 조회를 할 수 있어야 한다.

#### 관리자

1. 관리자는 브랜드 목록과 상세 정보를 조회할 수 있어야 한다.
2. 관리자는 브랜드 정보를 등록, 수정, 삭제할 수 있어야 한다.

#### 공통

1. 브랜드가 삭제될 때 해당 브랜드에 속한 상품도 삭제되어야 한다.

### 제공 기능

| 기능 | 인증 | 설명 |
| --- | --- | --- |
| 브랜드 조회 | 불필요 | 브랜드 정보를 조회한다. |
| 브랜드 관리 조회 | ADMIN 필요 | 등록된 브랜드 목록과 상세 정보를 조회한다. |
| 브랜드 관리 | ADMIN 필요 | 브랜드를 등록, 수정, 삭제한다. |

### 도메인 규칙

- 브랜드명(`name`)은 중복될 수 없다.
- 브랜드 등록 시 브랜드명은 필수이며, 설명(`description`)과 로고 이미지(`logoUrl`)를 함께 가질 수 있다.
- 브랜드 수정에서는 브랜드명, 설명, 로고 이미지를 변경할 수 있다.
- 브랜드를 삭제하면 해당 브랜드에 속한 상품도 함께 삭제한다. 같은 트랜잭션에서 brandId에 속한 상품을 모두 soft delete한다.
- soft delete는 `deleted_at`을 채우는 UPDATE이므로, 삭제 전에 대상 브랜드의 존재를 먼저 확인한다.
- 삭제된 브랜드와 상품의 복구 정책은 현재 범위에서 다루지 않는다.
- 고객과 관리자에게 제공하는 브랜드 정보는 현재 범위에서는 동일하게 본다.

### 인증과 권한

- 브랜드 조회는 사용자 인증 없이 가능하다.
- 브랜드 관리 조회, 등록, 수정, 삭제는 관리자 권한이 필요하다.

### 설계 메모

- `Brand`는 별도 Aggregate로 본다.
- `Product`는 `Brand`와 다른 Aggregate로 보고, 브랜드 식별자를 통해 브랜드에 소속된다.
- 브랜드 삭제 시 하위 상품 삭제는 `Brand` 내부 불변식이 아니라 여러 Aggregate를 조율하는 유스케이스 정책으로 본다.
- 브랜드 조회(사용자)와 브랜드 관리(관리자, 등록/수정/삭제)는 액터와 관심사가 다르므로, 조회는 `BrandService`, 관리 기능은 `BrandAdminService`로 분리한다. 두 서비스는 `Brand` 도메인 모델과 `BrandRepository`를 공유한다.

## Products

### 요구사항

#### 사용자

1. 사용자는 상품 목록과 상세 정보를 조회할 수 있다.

#### 관리자

1. 관리자는 상품 목록과 상세 정보를 조회할 수 있다.
2. 관리자는 상품을 등록, 수정, 삭제할 수 있다.

#### 공통

1. 상품은 특정 브랜드에 속한다.
2. 주문 시 상품 재고가 확인되고 차감되어야 한다.

### 제공 기능

| 기능 | 인증 | 설명 |
| --- | --- | --- |
| 상품 목록 조회 | 불필요 | 상품 목록을 조회하고 필요한 조건으로 탐색한다. |
| 상품 상세 조회 | 불필요 | 단일 상품 정보를 조회한다. |
| 상품 관리 조회 | ADMIN 필요 | 등록된 상품 목록과 상세 정보를 조회한다. |
| 상품 관리 | ADMIN 필요 | 상품을 등록, 수정, 삭제한다. |

### 도메인 규칙

- 상품은 하나의 브랜드에 속한다.
- 상품을 등록할 때 연결할 브랜드는 이미 존재해야 한다.
- 상품의 브랜드는 상품 수정 과정에서 변경하지 않는다.
- 상품은 판매 상태(`status`)를 가진다: `ON_SALE`(판매중), `SUSPENDED`(판매중지). 관리자가 `ON_SALE` ↔ `SUSPENDED`로 전환한다.
- 품절은 별도 상태로 저장하지 않고 재고(`ProductStock.quantity == 0`)로 판단한다. 재고 차감/복구는 `status`를 바꾸지 않아, 재고가 품절의 SSOT다.
- 품절은 주문 시 재고 차감(락)에서 자연히 강제되므로 상시 재고 조회가 필요 없고, 목록·상세의 품절 표시가 필요할 때만 그 시점에 재고로 계산한다.
- `SUSPENDED` 상품은 주문할 수 없고, 사용자 상품 목록·상세 조회에서 제외한다(삭제된 상품과 동일하게 취급). 관리자 조회에는 포함한다.
- 상품은 대표 이미지(`thumbnailUrl`) 1장을 가진다. 다중 이미지 갤러리는 현재 범위에서 다루지 않으며, 필요해지면 별도 `ProductImage` 테이블로 확장한다. 이미지는 URL 경로만 저장하고 파일 업로드 처리는 다루지 않는다.
- 상품 재고는 `ProductStock`으로 관리한다.
- `ProductStock.decrease(qty)`는 잔여 재고가 부족하면 예외를 발생시켜 음수 진입을 막는다.
- 관리자 상품 조회에는 재고 정보를 포함한다.

### 인증과 권한

- 상품 조회는 인증 없이 가능하다.
- 상품 관리 조회, 등록, 수정, 삭제는 관리자 권한이 필요하다.

### 설계 메모

- `Product`는 `Brand` 객체를 소유하지 않고 `brandId`로 브랜드를 참조한다.
- `ProductStock`은 Product Aggregate 안의 구성요소로 본다.
- 재고는 상품 생명주기에 종속되지만, 변경 빈도와 동시성 특성이 상품 기본 정보와 달라 별도 테이블로 관리한다.
- 주문 도메인은 주문 시점의 상품 정보를 스냅샷으로 저장해야 하므로, 상품이 삭제되어도 기존 주문 이력이 훼손되지 않아야 한다.
- 상품 목록 정렬 옵션은 도메인 필드와 다음과 같이 매핑한다.

  | 정렬 옵션 | 도메인 필드 |
  | --- | --- |
  | `latest` | `Product.createdAt DESC` |
  | `price_asc` | `Product.price ASC` |
  | `likes_desc` | `Product.likeCount DESC` (집계 컬럼) |

- 좋아요 수 정렬은 `Product.likeCount` 집계 컬럼으로 처리해 매 조회마다 전체 집계하지 않는다. `likeCount` 갱신 방식은 Likes 설계 메모 참고.

## Likes

### 요구사항

#### 사용자

1. 사용자는 상품에 좋아요를 등록할 수 있다.
2. 사용자는 상품 좋아요를 취소할 수 있다.
3. 사용자는 자신이 좋아요한 상품 목록을 조회할 수 있다.

#### 공통

1. 좋아요는 상품 탐색의 기준으로 사용될 수 있다.

### 제공 기능

| 기능 | 인증 | 설명 |
| --- | --- | --- |
| 좋아요 등록 | USER 필요 | 상품에 좋아요를 등록한다. |
| 좋아요 취소 | USER 필요 | 상품 좋아요를 취소한다. |
| 좋아요 목록 조회 | USER 필요 | 사용자가 좋아요한 상품 목록을 조회한다. |

### 도메인 규칙

- `Like`는 사용자와 상품 사이의 선호 관계를 표현한다.
- 현재 범위에서 `Like`는 상품 좋아요를 의미한다.
- 같은 사용자와 상품 조합은 한 row로 관리한다. 취소는 `deleted_at`을 채워 비활성화하고, 재등록은 `deleted_at`을 비워 복구한다.
- 좋아요 등록과 취소는 멱등하게 처리한다. 활성 상태에서 다시 등록 요청이 와도, 이미 취소된 상태에서 다시 취소 요청이 와도 같은 상태를 유지한다.
- 삭제된 상품과 판매중지(`SUSPENDED`) 상품에는 좋아요를 등록할 수 없다.
- 좋아요 목록에서는 삭제된 상품을 제외한다. 판매중지(`SUSPENDED`) 상품은 목록에 남기되 '판매중지' 상태로 표시한다(사용자가 찜한 항목을 보존).

### 인증과 권한

- 좋아요 등록, 취소, 목록 조회는 일반 사용자 인증이 필요하다.
- 인증된 `userId`를 기준으로 좋아요를 등록하거나 조회한다.

### 설계 메모

- `Like`는 `User`나 `Product` 내부 컬렉션이 아니라 별도 Aggregate로 본다.
- `Like`는 `userId`, `productId`로 `User`, `Product`를 참조한다.
- 취소된 좋아요를 다시 등록하면 기존 좋아요를 복구한다.
- 좋아요 등록과 취소는 별도 엔드포인트로 분리해 동작 자체가 멱등하게 보장된다. 응답은 두 경우 모두 `200 OK`로 통일하고 현재 좋아요 상태를 본문에 담는다.
- 상품별 좋아요 수는 `Product.likeCount` 집계 컬럼에 비정규화한다(`likes`가 SSOT). 갱신 규칙:
  - 좋아요 등록/취소 커밋 후 `@TransactionalEventListener(AFTER_COMMIT)` 이벤트로 해당 상품을 recount해 `likeCount`를 갱신한다. `LikeService`가 아닌 projection 핸들러가 처리해 애그리거트 경계를 유지한다.
  - recount는 멱등이라 이벤트 중복/유실에 강건하고, 배치(`commerce-batch`) 전체 재계산을 안전망으로 둔다.
  - 최종 일관성이라 정렬·표시가 순간적으로 lag날 수 있다.
- 좋아요 변경 이력 전체가 필요해지면 별도 history 모델을 검토한다.
- 상품 외 다른 대상에 좋아요가 생기면 `Like` 이름과 모델 범위를 다시 검토한다.

## Orders

### 요구사항

#### 사용자

1. 사용자는 상품을 주문할 수 있다.
2. 사용자는 본인의 주문 목록을 조회할 수 있다.

#### 관리자

1. 관리자는 전체 주문을 조회할 수 있다.

#### 공통

1. 주문에는 주문 당시의 상품 정보가 저장되어야 한다.
2. 주문 시 상품 재고가 확인되고 차감되어야 한다.

### 제공 기능

| 기능 | 인증 | 설명 |
| --- | --- | --- |
| 주문 요청 | USER 필요 | 상품을 주문한다. |
| 주문 목록 조회 | USER 필요 | 사용자의 주문 목록을 조회한다. |
| 주문 관리 조회 | ADMIN 필요 | 전체 주문을 조회한다. |

### 도메인 규칙

- `Order`는 별도 Aggregate로 본다.
- `OrderItem`은 Order Aggregate 안의 구성요소로 본다.
- 주문은 사람이 식별할 수 있는 주문번호(`orderNumber`)를 가진다. 주문 생성 시 1회 생성되고 변경되지 않으며, 중복되지 않도록 `UNIQUE` 제약을 둔다. PK(`id`)와 별개의 식별자다.
- 주문번호는 `yyyyMMdd + 일별 시퀀스`(예: `20260528-000123`) 형식으로 생성한다. 일별 채번 카운터는 동시 주문에서 충돌하지 않도록 원자적으로 처리한다(채번 테이블 또는 시퀀스).
- 주문 생성이 성공하면 상품 재고가 확보된 것으로 본다.
- 여러 상품 중 하나라도 주문할 수 없으면 전체 주문을 생성하지 않는다.
- 삭제된 상품과 판매중지(`SUSPENDED`) 상품은 주문할 수 없다. 품절(재고 부족) 상품은 재고 차감 단계에서 자연히 막힌다.
- 주문에는 주문 당시 상품 정보와 가격을 스냅샷으로 저장한다.
- 주문 스냅샷에는 상품 식별자·상품명·가격뿐 아니라 브랜드 식별자와 브랜드명도 포함한다. 브랜드가 변경되거나 삭제된 이후에도 주문 이력에서 당시 브랜드를 식별할 수 있어야 하기 때문이다.
- 주문 총액(`totalAmount`)은 주문 생성 시 `OrderItem`의 합으로 산정해 `Order`에 함께 저장한다.
- 주문에는 배송지·수령인 정보(수령인명, 연락처, 우편번호, 주소)를 주문 시점 정보로 저장한다.
- 결제수단(`paymentMethod`)은 `CARD` 또는 `CASH`로 저장한다. 현재 범위에서는 결제 처리 로직 없이 사용자가 선택한 값만 기록하며, 실제 결제 전이·검증은 결제 도메인이 추가될 때 다룬다.
- 주문 생성 시 초기 상태는 `PENDING`이다. 현재 범위에서는 결제 도메인이 없으므로 주문 생성 직후에도 `PENDING` 상태로 종료된다.
- 결제 도메인이 추가되면 결제 성공 시 `PAID`, 결제 실패 시 `FAILED`로 전이한다.
- 주문 취소 기능은 현재 요구사항에 없으므로 `CANCELED` 상태는 두지 않는다.

### 인증과 권한

- 일반 사용자는 본인의 주문만 조회할 수 있다.
- 관리자는 전체 주문을 조회할 수 있다.
- 주문 목록 조회의 기간 조건은 선택값으로 본다.

### 설계 메모

- 주문 생성과 재고 차감은 현재 범위에서는 하나의 트랜잭션으로 처리한다.
- 주문은 `Product`와 `Brand`를 직접 소유하지 않고, 주문 당시 필요한 정보를 스냅샷으로 저장한다.
- 배송지·수령인 정보는 `ShippingDestination` 값 객체(VO)로 묶어 `Order`가 보유한다. DB에는 `@Embedded`로 ORDERS 테이블 컬럼에 평면 저장하며, 주문 시점 스냅샷이라 회원 주소 변경과 무관하게 보존된다. 분할 배송(1:N)이나 배송 라이프사이클(상태·송장번호)이 생기면 별도 `ORDER_SHIPPING` 테이블로 승격을 검토한다.
- 한 주문에 여러 상품이 포함될 때, 동시 주문에서 발생할 수 있는 deadlock을 피하기 위해 재고 락은 `productId` 오름차순으로 잡는다.
- 결제 통합 지점은 `PaymentService.pay()` 인터페이스로 둔다. 목표 흐름(결제 도메인 추가 시 구현 예정)은 트랜잭션을 분리한다:
  1. **트랜잭션 1**: 재고 차감 + 주문 `PENDING` 저장 후 커밋. (외부 결제 호출 동안 재고 락을 잡지 않기 위해 여기서 끊는다.)
  2. `PaymentService.pay()` 호출 — 외부 결제 시스템, DB 트랜잭션 바깥.
  3. **트랜잭션 2/3**: 결제 결과에 따라 별도 트랜잭션에서 전이. 성공 → `PAID`. 실패 → 보상(재고 복구 + `FAILED` 전이). 트랜잭션 1이 이미 커밋됐으므로 DB 롤백이 아니라 명시적 보상이다.
- **현재 범위**: `PaymentService.pay()`는 stub(껍데기)으로만 두고 위 2·3은 구현하지 않는다. 주문은 `PENDING`으로 종료된다. 위 목표 흐름과 아래 TODO는 stub의 주석/TODO로 남겨, 결제 도메인 추가 시 채운다.

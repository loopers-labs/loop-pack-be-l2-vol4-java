# 01. 요구사항 명세서

## 0. 이 문서의 역할

Loopers의 무엇·누구·왜를 **기획·디자인·개발이 공통으로 쓰는 어휘와 흐름**으로 정의한다. 도메인 개념, 사용자 시나리오, 비즈니스 규칙만 다룬다.

이 문서는 후속 설계 문서의 출발점이다:
- 시스템 내부의 행동 흐름 → [`02-sequence-diagrams.md`](./02-sequence-diagrams.md)
- 도메인 객체와 책임 → [`03-class-diagram.md`](./03-class-diagram.md)
- 데이터 모델 → [`04-erd.md`](./04-erd.md)

API endpoint·요청/응답 스키마·에러 코드·구현 전략(쿼리·락·캐시 등)은 본 문서의 관심사가 아니다.

## 1. 개요

Loopers는 여러 브랜드의 상품을 둘러보고, 마음에 드는 상품에 좋아요를 누르며, 여러 상품을 한 번에 주문·결제할 수 있는 감성 이커머스 서비스다.

- 회원(User)은 가입 후 비밀번호를 직접 관리하며, 상품을 탐색·좋아요·주문한다.
- 비회원(Guest)은 카탈로그(브랜드·상품)만 둘러볼 수 있다.
- 운영자(Admin)는 브랜드·상품 카탈로그를 관리하고, 전체 주문을 모니터링한다.
- 결제는 외부 결제 시스템(Payment Gateway)에 위임한다.

사용자의 모든 행동(상품 조회, 좋아요, 주문 등)은 활동 기록으로 남아 추후 랭킹·추천 같은 데이터 기반 기능의 토대가 된다.

## 2. 등장 인물

| 액터 | 정체성 | 할 수 있는 것 | 할 수 없는 것 |
| --- | --- | --- | --- |
| **User** | 가입한 회원 | 카탈로그 둘러보기, 좋아요, 주문·결제, 내 주문 조회, 내 비밀번호 변경 | 다른 User의 자원에 접근 |
| **Guest** | 비회원 방문자 | 카탈로그 둘러보기 (브랜드·상품 목록/상세) | 좋아요·주문·내 정보 관련 모든 기능 |
| **Admin** | 운영자 | 브랜드·상품 등록/수정/삭제, 전체 주문 모니터링 | 대고객 기능 호출 |
| **Payment Gateway** | 외부 결제 시스템 | 주문 결제 처리·결과 통보 | 도메인 내부 상태 변경 |

**격리 원칙** — User는 다른 User의 좋아요·주문·비밀번호에 접근할 수 없다. 본인 외 자원에 접근을 시도하면 "찾을 수 없음" 안내로만 응대한다 (자원의 존재 여부 노출 방지).

## 3. 도메인 어휘

본 절은 명세 본문에서 사용하는 핵심 개념을 정의한다. 데이터 모델 상세는 [`04-erd.md`](./04-erd.md), 객체 설계는 [`03-class-diagram.md`](./03-class-diagram.md).

### 3.1 핵심 개념

- **브랜드(Brand)** — 상품을 묶는 운영 단위. 운영자가 등록·삭제한다.
- **상품(Product)** — 판매 대상. 브랜드에 속하며 가격·재고·이미지·설명을 가진다.
- **좋아요(Like)** — User가 상품에 대해 "마음에 든다"고 표시한 행위. 같은 사용자가 같은 상품에 두 번 누르는 일은 없다.
- **주문(Order)** — User가 한 번의 결제로 묶어서 사겠다고 선언한 단위. 결제 상태(대기·완료·실패)를 가진다.
- **주문 항목(Order Item)** — 주문에 포함된 개별 상품 + 수량. 주문 시점의 상품 이름·단가·브랜드명·이미지를 함께 보존한다 (사후의 상품 변경에 흔들리지 않는다).
- **결제 수단(Payment Method)** — 주문 시 사용자가 선택. 현 단계에서는 카드 결제만 지원.

### 3.2 관계

- 한 브랜드는 여러 상품을 가진다.
- 한 사용자는 여러 상품에 좋아요를 누를 수 있다. 한 상품도 여러 사용자에게서 좋아요를 받을 수 있다. 단 같은 (사용자, 상품) 쌍은 하나뿐이다.
- 한 사용자는 여러 주문을 한다. 한 주문은 여러 주문 항목을 가진다.
- 주문 항목은 상품을 가리키되, 상품의 이름·가격·이미지를 **주문 시점 값으로 별도 보존**한다.

### 3.3 활성 / 비활성

브랜드·상품·좋아요는 "활성"과 "비활성" 상태를 가진다. 비활성 자원은 대고객 화면에서 보이지 않는다. 운영자는 비활성 자원도 조회할 수 있다. 비활성된 자원이라 해도 이미 발생한 주문 내역에는 영향을 주지 않는다 (주문 항목의 스냅샷 덕분).

### 3.4 좋아요 수

각 상품은 "좋아요 수"를 가진다. 사용자가 누른 좋아요 변화는 좋아요 수에 즉시 반영된다. 좋아요 수는 상품 목록에서 "인기순" 정렬의 기준이 된다.

### 3.5 주문 상태

주문은 다음 셋 중 하나의 상태를 가진다:
- **결제 대기 (PENDING)** — 주문은 생성되었으나 외부 결제가 아직 확정되지 않은 상태
- **결제 완료 (PAID)** — 외부 결제까지 성공해 거래가 확정된 상태
- **결제 실패 (FAILED)** — 결제 시도가 실패해 거래가 무효가 된 상태

상태는 한 방향으로만 전이한다: 결제 대기 → 결제 완료, 또는 결제 대기 → 결제 실패. 결제 완료된 주문은 실패로 되돌리지 않으며, 그 반대도 마찬가지다.

## 4. 유저 스토리

### 4.1 User (회원)

- 사용자는 본인의 비밀번호를 변경할 수 있다.
- 사용자는 브랜드 정보를 조회할 수 있다.
- 사용자는 상품 목록을 브랜드·정렬·페이지 조건으로 둘러볼 수 있다.
- 사용자는 상품 목록을 최신순·가격 오름차순·가격 내림차순·좋아요 많은 순으로 정렬할 수 있다.
- 사용자는 특정 상품의 상세 정보(사진·설명·가격·재고 여부)를 볼 수 있다.
- 사용자는 상품에 좋아요를 누를 수 있다.
- 사용자는 좋아요한 상품의 좋아요를 취소할 수 있다.
- 사용자는 본인이 좋아요한 상품 목록을 조회할 수 있다.
- 사용자는 여러 상품을 한 번에 주문·결제할 수 있다.
- 사용자는 본인의 주문 목록과 상세를 언제든 조회할 수 있다.
- 사용자는 주문 후 상품의 가격이나 노출 상태가 변하더라도 주문 내역에서 주문 시점의 정보를 그대로 확인할 수 있다.

### 4.2 Guest (비회원)

- 비회원은 브랜드 정보를 조회할 수 있다.
- 비회원은 상품 목록과 상세를 조회할 수 있다.
- 비회원이 좋아요·주문·내 정보 기능을 시도하면 "로그인이 필요합니다" 안내를 받는다.

### 4.3 Admin (운영자)

- 운영자는 브랜드를 등록·수정·조회·삭제할 수 있다.
- 운영자는 상품을 등록·수정·조회·삭제할 수 있다.
- 운영자는 브랜드를 삭제하면 소속 상품과 그 상품의 좋아요가 함께 비활성된다. 단, 이미 발생한 주문 내역은 그대로 보존된다.
- 운영자는 본인 격리 규칙과 무관하게 전체 사용자의 주문 내역을 조회할 수 있다.
- 운영자가 카탈로그·주문에 가한 액션은 모두 활동 기록으로 남는다.

### 4.4 시스템 — 결제 처리

- 시스템은 주문 생성 시 재고를 차감한다.
- 시스템은 외부 결제 시스템(Payment Gateway)에 결제를 위임한다.
- 시스템은 결제 성공 시 주문을 결제 완료로 확정한다.
- 시스템은 결제 실패 시 차감한 재고를 원복하고 주문을 실패로 마감한다.
- 시스템은 결제 응답이 지연되거나 누락될 경우 주문을 결제 대기로 두고 추후 결제 시스템에 재확인하여 최종 상태를 확정한다.

## 6. 유스케이스

> 각 유스케이스 구조:
> - **액터** / **사전 조건** / **트리거**
> - **정상 흐름** — 가장 흔한 진행
> - **분기** — 정상 범위 안의 변형
> - **실패 흐름** — 사용자가 경험할 수 있는 실패
> - **사후 조건** — 흐름 종료 시 무엇이 바뀌어 있나
> - **규칙·정책** — 본 흐름에 적용되는 비즈니스 룰 (횡단 규칙은 §7)

### UC-01. 내 비밀번호 변경

- **액터**: User
- **사전 조건**: User가 본인으로 식별되어 있다
- **트리거**: 사용자가 새 비밀번호로 교체하고자 한다

**정상 흐름**
1. 사용자가 새 비밀번호를 제출한다
2. 시스템은 새 비밀번호가 비밀번호 정책(§7.1)을 만족하는지 확인한다
3. 시스템은 새 비밀번호가 현재 비밀번호와 다른지 확인한다
4. 시스템은 새 비밀번호를 안전하게 저장한다
5. 시스템은 변경 사건을 활동 기록에 남긴다

**실패 흐름**
- 새 비밀번호가 정책을 위반함 → 사용자에게 위반 사유 안내, 변경 미적용
- 새 비밀번호가 현재 비밀번호와 동일함 → "현재와 동일합니다" 안내
- 본인 식별 실패 → "로그인이 필요합니다" 안내

**사후 조건**
- 사용자의 비밀번호가 새 값으로 갱신됨. 이후 인증은 새 비밀번호로만 가능 (이전 비밀번호는 즉시 무효)
- 활동 기록에 "비밀번호 변경" 사건 1건

**규칙·정책**
- 비밀번호 정책 상세는 §7.1
- 평문 비밀번호는 어디에도 남기지 않는다

---

### UC-02. 브랜드 조회

- **액터**: User, Guest
- **사전 조건**: 없음
- **트리거**: 사용자가 특정 브랜드 정보를 보고자 한다

**정상 흐름**
1. 사용자가 브랜드를 지정해 조회를 요청한다
2. 시스템은 활성 브랜드 정보를 보여준다

**실패 흐름**
- 브랜드가 존재하지 않거나 비활성 상태 → "찾을 수 없습니다" 안내

**사후 조건**
- 변경 없음 (조회)

---

### UC-03. 상품 목록 둘러보기

- **액터**: User, Guest
- **사전 조건**: 없음
- **트리거**: 사용자가 조건(브랜드·정렬·페이지)에 맞춰 상품 목록을 둘러보고자 한다

**정상 흐름**
1. 사용자가 브랜드 필터(선택), 정렬 기준, 페이지를 지정한다
2. 시스템은 활성 상품만 모아 조건에 따라 정렬·페이지로 잘라 보여준다
3. (User로 식별된 경우) 각 상품에 대해 "내가 좋아요했는지" 표시도 함께 보여준다

**정렬 기준**
- 최신순 (기본)
- 가격 오름차순 / 가격 내림차순
- 좋아요 많은 순

**분기**
- 브랜드 필터 미지정 → 전체 상품 대상
- 결과가 0건 → 빈 페이지 안내

**실패 흐름**
- 페이지·정렬 기준이 잘못된 값 → "잘못된 요청" 안내

**사후 조건**
- 변경 없음 (조회)

**규칙·정책**
- 비활성 브랜드에 속한 상품은 목록에서 제외
- 비활성 상품은 목록에서 제외
- 좋아요 표시는 Guest에게 항상 미표시
- 페이지 1회 호출당 상품 수 상한은 §7.2

---

### UC-04. 상품 상세 조회

- **액터**: User, Guest
- **사전 조건**: 없음
- **트리거**: 사용자가 특정 상품의 상세 정보를 보고자 한다

**정상 흐름**
1. 사용자가 상품을 지정해 조회를 요청한다
2. 시스템은 활성 상품의 상세 정보(사진·설명·가격·재고 여부·좋아요 수)를 보여준다
3. (User로 식별된 경우) "내가 좋아요했는지" 표시도 함께 보여준다

**실패 흐름**
- 상품이 존재하지 않거나, 상품·소속 브랜드가 비활성 상태 → "찾을 수 없습니다" 안내

**사후 조건**
- 변경 없음 (조회)

**규칙·정책**
- 재고 수치는 직접 노출하지 않고 "재고 있음/없음"만 보여준다

---

### UC-05. 상품 좋아요 누르기

- **액터**: User
- **사전 조건**: User로 식별되어 있다
- **트리거**: 사용자가 상품에 좋아요를 누른다

**정상 흐름**
1. 사용자가 좋아요 대상 상품을 지정한다
2. 시스템은 상품이 활성 상태인지 확인한다
3. 시스템은 (사용자, 상품)에 대한 좋아요를 활성 상태로 만든다 (최초 또는 취소 후 재등록)
4. 시스템은 상품의 좋아요 수를 1 증가시킨다 (실제로 상태가 변했을 때만)
5. 시스템은 좋아요 사건을 활동 기록에 남긴다

**분기**
- 이미 좋아요 상태인 경우 → 변화 없음. 사용자에게는 동일한 성공 응답으로 보임

**실패 흐름**
- 본인 식별 실패 → "로그인이 필요합니다"
- 상품이 존재하지 않거나 비활성 → "찾을 수 없습니다"

**사후 조건**
- (사용자, 상품) 좋아요가 활성 상태
- 실제 상태 변화가 있던 경우 좋아요 수 +1, 활동 기록 1건

**규칙·정책**
- 같은 (사용자, 상품)에 대해 반복 호출해도 최종 결과는 같다 (이중 카운트 없음)
- 좋아요 수 반영은 §7.3

### UC-08. 상품 주문·결제

- **액터**: User, Payment Gateway
- **사전 조건**: User로 식별되어 있다
- **트리거**: 사용자가 여러 상품을 한 번에 주문·결제하고자 한다

**정상 흐름**
1. 사용자가 구매할 상품 목록(상품 + 수량)과 결제 수단을 제출한다
2. 시스템은 입력을 확인한다 (상품 1개 이상, 수량 1 이상)
3. 시스템은 모든 상품이 활성 상태이며 요청 수량만큼 재고가 남았는지 확인한다
4. 시스템은 요청 항목별 재고를 차감하고, 주문을 **결제 대기** 상태로 생성한다 (이때 상품명·단가·브랜드명·이미지를 주문 항목에 함께 보존한다)
5. 시스템은 외부 결제 시스템에 결제를 위임한다
6. 결제 결과에 따라:
   - **성공** → 주문을 **결제 완료**로 확정한다. 활동 기록에 "주문 완료" 사건을 남긴다
   - **실패** → 차감한 재고를 원복하고, 주문을 **결제 실패**로 마감한다. 활동 기록에 "주문 실패" 사건을 남긴다
   - **응답 지연/누락** → 주문을 결제 대기 그대로 둔다. 시스템은 추후 외부 결제 시스템에 다시 확인하여 최종 상태(완료 또는 실패)를 확정한다

**실패 흐름**
- 본인 식별 실패 → "로그인이 필요합니다"
- 주문 항목이 비어있거나 수량이 부적절 → "잘못된 요청" 안내
- 동일 상품을 같은 주문에 두 번 담음 → "잘못된 요청" 안내
- 상품 중 하나라도 존재하지 않거나 비활성 → "찾을 수 없습니다" 안내 (어떤 상품이 누락되었는지 함께 안내)
- 상품 중 하나라도 재고가 부족함 → 주문 전체 실패. 어떤 상품의 재고가 얼마나 부족한지 안내

**사후 조건**
- 결제 완료 → 주문 1건 + 주문 항목 N건 보존. 재고 N건 감소. 활동 기록 1건
- 결제 실패 → 주문 1건이 실패 상태로 보존. 재고는 차감 시점의 직전 상태로 원복. 활동 기록 1건
- 결제 응답 지연 → 주문은 결제 대기, 재고는 차감 상태 유지. 추후 사이클이 최종 상태 확정

**규칙·정책**
- **전체 실패 원칙** — 항목 중 하나라도 실패하면 주문 전체가 실패한다. 부분 성공 없음
- **주문 시점 스냅샷** — 주문 항목의 상품명·단가·브랜드명·이미지는 주문 시점의 값으로 보존한다. 이후 상품/브랜드 변경·삭제와 무관하다
- **외부 결제와의 경계** — 외부 결제 시스템 호출은 주문 생성과 별도의 단계로 다룬다. 결제 실패 시 재고 원복은 시스템이 책임진다
- **중복 호출 방어** — 같은 주문 요청이 짧은 시간 안에 중복으로 들어와도 결제는 한 번만 일어난다

### UC-10. 브랜드 관리 (Admin)

- **액터**: Admin
- **사전 조건**: Admin으로 식별되어 있다
- **트리거**: 운영자가 브랜드를 등록·수정·조회·삭제한다

**정상 흐름**
- **등록** — 운영자가 브랜드 이름과 설명을 입력한다. 시스템은 활성 브랜드로 신규 등록한다
- **수정** — 운영자가 활성 브랜드의 이름·설명을 변경한다. 시스템은 변경을 적용한다
- **조회** — 운영자는 활성 브랜드만 보거나, "비활성 포함" 옵션으로 비활성 브랜드까지 함께 본다
- **삭제** — 운영자가 브랜드를 삭제하면 해당 브랜드를 비활성 처리한다. 동시에 §7.5의 비활성 전파에 따라 소속 상품과 그 상품의 좋아요까지 비활성된다. **단 이미 발생한 주문 내역은 영향 없음**

**실패 흐름**
- Admin 식별 실패 → 접근 거부
- 대상 브랜드가 이미 비활성이거나 존재하지 않음 → "찾을 수 없습니다"
- 이름이 비어있거나 길이 초과 → "잘못된 입력" 안내

**사후 조건**
- 등록·수정·삭제 시 브랜드(또는 비활성 전파) 변경
- 활동 기록(운영자용)에 액션 1건

**규칙·정책**
- 모든 삭제는 비활성 처리이며, 데이터를 물리적으로 제거하지 않는다 (§7.5)
- 운영자의 모든 카탈로그 액션은 활동 기록에 남는다

### UC-11. 상품 관리 (Admin)

- **액터**: Admin
- **사전 조건**: Admin으로 식별되어 있다
- **트리거**: 운영자가 상품을 등록·수정·조회·삭제한다

**정상 흐름**
- **등록** — 운영자가 상품의 브랜드·이름·가격·재고·이미지·설명을 입력한다. 시스템은 해당 브랜드 아래에 활성 상품으로 등록한다 (좋아요 수는 0에서 시작)
- **수정** — 운영자가 상품 정보를 변경한다
- **조회** — 활성 상품만 보거나 "비활성 포함"으로 함께 조회한다
- **삭제** — 상품을 비활성 처리한다. 해당 상품의 좋아요도 §7.5에 따라 함께 비활성된다. 이미 발생한 주문 내역은 영향 없음

**실패 흐름**
- Admin 식별 실패 → 접근 거부
- 대상 상품이 이미 비활성이거나 존재하지 않음 → "찾을 수 없습니다"
- 등록·수정 시 브랜드가 존재하지 않거나 비활성 → "잘못된 입력"
- 가격·재고가 음수, 이름이 비어 있음 → "잘못된 입력"

**사후 조건**
- 등록·수정·삭제별 상품 변경 + 활동 기록 1건

**규칙·정책**
- 가격·재고는 0 이상이어야 한다
- 비활성 브랜드 소속으로 상품을 등록·이동할 수 없다
- 상품 가격을 변경해도 이미 발생한 주문 내역은 변하지 않는다 (스냅샷)

## 7. 비즈니스 규칙·정책

각 UC를 가로지르는 규칙. 본문에서 §7.x 참조로 호출한다.

### 7.1 비밀번호 정책

- 길이: 8자 이상, 16자 이하
- 허용 문자: 영문 대소문자, 숫자, 지정 특수문자(`! @ # $ % ^ & * ( ) _ + - = [ ] { } | \ ; : ' " , . < > / ?`). 한글 등 그 외 문자 포함 시 거부
- 본인의 생년월일을 비밀번호에 포함할 수 없다 (`yyyyMMdd` · `yyMMdd` · `MMdd` 어느 형식이든 부분 일치 시 거부)
- 평문 비밀번호는 어디에도 저장·노출되지 않는다 (활동 기록 포함)
- 비밀번호가 변경되면 이전 비밀번호는 즉시 무효가 된다

### 7.2 목록 조회 정책

- 페이지 단위로만 보여준다. 한 페이지의 상한은 100건
- 같은 정렬 조건에서 동률(예: 동일 시간에 등록된 두 상품)이 발생해도 페이지 경계가 일관되게 유지된다 — 같은 조회를 두 번 해도 결과 순서가 흔들리지 않는다

### 7.3 좋아요 수 표시

- 상품의 좋아요 수는 사용자가 좋아요를 누르거나 취소하는 즉시 화면에 반영된다
- 좋아요가 같은 사용자에 의해 여러 번 눌러져도 한 번만 카운트된다
- 좋아요 수는 음수가 되지 않는다

### 7.4 본인 자원 접근 정책

- 좋아요·주문·비밀번호는 본인 자원이다. 본인 외에는 접근할 수 없다
- 본인이 아닌 사용자가 다른 사용자의 자원에 접근을 시도하면 **자원 존재 자체를 알리지 않는다** — "찾을 수 없습니다"로만 응대 (예: 다른 사람의 주문 번호를 추측해 조회를 시도해도, 시스템은 그 주문이 있는지 없는지 누설하지 않는다)
- 운영자는 이 격리 규칙의 대상이 아니다

### 7.5 비활성 처리와 전파

- 브랜드·상품·좋아요는 삭제 요청이 와도 데이터를 물리적으로 지우지 않는다. 대신 **비활성** 상태로 표시하고 대고객 화면에서 보이지 않게 한다
- 전파 규칙:
  - 브랜드를 비활성하면 → 소속된 모든 상품을 비활성하고 → 그 상품에 걸린 좋아요도 모두 비활성한다
  - 상품을 비활성하면 → 그 상품에 걸린 좋아요를 모두 비활성한다
  - 좋아요를 취소하면 → 해당 좋아요만 비활성한다
- 이미 발생한 주문과 주문 항목은 이 전파의 영향을 받지 않는다. 주문 항목은 주문 시점의 스냅샷을 그대로 유지한다
- 운영자는 비활성 자원도 조회·복구 검토할 수 있다 (조회 옵션으로)

### 7.6 주문과 외부 결제의 경계

- 주문 생성과 외부 결제 호출은 별도 단계로 다룬다
- 외부 결제 결과가 실패인 경우 시스템은 차감했던 재고를 원복한다
- 외부 결제의 응답이 지연되거나 누락되는 경우 주문은 결제 대기 상태로 두고, 추후 사이클이 외부 결제 시스템에 재확인해 최종 상태를 확정한다
- 같은 주문 요청의 중복 호출은 결제를 두 번 일으키지 않는다

### 7.7 활동 기록

사용자·운영자·시스템의 주요 행동을 활동 기록으로 남긴다. 추후 랭킹·추천·운영 감사의 토대가 된다.

| 대상 | 기록 사건 |
| --- | --- |
| 대고객 | 브랜드/상품 조회·검색, 좋아요·좋아요 취소, 주문 생성·완료·실패, 비밀번호 변경 |
| 운영자 | 브랜드/상품 등록·수정·삭제, 주문 조회 (누가 누구의 주문을 봤는지) |
| 시스템 | 주문 상태 전이 (결제 대기 → 완료/실패) |

개인 식별 가능 정보·결제 카드 상세·비밀번호는 활동 기록에서 마스킹되거나 제외된다.

### 7.8 시간 표시

- 시스템 내부의 모든 시각은 UTC 기준이다
- 사용자 화면에 보이는 시간은 사용자 환경의 로컬 시간대로 변환되어 표시된다 (변환은 사용자 화면 쪽 책임)

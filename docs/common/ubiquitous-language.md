# 00. Ubiquitous Language

## 목적

이 문서는 루퍼스의 "우리가 함께 만들어갈 단 하나의 감성 이커머스" 프로젝트를 진행하는 동안 사용할 공통 언어를 정의한다.
요구사항, 시퀀스 다이어그램, 클래스 다이어그램, ERD, 코드에서 같은 개념을 같은 이름으로 다루는 것을 목표로 한다.

## 범위

현재 문서는 메인 이커머스 시나리오 중 다음 도메인을 우선 정의한다.

- 브랜드 조회
- 상품 목록 조회
- 상품 상세 조회
- 상품 좋아요 등록/취소
- 주문 생성
- 주문 생성 이후 결제 연동 흐름

회원가입, 내 정보 조회, 비밀번호 변경은 현재 용어 정의 범위에서 제외한다.
다만 좋아요와 주문은 로그인 사용자를 전제로 하므로, 사용자 식별 정보는 외부에서 이미 검증되어 전달된 값으로 본다.

## 핵심 용어

| 한글 용어 | 영문 용어 | 설명 |
| --- | --- | --- |
| 사용자 | User | 서비스를 이용하는 고객. 좋아요와 주문의 주체다. |
| 로그인 ID | Login ID | 사용자를 식별하기 위해 요청 헤더로 전달되는 값이다. |
| 브랜드 | Brand | 상품을 소유하거나 판매 주체로 표현되는 단위다. |
| 상품 | Product | 사용자가 조회하고 좋아요하거나 주문할 수 있는 판매 대상이다. |
| 상품 목록 | Product List | 조건과 정렬 기준에 따라 조회되는 상품들의 목록이다. |
| 상품 상세 | Product Detail | 단일 상품의 가격, 재고, 브랜드, 좋아요 수 등을 포함한 상세 정보다. |
| 좋아요 | Like | 사용자가 특정 상품에 관심을 표현한 상태다. |
| 좋아요 수 | Like Count | 상품에 등록된 좋아요의 개수다. 정렬과 화면 표시의 기준으로 사용될 수 있다. |
| 주문 | Order | 사용자가 하나 이상의 상품 구매를 요청한 결과다. |
| 주문 항목 | Order Item | 주문에 포함된 개별 상품과 수량 정보다. |
| 주문 상품 스냅샷 | Order Product Snapshot | 주문 시점의 상품명, 가격, 브랜드명 등 주문 이력 보존을 위한 복사 데이터다. |
| 결제 | Payment | 주문 금액을 외부 결제 시스템을 통해 승인받는 행위다. |
| 결제 시스템 | Payment Gateway | 결제 승인을 처리하는 외부 시스템이다. |
| 재고 | Stock | 상품을 주문할 수 있는 남은 수량이다. |
| 재고 차감 | Stock Deduction | 주문 생성 과정에서 주문 수량만큼 상품 재고를 줄이는 행위다. |
| 판매 상태 | Product Status | 상품이 주문 가능한 상태인지 표현하는 값이다. |

## 도메인별 언어

### Brand

브랜드는 상품을 묶는 기준이다.
사용자는 브랜드 정보를 조회할 수 있고, 상품 목록은 브랜드 기준으로 필터링될 수 있다.

| 용어 | 권장 이름 | 설명 |
| --- | --- | --- |
| 브랜드 식별자 | `brandId` | 브랜드를 식별하는 고유 ID다. |
| 브랜드명 | `brandName` | 사용자에게 노출되는 브랜드 이름이다. |
| 브랜드 설명 | `description` | 브랜드를 설명하는 문구다. |

### Product

상품은 사용자가 조회, 좋아요, 주문하는 중심 대상이다.
상품은 반드시 하나의 브랜드에 속한다.

| 용어 | 권장 이름 | 설명 |
| --- | --- | --- |
| 상품 식별자 | `productId` | 상품을 식별하는 고유 ID다. |
| 상품명 | `productName` | 사용자에게 노출되는 상품 이름이다. |
| 가격 | `price` | 주문 금액 계산에 사용되는 상품의 현재 판매 가격이다. |
| 재고 | `stock` | 주문 가능한 남은 수량이다. |
| 상품 상태 | `productStatus` | 상품의 판매 가능 여부를 표현한다. |
| 최신순 | `latest` | 상품 목록의 기본 정렬 기준이다. |
| 가격 낮은순 | `price_asc` | 낮은 가격부터 상품을 정렬하는 기준이다. |
| 좋아요 많은순 | `likes_desc` | 좋아요 수가 많은 상품부터 정렬하는 기준이다. |

### Like

좋아요는 사용자와 상품 사이의 관계다.
같은 사용자는 같은 상품에 좋아요를 하나만 가질 수 있다.

| 용어 | 권장 이름 | 설명 |
| --- | --- | --- |
| 좋아요 등록 | `like` | 사용자가 상품에 좋아요 상태를 만드는 행위다. |
| 좋아요 취소 | `unlike` | 사용자가 상품의 좋아요 상태를 제거하는 행위다. |
| 좋아요 여부 | `liked` | 사용자가 해당 상품을 좋아요했는지 나타내는 값이다. |
| 좋아요한 상품 목록 | `likedProducts` | 사용자가 좋아요한 상품들의 목록이다. |
| 멱등 좋아요 | `idempotent like` | 같은 좋아요 등록 요청이 반복되어도 결과 상태가 하나의 좋아요로 유지되는 성질이다. |

### Order

주문은 사용자가 여러 상품을 한 번에 구매 요청한 결과다.
주문 생성 시 상품 재고 확인과 재고 차감이 함께 고려되어야 한다.

| 용어 | 권장 이름 | 설명 |
| --- | --- | --- |
| 주문 식별자 | `orderId` | 주문을 식별하는 고유 ID다. |
| 주문 생성 | `placeOrder` | 사용자가 주문을 요청하고 시스템이 주문을 생성하는 행위다. |
| 주문 수량 | `quantity` | 주문 항목에 포함된 상품 개수다. |
| 주문 총액 | `totalAmount` | 주문 항목들의 가격과 수량을 기준으로 계산한 금액이다. |
| 주문 상태 | `orderStatus` | 주문의 진행 상태를 표현한다. |
| 주문 목록 | `orders` | 사용자의 주문 이력 목록이다. |
| 주문 상세 | `orderDetail` | 단일 주문의 상태, 주문 항목, 금액 정보를 포함한 상세 정보다. |

### Payment

결제는 주문 생성 이후 외부 결제 시스템과 연결되는 흐름이다.
2주차에서는 실제 결제 구현보다 주문과 외부 시스템의 책임 경계를 명확히 표현하는 것을 우선한다.

| 용어 | 권장 이름 | 설명 |
| --- | --- | --- |
| 결제 요청 | `requestPayment` | 주문 금액에 대한 결제 승인을 외부 시스템에 요청하는 행위다. |
| 결제 승인 | `paymentApproved` | 외부 결제 시스템이 결제를 성공 처리한 상태다. |
| 결제 실패 | `paymentFailed` | 외부 결제 시스템이 결제를 거절하거나 오류를 반환한 상태다. |
| 보상 처리 | `compensation` | 결제 실패 등으로 이미 수행한 재고 차감이나 주문 상태 변경을 되돌리거나 보정하는 행위다. |

## 상태 용어

### Product Status

| 상태 | 설명 |
| --- | --- |
| `ON_SALE` | 판매 중이며 주문 가능한 상태다. |
| `SOLD_OUT` | 재고가 없어 주문할 수 없는 상태다. |
| `HIDDEN` | 사용자에게 노출하지 않는 상태다. |
| `DELETED` | 삭제 처리된 상태다. 물리 삭제를 선택한다면 사용하지 않을 수 있다. |

### Order Status

| 상태 | 설명 |
| --- | --- |
| `CREATED` | 주문이 생성되었지만 결제가 완료되지 않은 상태다. |
| `PAID` | 결제가 승인되어 주문이 확정된 상태다. |
| `FAILED` | 주문 또는 결제 처리 중 실패한 상태다. |
| `CANCELED` | 주문이 취소된 상태다. |

### Payment Status

| 상태 | 설명 |
| --- | --- |
| `READY` | 결제 요청 전 대기 상태다. |
| `APPROVED` | 결제가 승인된 상태다. |
| `FAILED` | 결제가 실패한 상태다. |
| `CANCELED` | 승인된 결제가 취소된 상태다. |

## 용어 사용 원칙

- 문서와 코드에서는 `Product`, `Brand`, `Like`, `Order`, `OrderItem`, `Payment`를 기준 용어로 사용한다.
- `Item`은 주문 항목을 의미할 때만 사용하고, 판매 대상은 `Product`로 부른다.
- `Member`, `Customer`, `Account` 대신 2주차 문서에서는 `User`로 통일한다.
- 좋아요 등록/취소는 토글이 아니라 각각 `like`, `unlike`로 표현한다.
- 주문 생성은 단순 저장이 아니라 재고 확인, 재고 차감, 주문 항목 스냅샷 생성을 포함하는 유스케이스로 본다.
- 결제 시스템은 내부 도메인이 아니라 외부 시스템으로 표현한다.

## 용어 미정 항목

현재 문서에서는 아래 용어를 임시로 사용한다.
요구사항과 설계가 구체화되면 실제 코드와 문서에서 사용할 이름으로 확정한다.

| 후보 용어 | 현재 사용 | 확정 필요 이유 |
| --- | --- | --- |
| 사용자 / 회원 | `User` | 1주차 회원 도메인과 연결할 때 `User`와 `Member` 중 코드 명칭을 통일해야 한다. |
| 상품 항목 / 주문 항목 | `OrderItem` | `Item`은 판매 대상이 아니라 주문에 포함된 상품 행을 의미하도록 제한할지 확정해야 한다. |
| 결제 시스템 / 결제 게이트웨이 | `Payment Gateway` | 실제 외부 연동을 단순 외부 시스템으로 볼지, 게이트웨이 인터페이스로 모델링할지 확정해야 한다. |
| 상품 상태 / 판매 상태 | `Product Status` | 상태가 상품 노출 여부까지 포함하는지, 주문 가능 여부만 표현하는지 확정해야 한다. |

## 3. 핵심 결론 (보고서에 반영할 것)
- 좋아요 정렬 구조: **비정규화(`likes_count` 컬럼) 채택** (MySQL 네이티브 MV 부재 + 조인 회귀로 MV 제외).
- 인덱스 최종 픽: **복합 DESC** `idx_brand_active_likes_desc (brand_id, deleted_at, likes_count DESC, id DESC)` + `idx_active_likes_desc (deleted_at, likes_count DESC, id DESC)`.
  - 단일=filesort 잔존(무인덱스급 218~262ms). ASC backward=딥오프셋 페널티(13~34ms). 커버링=인덱스 2배 비용 대비 page0 실익 없음 → 보류.
  - DESC는 `@Index`로 방향 표현 불가 → DDL 관리(local/test `import.sql`, 운영 `migration_product_indexes.sql`).
- count 동기화: ① 동기 원자 UPDATE(`x=x+1`, lost update 차단) → ② hot row 해소 위해 Kafka 비동기 집계 + 10분 reconcile(결과적 일관성)로 진화. 진실원천=`product_like`, `likes_count`=파생 캐시.

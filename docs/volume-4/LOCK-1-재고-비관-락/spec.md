# Spec: LOCK-1 재고 차감 비관적 락

**소스**: `docs/volume-4/01-requirements.md` — 결정 10 / ORD-7 비기능 요구사항
**작성일**: 2026-06-10
**상태**: Draft

## 시나리오 요약

인기 상품에 여러 주문이 동시에 몰릴 때 재고가 정상적으로 차감되어 과판매가 발생하지 않도록, 주문 흐름의 재고 차감을 비관적 락으로 직렬화한다. 단계 3에서 재고 차감을 원자적 단일 UPDATE에서 read-modify-write(`getActiveById` 조회 → `ProductModel.decreaseStock` 메모리 차감)로 전환하면서, 같은 상품에 동시 주문이 몰리면 여러 트랜잭션이 같은 재고 값을 읽어 도메인의 재고 검증이 모두 통과한 뒤 서로의 차감을 덮어쓰는 Lost Update(재고 1개에 9건 주문 성공)가 재현되었다. 이를 상품 행을 잠금 조회(`SELECT ... FOR UPDATE`)한 뒤 차감하는 비관적 락으로 막는다. 락은 트랜잭션 커밋까지 유지되어 같은 상품의 동시 차감이 직렬로 처리되며, 다중 항목 주문은 상품 식별자 순으로 잠금을 획득해 데드락을 피한다. 단건·다건 주문의 기능적 동작은 단계 3과 동일하며(회귀 없음), 동시성 안전만 추가된다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 재고가 1개인 상품, **When** 10명이 같은 상품을 동시에 1개씩 주문, **Then** 정확히 1건만 성공하고 나머지 9건은 재고 부족(409)으로 실패하며 최종 재고는 0이다.
2. **Given** 재고가 N개인 상품, **When** N건을 초과하는 동시 주문, **Then** 성공 건수는 정확히 N이고 재고는 0이며 음수가 되지 않는다.
3. **Given** 재고가 충분한 상품, **When** 단건 주문(비동시), **Then** 단계 3과 동일하게 재고가 차감되고 주문이 생성된다(기능 회귀 없음).

### Exception Flow
1. **Given** 재고가 요청 수량보다 적은 상품, **When** 주문, **Then** 409 CONFLICT, 차감 없음(단계 3 동작 유지).

### 비즈니스 규칙
- 같은 상품에 대한 동시 재고 차감은 비관적 락으로 직렬화되어, 재고보다 많은 수량이 팔리지 않는다.
- 락은 차감 트랜잭션 커밋까지 유지된다(읽은 재고가 차감 완료 전까지 다른 트랜잭션에 의해 바뀌지 않음).
- 한 주문이 여러 상품을 다룰 때는 상품 식별자 순으로 잠금을 획득해 교차 대기(데드락)를 피한다.

## 엣지 케이스
- 재고 1 + 동시 N(N≫1): 성공 1, 재고 0.
- 재고 N + 동시 M(M>N): 성공 N, 재고 0, 음수 없음.
- 다중 항목 주문 동시 실행: 상품 식별자 순 잠금으로 데드락 미발생.
- 단건/비동시 주문: 락이 동작에 영향을 주지 않음(회귀 없음).

## 기능 요구사항
- **FR-001**: 시스템은 재고 차감 시 대상 상품 행을 비관적 쓰기 락으로 잠금 조회해야 한다.
- **FR-002**: 시스템은 잠금 조회한 상품에 대해서만 재고를 차감하며, 락은 트랜잭션 커밋까지 유지되어 같은 상품의 동시 차감을 직렬화해야 한다.
- **FR-003**: 시스템은 다중 항목 주문에서 상품 식별자 순으로 잠금을 획득해 데드락을 피해야 한다.
- **FR-004**: 시스템은 동시 주문 수가 재고를 초과해도 성공 건수가 재고를 넘지 않고 재고가 음수가 되지 않도록 보장해야 한다.

> **설계 출처**: 본 흐름은 `02-sequence-diagrams.md`의 ORD-7 다이어그램(재고 `getByIdForUpdate` 잠금 조회 → `ProductModel.decreaseStock`)과 `01-requirements.md` 결정 10을 따른다.

## 관련 엔티티
- **ProductRepository** (메서드 추가): 비관적 쓰기 락으로 활성 상품을 단건 조회하는 잠금 조회 메서드. 부재 시 NOT_FOUND(기존 `getActiveById` 응답 어휘 계승). 기존 `getActiveById`(락 없음)는 조회 전용으로 유지된다.
- **ProductModel** (기존): `decreaseStock(quantity)` — 단계 3에서 추가된 RMW 차감 메서드. 재고 부족 시 CONFLICT. 변경 없음.
- **OrderFacade** (기존, 변경): 주문 항목 처리 루프에서 재고를 차감할 상품을 잠금 조회 메서드로 가져온다. 상품 식별자 정렬(데드락 방지)은 단계 3 흐름 유지. 단가·브랜드명 스냅샷용 조회와 차감용 잠금 조회의 관계는 plan에서 확정한다.
- **재사용**: `@Transactional` 트랜잭션 경계(ORD-7), `ErrorType`(NOT_FOUND·CONFLICT).

## 관련 결정
- **결정 10 (재고 차감 동시성 — 비관적 락)**: 충돌 빈도가 높은 인기 상품 특성상 낙관적 락은 재시도 폭증을 유발하므로 비관적 락으로 직렬화. 원자적 UPDATE(대안 C)는 성능·단순성 우위이나 락 학습 목표·RMW 확장성에서 비관적 락 채택. 데드락은 상품 식별자 순 잠금 획득으로 회피.

## 성공 기준 / 범위 밖
- **성공**: `ConcurrentStockDecrease` 통과(성공 1·재고 0) + 기존 단일스레드 단위/통합/E2E 전부 그린(회귀 없음).
- **범위 밖**: 쿠폰 사용 동시성(LOCK-2), 좋아요 동시성(단계 5), 락 타임아웃·재시도 정책 튜닝, 분산 락.

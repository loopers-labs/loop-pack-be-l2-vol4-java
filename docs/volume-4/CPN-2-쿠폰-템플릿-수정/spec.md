# Spec: CPN-2 쿠폰 템플릿 수정

**소스**: `docs/volume-4/01-requirements.md` — CPN-2
**작성일**: 2026-06-10
**상태**: Draft

## 시나리오 요약

관리자가 기존 쿠폰 템플릿의 이름·할인 타입·할인 값·최소 주문 금액·만료 시각을 수정한다. 관리자 인증을 통과한 요청만 허용하며, 대상 템플릿이 활성 상태(삭제되지 않음)로 존재하고 입력값이 등록(CPN-1)과 동일한 허용 범위 안에 있으면 정보를 갱신하고 식별자를 반환한다. 수정은 이후 발급분에만 영향을 주며, 이미 발급된 쿠폰은 발급 시점 스냅샷을 유지한다(결정 3).

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 관리자 인증 + 활성 템플릿 존재, **When** 새 이름·타입·할인 값·최소 주문 금액·미래 만료 시각으로 수정 요청, **Then** 정보가 갱신되고 식별자가 반환된다(200).
2. **Given** 활성 템플릿 존재, **When** 기존과 동일한 값으로 수정 요청, **Then** 정상 처리되고 갱신 시각만 새로 찍힌다.
3. **Given** 활성 템플릿 존재, **When** 최소 주문 금액을 생략해 수정 요청, **Then** 최소 주문 금액 없이(null) 정상 갱신된다.

### Exception Flow
1. **Given** 관리자 인증 실패, **When** 수정 요청, **Then** 403 FORBIDDEN.
2. **Given** 대상 템플릿이 부재거나 이미 삭제됨, **When** 수정 요청, **Then** 404 NOT_FOUND.
3. **Given** 관리자 인증 통과, **When** 이름이 빈 값·100자 초과, **Then** 400 BAD_REQUEST.
4. **Given** 관리자 인증 통과, **When** 할인 타입이 허용되지 않는 값, **Then** 400 BAD_REQUEST.
5. **Given** 관리자 인증 통과, **When** 정률 값이 1~100 밖이거나 정액 값이 1 미만, **Then** 400 BAD_REQUEST.
6. **Given** 관리자 인증 통과, **When** 최소 주문 금액 지정 시 0 미만, **Then** 400 BAD_REQUEST.
7. **Given** 관리자 인증 통과, **When** 만료 시각을 현재 시각 이전으로 수정, **Then** 400 BAD_REQUEST.

### 비즈니스 규칙
- 입력값 허용 범위는 등록(CPN-1)과 동일하다(이름 1~100자, 타입 FIXED/RATE, 정률 1~100·정액 ≥1, 최소금액 선택 ≥0(미지정 시 0), 만료 현재 이후).
- 만료 시각을 현재 시각 이전으로 수정하는 것도 거부한다 — 신규 발급의 조기 종료가 필요하면 수정이 아니라 삭제(CPN-3)로 처리한다.
- 수정은 이후 발급분에만 영향을 준다. 이미 발급된 쿠폰은 발급 시점 정보를 그대로 유지한다(결정 3, 스냅샷). 발급 쿠폰(`UserCoupon`)은 CPN-6 범위라 본 cycle 검증 대상은 없고 규칙만 명시한다.
- 동일한 값으로 수정해도 정상 처리되며 갱신 시각만 새로 찍힌다(CPN-1 이름 중복 허용과 일관 — 중복 검사 없음).
- 쿠폰 이름은 중복 허용이므로 BRD-5와 달리 이름 중복(409) 검사가 없다.

## 엣지 케이스
- 이름 0/1/100/101자, 정률 0/1/100/101, 정액 0/1, 최소금액 미입력(0)/0/-1, 만료 과거/미래 (CPN-1과 동일 경계).
- 대상 미존재(404) / 이미 삭제됨(404).
- 동일 값 수정(통과, 갱신 시각만) / 최소금액 있던 것을 생략 수정(0으로 갱신).
- 권한 경계: 회원 헤더만 있고 admin 헤더 없으면 403.

## 기능 요구사항
- **FR-001**: 시스템은 관리자 인증 통과 요청만 수정을 허용한다(실패 403).
- **FR-002**: 시스템은 대상 템플릿이 활성 상태로 존재하는지 확인한다(부재/삭제 404).
- **FR-003**: 시스템은 입력값을 등록(CPN-1)과 동일한 범위로 검증한다(위반 400).
- **FR-004**: 시스템은 만료 시각이 현재 시각 이후인지 검증한다(이전이면 400).
- **FR-005**: 시스템은 템플릿 정보를 갱신하고 식별자를 반환한다.

## 관련 엔티티
- **CouponModel**: 자기 속성 갱신 행위 `update(name, type, value, minOrderAmount, expiredAt)` 신규 추가 — 생성과 동일하게 `Name.from`·`DiscountType.validate`·`ExpiredAt.of`·`MinOrderAmount`로 재검증.
- **CouponRepository**: 활성 단건 조회(`getActiveById` — 부재/삭제 시 NOT_FOUND) 신규.
- **재사용**: `AdminAuthInterceptor`, `ErrorType`(BAD_REQUEST·FORBIDDEN·NOT_FOUND), CPN-1 VO·enum.

## 관련 결정
- **결정 3 (스냅샷)**: 수정은 이후 발급분에만 영향. 이미 발급된 쿠폰은 발급 시점 정보 유지(본 cycle은 규칙만, 검증은 CPN-6 이후).
- **CPN-1 review 결정 계승**: discountType은 `DiscountType` enum 수신(허용 외 값 advice가 400), 할인 값 검증은 `DiscountType.validate`가 단일 소유, 만료 검증은 `ExpiredAt` VO.
- **CPN-1 신규 결정 계승 (이름 중복 허용)**: 이름 중복 검사·409 없음.

## 성공 기준 / 범위 밖
- **성공**: 위 모든 수용 시나리오·테스트 green. `PUT /api-admin/v1/coupons/{couponId}`가 인증·존재·검증·갱신 분기를 명세대로 처리.
- **범위 밖**: 발급 쿠폰 스냅샷 불변 검증(CPN-6 이후), 삭제(CPN-3), 동시 수정 race.

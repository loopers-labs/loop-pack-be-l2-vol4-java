# Spec: CPN-6 쿠폰 발급

**소스**: `docs/volume-4/01-requirements.md` — CPN-6
**작성일**: 2026-06-10
**상태**: Draft

## 시나리오 요약

로그인한 회원이 특정 쿠폰 템플릿으로부터 자신의 쿠폰 한 장을 발급받는다. 회원 인증을 통과한 요청만 허용하며, 대상 템플릿이 활성 상태(삭제되지 않음)로 존재하고 만료 시각이 지나지 않았으며 해당 회원이 그 템플릿에서 아직 발급받은 이력이 없으면, 발급 시점의 템플릿 정보(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)를 스냅샷으로 복사한 발급 쿠폰을 생성하고 그 식별자를 반환한다. Coupon 도메인의 두 번째 aggregate라 발급 쿠폰(`UserCoupon`) 골격(`UserCoupon` Model + `UserCouponRepository`)을 새로 세운다. 회원 인증 토대(`@LoginUser`)와 Coupon 템플릿 조회(`CouponRepository.getActiveById`)는 기존 것을 재사용한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 회원 인증을 통과하고 활성·미만료 템플릿이 존재하며 그 템플릿 발급 이력이 없을 때, **When** 발급을 요청하면, **Then** 발급 시점 템플릿 정보를 스냅샷한 발급 쿠폰이 생성되고 발급 쿠폰 식별자가 반환된다(201).
2. **Given** 동일 회원이 서로 다른 두 템플릿에서 각각 발급받을 때, **When** 두 번째 템플릿 발급을 요청하면, **Then** 정상 발급된다(1인 1매는 템플릿 단위).
3. **Given** 한 템플릿에서 서로 다른 두 회원이 각각 발급받을 때, **When** 두 번째 회원이 발급을 요청하면, **Then** 정상 발급된다(중복 판정은 회원·템플릿 쌍 단위).

### Exception Flow
1. **Given** 회원 인증 정보가 없거나 올바르지 않을 때, **When** 발급을 요청하면, **Then** 인증 실패로 응답한다(401 UNAUTHORIZED).
2. **Given** 대상 템플릿이 존재하지 않거나 이미 삭제됐을 때, **When** 발급을 요청하면, **Then** 해당 자원을 찾을 수 없다고 응답한다(404 NOT_FOUND).
3. **Given** 대상 템플릿의 만료 시각이 현재 시각을 지났을 때, **When** 발급을 요청하면, **Then** 자원 충돌로 응답한다(409 CONFLICT).
4. **Given** 해당 회원이 그 템플릿에서 이미 발급받은 이력이 있을 때, **When** 같은 템플릿 발급을 다시 요청하면, **Then** 자원 충돌로 응답한다(409 CONFLICT).
5. **Given** 회원이 한 템플릿의 쿠폰을 발급받아 사용까지 마친 뒤, **When** 같은 템플릿 발급을 다시 요청하면, **Then** 자원 충돌로 응답한다(409 CONFLICT — 발급 이력 기준이라 사용 후에도 재발급 불가).

### 비즈니스 규칙
- 한 회원은 한 템플릿에서 한 장만 발급받을 수 있다(1인 1매). 기준은 발급 이력이며, 발급받은 쿠폰을 사용한 뒤에도 같은 템플릿에서 재발급할 수 없다. (결정 5)
- 발급 가능 조건은 템플릿이 삭제되지 않고(결정 4) 만료 시각이 지나지 않은 것(결정 2)이다.
- 발급 쿠폰은 발급 시점의 템플릿 정보(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)를 스냅샷으로 기록한다. 이후 템플릿이 수정·삭제되어도 발급 쿠폰의 조건은 변하지 않는다. (결정 3)
- 발급 쿠폰은 생성 시점부터 사용 가능하다(사용 시각 없음).
- 만료 판정의 기준 시각은 표현 계층이 요청 단위로 확정해 주입한다(`DateTimeUtil`). 한 요청 내 모든 시각 비교는 동일 기준을 쓴다.

## 엣지 케이스
- 발급 이력: 없음(통과) / 미사용 1장 존재(409) / 사용 완료 1장 존재(409).
- 템플릿 상태: 활성·미만료(통과) / 삭제됨(404) / 부재(404) / 만료(409).
- 만료 경계: 만료 시각이 현재와 같거나 이후(통과) / 현재 이전(409).
- 중복 판정 범위: 같은 회원·다른 템플릿(통과), 다른 회원·같은 템플릿(통과), 같은 회원·같은 템플릿(409).
- 권한 경계: 회원 인증 헤더(`X-Loopers-LoginId/Pw`)가 없거나 비밀번호 불일치면 401.
- 동시성(동일 회원·동일 템플릿 동시 발급 → 한 장만)은 **단계 3~4 범위**라 본 cycle 검증 대상이 아니다.

## 기능 요구사항
- **FR-001**: 시스템은 회원 인증(`X-Loopers-LoginId/Pw`)을 통과한 요청만 발급을 허용해야 한다. 실패 시 401로 응답한다.
- **FR-002**: 시스템은 대상 쿠폰 템플릿이 활성 상태(삭제되지 않음)로 존재하는지 확인해야 한다. 부재·삭제 시 404로 응답한다.
- **FR-003**: 시스템은 대상 템플릿의 만료 시각이 현재 시각 이후인지 확인해야 한다. 지났으면 409로 응답한다.
- **FR-004**: 시스템은 해당 회원이 그 템플릿에서 발급받은 이력이 있는지 확인해야 한다. 이력이 있으면 409로 응답한다(1인 1매).
- **FR-005**: 시스템은 발급 시점 템플릿 정보(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)를 스냅샷으로 복사한 발급 쿠폰을 생성하고, 생성된 발급 쿠폰의 식별자를 반환해야 한다.

## 관련 엔티티
- **UserCoupon** (신규 aggregate): 소유 회원 ID(`userId`)·템플릿 ID(`couponId`)와 스냅샷(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)·사용 시각(`usedAt`, 미사용 시 없음) 보유. 스냅샷은 템플릿에서 이미 VO로 검증된 값이라 원시 타입으로 보유하고 재검증하지 않는다(`OrderItem`의 상품 스냅샷과 동일 원리). 이번 cycle은 발급(`issue`) 정적 팩토리만 만든다 — `calculateDiscount`·`use`·`isOwnedBy`·`getStatus`·`isAvailable`은 CPN-7·ORD-7 범위라 도입하지 않는다.
- **Coupon** (기존, 메서드 추가): 만료 질의 `isExpired(now)` 신규 추가(발급 가능 판단의 일부). CPN-1에서 범위 밖으로 미뤘던 메서드를 이번에 추가한다.
- **UserCouponRepository** (신규): 발급 쿠폰 저장, 회원·템플릿 발급 이력 존재 확인.
- **재사용**: `CouponRepository.getActiveById`(CPN-2, 부재/삭제 시 NOT_FOUND), `UserRepository.getActiveById`(회원 존재 확인, ORD 패턴 계승), `@LoginUser AuthenticatedUser`(회원 인증), `ErrorType`(UNAUTHENTICATED·NOT_FOUND·CONFLICT), `DateTimeUtil`(기준 시각 주입).

## 관련 결정
- **결정 1 (템플릿·발급 쿠폰 분리)**: `UserCoupon`은 템플릿(`Coupon`)과 별도 aggregate. 템플릿을 `couponId`로 ID 참조하되 할인 조건·만료를 발급 시점 스냅샷으로 복사 보유한다.
- **결정 2 (만료 시각 의미)**: 템플릿의 만료 시각은 발급 가능 기한이며, 발급 시 쿠폰에 복사되어 그 쿠폰의 사용 기한으로 고정된다. 발급 가능 판정은 템플릿 만료 시각 기준.
- **결정 3 (스냅샷)**: 발급 쿠폰은 발급 시점 템플릿 정보를 그대로 기록한다. 스냅샷 값은 템플릿 생성 시 검증된 값이라 발급 쿠폰에서 재검증하지 않는다.
- **결정 4 (삭제와 독립)**: 발급 가능 조건은 템플릿이 삭제되지 않은 것. 삭제된 템플릿으로는 발급할 수 없다(`getActiveById`가 NOT_FOUND).
- **결정 5 (1인 1매)**: 회원·템플릿 쌍 기준 발급 이력으로 중복을 판정. 사용 후에도 재발급 불가. 재발급 요청은 자원 충돌(409)로 명시 응답한다(좋아요 멱등 정상 응답과 의도된 차이).
- **결정 7 어휘 계승**: 만료·중복 발급은 상태 불일치라 CONFLICT, 템플릿 부재·삭제는 NOT_FOUND.
- **설계 문서 정정 필요 (analyze에서 정식 보고)**: `04-erd.md`의 `user_coupons.min_order_amount`가 `NULL`로 기재돼 있으나, 스냅샷 원천인 `Coupon.MinOrderAmount`가 항상 non-null(미지정 시 0)이고 `03-class-diagram.md`는 `int`(non-null)로 표기한다. 발급 쿠폰 스냅샷도 NOT NULL이어야 정합하므로 ERD를 `NOT NULL`로 정정한다.

## 성공 기준 / 범위 밖
- **성공**: 위 모든 수용 시나리오·테스트 계획이 green. `POST /api/v1/coupons/{couponId}/issue`가 인증·존재·만료·중복·발급 분기를 명세대로 처리.
- **범위 밖**: 내 쿠폰 목록(CPN-7)·발급 내역(CPN-8)·주문 적용(ORD-7), `UserCoupon.use`·`calculateDiscount`·`getStatus`·`isOwnedBy`·`isAvailable`, `UserCouponStatus` enum, `DiscountType.calculate`(할인 계산), 동시 발급 race(단계 3~4).

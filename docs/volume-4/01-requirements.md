# '감성 이커머스' 요구사항 명세

본 문서는 '감성 이커머스'의 기능에 대한 합의 사항을 기록한다. 한 줄 요청만으로는 드러나지 않는 전제·예외·정책을 풀어 적고, 트레이드오프가 있는 결정은 핵심 의사결정 분기에 그 이유와 함께 남긴다. 브랜드·상품·좋아요·주문 도메인의 기존 합의는 `docs/volume-2/01-requirements.md`를 따르며, 본 문서는 쿠폰 도메인과 주문 변경분만 다룬다.

---

## 도메인 및 용어 정의

### 쿠폰 (Coupon)

| 한글 | 영어 | 의미 |
|---|---|---|
| 쿠폰 템플릿 | `Coupon` | 관리자가 정의하는 할인 정책. 이름·할인 타입·할인 값·최소 주문 금액·만료 시각을 가진다. |
| 발급 쿠폰 | `UserCoupon` | 템플릿으로부터 회원에게 발급된 쿠폰 한 장. 발급 시점의 템플릿 정보(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)를 스냅샷으로 보유하며, 회원이 소유하고 한 번만 사용할 수 있다. (결정 1·3 참조) |
| 할인 타입 | `DiscountType` | 정액(`FIXED`) 또는 정률(`RATE`). |
| 할인 값 | `DiscountValue` | 정액이면 원 단위 금액(1 이상), 정률이면 퍼센트(1 이상 100 이하)인 정수. |
| 최소 주문 금액 | `MinOrderAmount` | 쿠폰을 적용할 수 있는 최소 주문 금액 조건. 선택 입력이며, 미지정 시 0(제약 없음)으로 취급한다. |
| 만료 시각 | `ExpiredAt` | 템플릿에서는 발급 가능 기한. 발급 시 쿠폰에 복사되어 그 쿠폰의 사용 기한으로 고정된다. (결정 2 참조) |
| 쿠폰 상태 | `UserCouponStatus` | 발급 쿠폰의 상태. 사용 가능(`AVAILABLE`)·사용 완료(`USED`)·만료(`EXPIRED`). 만료는 발급 쿠폰에 기록된 만료 시각 경과로 판정하며, 사용 완료가 만료보다 우선한다. |

### 주문 (Order)

| 한글 | 영어 | 의미 |
|---|---|---|
| 원 주문 금액 | `OriginalAmount` | 쿠폰 적용 전 주문 항목들의 (`단가 × 수량`) 합계. |
| 할인 금액 | `DiscountAmount` | 적용된 쿠폰으로 차감된 금액. 쿠폰 미적용 시 0. |
| 최종 결제 금액 | `FinalAmount` | `원 주문 금액 − 할인 금액`. 0 이상. |

## 쿠폰 (Coupon)

### CPN-1. 관리자는 새로운 쿠폰 템플릿을 등록할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 쿠폰 이름·할인 타입(정액/정률)·할인 값·최소 주문 금액(선택)·만료 시각을 포함해 쿠폰 템플릿 등록을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 입력값이 허용 범위 안에 있는지 검사한다.
    4. 새 쿠폰 템플릿을 생성한다.
    5. 생성된 쿠폰 템플릿의 식별자를 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 쿠폰 이름이 허용 범위(1자 이상 100자 이하)를 벗어나면, 시스템은 입력 검증 실패를 반환한다.
    - 할인 타입이 정액·정률 어느 쪽도 아니면, 시스템은 입력 검증 실패를 반환한다.
    - 정률 쿠폰의 할인 값이 1 이상 100 이하를 벗어나면, 시스템은 입력 검증 실패를 반환한다.
    - 정액 쿠폰의 할인 값이 1 미만이면, 시스템은 입력 검증 실패를 반환한다.
    - 최소 주문 금액이 지정되었는데 0 미만이면, 시스템은 입력 검증 실패를 반환한다.
    - 만료 시각이 현재 시각 이전이면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 할인 타입은 정액(`FIXED`)과 정률(`RATE`) 두 가지다.
- 정액 쿠폰의 할인 값은 원 단위 할인 금액으로 1 이상의 정수다.
- 정률 쿠폰의 할인 값은 퍼센트로 1 이상 100 이하의 정수다.
- 최소 주문 금액은 선택 입력이다. 지정하면 0 이상의 정수이며, 지정하지 않으면 0으로 취급되어 주문 금액 제약 없이 적용할 수 있다.
- 만료 시각은 이 템플릿의 발급 가능 기한이며, 발급 시 쿠폰에 기록되어 그 쿠폰의 사용 기한이 된다. (결정 2 참조)
- 쿠폰 이름은 중복될 수 있다. 템플릿의 식별은 식별자로만 한다. (브랜드 이름 중복 금지와 의도된 차이 — 같은 이름의 할인 행사를 기간을 달리해 반복 발행하는 운영이 자연스럽다.)

**관련 API**
- `POST /api-admin/v1/coupons`

### CPN-2. 관리자는 등록된 쿠폰 템플릿을 수정할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.
- 대상 쿠폰 템플릿이 삭제되지 않은 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 대상 쿠폰 템플릿과 새 이름·할인 타입·할인 값·최소 주문 금액(선택)·만료 시각을 포함해 수정을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 쿠폰 템플릿이 존재하고 삭제되지 않았는지 점검한다.
    4. 입력값이 허용 범위 안에 있는지 검사한다.
    5. 쿠폰 템플릿 정보를 갱신한다.
    6. 갱신된 쿠폰 템플릿의 식별자를 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 쿠폰 템플릿이 존재하지 않거나 이미 삭제된 상태이면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 입력값이 등록(CPN-1)과 동일한 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 입력값 허용 범위는 등록(CPN-1)과 동일하다. 만료 시각을 현재 시각 이전으로 수정하는 것도 동일하게 거부한다 — 신규 발급의 조기 종료가 필요하면 수정이 아니라 삭제로 처리한다.
- 수정은 이후 발급분에만 영향을 준다. 이미 발급된 쿠폰은 발급 시점에 기록된 정보(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)를 그대로 유지한다. (결정 3 참조)
- 동일한 값으로 수정을 요청해도 정상 처리되며 갱신 시각만 새로 찍힌다. (volume-2 BRD-5와 동일 정책)

**관련 API**
- `PUT /api-admin/v1/coupons/{couponId}`

### CPN-3. 관리자는 등록된 쿠폰 템플릿을 삭제할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 대상 쿠폰 템플릿 삭제를 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 쿠폰 템플릿이 존재하고 삭제되지 않은 경우, 템플릿을 삭제 처리한다.
    4. 대상 쿠폰 템플릿이 존재하지 않거나 이미 삭제된 경우, 별도 동작 없이 현재 상태를 유지한다.
    5. 삭제가 완료되었음을 알린다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.

**비즈니스 규칙**
- 삭제는 신규 발급의 중단을 의미한다. 삭제된 템플릿으로는 더 이상 발급할 수 없다.
- 이미 발급된 쿠폰은 발급 시점에 기록된 정보(결정 3 참조)로 독립적으로 동작하므로 삭제의 영향을 받지 않으며, 자신의 만료 시각까지 정상적으로 사용할 수 있다. (결정 4 참조)
- 삭제는 기술적으로 soft delete 방식으로, 행은 보존하되 외부 노출에서는 존재하지 않는 것으로 처리한다. (volume-2 결정 7과 일관)
- 존재하지 않거나 이미 삭제된 템플릿에 삭제 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 삭제). (volume-2 결정 6과 동일 정책)

**관련 API**
- `DELETE /api-admin/v1/coupons/{couponId}`

### CPN-4. 관리자는 등록된 쿠폰 템플릿 목록을 조회할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 페이지 정보를 포함해 쿠폰 템플릿 목록을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 삭제되지 않은 쿠폰 템플릿 한 페이지 분량과 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)를 반환한다.
    4. 응답의 각 항목에는 템플릿의 식별자·이름·할인 타입·할인 값·최소 주문 금액·만료 시각·등록 시각·갱신 시각이 포함된다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.

**비즈니스 규칙**
- 응답에는 삭제된 템플릿은 포함되지 않는다.
- 만료 시각이 지난 템플릿은 포함된다. 만료는 발급·사용의 차단 조건이지 존재의 소멸이 아니며, 관리자 목록은 운영 이력 확인이 목적이다.
- 페이지 크기를 지정하지 않으면 한 페이지에 20건을 반환한다.
- 결과는 등록 시각 내림차순으로 정렬한다.

**관련 API**
- `GET /api-admin/v1/coupons?page=0&size=20`

### CPN-5. 관리자는 특정 쿠폰 템플릿의 상세 정보를 조회할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 특정 쿠폰 템플릿의 상세 정보를 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 템플릿의 식별자·이름·할인 타입·할인 값·최소 주문 금액·만료 시각·등록 시각·갱신 시각을 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 요청한 쿠폰 템플릿이 존재하지 않거나 삭제되었으면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 만료 시각이 지난 템플릿도 정상 조회된다. (CPN-4와 동일 정책)

**관련 API**
- `GET /api-admin/v1/coupons/{couponId}`

### CPN-6. 회원은 특정 쿠폰 템플릿의 쿠폰을 발급받을 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.
- 대상 쿠폰 템플릿이 삭제되지 않고 만료되지 않은 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 특정 쿠폰 템플릿의 발급을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 대상 쿠폰 템플릿이 존재하고 삭제되지 않았는지 점검한다.
    4. 템플릿의 만료 시각이 지나지 않았는지 확인한다.
    5. 해당 회원이 이 템플릿에서 이미 발급받은 이력이 있는지 살핀다.
    6. 이력이 없으면 템플릿의 정보(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)를 기록한 발급 쿠폰을 생성한다.
    7. 생성된 발급 쿠폰의 식별자를 반환한다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 쿠폰 템플릿이 존재하지 않거나 삭제된 경우, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 템플릿의 만료 시각이 지난 경우, 시스템은 자원 충돌로 응답한다.
    - 해당 회원이 이 템플릿에서 이미 발급받은 이력이 있으면, 시스템은 자원 충돌로 응답한다.

**비즈니스 규칙**
- 한 회원은 한 템플릿에서 한 장만 발급받을 수 있다(1인 1매). 기준은 발급 이력이며, 발급받은 쿠폰을 사용한 뒤에도 같은 템플릿에서 재발급할 수 없다. (결정 5 참조)
- 발급 가능 조건은 템플릿이 삭제되지 않고(결정 4 참조) 만료 시각이 지나지 않은 것(결정 2 참조)이다.
- 발급 쿠폰은 발급 시점의 템플릿 정보를 스냅샷으로 기록한다. 이후 템플릿이 수정·삭제되어도 발급 쿠폰의 조건은 변하지 않는다. (결정 3 참조)
- 발급 쿠폰은 생성 시점부터 사용 가능하다.

**비기능 요구사항**
- 동일 회원이 동일 템플릿에 동시에 발급을 요청해도 발급 쿠폰은 한 장만 생성되어야 한다.

**관련 API**
- `POST /api/v1/coupons/{couponId}/issue`

### CPN-7. 회원은 자신이 발급받은 쿠폰 목록을 상태와 함께 조회할 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 자신의 쿠폰 목록을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 인증된 회원이 발급받은 쿠폰 전체를 반환한다.
    4. 응답의 각 항목에는 발급 쿠폰의 식별자·쿠폰 이름·할인 타입·할인 값·최소 주문 금액·만료 시각·상태(사용 가능/사용 완료/만료)가 포함된다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.

**비즈니스 규칙**
- 상태는 세 가지다: 사용 가능(`AVAILABLE`)·사용 완료(`USED`)·만료(`EXPIRED`).
- 만료는 발급 쿠폰에 기록된 만료 시각이 현재 시각을 지났는지로 판정한다. (결정 2 참조)
- 사용 완료가 만료보다 우선한다. 사용된 쿠폰은 만료 시각이 지나도 사용 완료로 표시된다.
- 템플릿이 삭제된 발급 쿠폰도 목록에 포함된다. (결정 4 참조)
- 페이지네이션 없이 전체를 반환한다. 1인 1매 정책(결정 5 참조)으로 회원당 쿠폰 수는 그간 등록된 템플릿 총수를 넘지 않는다.
- 결과는 발급 시각 내림차순으로 정렬한다.

**관련 API**
- `GET /api/v1/users/me/coupons`

### CPN-8. 관리자는 특정 쿠폰 템플릿의 발급 내역을 조회할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 특정 쿠폰 템플릿의 발급 내역을 페이지 정보와 함께 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 쿠폰 템플릿이 존재하고 삭제되지 않았는지 점검한다.
    4. 해당 템플릿에서 발급된 쿠폰 한 페이지 분량과 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)를 반환한다.
    5. 응답의 각 항목에는 발급 쿠폰의 식별자·발급받은 회원의 식별자·상태(사용 가능/사용 완료/만료)·발급 시각이 포함된다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 쿠폰 템플릿이 존재하지 않거나 삭제되었으면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 삭제된 템플릿의 발급 내역은 조회할 수 없다. 템플릿 목록·상세(CPN-4·5)와 동일한 노출 정책이다. 발급 쿠폰 자체는 스냅샷으로 독립적으로 살아 있으며(결정 4 참조) 회원의 쿠폰 목록(CPN-7)에서 확인된다.
- 상태 판정 규칙은 내 쿠폰 목록(CPN-7)과 동일하다.
- 페이지 크기를 지정하지 않으면 한 페이지에 20건을 반환한다.
- 결과는 발급 시각 내림차순으로 정렬한다.

**관련 API**
- `GET /api-admin/v1/coupons/{couponId}/issues?page=0&size=20`

## 주문 (Order)

### ORD-7. 회원은 주문에 보유한 쿠폰 한 장을 적용해 할인받을 수 있다.

ORD-1·ORD-2의 `POST /api/v1/orders`를 확장한다. 요청에 발급 쿠폰 식별자(`userCouponId`, 선택)가 추가되며, 주문 흐름에 쿠폰 검증·사용·할인 계산 단계가 끼어든다.

**전제 조건**
- 회원이 로그인 상태여야 한다.
- 주문 항목이 1개 이상 포함되어 있어야 한다. (ORD-2 전제 승계)
- 쿠폰을 적용하는 경우, 해당 발급 쿠폰이 본인 소유이며 사용 가능(미사용·미만료) 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 주문 항목들과 선택적으로 발급 쿠폰 식별자(`userCouponId`)를 포함해 주문을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 모든 항목의 상품이 존재하고 삭제되지 않았는지 점검한 뒤, 상품 식별자 순으로 정렬해 재고를 요청 수량만큼 차감한다. (ORD-2 흐름 승계)
    4. 쿠폰이 지정된 경우, 발급 쿠폰이 존재하고 본인 소유이며 사용 가능(미사용·미만료) 상태인지 점검한다.
    5. 할인 전 주문 금액(항목들의 `단가 × 수량` 합계)이 쿠폰의 최소 주문 금액 이상인지 확인한다.
    6. 할인 타입에 따라 할인 금액을 계산하고, 최종 결제 금액을 산출한다.
    7. 적용된 쿠폰을 사용 완료(`USED`)로 전이한다.
    8. 주문·주문 항목·금액 스냅샷(원 주문 금액·할인 금액·최종 결제 금액·적용 쿠폰 식별자)을 같은 트랜잭션 안에서 저장한다.
    9. 생성된 주문의 식별자·상태·주문 시각·원 주문 금액·할인 금액·최종 결제 금액·주문 항목 스냅샷을 반환한다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 어느 한 항목의 상품이 존재하지 않거나 삭제된 경우, 시스템은 해당 자원을 찾을 수 없다고 응답하며 이미 차감된 재고는 되돌린다. (ORD-2 승계)
    - 어느 한 항목의 재고가 요청 수량에 미치지 못하면, 시스템은 자원 충돌로 응답하며 이미 차감된 재고는 되돌린다. (ORD-2 승계)
    - 지정한 발급 쿠폰이 존재하지 않거나 본인 소유가 아니면, 시스템은 해당 자원을 찾을 수 없다고 응답하며 차감된 재고는 되돌린다. (결정 7 참조)
    - 지정한 발급 쿠폰이 이미 사용되었거나 만료된 경우, 시스템은 자원 충돌로 응답하며 차감된 재고는 되돌린다.
    - 할인 전 주문 금액이 쿠폰의 최소 주문 금액에 미치지 못하면, 시스템은 자원 충돌로 응답하며 차감된 재고는 되돌린다.

**비즈니스 규칙**
- 쿠폰은 주문 1건당 1장만 적용할 수 있다.
- 최소 주문 금액 충족 여부는 할인 적용 전 주문 금액을 기준으로 판정한다. (결정 8 참조)
- 정액 쿠폰의 할인 금액은 `min(쿠폰 할인 값, 할인 전 주문 금액)`이다. 할인액이 주문 금액을 넘지 않아 최종 결제 금액은 0 이상이다. (결정 8 참조)
- 정률 쿠폰의 할인 금액은 `할인 전 주문 금액 × 할인 값 ÷ 100`을 소수점 이하 내림한 정수다. (결정 8 참조)
- 최종 결제 금액은 `할인 전 주문 금액 − 할인 금액`이다.
- 주문 스냅샷에는 원 주문 금액·할인 금액·최종 결제 금액·적용 쿠폰 식별자가 기록된다. (결정 6 참조)
- 쿠폰을 적용하지 않은 주문도 동일한 스냅샷 구조를 가지며, 할인 금액은 0, 적용 쿠폰 식별자는 비어 있다.
- 쿠폰의 사용 완료 전이는 주문 성공과 같은 트랜잭션 안에서 이뤄진다. 어느 단계라도 실패하면 쿠폰은 다시 미사용 상태로 되돌아간다.

**비기능 요구사항**
- 재고 차감·쿠폰 사용·주문 저장·주문 항목 저장 중 어느 하나라도 실패하면 전체가 처음 상태로 되돌아간다.
- 동시에 같은 쿠폰으로 여러 주문이 요청되어도 쿠폰은 단 한 번만 사용되어야 한다.
- 동시에 같은 상품에 주문이 몰리는 상황에서도 재고가 음수가 되지 않아야 한다. (volume-2 결정 4 승계)

**관련 API**
- `POST /api/v1/orders`

## 핵심 의사결정 분기

### 결정 1. 쿠폰 템플릿·발급 쿠폰 분리와 주문 요청 식별자

**문제**: 관리자가 등록하는 할인 정책과 회원이 발급받는 쿠폰을 어떻게 모델링하고, 주문 요청의 쿠폰 식별자가 무엇을 가리키게 할지. (템플릿과 발급 쿠폰의 분리 자체는 관리자 API가 템플릿 CRUD·발급 내역 조회로, 대고객 API가 발급·내 쿠폰 목록으로 갈라져 있어 요구사항에 내재되어 있다.)

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 템플릿(`Coupon`)·발급 쿠폰(`UserCoupon`) 분리, 주문 요청은 발급 쿠폰 식별자(`userCouponId`) | 내 쿠폰 목록이 발급 쿠폰을 반환하므로 클라이언트 ID 체계가 일관되고, 소유·상태 검증이 발급 쿠폰 한 곳에 모인다. 중복 발급을 허용해도 모호함 없음. |
| B | 분리는 동일, 주문 요청은 템플릿 식별자 | 발급 API 경로(`/coupons/{couponId}/issue`)와 체계가 통일되나, 서버가 (회원, 템플릿)으로 발급 쿠폰을 역추적해야 하고 중복 발급 허용 시 어느 장을 쓸지 모호해진다. |

**사유**: 주문에서 검증·사용 처리의 대상은 발급 쿠폰이므로, 요청 식별자도 발급 쿠폰을 직접 가리키는 것이 명확하다. 과제 원문의 요청 예시 필드명은 `couponId`이나 템플릿 식별자와의 혼동을 차단하기 위해 `userCouponId`로 의도적으로 이탈한다.

### 결정 2. 만료 시각의 의미

**문제**: 템플릿의 만료 시각이 발급 가능 기한인지, 발급 쿠폰의 사용 기한인지. 내 쿠폰 목록의 `EXPIRED` 상태 판정 기준과 직결된다.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 템플릿에서는 발급 가능 기한, 발급 시 쿠폰에 복사되어 그 쿠폰의 사용 기한으로 고정 | 필드 하나로 발급·사용 수명을 모두 다루면서 만료 판정이 발급 쿠폰 안에서 닫힌다(결정 3의 스냅샷 원칙과 정합). 템플릿 수정 전후 발급분의 사용 기한이 서로 다를 수 있다. |
| B | 발급 쿠폰이 템플릿의 만료 시각을 실시간 참조 (단일 원천) | 정보 중복이 없으나 템플릿 수정·삭제와 발급 쿠폰 수명이 결합된다 — 연장이 기발급분에 소급되고, 삭제된 템플릿에 대한 판정 의존이 남는다. |
| C | 발급 가능 기한 + 발급 쿠폰별 사용 기한(발급일 + 유효기간) | 실서비스에 흔한 모델이나, 유효기간 필드가 과제 요청 예시에 없어 모델 확장이 필요하다. 본 라운드 과설계. |

**사유**: 처음에는 B(템플릿 단일 원천)를 단순성을 이유로 채택했으나, 결정 3 재논의(스냅샷 전환)와 함께 A로 정정했다. 발급 쿠폰이 발급 시점 계약으로 고정되어 템플릿과 독립적으로 만료를 판정한다.

### 결정 3. 템플릿 수정의 효력 범위와 발급 쿠폰 스냅샷

**문제**: 템플릿 수정(이름·할인 타입·할인 값·최소 주문 금액·만료 시각)이 이미 발급된 미사용 쿠폰에 어떤 효력을 가질지. 발급 쿠폰이 템플릿 정보를 어떻게 보유하는지와 한 몸인 결정.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 발급 시 템플릿 정보 전체를 발급 쿠폰에 기록(스냅샷), 수정은 이후 발급분에만 영향 | 발급 혜택이 발급 시점 계약으로 고정되어 소급이 원천 차단되고, 만료 판정·최소 주문 금액 검증·할인 계산이 발급 쿠폰 하나로 닫힌다. 같은 템플릿의 쿠폰이라도 발급 시점에 따라 조건이 다를 수 있고, 정보가 중복 보관된다. |
| B | 발급 쿠폰이 템플릿을 실시간 참조, 수정이 미사용 발급 쿠폰에 즉시 소급 | 정보 중복이 없고 모델이 가볍다. 회원이 받은 혜택이 사후에 바뀔 수 있고, 발급 쿠폰의 판정이 템플릿에 결합되어 삭제된 템플릿의 노출 정책과 충돌한다(발급 내역은 보이는데 그 머리인 템플릿 상세는 자원 부재). |
| C | 발급된 쿠폰이 하나라도 있으면 할인 조건 수정 잠금 | 소급은 막히나 필드별 수정 가능 분기와 발급 존재 확인이 더해져 명세·구현 모두 무겁다. |

**사유**: 처음에는 B(소급 수용)를 단순성을 이유로 채택했으나, 삭제된 템플릿을 둘러싼 관리자 조회 정합성이 깨지는 문제가 드러나 재논의했다. 주문 항목이 상품 정보를 스냅샷하는 기존 원리(volume-2 결정 5)와 동일하게 A로 정정 — 발급 쿠폰이 자족적이 되면서 수정 소급·삭제 영향·만료 판정·관리자 노출 정책이 한 번에 정리된다.

### 결정 4. 템플릿 삭제와 발급 쿠폰의 수명

**문제**: 템플릿이 삭제될 때 이미 발급된 미사용 쿠폰을 어떻게 처리할지. "더 이상 발급하지 않는 쿠폰"과 "존재하지 않는 쿠폰"을 분리할지 합칠지의 결정.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 삭제 = 신규 발급 중단만. 발급 쿠폰은 자신의 만료 시각까지 정상 사용 가능 | 회원에게 노출된 혜택이 보호되고 내 쿠폰 목록 표시도 기존 상태 체계 그대로라 단순하다. 발급 쿠폰이 스냅샷으로 자족적이라(결정 3 참조) 별도 장치 없이 성립하며, 삭제가 발급 쿠폰과 독립적이어서 구현 순서 제약도 없다. |
| B | 삭제 시 미사용 발급 쿠폰도 사용 불가 (cascade 무효화) | 브랜드 삭제 → 상품 cascade(volume-2 BRD-6)와 결이 같으나, 회원 혜택을 박탈하고 내 쿠폰 목록의 표시 상태(만료도 사용도 아님)가 모호해져 상태 체계 확장이 필요하다. |
| C | 발급된 쿠폰이 있으면 삭제 거부 | 정합성은 강하나 한 번이라도 발급되면 영구 삭제 불가로 운영이 경직되고, 멱등 삭제(volume-2 결정 6) 정책과도 충돌한다. |

**사유**: 쿠폰은 회원에게 약속된 혜택이라는 도메인 특성상, 운영 중단(발급 중단)과 발급분 무효화는 다른 행위다. 가장 단순하면서 회원 신뢰를 보존하는 A를 택한다. 긴급히 발급분까지 막아야 하는 시나리오(프로모션 사고 등)는 본 라운드 범위 밖으로 두고, 필요해지면 발급 쿠폰 회수(무효화) 기능을 별도 스토리로 도입한다.

### 결정 5. 동일 템플릿 중복 발급 정책

**문제**: 한 회원이 같은 템플릿에서 쿠폰을 여러 장 발급받을 수 있는지. 발급 동시성 이슈의 존재 여부까지 파생되는 결정.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 1인 1매 — 발급 이력 기준, 사용 후에도 재발급 불가 | 일반적인 쿠폰 서비스 관행과 일치한다. 중복 판정·응답 정책이 명세에 추가되고, 동일 회원의 동시 발급 요청에서 한 장만 남기는 동시성 보장이 필요해진다. |
| B | 무제한 허용 | 발급이 경합 없는 단순 생성이라 명세·구현 모두 가볍다. 같은 쿠폰을 무한정 쌓을 수 있어 할인 혜택의 도메인 의미가 어색해진다. |

**사유**: 쿠폰은 회원당 한 번의 혜택이라는 도메인 의미를 따른다. 재발급 요청은 자원 충돌로 명시 응답한다 — 좋아요 등록(volume-2 LIK-1)의 멱등 정상 응답과 의도된 차이로, 좋아요는 상태 토글이지만 발급은 혜택 수령 행위라 "이미 받은 쿠폰"임을 클라이언트가 구분해 안내할 수 있어야 한다.

### 결정 6. 쿠폰 적용 주문의 금액 스냅샷 범위

**문제**: 쿠폰을 적용한 주문에서 금액 정보를 어디까지 기록할지. 주문 시점의 할인 거래를 사후에 재현·추적할 수 있어야 한다.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 원 주문 금액·할인 금액·최종 결제 금액 3분리 + 적용 쿠폰 식별자 | 정산·CS·환불에서 "얼마짜리를 얼마 할인해 얼마 냈는지"와 "어느 쿠폰이 쓰였는지"가 모두 추적된다. 필드가 늘고, 세 금액 간 정합(`최종 = 원금 − 할인`)을 저장 시 보장해야 한다. |
| B | 할인 금액·최종 결제 금액만 (원금액은 항목 합계로 재계산) | 저장이 가볍다. 적용 쿠폰을 남기지 않으면 어느 쿠폰이 쓰였는지 추적 불가, 항목 단가가 스냅샷이라 재계산은 가능하나 도출 부담. |
| C | 최종 결제 금액만 | 가장 가볍다. 할인 내역 자체가 소실되어 영수증·정산에서 할인 사실을 복원할 수 없다. |

**사유**: 과제 원문이 "쿠폰 적용 전 금액·할인 금액·최종 결제 금액 모두 포함"을 명시한다. 여기에 적용 쿠폰 식별자를 더해, 어느 발급 쿠폰이 이 주문에 묶였는지를 보존한다 — 할인액만 남기면 정산·CS 추적이 끊긴다. 주문 항목이 상품 정보를 스냅샷하는 원리(volume-2 결정 5)의 금액 버전이다.

### 결정 7. 쿠폰 검증 실패의 응답 어휘

**문제**: 주문 시 쿠폰 검증이 실패하는 여러 경우를 "자원 부재"와 "자원 충돌" 중 무엇으로 응답할지.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 부재·타인 소유 → 자원 부재 / 사용됨·만료·최소금액 미달 → 자원 충돌 | 존재 추론 차단과 상태 불일치를 의미로 가른다. 두 갈래 분기 기준을 명세·구현이 함께 지켜야 한다. |
| B | 모든 쿠폰 검증 실패 → 자원 충돌 | 단일 규칙이라 단순하다. 타인 소유 쿠폰 식별자를 넣어보면 "그 쿠폰이 존재하긴 한다"는 사실이 응답 차이로 새어, 발급 쿠폰 식별자 enumeration이 가능해진다. |
| C | 모든 쿠폰 검증 실패 → 입력 검증 실패 | 클라이언트가 요청 단계 오류로 인지하기 쉽다. 재고·상태 같은 서버 측 상태 충돌을 입력 오류로 뭉뚱그려 의미가 흐려진다. |

**사유**: 타인 소유·부재 쿠폰을 자원 부재로 통일하는 것은 본인 것이 아닌 발급 쿠폰의 존재 여부가 응답으로 추론되지 않도록 하기 위함이다 — 타인의 주문 접근을 자원 부재로 응답하는 volume-2 ORD-4 패턴과 대칭이다. 사용됨·만료·최소금액 미달은 쿠폰이 본인 것임이 전제된 상태 불일치이므로 자원 충돌로 구분한다.

### 결정 8. 할인 금액 계산 규칙

**문제**: 할인 금액 계산에서 갈리는 세 지점 — 최소 주문 금액의 판정 기준, 정률 끝수 처리, 정액 상한.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 최소금액은 할인 전 금액 기준 / 정률은 내림 / 정액은 주문 금액으로 캡(최종 0 이상) | 쿠폰의 일반 관행("N원 이상 구매 시 할인")과 일치하고, 내림은 매출 손해가 없으며, 캡으로 음수 결제가 원천 차단된다. 정률 내림으로 고객 체감 할인액이 미세하게 줄 수 있다. |
| B | 최소금액은 할인 후 금액 기준 | 할인을 알아야 기준을 따지는데 기준을 통과해야 할인하는 순환이라 비직관적이고 실서비스에 드물다. |
| C | 정률 반올림 / 정액 상한 없음 | 반올림은 의도치 않게 할인액이 올라 매출 정합성이 흐려지고, 상한이 없으면 최종 결제 금액이 음수가 될 수 있어 별도 0 하한이 또 필요하다. |

**사유**: 세 규칙 모두 "운영자 매출을 해치지 않으면서 도메인 직관에 맞는 쪽"으로 일관 채택. 최소금액은 할인 전 금액 기준이라야 순환이 없고, 정률 내림과 정액 캡은 각각 매출 과다 할인과 음수 결제를 막는다. 세 규칙이 한 계산 흐름 안에서 함께 작동하므로 하나의 결정으로 묶는다.

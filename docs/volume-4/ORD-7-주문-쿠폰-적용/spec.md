# Spec: ORD-7 주문 쿠폰 적용

**소스**: `docs/volume-4/01-requirements.md` — ORD-7
**작성일**: 2026-06-10
**상태**: Draft

## 시나리오 요약

로그인한 회원이 기존 주문 생성(`POST /api/v1/orders`, ORD-1·2)에 보유한 발급 쿠폰 한 장(`userCouponId`, 선택)을 적용해 할인받는다. 요청에 발급 쿠폰 식별자가 추가되며, 재고 차감 뒤 쿠폰 검증·할인 계산·사용 처리 단계가 주문 흐름에 끼어든다. 쿠폰이 본인 소유이며 사용 가능(미사용·미만료)하고 할인 전 주문 금액이 최소 주문 금액 이상이면, 타입별로 할인 금액을 계산해 최종 결제 금액을 산출하고, 쿠폰을 사용 완료(`USED`)로 전이한 뒤 주문·주문 항목·금액 스냅샷(원 주문 금액·할인 금액·최종 결제 금액·적용 쿠폰 식별자)을 같은 트랜잭션 안에서 저장한다. 어느 단계라도 실패하면 재고·쿠폰을 포함해 전체가 처음 상태로 되돌아간다. 쿠폰을 적용하지 않은 주문도 동일한 스냅샷 구조(할인 0, 적용 쿠폰 없음)를 가진다. 이 변경으로 기존 주문의 금액 모델이 단일 총액(`totalPrice`)에서 세 금액으로 분리된다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 회원 인증 + 재고 충분한 상품 + 본인 소유·사용 가능 발급 쿠폰 + 할인 전 금액이 최소 주문 금액 이상, **When** `userCouponId`를 포함해 주문 요청, **Then** 재고가 차감되고 쿠폰이 `USED`로 전이되며, 원 주문 금액·할인 금액·최종 결제 금액·적용 쿠폰 식별자·항목 스냅샷이 저장·반환된다(201).
2. **Given** 회원 인증 + 재고 충분한 상품, **When** `userCouponId` 없이 주문 요청, **Then** 할인 금액 0·적용 쿠폰 없음으로 동일 스냅샷 구조의 주문이 생성된다(201, 원 주문 금액 = 최종 결제 금액).
3. **Given** 정액(`FIXED`) 쿠폰 적용 + 할인 값이 할인 전 금액보다 큼, **When** 주문 요청, **Then** 할인 금액은 `min(할인 값, 할인 전 금액)`으로 캡되어 최종 결제 금액이 0 이상이 된다.
4. **Given** 정률(`RATE`) 쿠폰 적용, **When** 주문 요청, **Then** 할인 금액은 `할인 전 금액 × 할인 값 ÷ 100`을 소수점 이하 내림한 정수가 된다.

### Exception Flow
1. **Given** 회원 인증 실패, **When** 주문 요청, **Then** 401 UNAUTHORIZED.
2. **Given** 어느 한 항목의 상품이 부재·삭제됨, **When** 주문 요청, **Then** 404 NOT_FOUND, 차감된 재고는 원복(ORD-2 승계).
3. **Given** 어느 한 항목의 재고가 요청 수량 미만, **When** 주문 요청, **Then** 409 CONFLICT, 차감된 재고는 원복(ORD-2 승계).
4. **Given** 지정한 발급 쿠폰이 부재하거나 본인 소유가 아님, **When** 주문 요청, **Then** 404 NOT_FOUND, 차감된 재고는 원복(결정 7).
5. **Given** 지정한 발급 쿠폰이 이미 사용됐거나 만료됨, **When** 주문 요청, **Then** 409 CONFLICT, 차감된 재고는 원복.
6. **Given** 할인 전 주문 금액이 쿠폰의 최소 주문 금액 미만, **When** 주문 요청, **Then** 409 CONFLICT, 차감된 재고는 원복.

### 비즈니스 규칙
- 쿠폰은 주문 1건당 1장만 적용한다.
- 최소 주문 금액 충족 여부는 **할인 적용 전 주문 금액**(항목 `단가 × 수량` 합계)을 기준으로 판정한다(결정 8).
- 정액 쿠폰의 할인 금액은 `min(할인 값, 할인 전 금액)`, 정률 쿠폰은 `할인 전 금액 × 할인 값 ÷ 100`의 내림(결정 8). 최종 결제 금액 = `할인 전 금액 − 할인 금액`(0 이상).
- 주문 스냅샷에 원 주문 금액·할인 금액·최종 결제 금액·적용 쿠폰 식별자를 기록한다(결정 6). 쿠폰 미적용 주문도 같은 구조이며 할인 금액 0·적용 쿠폰 식별자 없음.
- 쿠폰 사용 완료 전이는 주문 성공과 같은 트랜잭션 안에서 이뤄진다. 어느 단계라도 실패하면 쿠폰은 다시 미사용으로 되돌아간다.
- 부재·타인 소유 쿠폰은 자원 부재(404), 사용됨·만료·최소금액 미달은 자원 충돌(409)로 응답한다(결정 7 — 타인 쿠폰 존재 여부 추론 차단).

## 엣지 케이스
- 쿠폰 미적용(할인 0) / 적용(정액·정률) / 정액 할인 값 ≥ 주문 금액(최종 0) / 정률 내림 경계(나누어떨어지지 않는 금액).
- 최소 주문 금액 경계: 할인 전 금액 == 최소금액(통과) / 미달(409).
- 쿠폰 상태: 사용 가능(통과) / 이미 사용(409) / 만료(409) / 타인 소유(404) / 부재(404).
- 원자성: 재고 부족·상품 부재로 실패 시 재고 원복 + 쿠폰 미사용 유지(쿠폰 USED 미전이).
- 권한 경계: 회원 인증 헤더 없으면 401.

## 기능 요구사항
- **FR-001**: 시스템은 회원 인증 통과 요청만 허용한다(실패 401).
- **FR-002**: 시스템은 모든 항목 상품의 존재·재고를 확인하고 상품 식별자 순으로 재고를 차감한다(ORD-2 흐름 승계).
- **FR-003**: 쿠폰이 지정되면 발급 쿠폰이 존재하고 본인 소유이며 사용 가능(미사용·미만료)인지 검증한다(부재/타인 404, 사용됨/만료 409).
- **FR-004**: 시스템은 할인 전 주문 금액이 쿠폰 최소 주문 금액 이상인지 검증한다(미달 409).
- **FR-005**: 시스템은 할인 타입에 따라 할인 금액을 계산하고 최종 결제 금액을 산출한다(정액 캡·정률 내림, 결정 8).
- **FR-006**: 시스템은 적용된 쿠폰을 `USED`로 전이한다.
- **FR-007**: 시스템은 주문·주문 항목·금액 스냅샷(원 주문 금액·할인 금액·최종 결제 금액·적용 쿠폰 식별자)을 같은 트랜잭션에서 저장하고, 어느 단계라도 실패하면 전체를 롤백한다.
- **FR-008**: 시스템은 생성된 주문의 식별자·상태·주문 시각·원 주문 금액·할인 금액·최종 결제 금액·항목 스냅샷을 반환한다.

> **설계 출처**: 본 흐름은 `02-sequence-diagrams.md`의 ORD-7 다이어그램을 따른다(레이어·메서드 호출 연쇄). 아래 메서드·레포지토리 시그니처는 그 다이어그램과 정합한다.

## 관련 엔티티
- **Order** (기존, 변경): 단일 총액(`totalPrice`)을 원 주문 금액·할인 금액·최종 결제 금액 세 금액으로 분리하고 적용 발급 쿠폰을 `userCouponId`(nullable)로 참조한다(결정 6). 정적 팩토리가 세 금액과 `userCouponId`를 스냅샷한다. 쿠폰 미적용 시 응용 계층이 할인 0·`userCouponId` 없이 호출. *(팩토리 시그니처는 02 다이어그램 `of(items, originalAmount, discountAmount, finalAmount, userCouponId)`와 03 클래스 다이어그램 `create(userId, items, discountAmount, userCouponId)`가 어긋남 — analyze에서 합의, 아래 관련 결정 참조.)*
- **UserCoupon** (기존, 메서드 추가): `isAvailable(now)`(미사용·미만료 질의 — `getStatus` 기반, 02 다이어그램의 사용 가능 분기), `calculateDiscount(orderAmount)`(최소 주문 금액 검증 통합 — 할인 전 금액이 `minOrderAmount` 미달 시 CONFLICT, 통과 시 `DiscountType.calculate`에 위임해 할인액 산출), `use(now)`(사용 완료 전이 — `usedAt` 기록). 소유 검증은 `UserCoupon` 메서드가 아니라 레포지토리 조회로 처리한다(아래). CPN-6에서 미뤘던 메서드를 이번에 추가한다.
- **DiscountType** (기존, 메서드 추가): `calculate(orderAmount, value)` — 정액 `min(value, orderAmount)`, 정률 `orderAmount × value ÷ 100` 내림(결정 8). `validate`와 같은 template method 다형 디스패치.
- **UserCouponRepository** (메서드 추가): `getActiveByIdAndUserId(userCouponId, userId)` — 본인 소유 발급 쿠폰 단건 조회, 부재·타인 소유면 NOT_FOUND(결정 7 — 부재와 타인 소유를 한 조회로 묶어 존재 추론 차단, 기존 `OrderRepository.getActiveByIdAndUserId` 패턴 계승).
- **OrderRepository** (기존): 주문·항목·금액 스냅샷 저장(ORD-2 재사용).
- **재사용**: `@LoginUser AuthenticatedUser`, `DateTimeUtil`(주문 시각·쿠폰 만료 기준·`usedAt`), 재고 원자 차감·정렬·트랜잭션(ORD-2), `ErrorType`(UNAUTHENTICATED·NOT_FOUND·CONFLICT).

## 관련 결정
- **결정 6 (금액 스냅샷 3분리 + 적용 쿠폰 식별자)**: 원 주문 금액·할인 금액·최종 결제 금액 + `userCouponId`를 주문에 기록. 단일 `totalPrice` 대체.
- **결정 7 (응답 어휘)**: 부재·타인 소유 → 404, 사용됨·만료·최소금액 미달 → 409.
- **결정 8 (할인 계산)**: 최소금액은 할인 전 금액 기준, 정률 내림, 정액 주문 금액 캡.
- **소유 검증 방식 (02 다이어그램)**: 부재·타인 소유를 `UserCouponRepository.getActiveByIdAndUserId` 한 조회로 묶어 404. 별도 `isOwnedBy` 도메인 메서드를 두지 않는다(기존 주문 소유 검증과 동일 패턴).
- **설계 문서 정합 (analyze 보고)**: Order 정적 팩토리 시그니처가 `02-sequence-diagrams.md`(`of(items, originalAmount, discountAmount, finalAmount, userCouponId)` — 세 금액 전달)와 `03-class-diagram.md`(`create(userId, items, discountAmount, userCouponId)` — 항목·할인으로 원금·최종 산출)에서 어긋난다. analyze에서 어느 쪽으로 통일할지 합의한다(세 금액 정합을 팩토리가 자기 책임으로 보장하는 `create` 쪽이 후보).
- **트랜잭션 원자성(롤백)**: 동시성과 별개로 본 cycle에서 일반 테스트로 검증(재고·쿠폰 USED 전이 포함 전체 롤백).

## 성공 기준 / 범위 밖
- **성공**: 위 모든 수용 시나리오·테스트 계획 green. `POST /api/v1/orders`가 쿠폰 검증·할인 계산·사용 전이·세 금액 스냅샷·전체 롤백을 명세대로 처리.
- **기존 계약 변경 (구현 영향)**: 주문 응답·저장 모델의 `totalPrice`가 `originalAmount`·`discountAmount`·`finalAmount`로 대체된다. 기존 ORD-1·2 주문 생성/조회 테스트·E2E·픽스처·`OrderInfo`·`OrderV1Dto`도 동반 수정 대상(plan에서 변경 범위 확정).
- **범위 밖**: 동시성(같은 쿠폰 동시 주문 1회만·재고 음수 방지 — 단계 3~4), 다중 쿠폰 적용, 쿠폰 외 결제수단, 주문 취소·환불에 따른 쿠폰 복원.

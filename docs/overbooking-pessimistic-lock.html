재고 오버부킹 문제와
비관적 락으로 해결하기

2024 · Java 17 · Spring Boot 3 · JPA

문제 발견

주문 생성 로직을 리뷰하던 중 동시 요청 시 재고가 실제보다 더 많이 차감되는 오버부킹 위험을 발견했다. 재고 확인과 차감이 두 단계로 분리되어 있는 것이 원인이었다. 

// OrderService.createOrder()

// Step 1. 재고 확인
if (!product.hasEnoughStock(item.quantity())) {
    throw new IllegalArgumentException("재고가 부족합니다.");
}

// Step 2. 재고 차감 (확인과 차감 사이에 간격 존재)
product.decreaseStock(item.quantity());
productRepository.save(product);

원인 분석 — Race Condition

두 요청이 동시에 들어오면 읽기 → 검증 → 쓰기 사이의 간격에서 Race Condition이 발생한다.

시점 스레드 A 스레드 B DB stock
T1stock=1 조회stock=1 조회1
T21 >= 1 → 통과1 >= 1 → 통과1
T3stock=0 저장stock=0 저장 (A 덮어씀)0
결과주문 성공주문 성공오버부킹 발생
🚨
핵심 문제 두 트랜잭션이 모두 stock=1을 읽고, 모두 검증을 통과한 후, 모두 stock=0을 저장한다. 재고는 1개인데 주문은 2개가 성공하는 오버부킹이 발생한다.

해결 방법 비교

방법 원리 장점 단점
비관적 락 SELECT FOR UPDATE 정합성 확실 처리량 저하
낙관적 락 @Version 충돌 감지 처리량 높음 재시도 로직 필요
원자적 UPDATE DB 단 처리 구현 단순 복잡한 로직 적용 어려움
💡
재고 시스템은 정합성이 최우선이므로 비관적 락을 선택했다.

비관적 락 구현

비관적 락은 조회 시점에 행을 선점한다. 트랜잭션 A가 락을 잡고 있는 동안 트랜잭션 B는 커밋될 때까지 대기한다.

→ 스레드 A: SELECT ... FOR UPDATE 락 획득
→ 스레드 B: SELECT ... FOR UPDATE 대기 중...
→ 스레드 A: stock 차감 → COMMIT → 락 해제
→ 스레드 B: 이제 읽음 → stock=0 → 재고 부족 예외

1. 도메인 인터페이스 — ProductRepository

도메인이 인터페이스를 소유하고 인프라가 구현하는 DIP 원칙을 유지한다.

변경 전
Optional<Product> findById(Long id);
변경 후
Optional<Product> findById(Long id);

Optional<Product> findByIdWithLock(Long id);

2. JPA 구현 — ProductJpaRepository

@Lock(PESSIMISTIC_WRITE) 하나로 Spring Data JPA가 자동으로 SELECT ... FOR UPDATE를 실행해준다.

변경 전
public interface ProductJpaRepository
    extends JpaRepository<Product, Long> {

    List<Product> findAllByBrandId(Long brandId);
}
변경 후
public interface ProductJpaRepository
    extends JpaRepository<Product, Long> {

    List<Product> findAllByBrandId(Long brandId);

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    Optional<Product> findWithLockById(Long id);
}

3. 구현체 — ProductRepositoryImpl

변경 전
@Override
public Optional<Product> findById(Long id) {
    return jpaRepository.findById(id);
}
변경 후
@Override
public Optional<Product> findById(Long id) {
    return jpaRepository.findById(id);
}

@Override
public Optional<Product> findByIdWithLock(Long id) {
    return jpaRepository.findWithLockById(id);
}

4. 서비스 — OrderService

변경 전
Product product =
    productRepository
        .findById(item.productId())
        .orElseThrow(...);
변경 후
Product product =
    productRepository
        .findByIdWithLock(item.productId())
        .orElseThrow(...);

5. 단위 테스트 — FakeProductRepository

단위 테스트에서는 실제 DB 락이 없으므로 findById로 위임한다. 동시성 검증은 통합 테스트에서 수행한다. 

@Override
public Optional<Product> findByIdWithLock(Long id) {
    return findById(id); // 단위 테스트는 동시성 검증 대상 아님
}

주의사항

⚠️
@Transactional 필수 비관적 락은 트랜잭션 범위 안에서만 동작한다. @Transactional이 없으면 조회 즉시 락이 해제되어 의미가 없어진다. 
@Service
@Transactional  // 필수 — 트랜잭션이 커밋될 때까지 락 유지
public class OrderService {
    ...
}

# ADR-007: 주문 생성 흐름 설계

- 날짜: 2026-05-20
- 상태: 승인됨

## 결정

주문 생성 흐름을 아래 순서로 한다.

```
1. 상품 조회   — 존재 확인 + 스냅샷 데이터 수집 (BRAND, INVENTORY JOIN 포함)
2. 재고 확인   — fast fail (락 없음, product.quantity 직접 비교)
3. 주문 생성   — OrderEntity + OrderItemEntity INSERT (@Transactional 시작)
4. 재고 차감   — SELECT ... FOR UPDATE → productInventory.deduct(quantity)
               — 실패 시 @Transactional 전체 롤백 (주문 생성 포함)
```

## 근거

### 고려한 대안

#### Option 1. 재고 확인 → 재고 차감 → 주문 생성

재고를 먼저 차감하고, 성공 시에만 주문을 생성하는 방식이다.

```
1. 상품 조회
2. 재고 확인 (fast fail)
3. 재고 차감 (FOR UPDATE)
4. 주문 생성
```

- **장점**: 재고 차감 실패 시 주문 생성 자체를 시도하지 않아 롤백 범위가 줄어든다.
- **단점**: 스냅샷 데이터(상품명, 가격)를 1번에서 이미 수집했음에도 주문 생성이 마지막에 위치해 흐름이 부자연스럽다. 실제로 재고 차감 실패와 주문 생성 실패를 트랜잭션 롤백으로 처리하는 것과 결과적으로 차이가 없다.

---

#### Option 2. 재고 확인 → 주문 생성 → 재고 차감 (채택)

주문을 먼저 생성한 뒤 재고를 차감하고, 실패 시 전체를 롤백하는 방식이다. Shopify, Amazon, WooCommerce, 우아한형제들 배민 선물하기 등 국내외 이커머스의 일반적인 패턴이다.

```
1. 상품 조회 (트랜잭션 외부, 읽기 전용)
2. 재고 확인 — product.quantity < 요청수량이면 즉시 400
3. @Transactional 시작
   주문 생성 → 재고 차감 (FOR UPDATE) → Commit
   재고 부족 시 전체 Rollback
```

- **장점**: 스냅샷 수집 → 주문 생성 → 재고 차감의 흐름이 자연스럽고 코드 가독성이 높다. 트랜잭션 경계가 명확하다. 재고 확인(2번)이 명백한 재고 부족을 사전 차단하여 불필요한 트랜잭션 시작을 줄인다. 실제 동시성 보장은 FOR UPDATE 락(4번)이 담당하므로 정합성도 확보된다.
- **단점**: 재고 차감 실패 시 주문 생성까지 롤백된다. 트랜잭션 롤백 범위가 Option 1보다 넓다.

## 향후 고려사항

결제 기능이 추가될 경우 아래 방식으로 전환을 고려한다.

- **재고 예약(Reservation) 도입**: 결제 대기 중 재고를 soft hold → 결제 확정 후 최종 차감
- **락 전략 확장**: 트래픽 증가 시 낙관적 락(`@Version`) 또는 Redis 분산 락(Redisson)으로 전환

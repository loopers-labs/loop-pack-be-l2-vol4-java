# ADR-032: 쿠폰 적용 시 주문 생성 흐름 변경

- 날짜: 2026-06-09
- 상태: 승인됨
- 관련: [ADR-007](./007-order-creation-flow.md), [ADR-031](./031-coupon-pessimistic-lock.md)

## 결정

쿠폰을 적용하는 주문 생성의 트랜잭션 내부 순서를 아래와 같이 변경한다.

```
트랜잭션 외부 (읽기 전용)
  1. 상품 조회 (INVENTORY JOIN)
  2. 재고 fast-fail (product.quantity < 요청수량이면 400)

@Transactional 시작
  3. 쿠폰 유효성 검증 및 사용 처리 (PESSIMISTIC_WRITE 락)
     — 소유권·상태 검증은 락 획득 후 단일 지점에서만 수행
  4. 재고 차감 (SELECT ... FOR UPDATE, productId 오름차순)
  5. 주문 엔티티 생성 및 저장 (ORDER INSERT)
Commit / 실패 시 전체 Rollback
```

## 근거

### 기존 흐름의 문제 (ADR-007 Option 2)

```
@Transactional
  1. 주문 생성 (INSERT)
  2. 재고 차감 (FOR UPDATE)
```

쿠폰이 도입되면 트랜잭션 안에 세 작업이 묶인다. 기존 "주문 생성 먼저" 방식을 그대로 유지하면:

```
@Transactional
  1. 주문 생성 (INSERT)
  2. 쿠폰 사용 (PESSIMISTIC_WRITE)
  3. 재고 차감 (FOR UPDATE)
```

- 쿠폰 사용 실패 또는 재고 부족 시, **이미 INSERT된 주문**을 롤백해야 한다.
- 주문 INSERT는 성공 가능성이 가장 높은 단순 쓰기 작업이다. 이를 먼저 수행하면 실패 확률이 높은 쿠폰/재고 실패 시 불필요한 롤백이 발생한다.

### 새 흐름의 장점

#### 1. 실패 확률 순으로 정렬 — 빠른 실패 (Fail Fast)

| 순서 | 작업 | 실패 원인 |
|---|---|---|
| 4 | 쿠폰 사용 | 이미 USED, EXPIRED, 타인 소유 — 동시 요청 시 가장 빈번한 실패 지점 |
| 5 | 재고 차감 | 재고 부족 — 트래픽 집중 시 빈번한 실패 |
| 6 | 주문 생성 | 거의 실패하지 않음 — 순수 INSERT |

실패 확률이 높은 쿠폰 사용을 먼저 수행하면, 재고 차감과 주문 INSERT를 시도조차 하지 않고 빠르게 실패할 수 있다.

#### 2. 주문은 "성공의 기록"

주문 엔티티는 쿠폰 할인과 재고 차감이 모두 확정된 이후에만 의미가 있다. 선행 조건이 충족된 뒤 마지막에 저장하는 것이 의미적으로 자연스럽다.

#### 3. 락 획득 순서의 일관성 (데드락 방지)

다중 트랜잭션이 동시에 진입할 때:
- 쿠폰 락 → 재고 락 순으로 획득을 고정한다.
- 역순(재고 락 → 쿠폰 락)이 혼재하면 데드락 위험이 있다.
- 같은 쿠폰에 대한 락은 단일 row이므로 순서가 자연스럽게 직렬화된다.

### 고려한 대안

#### 쿠폰 사용을 트랜잭션 외부에서 처리

쿠폰 사용(`AVAILABLE → USED`)을 별도 트랜잭션에서 먼저 처리하고, 이후 재고 차감 + 주문 생성을 묶는 방식.

- **장점**: 트랜잭션 크기 축소.
- **단점**: 재고 차감 실패 시 쿠폰 상태를 별도로 복구(보상 트랜잭션)해야 한다. 분산 트랜잭션 복잡도가 발생하며 현재 요구사항 범위를 초과한다.

→ 현 시점에서는 단일 @Transactional 범위 유지가 적합하다.

## 향후 고려사항

- 트래픽 증가 시 쿠폰 락을 Redis 분산 락(Redisson)으로 전환 가능.
- 결제 기능 도입 시 쿠폰 사용 → 결제 → 재고 차감 순으로 확장을 검토한다.

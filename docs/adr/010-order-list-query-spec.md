# ADR-010: 내 주문 목록 조회 — 날짜 필터 + 페이지네이션

- 날짜: 2026-05-21
- 상태: 승인됨

## 결정

`GET /api/v1/orders` 요청은 `startAt` / `endAt` 날짜 범위 필터와 `page` / `size` 페이지네이션을 함께 사용한다.

```
GET /api/v1/orders?startAt=2026-05-01&endAt=2026-05-31&page=0&size=20
```

- `page` 미전달 시 기본값 `0`
- `size` 미전달 시 기본값 `20`
- 응답: `Page<OrderInfo>` 구조 (`content / page / size / totalElements`)

## 근거

유저의 주문 목록은 개인 범위이므로 전체 데이터가 방대하지 않다. 그러나 장기 사용자의 경우 누적 주문 수가 많아질 수 있으며, 날짜 범위 없이 전체를 반환하면 응답 크기가 무제한으로 커진다. 날짜 필터로 조회 범위를 제한하고, 그 안에서 페이지네이션으로 응답 크기를 제어한다.

### 고려한 대안

#### Option 1. 날짜 필터만 사용 (페이지네이션 없음)

`startAt` / `endAt`으로 범위를 좁히되 페이지네이션 없이 `List`로 반환하는 방식이다.

```
GET /api/v1/orders?startAt=2026-05-01&endAt=2026-05-31
→ List<OrderInfo> 반환
```

- **장점**: 구현이 단순하다. 클라이언트가 페이지 상태를 관리하지 않아도 된다.
- **단점**: 날짜 범위가 길 경우(예: 연간 조회) 응답 크기가 무제한으로 커질 수 있다. 응답 지연 및 메모리 이슈가 발생할 가능성이 있다.

---

#### Option 2. 페이지네이션만 사용 (날짜 필터 없음)

`page` / `size`로 전체 주문 목록을 페이지 단위로 반환하는 방식이다.

```
GET /api/v1/orders?page=0&size=20
→ Page<OrderInfo> 반환
```

- **장점**: 구현이 단순하다.
- **단점**: 기간별 조회(예: "이번 달 주문만 보기")가 불가능하다. UX상 날짜 기반 탐색이 자연스럽지 않다.

---

#### Option 3. 날짜 필터 + 페이지네이션 (채택)

`startAt` / `endAt`으로 조회 범위를 제한하고, 그 결과를 `page` / `size`로 페이지네이션하는 방식이다.

```
GET /api/v1/orders?startAt=2026-05-01&endAt=2026-05-31&page=0&size=20
→ Page<OrderInfo> 반환
```

- **장점**: 날짜 범위로 의미 있는 단위(월별, 분기별)로 탐색이 가능하다. 페이지네이션으로 응답 크기를 제어할 수 있다. 두 파라미터 모두 선택사항으로 설계하면 기본값으로 최근 전체를 페이지 단위로 조회하는 것도 허용된다.
- **단점**: 클라이언트가 날짜 파라미터와 페이지 파라미터를 함께 관리해야 한다.

## 파라미터 기본값

| 파라미터 | 기본값 | 설명 |
|---|---|---|
| `startAt` | 없음 (제한 없음) | 미전달 시 최초 주문부터 조회 |
| `endAt` | 없음 (제한 없음) | 미전달 시 현재까지 조회 |
| `page` | `0` | 첫 번째 페이지 |
| `size` | `20` | 페이지당 항목 수 |

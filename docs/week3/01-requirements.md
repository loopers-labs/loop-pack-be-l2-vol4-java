## 1. 개요

쿠폰은 사용자가 주문을 결제할 때 **할인을 받을 수 있는 권리**다.

- 운영자(Admin)는 쿠폰의 틀, 즉 **쿠폰 템플릿**을 등록한다. 템플릿은 할인 방식(정액/정률)·할인 값·최소 주문 금액·만료 시각을 정의한다.
- 사용자(User)는 템플릿으로부터 쿠폰을 **발급**받아 본인 소유의 쿠폰을 보유한다.
- 사용자는 주문 시 보유한 쿠폰 한 장을 적용해 결제 금액을 깎는다.
- 한 번 사용한 쿠폰은 **재사용할 수 없다**. 존재하지 않거나 사용할 수 없는 쿠폰으로 주문하면 주문은 실패한다.

쿠폰은 주문·결제 금액을 직접 바꾸므로, "한 장의 쿠폰이 정확히 한 번만 쓰인다"는 보장이 이 기능의 핵심이다.

## 2. 등장 인물

기존 액터 정의([week2 §2](../week2/01-requirements.md))에 쿠폰 관련 능력을 더한다.

| 액터 | 정체성 | 쿠폰 관련 할 수 있는 것 | 할 수 없는 것 |
| --- | --- | --- | --- |
| **User** | 가입한 회원 | 쿠폰 발급, 내 쿠폰 목록 조회, 주문 시 본인 쿠폰 1장 적용 | 다른 User의 쿠폰 적용·조회, 템플릿 관리 |

| **Admin** | 운영자 | 쿠폰 템플릿 등록·수정·조회·삭제, 발급 내역 조회 | 특정 사용자를 대신해 발급·사용 |

## 3. 도메인 어휘

### 3.1 핵심 개념

- **쿠폰 템플릿(Coupon)** — 운영자가 정의하는 할인의 틀. 이름·할인 방식·할인 값·최소 주문 금액·만료 시각을 가진다. 발급의 원본이며, 그 자체로는 특정 사용자에게 속하지 않는다.
- **발급 쿠폰(User Coupon)** — 템플릿으로부터 특정 사용자에게 발급된 한 장의 쿠폰. "사용자가 소유한 쿠폰"이 바로 이것이다. 사용 가능/사용 완료 상태를 가지며 재사용되지 않는다.
- **할인 방식(Coupon Type)** — 쿠폰이 금액을 깎는 방식.
  - **정액(FIXED)** — 정해진 금액만큼 깎는다. 할인 값은 "원" 단위.
  - **정률(RATE)** — 주문 금액의 정해진 비율만큼 깎는다. 할인 값은 "%" 단위.
- **할인 값(value)** — 정액이면 깎는 금액(원), 정률이면 깎는 비율(%).
- **최소 주문 금액(minOrderAmount)** — 쿠폰을 쓸 수 있는 최소 주문 금액 조건. 선택 항목이며, 지정 시 주문 금액이 이 값 미만이면 쿠폰을 적용할 수 없다.
- **만료 시각(expiredAt)** — 이 시각이 지나면 쿠폰을 더 이상 사용할 수 없다.
- **할인 금액(discountAmount)** — 쿠폰 적용으로 실제로 깎인 금액. 쿠폰 적용 전 금액을 넘지 않는다.
- **최종 결제 금액(finalAmount)** — 쿠폰 적용 전 금액에서 할인 금액을 뺀, 외부 결제 시스템에 실제로 청구되는 금액.

### 3.2 관계

- 한 템플릿(Coupon)으로부터 여러 발급 쿠폰(User Coupon)이 만들어진다.
- 한 발급 쿠폰은 정확히 한 명의 사용자에게 속한다.
- 한 발급 쿠폰은 최대 한 건의 주문에만 쓰인다. 한 주문에는 최대 한 장의 쿠폰만 적용된다.
- 발급 쿠폰은 템플릿을 가리키되, 사용 시점의 할인 결과(할인 금액)는 **주문 쪽에 별도 보존**한다(스냅샷). 사후 템플릿 변경에 흔들리지 않는다.

### 3.3 발급 쿠폰의 상태

발급 쿠폰은 사용자에게 다음 셋 중 하나의 상태로 보인다:

- **사용 가능 (AVAILABLE)** — 아직 쓰지 않았고 만료되지 않은 상태. 주문에 적용할 수 있다.
- **사용 완료 (USED)** — 어떤 주문에 이미 적용된 상태. 다시 쓸 수 없다.
- **만료 (EXPIRED)** — 사용하지 않은 채 만료 시각이 지난 상태. 쓸 수 없다.

**만료는 별도 행위가 아니라 시간에 따라 파생되는 상태**다. 즉 사용하지 않은 쿠폰은 만료 시각이 지나는 순간 자동으로 만료로 간주된다(별도의 만료 처리 작업 없이 조회 시점에 판정).

상태 전이는 한 방향이다: 사용 가능 → 사용 완료. 만료는 "사용 가능"한 쿠폰을 시간이 지나 더 못 쓰게 만드는 판정이며, 이미 사용 완료된 쿠폰의 상태를 바꾸지 않는다.

## 4. 유저 스토리

### 4.1 User (회원)

- 사용자는 쿠폰 템플릿으로부터 쿠폰을 발급받을 수 있다.
- 사용자는 본인이 보유한 쿠폰 목록을, 사용 가능/사용 완료/만료 상태와 함께 조회할 수 있다.
- 사용자는 주문할 때 보유한 사용 가능한 쿠폰 한 장을 골라 할인을 받을 수 있다.
- 사용자는 쿠폰 없이도 기존처럼 주문할 수 있다(쿠폰은 선택).
- 사용자는 주문 내역에서 쿠폰 적용 전 금액·할인 금액·최종 결제 금액을 함께 확인할 수 있다.
- 사용자는 한 번 사용한 쿠폰을 다시 쓸 수 없다.
- 사용자는 결제가 실패하면 적용했던 쿠폰을 다시 사용 가능한 상태로 돌려받는다.

### 4.2 Admin (운영자)

- 운영자는 쿠폰 템플릿을 등록할 수 있다(정액/정률 지정).
- 운영자는 쿠폰 템플릿 목록과 상세를 조회할 수 있다.
- 운영자는 쿠폰 템플릿을 수정·삭제할 수 있다.
- 운영자는 특정 쿠폰 템플릿의 발급 내역을 조회할 수 있다.

### 4.3 시스템 — 쿠폰 적용

- 시스템은 주문에 쿠폰이 지정되면, 쿠폰이 해당 사용자 소유이며 사용 가능한지 검증한다.
- 시스템은 검증을 통과한 쿠폰의 할인 금액을 계산해 최종 결제 금액을 정한다.
- 시스템은 주문이 성립하는 순간 해당 쿠폰을 사용 완료로 바꾸고, 같은 쿠폰이 동시에 두 번 사용되지 않도록 보장한다.
- 시스템은 결제가 실패하면 차감한 재고를 원복하듯, 사용 처리한 쿠폰도 사용 가능한 상태로 원복한다.

## 5. 기능 카탈로그

엔드포인트 표는 입력 명세를 그대로 옮긴다. 요청/응답 스키마 상세는 후속 설계 문서가 다룬다.

### 5.1 대고객 API

| METHOD | URI | user_required | 설명 |
| --- | --- | --- | --- |
| POST | `/api/v1/coupons/{couponId}/issue` | O | 쿠폰 발급 요청 |
| GET | `/api/v1/users/me/coupons` | O | 내 쿠폰 목록 조회 |

> 내 쿠폰 목록 조회 시 사용 가능한 쿠폰(AVAILABLE) / 사용 완료(USED) / 만료(EXPIRED) 상태를 함께 반환한다.

### 5.2 쿠폰 ADMIN API

| METHOD | URI | ldap_required | 설명 |
| --- | --- | --- | --- |
| GET | `/api-admin/v1/coupons?page=0&size=20` | O | 쿠폰 템플릿 목록 조회 |
| GET | `/api-admin/v1/coupons/{couponId}` | O | 쿠폰 템플릿 상세 조회 |
| POST | `/api-admin/v1/coupons` | O | 쿠폰 템플릿 등록 — 정액(`FIXED`)/정률(`RATE`) 타입 지정 |
| PUT | `/api-admin/v1/coupons/{couponId}` | O | 쿠폰 템플릿 수정 |
| DELETE | `/api-admin/v1/coupons/{couponId}` | O | 쿠폰 템플릿 삭제 |
| GET | `/api-admin/v1/coupons/{couponId}/issues?page=0&size=20` | O | 특정 쿠폰의 발급 내역 조회 |

쿠폰 템플릿 등록 요청 예시:

```json
{
  "name": "신규가입 10% 할인",
  "type": "RATE",          // FIXED | RATE
  "value": 10,             // 정률: 퍼센트(%), 정액: 할인 금액(원)
  "minOrderAmount": 10000, // 최소 주문 금액 조건 (선택)
  "expiredAt": "2026-12-31T23:59:59"
}
```

### 5.3 주문 API 변경

기존 주문 요청에 적용할 쿠폰 식별자를 더한다. 미적용 시 생략 가능(NULLABLE).

```json
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ],
  "couponId": 42
}
```

> 여기서 `couponId`는 **쿠폰 템플릿의 식별자**다(명세 전반에서 `couponId`는 템플릿을 가리킨다 — 발급 API 경로 `{couponId}`, Admin API `{couponId}` 모두 템플릿). 시스템은 이 템플릿에 대해 **해당 사용자가 보유한 사용 가능한 발급분 한 장을 골라** 적용한다(§7.3 발급분 선택 규칙). "발급 쿠폰"(User Coupon)은 별도 식별자를 갖되, 주문 요청에는 노출하지 않는다.

## 6. 유스케이스

기존 week2 UC-01~12에 이어 번호를 매긴다.

### UC-13. 쿠폰 발급

사용자가 쿠폰 템플릿을 골라 발급을 요청한다. 시스템은 템플릿이 존재하는지 확인하고, 해당 사용자 앞으로 사용 가능한 발급 쿠폰 한 장을 만든다.

- 존재하지 않는 템플릿으로 요청하면 발급은 실패한다.
- 이미 만료된 템플릿으로 발급 요청하면 거부된다(§9 Q2).

### UC-14. 내 쿠폰 목록 조회

사용자가 본인이 보유한 발급 쿠폰을 조회한다. 각 쿠폰은 사용 가능/사용 완료/만료 상태와 함께 반환된다. 만료 상태는 조회 시점의 시간으로 판정한다.

### UC-15. 쿠폰 템플릿 관리 (Admin)

운영자가 쿠폰 템플릿을 등록·조회·수정·삭제한다.

- 등록 시 할인 방식(정액/정률)·할인 값·만료 시각은 필수, 최소 주문 금액은 선택이다.
- 삭제는 soft delete이며, 이미 사용된 쿠폰은 주문 스냅샷으로 보존된다(§9 Q3).

### UC-16. 특정 쿠폰 발급 내역 조회 (Admin)

운영자가 특정 템플릿에서 발급된 쿠폰들의 내역(누구에게, 언제, 현재 상태)을 페이지 단위로 조회한다.

### UC-17. 쿠폰을 적용한 주문 (성공)

사용자가 쿠폰을 지정해 주문한다. 시스템은 ① 쿠폰이 본인 소유이며 사용 가능한지, 최소 주문 금액 조건을 만족하는지 검증하고, ② 할인 금액을 계산해 최종 결제 금액을 정하고, ③ 재고 차감과 함께 쿠폰을 사용 완료로 바꾼 뒤, ④ 최종 결제 금액으로 외부 결제를 진행한다. 결제까지 성공하면 주문이 확정되고, 주문 내역에는 쿠폰 적용 전 금액·할인 금액·최종 결제 금액이 모두 남는다.

### UC-18. 쿠폰을 적용한 주문 (실패 — 잘못된 쿠폰)

지정한 쿠폰이 존재하지 않거나, 이미 사용되었거나, 만료되었거나, 타 유저 소유이거나, 최소 주문 금액을 만족하지 않으면 **주문 전체가 실패한다**. 이때 재고 차감·쿠폰 사용 처리는 일어나지 않는다(또는 일어났더라도 원복된다).

### UC-19. 쿠폰을 적용한 주문 (결제 실패)

검증을 통과해 쿠폰을 사용 처리하고 재고를 차감했으나 외부 결제가 실패하면, 시스템은 차감한 재고를 원복하고 **사용 처리한 쿠폰도 사용 가능한 상태로 원복**한 뒤 주문을 실패로 마감한다.

### UC-20. 동일 쿠폰 동시 사용 시도

같은 사용자(또는 동일 쿠폰)에 대해 두 건의 주문이 거의 동시에 같은 쿠폰을 적용하려 하면, **정확히 한 건만 성공**하고 나머지는 실패한다. 한 장의 쿠폰이 두 주문에 중복 적용되는 일은 없어야 한다.

## 7. 비즈니스 규칙·정책

### 7.1 할인 계산

- **정액(FIXED)**: 할인 금액 = `min(value, 쿠폰 적용 전 금액)`. 할인이 주문 금액을 넘지 않는다(최종 결제 금액은 0 미만이 될 수 없다).
- **정률(RATE)**: 할인 금액 = `floor(쿠폰 적용 전 금액 × value ÷ 100)`. 원 단위 미만은 버린다.
- 최종 결제 금액 = `쿠폰 적용 전 금액 − 할인 금액`.
- 쿠폰 적용 전 금액 = 주문 항목 라인 합계(week2의 totalAmount). 즉 쿠폰은 배송비 등 다른 항목이 아니라 상품 금액 합계에만 작용한다.

### 7.2 적용 가능 조건 (주문 실패 사유)

다음 중 하나라도 해당하면 쿠폰 적용은 거부되고 **주문은 실패**한다:

- 지정한 쿠폰이 존재하지 않음
- 이미 사용 완료(USED)된 쿠폰
- 만료(만료 시각 경과)된 쿠폰
- 타 유저 소유 쿠폰 — "찾을 수 없음"으로 응대(존재/소유 노출 방지, §2 격리 원칙)
- 최소 주문 금액(minOrderAmount) 미달

### 7.3 1주문 1쿠폰 · 발급분 선택

- 한 주문에는 쿠폰을 최대 한 장만 적용한다. 쿠폰 미적용 주문도 정상 동작한다(couponId 생략).
- 주문 요청의 `couponId`는 템플릿 식별자다. 사용자가 같은 템플릿의 발급분을 여러 장 보유할 수 있으므로(§7.8, 중복 발급 허용), 시스템은 그중 **가장 먼저 발급된 사용 가능한 발급분 한 장**을 선택해 사용한다. 사용 가능한 발급분이 없으면(전부 USED/EXPIRED이거나 미보유) 주문은 실패한다(§7.2).

### 7.4 재사용 불가 · 동시성

- 주문이 성립하면 적용한 쿠폰은 **즉시 사용 완료(USED)**로 바뀐다.
- 한 장의 쿠폰은 평생 한 번만 사용된다. 동시에 들어온 두 주문이 같은 쿠폰을 쓰려 해도 **정확히 한 건만** 성공한다(UC-20).
- 이 동시성 보장 메커니즘(낙관적 락 / 비관적 락)의 트레이드오프는 본 명세의 관심사가 아니라 [`03-class-diagram.md`](./03-class-diagram.md) 및 구현 단계에서 결정·비교한다.

### 7.5 만료의 파생 판정

쿠폰의 만료는 저장된 상태가 아니라 **조회·사용 시점에 만료 시각과 현재 시각을 비교해 판정**한다. 별도의 배치성 만료 처리 작업을 두지 않는다(week2 §7.8 UTC 저장 원칙을 따른다).

### 7.6 결제 실패 시 원복

외부 결제가 실패하면 week2 §7.6의 재고 원복과 동일한 원칙으로 **사용 처리한 쿠폰을 사용 가능 상태로 원복**한다. 재고 원복과 쿠폰 원복은 같은 결제 실패 처리의 일부다. 결제 응답이 지연(TIMEOUT)되어 주문이 결제 대기로 남는 경우, 쿠폰은 사용 처리된 채로 둔다(추후 재확인으로 최종 확정).

### 7.7 주문 스냅샷

주문에는 쿠폰 적용 결과를 다음 세 값으로 모두 보존한다:

- **쿠폰 적용 전 금액** — 할인을 적용하기 전, 주문 항목 라인 합계
- **할인 금액** — 쿠폰으로 실제 깎인 금액(미적용 시 0)
- **최종 결제 금액** — 외부 결제에 청구된 금액

이 값들은 사후의 템플릿 변경·삭제에 흔들리지 않는다(주문 항목 스냅샷과 같은 원칙).

### 7.8 발급 정책

- 발급은 사용자 본인 앞으로만 이루어진다(인증된 사용자 = 소유자).
- 동일 사용자의 동일 템플릿 중복 발급은 허용된다(§9 Q4). 적용 시 §7.3 선택 규칙으로 한 장을 고른다.

## 9. 결정 사항 (확정)

> 초안의 미해결 질문은 모두 확정되었다(2026-06-03). 아래 결정을 02~04 설계의 전제로 삼는다.

- **Q1. 주문 요청의 `couponId` 의미** → **템플릿 식별자**로 확정. 시스템이 해당 템플릿의 "내 사용 가능한 발급분 한 장"(가장 먼저 발급된 것)을 골라 적용한다(§5.3, §7.3).
- **Q2. 만료된 템플릿의 발급** → **거부**. 만료 시각이 지난 템플릿에 대한 발급 요청은 실패한다.
- **Q3. 발급분이 있는 템플릿의 수정·삭제** → 템플릿은 **soft delete**(기존 Brand/Product 컨벤션). 이미 사용된 쿠폰은 주문 스냅샷(적용 전/할인/최종 금액)으로 **불변** 보존된다. soft delete된 템플릿의 미사용 발급분은 더 이상 적용 불가(비활성 템플릿).
- **Q4. 동일 템플릿 중복 발급** → **제약 없음**. 한 사용자가 같은 템플릿을 여러 번 발급받을 수 있다(§7.3 선택 규칙으로 처리).
- **Q5. 최소 주문 금액 기준** → **쿠폰 적용 전 금액**(상품 라인 합계) 기준(§7.1).
- **Q6. 정률 반올림 방향** → **버림(floor)**. 원 단위 미만은 버린다(§7.1).

### 협력 흐름 원칙
- **애그리거트 및 연관관계 매핑**:
    - `Order` ↔ `OrderItem` (1:N), `Product` ↔ `Stock` (1:1)과 같이 도메인 응집도가 높고 생명주기를 함께하는 경우에는 양방향 매핑을 사용하여 객체 탐색을 용이하게 한다.
    - 애그리거트 루트(부모)가 자식의 생명주기를 전적으로 관리하는 경우, 자식 엔티티(`Stock`, `OrderItem`)는 별도의 타임스탬프(`BaseTimeEntity`)를 가지지 않고 순수 필드 위주로 구성하여 경량화한다.
    - `Brand` ↔ `Product`와 같이 데이터 양이 많거나 성능 이슈가 예상되는 관계는 `Product`에서 `brandId`를 참조하는 등 단방향 또는 ID 참조를 지향한다.
- **영속성 전이(Cascade) 전략**:
    - **주문 애그리거트**: `Order` → `OrderItem`은 생성 시의 편의성을 위해 `CascadeType.PERSIST`만 적용한다. 주문 상품은 임의로 삭제되어서는 안 되므로 `orphanRemoval`은 사용하지 않는다.
    - **상품 애그리거트**: `Product` → `Stock`은 강력한 1:1 결합이므로 `CascadeType.ALL`을 적용한다.
    - **브랜드-상품 관계**: 대규모 데이터 보호를 위해 영속성 전이를 사용하지 않으며, 필요 시 도메인 서비스에서 벌크 연산으로 처리한다.
- **물리 제약조건 방지**: 모든 연관관계 매핑 시 `foreignKey = @ForeignKey(ConstraintMode.NO_CONSTRAINT)`를 사용하여 DB 레벨의 물리적 외래키 생성을 방지하고 논리적 매핑만 유지한다.
- **Facade 패턴**: 복합 도메인 정보가 필요한 경우(예: 상품 상세 조회) Facade에서 각 도메인 서비스를 호출하여 결과를 조립한다.
- **상태 없음**: 도메인 서비스는 상태를 가지지 않으며, 도메인 객체 간의 협력을 조율하는 역할에 집중한다.

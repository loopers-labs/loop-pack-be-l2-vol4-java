# Round 4 주문 트랜잭션 정책

## 배경

3주차 구현에서는 여러 상품 주문 중 일부 상품의 재고가 부족하면 가능한 상품만 주문하고 실패 상품은 `OrderFailure`로 응답하는 부분 주문 정책을 선택했다.

Round 4 과제는 주문 시 재고, 주문, 쿠폰의 정합성을 하나의 트랜잭션으로 보장하고, 동일 상품/동일 쿠폰에 대한 동시성 이슈를 제어하는 것을 요구한다. 특히 쿠폰은 주문 성공 시 즉시 `USED` 상태가 되어야 하며, 재사용되면 안 된다.

따라서 Round 4 구현에서는 부분 주문 UX보다 트랜잭션 원자성과 동시성 정합성을 우선한다.

## 결정

Round 4 주문 생성은 all-or-nothing 정책을 따른다.

- 주문 요청의 모든 상품이 존재하고 주문 수량만큼 재고가 충분해야 주문을 생성한다.
- 상품 없음, 재고 부족, 잘못된 수량 중 하나라도 있으면 주문은 생성하지 않는다.
- 쿠폰 ID가 전달되면 해당 쿠폰이 요청 사용자 소유이고, 사용 가능하며, 만료되지 않았고, 주문 금액 조건을 만족해야 한다.
- 쿠폰 검증 또는 사용 처리 실패 시 재고 차감과 주문 저장은 모두 롤백한다.
- 주문 성공 시 상품 재고 차감, 쿠폰 `USED` 변경, 주문 저장은 하나의 트랜잭션 안에서 커밋한다.
- 주문 스냅샷에는 쿠폰 적용 전 금액, 할인 금액, 최종 결제 금액을 저장한다.
- 동일 쿠폰을 같은 회원에게 동시에 발급하는 경우 DB unique constraint를 최종 방어선으로 두고, 충돌은 `CONFLICT` 비즈니스 예외로 변환한다.

## 대안과 트레이드오프

### A. 부분 주문 정책 유지

- 장점: 사용자는 구매 가능한 상품을 놓치지 않는다.
- 단점: 쿠폰 최소 주문 금액, 할인 금액, 실패 항목 안내, 결제 금액 스냅샷 기준이 복잡해진다.
- 판단: Round 4의 원자성 요구와 동시성 테스트 기준에는 맞지 않는다.

### B. 주문 전체 실패 정책 전환

- 장점: 재고 차감, 쿠폰 사용, 주문 저장의 성공/실패 기준이 명확하다.
- 장점: 락 전략과 동시성 테스트의 기대값을 단순하게 둘 수 있다.
- 단점: 일부 상품만이라도 구매하려는 사용자 경험은 약해진다.
- 판단: Round 4 구현 범위에서는 이 선택이 더 적절하다.

## 구현 영향

- `OrderFacade.createOrder`는 주문 생성 유스케이스의 트랜잭션 경계를 가진다.
- 상품 조회는 일반 조회와 주문 생성용 락 조회를 분리한다.
- 쿠폰 사용 조회도 동시 사용을 막을 수 있는 락 또는 버전 검증 경로를 둔다.
- `OrderProductProcessService`는 실패 항목 수집보다 전체 주문 가능 여부 검증과 주문 라인 생성 책임으로 재정의한다.
- Round 4 주문 생성에서는 실패 응답을 `ApiControllerAdvice`의 에러 응답으로 일원화한다.

## 락 획득 순서

주문 생성처럼 여러 도메인의 상태를 함께 변경하는 흐름에서는 락 획득 순서를 고정한다.

1. `Product` row를 product id 오름차순으로 잠근다.
2. 주문 가능 여부를 검증하고 재고를 차감한다.
3. 쿠폰이 있으면 `IssuedCoupon` row를 잠근 뒤 사용 처리한다.
4. 변경된 상품과 주문을 저장한다.

현재 정책은 `Product -> IssuedCoupon` 순서를 따른다.
향후 다른 유스케이스에서 상품과 발급 쿠폰을 함께 잠가야 한다면 같은 순서를 유지한다.
반대 순서인 `IssuedCoupon -> Product` 흐름이 생기면 주문 생성과 교차 대기할 수 있으므로 별도 검토 대상이다.

## 테스트 기준

- 동일 상품에 여러 주문이 동시에 들어와도 최종 재고는 성공 주문 수만큼만 차감된다.
- 재고가 부족한 동시 주문은 실패하고, 실패한 주문의 쿠폰 사용과 주문 저장은 남지 않는다.
- 동일 쿠폰으로 여러 요청이 동시에 주문해도 쿠폰은 한 번만 `USED`가 된다.
- 쿠폰 검증 실패 시 재고와 주문은 변경되지 않는다.

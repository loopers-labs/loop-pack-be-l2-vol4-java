# Spec: PAY-2 콜백 확정

**소스**: docs/volume-6/01-requirements.md / PAY-2
**작성일**: 2026-06-25
**상태**: Draft

## 시나리오 요약

외부 결제 시스템(PaymentGateway)이 비동기 처리 결과(성공/실패와 사유)를 **콜백**으로 통보하면, 시스템은 콜백이 가리키는 결제를 **주문 식별자**(우리 멱등 닻)로 조회해, 아직 확정되지 않은(`PENDING`) 결제를 처리 결과에 따라 성공(`SUCCESS`)/실패(`FAILED`)로 확정하고, 그에 맞춰 **주문 상태를 결제 완료(`PAID`)/결제 실패(`PAYMENT_FAILED`)로 함께 전이**한다. 결제의 종료 상태는 불변이므로, 같은 결과의 콜백이 중복 도착해도 결제·주문 후처리는 한 번만 일어난다(멱등). 콜백은 PG 시스템이 호출하므로 회원 인증 헤더가 없으며, Stage 0에서는 콜백 본문을 신뢰해 확정한다.

> **Stage 0 범위**: 콜백 본문의 PG 교차검증(위변조 방어)·콜백 누락 폴링 보정·콜백과 폴링의 동시 확정 경합 제어는 모두 후속 Stage(6·7)의 범위다. 본 시나리오는 단일 콜백을 받아 결제·주문을 확정하고 중복 콜백을 멱등 흡수하는 happy path 완결까지만 다룬다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** `PENDING` 결제가 존재할 때, **When** 처리 성공 콜백이 도착하면, **Then** 결제가 `SUCCESS`로 확정되고 대상 주문이 `PAID`로 전이되며 정상 수신(200)으로 응답한다.
2. **Given** `PENDING` 결제가 존재할 때, **When** 처리 실패 콜백(사유 포함)이 도착하면, **Then** 결제가 `FAILED`로 확정되고 실패 사유가 결제에 기록되며 대상 주문이 `PAYMENT_FAILED`로 전이되고 정상 수신(200)으로 응답한다.

### Alternate Flow
1. **Given** 이미 확정된(`SUCCESS`/`FAILED`) 결제일 때, **When** 같은 결제에 대한 콜백이 (중복) 도착하면, **Then** 결제·주문 상태를 변경하지 않고 정상 수신(200)으로 응답한다(멱등).

### Exception Flow
1. **Given** 콜백이 가리키는 주문 식별자에 해당하는 결제가 존재하지 않을 때, **When** 콜백이 도착하면, **Then** 자원 부재(NOT_FOUND)로 응답한다.
2. **Given** 결제가 존재할 때, **When** 콜백의 거래 식별자가 결제에 기록된 거래 식별자와 일치하지 않으면, **Then** 권한 없음(FORBIDDEN)으로 거절하고 결제·주문 상태를 바꾸지 않는다(콜백 위조 방어).

### 비즈니스 규칙
- 결제의 종료 상태(`SUCCESS`/`FAILED`)는 변경되지 않는다. 같은 결과의 콜백이 중복 도착해도 결제·주문 상태는 한 번만 전이한다.
- 결제 결과와 주문 상태는 함께 확정된다. 성공은 주문 `PAID`, 실패는 주문 `PAYMENT_FAILED`에 대응한다.
- 결제 실패 사유(한도 초과·잘못된 카드 등)는 결제에 기록한다.
- 콜백이 가리키는 결제는 **주문 식별자**로 조회한다(우리 멱등 닻). 거래 식별자도 함께 오지만, 타임아웃으로 거래 식별자가 부재할 수 있어 주문 식별자를 조회 기준으로 둔다(결정 1).

## 엣지 케이스

- **중복 콜백**: 동일 결과 콜백이 2회 이상 도착해도 주문 후처리(상태 전이)는 정확히 한 번. 결제의 종료 상태 불변(전이 가드)이 이를 보장한다.
- **콜백 주문 식별자 역매핑**: 콜백 본문의 주문 식별자는 우리가 접수 때 보낸 형식(6자리 0-padding 문자열)으로 되돌아온다. 인바운드 경계에서 우리 `Long` 식별자로 역매핑한다(PAY-1 아웃바운드 매핑의 대칭).
- **예상치 못한 `PENDING` 콜백**: 콜백은 처리 결과(성공/실패) 통보가 본질이라 `PENDING`으로 오지 않지만, 도착해도 종료 전이를 하지 않는다(방어적 무시 — 최종 정합은 Stage 6 폴링이 담당).
- **인증/위조 방어**: 콜백은 PG 시스템이 호출하므로 회원 인증 헤더가 없다(공개 엔드포인트). 대신 콜백의 **거래 식별자가 접수 때 기록한 값과 일치**해야 확정한다 — 임의 주문에 결과를 위조하는 공격을 막는 경량 진위 검증. 거래 식별자는 비밀이 아니므로 완전한 방어는 아니며, HMAC 서명·replay 방지·PG 서버-투-서버 결과 조회는 Stage 6 범위(현 pg-simulator는 서명을 보내지 않는다).
- **입력 검증**: 공개 엔드포인트이므로 `orderId`(숫자)·`transactionKey`(비어 있지 않음)·`status`(허용 enum) 형식을 경계에서 검증하고 위반 시 입력 오류로 거절한다.
- **동시 확정 경합**(콜백 스레드 ↔ 폴링 스레드) → 범위 밖(Stage 7 조건부 UPDATE).

## 기능 요구사항

- **FR-001**: 시스템은 콜백을 받아 주문 식별자로 결제를 조회해야 한다.
- **FR-002**: 시스템은 콜백이 가리키는 결제가 없으면 자원 부재로 응답해야 한다.
- **FR-003**: 시스템은 `PENDING` 결제를 처리 결과에 따라 `SUCCESS` 또는 `FAILED`로 확정해야 한다.
- **FR-004**: 시스템은 실패 콜백의 사유를 결제에 기록해야 한다.
- **FR-005**: 시스템은 결제 확정에 맞춰 주문을 `PAID`(성공)/`PAYMENT_FAILED`(실패)로 전이해야 한다.
- **FR-006**: 시스템은 이미 종료 상태인 결제에 대한 콜백이면 상태를 변경하지 않고 정상 수신으로 응답해야 한다(멱등).
- **FR-007**: 시스템은 콜백을 정상 수신했음을 응답해야 한다.

## 관련 엔티티

- **Payment** (기존, PAY-1에서 생성): `succeed()`·`fail(reason)`(`PENDING`에서만 종료 상태로 전이하는 가드)·`isTerminal()`를 본 시나리오에서 실제 사용한다. PAY-1에서 미룬 전이 메서드 단위 테스트를 본 cycle에서 보강한다.
  - **PaymentRepository**: 주문 식별자로 결제를 조회하는 기능 추가(PAY-1엔 존재 검사·저장만 있었다).
- **Order** (기존, 변경분): 결제 결과를 반영하는 상태 전이를 더한다.
  - `OrderStatus`에 `PAID`·`PAYMENT_FAILED` 추가(현재 `CREATED`만 존재).
  - `markPaid`(`CREATED`→`PAID`)·`markPaymentFailed`(`CREATED`→`PAYMENT_FAILED`) 추가.

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| Model 단위 | PaymentModel | `succeed`/`fail`가 `PENDING`에서만 전이하고 종료 상태에선 무변경(멱등 가드), `fail`이 사유 기록, `isTerminal` 질의 |
| Model 단위 | OrderModel | `markPaid`·`markPaymentFailed`가 `CREATED`에서 각 종료 상태로 전이 |
| Service 단위 | PaymentFacade | 성공 콜백→결제 SUCCESS+주문 PAID, 실패 콜백→결제 FAILED(사유)+주문 PAYMENT_FAILED, 이미 종료→무변경·주문 미전이(멱등), 결제 없음→NOT_FOUND |
| Integration | PaymentRepository | 주문 식별자로 결제 조회(존재/부재) |
| E2E | `POST /api/v1/payments/callback` | 성공/실패 확정 후 상태, 중복 콜백 멱등, 결제 없음(NOT_FOUND)의 status + meta.result + errorCode |

> 단언 범위는 CLAUDE.md 규약을 따른다: 단위는 ErrorType까지, E2E는 status+meta.result+errorCode까지. 에러 메시지 문구는 단언하지 않는다.

## 관련 결정

- **결정 1 (멱등 기준 = 주문 식별자)**: 콜백 결제 조회를 주문 식별자로 한다. 거래 식별자는 부재 가능성이 있어 조회 기준이 아니다.
- **종료 상태 불변(멱등)**: 결제의 `SUCCESS`/`FAILED`는 재전이하지 않으며, 중복 콜백의 후처리를 한 번으로 만든다.

## 성공 기준 / 범위 밖

- **성공**: 성공 콜백이 PENDING 결제를 `SUCCESS`+주문 `PAID`로, 실패 콜백이 `FAILED`(사유)+주문 `PAYMENT_FAILED`로 확정하고, 중복 콜백이 멱등하며, 없는 결제 콜백이 NOT_FOUND. 위 테스트 계획이 모두 green.
- **범위 밖**: 콜백 HMAC 서명·replay 방지·PG 서버-투-서버 결과 조회(완전한 위변조 방어, Stage 6 — 거래 식별자 일치 검증은 Stage 0에서 선반영), 콜백 누락 폴링 보정(Stage 6), 콜백·폴링 동시 확정 경합 제어(Stage 7 조건부 UPDATE), 재고·쿠폰 보상 복원.

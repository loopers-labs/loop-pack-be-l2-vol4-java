# Spec: PAY-1 결제 요청

**작성일**: 2026-06-25
**상태**: Draft

## 시나리오 요약

로그인한 **회원**이 자신의 **주문**에 대해 카드 결제를 요청한다. 회원은 주문 식별자와 카드 정보(카드 종류·카드 번호)만 보내고, 결제 금액은 서버가 대상 주문의 최종 결제 금액에서 도출한다(요청 본문의 금액을 신뢰하지 않음). 시스템은 본인 소유의 결제 가능한(`CREATED`) 주문인지, 그 주문에 이미 결제가 있는지 확인한 뒤, 결제를 **접수 대기(`PENDING`)** 로 생성·저장하고 **외부 결제 시스템(PaymentGateway)** 에 접수를 요청한다. 접수 응답으로 받은 거래 식별자를 결제에 보조 핸들로 기록하고, 접수 결과(`PENDING`)를 반환한다. 결제의 최종 성공/실패는 이후 비동기 콜백(PAY-2)으로 확정되므로, 이 시나리오는 **요청(동기 접수)과 결과(비동기 확정)가 분리되는 첫 지점**이다.

> **Stage 0 범위**: 타임아웃·재시도·서킷 브레이커·fallback·폴링 등 회복 장치는 의도적으로 넣지 않는다. 외부 접수 호출은 트랜잭션 안에서 동기로 수행하며, happy path 완결만 목표로 한다.

## 수용 시나리오 (Given/When/Then)

### Main Flow
1. **Given** 본인 소유의 `CREATED` 주문이 있고 그 주문에 결제가 없으며 유효한 카드 정보가 주어졌을 때, **When** 회원이 결제를 요청하면, **Then** 결제가 `PENDING`으로 생성·저장되고 외부 결제 시스템에 접수 요청이 나가며 발급받은 거래 식별자가 결제에 기록되고, 접수 결과(`PENDING` 상태와 식별 정보)가 반환된다.
2. **Given** Main Flow가 성공했을 때, **When** 결제 금액을 확인하면, **Then** 결제 금액은 요청 본문이 아니라 대상 주문의 최종 결제 금액과 일치한다.

### Exception Flow
1. **Given** 회원 인증 정보가 없거나 잘못되었을 때, **When** 결제를 요청하면, **Then** 인증 실패로 응답한다.
2. **Given** 대상 주문이 존재하지 않을 때, **When** 결제를 요청하면, **Then** 자원 부재(NOT_FOUND)로 응답한다.
3. **Given** 대상 주문이 타인 소유일 때, **When** 결제를 요청하면, **Then** 자원 부재(NOT_FOUND)로 응답한다(존재 노출 방지).
4. **Given** 대상 주문에 이미 결제가 존재할 때(상태 불문), **When** 결제를 요청하면, **Then** 자원 충돌(CONFLICT)로 응답한다.
5. **Given** 지원하지 않는 카드 종류 또는 형식이 잘못된 카드 번호가 주어졌을 때, **When** 결제를 요청하면, **Then** 입력 검증 실패(BAD_REQUEST)로 응답한다.

### 비즈니스 규칙
- 한 주문은 한 번만 결제된다. 결제 멱등의 기준은 **주문 식별자**이며, 같은 주문에 둘 이상의 결제가 생성되지 않는다(결정 1).
- 결제 금액은 **대상 주문의 최종 결제 금액**에서 도출한다(결정 2).
- 카드 종류는 시스템이 지원하는 우리 도메인 카드 종류만 허용하며, 외부 결제 시스템 표현과 별개로 정의·매핑한다(결정 3).
- 카드 번호는 `xxxx-xxxx-xxxx-xxxx` 형식이어야 한다.
- 접수와 결과 확정은 분리된다. 접수 시점 결과를 알 수 없어 결제는 `PENDING`으로 시작하고, 최종 결과는 콜백(PAY-2)으로 확정한다.

## 엣지 케이스

- **동시성("따닥")**: 같은 주문에 동시에 결제 요청이 들어와도 결제는 한 건만 생성되어야 한다. 응용 계층의 존재 검사만으로는 race를 막지 못하므로, 주문 식별자 **유일 제약**이 두 번째 생성을 차단하고 이를 자원 충돌로 번역한다.
- **거래 식별자 부재 가능성**: 거래 식별자는 외부가 발급하는 보조 핸들이라 접수 응답을 받지 못하면 부재할 수 있다(Stage 0 happy path에서는 수신을 전제, nullable 허용은 후속 단계 대비).
- **이미 결제된 주문 재요청**: 종료 상태(SUCCESS/FAILED)든 진행 중(PENDING)이든, 결제가 존재하면 자원 충돌로 거절한다(주문당 1건).

## 기능 요구사항

- **FR-001**: 시스템은 결제 요청 시 회원 인증을 확인해야 한다.
- **FR-002**: 시스템은 대상 주문이 존재하고 요청 회원 본인 소유인지 확인하고, 아니면 자원 부재로 응답해야 한다.
- **FR-003**: 시스템은 대상 주문에 이미 결제가 있으면 자원 충돌로 응답해야 한다.
- **FR-004**: 시스템은 카드 종류가 지원 범위인지, 카드 번호가 형식에 맞는지 검증하고 위반 시 입력 검증 실패로 응답해야 한다.
- **FR-005**: 시스템은 결제 금액을 대상 주문의 최종 결제 금액에서 도출해야 한다(요청 본문 금액 무시).
- **FR-006**: 시스템은 결제를 `PENDING` 상태로 생성·저장해야 한다.
- **FR-007**: 시스템은 외부 결제 시스템에 결제 접수를 요청하고, 우리 카드 종류를 외부 표현으로 매핑해 전송해야 한다.
- **FR-008**: 시스템은 접수 응답의 거래 식별자를 결제에 보조 핸들로 기록해야 한다.
- **FR-009**: 시스템은 같은 주문에 대한 동시 결제 요청에도 결제가 한 건만 생성되도록 유일 제약으로 보장해야 한다.
- **FR-010**: 시스템은 접수 결과(결제 상태·식별 정보)를 반환해야 한다.

## 관련 엔티티

- **Payment** (신규 aggregate 골격 필요): 주문 ID·회원 ID·결제 금액·카드 종류·카드 번호·거래 식별자(보조, nullable)·상태·사유(nullable)·접수 시각을 보유. 접수 시 `PENDING`으로 시작. 본 시나리오에서 새로 만든다.
  - **CardNo** (VO): 카드 번호 `xxxx-xxxx-xxxx-xxxx` 형식 검증을 값의 원천 지점에 단일화.
  - **CardType** (Enum): 우리 도메인 카드 종류(SAMSUNG/KB/HYUNDAI). 외부 표현 매핑은 인프라 어댑터 책임.
  - **PaymentStatus** (Enum): PENDING/SUCCESS/FAILED. 본 시나리오는 PENDING 생성까지.
  - **PaymentRepository**: 결제 저장·주문별 존재 검사.
- **PaymentGateway 포트**: 외부 결제 시스템 접수 요청 경계(멀티 PG를 염두에 둔 추상화). 본 시나리오에서 새로 정의한다.
- **Order** (기존, 참조): 본인 소유·결제 가능 상태(`CREATED`) 확인과 최종 결제 금액 도출의 출처. 본 시나리오에서 Order 구조는 변경하지 않는다(상태 전이 추가는 PAY-2).

## 테스트 계획

| 레벨 | 대상 | 무엇을 단언하는가 |
|------|------|------------------|
| VO 단위 | CardNo | `xxxx-xxxx-xxxx-xxxx` 형식 통과/거부(자릿수·구분자·비숫자 경계값), 위반 시 ErrorType |
| Model 단위 | Payment | `create`가 `PENDING`·접수 시각으로 시작하고 거래 식별자는 부재로 시작, `recordTransactionKey`가 거래 식별자를 기록 |
| Service 단위 | PaymentFacade | 주문 부재/타인 소유→자원 부재, 기존 결제 존재→자원 충돌, 금액이 주문 최종 결제 금액에서 도출, 외부 접수 요청 호출과 거래 식별자 기록 협력 |
| Integration | PaymentRepository | 결제 영속화, 주문 식별자 유일 제약 위반 시 충돌, 주문별 존재 검사 |
| E2E | `POST /api/v1/payments` | 성공: status + meta.result=SUCCESS + 응답 키(결제 상태=PENDING 등). 실패: 주문 부재(NOT_FOUND)·기존 결제(CONFLICT)·잘못된 카드(BAD_REQUEST)의 status + meta.result + errorCode |

> 단언 범위는 CLAUDE.md 테스트 규약을 따른다: 단위는 ErrorType까지, E2E는 status+meta.result+errorCode까지. 에러 메시지 문구는 어디서도 단언하지 않는다.

## 관련 결정

- **결정 1 (멱등 기준 = 주문 식별자)**: 주문당 결제 1건, 유일 제약으로 동시 요청 차단. 동일 주문 재결제는 범위 밖.
- **결정 2 (금액 출처 = 주문)**: 위변조 차단을 위해 서버가 주문에서 도출.
- **결정 3 (카드 종류 = 도메인 enum 매핑)**: 외부 표현은 어댑터 경계에서만 변환.

## 성공 기준 / 범위 밖

- **성공**: happy path 1건이 요청 → 본인/금액/멱등 검증 → `PENDING` 결제 생성·저장 → 외부 접수 요청 → 거래 식별자 기록 → 접수 응답까지 흐르고, 위 테스트 계획이 모두 green.
- **범위 밖**: 타임아웃·재시도·서킷·fallback·폴링(후속 Stage), 콜백 결과 확정·주문 상태 전이(PAY-2), 결제 결과 조회 API, 재고/쿠폰 보상 복원, 카드 번호 마스킹·암호화.

# '감성 이커머스' 요구사항 명세

본 문서는 '감성 이커머스'의 기능에 대한 합의 사항을 기록한다. 한 줄 요청만으로는 드러나지 않는 전제·예외·정책을 풀어 적고, 트레이드오프가 있는 결정은 [핵심 의사결정 분기](#핵심-의사결정-분기)에 그 이유와 함께 남긴다. 회원·브랜드·상품·좋아요·주문·쿠폰 도메인의 기존 합의는 `docs/volume-2`~`volume-4`의 명세를 따르며, 본 문서는 결제 도메인과 주문 상태 변경분만 다룬다.

---

## 도메인 및 용어 정의

### 결제 (Payment)

| 한글 | 영어 | 의미 |
|---|---|---|
| 결제 | `Payment` | 한 주문에 대한 한 번의 결제. 주문 식별자를 멱등 기준으로 하며(결정 1), 접수된 뒤 외부 결제 시스템의 결과로 확정된다. |
| 결제 상태 | `PaymentStatus` | 접수 대기(`PENDING`)·성공(`SUCCESS`)·실패(`FAILED`). 접수 시점에는 결과를 알 수 없어 `PENDING`으로 시작하고, 종료 상태(`SUCCESS`/`FAILED`)는 한 번 정해지면 바뀌지 않는다. |
| 결제 금액 | `Amount` | 결제 대상 금액. 요청 본문이 아니라 대상 주문의 최종 결제 금액에서 도출한다(결정 2). |
| 카드 종류 | `CardType` | 시스템이 지원하는 카드 종류(예: 삼성·KB·현대). 외부 결제 시스템의 카드 종류와 별개로 우리 도메인에 정의하고 매핑한다(결정 3). |
| 카드 번호 | `CardNo` | `xxxx-xxxx-xxxx-xxxx` 형식의 카드 번호. |
| 거래 식별자 | `TransactionKey` | 외부 결제 시스템이 발급하는 거래 식별 핸들. 결제에 보조로 기록하며, 접수 결과를 받지 못하면 부재할 수 있다. |

### 주문 (Order)

| 한글 | 영어 | 의미 |
|---|---|---|
| 주문 상태 | `OrderStatus` | 주문의 진행 상태. 주문 생성됨(`CREATED`)·결제 완료(`PAID`)·결제 실패(`PAYMENT_FAILED`). 결제 가능한 상태는 `CREATED`이며, 결제 결과 확정 시 `PAID`/`PAYMENT_FAILED`로 전이한다(PAY-2). |

## 결제 (Payment)

### PAY-1. 회원은 주문에 대해 카드 결제를 요청한다.

**전제 조건**
- 회원 인증 정보를 보유한 상태여야 한다.
- 대상 주문이 존재하고 요청 회원 본인의 주문이어야 한다.
- 대상 주문에 완료되었거나 진행 중인 결제가 없어야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 주문 식별자와 카드 정보(카드 종류·카드 번호)를 포함해 결제를 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 대상 주문이 존재하고 본인 소유이며 결제 가능한 상태인지 점검한다.
    4. 카드 정보의 형식이 유효한지 검사한다.
    5. 결제 금액을 대상 주문의 최종 결제 금액으로 확정한다.
    6. 결제를 접수 대기(`PENDING`) 상태로 생성한다.
    7. 외부 결제 시스템에 결제 접수를 요청하고, 발급받은 거래 식별자를 결제에 기록한다.
    8. 접수된 결제의 상태와 식별 정보를 반환한다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 주문이 존재하지 않거나 본인 주문이 아니면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 대상 주문에 이미 완료되었거나 진행 중인 결제가 있으면, 시스템은 자원 충돌로 응답한다.
    - 카드 종류가 지원 범위를 벗어나거나 카드 번호 형식이 올바르지 않으면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 한 주문은 한 번만 결제된다. 결제 멱등의 기준은 주문 식별자이며, 같은 주문에 둘 이상의 결제가 생성되지 않는다(결정 1).
- 결제 금액은 요청 본문이 아니라 대상 주문의 최종 결제 금액으로 확정한다(결정 2).
- 카드 종류는 시스템이 지원하는 종류만 허용하며, 외부 결제 시스템의 카드 종류와 별개로 우리 도메인의 카드 종류로 정의해 매핑한다(결정 3).
- 카드 번호는 `xxxx-xxxx-xxxx-xxxx` 형식이어야 한다.
- 결제의 접수와 결과 확정은 분리된다. 접수 시점에는 결과를 알 수 없어 결제는 `PENDING`으로 시작하고, 최종 결과는 외부 결제 시스템의 콜백으로 확정한다(PAY-2).

**비기능 요구사항**
- 같은 주문에 동시에 결제 요청이 들어와도 결제는 한 건만 생성되어야 한다.

**관련 API**
- `POST /api/v1/payments`

### PAY-2. 외부 결제 시스템의 처리 결과 콜백을 받아 결제·주문 상태를 확정한다.

**전제 조건**
- 콜백이 가리키는 결제가 이전에 접수되어 존재해야 한다.

**기능 요구사항**
- Main Flow
    1. 외부 결제 시스템이 처리 결과(성공/실패와 사유)를 콜백으로 통보한다.
    2. 시스템은 콜백이 가리키는 결제를 식별한다.
    3. 해당 결제가 아직 확정되지 않은(`PENDING`) 상태인지 확인한다.
    4. 처리 결과에 따라 결제를 성공(`SUCCESS`) 또는 실패(`FAILED`)로 확정한다.
    5. 결제 결과에 맞춰 주문 상태를 결제 완료(`PAID`) 또는 결제 실패(`PAYMENT_FAILED`)로 전이한다.
    6. 콜백을 정상 수신했음을 응답한다.
- Alternate Flow
    - 이미 확정된(`SUCCESS`/`FAILED`) 결제에 대한 콜백이면, 상태를 변경하지 않고 정상 수신으로 응답한다.
- Exception Flow
    - 콜백이 가리키는 결제가 존재하지 않으면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 결제의 종료 상태(`SUCCESS`/`FAILED`)는 변경되지 않는다. 같은 결과의 콜백이 중복 도착해도 결제·주문 상태는 한 번만 전이한다.
- 결제 결과와 주문 상태는 함께 확정된다. 결제 성공은 주문 결제 완료(`PAID`), 결제 실패는 주문 결제 실패(`PAYMENT_FAILED`)에 대응한다.
- 결제 실패 사유(예: 한도 초과·잘못된 카드)는 결제에 기록한다.

**비기능 요구사항**
- 결과 콜백이 중복 도착해도 결제·주문의 후처리는 한 번만 일어나야 한다.

**관련 API**
- `POST /api/v1/payments/callback`

## 핵심 의사결정 분기

### 결정 1. 결제 멱등의 기준

**문제**: 같은 주문에 결제 요청이 중복될 수 있다(따닥 클릭, 타임아웃 후 재시도, 콜백 중복). 무엇을 기준으로 "같은 결제"로 보고 한 건만 남길 것인가.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 주문 식별자를 멱등 기준으로 두고 한 주문당 결제 한 건만 생성(주문 식별자 유일 제약) | 주문이 이미 우리가 가진 식별자라 별도 키 발급이 불필요. 주문 생성 시 재고·쿠폰이 그 주문에 소비되어 결제 단위가 주문과 1:1이라 자연스럽다. 결제 실패는 주문 종료로 다루며, 동일 주문 재결제는 별도 주제 |
| B | 클라이언트가 멱등 키를 발급해 전달 | 주문 생성 전 구간까지 멱등 보장이 가능하나 클라이언트 신뢰에 의존하고 키 관리 부담이 생김 |
| C | 외부 시스템이 발급하는 거래 식별자에 의존 | 타임아웃으로 거래 식별자를 받지 못하면 추적이 불가능(닭-달걀). 외부 키 단독 의존은 위험 |

**사유**: 주문은 이미 우리가 가진 식별자(닻)이고 외부 시스템은 신뢰할 수 없으므로, 주문 식별자를 정합성의 기준으로 둔다. 주문 생성 시점에 재고·쿠폰이 그 주문에 소비되어 결제 단위가 주문 단위와 1:1이므로, 별도 멱등 키나 시도별 다중 결제 행 없이 주문 식별자 하나로 충분하다. 외부 거래 식별자는 보조 핸들로 매핑 저장하며, 같은 거래가 두 결제에 매핑되지 않도록 별도의 유일 제약을 둔다. 결제 실패는 주문의 종료 상태이며, 동일 주문 재결제(실패 후 재시도)는 재고·쿠폰 보상 또는 예약 모델이 갖춰져야 안전하므로 이번 범위 밖이다.

**추후 개선**: 동일 주문 재결제는 선차감된 재고·쿠폰의 보상 복원 또는 예약 모델과 함께 도입할 수 있다.

### 결정 2. 결제 금액의 출처

**문제**: 결제 금액을 요청 본문으로 받을지, 대상 주문에서 도출할지.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 요청은 주문 식별자·카드 정보만 받고, 결제 금액은 대상 주문의 최종 결제 금액으로 서버가 확정 | 금액 위변조를 차단. 단 주문과 결제 금액이 1:1이어야 함(부분 결제 불가) |
| B | 요청 본문으로 결제 금액을 받아 사용 | 부분·복합 결제에 유연하나 클라이언트가 보낸 금액을 신뢰해야 해 위변조 위험 |

**사유**: 금액은 신뢰 경계 밖에서 오면 안 되는 값이므로 서버가 주문에서 도출한다. 부분 결제는 이번 범위 밖이다.

### 결정 3. 카드 종류의 표현

**문제**: 외부 결제 시스템의 카드 종류를 도메인에서 그대로 쓸지, 우리 타입으로 정의·매핑할지.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 우리 도메인의 카드 종류를 정의하고, 외부 표현은 어댑터 경계에서 매핑 | 외부 스펙 변화·다중 결제사에 도메인이 흔들리지 않음. 매핑 코드 한 겹이 추가됨 |
| B | 외부 시스템의 카드 종류를 도메인에서 직접 사용 | 변환 코드가 없으나 외부 스펙에 도메인이 직접 결합됨 |

**사유**: 결제사가 하나여도 카드 종류 표현이 같으리라는 보장이 없고, 도메인을 외부 스펙에서 분리하기 위해 우리 타입으로 매핑한다.

### 📌 Q1. [정책] 정률(RATE) 쿠폰의 최대 할인 금액 한도(Limit) 정책이 존재하나요?

• 선택지 A (단순형): 한도 없음. 정률(%) 비율 그대로 무제한 적용합니다. (구현이 간단합니다)
• **선택지 B (안전형 - 추천): 쿠폰 템플릿에  maxDiscountAmount  필드를 nullable하게 정의하고, 한도가 설정된 경우 할인
금액이 이 한도를 넘지 않도록 차감액을 제한합니다. (현업 비즈니스에서 예산 관리를 위해 일반적으로 사용하는
방식입니다)**

### 📌 Q2. [정책/기술] 재고 및 쿠폰의 동시성 제어 락(Lock) 전략을 어떻게 구성할까요?

• 선택지 A (비관적 락 일괄 적용):
• 상품 재고와 쿠폰 사용 모두 **비관적 락(Pessimistic Write Lock)**을 적용합니다.
• 대량의 동시 주문 시에도 DB단에서 락을 통해 정합성을 안전하게 지키지만, 트랜잭션 대기 시간이 늘어날 수
있습니다.
• 선택지 B (낙관적 락 일괄 적용):
• 상품 재고와 쿠폰 사용 모두 **낙관적 락(Optimistic Lock with  @Version )**을 적용합니다.
• 락 대기 시간이 없어 기본 성능이 좋으나, 충돌 발생 시 사용자는 실패하게 되며 필요 시 애플리케이션 단에서
재시도(Retry) 로직을 개발해야 합니다.
• 선택지 C (도메인별 하이브리드 전략 - 추천):
• 상품 재고: 여러 명이 동시에 구매하여 충돌이 빈번하므로 비관적 락을 적용하여 대기 큐 형태로 처리.
• 쿠폰 사용: 동일 사용자가 본인의 쿠폰을 동시에 두 번 쓰는 특수 케이스(충돌 빈도 극히 낮음)이므로 낙관적 락을
적용하여 동시 요청 시 버전 오류 예외로 방어.

### 1. 사용자 본인의 악의적 어뷰징 (Double Spending)

사용자가 여러 개의 브라우저 탭을 열거나, 여러 기기에서 동시에 주문 API를 호출하는 경우(또는 스크립트를 통한 API
동시 다발적 호출) 발생합니다.

• 상황: 쿠폰 사용 주문 API 요청이 거의 동시에 2개 들어옵니다.
• 문제 발생 흐름:
1. 요청 A가 데이터베이스에서 사용자 쿠폰의 상태를 조회합니다. ( AVAILABLE  상태 확인)
2. 요청 B가 요청 A가 상태를  USED 로 변경하고 커밋하기 직전에 동일한 쿠폰 상태를 조회합니다. (역시  AVAILABLE
로 보임)
3. 요청 A와 요청 B 모두 쿠폰 적용 및 주문 처리를 완료하여, 결과적으로 1장의 쿠폰으로 2번의 할인 혜택을 받게
됩니다.


### 2. 쿠폰 발급 단계에서의 선착순 경합 (Template Stock)

사용 중이 아닌 쿠폰 발급(Issue) 단계에서의 동시성 이슈입니다.

• 상황: "선착순 100명 한정 쿠폰 발급"과 같은 이벤트가 진행될 때 발생합니다.
• 문제 발생 흐름:
• 수천 명의 사용자가 동시에 발급 버튼을 눌렀을 때, 쿠폰 템플릿의  최대 발급 수량  또는  남은 수량 을 정확하게
검증하고 차감하지 않으면 100명보다 훨씬 많은 사용자에게 쿠폰이 발급되는 초과 발급(Over-issuance) 문제가
발생합니다.

week4 요구사항의 체크리스트( 동일한 쿠폰으로 여러 기기에서 동시에 주문해도, 쿠폰은 단 한번만 사용되어야 한다 )는
바로 1번(Double Spending) 시나리오를 방어하기 위한 것입니다.

• 이러한 중복 사용 시도는 발생 빈도가 극히 낮으므로(비정상적인 어뷰징 패턴), DB 리소스를 많이 점유하는 비관적
락보다는 **낙관적 락(Version 검증 또는 JPA의  @Version )**을 통해 첫 번째 요청만 성공시키고 두 번째 동시 요청은
버전 불일치 에러를 뱉게 만드는 것이 성능과 자원 효율 측면에서 가장 합리적입니다.

### 1. 비관적 락 (Pessimistic Lock) ➡️ "선점 후 대기(줄서기)"

• 동작: 첫 번째 요청이 데이터를 선점하여 잠금(Lock)을 걸면, 나머지 요청들은 **DB 커넥션을 유지한 채 락이 풀릴
때까지 대기(Wait)**합니다.
• 재고 차감 시나리오에서의 장점:
• 앞사람이 재고를 1개 줄이고 트랜잭션을 끝내면(예: 100개 ➡️ 99개), 뒤에서 기다리던 사람은 변경된 최신
상태(99개)를 바탕으로 이어서 남은 재고를 차감할 수 있습니다.
• 따라서 충돌이 발생해도 뒤에 줄 서 있던 요청들이 실패하지 않고 순차적으로 안전하게 주문을 성공시킬 수 있습니다.
• 단점: 대기하는 요청이 많아지면 DB 커넥션을 오래 잡고 있게 되어 시스템 전체가 느려질 수 있고, 설정된  Lock Timeout
시간 동안 락을 획득하지 못하면 예외가 발생할 수 있습니다.

### 2. 낙관적 락 (Optimistic Lock) ➡️ "락 없이 일단 진행 후 검증(버전 확인)"

• 동작: 락을 걸지 않고 동시에 데이터를 읽어간 뒤, 수정(Commit)을 시도하는 시점에 버전 번호가 올라갔는지 체크합니다.
가장 먼저 커밋한 요청만 성공하고, 나머지 동시 요청들은 대기 없이 즉시 버전 충돌 예외를 뱉으며 실패 처리됩니다.
• 장점: DB 커넥션을 물고 대기하지 않으므로 DB 리소스 낭비가 없고, 충돌이 거의 없는 환경에서는 성능이 매우 빠릅니다.
• 단점: 충돌 발생 시 무조건 예외가 발생하므로, 비즈니스적으로 실패한 요청을 자동으로 재처리하려면 애플리케이션
단에서 재시도(Retry) 루프를 돌리는 코드를 작성해야 합니다.

### ⚠️ 설계의 잠재 리스크 및 대안 제안

#### 1. 비관적 락 대기로 인한 DB 커넥션 고갈 위험

• 리스크: 선착순 할인 등의 핫 스폿(Hot-spot) 상품 주문 시, 수많은 쓰레드가 재고 엔티티의  FOR UPDATE  쓰기 잠금을
획득하기 위해 대기하게 됩니다. 이로 인해 DB 커넥션 풀(Connection Pool)이 빠르게 고갈되어 다른 단순 조회 API들까지
먹통이 될 수 있습니다.
• 대안 선택지:
• 선택지 A: DB 커넥션 풀 사이즈 확장 및  Lock Timeout  시간을 짧게(예: 3초) 가져가 락을 오래 잡지 못하도록
빠르게 실패(Fail-Fast) 처리합니다.
• 선택지 B: 향후 트래픽이 커질 경우 Redis 등을 이용한 분산 락(Distributed Lock) 또는 분산 메시지
큐(Kafka/RabbitMQ)를 도입하여 애플리케이션 단에서 진입 트래픽을 제어합니다.


#### 2. 단일 트랜잭션의 비대화

• 리스크: 현재 주문 트랜잭션 내에서  쿠폰 검증 및 상태 변경  ->  재고 차감  ->  주문서 생성 까지 모두 처리됩니다.
향후 타 시스템(예: 외부 PG 결제사 연동, 알림톡 발송 등)이 동일 트랜잭션에 추가되면 네트워크 지연(I/O)으로 인해
트랜잭션 점유 시간이 기하급수적으로 늘어나 데드락 및 성능 저하가 초래됩니다.
• 대안 선택지:
• 선택지 A: 외부 결제 연동은 주문 트랜잭션이 끝난 후 수행하도록 비즈니스 프로세스를 단계적으로 분리합니다.
(주문 생성:  PENDING  ➡️ 외부 결제 성공 ➡️ 결제 및 재고 차감 확정)
• 선택지 B: 트랜잭션 아웃박스 패턴(Transactional Outbox Pattern)을 활용해 비핵심 기능(알림, 메일 등)은 이벤트를
발행하여 비동기로 독립 처리합니다.

### ❓ 질문 1. 쿠폰 발급 시 선착순 수량 제한(최대 발급 수량) 정책이 존재하나요?

만약 존재한다면, 여러 명이 동시에 발급을 요청할 때 한도를 초과하여 발급되지 않도록 동시성을 제어해야 합니다. 아래의
선택지 중 어떤 방식을 선호하시나요?

• 선택지 A: 선착순 수량 제한이 없음
• 영향: 쿠폰 템플릿의 유효기간 내에는 무제한으로 발급 가능합니다. 동시성 제어 락을 고려할 필요가 없어 구현이
매우 단순합니다.
• 선택지 B: 선착순 제한이 있으며, DB 비관적 락으로 제어 (추천)
• 영향:  COUPON_TEMPLATES  테이블에 최대 수량( max_issue_count )과 현재 발급 수량( issued_count ) 필드를 두고,
발급 시점에 템플릿 레코드에 비관적 락( FOR UPDATE )을 걸어 순차적으로 차감합니다. 안전하지만 동시 요청이 몰릴
때 DB 커넥션 대기가 발생할 수 있습니다.
• 선택지 C: 선착순 제한이 있으며, Redis 등을 활용해 캐시 레이어에서 처리
• 영향: DB 락 없이 고성능으로 선착순을 처리할 수 있으나, 현재 인프라 구조상 Redis 연동 로직과 데이터 동기화(DB
저장)를 위한 배치나 비동기 처리가 추가되어 시스템 복잡도가 늘어납니다.

> 애플리케이션 레벨에서 락을 거는 방식으로 제어를 할 수는 없을까?

자바의  synchronized 나  ReentrantLock 과 같이 애플리케이션 레벨(메모리 상)에서 락을 걸어 제어하는 방식도 좋은
대안이 될 수 있습니다.

다만, 이 방식은 아키텍처 구성과 트랜잭션 범위에 따라 다음과 같은 장단점과 제약 사항이 있습니다.

### ⚖️ 애플리케이션 레벨 락의 트레이드오프

1. 서버 확장성(Scale-Out)의 한계
   • 단일 서버 환경에서는 메모리가 공유되므로 문제없이 동작합니다.
   • 하지만 실제 상용 서비스처럼 **다중 서버(서버 2대 이상)**로 로드밸런싱이 될 경우, 각 서버의 메모리가
   독립적이므로 다른 서버에서 동시에 들어오는 요청을 막을 수 없어 초과 발급(동시성 유실) 문제가 여전히 발생합니다.
2. 트랜잭션 프록시와의 정합성 문제
   • 스프링의  @Transactional 은 메서드 실행 완료 후 프록시가 실제 DB 커밋을 완료하는 시점에 락을 해제해야
   안전합니다.
   • 만약 메서드 내부에서 자바 락을 해제하고 나갔는데 DB 커밋이 완료되지 않은 짧은 순간(찰나의 순간)에 다른
   스레드가 들어오면, 아직 커밋되지 않은 데이터를 읽어 중복 발급이 발생하는 **경쟁 상태(Race Condition)**가 발생할
   수 있습니다.

### ❓ 이 한계를 감안했을 때, 어떤 방향으로 정리할까요?

• 선택지 A: 단일 WAS 환경을 가정하고 애플리케이션 락 사용
• 내용: 서비스 규모가 작고 단일 WAS 인스턴스만 구동된다고 가정하며, 트랜잭션 커밋 정합성을 맞춘 별도의 Lock
Helper 클래스 등을 설계하여 적용합니다. DB 부하는 거의 없습니다.
• 선택지 B: 다중 서버 확장을 대비해 DB 비관적 락( SELECT FOR UPDATE ) 채택
• 내용: 향후 다중 WAS 구성을 대비하여, 데이터베이스를 공유 저장소로 삼아 DB 단에서 확실하게 동시성을 제어합니다.
• 선택지 C: Redis를 활용한 분산 락(Distributed Lock) 도입
• 내용: 애플리케이션 레벨의 가벼움과 다중 서버 지원을 모두 챙기기 위해, 외부 공유 캐시인 Redis를 사용하여 분산
락을 구현합니다.

상품 좋아요 등록 API 롤백 방지 개선
• 이유: JPA의  @Transactional  블록 내에서 중복 키 DB 예외(Unique Constraint Violation)가 발생하면, 비록
애플리케이션( catch ) 단에서 예외 처리를 하더라도 트랜잭션이 강제로  rollback-only  상태로 마킹되어 정상적인
트랜잭션 종료(커밋)가 불가능해집니다. 이를 해결하기 위해 트랜잭션 진입 전 혹은 초입에서 기존 좋아요 존재 여부
선조회(Exist Check) 흐름을 추가하여 예외 발생 없이 분기(멱등성 리턴)되도록 수정했습니다.

### ❓ 질문 2. 내 쿠폰 목록 조회 시  EXPIRED (만료) 상태를 판단하고 반환하는 아키텍처 전략은 무엇인가요?

사용자가 쿠폰 목록을 조회할 때, 만료 시간이 지난 쿠폰들을 필터링하고 상태를 결정하는 흐름에 대한 결정입니다.

• 선택지 A: 실시간 동적 판단 (WAS 가공)
• 내용: DB 조회 시점에는 사용하지 않은 쿠폰 정보를 조회해 오고, 애플리케이션(WAS) 레이어에서 쿠폰 템플릿의
만료일( expiredAt )과 현재 시각( LocalDateTime.now() )을 대조하여 실시간으로 상태를  EXPIRED 로 매핑해
응답합니다. DB의 물리적 상태값은 변경하지 않습니다.
• 영향: 배치가 필요 없고 데이터 일치성이 즉시 보장되므로 구현이 단순합니다. 다만, 발급 이력이 대량으로 쌓였을
때 정적 인덱스를 활용하기 모호해지며, DTO로 변환하는 시점에 매번 시각 비교 연산이 실행됩니다.
• 선택지 B: 배치 스케줄러 기반 물리 상태 동기화
• 내용: 주기적인 스케줄러 배치 작업(예: 매시간 또는 매일 자정)을 운용하여, 만료 시각이 지난 쿠폰의 물리적
상태를 DB 테이블 상에서  EXPIRED 로 변경합니다. 조회 시에는  status  컬럼 조건으로 단순 인덱스 조회만
수행합니다.
• 영향: 대용량 트래픽 조건에서 단순한 인덱스 스캔을 적용할 수 있으므로 조회 성능이 매우 우수합니다. 하지만 배치
주기 사이의 미세한 오차 시간 동안에는 만료 시각이 넘었어도 DB에는  AVAILABLE  상태로 남아있을 수 있으므로,
주문(사용) 요청 시점에 만료일 필드를 한 번 더 검증하는 방어 로직이 필수적입니다.

### ⚖️ 3가지 설계 방식 비교
#### 1. 선택지 A: Coupon 도메인(Entity) 내부 처리
• 구현:  CouponIssue  또는  CouponTemplate  엔티티 내에  calculateDiscountAmount(orderAmount)  구현.
• 장점: 객체지향적인 캡슐화가 가장 잘 이루어집니다. 본인의 데이터(할인율, 최소금액 등)를 가지고 본인이 계산하므로
응집도가 높습니다.
• 단점: 계산을 위해 주문 금액( orderAmount )을 외부에서 주입받아야 하며, 만약 나중에 "특정 브랜드의 상품에만
적용되는 쿠폰"처럼 주문의 상세 정보(상품 ID, 브랜드 ID 등)가 필요해지면, Coupon 엔티티가 Order 도메인의 복잡한
구조를 알아야 하는 결합도가 발생합니다.

#### 2. 선택지 B: Order 도메인(Service/Entity) 내부 처리

• 구현:  OrderService 에서 직접 쿠폰의 값을 읽어와 수식을 계산.
• 장점: 주문 전체의 결제 금액이 산출되는 흐름 속에서 직관적으로 파악하기 좋습니다.
• 단점: 쿠폰 정책(정액/정률 등)이 바뀔 때 주문 도메인의 핵심 로직이 흔들리며, 단일 책임 원칙(SRP)을 위배하여 주문
서비스의 덩치가 불필요하게 커집니다.

#### 3. 선택지 C (제안):  DiscountService  (도메인 서비스) 도입

• 구현: 두 도메인에 속하지 않는 독립적인  DiscountService  클래스를 정의하고, 주문 금액과 쿠폰 정보를 넘겨받아 할인
조건 검증 및 최종 할인 금액을 산출.
@Component
class DiscountService {
fun calculateDiscount(coupon: CouponIssue, orderAmount: BigDecimal): BigDecimal {
// 1. 최소 주문 금액 조건 검증 (minOrderAmount)
// 2. 쿠폰 타입별(정액/정률) 할인액 계산
// 3. 정률일 경우 최대 할인 한도(maxDiscountAmount) 적용
// 4. 최종 할인 금액 반환
}
}

• 장점 (결합도 완화 및 확장성):
• 도메인 간 결합 최소화: Coupon 엔티티가 Order 구조를 몰라도 되고, Order 엔티티도 Coupon의 구체적인 할인
공식(정액/정률 분기 등)을 알 필요가 없어집니다.
• 할인 정책의 확장성: 추후 쿠폰 외에 "회원 등급별 할인", "기간 한정 이벤트 브랜드 할인", "장바구니 묶음 할인"
등 다양한 할인 정책이 추가될 때, 주문이나 쿠폰 코드를 수정할 필요 없이  DiscountService  내에서 할인 규칙들을
조합하여 최종 할인액을 구하는 구조로 유연하게 확장할 수 있습니다.
• 단점: 현재 구조와 같이 단순 쿠폰 할인만 존재하는 상황에서는 클래스 파일이 하나 늘어나는 아키텍처적 부담(다소 과한
추상화)이 있을 수 있습니다.

### ❓ 질문 4. 재고 차감 시 데드락 방지를 위해 상품 ID 정렬 방식을 적용하고, 단일 트랜잭션으로 처리하는 방향이
맞을까요?
• 선택지 A: 단일 트랜잭션 내 처리 + 상품 ID 오름차순 정렬 후 비관적 락 획득 (추천)
• 내용:  OrderFacade 가 트랜잭션을 시작하고, 여러 상품을 동시에 주문할 때 발생할 수 있는 데드락(예: 유저 1은
상품 1->2 순서로, 유저 2는 상품 2->1 순서로 락을 요청하여 서로 대기하는 상황)을 막기 위해 상품 ID(PK) 기준으로
오름차순 정렬한 뒤 순차적으로 재고 비관적 락( SELECT FOR UPDATE )을 획득합니다. 그 후 쿠폰 처리(낙관적 락)와
주문 저장을 거쳐 커밋합니다.
• 영향: 가장 확실하게 데드락을 예방하고, 재고가 부족하거나 쿠폰 사용이 실패했을 때 트랜잭션 롤백 메커니즘을
통해 완벽한 원자성을 보장합니다.
• 선택지 B: 락 순서 정렬 없이 즉시 획득 및 락 획득 실패 시 재시도(Retry) 처리
• 내용: 요청 들어온 순서대로 락을 획득하며, 데드락이 발생하여 데이터베이스 예외가 터지면 애플리케이션 단에서
이를 캐치해 주문 요청을 재시도(Retry)하거나 에러를 반환합니다.
• 영향: 구현은 단순해 보이지만 데드락 발생 빈도가 높아지며, 재시도 로직으로 인해 시스템 전체 부하가 가중될
위험이 있습니다.

### 1. 현업에서 10건 미만에 Batch Insert를 잘 쓰지 않는 이유
#### ① JPA와 MySQL  IDENTITY  전략의 한계

• 대부분의 국내 자바 백엔드 환경에서는 Spring Data JPA + MySQL/MariaDB 조합을 많이 사용하며, PK 생성 전략으로
GenerationType.IDENTITY (Auto-increment)를 선택합니다.
• 하지만 Hibernate는  IDENTITY  전략을 사용할 경우, DB에 쓰기(Insert)를 실행해야만 ID 값을 알 수 있기 때문에 영속성
컨텍스트 쓰기 지연 및 Batch Insert를 비활성화합니다.
• 이를 우회하여 Batch Insert를 강제하려면  JdbcTemplate 을 따로 사용해 직접 SQL을 작성하거나, PK 생성 전략을
Sequence 나  Table  등으로 변경해야 합니다. 10건 미만의 저장을 위해 이처럼 구현 복잡도를 높이는 것은 득보다 실이
큽니다.

#### ② 미미한 성능 이점 (조기 최적화 방지)

• Batch Insert의 핵심 이점은 네트워크 왕복 횟수(Network Round-trip)의 감소입니다.
• 데이터가 3~5건인 경우, 단건  INSERT  5번과 Batch  INSERT  1번의 차이는 밀리초(ms) 단위 수준이며 전체 주문 API
성능에 미치는 영향이 극히 미미합니다.
• 현업에서는 코드의 가독성, 유지보수 편의성, 표준 JPA 활용(예: 단순히  orderItemRepository.saveAll(items)  호출)을
우선시합니다.

### 2. 그렇다면 Batch Insert는 현업에서 언제 사용할까요?

실무에서 Batch Insert는 다음과 같이 데이터의 물리적 크기가 크거나 대량 처리가 발생하여 네트워크 지연이 병목이 되는
시점에 적극적으로 도입합니다.

• 대용량 파일 업로드 및 데이터 동기화: 엑셀/CSV 업로드 처리, 외부 API 연동을 통한 파트너사 상품 등록 (수백~수만 건
이상).
• 배치(Batch) 프로세스: 야간 정산 작업, 일괄 상태 변경, 통계 집계 등 수천 건 이상의 데이터를 주기적으로 저장하는
작업.
• B2B 주문 시스템: 대량의 자재나 부품을 발주하는 B2B 이커머스 등, 한 번의 주문에 수백~수천 개의 품목이 포함되는
경우.
• 장바구니 일괄 주문 전환: 장바구니에 담긴 수십 개의 상품을 한 번에 주문으로 전환할 때 성능 테스트 결과 병목이
확인된 경우.

결론부터 말씀드리면, 질문하신 대로 생명주기(Life Cycle)를 기준으로 보았을 때  CouponTemplate 과  CouponIssue 는
서로 다른 애그리거트(Aggregate)로 분리하는 것이 맞습니다.
현업 및 도메인 주도 설계(DDD) 관점에서 이를 같은 애그리거트로 묶지 않는 구체적인 이유는 다음과 같습니다.
──────
### 1. 생명주기(Life Cycle)와 주체의 분리
•  CouponTemplate  (쿠폰 템플릿):
• 생성 주체: 관리자(Admin)
• 생명주기: 쿠폰 정책이 기획/등록되는 시점에 생성되어 수정 및 만료 처리될 때까지 장기적으로 유지됩니다. 개별
고객의 행동과 무관하게 존재합니다.
•  CouponIssue  (쿠폰 발급 내역):
• 생성 주체: 일반 고객(User)
• 생명주기: 특정 고객이 쿠폰 발급 버튼을 누르는 시점에 생성(발급)되며, 주문 결제 시 사용( USED )되거나 기간
만료( EXPIRED )되는 상대적으로 짧고 동적인 생명주기를 가집니다.
• 결론: 두 객체는 생성 시점, 변경 시점, 소멸 시점이 완전히 다르고 행위의 주체도 다르므로 생명주기가 일치하지
않습니다.
──────
### 2. 트랜잭션 경계와 동시성 제어 (가장 중요한 현실적 이유)

DDD에서 애그리거트는 **"트랜잭션 일관성을 유지해야 하는 최소한의 단위"**입니다. 만약 두 객체를 하나의 애그리거트로
묶는다면 애그리거트 루트(Aggregate Root)는  CouponTemplate 이 되어야 합니다.

• 동시성 병목 발생:
• 동일한 쿠폰 템플릿(예: 선착순 1,000명 쿠폰)에 대해 수많은 사용자가 동시에 발급을 요청할 때, 하나의 애그리거트
안에서 변경이 일어나야 하므로  CouponTemplate  로우(Row)에 대한 수정 권한이나 락(Lock)을 획득해야 합니다.
• 이로 인해 DB 락 경합이 극도로 심해져 쿠폰 발급 성능이 심각하게 저하되거나 데드락/낙관적 락 충돌이 대량
발생하게 됩니다.
• 애그리거트 분리를 통한 해결:
•  CouponTemplate 과  CouponIssue 를 독립된 애그리거트로 분리합니다.
• 발급 시  CouponTemplate 은 단순 조회(Read-Only)만 하거나 Redis 등을 이용해 잔여 수량만 빠르게 검증하고, 실제
쓰기(Write) 작업은 개별 고객의  CouponIssue  엔티티를 독립적으로 데이터베이스에  INSERT  합니다.
• 이를 통해 동시성 문제를 격리하고 처리량을 극대화할 수 있습니다

할인 정책(비즈니스 룰)을 데이터베이스 데이터로 관리하는 방식(현재 구조)과 순수하게 애플리케이션 코드로 표현하는
방식은 아키텍처 관점에서 뚜렷한 장단점(Trade-off)을 가집니다.

두 방식의 특징을 설계 리뷰 관점에서 비교 분석하여 정리해 드립니다.
──────
### 1. DB 기반 관리 (현재 구조)
할인의 타입( FIXED / RATE ) 및 한도, 조건 등 비즈니스 정책의 핵심 조건(Parameter)들을 03-class-diagram.md 테이블에
저장하여 동적으로 결합하는 방식입니다.
• 장점 (운영 유연성 및 정산 관리):
• 무배포 실시간 적용: 마케팅 부서에서 새로운 이벤트 쿠폰(예: "신년 맞이 15% 할인")을 기획했을 때, 개발자의 코드
변경이나 서버 재배포 없이 어드민 API 혹은 DB Insert만으로 즉시 적용이 가능합니다.
• 정확한 데이터 정산(Auditing): 할인 조건이 DB에 영속화되어 있으므로, 과거에 어떤 정책 조건으로 쿠폰이 발급되어
사용되었는지 관계형 모델을 통해 쉽게 통계 및 정산을 낼 수 있습니다.
• 단점 (비즈니스 복잡도의 제한):
• 조건이 다양해질 때의 테이블 비대화: "첫 구매 유저 중 특정 카테고리 상품을 2개 이상 샀을 때 적용되는 할인"처럼
조건이 복잡해질수록 이를 표현하기 위한 DB 컬럼이 기하급수적으로 늘어나거나 JSON 필드 등을 남용하게 되어 DB
정합성이 깨질 수 있습니다.
• 런타임 에러 취약성: 운영자가 어드민 툴에서 할인율을 실수로  10%  대신  100% 로 잘못 입력하는 등의 데이터 입력
휴먼 에러가 즉시 비즈니스 장애로 이어집니다.

──────
### 2. 애플리케이션 코드 기반 관리 (예: 전략 패턴 적용)

DiscountPolicy  인터페이스를 두고, 구체적인 할인 클래스들( FixedDiscountPolicy ,  RateDiscountPolicy ,
FirstOrderDiscountPolicy  등)을 애플리케이션 코드로 구현하여 쿠폰 발행 시 해당 정책 식별자(ID)만 매핑하는
방식입니다.

• 장점 (설계 유연성 및 안전성):
• 복잡한 정책의 정교한 캡슐화: 프로그래밍 언어의 다형성, 조건문, 풍부한 도메인 표현력을 100% 활용할 수 있어,
매우 복잡한 조건(예: 회원 등급별 + 카테고리 교차 할인)도 가독성 높고 안전하게 코드로 구조화할 수 있습니다.
• 컴파일 시점 검증 및 테스트 용이성: 로직이 코드로 완벽하게 격리되어 있어 단위 테스트(Unit Test)를 통해 경계값
오류를 철저히 검증할 수 있으며, 빌드 단계에서 실수를 예방할 수 있습니다.
• 단점 (잦은 배포와 유지보수성 저하):
• 정책 변경 시 재배포 필수: 단순한 할인 금액 조정(예:  1,000원  ➡️  1,500원 )이나 조건 변경이 일어날 때마다
코드를 수정하고, 빌드 및 테스트를 수행한 후 실 서버를 재배포해야 하므로 비즈니스 피드백 루프가 느려집니다.
• 과거 정책의 이력 추적성 저하: 소스 코드 상에서 과거에 쓰이던 할인 정책 클래스를 리팩토링하거나 삭제하면, 과거
주문 이력을 정확히 재현하기 어렵습니다. 이를 방어하기 위해 주문 정보에 가격 및 할인 스냅샷을 영속화하는 로직이
더 강력하게 강제됩니다.

> 외부 결제 모듈을 사용하는 경우 결제 도메인을 꼭 정의해야 할까?

▸ Thought Process
Considering Domain Definitions
외부 결제 모듈(PG사)을 사용하더라도 내부에 결제( Payment ) 도메인을 명시적으로 정의할지 여부는 비즈니스의 복잡도와
정산 요구사항에 따른 장단점(Trade-off)이 뚜렷하게 나뉩니다.

반드시 정의해야 하는 것은 아니며, 설계 시 아래 두 가지 선택지를 고려할 수 있습니다.
──────
### 선택지 A: 결제 도메인을 정의하지 않는 방식 (주문서 통합형)
결제 테이블을 별도로 두지 않고, 04-erd.md 테이블 내에 PG 승인 번호( pg_transaction_id ), 결제 수단, 결제 완료
시점 등의 메타데이터 컬럼들을 추가하여 관리하는 방식입니다.

• 장점 (극도의 단순함):
• 새로운 엔티티나 테이블을 생성하고 조인할 필요가 없어 데이터 모델이 가벼워지고 데이터 조회 쿼리가 단순해집니다.
• 주문 상태를  COMPLETED 로 변경하면서 결제 식별자를 동시에 쓰기 때문에, 두 테이블 간의 일관성이 깨질 위험이
원천 차단됩니다.
• 단점 (확장성 부재):
• 복합 결제 불가: 한 주문에 대해 "포인트 + 신용카드" 혹은 "쿠폰 + 간편결제"와 같이 여러 결제 수단을 혼합하여
지불하는 구조를 표현하기 매우 어렵습니다.
• 부분 취소 및 환불 관리의 한계: 주문한 상품 3개 중 1개만 부분 취소하여 일부 금액만 환불해야 하는 상황이
발생하면, 환불 이력을 주문서 한 줄만으로 추적하기 어려워 데이터 모델이 복잡해지기 시작합니다.
• 결제 재시도 이력 분실: 사용자가 카드 한도 초과 등으로 첫 번째 결제에 실패하고 다른 수단으로 성공했을 때,
실패한 시도와 성공한 시도에 대한 이력(Audit Log)을 역추적할 수 없습니다.

──────
### 선택지 B: 결제 도메인을 분리 정의하는 방식 (독립 도메인형 - 추천)

주문과 별개로  Payment  엔티티를 독립적인 생명주기를 가진 도메인으로 설계하는 방식입니다.

• 장점 (비즈니스 확장성과 정산 신뢰성):
• 단일 책임 원칙(SRP): 주문은 "어떤 상품을 살 것인가"에 집중하고, 결제는 "돈을 어떻게 치불할 것인가"에만
집중하여 도메인 간 결합도가 완화됩니다.
• 다양한 결제 흐름 대응: 다중 결제 수단 적용, 부분 환불 이력 누적 관리, 결제 실패 및 취소 로그 보존 등이
유연하게 가능해집니다.
• 정산 대조(Reconciliation) 용이: 매일 자정 PG사의 정산 데이터와 내부 시스템의 결제( Payment ) 레코드를
대조하여 금액이 맞는지 검증하는 정산 배치나 감사 작업을 훨씬 안정적으로 설계할 수 있습니다.
• 단점 (아키텍처 비용):
• 테이블 간 외래키 매핑이 추가되고, 주문과 결제 생성 간의 원자적 정합성(두 테이블에 데이터가 모두 정상
입력되는지)을 관리하는 개발 비용이 발생합니다.

──────
### 💡 설계 권장안 및 요약

• 현재 및 단기 요구사항이 단순한 경우 (단일 수단 결제, 단순 전체 취소만 존재):
• 비대해진 아키텍처를 지양하는 관점에서 **선택지 A(주문 테이블에 통합)**가 효율적입니다.
• 장기적으로 포인트 복합 결제, 부분 환불, PG사 데이터와의 일일 정산 자동화가 계획된 경우:
• 처음부터 **선택지 B(결제 도메인 신설)**를 설계하여 도메인 간의 책임을 확실히 나누는 것이 장기적인 유지보수
관점에서 훨씬 유리합니다.

> 이벤트 기반 서버가 아닌데도 보상 트랜잭션이라는 걸 사용할 수 있어?

▸ Thought Process
Considering Transaction Applicability
네, 이벤트 기반 아키텍처가 아니더라도 보상 트랜잭션(Compensating Transaction) 개념은 얼마든지 사용할 수 있으며,
실제로 동기식 웹 서버에서도 매우 흔하게 사용됩니다.
보상 트랜잭션의 본질은 아키텍처의 형태(동기/비동기)가 아니라, **"이미 되돌릴 수 없이 완료(커밋)된 물리적 작업을
논리적으로 취소(롤백)하기 위한 반대 작업"**이기 때문입니다.

동기식 서버에서 보상 트랜잭션이 작동하는 방식과 이벤트 기반 아키텍처와의 차이점을 설명해 드립니다.
──────
### 1. 동기식(MVC 등) 아키텍처에서의 보상 트랜잭션 작동 방식

외부 PG사를 사용하는 결제 확정 시나리오에서 동기식 보상 트랜잭션은 주로  try-catch  블록 내의 예외 처리 흐름으로
구현됩니다.
// PaymentFacade 내 동기식 보상 트랜잭션 예시
public void confirmPayment(Long orderId, String pgToken) {
// 1. 외부 PG 승인 (되돌릴 수 없는 물리적 커밋)
PgResponse pgResult = pgClient.approve(pgToken);

        try {
            // 2. 내부 데이터베이스 작업 (재고 차감, 주문 상태 COMPLETED 변경)
            // 이 안에서 예외(재고 부족, DB 다운 등)가 발생하면 JPA 트랜잭션은 롤백됩니다.
            orderPaymentService.completeOrderAndDecreaseStock(orderId, pgResult);

        } catch (Exception e) {
            // [동기식 보상 트랜잭션 작동]
            // 내부 DB 작업은 롤백되었지만 외부 PG에 지불된 돈은 남아있으므로,
            // catch 블록에서 즉시 외부 PG 환불 API를 동기식(HTTP)으로 호출하여 취소합니다.
            pgClient.refund(pgResult.getApprovedId(), "내부 재고 부족으로 인한 자동 환불");

            throw new CoreException(ErrorType.SYSTEM_ERROR, "주문 처리 중 오류가 발생하여 결제가 자동 취소되었습니다.
");
}
}

• 동작 원리: 데이터베이스 트랜잭션( @Transactional )은 내부 DB에만 영향력을 미치기 때문에, 외부 시스템(PG)의 변경
사항을 되돌리기 위해 코드를 통해 명시적인 취소 API(보상 트랜잭션)를 동기 호출하는 것입니다.
──────
### 2. 동기식 vs 이벤트 기반(비동기) 보상 트랜잭션 비교

   구분          | 동기식 보상 트랜잭션 (예외 처리형)              | 이벤트 기반 보상 트랜잭션 (Saga 패턴 등)
  ---------------|-------------------------------------------------|-------------------------------------------------
   호출 아키텍처 | API 요청 스레드 내에서 즉시 동기(HTTP) 호출     | 메시지 큐(Kafka 등)의 실패 이벤트를 구독하여
|                                                 | 비동기 처리
   장점          | 구조가 단순하며, 고객이 즉시 결제 취소 여부를   | 특정 서버 장애 시에도 큐에 메시지가 보관되어
| 화면에서 확인할 수 있음                         | 언젠가 반드시 취소가 실행됨 (신뢰성 높음)
   단점          | 보상 트랜잭션 호출 자체의 실패 대책이 필요함    | 최종 일관성(Eventual Consistency)을 지향하므로
| (예: 환불 API 호출 중 타임아웃 발생 시 대응법)  | 일시적인 데이터 불일치 구간이 발생하며 구현
|                                                 | 복잡도가 높음
──────
### 💡 동기식 보상 트랜잭션 적용 시 추가 고려사항
앞서 제안해 드린 order_payment_separation_plan.md의 결제 승인 API 다이어그램에서도 동기식 보상 트랜잭션 흐름을
표현하고 있습니다.

다만, 만약 환불 API(보상 트랜잭션)마저 네트워크 일시 장애 등으로 실패하는 극단적인 상황에 대비하기 위해, 실무에서는
다음 대안을 상호 보완적으로 함께 사용합니다.

• 보안책: 결제 승인이 났으나 내부 주문 처리에 실패한 내역을 별도의  PaymentHistory(status=PENDING_REFUND)  상태로
DB에 기록해 두고, 배치 스케줄러가 정기적으로 돌며 PG사의 실 거래 내역과 대조해 미환불 건을 자동 환불(보상 트랜잭션
재시도) 처리하도록 설계합니다.

────────────────────────────────────────────────────────────

### Q1. [정책] 재고 차감 및 락 획득 시점

주문과 결제가 분리됨에 따라, 상품의 재고를 어느 시점에 차감하고 락(Lock)을 획득할지 결정해야 합니다.

• 선택지 A: 주문 생성 시점에 재고 가선점 (추천)
• 설명: 주문 생성 API 요청 시 재고를 차감하고, 일정 시간(예: 30분) 내에 결제가 완료되지 않으면 주문을 취소하고
재고를 다시 채워 넣습니다.
• 장점: 사용자가 결제 과정을 진행하는 도중에 다른 사람이 재고를 채가서 결제가 실패하는 불쾌한 경험(결제 승인
단계에서의 품절)을 방지할 수 있습니다.
• 단점: 결제를 안 하고 이탈한 '좀비 주문'의 재고 복구를 위해 만료 스케줄러 등을 구현해야 하므로 시스템 복잡도가
증가합니다.
• 선택지 B: 결제 승인 시점에 재고 차감
• 설명: 주문서 생성 시에는 재고를 건드리지 않고, 실제 카드/계좌 이체 승인 처리가 수행되는 시점에 재고를
차감합니다.
• 장점: 결제 시간 만료 처리나 재고 복구 배치를 구현할 필요가 없어 구현이 매우 단순합니다.
• 단점: 장바구니에서 주문서를 작성하고 카드 번호까지 다 입력한 뒤 승인 단추를 눌렀을 때, 동시 주문으로 인해
재고 부족으로 결제가 튕길 수 있습니다.

### Q2. [경계] 쿠폰 검증 및 적용 시점 (금액 계산의 주체)
주문과 결제가 분리되었을 때, 할인 혜택(쿠폰)을 적용하고 최종 결제 금액을 확정하는 책임을 어디에 둘지 결정해야
합니다.

• 선택지 A: 주문 생성 시점에 쿠폰 적용 (추천)
• 설명: 주문 생성 API 요청 시 사용할 쿠폰 ID를 받아 검증한 뒤, 적용 금액 및 최종 결제 금액( totalPaymentAmount
)을 계산하여 주문서에 확정(스냅샷)합니다. 결제 도메인은 주문서에 확정된 최종 금액에 대해서만 결제 승인 요청을
처리합니다.
• 장점: 주문 단계에서 금액 계산과 검증이 모두 완결되므로 결제 도메인이 쿠폰 비즈니스 규칙을 알 필요가 없어
도메인 간 결합도가 낮아집니다.
• 단점: 결제 단계로 진입한 후에는 사용자가 쿠폰을 바꾸고 싶어도 변경이 불가능하며, 주문을 취소하고 다시
주문해야 합니다.
• 선택지 B: 결제 요청 시점에 쿠폰 적용
• 설명: 주문서 생성 시에는 상품 가격의 원가로만 주문을 넣고, 실제 결제 요청 API를 보낼 때 결제 수단과 사용할
쿠폰 ID를 함께 받아 최종 결제액을 결정합니다.
• 장점: 사용자가 결제 화면에서 마음대로 쿠폰을 적용하거나 변경해 볼 수 있어 결제 시점의 유연성이 높습니다.
• 단점: 주문 금액 정보가 유동적으로 변하게 되므로 결제 승인 전후로 주문서와 결제 정보의 일관성을 검증하는
로직이 복잡해집니다.

### Q3. [확장] 외부 결제 연동 인터페이스 설계 방식

나중에 실제 PG사(토스페이먼츠, KG이니시스 등) 결제 승인 API를 호출하게 될 텐데, 현재 모킹(Mocking) 단계를 어떻게
설계할지에 대한 질문입니다.

• 선택지 A: DIP(의존성 역전 원칙) 기반 인터페이스 설계 (추천)
• 설명: 도메인 영역에  PaymentGateway 라는 인터페이스(포트)를 정의하고, 이를 구현하는 가짜 구현체
MockPaymentGateway 를 외부 인프라 영역에 만듭니다.
• 장점: 객체지향적인 설계(DIP)를 준수합니다. 나중에 실제 PG 연동 모듈을 개발할 때 비즈니스 로직이 담긴 서비스
코드는 전혀 건드리지 않고, 새로운 실제 PG사 구현체(예:  TossPaymentGateway )를 구현해 갈아 끼우기만 하면 됩니다.
• 단점: 인터페이스 구조를 설계하고 구현 클래스를 분리해야 하므로 초기 설계 비용이 약간 더 듭니다.
• 선택지 B: 단순 서비스 내부 가짜(Dummy) 로직 작성
• 설명: 별도의 인터페이스나 클래스 분리 없이, 결제 처리를 담당하는  PaymentService  클래스 내부 메소드에서
무조건 성공 결과를 반환하도록 하드코딩하거나 간단한 로그만 찍습니다.
• 장점: 추가 클래스나 인터페이스 정의가 없어 빠르게 개발할 수 있습니다.
• 단점: 향후 실제 외부 결제를 도입할 때 비즈니스 서비스 로직을 직접 수정해야 하므로 영향 범위가 넓어집니다.

• 주문 생성 flow 축소 & 결제 승인 flow 신설
• Before: 주문 생성 API 트랜잭션 하나 안에서  상품조회  ->  쿠폰사용(USED)  ->  재고차감(Pessimistic Lock)  ->
주문등록 을 모두 실행.
• After:
1. 주문 생성 API:  상품조회  ->  쿠폰유효성 검증(금액확정)  ->  재고가선점(차감)  ->  주문등록(PENDING)  후
트랜잭션 종료.
2. 결제 승인 API (신설):
• 트랜잭션 외부에서  PaymentGateway 를 통해 외부 PG사에 결제 승인 요청.
• 결제 승인 성공 시 트랜잭션을 열어  PAYMENTS  데이터 저장 ->  ORDERS  상태 변경( COMPLETED ) ->
COUPONS  상태 사용 완료( USED ) 처리.
• DB 반영 중 낙관적 락 등의 오류 발생 시 외부 PG사에 결제 취소(보상 트랜잭션) API를 연쇄 호출하는 예외
처리 흐름을 묘사.
   상품의 좋아요 수( like_count )와 같은 역정규화 컬럼을 동시성 환경에서 가감할 때 사용하는 비관적 락 방식과 원자적
   Update 쿼리 방식을 비교해 드립니다.

두 방식은 성능과 정합성 유지 관점에서 뚜렷한 트레이드오프를 가집니다.

### 📊 비관적 락 vs 원자적 Update 쿼리 비교

   비교 항목              | 비관적 락 (SELECT FOR UPDATE)              | 원자적 Update (like_count = like_count + 1)
  ------------------------|--------------------------------------------|---------------------------------------------
   기본 메커니즘          | SELECT 시점에 레코드 쓰기 락을 획득하고    | SELECT 시점에는 락을 잡지 않음. 쿼리가
| 트랜잭션이 끝날 때까지 락 유지.            | 실행되는 아주 짧은 순간 DB 엔진의 자체
| 애플리케이션 메모리에서 값을 변경한 후     | Lock(Latch)을 이용하여 값을 증가시킴.
| 더티 체킹으로 반영.                        |
   동시성 및 처리량       | 낮음 (동일 상품에 좋아요가 몰릴 경우       | 매우 높음 (락 점유 시간이 극도로 짧아 동시
| 트랜잭션 대기열이 길어지고 병목 발생)      | 처리 성능이 우수함)
   데드락(Deadlock) 위험  | 비교적 높음 (여러 트랜잭션이 상품          | 거의 없음 (단일 행에 대한 짧은 업데이트로
| 조회/주문 등과 얽힐 경우 발생 가능)        | 안전함)
   영속성 컨텍스트 동기화 | 자동 동기화 (JPA 영속성 컨텍스트 내부      | 동기화 안 됨 (JPA를 건너뛰고 DB 값을
| 엔티티의 상태와 DB 상태가 완벽히 일치함)   | 변경하므로, 1차 캐시의 엔티티 값은 과거
|                                            | 값으로 남음.  clear 나  refresh  필요)
   비즈니스 규칙 검증     | 매우 용이 (현재 값을 조회하여 음수 방지,   | 제한적 (쿼리 조건절  WHERE like_count > 0
| 유효 범위 확인 등을 메모리 상에서 안전하게 | 등 DB 수준의 조건 제한만 가능)
| 검증 가능)                                 |

### 🔍 구체적인 동작 방식 차이

#### 1️⃣ 비관적 락 (Pessimistic Lock)

    // 1. SELECT ... FOR UPDATE 쿼리 발생 (락 획득)
    Product product = productRepository.findByIdForUpdate(productId);

    // 2. 애플리케이션 메모리 상에서 값 검증 및 계산
    product.increaseLikeCount();

    // 3. 트랜잭션 종료 시점에 Dirty Checking으로 UPDATE 쿼리 실행 후 락 해제

• 특징: 상품 엔티티를 영속성 컨텍스트로 온전히 가져온 뒤 안전하게 비즈니스 로직을 태웁니다.
• 추천 상황: 좋아요 증감 뿐만 아니라, 상품의 상태나 재고처럼 엄격한 규칙(예: 음수 불가능, 특정 등급 이상만 가증
등)을 애플리케이션 레벨에서 꼼꼼히 체크해야 할 때 적합합니다.

#### 2️⃣ 원자적 Update 쿼리 (Atomic Update)

    // JPA 벌크 연산 또는 JPQL 실행 (SELECT 없이 바로 반영)
    productRepository.incrementLikeCount(productId);

• 특징: DB의 원자적 연산( like_count = like_count + 1 )을 사용하므로, 동시에 100명이 누르더라도 소실(Lost Update)
없이 정확히 100이 증가합니다.
• 추천 상황: 대규모 이벤트 등으로 특정 인기 상품에 초당 수천 건의 좋아요가 발생하여 처리량(Throughput)이 극도로
중요할 때, 그리고 단순 1씩 더하고 빼는 정수 연산만 요구될 때 매우 유용합니다.

### 💡 본 프로젝트의 상황 분석 및 설계 권장안

현재  01-requirements.md 의 **"2.5 좋아요(Likes) 및 정렬 정책"**을 보면 다음과 같은 규칙이 명시되어 있습니다.

│ *"이미 좋아요를 누른 상품에 대해 중복 요청이 올 경우 ... **비관적 락으로 상품을 선점한 즉시 좋아요 이력을
│ 선조회(Exist Check)하여 분기(멱등성 리턴)*해야 한다."

이 멱등성 비즈니스 조건으로 인해 다음과 같은 트레이드오프가 존재합니다.

1. 현재 기존 설계 (비관적 락 유지):
   • 좋아요 이력이 있는지 검사( Exist Check )하고 상품의  like_count 를 수정하는 전체 과정을 하나의 락( FOR UPDATE
   )으로 묶어 멱등성과 정합성을 단순하게 보장합니다.
2. 원자적 Update 쿼리로 변경할 경우:
   •  Exist Check 는 락 없이 조회하여 중복 등록 여부를 판단한 뒤, 이력이 없으면  INSERT Like  +  UPDATE product
   SET like_count = like_count + 1 을 실행해야 합니다.
   • 이 경우, 아주 미세한 찰나의 순간에 동일 유저가 두 번 클릭하면 중복  INSERT 가 일어날 수 있으므로, **
   PRODUCT_LIKES  테이블에  (user_id, product_id)  복합 유니크 제약조건(Unique Key)**을 걸어 DB 수준에서 멱등성을
   강제해 주어야 합니다.


따라서 동시 접속자가 아주 많고 성능 향상이 목표라면,  PRODUCT_LIKES  테이블에 Unique Key를 단단히 걸고 원자적
Update 쿼리 방식으로 전환하는 것이 성능적으로 훨씬 유리한 선택이 될 수 있습니다.

주문 요청 API와 결제 승인 API를 하나로 합치지 않고 2단계(2-Phase)로 분리한 이유는 현대적인 이커머스 시스템의
인프라적 제약과 웹/앱 클라이언트의 결제 흐름 때문입니다.
크게 4가지 관점에서 설명해 드릴 수 있습니다.
──────
### 1️⃣ 외부 시스템(PG사)의 네트워크 지연 및 DB 커넥션 고갈 방지

• 하나의 API에서 처리할 때의 문제:
1. 내부 DB 트랜잭션을 시작하고 재고에 비관적 락( FOR UPDATE )을 획득합니다.
2. 그 상태에서 외부 PG사(토스, 이니시스 등) 승인 API를 호출합니다.
3. 만약 외부 PG사 응답이 네트워크 장애로 인해 3~5초 이상 지연(Timeout)된다면, 그 시간 동안 DB 커넥션과 재고
락이 묶여있게 됩니다.
4. 이는 순식간에 DB 커넥션 풀 고갈로 이어져, 결제와 전혀 상관없는 다른 사용자들까지 시스템을 이용하지 못하게
만듭니다.
• 분리 시 해결 방법: 주문 API는 내부 DB 작업만 빠르게 끝내고 락을 놓아줍니다. 이후 결제 승인 API에서는 외부 PG 승인
요청을 **내부 DB 트랜잭션이 시작되기 전(혹은 격리된 상태)**에서 처리하므로 외부 지연이 내부 DB 정합성이나 성능을
위협하지 않습니다.
──────
### 2️⃣ 현대 이커머스 PG SDK(결제창)의 클라이언트 흐름과의 정합성

현업에서 사용하는 외부 결제창(토스페이먼츠, 포트원 등)은 다음과 같은 순서로 작동합니다.

1. 주문서 생성: 고객이 '결제하기'를 누르면 백엔드로 주문 생성 요청을 보냅니다. 백엔드는 주문 ID와 최종 결제 금액을
   확정하여 응답합니다. (이때 주문 상태는  PENDING )
2. 결제창 호출: 프론트엔드는 백엔드로부터 받은 주문 ID와 확정 금액을 외부 결제 SDK에 넘겨주며 카드 인증창을
   띄웁니다.
3. 인증 완료 및 승인 요청: 고객이 카드 비밀번호를 입력하여 인증을 완료하면, PG사로부터 가승인 키(예:  paymentKey
   )가 프론트엔드로 전달됩니다.
4. 최종 승인: 프론트엔드는 이 인증 키를 백엔드의 결제 승인 API로 전송하고, 백엔드가 최종적으로 PG사에 승인 요청을
   넣어 결제를 완료합니다.

이 브라우저/앱 단의 인증 단계(2번, 3번)가 중간에 들어가야 하기 때문에, 백엔드는 결제창을 띄우기 전의 단계와 결제창
인증 후 승인하는 단계를 분리하여 API를 제공할 수밖에 없습니다.
──────
### 3 사용자 경험(UX) 개선 및 이탈률 방지

• 하나의 API일 때: 결제 카드 한도 초과, 인증 번호 오입력 등으로 결제가 한 번 실패하면 주문 정보 자체가 없어지므로,
사용자는 장바구니에 상품을 처음부터 다시 담아 주문을 처음부터 시작해야 합니다.
• 분리되어 있을 때: 결제 승인이 실패하더라도 주문 자체는  PENDING  상태로 저장되어 있습니다. 사용자는 장바구니를
다시 담을 필요 없이, 주문 내역 화면에서 "결제 수단을 카카오페이로 변경해서 재시도" 하거나 **"다른 카드로 결제
재시도"**를 바로 진행할 수 있어 이탈율이 획기적으로 줄어듭니다.
──────
### 4 보상 트랜잭션(Rollback) 및 예외 처리 통제 용이성

• 주문 생성과 외부 결제 승인을 별도 단계로 다루면, 특정 단계에서 오류가 났을 때 무엇을 원복해야 하는지가 매우
명확해집니다.
• 외부 결제 승인은 성공했으나 DB 쓰기에 실패한 경우: 외부 PG사에 결제 취소(보상 트랜잭션) API를 안전하게
전송합니다.
• 외부 결제가 실패한 경우: 재고 선점만 취소 처리하면 되고 외부 PG사에는 따로 취소 요청을 날릴 필요가 없습니다.

### 💡 요약

주문과 결제를 분리하는 것은 **"결제 대행사라는 외부 시스템과의 통신 시간 동안 우리의 소중한 DB 자원(락, 커넥션)을
대기 상태로 낭비하지 않겠다"**는 시스템 안정을 위한 필수적인 설계 판단이며, 클라이언트 결제창 연동의 물리적 흐름을
맞추기 위한 선택입니다.

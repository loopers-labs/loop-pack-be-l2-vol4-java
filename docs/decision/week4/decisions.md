# 4주차 아키텍처 및 정책 의사결정 내역 (통합본)

### 1. 쿠폰 템플릿 수정/삭제 시 기발급 쿠폰 보호를 위한 아키텍처 개편

#### 문제 정의
- **현재 동작/제약:** 쿠폰 발급(`CouponIssue`) 엔티티가 발급 시점의 혜택 데이터를 가지지 않고, 템플릿(`CouponTemplate`)의 ID만을 참조하여 쿠폰 결제 및 조회 시 매번 템플릿 정보를 다시 불러옴.
- **문제(또는 리스크):** 관리자가 템플릿의 할인 금액을 수정하면, 기존에 쿠폰을 정상적으로 발급받은 사용자의 혜택도 강제로 소급 변경됨(심각한 고객 클레임 유발). 또한, 템플릿 삭제 시 이미 쿠폰을 가진 사용자가 쿠폰을 쓰려 할 때 템플릿을 찾을 수 없어 런타임 에러(`NOT_FOUND`)가 발생하여 결제 시스템에 장애를 유발함.
- **성공 기준(완료 정의):** 기존 발급 쿠폰에 영향을 주지 않으면서, 새로운 발급 시에만 변경된 템플릿 정보가 반영되도록 아키텍처를 개편하고 테스트 검증을 완료할 것.

#### 선택지와 결정
- **고려한 대안:**
  - **A (참조 구조 유지 및 템플릿 불변 객체화):** `CouponIssue`는 기존처럼 템플릿 ID만 참조하되, 템플릿 자체의 수정을 정책적으로 금지. 템플릿 삭제 시엔 `status` 필드를 통해 논리적(Soft) 중지 상태로만 변경하여 결제 시 "관리자에 의해 중지됨"으로 우아하게 예외 처리.
  - **B (스냅샷 패턴 적용):** 쿠폰 발급 시점에 템플릿의 주요 혜택 정보(할인 타입, 금액, 조건, 만료일 등)를 `CouponIssue` 컬럼으로 직접 복사(스냅샷)하여 저장.
- **최종 결정:** **대안 B (스냅샷 패턴 적용)**
- **트레이드오프:** `CouponIssue` 테이블에 저장되는 데이터량이 늘어나지만(데이터 정규화 관점에서는 중복 발생), 템플릿이 변경되거나 삭제되더라도 **이미 발급된 쿠폰의 불변성을 완벽하게 보장**하여 운영 안정성과 고객 경험이 극대화된다는 비즈니스적 이득이 훨씬 큼.
- **추후 개선 여지(있다면):** 만약 향후 "기존 발급된 쿠폰들의 혜택까지 관리자가 일괄 수정/회수하고 싶다"는 요구사항이 들어오면, 스냅샷 구조 내에서 `CouponIssue` 대상 데이터들을 일괄 수정(Bulk Update)하는 관리자 전용 배치 API를 별도 개발해야 함.

### 2. 동시성 제어 및 락(Lock) 전략 (도메인별 하이브리드)

#### 문제 정의
- **현재 동작/제약:** 주문 시 재고 차감 및 쿠폰 사용 로직에 동시성 제어가 없으면 초과 판매되거나 동일 쿠폰이 여러 번 사용(Double Spending)될 위험 존재.
- **문제(또는 리스크):** 락을 잘못 사용할 경우, 정합성은 지켜지더라도 트랜잭션 대기열이 폭증하여 데드락이나 타임아웃에 의한 전체 시스템 장애로 이어질 수 있음.
- **성공 기준(완료 정의):** 데이터 정합성을 100% 보장하면서도 시스템 대기 시간을 최소화하는 최적의 락 전략 도입.

#### 선택지와 결정
- **고려한 대안:**
  - **A (비관적 락 일괄):** DB단 잠금으로 정합성은 확실하나 전체 트랜잭션 대기열 증가 우려.
  - **B (낙관적 락 일괄):** 대기 시간 제거로 기본 성능 향상, 단 잦은 충돌 시 사용자 실패율 및 Retry 비용 증가.
  - **C (도메인별 하이브리드 적용):** 각 도메인의 비즈니스 특성에 맞게 분리 적용.
- **최종 결정:** **대안 C (하이브리드 적용)**
  - 빈번한 경합이 예상되는 '재고 차감'은 비관적 락을 통해 대기 큐로 순차 처리 유도.
  - 드물게 발생하는 '쿠폰 중복 사용(어뷰징)'은 낙관적 락(Version)으로 즉각 예외를 던져 차단.
- **트레이드오프:** 도메인마다 다른 락 전략을 유지보수해야 하므로 개발 복잡도는 상승하나, 자원 효율성과 시스템 전체의 처리량(Throughput)이 극대화됨.

### 3. 주문 및 외부 결제(PG) 단일 트랜잭션 처리

#### 문제 정의
- **현재 동작/제약:** 주문 생성부터 결제 승인 요청, 재고 차감을 하나의 거대한 트랜잭션으로 묶을지, 외부 API 통신을 분리할지에 대한 고민.
- **문제(또는 리스크):** 기존에는 PG사 네트워크 지연 시 내부 DB 커넥션 고갈을 방지하기 위해 트랜잭션을 분리했으나, 이 경우 결제 실패 시 수동으로 DB 상태를 원복해야 하는 '보상 트랜잭션' 로직의 복잡도가 크게 증가함.
- **성공 기준(완료 정의):** 개발 생산성과 데이터 정합성을 가장 우선하여 코드 구조를 단순화하고, 트랜잭션 관리를 스프링 프레임워크에 위임함.

#### 선택지와 결정
- **고려한 대안:**
  - **A (주문/결제 단계 분리):** 내부 DB 작업(주문 상태 PENDING 생성) 후 트랜잭션 외부에서 결제 승인. 지연 격리는 되나 복잡한 롤백(보상 트랜잭션) 코드 필수.
  - **B (단일 트랜잭션 진행):** 주문 검증부터 재고 차감, 생성, PG사 승인, 완료 처리까지 전부 하나의 `@Transactional`로 묶어 처리. 예외 발생 시 프레임워크가 알아서 롤백.
- **최종 결정:** **대안 B (단일 트랜잭션 진행)**
- **트레이드오프:** 외부 PG사 지연이 DB 커넥션 점유로 이어질 수 있는 병목 리스크는 존재하나, 복잡한 보상 트랜잭션 코드를 짤 필요 없이 데이터 정합성과 롤백 안정성을 프레임워크 수준에서 100% 쉽게 보장받음.
- **추후 개선 여지(있다면):** 향후 트래픽이 폭증하여 커넥션 풀 고갈 이슈가 실제로 발생할 경우, 비동기 큐(Kafka/RabbitMQ)를 도입하거나 다시 단계 분리 트랜잭션(SAGA 패턴 등)으로 고도화 검토.

### 4. 결제 도메인 독립성 및 DIP 기반 설계

#### 문제 정의
- **현재 동작/제약:** 결제 관련 로직이나 데이터가 주문 도메인 내부에 종속될 가능성.
- **문제(또는 리스크):** 향후 다중 결제(복합 결제), 환불 정책 변경, 타 PG사 추가 연동 시 주문 핵심 로직을 매번 수정해야 하는 OCP(개방 폐쇄 원칙) 위반 우려.
- **성공 기준(완료 정의):** 주문 비즈니스 로직 수정 없이 결제 로직 확장이 가능한 구조 확립.

#### 선택지와 결정
- **고려한 대안:**
  - **A (주문서 통합 및 하드코딩):** 주문 테이블 컬럼에 결제 정보를 관리하고 인프라 코드를 바로 호출. (개발 속도 빠름)
  - **B (독립 도메인 및 DIP 기반 포트 설계):** 결제(Payment) 도메인을 별도로 분리하고, 추상화된 PaymentGateway 인터페이스(Port)를 두어 인프라 구현체를 역전시킴.
- **최종 결정:** **대안 B (결제 독립 및 DIP 설계)**
- **트레이드오프:** 아키텍처 계층과 매핑해야 할 DTO, 도메인 클래스 수 등 설계 비용이 증가하지만, 유지보수성과 확장성이 압도적으로 우수함.

## 🤔 고민한 점 / 막혔던 부분
<!--
학습 과정에서 어려웠던 것, 아직 해결 못한 것을 자유롭게 기록한다.
-->
- 기존에 `BaseSoftDeleteEntity`를 상속받아 Soft Delete를 의도했으나, 정작 엔티티에 `@SQLDelete` 어노테이션이 누락되어 있어 `deleteById()` 호출 시 데이터가 물리적으로 하드 삭제되는 버그를 발견했습니다. 이로 인해 템플릿 삭제 시 결제 로직이 깨지는 원인을 추적하는 데 고민의 시간이 있었습니다.
- 템플릿의 스냅샷 데이터를 `CouponIssue`에 직접 매핑하게 되면서, 의존하고 있던 다수의 `CouponService` 비즈니스 로직과 단위/통합/동시성 테스트의 생성자들을 일괄적으로 수정하고 동기화하는 작업에 주의를 기울여야 했습니다.
- **결제 트랜잭션 설계 수정:** 초기에는 DB 커넥션 보호를 위해 외부 통신 로직을 트랜잭션 밖으로 뺐으나, 수동 보상 트랜잭션 로직이 지나치게 복잡해지는 단점을 겪었습니다. 이에 팀과 논의하여 유지보수성과 데이터 정합성 보장의 편리함을 우선시하기 위해 단일 트랜잭션 구조로 다시 합치고 코드를 리팩토링했습니다.

- 동시성과 대용량 트래픽 처리를 고려하며 코드를 설계하는 과정에서, 단순히 요구사항을 만족하는 것을 넘어 장애 상황(네트워크 타임아웃, 스레드 경합 등)을 가정해보며 아키텍처를 방어적으로 짜는 습관을 기를 수 있었습니다.

  현재 단일 트랜잭션 구조는 구현이 극도로 단순해지고 정합성이 쉽게 유지된다는 강력한 장점이 있으나, 반드시 다음의
  리스크를 인지해야 합니다.

    1. 트랜잭션 비대화에 따른 커넥션 풀 고갈 리스크
       • 위험: 외부 PG사 시스템에 대규모 지연 장애가 발생하면, 우리 서버의 DB 커넥션과  Stock 의 비관적 락이 반환되지
       못한 채 무한정 대기하게 되어 전체 서비스(조회 API 포함) 장애로 이어질 수 있습니다.
       • 완화 선택지:
       • 선택 A: PG 요청(HttpClient)의 Read Timeout을 3~5초 이내로 극단적으로 짧게 설정하여 강제로 DB 커넥션을
       반환하도록 설정. (현재 도입하기 가장 쉬운 방식)
       • 선택 B: 향후 트래픽이 커질 경우, 메시지 큐(Kafka/Redis)를 도입하여 주문 생성(재고 차감)과 실제 결제
       승인을 비동기 이벤트 스트림으로 완전히 격리(Eventual Consistency).

    2. 비관적 락 타임아웃에 의한 결제 실패 빈도 증가
       • 위험: 단일 트랜잭션 안에 락 점유 로직과 외부 통신 로직이 공존하므로, 인기 상품의 경우 재고 락 대기 시간(
       lock_wait_timeout )이 길어져 정상 고객들의 결제가 튕길 수 있습니다.

# week4 인사이트

> 날짜: 2026-06-10
> 주제: 스프링 트랜잭션 / 동시성 제어 (Lost Update, 락, 원자적 업데이트) 딥다이브

## 개념 정리

### 더티체킹으로 좋아요를 처리하면 왜 동시 요청 시 값이 손실되는가 (Lost Update)
각 트랜잭션이 `findById` 시점에 **자기만의 스냅샷**을 가진다. 거의 동시에 들어오면 여러 트랜잭션이 같은 초기값(`likeCount=0`)을 함께 읽고, 메모리에서 각자 +1 한 뒤 더티체킹으로 UPDATE를 날린다. 이때 나가는 SQL은 `like_count = like_count + 1`이 아니라 **`like_count = 1`이라는 절대값**이라, 서로의 증가분을 덮어쓴다 → 갱신 손실.
- 결과는 "정확히 1"이 아니라 **1~몇 개 사이로 매번 달라진다.** 모든 스레드가 같은 순간에 읽는 게 아니라, 일부는 다른 트랜잭션이 커밋한 뒤 값을 읽어 더 올릴 수도 있기 때문.
- **예외가 안 난다.** 모든 요청이 정상 처리된 듯 보이지만 데이터만 조용히 오염된다 — 이게 Lost Update가 위험한 이유.

### 메모리 / 영속성 컨텍스트 / 1차 캐시는 같은 말인가?
같지 않다. **포함 관계**다: `메모리(JVM 힙) ⊃ 영속성 컨텍스트 ⊃ 1차 캐시`.
- **영속성 컨텍스트**: JPA가 엔티티 생명주기를 관리하는 논리적 영역. 1차 캐시·변경감지·쓰기지연·동일성보장을 제공.
- **1차 캐시**: 영속성 컨텍스트 안의 `Map<타입+ID, 엔티티>` 자료구조.
- `clearAutomatically`는 **영속성 컨텍스트 전체**를 비운다 (1차 캐시 포함).

### detached(준영속) 상태는 객체가 다른 메모리로 옮겨가는 것인가?
아니다. **물리적 위치(힙)는 그대로**고, 바뀌는 건 "영속성 컨텍스트가 그 객체를 관리 목록에서 빼는" **논리적 관계**뿐이다.
- `clear()`는 1차 캐시(Map)의 엔트리만 제거한다. 변수가 여전히 그 객체를 참조하므로 객체는 힙에 그대로 남는다(GC 안 됨).
- detached 객체의 **이미 로딩된 단순 필드 읽기는 에러 없이 동작**한다. 단지 낡은(stale) 값일 뿐. 진짜 에러는 ① 지연 로딩 접근(`LazyInitializationException`) ② 변경해도 dirty checking 무력화(조용히 무시).

### @Modifying 과 clearAutomatically 의 의미
- `@Modifying`: `@Query`는 기본 SELECT를 가정하므로, UPDATE/DELETE(DML)임을 알려줘야 한다. 없으면 실행 시 예외.
- `clearAutomatically=true`: `@Modifying` 쿼리는 1차 캐시를 거치지 않고 DB로 직행한다. 그래서 DB와 1차 캐시 속 엔티티 값이 불일치해진다. UPDATE 직후 영속성 컨텍스트를 비워, 다음 조회가 캐시가 아닌 DB에서 최신값을 읽도록 강제한다.

### 원자적 UPDATE는 왜 동시성에 안전한가
`UPDATE ... SET like_count = like_count + 1`의 우변은 **메모리 스냅샷이 아니라 UPDATE 실행 순간 DB 행의 실제 값**이다. InnoDB가 UPDATE 시 행에 X락을 잡아 요청들이 순차 처리되므로 손실이 없다.
- 핵심은 **락 보유 시간**: 원자적 UPDATE는 "UPDATE 한 문장 + 커밋"으로 락 구간이 극히 짧아(메모리 연산) 경합에 강하다. 반면 비관적 락은 조회~커밋까지 락을 잡아 보유 시간이 길다.

### 핫 카운터와 카운터 분산(sharding) 트레이드오프
한 행에 트래픽이 몰리면(핫 로우) 원자적 UPDATE도 직렬화 병목이 생긴다. 카운터를 N개 슬롯 행으로 쪼개 쓰기를 1/N로 분산하고 읽을 때 SUM 한다.
- **본질은 쓰기 부하를 읽기로 떠넘기는 것** → 쓰기 ≫ 읽기일 때만 유리.
- 좋아요는 읽기가 더 많아 순수 sharding이 안 맞을 수 있고, 강한 정합성도 불필요 → Redis INCR + 주기적 flush가 더 자연스럽다.
- 정합성이 돈과 직결되는 재고·쿠폰·잔액은 락으로 강한 정합성을 보장해야 한다.

### 비관적 락 vs 낙관적 락의 본질적 차이
- **비관적 락**: "충돌 분명히 난다 → 미리 잠근다." `SELECT ... FOR UPDATE`로 조회 시점부터 X락. 충돌 시 **기다렸다가** 진입하므로, 깨어날 땐 이미 최신값을 본다 → 재시도 불필요.
- **낙관적 락**: "충돌 거의 없다 → 안 잠그고 진행, 저장 시 검사." `@Version` 필드로 UPDATE에 `WHERE version=:old`를 자동 부착. 충돌 시 **기다리지 않고 그냥 진행했다가 저장 순간 터진다**(`OptimisticLockException`).

### 둘 다 예외를 던지는데 뭐가 다른가 (핵심)
쿠폰처럼 "한 번 USED면 영원히 USED"인 경우엔 결과가 같아 보이지만, 예외의 **성격**이 다르다:
- **비관적의 예외** = "최신값을 보고 판단했더니 진짜 안 됨" → **사실의 통보.** 재시도해도 소용없음.
- **낙관적의 예외** = "낡은 값으로 진행하다 타이밍에 밀림" → **타이밍의 통보.** 다시 읽으면 상황이 다를 수 있어 재시도가 의미 있을 수 있음 (예: 재고는 충돌해도 9개 남아있어 재시도로 성공 가능).

### 재시도(Retry) 책임은 어디서 나오는가
락의 성질이 아니라 **업무가 충돌을 어떻게 다뤄야 하느냐**에서 나온다.
- 쿠폰: 충돌 = 거절이 정답 → 재시도 금지 (다시 해도 USED).
- 재고: 충돌 = 그래도 처리돼야 함 → 재시도 필요.
- 비관적 락은 "기다림"이 곧 재처리라 코드가 단순하고, 낙관적 락은 DB가 예외만 던지므로 재시도/포기를 내가 코드로 결정해야 한다.

### 원자적 조건부 UPDATE vs 낙관적 락
둘 다 WHERE 조건으로 막고 0건이면 실패라 닮았지만:
- 원자적 조건부(`SET status='USED' WHERE status='AVAILABLE'`): 내가 직접 SQL에 의미 컬럼 조건 작성. 가장 빠르나 **도메인 규칙이 SQL로 새어나간다.** 부가 검증(소유자·만료·할인)은 UPDATE 밖에서 따로 배치해야 함.
- 낙관적 락(`@Version`): JPA가 버전 조건을 자동 생성. **도메인 메서드(`use()`)를 그대로 쓰면서** 충돌 감지 → 아키텍처상 깔끔.

## 설계 결정

### 쿠폰 사용 동시성 제어를 비관적 락 → 낙관적 락으로 전환
- **결정:** `IssuedCoupon`에 `@Version`을 추가하고, `findByIdWithLock`(PESSIMISTIC_WRITE) 대신 일반 `findById` + 도메인 `use()` 흐름으로 변경.
- **이유:**
  1. 쿠폰 동시 사용 충돌은 빈도가 극히 낮다(더블클릭·재전송 수준). 낙관적 락의 유일한 약점(충돌 시 재시도 폭증)이 여기선 발생하지 않음.
  2. 충돌 시 정답이 "거절"이라 재시도가 불필요 → 낙관적 락의 실패 처리 부담도 사라짐.
  3. `use()` 도메인 메서드에 담긴 상태 전이 규칙을 그대로 유지 → "비즈니스 규칙은 도메인에" 원칙과 부합.
  4. 비관적 락은 충돌이 거의 없는데도 모든 주문이 락을 잡고 기다릴 잠재 비용을 떠안는다. 쿠폰엔 과투자.
- **단서:** 현행 비관적 락도 동작하고 테스트가 통과한다면 전환은 버그 수정이 아닌 성능 최적화다. 실제 병목 측정 전엔 현행 유지도 정당 — 이번엔 학습 목적의 전환.

## 오해 교정

### 더티체킹 동시 좋아요의 최종값
- **오해:** 30명이 동시에 누르면 최종 좋아요가 "정확히 1개" 찍힌다.
- **정정:** 1~몇 개 사이로 매번 달라진다. 모든 스레드가 같은 순간에 읽지 않아, 일부는 커밋된 값을 읽어 더 올릴 수 있다. "정확히 1"이 아니라 "30보다 한참 작은 들쭉날쭉한 값".

### detached 상태의 의미
- **오해:** 영속성 컨텍스트가 "끊겼다"는 건 객체가 다른 메모리 공간으로 이동한 것이고, 접근하면 에러가 난다.
- **정정:** 객체는 같은 힙에 그대로 있다. 영속성 컨텍스트의 관리 목록(Map)에서 빠진 논리적 단절일 뿐. 이미 로딩된 필드 읽기는 에러 없이 동작한다(낡은 값일 뿐).

### "둘 다 예외 던지니 비관적·낙관적이 같다"
- **오해:** 쿠폰에서 비관적이든 낙관적이든 결국 예외로 거절되니 차이가 없다.
- **정정:** 예외의 성격이 다르다. 비관적은 "사실의 통보"(재시도 무의미), 낙관적은 "타이밍의 통보"(다시 보면 다를 수 있음). 재고처럼 재시도가 필요한 경우에 이 차이가 결정적으로 갈린다.

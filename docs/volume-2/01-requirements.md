# '감성 이커머스' 요구사항 명세

본 문서는 '감성 이커머스'의 기능에 대한 합의 사항을 기록한다. 한 줄 요청만으로는 드러나지 않는 전제·예외·정책을 풀어 적고, 트레이드오프가 있는 결정은 핵심 의사결정 분기에 그 이유와 함께 남긴다.

---

## 도메인 및 용어 정의

### 브랜드 (Brand)

| 한글 | 영어            | 의미 |
|---|---------------|---|
| 브랜드 | `Brand`       | 상품을 묶는 상위 카테고리. |

### 상품 (Product)

| 한글 | 영어            | 의미 |
|---|---------------|---|
| 상품 | `Product`     | 판매 대상 아이템. 한 브랜드에 속한다. 한 번 등록된 후 소속 브랜드를 바꿀 수 없다. |
| 가격 | `Price`       | 상품의 판매 단가. 0 이상의 정수. |
| 재고 | `Stock`       | 상품의 남은 수량. 0 이상의 정수. |

### 좋아요 (Like)

| 한글 | 영어 | 의미 |
|---|---|---|
| 좋아요 | `Like` | 회원이 상품에 표시하는 관심. 회원당 상품당 1건. |

### 주문 (Order)

| 한글 | 영어            | 의미                                                                    |
|---|---------------|-----------------------------------------------------------------------|
| 주문 | `Order`       | 회원이 상품 구매 의사를 표명한 행위. 1개 이상의 주문 항목으로 구성된다.                            |
| 주문 항목 | `OrderItem`   | 주문에 포함된 상품 한 줄. 주문 시점의 상품 정보(상품명·소속 브랜드명·단가)와 수량이 스냅샷으로 기록된다. (결정 5 참조) |
| 주문 상태 | `OrderStatus` | 주문의 생명주기 상태. 본 라운드에서는 생성 상태만 진입한다. 결제 도입 이후 결제 완료·결제 취소 상태가 추가될 자리.   |
| 수량 | `Quantity`    | 주문 항목의 구매 개수. 1 이상의 정수.                                               |
| 단가 | `UnitPrice`   | 주문 항목에 된 주문 시점 상품 가격. 이후 상품 가격이 변경되어도 본 값은 변하지 않는다.                   |
| 총 결제 금액 | `TotalPrice`  | 주문 항목들의 (`단가 × 수량`) 합계.                                               |

## 액터 정의

| 한글 | 영어 | 의미 |
|---|---|---|
| 회원 | `User` | 서비스에 가입해 로그인한 상태로 시스템을 이용하는 사용자. 헤더 `X-Loopers-LoginId`(로그인 ID)와 `X-Loopers-LoginPw`(비밀번호)로 식별. |
| 비회원 | `Guest` | 가입하지 않았거나 로그인하지 않은 상태로 시스템을 이용하는 사용자. |
| 관리자 | `Admin` | 운영자 권한으로 서비스 자원을 운영하는 사용자. 회원과 다른 인증 경로(LDAP 헤더 `X-Loopers-Ldap: loopers.admin`)로 식별. |

## 브랜드 (Brand)

### BRD-1. 회원 또는 비회원은 특정 브랜드의 정보를 확인할 수 있다.

**전제 조건**
- 별도 로그인 없이도 이용할 수 있다.

**기능 요구사항**
- Main Flow
    1. 회원 또는 비회원이 특정 브랜드의 정보를 요청한다.
    2. 시스템은 해당 브랜드가 삭제되지 않고 존재하는지 확인한다.
    3. 시스템은 브랜드의 식별자·이름·설명을 반환한다.
- Exception Flow
    - 요청한 브랜드가 존재하지 않거나 삭제되었으면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 관리자 전용 정보(등록·갱신 시각, 운영 메모 등)는 본 응답에서 제외된다.

**관련 API**
- `GET /api/v1/brands/{brandId}`

### BRD-2. 관리자는 등록된 브랜드 목록을 조회할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 페이지 정보를 포함해 브랜드 목록을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 삭제되지 않은 브랜드 한 페이지 분량과 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)를 반환한다.
    4. 응답의 각 항목에는 브랜드의 식별자·이름·설명·등록 시각·갱신 시각이 포함된다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 페이지 번호가 0 미만이거나 페이지 크기가 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 응답에는 삭제된 브랜드는 포함되지 않는다.
- 페이지 크기를 지정하지 않으면 한 페이지에 20건을 반환하며 1 이상 100 이하만 허용한다.
- 결과는 등록 시각 내림차순으로 정렬한다.

**관련 API**
- `GET /api-admin/v1/brands?page=0&size=20`

### BRD-3. 관리자는 특정 브랜드의 상세 정보를 조회할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 특정 브랜드의 상세 정보를 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 브랜드의 식별자·이름·설명·등록 시각·갱신 시각을 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 요청한 브랜드가 존재하지 않거나 삭제되었으면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**관련 API**
- `GET /api-admin/v1/brands/{brandId}`

### BRD-4. 관리자는 새로운 브랜드를 등록할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.
- 등록하려는 브랜드 이름이 삭제되지 않은 다른 브랜드와 중복되지 않아야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 브랜드 이름과 설명을 포함해 등록을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 입력값이 허용 범위 안에 있는지 검사한다.
    4. 입력된 이름이 삭제되지 않은 브랜드 중에 이미 사용되는지 살핀다.
    5. 중복이 없으면 새 브랜드를 생성한다.
    6. 생성된 브랜드의 식별자를 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 브랜드 이름이 허용 범위(1자 이상 50자 이하)를 벗어나면, 시스템은 입력 검증 실패를 반환한다.
    - 동일한 이름의 삭제되지 않은 브랜드가 이미 존재하면, 시스템은 자원 충돌로 응답한다.

**비즈니스 규칙**
- 브랜드 이름은 1자 이상 50자 이하 문자열이다.
- 삭제된 브랜드 이름은 중복 검사 대상에서 제외한다.

**관련 API**
- `POST /api-admin/v1/brands`

### BRD-5. 관리자는 등록된 브랜드 정보를 수정할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.
- 대상 브랜드가 삭제되지 않은 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 대상 브랜드와 새 이름·설명을 포함해 수정을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 브랜드가 존재하고 삭제되지 않았는지 점검한다.
    4. 입력값이 허용 범위 안에 있는지 검사한다.
    5. 새 이름이 다른 삭제되지 않은 브랜드와 중복되는지 살핀다.
    6. 중복이 없으면 브랜드 정보를 갱신한다.
    7. 갱신된 브랜드의 식별자를 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 브랜드가 존재하지 않거나 이미 삭제된 상태이면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 브랜드 이름이 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.
    - 새 이름이 다른 삭제되지 않은 브랜드와 중복되면, 시스템은 자원 충돌로 응답한다.

**비즈니스 규칙**
- 동일한 이름으로 수정을 요청해도 정상 처리되며 갱신 시각만 새로 찍힌다.

**관련 API**
- `PUT /api-admin/v1/brands/{brandId}`

### BRD-6. 관리자는 등록된 브랜드를 삭제할 수 있다. 삭제 시 해당 브랜드의 상품도 함께 삭제된다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 대상 브랜드 삭제를 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 브랜드가 존재하고 삭제되지 않은 경우, 해당 브랜드와 소속 삭제되지 않은 상품들을 같은 트랜잭션 안에서 삭제한다.
    4. 대상 브랜드가 존재하지 않거나 이미 삭제된 경우, 별도 동작 없이 현재 상태를 유지한다.
    5. 삭제가 완료되었음을 알린다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.

**비즈니스 규칙**
- 브랜드 삭제 시 해당 브랜드의 삭제되지 않은 상품도 모두 함께 삭제된다. 데이터베이스 차원의 연쇄 제약이 아니라 애플리케이션이 같은 트랜잭션 안에서 처리한다.
- 삭제는 기술적으로 soft delete 방식으로, 행은 보존하되 외부 노출에서는 존재하지 않는 것으로 처리한다. (결정 7 참조)
- 존재하지 않거나 이미 삭제된 브랜드에 삭제 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 삭제). (결정 6 참조)

**비기능 요구사항**
- 브랜드와 소속 상품의 삭제는 같은 트랜잭션 안에서 처리하여, 일부만 삭제된 상태로 남는 비정합 상태가 발생하지 않아야 한다.

**관련 API**
- `DELETE /api-admin/v1/brands/{brandId}`

## 상품 (Product)

### PRD-1. 회원 또는 비회원은 상품 목록을 조회할 수 있다.

**전제 조건**
- 별도 로그인 없이도 이용할 수 있다.

**기능 요구사항**
- Main Flow
    1. 회원 또는 비회원이 선택적으로 브랜드 필터·정렬 기준·페이지 정보를 포함해 상품 목록을 요청한다.
    2. 시스템은 삭제되지 않은 상품 중 조건에 맞는 한 페이지 분량을 조회한다.
    3. 시스템은 페이지 내 상품들의 식별자·이름·소속 브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수와 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)를 반환한다.
- Exception Flow
    - 정렬 기준이 허용되지 않는 값이면, 시스템은 입력 검증 실패를 반환한다.
    - 페이지 번호가 0 미만이거나 페이지 크기가 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 정렬 기준은 최신 등록 순(기본), 가격 오름차순, 좋아요 많은 순 세 가지 중 하나를 선택한다.
- 페이지 크기를 지정하지 않으면 한 페이지에 20건을 반환하며 1 이상 100 이하만 허용한다.
- 응답에는 삭제된 상품과 소속 브랜드가 삭제된 상품은 포함되지 않는다.
- 재고는 정확한 수량 대신 가용 여부만 노출한다.
- 브랜드 필터로 지정된 브랜드가 존재하지 않거나 삭제된 경우, 빈 목록을 반환한다.
- 좋아요 수는 좋아요 테이블에서 매 조회마다 집계한 값을 사용한다.

**관련 API**
- `GET /api/v1/products?brandId={brandId}&sort=latest&page=0&size=20`

### PRD-2. 회원 또는 비회원은 특정 상품의 정보를 조회할 수 있다.

**전제 조건**
- 별도 로그인 없이도 이용할 수 있다.

**기능 요구사항**
- Main Flow
    1. 회원 또는 비회원이 특정 상품의 상세 정보를 요청한다.
    2. 시스템은 해당 상품이 삭제되지 않고 존재하는지 확인한다.
    3. 시스템은 상품의 식별자·이름·설명·소속 브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수를 반환한다.
- Exception Flow
    - 요청한 상품이 존재하지 않거나 삭제되었으면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 관리자 전용 정보(정확한 재고 수량, 등록·갱신 시각, 운영 메모 등)는 본 응답에 포함하지 않는다.
- 재고는 정확한 수량 대신 가용 여부만 노출한다. 주문 단계에서 실제 수량 부족이 발생하면 자원 충돌로 응답한다.
- 좋아요 수는 좋아요 테이블에서 매 조회마다 집계한 값을 사용한다.
- 응답에 노출되는 상품과 소속 브랜드는 모두 삭제되지 않은 상태여야 한다. 이 조건은 브랜드 삭제 시 소속 상품도 cascade 삭제되는 정책(BRD-6 참조)으로 보장된다.

**관련 API**
- `GET /api/v1/products/{productId}`

### PRD-3. 관리자는 등록된 상품 목록을 조회할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 선택적으로 브랜드 필터·페이지 정보를 포함해 상품 목록을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 삭제되지 않은 상품 한 페이지 분량과 페이지 메타 정보를 반환한다.
    4. 응답의 각 항목에는 상품의 식별자·이름·설명·소속 브랜드(식별자·이름)·가격·재고 수량·등록 시각·갱신 시각이 포함된다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 페이지 번호가 0 미만이거나 페이지 크기가 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 응답에는 삭제된 상품은 포함되지 않는다.
- 관리자 응답에는 재고 수량을 정확한 정수로 노출한다. (대고객 응답의 가용 여부와 대비)
- 페이지 크기를 지정하지 않으면 한 페이지에 20건을 반환하며 1 이상 100 이하만 허용한다.
- 결과는 등록 시각 내림차순으로 정렬한다.
- 브랜드 필터로 지정된 브랜드가 존재하지 않거나 삭제된 경우, 빈 목록을 반환한다. (PRD-1과 동일 정책)

**관련 API**
- `GET /api-admin/v1/products?page=0&size=20&brandId={brandId}`

**전제 조건**

**기능 요구사항**

- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.

### PRD-5. 관리자는 새로운 상품을 등록할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.
- 지정된 브랜드가 삭제되지 않은 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 소속 브랜드 식별자·상품 이름·설명·가격·초기 재고를 포함해 상품 등록을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 지정된 브랜드가 존재하고 삭제되지 않았는지 점검한다.
    4. 입력값이 허용 범위 안에 있는지 검사한다.
    5. 새 상품을 생성한다.
    6. 생성된 상품의 식별자를 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 지정된 브랜드가 존재하지 않거나 이미 삭제된 상태이면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 상품 이름이 허용 범위(1자 이상 100자 이하)를 벗어나면, 시스템은 입력 검증 실패를 반환한다.
    - 가격이 0 미만이거나 초기 재고가 0 미만이면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 상품 이름은 1자 이상 100자 이하 문자열이다.
- 가격은 0 이상의 정수이다.
- 초기 재고는 0 이상의 정수이다.

**관련 API**
- `POST /api-admin/v1/products`

### PRD-6. 관리자는 등록된 상품 정보를 수정할 수 있다.

**기능 요구사항**

    1. 관리자가 대상 상품과 새 이름·설명·가격·재고 수량을 포함해 수정을 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 상품이 존재하고 삭제되지 않았는지 점검한다.
    4. 입력값이 허용 범위 안에 있는지 검사한다.
    5. 상품 정보를 갱신한다.
    6. 갱신된 상품의 식별자를 반환한다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 상품이 존재하지 않거나 이미 삭제된 상태이면, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 상품 이름이 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.
    - 가격이 0 미만이거나 재고 수량이 0 미만이면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 소속 브랜드는 수정할 수 없다. 브랜드를 바꾸려면 상품을 삭제하고 새로 등록한다.
- 재고 수량의 임의 갱신은 관리자 권한 안에서만 가능하다. 회원의 주문 흐름은 차감 방향으로만 동작한다. (결정 2 참조)

**관련 API**
- `PUT /api-admin/v1/products/{productId}`

### PRD-7. 관리자는 등록된 상품을 삭제할 수 있다.

**전제 조건**
- 관리자 인증 정보를 보유한 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 관리자가 대상 상품 삭제를 요청한다.
    2. 시스템은 관리자 인증을 확인한다.
    3. 대상 상품이 존재하고 삭제되지 않은 경우, 상품을 삭제 처리한다.
    4. 대상 상품이 존재하지 않거나 이미 삭제된 경우, 별도 동작 없이 현재 상태를 유지한다.
    5. 삭제가 완료되었음을 알린다.
- Exception Flow
    - 관리자 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.

**비즈니스 규칙**
- 삭제는 기술적으로 soft delete 방식으로, 행은 보존하되 외부 노출에서는 존재하지 않는 것으로 처리한다. (결정 7 참조)
- 존재하지 않거나 이미 삭제된 상품에 삭제 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 삭제). (결정 6 참조)
- 이미 주문된 상품 항목은 주문 시점에 스냅샷이 기록되어 있어(결정 5 참조), 상품이 삭제되어도 과거 주문 표시는 유지된다.

**관련 API**
- `DELETE /api-admin/v1/products/{productId}`

## 좋아요 (Like)

### LIK-1. 회원은 특정 상품에 좋아요를 누를 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.
- 대상 상품이 삭제되지 않은 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 상품 목록 또는 상품 상세에서 특정 상품에 좋아요 등록을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 대상 상품이 존재하고 삭제되지 않았는지 점검한다.
    4. 해당 회원과 상품 조합의 좋아요 기록이 이미 있는지 살핀다.
    5. 기존 기록이 없으면 새로 저장하고 이미 있으면 별도 동작 없이 현재 상태를 유지한다.
    6. 좋아요 등록이 완료되었음을 알린다.
- Alternate Flow
    - 동일 회원이 동일 상품에 짧은 시간 안에 좋아요 등록을 여러 번 요청해도, 좋아요 행은 한 건만 유지된다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 상품이 존재하지 않거나 삭제된 경우, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 한 회원의 한 상품에 대한 좋아요는 한 건만 유지된다.
- 좋아요 등록 동작은 진입 경로(목록/상세)와 무관하게 같다.
- 이미 좋아요한 상품에 다시 등록 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 등록).
- 삭제된 상품(브랜드 삭제로 cascade 삭제된 상품 포함, BRD-6 참조)에는 좋아요를 새로 등록할 수 없다.

**비기능 요구사항**
- 동일 회원의 동일 상품에 대한 동시 좋아요 등록 요청에서도 좋아요 행은 한 건만 남아야 한다.

**관련 API**
- `POST /api/v1/products/{productId}/likes`

### LIK-2. 회원은 특정 상품에 누른 좋아요를 취소할 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.
- 대상 상품이 삭제되지 않은 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 상품 목록, 상품 상세, 또는 자신이 좋아요한 상품 목록에서 특정 상품의 좋아요 취소를 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 대상 상품이 존재하고 삭제되지 않았는지 점검한다.
    4. 해당 회원과 상품 조합의 좋아요 기록이 있는지 살핀다.
    5. 기록이 있으면 삭제하고 없으면 별도 동작 없이 현재 상태를 유지한다.
    6. 좋아요 취소가 완료되었음을 알린다.
- Alternate Flow
    - 동일 회원이 동일 상품에 짧은 시간 안에 좋아요 취소를 여러 번 요청해도, 결과 상태는 한 번 취소한 결과와 동일하다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 상품이 존재하지 않거나 삭제된 경우, 시스템은 해당 자원을 찾을 수 없다고 응답한다.

**비즈니스 규칙**
- 좋아요 취소 동작은 진입 경로(상품 목록/상세/내 좋아요 목록)와 무관하게 같다.
- 좋아요 기록이 없는 상품에 취소 요청이 들어와도 별도 처리 없이 정상 응답으로 마무리한다(멱등 취소).
- 삭제된 상품에 대한 좋아요 동작은 등록·취소 모두 일관되게 차단한다. 좋아요한 상품 목록 조회에서 삭제 상품은 노출되지 않아 취소 진입 자체가 닫혀 있고 잘못된 상품 식별자에 대한 직접 호출도 자원 부재로 응답하는 일관성을 유지한다. (LIK-1 등록 차단과 대칭)

**관련 API**
- `DELETE /api/v1/products/{productId}/likes`

### LIK-3. 회원은 자신이 좋아요를 누른 상품 목록을 조회할 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 페이지 정보를 포함해 좋아요한 상품 목록을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 요청 경로의 회원 식별자가 인증된 회원과 일치하는지 점검한다.
    4. 일치하면 인증된 회원이 좋아요한 삭제되지 않은 상품의 한 페이지 분량을, 일치하지 않으면 빈 목록을 페이지 메타 정보(현재 페이지 번호·페이지 크기·총 페이지 수·총 항목 수)와 함께 반환한다.
    5. 응답의 각 항목에는 상품의 식별자·이름·소속 브랜드(식별자·이름)·가격·재고 가용 여부·좋아요 수가 포함된다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 페이지 번호가 0 미만이거나 페이지 크기가 허용 범위를 벗어나면, 시스템은 입력 검증 실패를 반환한다.

**비즈니스 규칙**
- 정렬 기준은 최신 좋아요 순 하나로 고정한다.
- 좋아요한 시점 이후 상품이 삭제되었거나 소속 브랜드가 삭제된 경우, 해당 항목은 응답에서 제외된다. (LIK-2가 이 정책을 전제로 취소 차단을 결정함)
- 페이지 크기를 지정하지 않으면 한 페이지에 20건을 반환하며 1 이상 100 이하만 허용한다.
- 좋아요 수는 좋아요 테이블에서 매 조회마다 집계한 값을 사용한다.

**비기능 요구사항**
- 타인 또는 존재하지 않는 회원 식별자로 조회하더라도 빈 목록(정상 응답)을 반환하여, 다른 회원의 식별자 존재 여부·활동 정보가 응답으로부터 추론되지 않도록 한다.

**관련 API**
- `GET /api/v1/users/{userId}/likes?page=0&size=20`

## 주문 (Order)

### ORD-1. 회원은 특정 상품 하나를 즉시 주문할 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.
- 대상 상품이 삭제되지 않은 상태여야 한다.
- 주문 수량은 1 이상의 정수여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 상품 식별자와 수량을 포함해 주문을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 대상 상품이 존재하고 삭제되지 않았는지 점검한다.
    4. 상품의 재고가 요청 수량 이상인지 확인한 뒤 충분하면 요청 수량만큼 차감한다.
    5. 주문 시점의 상품 정보(상품명·소속 브랜드명·단가)와 수량을 주문 항목 스냅샷으로 함께 저장한다.
    6. 주문과 주문 항목은 같은 트랜잭션 안에서 저장된다.
    7. 생성된 주문의 식별자·상태·주문 시각·총 결제 금액·주문 항목 스냅샷을 반환한다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 대상 상품이 존재하지 않거나 삭제된 경우, 시스템은 해당 자원을 찾을 수 없다고 응답한다.
    - 수량이 1 미만이면, 시스템은 입력 검증 실패를 반환한다.
    - 상품의 재고가 요청 수량에 미치지 못하면, 시스템은 자원 충돌로 응답하며 재고는 변하지 않는다.

**비즈니스 규칙**
- 주문 항목은 주문 시점의 상품명·소속 브랜드명·단가를 스냅샷으로 저장한다. 이후 상품 정보가 변경되어도 과거 주문 표시는 유지된다. (결정 5 참조)
- 본 라운드의 주문 상태는 생성 상태만 진입한다. 결제 도입 이후 결제 완료·결제 취소 상태가 추가될 자리만 둔다.
- 재고는 주문 생성 시점에 차감한다. 결제 미구현 단계에서 예약/확정 분리를 끌어들이지 않는다. (결정 2 참조)
- 총 결제 금액은 `단가 × 수량`으로 계산한다.
- ORD-1은 단일 상품 단건 케이스를 다루며, 동일 API의 다중 항목 일반화 케이스는 ORD-2에서 정의한다.

**비기능 요구사항**
- 동시에 같은 상품에 주문이 몰리는 상황에서도 재고가 음수가 되지 않아야 한다. (결정 4 참조)
- 재고 차감·주문 저장·주문 항목 저장 중 어느 하나라도 실패하면 전체가 처음 상태로 되돌아간다.

**관련 API**
- `POST /api/v1/orders`

### ORD-2. 회원은 단일 또는 여러 상품을 한 번의 주문으로 묶어 주문할 수 있다.

**전제 조건**
- 회원이 로그인 상태여야 한다.
- 주문 항목이 1개 이상 포함되어 있어야 한다.
- 각 항목의 상품이 삭제되지 않은 상태여야 한다.
- 각 항목의 수량은 1 이상의 정수여야 한다.

**기능 요구사항**
- Main Flow
    1. 회원이 상품 식별자와 수량의 쌍을 1개 이상 묶어 주문을 요청한다.
    2. 시스템은 회원 인증을 확인한다.
    3. 모든 항목의 상품이 존재하고 삭제되지 않았는지 점검한다.
    4. 항목들을 상품 식별자 기준으로 일정한 순서로 정렬한 뒤, 정렬된 순서로 각 상품의 재고를 요청 수량만큼 차감한다. 재고가 충분한 경우에만 차감이 성공한다.
    5. 항목별 주문 시점의 상품 정보(상품명·소속 브랜드명·단가)와 수량을 주문 항목 스냅샷으로 저장할 준비를 한다.
    6. 주문과 주문 항목들은 같은 트랜잭션 안에서 저장된다.
    7. 생성된 주문의 식별자·상태·주문 시각·총 결제 금액·주문 항목 스냅샷 목록을 반환한다.
- Exception Flow
    - 회원 인증에 실패한 경우, 시스템은 인증 실패를 반환한다.
    - 항목이 비어 있거나 어느 한 항목의 수량이 1 미만이거나 같은 상품 식별자가 둘 이상 포함된 경우, 시스템은 입력 검증 실패를 반환한다.
    - 어느 한 항목의 상품이 존재하지 않거나 삭제된 경우, 시스템은 해당 자원을 찾을 수 없다고 응답하며 이미 차감된 다른 항목의 재고는 함께 되돌린다.
    - 어느 한 항목의 재고가 요청 수량에 미치지 못하면, 시스템은 자원 충돌로 응답하며 이미 차감된 다른 항목의 재고는 함께 되돌린다.

**비즈니스 규칙**
- 항목 간 정합성은 전부 성공 또는 전부 실패다. 한 항목이라도 실패하면 다른 항목 처리도 되돌린다.
- 같은 상품 식별자가 항목에 둘 이상 포함된 요청은 거부한다. 수량 합산 책임은 클라이언트가 진다.
- 총 결제 금액은 항목들의 (`단가 × 수량`) 합계로 계산한다.
- 주문 항목은 항목별로 상품명·소속 브랜드명·단가를 스냅샷으로 저장한다. (결정 5 참조)
- ORD-1과 동일한 `POST /api/v1/orders` API를 공유한다. ORD-1은 항목 수 1인 케이스, ORD-2는 항목 수 1 이상인 일반화 케이스.

**비기능 요구사항**
- 동시에 같은 상품 묶음에 주문이 몰리는 상황에서도 교착 상태가 발생하지 않아야 한다. 항목들을 상품 식별자 순서대로 정렬해 잠금 획득 순서를 일관되게 유지한다.
- 동시에 같은 상품에 주문이 몰리는 상황에서도 재고가 음수가 되지 않아야 한다. (결정 4 참조)
- 재고 차감·주문 저장·주문 항목 저장 중 어느 하나라도 실패하면 전체가 처음 상태로 되돌아간다.

**관련 API**
- `POST /api/v1/orders`

### ORD-3. 회원은 자신의 주문 내역을 특정 날짜 범위로 조회할 수 있다.

**문제**: 좋아요 수를 매 조회마다 집계할지, 누적 컬럼·캐시로 보관할지.

|---|---|---|
| **A (채택)** | 매 조회마다 좋아요 테이블에서 집계 | 정합성 완벽, 동시성 처리 불필요. 좋아요 누적이 늘수록 조회·정렬 비용 비례 증가. |
| B | 상품에 `likeCount` 컬럼 + 트랜잭션 안 원자적 증감 | 조회·정렬 비용 일정. 동시성 처리 필요, 토글 폭주 시 미세 오차 가능. |
| C | 좋아요 행은 그대로, 카운트는 캐시(Redis) | 조회 비용 O(1). 캐시 미스·장애 폴백, 주기 보정 잡 등 일관성 모델 복잡. |

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 상품 테이블의 컬럼 (`Product.stock`) | 단순. 상품 조회 시 함께 읽힘. 상품 정보 수정과 재고 차감이 같은 행 잠금 경쟁. |
| B | 별도 `Stock` 테이블 | 관심사·경합 분리. join/추가 쿼리, 모델 한 단계 더. |
| C | 외부 시스템(Redis 등) | 차감 빠름. 영속성·일관성 책임이 캐시 인프라 결정과 묶임. 과설계. |

**사유**: 본 라운드 규모에서 상품·재고 행 경합은 무시할 만하고 모델을 가장 단순하게 유지.
**추후 개선**: 상품 정보 수정과 재고 갱신이 실측상 경합하면 B안으로 분리.

### 결정 4. 재고 차감 동시성 메커니즘

**문제**: 동시 주문에서 음수 차감과 race를 어떻게 막을지.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | 원자적 조건부 갱신 (`UPDATE ... SET stock = stock - :qty WHERE id = :id AND stock >= :qty`) | 검사·차감 원자성을 DB가 한 쿼리에서 보장. RTT 1회. 잠금 안에서 보호할 추가 로직 어려움. |
| B | 비관 잠금 (`SELECT ... FOR UPDATE` 후 비교 후 `UPDATE`) | 잠금 안에서 여러 단계 로직 보호. RTT 2회, 인덱스·격리 수준에 따라 갭/넥스트키 잠금 가능. |
| C | 낙관 잠금 (`version` 컬럼 + 재시도) | 충돌 드물 때 효율적. 인기 상품에서 재시도 폭주 가능. |

**사유**: "검사·차감 원자성을 DB가 한 쿼리에서 보장"한다는 점이 핵심. 본 라운드에서 잠금 안에 보호할 추가 도메인 로직 없음. ORD-2 다중 주문에서는 교착이 생기지 않도록 항목을 상품 식별자 순으로 정렬해 잠금 순서를 일관되게 유지.
**추후 개선**: 잠금 안에서 보호할 추가 도메인 로직이 생기면 B안(비관 잠금)으로 전환 검토.

### 결정 5. 주문 항목 스냅샷 범위

**문제**: 상품 정보 변경에도 과거 주문 표시를 일관되게 유지하려면 어디까지 기록할지.

| 안 | 접근 | 트레이드오프                                               |
|---|---|------------------------------------------------------|
| A | 상품 식별자만 저장 | 저장 가벼움. 상품 변경 시 과거 주문 표시 깨지고, 상품 삭제 후 "이름 없는 항목"이 됨. |
| **B (채택)** | 상품 식별자 + 상품명·소속 브랜드명·단가·수량 | 영수증·CS·정산 정합성 보존. 상품·브랜드 변경에 무관하게 거래 표시 일관.          |
| C | 상품 정보 전체(설명·이미지 URL·기타 메타) + 브랜드 정보 전체 | 정보 손실 적음. 설명·이미지는 기록 가치 낮음, 이미지 URL은 스토리지 수명 종속.     |

**사유**: 기록 대상은 "그 시점 거래를 재현하는 데 재무·식별 가치가 있는 최소 집합"으로 한정. 상품명·브랜드명·단가는 영수증·CS·정산에 직접 영향. 수량은 항목 본질. 설명·이미지는 제외.
**추후 개선**: 상품 옵션·할인 도입 시 옵션명·할인 적용가 기록 추가.

### 결정 6. DELETE 자원의 멱등 처리 정책

**문제**: "존재하지 않거나 이미 삭제된 자원에 DELETE"가 들어왔을 때 응답 정책.

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| A | 자원 부재로 응답 | 명시적. 잘못된 입력 즉시 인지. 반복 DELETE가 200→404로 응답 갈려 RFC 멱등성 깨짐. |
| **B (채택)** | 모두 정상 응답 (완전 멱등) | RFC와 완전 일관. 자원 존재 여부 추론 불가(enumeration 방지). 잘못된 식별자도 정상 응답이라 디버깅 불리. |
| C | "이미 삭제"는 멱등, "존재하지 않음"은 자원 부재 (LIK-2 패턴) | 사실상 멱등 + 잘못된 입력 식별. 두 경우 구분 분기 로직 추가. |

**사유**: DELETE의 RFC 7231 의미와 정확히 일관. 디버깅 부담은 관리자 화면이라 운영 로그·관리자 도구로 보완 가능.
**추후 개선**: BRD-6·PRD-7에 동일 적용. LIK-2가 C안인 것은 의도된 차이 — 좋아요는 부수 자원이라 부모(상품)의 부재는 자원 부재로, 삭제 대상 자신의 부재는 멱등으로 통일.

### 결정 7. 소프트 삭제 정책의 도메인별 차이

**문제**: 각 도메인을 hard delete로 처리할지, soft delete로 처리할지.

| 안 | 접근 | 트레이드오프                                                  |
|---|---|---------------------------------------------------------|
| A | 전부 hard delete | 단순. 감사 추적·과거 주문 표시 가치 잃음, 기록 상품명·브랜드명의 원본 추적 불가.        |
| B | 전부 soft delete | 일관성. 좋아요처럼 토글 빈번한 자원에 활성 행 유니크 제약 복잡(MySQL 부분 인덱스 미지원). |
| **C (채택)** | 도메인별 분리 — 브랜드·상품·주문·주문 항목은 soft delete, 좋아요는 hard delete | 자원 성격별 최적. 정책이 도메인마다 다르다는 인지 부담.                        |

**사유**:
- **브랜드·상품**: 감사 추적·데이터 분석 가치 있고 결정 5(스냅샷)과 결합해 과거 주문 기록 정보의 원본 추적 가능. soft delete.
- **주문·주문 항목**: 영수증·정산·법령상 보관 의무. soft delete (본 라운드에 명시적 삭제 API 없으나 도입 시).
- **좋아요**: 토글 빈번, 이력 가치 낮음. 활성 행 유니크 보장이 복잡해져 hard delete가 단순.

**추후 개선**: 좋아요 이력이 추천·행동 분석에 필요해지면 토글 이벤트를 별도 이벤트 로그로 적재. 브랜드·상품의 soft delete 행이 운영 가시성에 필요해지면 `BrandStatus`·`ProductStatus` enum(`ACTIVE`/`INACTIVE`/`DELETED`)으로 분리 검토.

## 2. 비즈니스 규칙

### 인증 / 역할

- 사용자(`USER`)와 관리자(`ADMIN`)는 역할로 구분한다
- `USER`: 상품 조회, 찜, 주문, 주문 조회
- `ADMIN`: 브랜드/상품/재고 관리, 전체 주문 조회

### 브랜드 / 상품

- 상품은 반드시 등록된 브랜드에 속해야 한다
- 상품의 브랜드는 등록 이후 변경할 수 없다
- 브랜드 삭제 시 소속 상품은 판매중지(`INACTIVE`) 상태로 변경되며, 해당 상품의 찜 이력은 보존된다
- 상품은 이름, 가격, 재고 수량을 가지며 주문 요청 시 재고가 차감된다

### 주문

- 주문 수량은 1 이상이어야 한다
- `ACTIVE` 상태의 상품만 주문할 수 있다 (판매중지 상품 주문 불가)

## 3. 유스케이스 흐름

> 표기 규칙
> - **Main Flow**: 정상 성공 경로
> - **Alternate Flow**: 조건 분기에 따른 대안 경로 (오류 아님)
> - **Exception Flow**: 오류·예외 상황 (괄호 안 숫자는 분기 발생 단계)

---

### UC-001: 브랜드별 상품 조회

**시나리오**: 사용자가 특정 브랜드의 상품 라인업을 탐색하고 싶을 때, 브랜드 ID로 해당 브랜드의 판매 중인 상품 목록을 조회한다.

**액터**: 사용자

**Main Flow**
1. 사용자가 브랜드 ID를 지정하여 상품 목록을 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 해당 브랜드에 속한 `ACTIVE` 상태의 상품 목록을 반환한다

**Alternate Flow**
- (3) `ACTIVE` 상품이 없으면 → 빈 목록을 반환한다

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `404 NOT_FOUND`

---

### UC-002: 전체 상품 목록 조회

**시나리오**: 사용자가 플랫폼에서 판매 중인 모든 상품을 둘러보고 싶을 때, 브랜드 구분 없이 전체 상품 목록을 조회한다.

**액터**: 사용자

**Main Flow**
1. 사용자가 전체 상품 목록을 요청한다
2. 시스템이 `ACTIVE` 상태의 전체 상품을 조회한다
3. 상품 목록을 반환한다

**Alternate Flow**
- (2) `ACTIVE` 상품이 없으면 → 빈 목록을 반환한다

---

### UC-003: 상품 상세 조회

**시나리오**: 사용자가 특정 상품의 이름, 가격, 재고 등 상세 정보를 확인하고 싶을 때, 상품 ID로 상세 정보를 조회한다.

**액터**: 사용자

**Main Flow**
1. 사용자가 상품 ID를 지정하여 상세 정보를 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 상품 상세 정보를 반환한다

**Exception Flow**
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`

---

### UC-004: 상품 찜 / 찜 취소

**시나리오**: 사용자가 관심 있는 상품을 찜 목록에 추가하거나, 이미 찜한 상품을 찜 목록에서 제거하고 싶을 때 사용한다. 동일 요청으로 등록/취소를 토글한다.

**액터**: 사용자 (로그인 필요)

**Main Flow** (찜 등록)
1. 사용자가 특정 상품에 찜 요청을 한다
2. 시스템이 상품 존재 여부를 확인한다
3. 시스템이 해당 사용자의 기존 찜 이력을 확인한다
4. 찜 이력이 없으면 찜을 등록한다

**Alternate Flow** (찜 취소)
- (3) 이미 찜한 상품이면 → 기존 찜을 취소(삭제)한다

**Exception Flow**
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-005: 찜 목록 조회

**시나리오**: 사용자가 본인이 찜해 둔 상품 목록을 한눈에 확인하고 싶을 때 사용한다. 삭제된 상품은 목록에서 제외된다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 본인의 찜 목록을 요청한다
2. 시스템이 사용자의 찜 이력을 조회한다
3. `DELETED` 상태의 상품을 제외한 찜 목록을 반환한다

**Alternate Flow**
- (2) 찜한 상품이 없거나 모두 삭제된 경우 → 빈 목록을 반환한다

**Exception Flow**
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-006: 주문 요청

**시나리오**: 사용자가 원하는 상품을 수량을 지정하여 구매하고 싶을 때 사용한다. 외부 PG 결제가 성공해야 주문이 완료되고 재고가 차감된다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 상품 ID와 수량을 지정하여 주문을 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 시스템이 요청 수량과 현재 재고를 비교한다
4. 재고가 충분하면 주문을 생성한다
5. 재고를 요청 수량만큼 차감한다
6. 주문 완료 응답을 반환한다

**Exception Flow**
- (1) 주문 수량이 1 미만 → `400 BAD_REQUEST`
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`
- (2) 상품이 `ACTIVE` 상태가 아니면 → `400 BAD_REQUEST` (판매중지 포함)
- (3) 재고가 부족하면 → `400 BAD_REQUEST` (주문 미생성, 재고 변동 없음)
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-007: 주문 목록 조회

**시나리오**: 사용자가 본인의 주문 내역을 최신순으로 확인하고 싶을 때 사용한다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 본인의 주문 목록을 요청한다
2. 시스템이 해당 사용자의 주문을 최신순으로 조회한다
3. 주문 목록을 반환한다

**Alternate Flow**
- (2) 주문 내역이 없으면 → 빈 목록을 반환한다

**Exception Flow**
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-008: 주문 상세 조회

**시나리오**: 사용자가 특정 주문의 상품, 수량, 결제 정보 등 상세 내역을 확인하고 싶을 때 사용한다. 본인 주문만 조회할 수 있다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 주문 ID를 지정하여 상세 정보를 요청한다
2. 시스템이 주문 존재 여부를 확인한다
3. 시스템이 해당 주문의 소유자를 확인한다
4. 주문 상세 정보를 반환한다

**Exception Flow**
- (2) 주문이 존재하지 않으면 → `404 NOT_FOUND`
- (3) 본인 주문이 아니면 → `403 FORBIDDEN`
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

### UC-A001: 브랜드 등록

**시나리오**: 관리자가 새로운 브랜드를 플랫폼에 등록하고 싶을 때 사용한다. 등록 완료 시 생성된 브랜드 ID를 반환한다.

**액터**: 관리자

**Main Flow**
1. 관리자가 브랜드 이름을 입력하여 등록을 요청한다
2. 시스템이 브랜드명 중복 여부를 확인한다
3. 브랜드를 등록하고 생성된 브랜드 ID를 반환한다

**Exception Flow**
- (2) 브랜드명이 이미 사용 중 → `409 CONFLICT`
- (1) 필수 입력값 누락 → `400 BAD_REQUEST`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A002: 브랜드 수정

**시나리오**: 관리자가 기존 브랜드의 이름을 변경하고 싶을 때 사용한다. 변경할 이름이 현재 이름과 다를 경우에만 중복 여부를 검증한다.

**액터**: 관리자

**Main Flow**
1. 관리자가 브랜드 ID와 수정할 이름을 지정하여 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 변경할 이름이 현재 이름과 다른지 확인한다
4. 변경할 이름의 중복 여부를 확인한다
5. 브랜드 이름을 수정한다

**Alternate Flow**
- (3) 변경할 이름이 현재 이름과 동일하면 → 중복 확인(4단계)을 생략하고 수정을 진행한다

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `404 NOT_FOUND`
- (4) 변경할 이름이 이미 사용 중 → `409 CONFLICT`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A003: 브랜드 삭제

**시나리오**: 관리자가 브랜드를 플랫폼에서 제거하고 싶을 때 사용한다. 소속 상품은 함께 소프트 삭제되며, 기존 찜 이력은 데이터 무결성을 위해 보존된다.

**액터**: 관리자

**Main Flow**
1. 관리자가 브랜드 ID를 지정하여 삭제를 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 해당 브랜드에 소속된 상품을 판매중지(`INACTIVE`) 상태로 변경한다
4. 판매중지된 상품의 찜 이력은 보존한다
5. 브랜드를 삭제한다

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `404 NOT_FOUND`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A004: 상품 등록

**시나리오**: 관리자가 특정 브랜드에 속하는 새 상품을 플랫폼에 등록하고 싶을 때 사용한다. 상품은 `ACTIVE` 상태로 등록된다.

**액터**: 관리자

**Main Flow**
1. 관리자가 상품 정보(이름, 가격, 재고 수량, 브랜드 ID)를 입력하여 등록을 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 상품을 `ACTIVE` 상태로 등록하고 생성된 상품 ID를 반환한다

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `400 BAD_REQUEST`
- (1) 필수 입력값 누락 → `400 BAD_REQUEST`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A005: 상품 수정

**시나리오**: 관리자가 상품의 이름·가격을 변경하거나 재고를 추가·차감하고 싶을 때 사용한다. 브랜드 변경은 불가하며, 재고 차감 시 음수가 되지 않아야 한다.

**액터**: 관리자

**Main Flow** (정보 수정)
1. 관리자가 상품 ID와 수정할 정보(이름, 가격)를 지정하여 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 시스템이 브랜드 변경 시도 여부를 확인한다
4. 상품 정보를 수정한다

**Alternate Flow A** (재고 추가)
1. 관리자가 상품 ID와 추가할 수량(양수)을 지정하여 재고 변경을 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 재고를 지정 수량만큼 증가시킨다

**Alternate Flow B** (재고 차감)
1. 관리자가 상품 ID와 차감할 수량을 지정하여 재고 변경을 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 차감 후 재고가 0 이상인지 확인한다
4. 재고를 지정 수량만큼 감소시킨다

**Exception Flow**
- (Main-2, A-2, B-2) 상품이 존재하지 않으면 → `404 NOT_FOUND`
- (Main-3) 브랜드 변경 시도 → `400 BAD_REQUEST`
- (A-1, B-1) 재고 변경 수량이 1 미만 → `400 BAD_REQUEST`
- (B-3) 차감 후 재고가 음수가 되면 → `400 BAD_REQUEST`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A006: 상품 삭제

**시나리오**: 관리자가 특정 상품을 플랫폼에서 제거하고 싶을 때 사용한다. 소프트 삭제로 처리되며, 해당 상품의 찜 이력은 보존된다.

**액터**: 관리자

**Main Flow**
1. 관리자가 상품 ID를 지정하여 삭제를 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 상품을 소프트 삭제(`DELETED` 상태로 변경)한다
4. 해당 상품의 찜 이력은 보존한다

**Exception Flow**
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A007: 주문 목록 조회 (관리자)

**시나리오**: 관리자가 플랫폼 전체의 주문 현황을 파악하고 싶을 때 사용한다. 사용자 구분 없이 모든 주문을 최신순으로 조회한다.

**액터**: 관리자

**Main Flow**
1. 관리자가 전체 주문 목록을 요청한다
2. 시스템이 모든 주문을 최신순으로 조회한다
3. 주문 목록을 반환한다

**Alternate Flow**
- (2) 주문이 없으면 → 빈 목록을 반환한다

**Exception Flow**
- (1) 비관리자 접근 → `403 FORBIDDEN`
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-A008: 주문 상세 조회 (관리자)

**시나리오**: 관리자가 특정 주문의 상세 내역을 확인하고 싶을 때 사용한다. 어떤 사용자의 주문이든 조회 가능하다.

**액터**: 관리자

**Main Flow**
1. 관리자가 주문 ID를 지정하여 상세 정보를 요청한다
2. 시스템이 주문 존재 여부를 확인한다
3. 주문 상세 정보를 반환한다

**Exception Flow**
- (2) 주문이 존재하지 않으면 → `404 NOT_FOUND`
- (1) 비관리자 접근 → `403 FORBIDDEN`
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

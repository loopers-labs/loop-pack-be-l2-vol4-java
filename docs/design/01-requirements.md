## **0. 개요**

- 감성 이커머스 서비스의 상품, 브랜드, 좋아요, 주문 기능을 정의한다.
- 현재 문서에서 회원 기능 정의는 제외하며, 용어는 유비쿼터스 언어 기준을 따른다.
- 설계 범위
    - 상품 목록 / 상품 상세 / 브랜드 조회
    - 상품 좋아요 등록/취소 (멱등 동작)
    - 주문 생성 (재고 차감, 외부 시스템 연동)
    - 쿠폰 발급 및 적용 (정액/정률 할인, 재사용 불가)

## **1. 유비쿼터스 언어**

### **액터**

- 유저(`User`) : 회원가입을 완료하고 서비스를 이용하는 사용자. 주문, 좋아요 등의 행동의 주체
    - 게스트(`Guest`) : 비회원. 유저로 전환 가능.
    - 고객(`Customer`) : 유저와 게스트를 아울러 칭하는 용어.
- 어드민(`Admin`) : 브랜드, 상품, 주문을 관리하는 내부 관리자

### **주요 도메인**

| **한글** | **영문** | **정의** |
| --- | --- | --- |
| 브랜드 | `Brand` | 상품을 공급하는 판매 단위
모든 상품은 하나의 브랜드에 속함. |
| 상품 | `Product` | 브랜드가 판매하는 개별 판매 단위.
고객의 탐색, 주문, 좋아요의 대상.
가격(Price)과 재고(Stock)를 보유. |
| 좋아요 | `Like` | 로그인 한 고객이 특정 상품에 표현한 관심 관계.
고객-상품 쌍은 유일해야 함. |
| 주문 | `Order` | 고객이 상품을 구매하는 행위.
하나의 고객이 여러 개의 상품을 주문할 수 있으며, 동일한 상품을 여러 개, 여러 번 주문 할 수 있음. |
| 쿠폰 | `Coupon` | 어드민이 생성하는 쿠폰 마스터 데이터. 정액(`FIXED`) / 정률(`RATE`) 타입 및 할인 값, 최소 주문 금액 조건, 만료일을 보유. |
| 발급 쿠폰 | `IssuedCoupon` | 유저에게 발급된 쿠폰 인스턴스. 주문 시 1회만 사용 가능하며 사용 후 재사용 불가. |

### **상태 정의**

| **도메인** | **상태** | **값** | **비고**                                                          |
| --- | --- | --- |-----------------------------------------------------------------|
| `Order` | `OrderStatus` | `PENDING` / `CONFIRMED` / `CANCELED` | `PENDING` : 주문 요청, `CONFIRMED` : 주문 완료 (결제 완료), `CANCELED` : 취소 |
| `IssuedCoupon` | `CouponStatus` | `AVAILABLE` / `USED` / `EXPIRED` | `AVAILABLE` : 사용 가능, `USED` : 사용 완료, `EXPIRED` : 만료             |

## **2. 공통 정보**

### **URL 프리픽스**

- 대고객 기능은 `/api/v1` prefix를 통해 제공
- 어드민 기능은 `/api-admin/v1` prefix를 통해 제공

### **인증 체계**

- 유저 로그인이 필요한 기능은 `X-Loopers-LoginId` (로그인 ID), `X-Loopers-LoginPw` (비밀번호) 헤더를 통해 유저 식별
    - 인증/인가는 주요 스코프가 아니므로 구현하지 않음
    - 유저는 타 유저의 정보에 직접 접근할 수 없음
- 어드민 기능은 `X-Loopers-Ldap : loopers.admin` 헤더를 통해 식별해 제공

## **3. 제약 사항**

- 주문 정보에는 당시 상품의 정보가 스냅샷으로 저장되어야 함
- 주문 시에 상품 재고 확인 및 차감이 보장되어야 함
    - 동시에 동일 상품을 주문할 때 재고 초과 차감이 되지 않아야 함
- 회원이 좋아요를 등록/취소할 때 멱등성을 보장해야 함
- 상품 목록 / 주문 목록 조회는 허용 응답 시간 내에 처리되어야 함.
- 쿠폰은 주문 1건당 1장만 적용 가능하며, 사용된 쿠폰은 재사용이 불가능하다.
- 쿠폰 사용 처리와 재고 차감은 하나의 트랜잭션으로 처리되어야 하며 동시성 이슈(Lost Update)가 발생하지 않아야 한다.
- 주문 정보에는 쿠폰 적용 전 금액, 할인 금액, 최종 결제 금액이 스냅샷으로 저장되어야 한다.

### **브랜드 & 상품**

#### **[BRAND-001] 브랜드 정보 조회 (GET `/api/v1/brands/{brandId}`)**

- **유저 스토리**
    - 고객은 브랜드 정보를 조회할 수 있다.
    - 생성일자, 수정일자 등은 노출하지 않는다.
- **기능 흐름**
    1. 브랜드 ID로 브랜드 정보를 조회한다.
    2. 브랜드가 존재하지 않으면 `404` 반환.
    3. 브랜드 정보를 반환한다.

---

#### **[PRODUCT-001] 상품 목록 조회 (GET `/api/v1/products`)**

- **유저 스토리**
    - 고객은 상품 목록을 조회할 수 있다.
    - 고객은 브랜드를 기준으로 상품을 필터링 할 수 있다.
    - 고객은 최신순/가격순/좋아요 순으로 상품을 정렬할 수 있다.
- **기능 흐름**
    1. 요청 파라미터 (brandId, sort, page, size)를 받는다.
        - page와 size 값이 존재하지 않으면 `400` 반환
        - size는 최대 100개로 제한. 이 이상을 요청할 경우 `400` 반환.
    2. brandId 값이 있으면 해당 브랜드로 필터링한다.
        - 존재하지 않는 brandID이면 `404` 반환
    3. sort 조건을 적용한다.
        - latest : 최신 등록일 기준 (기본값)
        - price_asc : 낮은 가격 순
        - likes_desc : 좋아요 많은 순
        - 정의되지 않은 sort 값이면 `400` 반환
    4. page와 size 값으로 페이징을 적용해 반환
- **상품 목록 조회 쿼리 파라미터**

| **파라미터** | **예시** | **설명** |
| --- | --- | --- |
| `brandId` | `1` | 특정 브랜드의 상품만 필터링 |
| `sort` | `latest` / `price_asc` / `likes_desc` | 정렬 기준 |
| `page` | `0` | 페이지 번호 (기본값 0) |
| `size` | `20` | 페이지당 상품 수 (기본값 20) |

### 좋아요

#### [LIKE-001] 상품 좋아요 등록 (POST `api/v1/products/{productId}/likes`)

- **유저 스토리**
    - 로그인한 사용자는 상품을 찜할 수 있다.
    - 이미 찜한 상품을 다시 찜해도 무시된다.
- **기능 흐름**
    1. 헤더(`X-Loopers-LoginId`, `X-Loopers-LoginPw`)로 유저 식별
        - 헤더가 없거나 불일치할 경우 `401` 반환
        - 요청 헤더와 회원 정보가 맞지 않을 경우 `403` 반환
    2. productId로 상품을 조회한다
        - 존재하지 않거나 삭제되었으면 `404` 반환
    3. (user_id, product_id) 기준으로 좋아요 존재 여부를 확인한다
        - 이미 존재하면 무시 (멱등성 보장)
        - 존재하지 않으면 (회원, 상품) 쌍을 저장
    4. `200` OK 응답 반환

---

#### [ORDER-001] 주문 요청 (**POST** `/api/v1/orders` )

- **유저 스토리**
    - 유저는 여러 상품을 한 번에 주문할 수 있다.
    - 주문 시점의 상품 정보(상품명, 가격)가 스냅샷으로 저장된다.
    - 재고가 부족한 상품이 하나라도 있으면 주문 전체가 실패한다.

**Request Body**

```json
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ],
  "couponId": 42
}
```
> `couponId`는 미적용 시 생략 가능합니다 (nullable).

- **기능 흐름**
    1. `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더로 유저를 식별한다.
        - 헤더가 없거나 불일치하면 `401` 반환
    2. 요청된 `productId` 목록으로 상품을 조회한다.
        - 존재하지 않거나 Soft Delete된 상품이 포함되면 `404` 반환
    3. `couponId`가 제공된 경우 쿠폰 유효성 검증 및 사용 처리를 한다.
        - 쿠폰이 존재하지 않으면 `404` 반환
        - 쿠폰의 소유자가 요청 유저와 다르면 `403` 반환
        - 쿠폰 상태가 `AVAILABLE`이 아니거나 만료(`expiredAt` 초과)된 경우 `409` 반환
        - 최소 주문 금액 조건(`minOrderAmount`) 미충족 시 `400` 반환
        - 쿠폰 상태를 `USED`로 변경한다.
    4. 각 상품에 대해 비관적 락 **(`SELECT FOR UPDATE`)** 을 걸고 재고를 확인한다.
        - 재고가 부족한 상품이 하나라도 있으면 `409` 반환
    5. 모든 상품의 재고를 차감한다.
    6. 쿠폰 타입에 따라 할인 금액을 계산한다.
        - `FIXED`: 정액 차감
        - `RATE`: 총 금액 × 할인율(%)
        - 쿠폰 미적용 시 할인 금액은 `0`
    7. 외부 시스템 연동을 위해 `Outbox` 테이블에 이벤트 INSERT
        - 외부 시스템 실행은 다른 트랜잭션에서 outbox 테이블을 polling 하여 실행
    8. `Order`를 생성하고 상태를 `PENDING`으로 설정한다. 쿠폰 적용 전 금액(`originalPrice`), 할인 금액(`discountAmount`), 최종 결제 금액(`totalPrice`)을 저장한다.
    9. `OrderItem`을 생성한다. 이 때 상품명, 가격을 스냅샷으로 저장한다.
    10. `201` 응답과 함께 생성된 `orderId`를 반환한다.

- **비즈니스 규칙**
    - 쿠폰 사용 처리, 재고 확인·차감, 주문 생성은 하나의 트랜잭션으로 처리된다.
    - 트랜잭션 실패 시 쿠폰 상태 변경 및 재고 차감 모두 롤백된다.
    - 주문 생성 이후 상품 정보가 변경되어도 주문 내역은 변하지 않는다.
    - 쿠폰 적용 시 원래 금액, 할인 금액, 최종 결제 금액이 모두 `Order`에 스냅샷으로 저장된다.
    - 쿠폰 미적용 시 `discountAmount`는 `0`으로 저장된다.

| 파라미터 | 필수 | 설명 |

- **기능 흐름**

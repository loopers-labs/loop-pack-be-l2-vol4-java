## 2. 비즈니스 규칙

### 인증 / 역할

- 사용자(`USER`)와 관리자(`ADMIN`)는 역할로 구분한다
- `USER`: 상품 조회, 찜, 주문, 주문 조회
- `ADMIN`: 브랜드/상품/재고 관리, 전체 주문 조회

### 브랜드 / 상품

- 브랜드는 `ACTIVE` / `INACTIVE` 상태를 가진다
- 브랜드 이름은 최소 2글자 이상이어야 한다
- 브랜드 삭제 시 소프트 삭제한다 (`INACTIVE` 상태로 변경)
- 소프트 삭제된 브랜드는 존재하지 않는 것으로 간주한다
- 상품은 반드시 등록된 브랜드에 속해야 한다
- 상품의 브랜드는 등록 이후 변경할 수 없다
- 브랜드 삭제 시 소속 상품은 판매중지(`INACTIVE`) 상태로 변경되며, 해당 상품의 찜 이력은 보존된다
- 상품은 이름을 가지며, 가격과 재고 수량은 상품 재고(ProductStock) 단위로 관리된다
- 상품 재고는 옵션 단위로 분리될 수 있어 상품당 1개 이상 존재할 수 있다
- 재고 수량은 0 이상이어야 하며, 애플리케이션 레벨과 DB CHECK 제약 양쪽에서 보장한다
- 상품 이름은 최소 2글자 이상이어야 한다
- 주문 요청 시 해당 재고 항목의 수량이 차감된다

### 주문

- 주문 수량은 1 이상이어야 한다
- `ACTIVE` 상태의 상품만 주문할 수 있다 (판매중지 상품 주문 불가)

### 결제

- 결제는 주문 생성 시 함께 생성되며 초기 상태는 `PENDING`이다
- 한 주문에 결제 시도가 여러 번 존재할 수 있다 (재시도 허용)
- PG 승인 웹훅 수신 시 결제 상태를 `APPROVED`로, 주문 상태를 `COMPLETED`로 변경한다
- PG 실패 웹훅 수신 시 결제 상태를 `FAILED`로 변경한다. 주문은 `REQUESTED` 상태를 유지하며 사용자가 재시도할 수 있다
- 주문 생성 후 15분 내 `APPROVED` 결제가 없으면 배치가 진행 중인 결제를 `EXPIRED`로, 주문을 `CANCELLED`로 변경하고 재고를 원복한다 (보상 트랜잭션)
- 재시도는 주문이 `REQUESTED` 상태일 때만 가능하다

## 3. 유스케이스 흐름

> 표기 규칙
> - **Main Flow**: 정상 성공 경로
> - **Alternate Flow**: 조건 분기에 따른 대안 경로 (오류 아님)
> - **Exception Flow**: 오류·예외 상황 (괄호 안 숫자는 분기 발생 단계)

---

### UC-001: 브랜드별 상품 조회

**시나리오**: 사용자가 특정 브랜드의 상품 라인업을 탐색하고 싶을 때, 브랜드 ID로 해당 브랜드의 판매 중인 상품 목록을 조회한다.

**액터**: 사용자

**Main Flow**
1. 사용자가 브랜드 ID를 지정하여 상품 목록을 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 해당 브랜드에 속한 `ACTIVE` 상태의 상품 목록을 반환한다

**Alternate Flow**
- (3) `ACTIVE` 상품이 없으면 → 빈 목록을 반환한다

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `404 NOT_FOUND`

---

### UC-002: 전체 상품 목록 조회

**시나리오**: 사용자가 플랫폼에서 판매 중인 모든 상품을 둘러보고 싶을 때, 브랜드 구분 없이 전체 상품 목록을 조회한다.

**액터**: 사용자

**Main Flow**
1. 사용자가 전체 상품 목록을 요청한다
2. 시스템이 `ACTIVE` 상태의 전체 상품을 조회한다
3. 상품 목록을 반환한다

**Alternate Flow**
- (2) `ACTIVE` 상품이 없으면 → 빈 목록을 반환한다

---

### UC-003: 상품 상세 조회

**시나리오**: 사용자가 특정 상품의 이름, 가격, 재고 등 상세 정보를 확인하고 싶을 때, 상품 ID로 상세 정보를 조회한다.

**액터**: 사용자

**Main Flow**
1. 사용자가 상품 ID를 지정하여 상세 정보를 요청한다
2. 시스템이 상품 존재 여부를 확인한다
3. 상품 상세 정보를 반환한다

**Exception Flow**
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`

---

### UC-004: 상품 찜 / 찜 취소

**시나리오**: 사용자가 관심 있는 상품을 찜 목록에 추가하거나, 이미 찜한 상품을 찜 목록에서 제거하고 싶을 때 사용한다. 동일 요청으로 등록/취소를 토글한다.

**액터**: 사용자 (로그인 필요)

**Main Flow** (찜 등록)
1. 사용자가 특정 상품에 찜 요청을 한다
2. 시스템이 상품 존재 여부를 확인한다
3. 시스템이 해당 사용자의 기존 찜 이력을 확인한다
4. 찜 이력이 없으면 찜을 등록한다

**Alternate Flow** (찜 취소)
- (3) 이미 찜한 상품이면 → 기존 찜을 취소(삭제)한다

**Exception Flow**
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-005: 찜 목록 조회

**시나리오**: 사용자가 본인이 찜해 둔 상품 목록을 한눈에 확인하고 싶을 때 사용한다. 삭제된 상품은 목록에서 제외된다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 본인의 찜 목록을 요청한다
2. 시스템이 사용자의 찜 이력을 조회한다
3. 소프트 삭제된 상품을 제외한 찜 목록을 반환한다

**Alternate Flow**
- (2) 찜한 상품이 없거나 모두 삭제된 경우 → 빈 목록을 반환한다

**Exception Flow**
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-006: 주문 및 결제 요청

**시나리오**: 사용자가 원하는 상품 목록을 수량과 함께 지정하여 구매하고 싶을 때 사용한다. 재고를 즉시 차감하고 주문을 생성한 뒤, PG 결제창 호출에 필요한 결제 요청 정보를 반환한다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 상품 재고 목록(재고 ID, 수량)을 지정하여 주문을 요청한다
2. 시스템이 상품 재고 목록 각각의 존재 여부를 확인한다
3. 시스템이 상품 재고 목록 각각의 재고와 요청 수량을 비교한다
4. 재고를 요청 수량만큼 차감한다
5. 주문을 생성한다 (`REQUESTED` 상태)
6. 결제를 생성한다 (`PENDING` 상태)
7. 결제 요청 정보를 반환한다

**Exception Flow**
- (1) 재고 목록이 비어 있으면 → `400 BAD_REQUEST`
- (1) 재고 목록 중 수량이 1 미만인 항목이 있으면 → `400 BAD_REQUEST`
- (2) 재고 목록 중 존재하지 않는 재고 항목이 있으면 → `404 NOT_FOUND`
- (2) 재고 목록 중 연결된 상품이 `ACTIVE` 상태가 아닌 항목이 있으면 → `400 BAD_REQUEST` (판매중지 포함)
- (3) 재고 목록 중 재고가 부족한 항목이 있으면 → `400 BAD_REQUEST` (주문 미생성, 재고 변동 없음)
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-007: 주문 목록 조회

**시나리오**: 사용자가 본인의 주문 내역을 최신순으로 확인하고 싶을 때 사용한다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 본인의 주문 목록을 요청한다
2. 시스템이 해당 사용자의 주문을 최신순으로 조회한다
3. 주문 목록을 반환한다

**Alternate Flow**
- (2) 주문 내역이 없으면 → 빈 목록을 반환한다

**Exception Flow**
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-008: 주문 상세 조회

**시나리오**: 사용자가 특정 주문의 상품, 수량 등 상세 내역을 확인하고 싶을 때 사용한다. 본인 주문만 조회할 수 있다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 주문 ID를 지정하여 상세 정보를 요청한다
2. 시스템이 주문 존재 여부를 확인한다
3. 시스템이 해당 주문의 소유자를 확인한다
4. 주문 상세 정보를 반환한다

**Exception Flow**
- (2) 주문이 존재하지 않으면 → `404 NOT_FOUND`
- (3) 본인 주문이 아니면 → `403 FORBIDDEN`
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

---

### UC-009: 결제 승인 처리

**시나리오**: PG가 결제 승인 결과를 웹훅으로 전송하면, 시스템이 이를 검증하고 주문을 완료 처리한다. 재고는 주문 요청(UC-006) 시점에 이미 차감된다.

**액터**: PG (외부 시스템)

**Main Flow**
1. PG가 결제 승인 결과를 웹훅으로 전송한다
2. 시스템이 웹훅 유효성을 검증한다
3. 결제 상태를 `APPROVED`로 변경한다
4. 주문 상태를 `COMPLETED`로 변경한다

**Exception Flow**
- (2) 웹훅 유효성 검증 실패 → 처리 무시

---

### UC-010: 결제 실패 처리

**시나리오**: PG가 결제 실패 결과를 웹훅으로 전송하면, 시스템이 이를 검증하고 해당 결제를 실패 처리한다. 주문은 `REQUESTED` 상태를 유지하며, 사용자는 결제를 재시도할 수 있다.

**액터**: PG (외부 시스템)

**Main Flow**
1. PG가 결제 실패 결과를 웹훅으로 전송한다
2. 시스템이 웹훅 유효성을 검증한다
3. 결제 상태를 `FAILED`로 변경한다
4. 주문은 `REQUESTED` 상태를 유지한다 (재시도 가능)

**Exception Flow**
- (2) 웹훅 유효성 검증 실패 → 처리 무시

---

### UC-011: 미완료 결제 만료 처리 (배치)

**시나리오**: 주문 생성 후 15분이 지나도 승인된 결제가 없는 주문을 배치가 주기적으로 스캔하여 만료 처리한다. 진행 중인 결제를 `EXPIRED`로, 주문을 `CANCELLED`로 변경하고 차감된 재고를 전량 원복한다.

**액터**: 시스템 (배치, `commerce-batch`)

**Main Flow**
1. 배치가 주기적으로 실행된다
2. 생성된 지 15분이 경과했으나 `APPROVED` 결제가 없는 `REQUESTED` 주문을 조회한다
3. 해당 주문의 `PENDING` 상태 결제를 `EXPIRED`로 변경한다
4. 주문 상태를 `CANCELLED`로 변경한다
5. 주문 항목 각각에 대해 차감된 재고를 원복한다 (보상 트랜잭션)

**Alternate Flow**
- (2) 대상 주문이 없으면 → 처리 없이 종료

---

### UC-012: 결제 재시도

**시나리오**: PG 실패 등으로 결제가 실패한 사용자가 동일 주문에 대해 결제를 다시 시도하고 싶을 때 사용한다. 주문이 `REQUESTED` 상태일 때만 가능하다.

**액터**: 사용자 (로그인 필요)

**Main Flow**
1. 사용자가 주문 ID를 지정하여 결제 재시도를 요청한다
2. 시스템이 주문 존재 여부 및 소유자를 확인한다
3. 시스템이 주문이 `REQUESTED` 상태인지 확인한다
4. 새로운 결제를 생성한다 (`PENDING` 상태)
5. 결제 요청 정보를 반환한다

**Exception Flow**
- (2) 주문이 존재하지 않으면 → `404 NOT_FOUND`
- (2) 본인 주문이 아니면 → `403 FORBIDDEN`
- (3) 주문이 `REQUESTED` 상태가 아니면 → `400 BAD_REQUEST`
- (3) 주문 생성 후 15분이 경과했으면 → `400 BAD_REQUEST` (결제 시도 불가)
- (1) 비로그인 상태 → `401 UNAUTHORIZED`

### UC-A001: 브랜드 등록

**시나리오**: 관리자가 새로운 브랜드를 플랫폼에 등록하고 싶을 때 사용한다. 등록 완료 시 생성된 브랜드 ID를 반환한다.

**액터**: 관리자

**Main Flow**
1. 관리자가 브랜드 이름을 입력하여 등록을 요청한다
2. 시스템이 브랜드명 중복 여부를 확인한다
3. 브랜드를 등록하고 생성된 브랜드 ID를 반환한다

**Exception Flow**
- (2) 브랜드명이 이미 사용 중 → `409 CONFLICT`
- (1) 필수 입력값 누락 → `400 BAD_REQUEST`
- (1) 이름이 2글자 미만 → `400 BAD_REQUEST`
- (1) 비관리자 접근 → `403 FORBIDDEN`

---

### UC-A002: 브랜드 수정

**시나리오**: 관리자가 기존 브랜드의 이름을 변경하고 싶을 때 사용한다. 변경할 이름이 현재 이름과 다를 경우에만 중복 여부를 검증한다.

**액터**: 관리자

**Main Flow**
1. 관리자가 브랜드 ID와 수정할 이름을 지정하여 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 변경할 이름이 현재 이름과 다른지 확인한다
4. 변경할 이름의 중복 여부를 확인한다
5. 브랜드 이름을 수정한다

**Alternate Flow**
- (3) 변경할 이름이 현재 이름과 동일하면 → 중복 확인(4단계)을 생략하고 수정을 진행한다

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `404 NOT_FOUND`
- (4) 변경할 이름이 이미 사용 중 → `409 CONFLICT`
- (1) 이름이 2글자 미만 → `400 BAD_REQUEST`
- (1) 비관리자 접근 → `403 FORBIDDEN`

### UC-A003: 브랜드 삭제

**시나리오**: 관리자가 브랜드를 플랫폼에서 제거하고 싶을 때 사용한다. 소속 상품은 함께 소프트 삭제되며, 기존 찜 이력은 데이터 무결성을 위해 보존된다.

**액터**: 관리자

**Main Flow**
1. 관리자가 브랜드 ID를 지정하여 삭제를 요청한다
2. 시스템이 브랜드 존재 여부를 확인한다
3. 해당 브랜드에 소속된 상품을 판매중지(`INACTIVE`) 상태로 변경한다
4. 판매중지된 상품의 찜 이력은 보존한다
5. 브랜드를 소프트 삭제한다 (`INACTIVE` 상태로 변경)

**Exception Flow**
- (2) 브랜드가 존재하지 않으면 → `404 NOT_FOUND`
- (1) 비관리자 접근 → `403 FORBIDDEN`

### UC-A005: 상품 수정

**시나리오**: 관리자가 상품의 이름·가격·재고를 수정하고 싶을 때 사용한다. 요청에 포함된 키만 수정되며, 키가 없는 필드는 변경하지 않는다. 브랜드 변경은 불가하다. 재고는 증감량(델타) 방식으로 수정하며, 차감 후 재고가 음수가 되지 않아야 한다.

**액터**: 관리자

**Main Flow**
1. 관리자가 상품 ID와 수정할 필드(이름?, 가격?, 재고 증감량?)를 포함하여 요청한다
   - `stockQuantity`: 양수이면 재고 증가, 음수이면 재고 차감 (예: `+5` → 현재 재고 10개에서 15개, `-3` → 7개)
2. 시스템이 상품 존재 여부를 확인한다
3. 시스템이 브랜드 변경 시도 여부를 확인한다
4. 키가 존재하는 필드만 각각 검증한다
5. 키가 존재하는 필드만 수정한다

**Exception Flow**
- (2) 상품이 존재하지 않으면 → `404 NOT_FOUND`
- (3) 브랜드 변경 시도 → `400 BAD_REQUEST`
- (4) `name` 키가 있고 값이 2글자 미만 → `400 BAD_REQUEST`
- (4) `stockQuantity` 키가 있고 값이 0 → `400 BAD_REQUEST` (변경량이 없음)
- (4) `stockQuantity` 키가 있고 차감 후 재고가 음수 → `400 BAD_REQUEST`
- (1) 비관리자 접근 → `403 FORBIDDEN`

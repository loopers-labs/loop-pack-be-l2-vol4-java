## 2. 핵심 설계 결정

| 항목 | 결정 | 근거 |
|---|---|---|
| 도메인 | Brand, Product, Like, Order(+OrderItem) | 요구 기능이 이 네 개념을 중심으로 구성되어 도메인 경계와 일치한다. |
| 주문 F-06 범위 | 재고 차감 + 외부 연동 (포인트 차감 제외) | 포인트는 별도 도메인이 필요해 1차 범위에서 제외하고, 재고·외부 연동에 집중하기로 합의했다. |
| 주문 트랜잭션 | 단계 분리 + 보상 트랜잭션 | 외부 PG 호출을 트랜잭션에 묶으면 커넥션을 오래 점유하고 정합성이 위험하다. 단계를 분리해 PG 실패 시 재고를 보상한다. |
| 외부 연동 | PG 결제 승인 (동기 호출) | 결제 성공/실패가 주문 확정 여부를 결정하므로, 응답을 기다려 분기해야 한다. |
| 재고 차감 동시성 | 비관적 락 (`SELECT … FOR UPDATE`로 상품 조회 후 차감) | 재고는 경쟁이 한 행에 몰려 낙관적 락은 재시도가 폭증한다. 검증 로직을 도메인 모델에 유지하려면 조건부 UPDATE보다 락 조회 후 모델 메서드 차감이 적합하다. |
| Like 구조 | 복합키 `LikeId(userId, productId)`, `BaseEntity` 미상속, `likedAt` nullable 토글 | 복합 PK가 중복 좋아요를 스키마로 차단한다. `likedAt` 토글은 행을 보존해 취소 후 재등록이 UNIQUE 충돌 없이 UPDATE로 처리된다. `BaseEntity`는 자체 `@Id`를 가져 복합키와 양립할 수 없다. |
| 좋아요 수 집계 | `COUNT(*) WHERE liked_at IS NOT NULL` (비정규화 컬럼 없음) | 비정규화 카운터는 토글 시 동시성·정합성 관리 부담이 있다. 1차 범위에서는 정확성을 우선해 COUNT로 집계한다. |
| 멱등 정책 | 같은 요청 반복 시 `likedAt` 유지 (관측 상태 불변) | 멱등의 엄밀한 정의는 반복 요청 시 관측 상태가 불변인 것이다. `likedAt`을 갱신하면 시각이 바뀌어 엄밀한 멱등이 깨진다. |
| 주문 상태 | `PENDING` → `PAID` / `PENDING` → `FAILED` | 단계 분리 흐름의 상태 전이를 표현하는 최소 집합이다. |

## 3. 설계 전제 및 범위 가정

- 상품 목록 정렬은 최신순 / 가격 오름차순·내림차순 / 좋아요순을 지원한다.
- 페이지네이션은 offset 방식(`page`, `size`)을 사용한다.
- 상품 목록은 `brandId` 필터를 지원하며, "특정 브랜드의 상품 목록"도 이 필터로 처리한다.
- 소프트 삭제된 상품은 목록·상세에서 제외한다. 재고 0 상품은 노출하되 품절로 표시한다.
- 주문은 다건 상품(`OrderItem` N개)을 포함한다.
- `OrderItem`은 주문 시점의 상품명·단가를 스냅샷으로 저장한다.

## 4. 기능 요구사항

각 기능을 목표·대상, Happy Path, Unhappy Path, 비기능 요구사항으로 정의한다.

### F-01. 상품 목록 조회

**목표:** 사용자는 살 수 있는 상품을 탐색하고, 관리자는 판매중지 상품까지 포함해 전체 상품을 조회한다.
<br>
**대상:** 비로그인 포함 모든 사용자(판매중 상품만), 관리자(판매중·판매중지 모두). 1차 범위는 사용자 조회만 구현하며, 관리자 기능은 명세에만 둔다.

**Happy Path.**
1. 사용자(또는 관리자)가 상품 목록을 요청한다.
2. 시스템은 권한에 따라 노출 범위를 결정한다. 사용자에게는 판매중 상품만, 관리자에게는 판매중·판매중지 상품을 모두 노출한다.
3. 정렬 기준(최신순 / 가격순 / 좋아요순)에 따라 정렬한다. 기준이 없으면 최신순으로 보여준다.
4. 사용자가 특정 브랜드를 지정하면 해당 브랜드의 상품만 보여준다.
5. 각 상품에 좋아요 수를 함께 표시한다.
6. 목록은 페이지 단위로 제공되며, 다음 페이지를 이어서 볼 수 있다.

**Unhappy Path.**
- (관리자 기능 구현 시) 일반 사용자가 관리자 엔드포인트를 호출 → 권한 없음으로 응답한다.

### F-02. 상품 상세 조회

**목표:** 사용자는 살 수 있는 상품의 상세 정보를 확인해 구매를 판단한다. 관리자는 판매중지 상품까지 포함해 상세를 점검한다.
<br>
**대상:** 비로그인 포함 모든 사용자(판매중 상품만), 관리자(판매중·판매중지 모두). 응답 필드는 권한과 무관하게 동일하다. 1차 범위는 사용자 조회만 구현하며, 관리자 기능은 명세에만 둔다.

**Happy Path.**
1. 사용자(또는 관리자)가 특정 상품의 상세를 요청한다 (상품 ID 기준).
2. 시스템은 권한에 따라 노출 범위를 결정한다. 사용자에게는 판매중 상품만, 관리자에게는 판매중·판매중지 상품을 모두 노출한다.
3. 상품 정보(이름·설명·가격·재고·좋아요 수)와 소속 브랜드 정보를 함께 보여준다.
4. 재고가 0이면 품절로 표시한다.

**Unhappy Path.**
- 존재하지 않는 상품 → 404 응답.
- 소프트 삭제된 상품 → 404 응답.
- 사용자가 판매중지 상품을 조회 → 404 응답.
- 소속 브랜드가 소프트 삭제된 경우 → 404 응답.
- (관리자 기능 구현 시) 일반 사용자가 관리자 엔드포인트를 호출 → 권한 없음으로 응답한다.

### F-03. 브랜드 목록·검색 조회

**목표:** 사용자는 어떤 브랜드들이 있는지 훑거나 이름으로 검색해 원하는 브랜드를 찾는다. 관리자는 소프트 삭제된 브랜드까지 포함해 점검한다.
<br>
**대상:** 비로그인 포함 모든 사용자(삭제되지 않은 브랜드만), 관리자(소프트 삭제 포함). 1차 범위는 사용자 조회만 구현하며, 관리자 기능은 명세에만 둔다.

**Happy Path.**
1. 사용자(또는 관리자)가 브랜드 조회를 요청한다 (검색어는 선택값).
2. 시스템은 권한에 따라 노출 범위를 결정한다. 사용자에게는 소프트 삭제되지 않은 브랜드만, 관리자에게는 소프트 삭제된 브랜드까지 모두 노출한다.
3. 검색어 포함 여부에 따라 분기한다.
   - 검색어가 없으면 전체 브랜드 목록을 조회한다.
   - 검색어가 있으면 브랜드명이 검색어를 부분 포함하는 목록을 조회한다 (`name LIKE %검색어%`).
4. 브랜드명 가나다순으로 정렬해 페이지 단위로 보여준다. 다음 페이지를 이어서 볼 수 있다.
5. 일치하는 브랜드가 없으면 빈 목록을 반환한다.

**Unhappy Path.**
- (관리자 기능 구현 시) 일반 사용자가 관리자 엔드포인트를 호출 → 권한 없음으로 응답한다.

### F-04. 상품 좋아요 등록 (멱등)

**목표:** 로그인한 사용자가 관심 있는 상품에 좋아요를 표시한다. 같은 동작을 반복해도 결과는 한 번 한 것과 같다.
<br>
**대상:** 로그인한 사용자 (관리자 역할 포함, 권한 분기 없음).

**Happy Path.**
1. 로그인한 사용자가 특정 상품에 좋아요 등록을 요청한다 (상품 ID 기준).
2. 시스템은 해당 사용자가 그 상품에 좋아요 상태인지 확인한다.
   - 이미 좋아요 상태이면 추가 등록 없이 성공으로 응답한다 (멱등 처리).
   - 좋아요 상태가 아니면 좋아요를 등록한다.
3. 성공으로 응답한다.

**Unhappy Path.**
- 존재하지 않는 상품 → 404 응답.
- 소프트 삭제된 상품 → 404 응답.
- 판매중지 상품 → 404 응답.
- 비로그인 사용자의 요청 → 401 응답.
- 인증 헤더는 있으나 자격이 유효하지 않음(존재하지 않는 사용자·잘못된 비밀번호) → 401 응답.

### F-05. 상품 좋아요 취소 (멱등)

**목표:** 로그인한 사용자가 자신의 좋아요를 취소한다. 이미 취소된 상태에서 다시 취소해도 결과는 동일하다.
<br>
**대상:** 로그인한 사용자 (관리자 역할 포함, 권한 분기 없음).

**Happy Path.**
1. 로그인한 사용자가 특정 상품에 좋아요 취소를 요청한다 (상품 ID 기준).
2. 시스템은 해당 사용자가 그 상품에 좋아요 상태인지 확인한다.
   - 좋아요 상태가 아니면 추가 처리 없이 성공으로 응답한다 (멱등 처리).
   - 좋아요 상태이면 좋아요를 취소한다.
3. 성공으로 응답한다.

**Unhappy Path.**
- 존재하지 않는 상품 → 404 응답.
- 소프트 삭제된 상품 → 404 응답.
- 판매중지 상품 → 404 응답.
- 비로그인 사용자의 요청 → 401 응답.
- 인증 헤더는 있으나 자격이 유효하지 않음(존재하지 않는 사용자·잘못된 비밀번호) → 401 응답.

### F-06. 주문 생성 및 결제

**[F-06 유저 스토리]**

- 로그인한 사용자에 한해서만 주문 생성 및 재고차감 및 결제흐름을 진행하도록 한다.
- 사용자는 한 주문에 하나 이상의 (상품, 수량)을 담아 주문하며, 같은 상품을 중복으로 담으면 수량이 합산되어 한 항목으로 처리된다.
- 결제가 성공해야만 주문이 확정되고, 결제가 실패하거나 응답이 없으면 차감했던 재고가 원상 복구되며 주문은 실패로 남는다.
- 주문 항목의 금액은 주문 시점 가격으로 고정되어, 이후 상품 가격이 바뀌어도 변하지 않는다.
- 동시에 같은 상품 주문이 몰려도 실제 재고보다 많이 팔리지 않는다.

**[F-06 기능 흐름]**

1. 로그인 사용자가 하나 이상의 (상품, 수량)을 담아 주문을 요청한다.
2. 항목이 비어 있거나 수량이 0 이하면 `400`으로 거절하고 주문을 만들지 않는다. 같은 상품이 여러 번 담겨 있으면 수량을 합산해 한 항목으로 만든다.
3. **(트랜잭션 1)** 각 상품을 비관적 락(`SELECT … FOR UPDATE`)으로 조회해 존재·재고를 검증한다. 미존재·삭제 상품이면 `404`, 재고보다 많은 수량이면 `409`로 거절한다. 검증을 통과하면 주문을 `PENDING`으로 생성하고, 주문 시점의 상품명·단가를 항목에 스냅샷으로 기록한 뒤, 주문 수량만큼 재고를 차감하고 커밋한다.
4. **(트랜잭션 밖)** 외부 PG에 주문 총액의 결제 승인을 동기로 요청한다.
5. **승인 성공 — (트랜잭션 2)** 주문을 `PAID`로 확정하고 주문 결과를 응답한다.
6. **승인 실패·타임아웃 — (보상 트랜잭션)** 차감했던 재고를 복구하고 주문을 `FAILED`로 처리한 뒤 결제 실패를 응답한다.

**목표:** 구매자가 상품을 주문하고 외부 결제를 통해 주문을 확정한다. 결제가 실패하면 재고가 원상 복구된다.
<br>
**대상:** 로그인한 사용자.

**Happy Path.**
1. 로그인한 사용자가 하나 이상의 상품과 수량을 담아 주문을 요청한다.
2. 시스템은 각 상품의 존재와 재고를 확인한다.
3. 주문을 생성하고, 주문 시점의 상품명·가격을 주문 항목에 기록한다.
4. 주문한 수량만큼 상품 재고를 차감한다.
5. 외부 PG에 주문 총액의 결제 승인을 요청한다.
6. 승인이 성공하면 주문을 결제 완료 상태로 확정한다.
7. 사용자에게 주문 결과를 응답한다.

**Unhappy Path.**
- 주문 항목이 비어 있음(상품 0개) → 400 응답, 주문을 생성하지 않는다.
- 주문 수량이 0 이하 → 400 응답, 주문을 생성하지 않는다.
- 한 주문에 같은 상품을 여러 항목으로 중복 지정 → 동일 상품의 수량을 합산해 단일 항목으로 처리한다.
- 존재하지 않는·삭제된 상품 포함 → 404 응답, 주문을 생성하지 않는다.
- 재고보다 많은 수량 주문 → 409 응답, 주문을 생성하지 않는다.
- 동시 주문으로 재고 경합 발생 → 비관적 락으로 직렬화하고, 재고가 부족해진 요청은 409 응답으로 거절한다.
- PG 결제 승인 실패·타임아웃 → 차감했던 재고를 복구하고 주문을 실패 상태로 처리한 뒤 결제 실패를 응답한다.
- PG 승인은 성공했으나 이후 주문 확정 처리가 실패 → 주문이 `PENDING`에 잔류한다 (8장 리스크 참고).
- 비로그인 사용자의 요청 → 401 응답.

**비기능 요구사항.**
- 동시 주문이 몰려도 재고가 실제 수량보다 더 팔리지 않는다 (오버셀 방지).
- 외부 PG 호출은 트랜잭션 경계 밖에서 수행해, 외부 지연이 DB 커넥션을 점유하지 않는다.
- 결제 실패 시 재고와 주문 상태가 어긋나지 않도록 보상 처리한다.
- 주문 항목의 가격은 주문 시점 가격으로 고정되어, 이후 상품가가 바뀌어도 변하지 않는다.
- 주문 생성 흐름은 외부 PG 응답 시간을 제외하고 2초 이내에 응답한다.

## 7. 공통 규약

- 모든 응답은 `ApiResponse<T>`로 감싼다. 실패는 `ApiControllerAdvice`가 변환한다.
- 인증이 필요한 기능은 `@LoginUser String loginId` 파라미터로 사용자를 식별한다.
- 컨트롤러는 엔티티(`*Model`)를 응답으로 노출하지 않는다. `Info` → `Response` 두 단계로 변환한다.

## 8. 차후 고려 리스크

아래는 현 설계가 안고 있는 알려진 리스크다. 1차 구현 범위에서는 수용하되, 운영·확장 단계에서 차후 고려한다.

| 리스크 | 내용 | 차후 고려 방향 |
|---|---|---|
| 결제-주문 상태 불일치 | PG 결제 승인은 성공했으나 3단계 `confirm` 트랜잭션이 실패하면, 결제는 됐는데 주문이 `PENDING`에 멈춘다. 보상 트랜잭션으로도 메우지 못하는 구간이다. | `PENDING` 장기 방치 주문을 주기적으로 점검·정리하는 배치, 또는 `confirm` 실패 시 PG 결제 취소 호출. |
| 좋아요 COUNT 집계 비용 | 좋아요 수를 매 조회 시 `COUNT`로 계산한다. 상품 목록·좋아요순 정렬에서 상품 수에 비례해 쿼리·정렬 비용이 커진다. | 트래픽 증가 시 `product.like_count` 비정규화 컬럼으로 전환. |
| Like의 audit 불일치 | `LikeModel`은 `BaseEntity`를 상속하지 않아 `created_at`·`updated_at`·`deleted_at`이 없다. 다른 모델과 audit 패턴이 다르다. | 좋아요 이력 추적이 필요해지면 별도 audit 컬럼 추가 검토. |
| Like 행 단조 증가 | 좋아요 취소는 행을 삭제하지 않고 `likedAt`을 `null`로 둔다. 한 번이라도 좋아요한 `(user, product)` 쌍의 행이 영구히 누적된다. | 데이터량이 커지면 오래된 취소 행 아카이빙 검토. |

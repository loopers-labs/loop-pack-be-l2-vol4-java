# 01. 요구사항 명세

## 1. 서비스 개요

본 서비스는 감성 이커머스 플랫폼으로, 사용자가 브랜드별 상품을 탐색하고, 관심 상품에 좋아요를 등록하며, 여러 상품을 한 번에 주문할 수 있는 기능을 제공한다.

설계 범위는 다음과 같다.

- 상품 목록 조회 / 상품 상세 조회 / 브랜드 조회
- 상품 좋아요 등록 및 취소 (멱등 동작)
- 주문 생성 (재고 차감 보장, 상품 정보 스냅샷 저장)

다음 도메인은 설계 범위에서 제외한다.

- 회원가입
- 내 정보 조회
- 결제 (추후 별도 추가 예정)

---

## 2. 주요 사용자

| 사용자 유형 | 설명 |
| --- | --- |
| 일반 사용자 (User) | 상품을 탐색하고, 좋아요를 누르고, 주문을 생성하는 고객 |
| 어드민 (Admin) | 브랜드와 상품을 등록·수정·삭제하고, 주문을 관리하는 운영자 |

인증은 요청 헤더(`X-Loopers-LoginId`, `X-Loopers-LoginPw`)로 사용자를 식별하며, 인증/인가 구현 자체는 설계 범위에서 제외한다.

## 3. 유저 시나리오

### 3.1 일반 사용자 시나리오

사용자 김민준은 패션에 관심이 많은 20대 직장인이다.

1. 김민준은 앱을 열어 상품 목록을 스크롤하며 최신순으로 탐색한다.
2. 마음에 드는 상품을 클릭해 상세 정보(가격, 재고, 브랜드 등)를 확인한다.
3. 해당 브랜드의 다른 상품도 보고 싶어 브랜드 정보를 조회한다.
4. 마음에 드는 상품 두 가지에 좋아요를 등록해 나중에 다시 확인할 수 있도록 저장한다.
5. 상품 A 2개, 상품 B 1개를 한 번에 주문 요청한다.
6. 시스템은 재고를 확인하고 차감한 뒤 주문을 생성한다.
7. 김민준은 주문 내역을 조회하여 주문 상태를 확인한다.

### 3.2 어드민 시나리오

어드민 이수진은 이커머스 플랫폼의 운영 담당자이다.

1. 이수진은 새로운 패션 브랜드와 파트너십을 맺고 브랜드를 시스템에 등록한다.
2. 해당 브랜드의 신상품 3종을 상품명, 가격, 재고 수량과 함께 등록한다.
3. 상품 정보 오류를 발견하여 가격과 재고 수량을 수정한다. 단, 브랜드는 수정하지 않는다.
4. 시즌 종료로 특정 브랜드를 서비스에서 제거한다. 해당 브랜드의 모든 상품도 함께 삭제된다.
5. 주문 이상 여부를 확인하기 위해 전체 주문 목록을 페이지 단위로 조회한다.
6. 특정 주문의 상세 정보를 조회하여 주문 항목과 결제 상태를 확인한다.

## 5. 기능별 상세 요구사항

### 5.1 상품 목록 조회

#### 사용자 시나리오

사용자는 앱 진입 시 상품 목록을 탐색하며 원하는 상품을 찾는다. 특정 브랜드만 보거나 가격순/최신순으로 정렬할 수 있다.

#### 기본 흐름

1. 사용자가 상품 목록 화면에 진입한다.
2. 클라이언트는 `GET /api/v1/products`를 호출한다. 선택적으로 `brandId`, `sort`, `page`, `size` 파라미터를 포함한다.
3. 서버는 조건에 맞는 상품 목록을 페이지네이션하여 반환한다.
4. 클라이언트는 상품 목록을 화면에 표시한다.

#### 예외 흐름

- 존재하지 않는 `brandId`로 필터링 요청 시 빈 목록을 반환한다.
- `page`, `size` 파라미터가 유효하지 않은 경우 기본값(page=0, size=20)을 사용한다.

#### 기능 요구사항

- 사용자는 전체 상품 목록을 페이지 단위로 조회할 수 있어야 한다.
- `brandId` 파라미터로 특정 브랜드의 상품만 필터링할 수 있어야 한다.
- `sort` 파라미터로 최신순(`latest`), 가격 오름차순(`price_asc`), 좋아요 많은 순(`likes_desc`) 정렬을 지원한다.
- `latest` 정렬은 필수이며, 나머지 정렬 기준은 선택 구현이다.

#### 비기능 요구사항 및 정책

- 기본 페이지 크기는 20이다.
- 정렬 기준이 명시되지 않으면 최신순(`latest`)을 기본값으로 사용한다.

### 5.2 상품 상세 조회

#### 사용자 시나리오

사용자는 목록에서 관심 있는 상품을 클릭하여 상품명, 가격, 재고 수량, 브랜드 정보 등 상세 정보를 확인한다.

#### 기본 흐름

1. 사용자가 상품 목록에서 특정 상품을 클릭한다.
2. 클라이언트는 `GET /api/v1/products/{productId}`를 호출한다.
3. 서버는 상품 ID로 상품 정보를 조회한다.
4. 서버는 상품 상세 정보를 반환한다.
5. 클라이언트는 상품 상세 화면을 표시한다.

#### 예외 흐름

- 존재하지 않는 `productId`로 요청 시 404 응답을 반환한다.
- 삭제된 상품에 접근 시 404 응답을 반환한다.

#### 기능 요구사항

- 사용자는 상품 ID로 상품 상세 정보를 조회할 수 있어야 한다.
- 응답에는 상품명, 가격, 재고 수량, 판매 상태, 브랜드 정보가 포함되어야 한다.

---

### 5.3 브랜드 조회

#### 사용자 시나리오

사용자는 특정 상품의 브랜드 정보를 확인하거나, 브랜드 상세 페이지에서 해당 브랜드를 소개받는다.

#### 기본 흐름

1. 사용자가 브랜드 정보 조회를 요청한다.
2. 클라이언트는 `GET /api/v1/brands/{brandId}`를 호출한다.
3. 서버는 브랜드 ID로 브랜드 정보를 조회한다.
4. 서버는 브랜드 상세 정보를 반환한다.

#### 예외 흐름

- 존재하지 않는 `brandId`로 요청 시 404 응답을 반환한다.

#### 기능 요구사항

- 사용자는 브랜드 ID로 브랜드 정보를 조회할 수 있어야 한다.
- 응답에는 브랜드명이 포함되어야 한다.

---

### 5.4 상품 좋아요 등록

#### 사용자 시나리오

사용자는 관심 있는 상품을 나중에 다시 확인하기 위해 좋아요를 등록한다.

#### 기본 흐름

1. 사용자가 상품 상세 화면에서 좋아요 버튼을 클릭한다.
2. 클라이언트는 `POST /api/v1/products/{productId}/likes`를 호출한다.
3. 시스템은 요청 헤더에서 사용자를 식별한다.
4. 시스템은 해당 사용자가 이미 좋아요를 등록했는지 확인한다.
5. 기존 좋아요가 없으면 좋아요를 등록하고, 변경된 좋아요 상태를 반환한다.

#### 예외 흐름

- 로그인하지 않은 사용자가 요청하면 인증 오류(401)를 반환한다.
- 존재하지 않는 상품에 요청하면 404 응답을 반환한다.
- 이미 좋아요가 등록된 상품에 다시 요청하면 중복 등록하지 않고 정상 응답(200)을 반환한다 (멱등 동작).

#### 기능 요구사항

- 사용자는 상품에 좋아요를 등록할 수 있어야 한다.
- 하나의 사용자는 동일한 상품에 좋아요를 중복 등록할 수 없어야 한다.
- 좋아요 등록 요청은 멱등해야 한다. 이미 등록된 경우에도 오류 없이 성공 응답을 반환한다.

---

### 5.5 상품 좋아요 취소

#### 사용자 시나리오

사용자는 더 이상 관심이 없는 상품의 좋아요를 취소한다.

#### 기본 흐름

1. 사용자가 상품 상세 화면에서 좋아요 취소 버튼을 클릭한다.
2. 클라이언트는 `DELETE /api/v1/products/{productId}/likes`를 호출한다.
3. 시스템은 요청 헤더에서 사용자를 식별한다.
4. 시스템은 해당 사용자의 좋아요 등록 여부를 확인한다.
5. 좋아요가 등록된 경우 삭제하고 성공 응답을 반환한다.

#### 예외 흐름

- 로그인하지 않은 사용자가 요청하면 인증 오류(401)를 반환한다.
- 존재하지 않는 상품에 요청하면 404 응답을 반환한다.
- 좋아요가 등록되지 않은 상품에 취소 요청을 하면 오류 없이 정상 응답을 반환한다 (멱등 동작).

#### 기능 요구사항

- 사용자는 등록된 좋아요를 취소할 수 있어야 한다.
- 좋아요 취소 요청은 멱등해야 한다. 등록되지 않은 경우에도 오류 없이 성공 응답을 반환한다.

---

### 5.6 내 좋아요 상품 목록 조회

#### 사용자 시나리오

사용자는 이전에 좋아요를 등록한 상품 목록을 한 곳에서 모아 확인한다.

#### 기본 흐름

1. 사용자가 좋아요 목록 화면에 진입한다.
2. 클라이언트는 `GET /api/v1/users/{userId}/likes`를 호출한다.
3. 시스템은 요청 헤더에서 사용자를 식별하고, 본인의 좋아요 목록만 조회 가능한지 확인한다.
4. 시스템은 해당 사용자가 좋아요를 등록한 상품 목록을 반환한다.

#### 예외 흐름

- 로그인하지 않은 사용자가 요청하면 인증 오류(401)를 반환한다.
- 다른 사용자의 좋아요 목록에 접근하면 403 응답을 반환한다.
- 좋아요 등록 상품이 없으면 빈 목록을 반환한다.

#### 기능 요구사항

- 사용자는 자신이 좋아요를 등록한 상품 목록을 조회할 수 있어야 한다.
- 사용자는 타 사용자의 좋아요 목록에 접근할 수 없어야 한다.

---

### 5.7 주문 생성

#### 사용자 시나리오

사용자는 여러 상품을 선택한 후 한 번에 주문을 요청한다. 시스템은 재고를 확인하고 차감한 뒤 주문을 생성한다.

#### 기본 흐름

1. 사용자가 주문하기 버튼을 클릭한다.
2. 클라이언트는 `POST /api/v1/orders`를 호출한다. 요청 본문에 `[{ productId, quantity }]` 형태의 주문 항목을 포함한다.
3. 시스템은 요청 헤더에서 사용자를 식별한다.
4. 시스템은 각 상품의 존재 여부와 재고 수량을 확인한다.
5. 재고가 충분하면 각 상품의 재고를 차감한다.
6. 시스템은 주문 당시의 상품 정보(상품명, 가격)를 스냅샷으로 저장하여 주문을 생성한다.
7. 시스템은 생성된 주문 정보를 반환한다.

#### 예외 흐름

- 로그인하지 않은 사용자가 요청하면 인증 오류(401)를 반환한다.
- 주문 항목이 비어 있으면 400 응답을 반환한다.
- 존재하지 않는 상품 ID가 포함된 경우 400 응답을 반환한다.
- 재고가 부족한 상품이 하나라도 있으면 주문 전체가 생성되지 않고 400 응답을 반환한다.

#### 기능 요구사항

- 사용자는 여러 상품을 하나의 주문으로 묶어 요청할 수 있어야 한다.
- 주문 생성 시 모든 상품의 재고가 충분한지 확인해야 한다.
- 재고 확인 후 주문 수량만큼 재고를 차감해야 한다.
- 주문 항목에는 주문 당시의 상품명과 가격이 스냅샷으로 저장되어야 한다. 이후 상품 정보가 변경되어도 주문 당시 정보는 유지된다.
- 재고 부족 시 주문 전체를 실패 처리한다. 부분 성공은 허용하지 않는다.

#### 비기능 요구사항 및 정책

- 재고 차감은 동시 주문 환경에서 데이터 정합성을 보장해야 한다.
- 주문 생성 요청 본문 형식: `{ "items": [{ "productId": 1, "quantity": 2 }] }`

### 5.8 주문 목록 조회

#### 사용자 시나리오

사용자는 기간을 설정하여 자신의 주문 내역을 조회한다.

#### 기본 흐름

1. 사용자가 주문 내역 화면에 진입한다.
2. 클라이언트는 `GET /api/v1/orders?startAt={날짜}&endAt={날짜}`를 호출한다.
3. 시스템은 요청 헤더에서 사용자를 식별한다.
4. 시스템은 해당 사용자의 기간 내 주문 목록을 반환한다.

#### 예외 흐름

- 로그인하지 않은 사용자가 요청하면 인증 오류(401)를 반환한다.
- 조회 결과가 없으면 빈 목록을 반환한다.

#### 기능 요구사항

- 사용자는 `startAt`, `endAt` 날짜 파라미터를 이용하여 기간 내 주문 목록을 조회할 수 있어야 한다.
- 사용자는 자신의 주문 목록만 조회할 수 있어야 한다.

---

### 5.9 주문 상세 조회

#### 사용자 시나리오

사용자는 특정 주문의 상세 정보(주문 항목, 금액, 상태)를 확인한다.

#### 기본 흐름

1. 사용자가 주문 목록에서 특정 주문을 클릭한다.
2. 클라이언트는 `GET /api/v1/orders/{orderId}`를 호출한다.
3. 시스템은 요청 헤더에서 사용자를 식별한다.
4. 시스템은 해당 주문이 요청 사용자의 주문인지 확인한다.
5. 시스템은 주문 상세 정보를 반환한다.

#### 예외 흐름

- 로그인하지 않은 사용자가 요청하면 인증 오류(401)를 반환한다.
- 존재하지 않는 주문 ID로 요청 시 404 응답을 반환한다.
- 다른 사용자의 주문에 접근하면 403 응답을 반환한다.

#### 기능 요구사항

- 사용자는 자신의 주문 상세 정보를 조회할 수 있어야 한다.
- 응답에는 주문 항목별 상품명, 주문 당시 가격, 수량, 주문 총액, 주문 상태가 포함되어야 한다.
- 사용자는 타 사용자의 주문에 접근할 수 없어야 한다.

### 5.10 어드민 브랜드 관리

#### 기본 흐름

- 어드민은 브랜드 목록을 페이지 단위(`page`, `size`)로 조회할 수 있다.
- 어드민은 브랜드 ID로 브랜드 상세 정보를 조회할 수 있다.
- 어드민은 브랜드명을 입력하여 새 브랜드를 등록할 수 있다.
- 어드민은 브랜드명을 수정할 수 있다.
- 어드민은 브랜드를 삭제할 수 있다. 삭제 시 해당 브랜드에 속한 모든 상품도 함께 삭제된다.

#### 예외 흐름

- 존재하지 않는 브랜드 ID로 요청 시 404 응답을 반환한다.

#### 기능 요구사항

- 브랜드 삭제는 해당 브랜드의 상품을 모두 포함하여 삭제되어야 한다.
- 어드민은 `X-Loopers-Ldap` 헤더로 식별된다.

---

### 5.11 어드민 상품 관리

#### 기본 흐름

- 어드민은 상품 목록을 페이지 단위로 조회할 수 있다. `brandId`로 필터링할 수 있다.
- 어드민은 상품 ID로 상품 상세 정보를 조회할 수 있다.
- 어드민은 브랜드, 상품명, 가격, 재고 수량을 지정하여 상품을 등록할 수 있다.
- 어드민은 상품명, 가격, 재고 수량을 수정할 수 있다.
- 어드민은 상품을 삭제할 수 있다.

#### 예외 흐름

- 등록 시 존재하지 않는 브랜드 ID를 지정하면 400 응답을 반환한다.
- 존재하지 않는 상품 ID로 요청 시 404 응답을 반환한다.

#### 기능 요구사항

- 상품 등록 시 브랜드는 이미 등록된 브랜드여야 한다.
- 상품 수정 시 브랜드는 변경할 수 없다.

---

### 5.12 어드민 주문 조회

#### 기본 흐름

- 어드민은 전체 주문 목록을 페이지 단위로 조회할 수 있다.
- 어드민은 주문 ID로 특정 주문의 상세 정보를 조회할 수 있다.

#### 예외 흐름

- 존재하지 않는 주문 ID로 요청 시 404 응답을 반환한다.

#### 기능 요구사항

- 어드민은 소유자 제한 없이 모든 사용자의 주문에 접근할 수 있다.
- 주문 목록은 생성일 기준 내림차순으로 반환된다.

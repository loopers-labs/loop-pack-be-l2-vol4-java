# 설계 리뷰 — 상품 조회 읽기 최적화 (구조적 리스크 관점)

> 과제 "가정한 설계에 대해 점검하기"의 6관점 분석. 구현/설정값이 아니라 **구조적 리스크와 사고 관점**으로만 작성.

## 설계 요약
- **조회 API 목적**: 상품 목록(브랜드 필터 + 정렬) / 상품 상세
- **주요 조회 조건**: `brand_id` 등치 필터 + 정렬(LATEST/PRICE_ASC/LIKES_DESC) + 페이지네이션
- **테이블/데이터**: `product`(10만), `brand`(100), `likes`(SOT)
- **인덱스**: `(brand_id, like_count)`, `(like_count)`, `(brand_id, created_at)`, `(created_at)`, `(brand_id, price)`, `(price)`
- **비정규화**: `product.like_count` (좋아요 시 원자적 UPDATE 동기화)
- **캐시**: Redis Cache-Aside. 상세 `productDetail::{id}`(TTL 5분, 좋아요 시 evict), 목록 `productList::{brandId}:{sort}:{page}:{size}`(TTL 30초)

---

### 1. 이 설계가 성립하기 위해 반드시 참이어야 하는 전제

- **읽기가 쓰기를 압도한다.** like_count 인덱스 6개의 쓰기 비용을 감수하는 근거 전체가 이 전제 위에 있다. 쓰기 비중이 높아지면 인덱스가 부채가 된다.
- **조회 조건의 카디널리티 분포가 인덱스를 의미 있게 만든다.** brand_id가 거의 한 값에 몰리면 인덱스 선택도가 떨어진다.
- **like_count의 짧은 stale(목록 30초)이 비즈니스적으로 허용된다.** 좋아요 순위가 실시간이어야 한다면 목록 캐시 전략이 무너진다.
- **캐시는 보조 경로이지 필수 경로가 아니다.** Redis가 없어도 DB가 (느리더라도) 진실을 제공할 수 있어야 한다.
- **목록 조회의 키(조건 조합)가 폭발하지 않는다.** 조합이 무한히 늘면 캐시 적중률이 0에 수렴한다.

### 2. 트래픽이 10배 증가했을 때 가장 먼저 병목이 될 지점

- **목록 캐시 적중률 저하 → DB로의 낙수.** 조건 조합이 많은 목록은 키가 분산돼 적중률이 낮고, 트래픽이 커지면 캐시 미스가 그대로 DB 부하로 전파된다. (부하 테스트에서 캐시 없을 때 DB는 47 req/s가 한계였다.)
- 그 다음은 **인기 상품 like_count의 핫 로우 쓰기 경합**. 좋아요가 한 행에 몰리면 UPDATE 직렬화 + 인덱스 재정렬이 트랜잭션 점유 시간을 늘린다.

### 3. 캐시 적중률이 30% 이하로 떨어졌을 때 발생할 문제

- 미스의 70%가 DB로 직행한다. **캐시가 흡수하던 부하의 대부분이 DB로 옮겨가** 커넥션 점유 → 슬로우 쿼리 → 커넥션 고갈로 이어질 수 있다.
- 더 위험한 건 **"캐시가 있다는 가정 하에 DB가 과소 프로비저닝"** 되어 있을 때다. 적중률이 떨어지는 순간 DB가 설계 용량을 초과한다. 캐시의 이점이 클수록, 적중률 하락은 더 가파른 절벽이 된다.

### 4. 데이터 정합성이 깨질 수 있는 시나리오 (2가지+)

- **어드민 상품 수정 ↔ 좋아요 동시성**: 어드민이 product를 통째로 `save()`하는 사이 좋아요가 증가하면, 옛 like_count로 덮어써져 좋아요가 유실된다(lost update). like_count가 product에 비정규화돼 있어 발생한다.
- **목록 캐시의 stale**: 좋아요 순위가 바뀌어도 목록 캐시는 TTL(30초)까지 옛 순서를 보여준다. 의도된 허용이지만, "실시간 인기"가 요구되면 정합성 위반이다.
- **캐시 무효화 누락 경로**: 향후 like_count를 바꾸는 새 경로가 추가됐는데 상세 캐시 evict를 빠뜨리면, TTL이 만료될 때까지 낡은 값이 남는다.
- **(미래) Redis 카운터 도입 시**: 증분을 DB에 flush하기 전 장애가 나면 Redis와 DB가 어긋난다. SOT(`likes`)를 기준으로 재집계해야 보정된다.

### 5. 이 설계를 유지하면서 가장 나중까지 미룰 수 있는 개선

- **like_count의 CQRS 분리(요약 테이블/Redis 카운터/이벤트 MV)**. 핫 로우는 바이럴 스케일에서만 터지고, 분리는 오히려 좋아요순 정렬에 조인을 부활시켜 읽기를 해친다. 쓰기 병목이 실측될 때까지 미룬다.
- **목록의 ES 전환, 깊은 페이지네이션의 커서 전환** 도 트래픽/조건 다양성이 실제로 커지기 전까진 미룰 수 있다.

### 6. 반대로, 가장 먼저 손대야 할 위험 요소

- **캐시(Redis) 장애 시 가용성**. 부하 테스트에서 Redis 장애가 앱 hang으로 전파됨을 확인했고, 이는 트래픽과 무관하게 *지금 당장* 서비스 전체를 멈출 수 있는 위험이다. 1차로 빠른 실패(타임아웃)로 graceful degradation을 확보했고, 다음으로 서킷브레이커(회복 구간 포함)가 가장 먼저 손대야 할 영역이다.
- 그 다음은 **목록 캐시 적중률 모니터링** — 적중률이 곧 DB 생존의 선행 지표이기 때문이다.

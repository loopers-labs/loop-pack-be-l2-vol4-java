# Round 6 Quest Guide

## 1. 문서 목적

이 문서는 6주차 구현 과제인 PG 결제 연동과 Resilience 설계를 AI가 정확히 이해하고 수행할 수 있도록 정리한 요구사항 문서다.

AI는 이 문서를 기준으로 다음 작업을 수행한다.

- 외부 PG 시스템의 지연, 실패, 응답 유실 가능성을 고려한다.
- 결제 요청, 콜백, 상태 조회를 조합해 주문과 결제 상태를 안전하게 맞춘다.
- Timeout, Fallback, CircuitBreaker를 적용해 외부 장애가 내부 서비스 전체로 확산되지 않게 한다.
- 비동기 결제 특성상 발생할 수 있는 상태 불일치와 복구 시나리오를 점검한다.

## 2. 전체 목표

### 2.1 Implementation Quest

> 외부 시스템(PG)의 장애와 지연에 대응하는 Resilience 설계를 학습하고 적용한다.

`pg-simulator` 모듈을 활용해 비동기 결제 연동을 구현하고, 외부 시스템이 느리거나 실패하거나 응답을 잃어버리는 상황에서도 내부 시스템이 정상적으로 동작하도록 보호한다.

| 목표 | 설명 |
| --- | --- |
| PG 연동 | `commerce-api`에서 `pg-simulator`의 결제 요청, 결제 조회 API를 호출한다. |
| 비동기 결제 처리 | 결제 요청 성공과 최종 결제 성공을 분리해서 다룬다. |
| 상태 정합성 | PG 결제 상태와 내부 주문/결제 상태가 어긋나는 지점을 식별하고 복구한다. |
| 장애 대응 | Timeout, Fallback, CircuitBreaker로 외부 장애 확산을 막는다. |
| 복구 경로 | 콜백이 누락되거나 timeout이 발생해도 결제 상태 조회로 최종 상태를 반영한다. |

### 2.2 Must-Have

- Fallback
- Timeout
- CircuitBreaker

### 2.3 Nice-To-Have

- Retryer

## 3. 사전 설계 점검

### 3.1 외부 시스템 전제

PG 시스템은 항상 다음 문제가 발생할 수 있다고 가정한다.

- 응답이 지연될 수 있다.
- 요청이 실패할 수 있다.
- 같은 요청이 중복 실행될 수 있다.
- PG에서는 성공했지만 내부 시스템이 응답을 받지 못할 수 있다.
- 콜백이 늦게 오거나 오지 않을 수 있다.

### 3.2 핵심 점검 질문

구현 전에 다음 질문에 대한 답을 정리한다.

1. PG 호출은 내부 DB 트랜잭션 안에서 수행되는가, 밖에서 수행되는가?
2. PG 요청 timeout 이후 내부 결제 상태는 무엇으로 남길 것인가?
3. PG 요청은 실패했지만 실제 PG 결제가 진행된 경우 어떻게 확인할 것인가?
4. PG 콜백이 중복으로 도착하면 같은 결제 상태가 두 번 반영되지 않는가?
5. PG 콜백이 오지 않으면 어떤 주기 또는 API로 상태를 복구할 것인가?
6. CircuitBreaker가 열렸을 때 사용자는 어떤 응답을 받아야 하는가?
7. Retryer를 적용한다면 멱등성 키 또는 중복 방지 기준은 무엇인가?

### 3.3 AI Skill 작성 과제 요약

외부 시스템 연동 설계를 검토하는 Skill을 작성한다.
Skill은 설계를 대신 생성하는 목적이 아니라, 이미 작성된 설계의 리스크와 선택지를 드러내는 목적으로 사용한다.

Skill의 분석 흐름은 다음을 포함한다.

- 기능 흐름보다 외부 시스템의 불확실성을 먼저 분석한다.
- 내부 DB 트랜잭션과 외부 PG 호출의 경계를 검증한다.
- 호출 순서가 아니라 내부 상태와 PG 상태의 전이를 기준으로 정리한다.
- 중복 요청, 재시도, 콜백 중복에 대한 멱등성 여부를 확인한다.
- 장애 시나리오를 최소 3개 이상 만들고 데이터 정합성, 상태 불일치, 복구 가능성을 분석한다.
- 해결책을 정답처럼 단정하지 않고 선택지와 trade-off로 제시한다.

## 4. 구현 과제

### 4.1 결제 API 추가

주문에 대한 카드 결제 API를 추가한다.
요청에는 주문 ID와 결제 수단을 포함하고, 내부 시스템은 PG와 연동해 결제 처리를 시작한다.

```http
POST {{commerce-api}}/api/v1/payments
X-Loopers-LoginId:
X-Loopers-LoginPw:
Content-Type: application/json

{
  "orderId": "1351039135",
  "cardType": "SAMSUNG",
  "cardNo": "1234-5678-9814-1451"
}
```

### 4.2 PG 결제 요청

`pg-simulator`의 결제 요청 API를 호출한다.
PG는 비동기 결제를 제공하므로, 결제 요청 성공은 최종 결제 성공을 의미하지 않는다.

```http
POST {{pg-simulator}}/api/v1/payments
X-USER-ID:
Content-Type: application/json

{
  "orderId": "1351039135",
  "cardType": "SAMSUNG",
  "cardNo": "1234-5678-9814-1451",
  "amount": "5000",
  "callbackUrl": "http://localhost:8080/api/v1/payments/callback"
}
```

### 4.3 PG 결제 상태 조회

콜백이 지연되거나 누락될 수 있으므로 결제 상태 조회 API를 복구 경로로 사용한다.

```http
### 결제 정보 확인
GET {{pg-simulator}}/api/v1/payments/20250816:TR:9577c5
X-USER-ID:

### 주문에 엮인 결제 정보 조회
GET {{pg-simulator}}/api/v1/payments?orderId=1351039135
X-USER-ID:
```

### 4.4 Resilience 적용

PG 연동 구간에 다음 정책을 적용한다.

| 항목 | 구현 기준 |
| --- | --- |
| Timeout | PG 응답 지연이 내부 요청 스레드를 오래 점유하지 않도록 제한한다. |
| Fallback | PG 장애나 timeout 시 내부 시스템이 예측 가능한 응답을 반환하도록 한다. |
| CircuitBreaker | PG 실패가 누적되면 외부 호출을 차단해 장애 확산을 방지한다. |
| Retryer | 선택 과제다. 적용 시 중복 결제 방지와 멱등성 기준을 먼저 정한다. |

## 5. PG Simulator 동작 기준

`pg-simulator`는 별도 Spring Boot 애플리케이션으로 실행된다.
로컬 환경에서 PG 기반 카드 결제 흐름과 실패 시나리오를 재현하는 데 사용한다.

| 항목 | 값 |
| --- | --- |
| 결제 방식 | 비동기 결제 |
| 요청 성공 확률 | 60% |
| 요청 지연 | 100ms ~ 500ms |
| 처리 지연 | 1s ~ 5s |
| 최종 처리 결과 | 성공 70%, 한도 초과 20%, 잘못된 카드 10% |

비동기 결제에서는 요청 접수, 실제 처리, 콜백 전달, 내부 상태 반영이 서로 다른 시점에 발생한다.
따라서 내부 결제 상태는 기존 `PaymentStatus.REQUESTED/SUCCESS/FAILED/CANCELED/NOT_REQUIRED`와 목표 구조의 `PROCESSING` 필요 여부를 기준으로 명확히 표현해야 한다.

## 6. 분석 및 기록 기준

구현 결과와 함께 다음 내용을 기록한다.

- PG 호출 timeout 기준과 선택 이유
- CircuitBreaker 적용 위치와 열림/닫힘 기준
- Fallback 응답의 의미와 사용자에게 노출되는 상태
- PG 요청 실패, timeout, 콜백 누락, 콜백 중복 시나리오별 처리 방식
- 내부 주문 상태와 결제 상태가 변경되는 조건
- 결제 상태 조회 API를 활용한 복구 방식
- Retryer를 적용했다면 멱등성 보장 방식

## 7. 구현 체크리스트

### 7.1 PG 연동

- [ ] `commerce-api`에 주문 결제 API를 추가했다.
- [ ] RestTemplate 또는 FeignClient로 `pg-simulator`를 호출한다.
- [ ] PG 결제 요청 API를 연동했다.
- [ ] PG 결제 상태 조회 API를 연동했다.
- [ ] 콜백 방식과 상태 조회 방식을 함께 활용한다.

### 7.2 상태 정합성

- [ ] 결제 요청 성공과 최종 결제 성공을 분리해서 처리한다.
- [ ] PG timeout 이후에도 해당 결제 건의 최종 상태를 확인할 수 있다.
- [ ] 콜백이 오지 않아도 일정 주기 또는 수동 API로 상태를 복구할 수 있다.
- [ ] 중복 콜백 또는 중복 상태 조회 결과가 와도 결제 상태가 중복 반영되지 않는다.
- [ ] PG 결제 결과를 기반으로 주문 상태를 안전하게 변경한다.

### 7.3 Resilience

- [ ] PG 호출에 timeout을 설정했다.
- [ ] PG 장애 또는 timeout에 대한 fallback을 구현했다.
- [ ] CircuitBreaker를 적용해 장애 확산을 방지했다.
- [ ] CircuitBreaker가 열린 상태에서도 내부 시스템은 정상적으로 응답한다.
- [ ] Retryer를 적용했다면 중복 결제 방지 기준을 함께 구현했다.

### 7.4 검증

- [ ] PG 요청 성공, 요청 실패, timeout 시나리오를 검증했다.
- [ ] 최종 결제 성공, 한도 초과, 잘못된 카드 결과를 검증했다.
- [ ] 콜백 누락 후 상태 조회로 복구되는 흐름을 검증했다.
- [ ] 콜백 중복 또는 상태 조회 중복 호출의 멱등성을 검증했다.
- [ ] CircuitBreaker open 상태에서 fallback 응답을 검증했다.

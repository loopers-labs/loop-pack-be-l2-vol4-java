# Round 1 Quest Guide

## 1. 문서 목적

이 문서는 1주차 과제를 AI가 정확히 이해하고 수행할 수 있도록 정리한 요구사항 문서다.

AI는 이 문서를 기준으로 다음 작업을 수행한다.

- 프로젝트 개발 환경과 Agent 관련 설정을 정리한다.
- 회원 기능 요구사항을 구현한다.
- 단위 테스트, 통합 테스트, E2E 테스트를 작성하고 통과시킨다.
- 과제 진행 과정과 기술적 판단을 글로 정리한다.

## 2. 전체 목표

### 2.1 Implementation Quest

프로젝트에 자신의 브랜치를 기준으로 개발 환경을 세팅하고, Agent 관련 설정을 작성한다.

지정된 기능 요구사항을 구현하고, 필요한 테스트 케이스를 작성한 뒤 모든 테스트를 통과시키는 것을 목표로 한다.

### 2.2 Technical Writing Quest

이번 주에 학습한 내용과 과제 진행 과정을 돌아보고, 어떤 판단을 왜 했는지 글로 정리한다.

글은 단순 작업 기록이 아니라, 문제 상황, 선택지, 판단 근거, 구현 결과가 드러나야 한다.

### 3.2 CLAUDE.md 작성

프로젝트 루트에 `CLAUDE.md` 파일을 생성한다.

`CLAUDE.md`에는 최소한 다음 내용을 포함한다.

| 항목 | 내용 |
| --- | --- |
| 기술 스택 및 버전 | `gradle.properties`, `build.gradle.kts` 등을 참고해 프로젝트 기술 스택과 버전을 정리한다. |
| 모듈 구조 | 프로젝트의 주요 모듈과 역할을 정리한다. |
| 개발 규칙 | AI가 따라야 할 개발 원칙과 작업 흐름을 정리한다. |
| 주의사항 | AI가 절대 하지 말아야 할 행동과 우선순위를 정리한다. |

### 3.3 CLAUDE.md 생성 프롬프트 예시

```markdown
Claude Code
> 현재의 프로젝트를 분석해 CLAUDE.md 를 만들어줘.
  gradle.properties, build.gradle.kts 등을 참고해 현재 프로젝트의 주요 기술 스택 및 버전, 모듈 구조를 포함해줘.
```

## 4. AI 개발 규칙

### 4.1 진행 Workflow

- 방향성과 주요 의사결정은 개발자에게 제안하고, 최종 승인된 사항을 기반으로 작업한다.
- AI가 반복 작업을 하거나 요청하지 않은 기능 구현, 테스트 삭제를 진행하려 하면 개발자가 개입할 수 있어야 한다.
- AI는 임의 판단으로 요구사항을 확장하지 않는다.
- 설계 방향을 바꿔야 한다면 먼저 근거와 선택지를 제시한다.

### 4.2 TDD Workflow

모든 테스트는 3A 원칙을 따른다.

- Arrange: 테스트에 필요한 상태를 준비한다.
- Act: 검증할 동작을 실행한다.
- Assert: 결과를 명확히 검증한다.

TDD 흐름은 다음 순서를 따른다.

| 단계 | 목적 | 규칙 |
| --- | --- | --- |
| Red | 실패하는 테스트 작성 | 요구사항을 만족하는 테스트 케이스를 먼저 작성한다. |
| Green | 테스트 통과 코드 작성 | 테스트가 통과하는 최소 구현을 작성하고 오버엔지니어링을 피한다. |
| Refactor | 코드 품질 개선 | 불필요한 코드 제거, 객체지향적 구조 개선, unused import 제거를 수행한다. |

### 4.3 구현 주의사항

AI는 다음 행동을 하지 않는다.

- 실제로 동작하지 않는 코드를 작성하지 않는다.
- 불필요한 Mock 데이터에 의존해 기능을 구현하지 않는다.
- null-safety를 무시하지 않는다. Java에서는 필요한 경우 `Optional`을 활용한다.
- `println` 디버깅 코드를 남기지 않는다.
- 기존 코드 패턴을 확인하지 않은 채 새로운 구조를 만들지 않는다.

우선순위는 다음과 같다.

1. 실제 동작하는 해결책을 우선한다.
2. null-safety와 thread-safety를 고려한다.
3. 테스트 가능한 구조로 설계한다.
4. 기존 코드 패턴과 일관성을 유지한다.

## 7. 회원가입 요구사항

### 7.1 입력 정보

회원가입 요청에는 다음 정보가 필요하다.

| 필드 | 설명 |
| --- | --- |
| 로그인 ID | 사용자를 식별하는 ID |
| 비밀번호 | 로그인에 사용하는 비밀번호 |
| 이름 | 사용자 이름 |
| 생년월일 | 사용자 생년월일 |
| 이메일 | 사용자 이메일 |

### 7.2 기능 요구사항

- 이미 가입된 로그인 ID로는 가입할 수 없다.
- 로그인 ID, 이름, 이메일, 생년월일은 정해진 포맷에 맞아야 한다.
- 비밀번호는 암호화해서 저장한다.
- 회원가입 성공 시 생성된 유저 정보를 반환한다.

### 7.3 비밀번호 규칙

비밀번호는 다음 규칙을 만족해야 한다.

1. 8자 이상 16자 이하이다.
2. 영문 대문자, 영문 소문자, 숫자, 특수문자를 사용할 수 있다.
3. 생년월일은 비밀번호에 포함될 수 없다.

## 8. 내 정보 조회 요구사항

### 8.1 반환 정보

내 정보 조회 응답에는 다음 정보를 포함한다.

| 필드 | 설명 |
| --- | --- |
| 로그인 ID | 사용자 로그인 ID |
| 이름 | 마지막 글자가 마스킹된 이름 |
| 생년월일 | 사용자 생년월일 |
| 이메일 | 사용자 이메일 |

### 8.2 기능 요구사항

- 로그인 ID는 영문과 숫자만 허용한다.
- 이름은 마지막 글자를 `*`로 마스킹해 반환한다.
- 유효한 인증 정보가 제공되면 사용자 정보를 반환한다.

## 9. 비밀번호 수정 요구사항

### 9.1 입력 정보

비밀번호 수정 요청에는 다음 정보가 필요하다.

| 필드 | 설명 |
| --- | --- |
| 기존 비밀번호 | 현재 사용 중인 비밀번호 |
| 새 비밀번호 | 변경할 비밀번호 |

### 9.2 기능 요구사항

- 기존 비밀번호가 일치해야 비밀번호를 수정할 수 있다.
- 새 비밀번호는 비밀번호 규칙을 만족해야 한다.
- 현재 비밀번호와 같은 비밀번호는 새 비밀번호로 사용할 수 없다.
- 새 비밀번호에는 생년월일이 포함될 수 없다.

## 10. 테스트 요구사항

AI는 기능 구현 시 테스트를 함께 작성한다.

### 10.1 테스트 종류

| 테스트 종류 | 목적 |
| --- | --- |
| 단위 테스트 | 도메인 규칙과 예외 조건을 빠르게 검증한다. |
| 통합 테스트 | 서비스와 저장소가 함께 동작하는지 검증한다. |
| E2E 테스트 | 실제 API 요청과 응답이 요구사항을 만족하는지 검증한다. |

### 10.2 테스트 작성 규칙

- 모든 테스트는 Arrange, Act, Assert 구조로 작성한다.
- 정상 케이스와 실패 케이스를 모두 작성한다.
- API 요구사항은 가능하면 실제 HTTP 요청 기반 E2E 테스트로 검증한다.
- 이미 존재하는 테스트 패턴이 있으면 그 구조를 따른다.

## 12. Technical Writing Quest

### 12.1 작성 목표

이번 주에 학습한 내용과 구현 과정을 글로 정리한다.

글은 다음 질문에 답할 수 있어야 한다.

- 어떤 문제를 겪었는가?
- 어떤 선택지가 있었는가?
- 왜 그 방식을 선택했는가?
- 구현 후 어떤 장단점이 남았는가?

### 12.2 작성 기준

| 항목 | 기준 |
| --- | --- |
| 형식 | 블로그 |
| 길이 | 제한 없음 |
| 필수 요소 | 1줄 요약, 즉 TL;DR |
| 중심 내용 | 무엇을 했는지보다 왜 그렇게 판단했는지 |
| 예시 | 코드 비교, 흐름도, 리팩토링 전후 비교 등 |
| 톤 | 실력은 보이지만 자만하지 않고 고민이 읽히는 글 |

### 12.3 좋은 글의 특징

| 특징 | 예시 |
| --- | --- |
| 자신의 언어로 설명한다 | Stub과 Mock의 차이를 이번 테스트에서 어떻게 체감했는지 설명한다. |
| 판단 흐름이 드러난다 | 처음에는 도메인을 나누지 않았지만 테스트가 어려워져 분리했다는 과정을 설명한다. |
| 정보 나열보다 인사이트가 있다 | 테스트는 작성했지만 구조가 만족스럽지 않았고 다음에는 어떻게 바꿀지 설명한다. |

### 12.4 피해야 할 스타일

| 스타일 | 이유 |
| --- | --- |
| 많이 부족했고 반성합니다 | 회고가 아니라 일기처럼 보인다. |
| 개념 정의만 나열한다 | 내 판단이 아니라 요약문처럼 보인다. |
| 테스트가 진리다 | 너무 단정적이거나 오만해 보일 수 있다. |

### 12.5 추천 주제

- Claude와 협업하며 바뀐 개발자의 역할
- 테스트 가능한 구조를 만들기 위해 수행한 리팩토링
- Mock, Stub, Fake 중 실제 활용 경험과 나만의 구분 기준
- TDD 방식으로 접근하거나 테스트를 작성하며 어려웠던 점

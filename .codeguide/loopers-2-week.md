# Round 2 Quest Guide

## 1. 문서 목적

이 문서는 2주차 설계 과제를 AI가 정확히 이해하고 수행할 수 있도록 정리한 요구사항 문서다.

AI는 이 문서를 기준으로 다음 작업을 수행한다.

- 요구사항 분석용 Skill을 작성하거나 개선한다.
- 이커머스 도메인 설계 문서를 작성한다.
- 설계 문서에 기능 요구사항, 시퀀스 다이어그램, 클래스 설계, ERD를 포함한다.
- PR 제출에 필요한 리뷰 포인트를 정리한다.
- 기술 글쓰기 과제를 작성할 수 있도록 설계 의사결정과 고민 지점을 정리한다.

## 2. 전체 목표

### 2.1 Design Quest

상품, 브랜드, 좋아요, 주문, 결제, 재고, 외부 시스템 연동을 포함한 이커머스 도메인 설계를 완료한다.

설계 문서는 다음 주부터 개발 가능한 수준이어야 한다.

### 2.2 Technical Writing Quest

이번 주 설계 과정에서 어떤 판단을 했고 왜 그렇게 설계했는지 글로 정리한다.

글은 단순 요약이 아니라 설계 선택, 트레이드오프, 리스크, 보완 방향이 드러나야 한다.

## 3. 설계 범위

### 3.1 포함 도메인

| 도메인 | 포함 기능 |
| --- | --- |
| 상품 | 상품 목록 조회, 상품 상세 조회 |

| 외부 시스템 | 결제 시스템, 데이터 플랫폼 등 외부 연동 |

## 4. 요구사항 분석 Skill 작성

### 4.1 Skill 목적

요구사항 분석 Skill은 코드를 작성하기 전에 요구사항을 명확히 하고, 개발자와의 질문/답변을 통해 애매한 정책과 책임 경계를 드러내기 위한 도구다.

정리가 끝나면 다음 산출물을 작성할 수 있어야 한다.

- 요구사항 명세
- 시퀀스 다이어그램
- 클래스 다이어그램
- ERD

### 4.2 Skill 작성 위치

예시 경로:

```text
~/.claude/skills/requirements-analysis/SKILL.md
```

프로젝트나 사용 도구에 따라 실제 경로는 달라질 수 있다.

### 4.3 Skill 필수 흐름

요구사항 분석 Skill은 다음 흐름을 따라야 한다.

| 순서 | 단계 | 설명 |
| --- | --- | --- |
| 1 | 문제 상황 재해석 | 요구사항을 그대로 옮기지 말고, 어떤 문제를 해결하려는지 사용자/비즈니스/시스템 관점으로 재해석한다. |
| 2 | 애매한 요구사항 노출 | 확정되지 않은 정책, 예외 처리, 책임 경계, 확장 가능성을 명시한다. |
| 3 | 질문 제시 | 개발자가 답하기 쉬운 형태로 우선순위가 있는 질문을 제시한다. |
| 4 | 개념 모델 정의 | 액터, 핵심 도메인, 보조 도메인, 외부 시스템을 먼저 정의한다. |
| 5 | 다이어그램 작성 | 이유, 다이어그램, 해석 순서로 Mermaid 다이어그램을 작성한다. |
| 6 | 다이어그램 해석 | 책임 분리, 트랜잭션 경계, 변경 영향 지점을 설명한다. |
| 7 | 설계 리스크 언급 | 트랜잭션 비대화, 도메인 결합, 정책 변경 영향 등 리스크와 선택지를 제시한다. |

### 4.4 질문 유형

Skill은 최소한 다음 질문 유형을 포함해야 한다.

| 질문 유형 | 확인할 내용 |
| --- | --- |
| 정책 질문 | 기준 시점, 성공/실패 조건, 예외 처리 규칙 |
| 경계 질문 | 어떤 책임을 어느 도메인 또는 서비스가 담당하는지 |
| 확장 질문 | 향후 정책 변경 가능성과 설계 영향 |

### 4.5 다이어그램 사용 기준

| 다이어그램 | 검증할 내용 |
| --- | --- |
| 시퀀스 다이어그램 | 책임 분리, 호출 순서, 트랜잭션 경계 |
| 클래스 다이어그램 | 도메인 책임, 의존 방향, 응집도 |
| ERD | 영속성 구조, 관계의 주인, 정규화 여부, 제약 조건 |

### 4.6 Skill 사용 프롬프트 예시

```markdown
Claude Code
> /requirements-analysis 를 통해 아래 요구사항을 분석합니다.
  목표: 이커머스 주문/결제 흐름 설계
  요구사항
  - 상품 목록과 상세를 조회한다.
  - 상품 좋아요를 등록/취소한다.
  - 주문 생성 시 재고를 차감한다.
  - 결제 시스템과 외부 데이터 플랫폼을 연동한다.
  제안사항
  - 애매한 정책과 책임 경계를 먼저 질문해줘.
  - 합의된 내용을 바탕으로 시퀀스 다이어그램, 클래스 다이어그램, ERD를 작성해줘.
```

## 7. 도메인별 설계 체크포인트

### 7.1 상품과 브랜드

- 상품 목록에는 어떤 정보를 보여주는가?
- 상품 상세에는 어떤 추가 정보가 필요한가?
- 상품은 브랜드와 어떤 관계를 가지는가?
- 품절, 판매 중지 등 상품 상태를 어떻게 표현하는가?
- 노출하지 않을 상품 상태가 있다면 목록과 상세에 동일하게 적용되는가?

### 7.2 좋아요

- 같은 사용자가 같은 상품에 여러 번 좋아요를 요청하면 어떻게 처리하는가?
- 좋아요 취소는 멱등하게 동작하는가?
- 좋아요 수는 실시간 집계인가, 카운터인가?
- 좋아요 이력과 카운터가 어긋났을 때 복구 기준은 무엇인가?
- 판매 중지 또는 품절 상품에 대한 좋아요 등록/취소 정책은 무엇인가?

### 7.3 주문

- 주문 생성 시 어떤 검증을 먼저 수행하는가?
- 주문 항목은 상품 정보를 참조만 하는가, 주문 시점의 상품명/가격을 스냅샷으로 저장하는가?
- 주문 상태는 어떤 값들을 가지는가?
- 주문 생성과 결제 요청은 같은 트랜잭션인가, 분리된 흐름인가?

### 7.4 재고

- 주문 생성 시 재고를 언제 차감하는가?
- 동시 주문에서 재고가 음수가 되지 않게 어떻게 보장하는가?
- 재고 차감 실패는 주문 실패와 어떻게 연결되는가?
- 결제 실패 또는 취소 시 재고를 어떻게 복구하는가?

### 7.5 결제

- 결제 요청은 동기인가, 비동기인가?
- 외부 결제 시스템 호출은 어떤 경계에서 수행하는가?
- 결제 성공, 실패, 취소, 타임아웃 상태를 어떻게 표현하는가?
- 중복 결제 요청을 어떻게 막는가?
- 외부 결제 응답이 늦게 도착하면 어떻게 처리하는가?
- 내부 주문 상태와 외부 결제 상태가 어긋날 때 어떤 보상 처리를 하는가?

### 7.6 외부 시스템 연동

- 주문 성공 이벤트는 어떤 시점에 외부 데이터 플랫폼으로 전송하는가?
- 외부 전송 실패가 주문 성공을 실패로 바꾸는가?
- 외부 전송 실패 시 재시도 기준은 무엇인가?
- Outbox 패턴을 사용할 경우, 주문 상태 변경과 이벤트 저장은 같은 트랜잭션인가?

## 8. PR 제출 요구사항

### 8.1 제출 방식

- GitHub PR로 제출한다.
- PR에는 `.docs/design`의 4개 설계 문서를 포함한다.

### 8.2 PR 제목

```text
[2주차] 설계 문서 제출 - 홍길동
```

### 8.3 PR 본문에 포함할 내용

- 설계 중 고민한 지점
- 트랜잭션 경계 판단 근거
- 외부 시스템 실패 처리 방향
- 데이터 정합성을 위해 둔 제약
- 향후 확장 가능성을 고려한 부분

## 9. 설계 체크리스트

- [ ] 상품, 브랜드, 좋아요, 주문 도메인이 모두 포함되어 있는가?
- [ ] 기능 요구사항이 유저 중심으로 정리되어 있는가?
- [ ] 제외 도메인이 명확히 드러나는가?
- [ ] 시퀀스 다이어그램에서 책임 객체가 드러나는가?
- [ ] 시퀀스 다이어그램에서 트랜잭션 경계가 드러나는가?
- [ ] 클래스 구조가 도메인 책임과 의존 방향을 표현하는가?
- [ ] ERD가 데이터 정합성과 관계를 고려해 구성되어 있는가?
- [ ] 주문/결제 실패 시 보상 처리나 상태 관리 방향이 설명되어 있는가?
- [ ] PR 본문에 리뷰 포인트가 포함되어 있는가?

## 10. Technical Writing Quest

### 10.1 제출 방식

글쓰기 과제는 다음 두 방식 중 하나를 선택한다.

| 방식 | 설명 |
| --- | --- |
| GitHub Issue | 제공된 4개 포맷 중 하나를 선택해 작성한다. |
| Blog | 이번 주차 설계와 학습 내용을 자유롭게 작성한다. |

### 10.2 GitHub Issue 포맷

Issue 작성 시 다음 포맷 중 하나를 선택한다.

| 포맷 | 목적 |
| --- | --- |
| Design Doc | 설계 의사결정 중심 |
| Retrospective | 과제 회고와 트러블슈팅 |
| Challenge Story | 도전과 해결 과정을 압축한 서사 |
| Benchmark Report | 두 선택지의 직접 비교와 측정 |

제목 형식:

```text
[포맷] 키워드 (N주차 · K팀 · 이름)
```

### 10.3 Issue 작성 링크

- Java: https://github.com/loopers-labs/loop-pack-be-l2-vol4-java/issues/new/choose
- Kotlin: https://github.com/loopers-labs/loop-pack-be-l2-vol4-kotlin/issues/new/choose

### 10.4 Blog 작성 기준

| 항목 | 기준 |
| --- | --- |
| 형식 | 블로그 |
| 길이 | 제한 없음 |
| 필수 요소 | 1줄 요약, 즉 TL;DR |
| 중심 내용 | 무엇을 했는지보다 왜 그렇게 판단했는지 |
| 예시 | 코드 비교, 흐름도, 리팩토링 전후 비교 등 |
| 톤 | 실력은 보이지만 자만하지 않고 고민이 읽히는 글 |

### 10.5 추천 주제

- 좋아요 기능의 멱등성을 어떻게 보장할 수 있을까?
- 주문 생성 시 재고 차감을 어떻게 연결했는가?
- 결제 흐름에서 동기 처리와 비동기 처리 중 무엇을 선택했는가?

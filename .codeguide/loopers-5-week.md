# Round 5 Quest Guide

## 1. 문서 목적

이 문서는 5주차 구현 과제를 AI가 정확히 이해하고 수행할 수 있도록 정리한 요구사항 문서다.

AI는 이 문서를 기준으로 다음 작업을 수행한다.

- 조회 API의 병목 가능성을 분석한다.
- 상품 목록 조회, 좋아요 수 정렬, 상품 상세 조회의 성능을 개선한다.
- DB 인덱스, 비정규화, Redis 캐시를 활용해 읽기 최적화 구조를 실습한다.
- 개선 전후 성능 차이와 선택한 구조의 trade-off를 기록한다.

## 2. 전체 목표

### 2.1 Implementation Quest

> 실제 트래픽에서 자주 발생하는 조회 병목 문제를 구조적으로 개선한다.

좋아요 수 기반 정렬, 브랜드 필터링, 인기 상품 조회는 데이터가 많아질수록 성능 저하가 발생하기 쉽다.
이번 주차의 목표는 인덱스, 비정규화, 캐시를 활용해 조회 성능을 개선하고, 선택한 방식의 전제와 한계를 설명할 수 있게 만드는 것이다.

| 목표 | 설명 |
| --- | --- |
| DB Index | 브랜드 필터, 좋아요 순 정렬 등 조회 조건에 맞는 인덱스를 설계하고 `EXPLAIN`으로 확인한다. |
| 비정규화 | 좋아요 수 정렬을 위해 `like_count` 같은 읽기 최적화 필드를 도입하거나 대안을 선택한다. |
| Redis | 상품 목록과 상품 상세 조회에 캐시를 적용한다. |
| 캐시 전략 | TTL, 캐시 키, 무효화 전략 중 하나 이상을 설계하고 적용한다. |

### 2.2 Must-Have

- DB Index
- 비정규화
- Redis
- 캐시

### 2.3 Nice-To-Have

- Materialized View

## 3. 사전 설계 점검

### 3.1 점검 목적

구현 전에 읽기 최적화 설계가 어떤 전제에서 성립하는지 확인한다.
AI는 제공된 구조와 설계를 기준으로 리스크, 병목 지점, trade-off를 분석하는 데 활용한다.

점검할 관점은 다음과 같다.

- 조회 API의 목적과 주요 조회 조건
- 사용 테이블과 데이터 분포
- 적용한 인덱스의 의도
- 캐시 적용 위치와 캐시 키 전략
- TTL 또는 무효화 전략
- 트래픽 증가 시 가장 먼저 드러날 병목
- 정합성이 깨질 수 있는 시나리오

### 3.2 AI 리뷰 프롬프트 예시

```markdown
너는 대규모 트래픽 환경에서 백엔드 시스템 설계를 리뷰하는 시니어 아키텍트다.
아래는 내가 설계한 구조 및 구현 코드에 대한 내용이다.

[설계 설명]
- 조회 API 목적:
- 주요 조회 조건:
- 사용한 테이블/데이터:
- 해당 테이블의 인덱스:
- 캐시 적용 여부 및 위치:
- 캐시 키 전략:
- 캐시 TTL 가정:

위 내용을 기준으로 다음 관점에서 분석해줘.

1. 이 설계가 성립하기 위해 반드시 참이어야 하는 전제 조건은 무엇인가?
2. 트래픽이 10배 증가했을 때 가장 먼저 병목이 될 지점은 어디인가?
3. 캐시 적중률이 30% 이하로 떨어졌을 때 발생할 문제는?
4. 데이터 정합성이 깨질 수 있는 시나리오를 2가지 이상 제시해줘.
5. 이 설계를 유지하면서 가장 나중까지 미룰 수 있는 개선은 무엇인가?
6. 반대로, 가장 먼저 손대야 할 위험 요소는 무엇인가?

주의:
- 구현 코드나 설정 값은 제안하지 마라.
- 구조적 리스크와 사고 관점에서만 답변하라.
```

## 4. 구현 과제

아래 세 가지 성능 개선을 수행한다.
가능하면 모두 수행하고, 선택 이유와 AS-IS / TO-BE 비교를 블로그 또는 작업 기록에 남긴다.

### 4.1 상품 목록 조회 성능 개선

상품 목록 API에서 브랜드 필터와 좋아요 순 정렬을 처리하고, 대량 데이터에서 조회 성능을 개선한다.

요구사항:

- 상품 데이터를 10만 개 이상 준비한다.
- 각 컬럼의 값은 다양하게 분포하도록 준비한다.
- 브랜드 필터와 좋아요 순 정렬 기능을 구현한다.
- `EXPLAIN` 분석을 통해 인덱스 적용 전후 실행 계획을 비교한다.
- 성능 개선 전후 비교 결과를 포함한다.

### 4.2 좋아요 수 정렬 구조 개선

좋아요 수 기반 정렬이 조인과 집계에 의존하지 않도록 읽기 최적화 구조를 선택한다.

요구사항:

- 비정규화(`like_count`) 또는 Materialized View 중 하나를 선택한다.
- 선택한 구조로 좋아요 수 정렬 성능을 개선한다.
- 좋아요 등록/취소 시 좋아요 수 동기화 처리가 누락되어 있다면 함께 구현한다.
- 선택하지 않은 대안의 trade-off도 간단히 기록한다.

### 4.3 캐시 적용

자주 조회되는 상품 목록 API와 상품 상세 API에 Redis 캐시를 적용한다.

요구사항:

- 상품 상세 API에 Redis 캐시를 적용한다.
- 상품 목록 API에 Redis 캐시를 적용한다.
- TTL 설정, 캐시 키 설계, 무효화 전략 중 하나 이상을 포함한다.
- 캐시 미스 상황에서도 서비스가 정상 동작하도록 처리한다.

## 5. 분석 및 기록 기준

성능 개선 작업은 구현 결과만 남기지 않고, 다음 내용을 함께 정리한다.

- 어떤 조회 조건을 최적화했는가
- AS-IS 실행 계획과 병목 지점은 무엇이었는가
- TO-BE 구조에서 어떤 비용이 줄었는가
- 인덱스, 비정규화, 캐시 중 어떤 선택을 했고 왜 선택했는가
- 데이터 정합성, 캐시 무효화, 쓰기 비용 증가 같은 trade-off는 무엇인가
- 개선 전후 성능 비교 결과는 무엇인가

## 6. 구현 체크리스트

### 6.1 Index

- [ ] 상품 목록 API에서 `brandId` 기반 검색을 처리했다.
- [ ] 상품 목록 API에서 좋아요 순 정렬을 처리했다.
- [ ] 조회 필터와 정렬 조건별 유즈케이스를 분석했다.
- [ ] 인덱스를 적용하고 적용 전후 성능을 비교했다.
- [ ] `EXPLAIN` 결과를 기준으로 개선 효과를 설명할 수 있다.

### 6.2 Structure

- [ ] 상품 목록 조회 시 좋아요 수를 조회할 수 있도록 구조를 개선했다.
- [ ] 상품 상세 조회 시 좋아요 수를 조회할 수 있도록 구조를 개선했다.
- [ ] 좋아요 순 정렬이 가능하도록 구조를 개선했다.
- [ ] 좋아요 등록 시 상품 좋아요 수가 정상적으로 동기화된다.
- [ ] 좋아요 해제 시 상품 좋아요 수가 정상적으로 동기화된다.

### 6.3 Cache

- [ ] 상품 상세 API에 Redis 캐시를 적용했다.
- [ ] 상품 목록 API에 Redis 캐시를 적용했다.
- [ ] TTL 또는 무효화 전략을 적용했다.
- [ ] 캐시 키 전략을 설명할 수 있다.
- [ ] 캐시 미스 상황에서도 서비스가 정상 동작한다.

# Round 4 Quest Guide

## 1. 문서 목적

이 문서는 4주차 구현 과제를 AI가 정확히 이해하고 수행할 수 있도록 정리한 요구사항 문서다.

AI는 이 문서를 기준으로 다음 작업을 수행한다.

- 쿠폰 도메인을 구현한다.
- 주문 시 재고, 쿠폰, 주문 데이터의 정합성을 DB 트랜잭션으로 보장한다.
- 동시성 이슈가 발생할 수 있는 기능에 적절한 락을 적용한다.
- 동시성 테스트를 작성한다.
- 트랜잭션 분석 Skill을 작성하고, 구현된 기능을 지속적으로 점검하며 개선한다.

## 2. 전체 목표

### 2.1 Implementation Quest

> 주문 시 재고, 주문, 쿠폰의 정합성을 트랜잭션으로 보장하고 동시성 이슈를 제어한다.

| 목표 | 설명 |
| --- | --- |
| DB 트랜잭션 | 주문 처리 과정의 재고 차감, 쿠폰 사용, 주문 저장을 원자적으로 처리한다. |
| Lock | Lost Update가 발생하지 않도록 낙관적 락 또는 비관적 락을 적용한다. |
| 동시성 테스트 | 동일 상품과 동일 쿠폰에 대한 동시 요청을 검증한다. |
| 쿠폰 개념 | 정액, 정률 쿠폰과 발급, 사용, 만료 상태를 구현한다. |

### 2.2 Must-Have

- DB 트랜잭션
- Lock
- 동시성 테스트
- 쿠폰 개념

## 3. 쿠폰 도메인

### 3.1 기본 요구사항

- 주문 시 사용자가 소유한 쿠폰을 적용해 할인받을 수 있어야 한다.
- 쿠폰은 정액 쿠폰과 정률 쿠폰으로 구분한다.
- 발급된 쿠폰은 재사용할 수 없다.
- 존재하지 않거나 사용할 수 없는 쿠폰으로 주문하면 주문 전체가 실패해야 한다.

### 3.2 쿠폰 종류

| 타입 | 설명 |
| --- | --- |
| `FIXED` | 정액 할인. 지정된 금액만큼 할인한다. |
| `RATE` | 정률 할인. 지정된 비율만큼 할인한다. |

### 3.3 발급 쿠폰 상태

| 상태 | 설명 |
| --- | --- |
| `AVAILABLE` | 사용 가능한 쿠폰 |
| `USED` | 사용 완료된 쿠폰 |
| `EXPIRED` | 만료된 쿠폰 |

### 3.4 주문 적용 규칙

- 쿠폰은 주문 1건당 1장만 적용할 수 있다.
- 존재하지 않는 쿠폰, 이미 사용된 쿠폰, 만료된 쿠폰, 다른 사용자가 소유한 쿠폰으로 주문하면 실패한다.
- 주문 성공 시 사용한 쿠폰은 즉시 `USED` 상태로 변경되며 재사용할 수 없다.
- 주문 정보 스냅샷에는 쿠폰 적용 전 금액, 할인 금액, 최종 결제 금액이 모두 포함되어야 한다.

## 4. API 요구사항

### 4.1 대고객 쿠폰 API

| Method | URI | `user_required` | 설명 |
| --- | --- | --- | --- |
| `POST` | `/api/v1/coupons/{couponId}/issue` | O | 쿠폰 발급 요청 |
| `GET` | `/api/v1/users/me/coupons` | O | 내 쿠폰 목록 조회 |

쿠폰 목록 조회 시 `AVAILABLE`, `USED`, `EXPIRED` 상태를 함께 반환한다.

### 4.2 쿠폰 Admin API

| Method | URI | `ldap_required` | 설명 |
| --- | --- | --- | --- |
| `GET` | `/api-admin/v1/coupons?page=0&size=20` | O | 쿠폰 템플릿 목록 조회 |
| `GET` | `/api-admin/v1/coupons/{couponId}` | O | 쿠폰 템플릿 상세 조회 |
| `POST` | `/api-admin/v1/coupons` | O | 쿠폰 템플릿 등록. 정액(`FIXED`) 또는 정률(`RATE`) 타입을 지정한다. |
| `PUT` | `/api-admin/v1/coupons/{couponId}` | O | 쿠폰 템플릿 수정 |
| `DELETE` | `/api-admin/v1/coupons/{couponId}` | O | 쿠폰 템플릿 삭제 |
| `GET` | `/api-admin/v1/coupons/{couponId}/issues?page=0&size=20` | O | 특정 쿠폰의 발급 내역 조회 |

쿠폰 템플릿 등록 요청 예시:

```jsonc
{
  "name": "신규가입 10% 할인",
  "type": "RATE",             // FIXED | RATE
  "value": 10,                // 정률: 퍼센트(%), 정액: 할인 금액(원)
  "minOrderAmount": 10000,    // 최소 주문 금액 조건 (선택)
  "expiredAt": "2026-12-31T23:59:59"
}
```

### 4.3 주문 API 변경사항

쿠폰 적용 주문 요청 예시:

```jsonc
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ],
  "couponId": 42 // 미적용 시 생략 가능 (nullable)
}
```

## 5. 트랜잭션과 동시성 제어

### 5.1 정합성 보장

- 주문 API에 트랜잭션을 적용한다.
- 재고, 쿠폰, 주문 도메인의 정합성을 보장한다.
- 재고 차감 또는 쿠폰 사용 중 하나라도 실패하면 주문 관련 변경 사항을 모두 롤백한다.
- 주요 트랜잭션 경계는 Application Layer 또는 `OrderFacade`와 같은 애플리케이션 조립 계층에 둔다.

### 5.2 Lock 적용

- Lost Update가 발생하지 않도록 낙관적 락 또는 비관적 락을 적용한다.
- 각 도메인의 특성에 맞는 락 전략을 선택한다.
- 동일 쿠폰의 중복 사용과 동일 상품 재고의 잘못된 차감을 방지한다.

### 5.3 주문 처리 흐름

```text
1. 주문 요청
2. 주문을 위한 처리 (순서 무관)
   - 쿠폰 유효성 검증 및 사용 처리  // 동시성 이슈 위험 구간
   - 상품 재고 확인 및 차감         // 동시성 이슈 위험 구간
3. 주문 엔티티 생성 및 저장
```

## 6. 구현 보강

- 모든 API가 요구사항을 기반으로 동작해야 한다.
- 미비한 구현을 모두 완성한다.
- 동시성 이슈가 발생할 수 있는 기능의 테스트가 모두 성공해야 한다.

## 7. 구현 체크리스트

### 7.1 Coupon

- [ ] 쿠폰은 사용자가 소유하며, 이미 사용된 쿠폰은 다시 사용할 수 없다.
- [ ] 쿠폰 종류는 정액과 정률로 구분하며, 각 적용 로직을 구현한다.
- [ ] 발급된 쿠폰은 최대 한 번만 사용할 수 있다.

### 7.2 Order

- [ ] 주문 전체 흐름의 원자성을 보장한다.
- [ ] 사용할 수 없거나 존재하지 않는 쿠폰으로 주문하면 실패한다.
- [ ] 재고가 존재하지 않거나 부족하면 주문이 실패한다.
- [ ] 쿠폰, 재고 처리 중 하나라도 실패하면 모든 변경 사항을 롤백한다.
- [ ] 주문 성공 시 모든 변경 사항이 정상적으로 반영된다.

### 7.3 동시성 테스트

- [ ] 동일 상품에 대해 여러 사용자가 좋아요 또는 좋아요 취소를 요청해도 좋아요 수가 정상적으로 반영된다.
- [ ] 동일 쿠폰으로 여러 기기에서 동시에 주문해도 쿠폰은 한 번만 사용된다.
- [ ] 동일 상품에 대해 여러 주문이 동시에 요청되어도 재고가 정상적으로 차감된다.

## 8. 과제 집중 사항

모든 기능의 동작을 개발한 후 동시성, 멱등성, 일관성, 느린 조회, 동시 주문 등 실제 서비스에서 발생할 수 있는 문제를 해결한다.

낙관적 락(Optimistic Lock) 또는 비관적 락(Pessimistic Lock) 중 각 도메인의 특성에 맞는 전략을 선택해 적용한다. Application Layer 또는 `OrderFacade`에서의 트랜잭션 경계 설계가 핵심이다.

## 9. 트랜잭션 분석 Skill 작성

트랜잭션 분석 Skill을 작성하고, 구현된 기능을 지속적으로 점검하며 개선한다.

작성 예시: `~/.claude/skills/analyze-query/SKILL.md`

````markdown
---
name: analyze-query
description:
  대상이 되는 코드 범위를 탐색하고, Spring @Transactional, JPA, QueryDSL 기반의 코드에 대해 트랜잭션 범위, 영속성 컨텍스트, 쿼리 실행 시점 관점에서 분석한다.

  특히 다음을 중점적으로 점검한다.
  - 트랜잭션이 불필요하게 크게 잡혀 있지는 않은지
  - 조회/쓰기 로직이 하나의 트랜잭션에 혼합되어 있지는 않은지
  - JPA의 지연 로딩, flush 타이밍, 변경 감지로 인해
    의도치 않은 쿼리 또는 락이 발생할 가능성은 없는지

  단순한 정답 제시가 아니라, 현재 구조의 의도와 trade-off를 드러내고 개선 가능 지점을 선택적으로 판단할 수 있도록 돕는다.
---

### Analysis Scope

이 스킬은 아래 대상에 대해 분석한다.

- `@Transactional`이 선언된 클래스 또는 메서드
- Service, Facade, Application Layer 코드
- JPA Entity, Repository, QueryDSL 사용 코드
- 하나의 유즈케이스(요청 흐름) 단위

> 컨트롤러 -> 서비스 -> 레포지토리 전체 흐름을 기준으로 분석하며 특정 메서드만 떼어내어 판단하지 않는다.

### Analysis Checklist

#### 1. Transaction Boundary 분석

다음을 순서대로 확인한다.

- 트랜잭션 시작 지점은 어디인가?
  - Service, Facade, 그 외 계층
- 트랜잭션이 실제로 필요한 작업은 무엇인가?
  - 상태 변경(쓰기)
  - 단순 조회
- 트랜잭션 내부에서 수행되는 작업은 무엇인가?
  - 외부 API 호출
  - 복잡한 조회(QueryDSL)
  - 반복문 기반 처리

출력 예시:

```text
- 현재 트랜잭션 범위:
  OrderFacade.placeOrder()
    -> 유저 검증
    -> 상품 조회
    -> 주문 생성
    -> 결제 요청
    -> 재고 차감

- 트랜잭션이 필요한 핵심 작업:
  - 주문 생성
  - 재고 차감
```

#### 2. 불필요하게 큰 트랜잭션 식별

아래 패턴이 존재하는지 점검한다.

- Controller에서 `@Transactional`을 사용한다.
- 읽기 전용 로직이 쓰기 트랜잭션에 포함된다.
- 외부 시스템 호출이 트랜잭션 내부에 포함된다.
- 트랜잭션 내부에서 대량 조회 또는 복잡한 QueryDSL을 실행한다.
- 상태 변경 이후에도 트랜잭션이 길게 유지된다.

문제 후보 예시:

- 결제 API 호출이 트랜잭션 내부에 포함되어 있다.
- 주문 생성 이후 추천 상품 조회 로직까지 동일 트랜잭션에 포함되어 있다.

#### 3. JPA / 영속성 컨텍스트 관점 분석

다음을 중심으로 분석한다.

- Entity 변경이 언제 flush 되는가?
- 조회용 Entity가 변경 감지 대상인가?
- 지연 로딩으로 인해 트랜잭션 후반에 쿼리가 발생할 가능성이 있는가?
- `@Transactional(readOnly = true)`를 적용했는가?

체크리스트 예시:

```text
- 단순 조회인데 Entity 반환 후 변경 가능성이 존재하는가?
- DTO Projection 대신 Entity 조회를 사용하는가?
- QueryDSL 조회 결과가 영속성 컨텍스트에 포함되는가?
```

#### 4. Improvement Proposal

개선안은 강제하지 않고 선택지로 제시한다.

- 트랜잭션 분리
  - 조회와 쓰기 분리
  - Facade에서 orchestration, Service에서 최소 트랜잭션 유지
- `@Transactional(readOnly = true)` 적용
- DTO Projection(읽기 전용 모델) 도입
- 외부 호출 또는 이벤트 발행을 트랜잭션 외부로 이동
- Application Service와 Domain Service의 책임 재조정

개선안 예시:

```text
[개선안 1]
- 주문 생성과 결제 요청을 분리
- 주문 생성까지만 트랜잭션 유지
- 결제 요청은 트랜잭션 종료 후 수행

[고려 사항]
- 결제 실패 시 주문 상태 관리 필요
- 보상 트랜잭션 또는 상태 전이 설계 필요
```
````

# 실험 보고
## 재고 차감 동시성 제어: 비관적 락(A) vs 원자적 조건부 UPDATE(B)

- **Date**: 2026-06-12
- **관련 문서**: [`ADR-001-stock-lock-strategy.md`](../../decision/ADR-001-stock-lock-strategy.md) — "성능 우위가 유의미한지는 아직 측정하지 않았다"의 그 측정

---

## 측정 결과 (요약)

측정 전 질문은 "성능이 같은가"가 아니었다. 원자적 조건부 UPDATE(B)가 더 빠르다는 건 이미 알고 있었고, 잰 건 "그 차이가 유의미한가"였다.

결과: 차이는 실재하고 B가 우위다. 모든 동시성(10/50/100/200)에서 총시간 중앙값 기준 1.38~1.74× 빠르다. 다만 절대값으로는 요청당 약 0.6~1.3ms다.

- **임계 동시성은 없다.** 차이는 풀 대기가 전혀 없는 동시성 10에서도 이미 두 분포가 완전히 갈릴 만큼 존재한다(A 최소 31.3ms > B 최대 29.1ms). 차이의 원인이 커넥션 풀 포화가 아니라 직렬화 임계구역의 길이라는 뜻이다.
- 락 동작 방식이 같다는 건 확인됐다. 두 전략 모두 정합성 게이트(과판매 0) 통과, 데드락·에러 0, 풀 포화도(대기 스레드 최대치)도 동시성마다 완전히 동일(40/90/190). 직렬화되는 방식은 같고, 한 번 직렬화될 때 잡고 있는 시간이 다르다.
- 절대값 관점: A의 추가 비용은 경합 하 요청당 약 0.6~1.3ms다(총시간 차 ÷ 요청 수). A는 임계구역 안에 SELECT…FOR UPDATE 왕복과 엔티티 객체 생성, dirty checking flush가 들어가고(왕복 2회), B는 UPDATE 한 문장(왕복 1회)이다. 그 차이가 직렬화 큐를 따라 N배로 누적된 것이 총시간 차다.

> 그래서 선택의 질문은 "성능이 같으니 설계 가치만 보면 된다"가 아니라, "재고 규칙을 도메인에 두는 값으로 worst-case 처리량 약 30~40%(요청당 ~1ms)를 감수할 것인가"로 좁혀진다. 실제 주문 플로우에서는 두 전략이 공유하는 비용(주문 INSERT, 쿠폰, 커밋 flush)이 커서 상대 격차는 이보다 줄어든다(아래 한계 참조).

- p50/p95/p99는 측정 7 run의 요청 레이턴시를 풀링한 값이다. 레이턴시에는 행 락 대기와 커넥션 풀 대기가 포함된다(두 전략 동일 조건).
- 통계적 유의성: 동시성별 총시간 7:7 분포에 Mann-Whitney U 정확검정을 적용하면 전 구간 p < 0.01이다. 동시성 10/50은 두 분포가 아예 겹치지 않고, 100/200도 각 1개 run만 상호 범위에 걸친다.
- 풀 대기 최대(동시성 − 풀 10)는 두 전략이 동시성마다 정확히 같다. 풀 점유 패턴에는 전략 간 차이가 없다는 뜻이다. 비관락이 락 대기 동안 커넥션을 점유하는 만큼, 원자 UPDATE도 행 락 대기 동안 커넥션을 점유한다(InnoDB UPDATE가 같은 X락에서 블로킹되므로). 차이는 점유 '여부'가 아니라 점유 '시간'에서 났다.

## 정합성

재고 30, 동시 50 차감 → 기대: 성공 30, OUT_OF_STOCK 20, 최종 재고 0, 과판매 0, 기타 예외 0.

| 전략 | 결과 |
|---|---|
| A 비관락 | PASS · 성공 30/30, 품절 20, 최종재고 0, 기타예외 0 |
| B 원자 UPDATE | PASS · 성공 30/30, 품절 20, 최종재고 0, 기타예외 0 |

추가로 측정 run 56개(동시성 × 전략 × 7) 전부에서 성공 == 동시성, 최종 재고 == 0을 단언으로 검증했다.

## 방법

### 전략 구현 (유일한 변수 = 락 전략)

공통 인터페이스 `StockDecrementStrategy` 뒤에 두 구현을 뒀다. 호출 1건 = 독립 트랜잭션 1개.

- **A (비관락)**: 기존 운영 경로 그대로 `StockService.decrease`에 위임. `SELECT … FOR UPDATE`(`@Lock(PESSIMISTIC_WRITE)`) → 도메인 `StockModel.decrease()` → dirty checking flush. 임계구역 내 DB 왕복 2회.
- **B (원자 UPDATE)**: `UPDATE stock SET quantity = quantity - :n WHERE product_id = :id AND quantity >= :n`, 영향 0행이면 OUT_OF_STOCK. 임계구역 내 DB 왕복 1회.
    - `amount <= 0` 가드는 쿼리 앞단 자바 코드로 A와 동일하게 적용했다. 원자 UPDATE 한 문장은 이 입력 검증을 담지 못한다(`quantity >= :n`은 n ≤ 0이면 항상 참이라 차감이 그냥 성공해버린다). 전략 B를 채택하더라도 가드는 어차피 애플리케이션 코드에 남아야 한다.
    - 영향 0행은 "행 없음"과 "재고 부족"을 구분하지 못한다. A는 PRODUCT_NOT_FOUND와 OUT_OF_STOCK을 구분하지만, B 단독으로는 추가 조회 없이 불가능하다. 에러 의미에서 B가 잃는 부분이다.

## 해석( 왜 갈렸고, 왜 임계점이 없는가)

단일 행 고경합에서 두 전략 모두 행 X락으로 완전 직렬화된다(게이트와 풀 포화도가 그 증거다). 직렬화된 시스템의 총시간은 근사적으로 `N × 임계구역 길이`다. 그래서 세 가지가 따라온다.

1. **임계구역 길이가 곧 처리량이다.** A의 임계구역은 락 획득 SELECT 왕복 + 엔티티 객체 생성 + 도메인 로직 + flush UPDATE + 커밋이고, B의 임계구역은 조건부 UPDATE + 커밋이다. 측정된 차이(요청당 0.6~1.3ms)는 정확히 "왕복 1회 + JPA 경로 비용" 크기다.
2. **임계 동시성이 없는 이유.** 풀이 비포화인 동시성 10에서도 직렬화는 이미 일어나므로, 임계구역 차이는 처음부터 누적된다. 동시성을 올리면 절대 차이는 커지지만(직렬화 큐가 길어진다) 상대 배율은 오히려 약간 줄어든다(1.74× → 1.38×). 스레드 스케줄링·풀 대기처럼 두 전략이 공유하는 비용이 함께 커지며 격차를 희석하기 때문이다.
3. **풀은 원인이 아니다.** "비관락은 락 대기 동안 커넥션을 점유하니 풀에서 차이가 드러날 것"이라는 예상은 기각됐다. 원자 UPDATE도 행 락에 블로킹되는 동안 커넥션을 똑같이 점유한다. 풀 대기 최대치가 전략 간 완전히 동일(40/90/190)했다. 풀 대기는 두 전략에 똑같이 걸려서, 차이의 원인이 아니라 양쪽에 같은 배경으로 깔릴 뿐이다.

## 한계 (이 수치를 일반화하면 안 되는 범위)

- 재고 차감 단독 트랜잭션의 순수 비용이다. 실제 주문 플로우(`OrderFacade.placeOrder`)는 유저 조회·주문 생성·쿠폰 적용이 한 트랜잭션에 있고 재고 차감이 마지막이라, 임계구역에 커밋 flush(주문 INSERT 등)가 양쪽 공통으로 더해진다. 그래서 상대 격차(1.4~1.7×)는 실제 플로우에서 이보다 줄어든다. 절대 차이(요청당 ~1ms)가 더 이식 가능한 수치다.
- 로컬 Docker 단일 노드 측정이라 네트워크 RTT가 사실상 0에 가깝다. 애플리케이션↔DB가 분리된 운영망에선 왕복 1회 차이가 RTT만큼 더 벌어진다(A에 불리한 방향으로 격차 확대).
- 단일 행 worst-case 고경합만 측정했다. 경합이 낮으면 직렬화 자체가 드물어 차이는 무의미해진다.
- 동시성 200 > Tomcat 기본 워커 200 같은 서버 계층 효과는 측정 범위 밖이다(webEnvironment NONE).
- 낙관락(@Version)의 thundering herd는 이 실험 범위가 아니다(별도 실험).

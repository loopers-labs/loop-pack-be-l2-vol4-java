## Q1. 어디서부터 구현할 것인가 — 구현 순서의 판단 기준

**[질문]**
"쿠폰 템플릿부터" vs "주문 뼈대부터" 두 후보가 있다. 구현 순서를 정할 때 가장 중요한 기준 하나를 꼽으라면 무엇인가? 그 기준으로 봤을 때 두 도메인은 서로 어떤 의존 관계를 갖는가? (주문을 먼저 짜려면 무엇이 이미 있어야 하고, 쿠폰은 무엇에 의존하는가?)

**[답변]**
> 쿠폰이 있어야지 주문도 온전히 동작하는지 확인할 수 있고, 그 대신 주문에 대한 직관적인 이해가 부족한 상태로 쿠폰을 만들 것 같아

**[후속 질문: couponId가 NULLABLE이라는 점 — 쿠폰은 주문의 "전제"인가 "옵션"인가]**
quest의 주문 요청 스펙에서 `couponId`는 NULLABLE(미적용 시 생략 가능)이다. 그렇다면 쿠폰 없이도 주문이 성립하는가? "쿠폰이 있어야 주문이 온전히 동작한다"는 말과, "couponId 없이도 주문은 동작한다(쿠폰은 할인 옵션일 뿐)"는 말 중 어느 쪽이 더 정확한가? 의존 방향(누가 누구를 필요로 하는가)을 다시 생각해보면?

**[답변]**
> B이고, 쿠폰 -> 주문

**[후속 질문: B와 "쿠폰→주문"의 모순? — 쿠폰의 두 얼굴(발급 vs 사용)]**
B를 골랐다 = "주문은 쿠폰 없이도 동작한다"(주문은 쿠폰에 의존하지 않음). 그런데 화살표는 "쿠폰 → 주문"(쿠폰이 주문에 의존?)이라고 했다. 이 둘은 충돌하는 것처럼 보인다. 충돌이 아니라면, 쿠폰의 어떤 부분은 주문이 없어도 되고(독립), 어떤 부분은 주문이 있어야만 동작하는가(의존)? 쿠폰을 기능별로 쪼개보면?

**[답변]**
> 1, 2번 발급/조회는 독립, 사용은 주문 의존

**[정리: 쿠폰의 두 얼굴]**
- 쿠폰 템플릿 / 발급 / 조회 → 독립 (주문 없이 만들고 테스트 가능)
- 쿠폰 사용(검증·할인·USED) → 주문 흐름 안에서만 동작 = 주문에 의존
- 따라서 "쿠폰 → 주문" 화살표는 정확히 **쿠폰 '사용'** 부분을 가리킨다. B(주문은 쿠폰 없이도 동작)와 모순 아님.

## Q2. 그래서 첫 단위는? — 의존 그래프 기반 빌드 순서

**[질문]**
독립적인 조각(leaf)을 먼저 만들고, 의존(통합)은 마지막에 붙이는 게 점진적 구현의 정석이다. 지금 의존 그래프는:
- 독립 leaf: ① 쿠폰 발급/조회, ② 주문 뼈대(product에만 의존, product는 이미 존재)
- 통합: ③ 주문 + 쿠폰 사용

처음 worry("주문 직관 없이 쿠폰부터 만든다")는 이 분해로 해소되는가? 둘 다 독립 leaf라면 어느 것을 1번으로 두겠는가? 그 근거는?

**[답변]**
> 쿠폰 발급 먼저 하자

**[결정]**
- 첫 단위 = ① 쿠폰 발급/조회 (독립 leaf, 기존 product CRUD 패턴과 유사해 워밍업)
- 구현은 계층별로 끊어서 진행: domain → repository/infra → service/facade → controller/dto

## Q3. (구현 중) 쿠폰 도메인 모델링

**[멘토 설명: 두 엔티티 분리]**
- `Coupon`(템플릿/정책): name, type(FIXED/RATE), value, minOrderAmount, expiredAt
- `UserCoupon`(발급된 쿠폰): userId, couponId(템플릿 참조), status
- 발급 = 템플릿을 보고 UserCoupon 인스턴스를 찍어내는 것

**[가정 — 사용자 확인 필요]**
- EXPIRED 상태는 **저장하지 않고** 조회 시점에 템플릿 `expiredAt`과 현재시각을 비교해 계산한다.
  → 저장 status는 AVAILABLE / USED만 가짐. (대안: 배치로 EXPIRED flip)

**[사용자 지적: 기존 코드 일관성 / 엔티티 null 검증]**
- 첫 구현이 `Product`(단순 스타일)만 따라해 `User`(리치 스타일)와 불일치.
- 이 코드베이스 검증 레이어링: VO record 생성자(의미 검증) + `@Column(nullable=false)`(DB 안전망). DTO엔 jakarta.validation 미사용. → **엔티티 생성자 null 재검증은 중복**.
- 정정 방향: coupon 엔티티를 User 리치 스타일(`@Getter`+`@NoArgsConstructor(PROTECTED)`+`@Column` 명시, 엔티티 null 체크 제거)로 맞춘다.

## Q4. 쿠폰의 비즈니스 규칙은 어디에 둘까 — VO 도입 여부

**[질문]**
User 패턴은 검증·불변식을 VO에 몰아넣는다(Email/Name/Birth...). 쿠폰의 진짜 비즈니스 규칙은 `value > 0`, `RATE일 때 value ≤ 100`이다. 이 규칙(type+value 묶음)을 별도 VO(예: `Discount`)로 뺄까, 아니면 엔티티에 둘까? VO로 빼면 나중에 할인 계산(`apply(orderAmount)`)도 거기 자연스럽게 들어간다. trade-off는?

**[답변]**
> A

**[결정 + 컨벤션 확인]**
- (A) `Discount` VO 도입. type+value를 묶어 검증(value>0, RATE≤100). 나중에 할인계산도 여기 추가.
- VO 생성 위치 = **Facade** (user 패턴: `new LoginId(...)`를 Facade에서 만들어 Service로 전달).
- 엔티티는 VO를 받아 **언래핑한 원시값을 @Column에 저장**.

## Q5. VO 아닌 필드의 null 검증 — 기존 코드 확인 후 하이브리드 결정

**[질문]** VO에서 null 검증하면, VO 아닌 필드(name·expiredAt)는 어디서 막나?

**[기존 코드 근거]**
- `User` = 전부 VO → 엔티티 null 체크 0개.
- `ProductModel` = VO 없음 → **엔티티 생성자에서 `CoreException(BAD_REQUEST)` 검증** (깔끔한 400, 추가 클래스 0).
- 즉 "VO 아닌 필드는 엔티티 생성자에서 검증"이 기존 정답. → 처음에 null 체크 전부 지운 건 과했음(User 전용 규칙을 오적용).

**[결정: 하이브리드]**
- `type·value`(불변식 있음) → `Discount` VO
- `name·expiredAt·minOrderAmount`(단순 존재/범위) → 엔티티 생성자에서 Product 스타일 검증
- `UserCoupon`의 userId·couponId도 동일하게 엔티티 생성자 검증

**[Discount VO 용도]** ① 생성 시 불변식 검증 ② `getDiscount()`로 재구성 ③ 3단계에서 `apply(주문금액)` 할인 계산 캡슐화(FIXED/RATE 분기). 지금은 ③ 미구현.

## Q6. ★중요★ 코드 재확인 — 첫 분석이 pull 이전 옛 코드였음

**[사용자 지적]** "pull 안 받고 코드 이해시켰다, 전부 다시 봐. VO 아닌 null들 어떻게 풀고 있나?"

**[재분석 결과]** 현재 워킹트리(volume-4)는 첫 분석과 완전히 다름:
- `ProductModel` → `Product` + `Stock`/`Money`/`Quantity` **@Embeddable VO**로 리팩터
- `order`(Order·OrderItem·OrderLine·OrderService·OrderFacade·Controller), `like`, `brand` **이미 구현 완료**
- 주문 트랜잭션 경계: `OrderFacade.place`(@Transactional) + `OrderService.place`(@Transactional) — **이미 Facade 경계 완성**

**[null 전략 = "출처 기반 선택적 검증"]**
- 사용자 입력 + 잘못될 수 있는 값 → VO 또는 entity `validate()` → 400
- 내부 보장값(인증 userId, 검증된 path id) → 명시적 체크 생략, `@Column(nullable=false)`만 (증거: `OrderItem` 생성자 검증 0개)
- 장점: 보일러플레이트 절감, VO 보장. 단점: 내부값이 null로 뚫리면 flush 시 500(늦은 실패). → 상류 보장에 베팅.
- 대안(전부 방어적 null 체크)은 코드베이스가 이미 거부(OrderItem) → 일관성 위해 비권장.

**[coupon 새 컨벤션 적용 — 최종]**
- `Discount` = **@Embeddable VO 클래스**(record 아님), `@Embedded`로 Coupon에 저장. type+value 불변식.
- `Coupon`: name·expiredAt → private static `validate()`(admin 입력 필수, 400). minOrderAmount → **Money VO**(nullable).
- `UserCoupon`: userId·couponId **null 체크 제거** → @Column에 위임(OrderItem과 동일).
- 컴파일 통과 확인.

## Q7. @Transactional을 Facade에 붙이는 기준 (week4 핵심)

**[질문]** CouponFacade.issue엔 @Transactional이 없고 OrderFacade.place엔 있다. 왜?

**[정리]**
- `@Transactional` = "다 같이 성공 or 다 같이 롤백"의 **작업 단위 경계**를 긋는 것.
- **판단 규칙**: Facade가 부르는 서비스가
  - **1개** → 경계는 그 서비스에 있음 → Facade @Transactional **불필요** (ProductFacade, CouponFacade.issue)
  - **여러 개를 원자적으로 묶음** → 경계는 **Facade**여야 함 → @Transactional **필수** (OrderFacade.place)
- **왜 필수인가**: 쿠폰 USED와 주문 저장이 각 서비스에 흩어져 있는데, Facade에 경계가 없으면 `couponService.use`가 혼자 커밋된 뒤 주문이 실패해도 쿠폰은 USED로 남음 → 원자성 깨짐.
- **어떻게 묶이나**: Facade @Transactional 안에서 부르는 서비스들의 @Transactional은 기본 전파 `REQUIRED`로 **기존 트랜잭션에 합류** → 물리적으로 트랜잭션 1개.
- → week4의 "OrderFacade 트랜잭션 경계 설계가 핵심"의 정체.

**핵심 한 줄**: 한 덩어리로 묶일 작업이 여러 서비스에 흩어져 있으면, 묶는 자리(Facade)에 경계를 긋는다.

## Q8. Application Service vs Facade vs Domain Service — 뭐가 다른가

**[질문]** 셋이 그냥 이름만 다른 건가?

**[정리]**
- **사실 2개**: "Application Service" == "Facade" (이 프로젝트가 application 레이어를 Facade로 명명). Domain Service는 별개.
- **Domain Service** (`domain/`, `OrderService`/`CouponService`): 비즈니스 **규칙**(HOW). 엔티티·레포지토리 다룸. 재고 차감, 쿠폰 검증 등. = 요리사.
- **Facade = App Service** (`application/`, `OrderFacade`): 유스케이스 **조율**(WHAT/순서) + 트랜잭션 경계 + DTO(Info) 변환. 규칙 자체는 모름. = 홀 매니저.
- **왜 둘 다(얇은 Facade라도)**: ① 여러 도메인 조율(Order+Coupon) ② 트랜잭션 경계 소유 ③ 도메인↔컨트롤러 차단(Info 변환). 조율이 커질 자리를 일관되게 마련.

**핵심 한 줄**: Domain Service=규칙 실행(요리사), Facade(=App Service)=유스케이스 조율(홀 매니저).

## Q9. 왜 CouponRepository / UserCouponRepository 둘로 나눴나

**[질문]** coupon 리포지토리를 user용과 따로 둔 이유? 파일이 여러 개인데.

**[정리]**
- DDD: **애그리거트 루트 하나당 리포지토리 하나.**
- 판단 테스트: 둘이 한 단위로 저장/조회되고 하나가 다른 하나 없이 존재 의미 없으면 → 한 애그리거트.
- `Coupon`(템플릿, admin 관리) vs `UserCoupon`(유저 소유, userId로 조회, status 독립 변경) → **서로 id 참조만 하는 독립 루트 둘** → 리포지토리 둘.
- **코드베이스 대조**:
  - `OrderItem` = Order의 **자식** → 자기 port 없음. OrderRepositoryImpl이 OrderJpa+OrderItemJpa 둘 다 써서 한 번에 저장.
  - `Like` = user·product를 id 참조하는 **독립 루트** → 자기 리포지토리. → `UserCoupon`은 Like와 동급.
- 리포지토리 1개 = 3파일(port + JpaRepository + Impl). 의존성 역전(도메인이 JPA에 직접 안 묶임).

**핵심 한 줄**: 애그리거트 루트 수 = 리포지토리 수. OrderItem(자식)은 흡수, UserCoupon(독립 루트)은 분리.

## Q10. 발급 응답 DTO 분리 (IssuedCouponInfo)

- 문제: 발급 Facade가 rich `UserCouponInfo`를 만들려고 템플릿을 재조회(double read). 발급 응답엔 템플릿 상세 불필요.
- 원인: 발급/조회가 rich DTO 하나를 공유 → 발급도 템플릿 필드를 채워야 함.
- 해결: `IssuedCouponInfo`(id·couponId·status, UserCoupon만으로 조립) 분리. 발급은 이걸 반환 → 재조회 0.
- `CouponService.getCoupon` 제거(이 변경으로 죽은 코드. order 통합 때 재추가 가능).
- 보충 개념: UserCoupon은 Coupon을 안 가짐, couponId(Long)만 가짐 → 상세 필요 시 findById로 복원(id 참조).

## Q13. 락 전략 — 충돌했을 때 "누가 이기는 게" 비즈니스 정답인가

**[질문]**
subject 문서의 관점: 락 선택은 "충돌이 많냐 적냐"가 아니라 **"충돌 시 누가 성공하고 누가 실패/대기해야 하는가"**로 판단한다.

공유자원 3개 각각에 대해, 두 요청이 동시에 충돌한 상황을 상상하고 답하라:

1. **product.stock** — 재고 5개에 A·B가 동시 주문. 비즈니스적으로 올바른 결과는? (둘 다 성공해야 하나 / 한 명만 성공하고 나머지는 에러여도 되나 / 줄 세워서 차례로 처리해야 하나)
2. **product.like_count** — 서로 다른 유저 10명이 같은 상품에 동시 좋아요. 몇 명이 성공해야 하나?
3. **user_coupon.status** — 같은 유저가 두 기기에서 같은 쿠폰으로 동시 주문. 몇 건이 성공해야 하나?

각 답에서 낙관적 락(`@Version`, 충돌자는 예외로 실패) / 비관적 락(`FOR UPDATE`, 충돌자는 대기 후 차례로 진행) 중 무엇이 자연스럽게 따라 나오는가?

**[답변]**
> JPA에서는 두 가지 종류의 락 전략을 제공합니다. 단순히 "충돌이 많다/적다" 로 구분하기보다는 "누가 성공하고, 나머지는 실패하게 둘 것인가?" 또는 "모두가 기다리게 할 것인가?" 라는 설계 관점으로 접근하는 것이 더 실용적입니다. 왜 이 관점이 더 실용적이라고 하는거야? 근거는?

**[멘토 보충: "빈도" 기준의 두 가지 약점]**
- 반례: 좋아요는 충돌이 적다 → 빈도 기준이면 낙관적 락. 그런데 낙관적 락 = 충돌자는 예외로 실패. 드물게 충돌한 1명의 좋아요는 증발 → 빈도 예측이 맞았는데도 요구사항("좋아요 수 정상 반영") 위반.
- 약점 ①: 빈도는 설계 시점에 알 수 없는 **예측값**(트래픽·인기도·세일에 따라 변함). 반면 "진 쪽이 실패해도 되는가"는 요구사항에서 지금 확정적으로 읽힘 — quest 체크리스트가 그 답("쿠폰은 단 한 번만", "좋아요 수 정상 반영").
- 약점 ②: 빈도를 맞혀도 정합성은 별개. 빈도 기준은 "어느 쪽이 성능이 나은가"만 답함. 충돌이 1년에 한 번이어도 그 한 번이 틀리면 버그. 쿠폰 따닥은 충돌 극히 드물지만 "1명만 성공"이 정확히 원하는 동작 → 낙관적이 정답. 두 사례 모두 빈도는 결론에 기여 없음.
- 정리: 빈도 기준 = 성능 질문(예측 기반). 승패 기준 = 정합성 질문(요구사항 기반, 설계 시점 확정, 테스트로 증명 가능). 정합성 먼저, 성능은 둘 다 정답일 때의 타이브레이커.
- 덧: 낙관적 락 + 재시도로 "결국 모두 성공"을 만들 수도 있으나, 그 결정도 "진 쪽이 결국 성공해야 하는가?"라는 승패 질문에 먼저 답해야 나옴.

**[답변]**
> 그러면 왜 "누가 성공하고, 나머지는 실패하게 둘 것인가?" 또는 "모두가 기다리게 할 것인가?"의 관점을 따라 락 선택 관점이 달라져?

**[멘토 보충: 각 락에는 "패자 처리 방식"이 기계적으로 내장되어 있다]**
- 두 락은 "충돌을 막는 정도가 다른 도구"가 아니라, 충돌 시 진 쪽 처리 정책이 하드코딩된 서로 다른 도구.
- 낙관적 락: 읽을 때 안 막음. `@Version` 컬럼을 UPDATE 조건에 포함(`WHERE id=? AND version=3`) → 먼저 커밋한 쪽이 version을 올리면 뒤쪽 UPDATE는 0건 → `OptimisticLockingFailureException`. **패자는 그 자리에서 실패.** = "1명 성공, 나머지 탈락" 내장.
- 비관적 락: 읽는 순간 `FOR UPDATE`로 행 잠금. 두 번째 도착자는 예외가 아니라 **대기** → 승자 커밋 후 갱신된 최신 값을 읽고 이어서 진행. = "줄 세워 전원 차례 처리" 내장.
- 따라서 비즈니스 답("패자 실패 OK" vs "전원 처리")이 나오는 순간 동작이 어긋나는 도구가 탈락. 좋아요에 낙관적을 고르면 요구("전부 반영")를 지키기 위해 재시도 루프로 줄 세우기를 손수 재구현해야 함 — 비관적이 이미 내장한 동작을.
- 단, 이 질문은 유일한 정답을 찍는 게 아니라 **어긋나는 쪽을 탈락시키는 필터**. 두 락 모두 정합성이 지켜지는 자원도 있고, 그때 성능·대기 비용 같은 2차 기준이 등판 → Q13에서 자원별로 판정.

**[답변]**
> stock은 하나 실패하면 예외해야하니 낙관적락, likeCount은 동시 좋아요 ok userCoupon은 비관적락?

**[후속 질문: 재고 충분한데 실패하는 B / 어차피 실패할 운명인데 기다리는 쿠폰]**
1. stock 반례 — 재고 100개, A·B가 각각 1개씩 동시 주문. 낙관적 락이면 B의 UPDATE는 version 불일치로 예외. 재고가 99개나 남는데 B가 실패하는 게 비즈니스 정답인가? "재고 부족이라 실패"(검증 실패)와 "동시에 썼다고 실패"(락 충돌 실패)는 같은 실패인가?
2. userCoupon — 비관적이면 두 번째 기기는 대기했다가 USED를 보고 실패. 결과는 맞다. 그런데 어차피 실패할 운명의 요청을 기다리게 해서 얻는 게 있나? 낙관적이면 두 번째는 어떻게 되나?
3. likeCount — "동시 좋아요 ok"(전원 성공)면 그래서 어떤 락이 따라 나오나?

**[답변]**
> 아 모르겠어 알려줘

**[멘토 정리]**
판단 도구: **충돌 시 진 쪽이 어떻게 되어야 하는가** → 락의 내장 동작과 일치하는 쪽 선택.

| 자원 | 충돌 시 비즈니스 정답 | 선택 |
|---|---|---|
| product.stock | 재고가 남는 한 전원 차례 처리 | **비관적 락** |
| product.like_count | 전원 반영 | **비관적 락** |
| user_coupon.status | 1건만 성공, 나머지 실패가 옳음 | **낙관적 락** |

- **stock → 비관적**: 재고 100에 A·B 동시 주문이면 둘 다 성공해야 함. 낙관적이면 B는 재고 99개 남았는데 version 충돌로 실패 — 팔 물건을 못 팔고, 동시 주문 몰릴수록(가장 잘 팔리는 순간) 실패 폭증. 비관적이면 B는 잠깐 대기 후 최신 재고로 정상 차감, 재고 0일 때만 "재고 부족" 비즈니스 에러. **"재고 부족 실패"(검증 실패, 정당)와 "동시에 샀다는 이유의 실패"(락 충돌, 기술적)는 다른 실패** — 사용자가 처음 답한 "하나 실패하면 예외"는 전자였음.
- **like_count → 비관적**: 전원 성공 요구 → 패자를 탈락시키는 낙관적은 부적합(고르면 재시도 루프로 줄 세우기 재구현). 덧: 실무에선 `SET like_count = like_count + 1` 원자적 UPDATE로 read-modify-write 자체를 제거하기도 함. 이번 과제는 락 학습 목적이라 비관적으로.
- **coupon → 낙관적**: 중복 사용 시도는 실패가 정답 → 낙관적의 패자 처리(즉시 예외)가 요구와 정확히 일치. 비관적도 결과는 맞지만(대기→USED 확인→검증 실패) 죽을 요청에 대기 비용만 추가. version 컬럼 하나로 끝나 비용 최소. 충돌 예외는 Facade 트랜잭션 전체 롤백 → 그 주문의 재고 차감도 함께 복구(Q11 경계 덕).
- **코드 연결**: ① ProductRepository에 `@Lock(PESSIMISTIC_WRITE)` 주문/좋아요 전용 조회 메서드(일반 조회는 락 없음) ② UserCoupon에 `@Version` + `use()` 전이 메서드 ③ 낙관적 충돌 예외 → 주문 실패 응답 매핑.

**[멘토 정리]**
- 정확히 데드락(순환 대기): A는 B가 쥔 상품2를, B는 A가 쥔 상품1을 기다림.
- **해법 1 = 예방: 전역 락 순서 고정(정답).** 모든 트랜잭션이 같은 순서로 락을 잡으면 순환이 물리적으로 불가능. 구현 = 주문 items의 productId **오름차순 정렬 후** 그 순서로 FOR UPDATE 조회(`WHERE id IN (...) ORDER BY id FOR UPDATE` 한 방이면 잠금도 id 순). 정렬 한 줄로 끝나는 가장 싼 해법.
- **해법 2 = 감지/타임아웃: 안전망일 뿐.** InnoDB는 데드락 자동 감지 후 한쪽을 강제 롤백(`Deadlock found ...`) → 시스템은 안 멈추지만 희생자는 주문 실패. `innodb_lock_wait_timeout`(기본 50초)은 일반 대기 안전망. 예방이 1순위, 감지·타임아웃은 보험.
- **쿠폰은 데드락 행렬에 안 들어감**: 낙관적 락은 락을 안 잡음(version 비교만). 비관적 락 대상은 product 행들뿐 → 거기만 정렬하면 됨.
- **코드 연결**: 주문 경로에서 productId 정렬 → 정렬 순서로 락 조회. 기존 조회 로직이 어떤 순서로 도는지 구현 단계에서 확인 후 적용.

**[답변]**
> (ORDER BY id + FOR UPDATE로 잠금이 id 순서로 걸린다는 부분) 이게 이해가 안가 어케한다는거임?

**[멘토 보충: 락은 "한꺼번에"가 아니라 "행을 읽는 순서대로 하나씩" 걸린다]**
- FOR UPDATE는 결과 행 전체를 동시에 잠그지 않음. DB는 **행을 읽는 순서대로 한 행씩** 잠그고, 남이 잠근 행을 만나면 그 자리에서 대기. → "잠그는 순서"가 존재하고, 트랜잭션마다 다르면 데드락.
- 정렬 X 타임라인: t1) A 상품1 ✔ / B 상품2 ✔ → t2) A 상품2 대기 / B 상품1 대기 → 서로 영원히 대기(데드락).
- 정렬 O 타임라인: t1) A 상품1 ✔ / B 상품1 앞 **빈손 대기** → t2) A 상품2 ✔ → 커밋·해제 → t3) B가 1→2 차례로 획득·진행.
- 원리: 전원이 오름차순으로 잡으면 **대기는 항상 "더 큰 id" 방향으로만 발생**. 내가 id=5를 기다리면 난 5 미만만 쥔 상태고, 5를 쥔 상대는 앞으로 5 초과만 요청 → 나를 기다릴 수 없음. 한 방향 화살표로는 원을 못 그림 = 순환 대기 물리적 불가.
- 코드 두 모양(본질 동일 = 전원 같은 기준 줄 세우기): ① `@Lock(PESSIMISTIC_WRITE)` + `WHERE p.id IN :ids ORDER BY p.id` 한 방 쿼리(PK 인덱스를 id 순으로 훑으며 차례 잠금) ② 자바에서 id 정렬 후 하나씩 락 조회 루프. → **방법 ①로 결정.**

# 구현 중 결정 로그 (1~2단계)

## D1. Discount.apply — 할인 계산 (1단계 완료)
- 시그니처 `Money apply(Money orderAmount)` → **할인 금액** 반환(최종 금액 아님). 최종 = 원금 - 할인으로 유도.
- RATE 반올림: **절사(내림)** — `divide(100, 0, RoundingMode.DOWN)`.
- FIXED > 주문금액: **주문금액으로 캡** — 마지막에 `min(orderAmount)` 한 번, 분기 안 늘림.
- "if는 나쁜 설계?" 검토: 격언의 표적은 **타입 분기가 여러 곳에 복제**되는 것. 우리는 분기가 Discount.apply 한 곳 + enum switch는 컴파일러가 누락 케이스를 막음 → 다형성(enum 추상 메서드/클래스 분리) 전환은 현재 과함(타입 2개·분기 1곳·확장 계획 없음, 클래스 분리는 @Embeddable과 충돌). **전환 트리거**: 같은 type 분기가 2~3곳 생기는 순간 enum 다형성으로.

## D2. UserCoupon.use — 검증 위치 (2단계 완료)
- 만료 검증 위치: 서비스 vs **엔티티(채택)**. `use(expiredAt, now)`가 USED 중복·만료 모두 검사 후 전이. resolveStatus 재사용으로 판정 로직 한 곳 유지. `@Version` 추가(낙관적 락).
- 근거 = **코드베이스 일관성**: 같은 엔티티의 `resolveStatus(expiredAt, now)`(외부 재료를 파라미터로 받아 판단은 엔티티가), `User.authenticate(rawPassword, encoder)`, `Stock.decrease(quantity)` 전부 동형. 서비스에 불변식 검증을 둔 반례 0(서비스는 NOT_FOUND 조회 실패만).
- 넷플릭스 참고 검증: 공개 자료(Studio의 Hexagonal Architecture 포스트)는 오히려 Interactor(유스케이스 객체) 중심 — 외부 권위는 갈려 있어 근거가 못 됨. 결정 기준은 우리 코드 일관성.

## D3. 만료를 상태 컬럼 없이 시간 파생으로 — 근거 (멘토 QnA 스크린샷 논의)
- Q3 결정의 재확인. 사용자 질문: "상태 컬럼 안 두는 게 좋다는 근거가 뭔가? 스케줄러로는 정확히 10시에 만료 못 시킨다던데."
- **근거: 파생 가능한 값을 저장하면 진실이 두 곳이 된다.** 진짜 진실(now > expiredAt) vs 저장된 주장(status 컬럼). 두 곳은 반드시 어긋나는 창이 생김 — 스케줄러 기동·10만 row UPDATE 동안 "실제 만료인데 컬럼은 AVAILABLE" → 만료 쿠폰 사용 사고. 스케줄러 자체도 새 비용(대량 UPDATE 부하)·새 실패 모드(다운, 분산 중복 실행).
- 파생이면 진실 원천이 expiredAt 하나 → 10:00:00.000에 정확히 만료, **UPDATE 자체가 불필요**해 "정확히 만료 못 시킴" 문제가 성립 안 함. "심플하게"의 실체 = 동기화할 사본을 만들지 말라는 정합성 원칙.
- 저장 vs 파생 기준: **재계산 가능한가.** EXPIRED(재계산 가능) → 파생. USED(사건, 재계산 불가) → 저장. 그래서 status는 AVAILABLE/USED만.
- "row마다 스케줄링?" → 안 함. 컬럼이 불가피하면 주기 배치(`WHERE expired_at < now()` 일괄 UPDATE)로 지연을 수용하는 것.
- 극단까지 밀면 status 대신 used_at 타임스탬프로 완전 파생 가능하나, 현 설계도 원칙 충족 → 유지.

## D4. lock-free 대안 검토 (사용자 질문: "lock-free하게 해결할 수 있는 게 있어?")
- DB에서 "lock-free" = 잡고 기다리는 락 없이 **원자적 한 방 연산**으로 해결(CAS 계열). 행 갱신의 내부 짧은 락은 항상 존재 — 엄밀한 lock-free는 아님.
- **쿠폰: 이미 lock-free 계열.** @Version 낙관적 락의 실체 = `UPDATE ... WHERE version=3` = DB판 CAS. 이름과 달리 분류상 락이 아니라 충돌 감지. 수동 CAS(`WHERE status='AVAILABLE'` + 영향 행 수 검사)도 동일 효과지만 @Version이 내장이라 더 쌈.
- **좋아요: 진짜 대안 둘.** ① 원자적 UPDATE(`SET like_count = like_count + 1`) — read-modify-write 3단계가 한 문장이 되어 Lost Update 성립 불가. ② D3 적용 — like_count는 like 테이블 COUNT(*)로 재계산 가능한 파생값(비정규화 캐시), 컬럼을 버리면 동기화할 사본 자체가 소멸.
- **재고: 조건부 UPDATE(`SET stock=stock-? WHERE stock>=?`) 가능하나 비용** — ① 도메인 규칙이 Stock.decrease에서 SQL로 이사(@Modifying, 영속성 컨텍스트 불일치) ② 트랜잭션 내 UPDATE 행 락도 커밋까지 유지라 다중 상품 데드락 순서 문제 잔존 ③ 검증·스냅샷용 읽기가 어차피 필요해 FOR UPDATE가 자연스러움.
- **결론**: quest가 락 학습 과제이므로 합의 유지(재고·좋아요 비관적, 쿠폰 낙관적=사실상 CAS). 좋아요 원자적 UPDATE는 Technical Writing Quest 벤치마크 소재로 적합("비관적 락 vs 원자적 UPDATE" 측정).

## D5. lock-free / 낙관적 / 비관적 우선순위와 근거 (사용자: "진짜 lock-free가 좋다면 포함할 것")
- **우선순위 = 정확성을 만족하는 가장 약한(싼) 도구부터**: ⓪ 공유 자원 제거(파생·INSERT-only) → ① 원자적 한 문장 연산 → ② 낙관적(OCC) → ③ 비관적 → (④ 분산 락). 아래로 갈수록 강해지는 게 아니라 비싸짐.
- **근거 ① 직렬화 구간 길이 = 처리량 상한**(행당 처리량 ≤ 1/독점시간): lock-free는 UPDATE 한 문장 마이크로초 / 낙관적은 독점 없음(대신 충돌 시 한 작업 폐기) / 비관적은 읽기~커밋까지 독점(가장 김).
- **근거 ② 내려갈 때마다 새 실패 모드**: lock-free = 영향 행 0(즉시·명확) / 낙관적 = 커밋 시점 예외+재시도 판단 / 비관적 = 대기+데드락·타임아웃. "안전해 보여서"가 아니라 "필요해서" 내려가야 함.
- **격상 조건(거부권)**: ⑴ 원자 한 문장에 안 들어가는 다단계 작업(검증·스냅샷·타 테이블 참조) → lock-free 탈락. lock-free는 표현력을 포기하고 속도를 얻는 도구. ⑵ 전원 처리 요구 + 충돌 빈번 → 낙관적 탈락(재시도 폭풍). ⑶ 남는 게 비관적 — 가장 비싸지만 표현력 최광.
- **재적용 결과**: 좋아요 = `+1`이 완벽히 한 문장 → **원자적 UPDATE가 1순위**(기존 "비관적"은 교육 목적 타협이었음). 재고 = 다단계+전원 처리 → 비관적 유지. 쿠폰 = 다단계+패자 실패가 정답 → 낙관적(=CAS) 유지. 재고·쿠폰이 두 락을 커버하므로 과제 요구 충족 + 세 자원이 세 메커니즘 시연.
- → **결정: 좋아요 = 원자적 UPDATE로 변경** (사용자 확정). 최종: 재고 비관적 / 쿠폰 낙관적(OCC) / 좋아요 원자적 UPDATE.

**[정정 (사용자 지적: "왜 낙관적이 CAS야? CAS == lock-free 아냐?")]**
- 맞는 지적. "낙관적 락"은 락을 안 잡음 — 정식 명칭은 낙관적 동시성 제어(OCC), 본질은 상호 배제가 아니라 충돌 감지 → **비차단(lock-free 계열)이 맞다.** ①/②를 "lock-free냐 아니냐"로 가른 기존 표현은 부정확.
- ①과 ②의 진짜 차이 = **충돌 창의 크기**. ① 원자적 한 문장: 읽기-계산-쓰기가 DB 내부 한 문장으로 붕괴 → 충돌 창 0. `like_count = like_count + 1`은 비교조차 없는 fetch-and-add라 실패 모드 자체가 없음(전원 성공). ② OCC: 읽고(t1)-앱에서 생각하고(t2)-쓰는(t3) 전 구간이 충돌 창이고 CAS는 t3에서 검사기로만 등장. 충돌 시 t1~t2 작업 폐기.
- 차단/비차단 경계선은 ①/② 사이가 아니라 **②/③ 사이**. ①이 ②보다 위인 이유는 "lock-free라서"가 아니라 충돌 창 0 + 실패 모드 없음/즉각. 대가 = 한 문장 표현력 한계. ②는 충돌 창을 여는 대가로 다단계 로직 표현력을 삼(쿠폰이 ②인 이유 = 만료 검증 등 다단계).

## D7. (예약된 학습 이행) 트랜잭션 범위·전파 — OrderFacade 실코드 기준
- **물리 트랜잭션은 1개.** Controller → 프록시(스프링이 OrderFacade를 감싼 대리인)가 호출을 가로채 트랜잭션 시작(커넥션 점유) → 진짜 place() 실행 → 정상 종료 시 프록시가 커밋.
- 안에서 부르는 `orderService.prepareItems` / `couponService.use` / `orderService.complete`도 각자 @Transactional이지만 전파 기본값 **REQUIRED = "이미 굴러가는 트랜잭션이 있으면 새로 열지 않고 합류"** → 전부 Facade 트랜잭션 안. 같은 커넥션·같은 영속성 컨텍스트 공유 → prepareItems에서 수정한 Product(재고)와 use에서 수정한 UserCoupon(USED)이 커밋 시점에 함께 flush.
- **롤백 범위**: use()가 만료 쿠폰 BAD_REQUEST를 던지면 예외가 place()를 뚫고 프록시까지 → RuntimeException 감지 → **전체 롤백** → 이미 차감한 재고도 원복. "쿠폰만 USED, 주문 없음" 사고가 구조적으로 불가능.
- **반사실**: Facade에 @Transactional이 없으면 각 서비스가 자기 트랜잭션을 열고 **각자 커밋** — complete 실패 시 재고·쿠폰은 이미 확정 = Q11에서 걱정한 바로 그 사고.
- **self-invocation 함정**: 프록시는 밖에서 들어오는 호출만 가로챔. 같은 클래스 안에서 this.메서드() 호출은 프록시를 안 거쳐 @Transactional 무시. 우리 구조는 Facade가 **다른 빈**(orderService, couponService)을 호출하므로 안전.
- 주의(보너스): REQUIRED 합류 중 내부에서 예외가 나면 잡아서 무시해도 트랜잭션은 rollback-only로 마킹되어 커밋 시도 시 실패한다.
- 보충(사용자 질문 "트랜잭션 1개가 왜 중요?"): **롤백은 트랜잭션 단위로만 가능** — 커밋된 트랜잭션은 못 되돌림(Durability). 물리 트랜잭션 개수 = 원자성의 단위. "1개" = quest의 "하나라도 실패하면 모두 롤백" 충족 증명. 실용: 트랜잭션을 쪼개는 변경(REQUIRES_NEW 부착 / Facade @Transactional 제거 / self-invocation 리팩토링)을 리뷰에서 알아채는 데 쓰는 지식. 비관적 락 보유 시간 = Facade place() 전체 = 트랜잭션 경계에서 파생.

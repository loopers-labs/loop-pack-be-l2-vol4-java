# 유저 시나리오 기반 기능 정의 & 요구사항 명세

---

## 서비스 개요

좋아요를 누르고, 쿠폰을 쓰고, 카드로 결제하는 감성 이커머스.
사용자는 다양한 브랜드의 상품을 탐색하고, 좋아요·주문·결제로 이어지는 흐름을 경험한다.
유저의 행동은 모두 기록되어 랭킹·추천 등 다양한 기능 확장의 기반이 된다.

---

## 전체 서비스 시나리오

```
회원가입
  → 상품 탐색 (브랜드·상품 목록 조회)
    → 마음에 드는 상품에 좋아요
      → 쿠폰 발급
        → 여러 상품 한 번에 주문 & 결제
          → 유저 행동 기록 → 랭킹 / 추천 확장
```

---

## API 공통 규칙

| 구분 | Prefix | 인증 방식 |
|------|--------|-----------|
| 대고객 | `/api/v1` | `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 |
| 어드민 | `/api-admin/v1` | `X-Loopers-Ldap: loopers.admin` 헤더 |

- 인증/인가 구현은 스코프 외. 헤더로 유저를 식별한다.
- 유저는 타 유저의 정보에 직접 접근할 수 없다.

## 도메인별 요구사항

---

### 1. 유저 (Users)

#### 유저 스토리

- 로그인한 사용자는 비밀번호를 변경할 수 있다.
- 사용자는 타 유저의 정보에 접근할 수 없다.

---

#### 유스케이스

**1. 비밀번호 변경**

액터: 로그인 사용자

목적: 보안을 위한 비밀번호 갱신

1) Main Flow

(1) 사용자가 현재 비밀번호와 새 비밀번호를 입력한다.

(2) 시스템이 헤더로 유저를 식별한다.

(3) 시스템이 현재 비밀번호가 일치하는지 확인한다.

(4) 시스템이 새 비밀번호로 업데이트한다.

2) Alternate Flow

해당 없음

3) Exception Flow

→ 현재 비밀번호가 일치하지 않는 경우: 오류 메시지를 반환한다.

→ 새 비밀번호가 현재 비밀번호와 동일한 경우: 오류 메시지를 반환한다.

### 2. 브랜드 & 상품 (Brands / Products)

#### 유저 스토리

- 사용자는 특정 브랜드의 상세 정보를 조회할 수 있다.
- 사용자는 전체 상품 목록을 탐색할 수 있다.
- 사용자는 특정 브랜드의 상품만 필터링하여 볼 수 있다.
- 사용자는 최신순·가격순·좋아요순으로 상품을 정렬할 수 있다.
- 사용자는 상품의 상세 정보, 좋아요 여부, 과거 주문내역 여를 조회할 수 있다.


---

#### 유스케이스

**1. 상품 목록 조회**

액터: 사용자 (비회원 포함)

목적: 원하는 상품 탐색

조회 조건

| 파라미터 | 검색 방식 | 설명 |
|----------|-----------|------|
| `productName` | like | 상품명 부분 검색 |
| `brandName` | like | 브랜드명 부분 검색 |
| `categoryLarge` | 일치 | 대분류 카테고리 |
| `categoryMiddle` | 일치 | 중분류 카테고리 |
| `categorySmall` | 일치 | 소분류 카테고리 |
| `brandId` | 일치 | 특정 브랜드 필터 |
| `sort` | — | `latest` / `price_asc` / `likes_desc` (기본: `latest`) |
| `page` | — | 페이지 번호 (기본값 0) |
| `size` | — | 페이지당 상품 수 (기본값 20) |

1) Main Flow

(1) 사용자가 조회 조건(상품명, 브랜드명, 카테고리, brandId, sort 등)을 입력하여 목록 조회를 요청한다.

(2) 시스템이 productName이 있으면 상품명 like 검색을 적용한다.

(3) 시스템이 brandName이 있으면 브랜드명 like 검색을 적용한다.

(4) 시스템이 카테고리(대/중/소) 조건이 있으면 해당 카테고리로 필터링한다.

(5) 시스템이 brandId가 있으면 해당 브랜드로 필터링한다.

(6) 시스템이 sort 기준에 따라 정렬한다. (기본: 최신순)

(7) 시스템이 페이지네이션을 적용하여 목록을 반환한다.

2) Alternate Flow

→ 조회 조건이 없는 경우: 전체 상품을 대상으로 조회한다.

→ sort가 지정되지 않은 경우: 최신순(latest)으로 정렬한다.

→ 조건에 해당하는 상품이 없는 경우: 빈 목록을 반환한다.

3) Exception Flow

→ 지원하지 않는 sort 값이 입력된 경우: 오류 메시지를 반환한다.

→ 카테고리 대/중/소 계층이 맞지 않는 경우: 오류 메시지를 반환한다.

---

**2. 브랜드 목록 조회**

액터: 사용자 (비회원 포함)

목적: 전체 브랜드 탐색

1) Main Flow

(1) 사용자가 브랜드 목록 조회를 요청한다.

(2) 시스템이 전체 브랜드 목록을 반환한다.

2) Alternate Flow

→ 브랜드가 없는 경우: 빈 목록을 반환한다.

3) Exception Flow

해당 없음

---

**3. 브랜드 정보 조회**

액터: 사용자 (비회원 포함)

목적: 특정 브랜드의 상세 정보 확인

1) Main Flow

(1) 사용자가 brandId로 브랜드 정보 조회를 요청한다.

(2) 시스템이 해당 브랜드의 상세 정보를 반환한다.

2) Alternate Flow

해당 없음

3) Exception Flow

→ 존재하지 않는 브랜드인 경우: 오류 메시지를 반환한다.

---

**4. 상품 정보 조회**

액터: 사용자 (비회원 포함)

목적: 특정 상품의 상세 정보 확인

1) Main Flow

(1) 사용자가 productId로 상품 정보 조회를 요청한다.

(2) 시스템이 해당 상품의 상세 정보, 좋아요 여부, 과거 주문내역 여부를 반환한다.

2) Alternate Flow

해당 없음

3) Exception Flow

→ 존재하지 않는 상품인 경우: 오류 메시지를 반환한다.

### 2-ADMIN. 브랜드 & 상품 ADMIN

#### 유저 스토리

- 어드민은 브랜드와 상품을 등록·수정·삭제할 수 있다.
- 브랜드 삭제 시 해당 브랜드의 상품도 함께 삭제된다.
- 상품 등록 시 브랜드는 이미 등록된 브랜드여야 한다.
- 상품 수정 시 브랜드는 변경할 수 없다.
> 고객과 어드민에게 제공되는 상품·브랜드 응답 필드는 다를 수 있다.
> (예: 어드민은 재고·원가 등 내부 정보 포함, 고객은 노출용 정보만 제공)
---

#### 유스케이스

**1. 등록된 브랜드 목록 조회**

액터: 어드민

목적: 등록된 전체 브랜드 현황 파악

1) Main Flow

(1) 어드민이 브랜드 목록 조회를 요청한다.

(2) 시스템이 어드민 권한을 확인한다.

(3) 시스템이 전체 브랜드 목록을 페이지네이션하여 반환한다.

2) Alternate Flow

해당 없음

3) Exception Flow

→ 어드민 권한이 없는 경우: 인증 오류 메시지를 반환한다.

### 3. 좋아요 (Likes)

#### 유저 스토리

- 사용자는 마음에 드는 상품에 좋아요를 누를 수 있다.
- 이미 좋아요한 상품을 다시 누르면 좋아요가 취소된다.
- 사용자는 자신이 좋아요한 상품 목록을 볼 수 있다.
- 좋아요 수는 상품 정보에 반영된다.

---

#### 유스케이스

**1. 좋아요 등록**

액터: 로그인 사용자

목적: 관심 있는 상품 저장

1) Main Flow

(1) 사용자가 상품 상세 페이지에서 좋아요 버튼을 클릭한다.

(2) 시스템이 해당 상품의 존재 여부를 확인한다.

(3) 시스템이 좋아요 정보를 저장한다.

(4) 시스템이 좋아요 수를 증가시킨다.

2) Alternate Flow

해당 없음

3) Exception Flow

→ 로그인하지 않은 경우: 인증 오류 메시지를 반환한다.

- 결제 시 쿠폰을 적용하여 할인받을 수 있다.
- 결제가 완료되면 주문 상태가 변경된다.

→ 존재하지 않는 주문인 경우: 오류 메시지를 반환한다.

(3) 시스템이 결제 취소를 처리한다.

(4) 시스템이 주문 상태를 취소로 변경한다.

(5) 시스템이 차감되었던 재고를 복구한다.

(6) 시스템이 사용된 쿠폰을 복원한다.

(7) 시스템이 취소된 상품을 장바구니에 복원한다.

3) Exception Flow

→ 존재하지 않는 결제인 경우: 오류 메시지를 반환한다.

→ 이미 취소된 결제인 경우: 오류 메시지를 반환한다.

→ 취소 불가 상태인 경우 (배송 중 등): 오류 메시지를 반환한다.

→ 타 유저의 결제 취소 시도 시: 접근 거부 오류 메시지를 반환한다.

## 요약: 도메인별 핵심 제약 사항

| 도메인 | 핵심 제약 |
|--------|-----------|
| 유저 | loginId 중복 불가, 타 유저 정보 접근 불가 |
| 브랜드 & 상품 | 브랜드 삭제 시 하위 상품 함께 삭제 |
| 브랜드 & 상품 ADMIN | 등록 시 브랜드 존재 필수, 수정 시 브랜드 변경 불가, 어드민 권한 필수 |
| 좋아요 | 로그인 필수, 중복 좋아요 시 취소 처리, 좋아요 수 상품에 반영 |
| 주문 | 재고 확인 및 원자적 차감, 주문 시점 스냅샷 저장 필수 |
| 주문 ADMIN | 어드민 권한 필수, 수익 집계는 결제 완료 주문 기준 |
| 장바구니 | 로그인 필수, 동일 상품 재담기 시 수량 누적 |
| 결제 | 결제 완료 시 주문 상태 변경, 취소 시 재고·쿠폰 복원 필수 |

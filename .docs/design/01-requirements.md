# 01. 요구사항 명세

## 1. 설계 범위

**포함**
- 브랜드 조회(고객) / 브랜드 CRUD(어드민)
- 상품 목록·상세 조회(고객) / 상품 CRUD(어드민)
- 상품 좋아요 등록·취소 및 내 좋아요 목록 조회 (멱등)
- 주문 생성 및 결제 흐름 — 재고 차감, 포인트 차감, 외부 PG 연동
- 주문 조회(고객) / 주문 조회(어드민)

**제외**
- 회원가입 / 내 정보 조회 / 비밀번호 변경 (1주차 `volume-1` 범위)
- 쿠폰 (시나리오 언급되나 2주차 범위 외)
- 포인트 충전·적립 (명세에 없음 — 초기 잔액 보유를 가정)
- 동시성·일관성·랭킹·추천의 구체 구현 (시나리오 "나아가며" — 후속 주차)

## 3. 유저 시나리오 (User Stories)

> 형식: `US-XX  <역할>로서 <목적>을 위해 <행동>한다.`

- US-01  고객으로서 마음에 드는 상품을 찾기 위해 브랜드·정렬 기준으로 상품 목록을 둘러본다.
- US-02  고객으로서 구매를 결정하기 위해 상품 상세(가격·재고·브랜드)를 확인한다.
- US-03  고객으로서 브랜드를 알아보기 위해 브랜드 정보를 조회한다.
- US-04  고객으로서 관심 상품을 표시하기 위해 상품에 좋아요를 누른다.
- US-05  고객으로서 관심이 식은 상품의 좋아요를 취소한다.
- US-06  고객으로서 관심 상품을 모아보기 위해 내가 좋아요한 상품 목록을 조회한다.
- US-07  고객으로서 여러 상품을 한 번에 주문한다.
- US-08  고객으로서 보유 포인트로 주문을 결제하고, 외부 PG를 통해 결제가 처리된다.
- US-09  고객으로서 본인의 주문 내역(목록·상세)을 조회한다.
- US-10  어드민으로서 브랜드와 상품을 등록·수정·삭제한다.
- US-11  어드민으로서 전체 주문 목록과 상세를 조회한다.

## 4. 기능 요구사항

### 4.1 브랜드

- 고객 — `GET /api/v1/brands/{brandId}` : 브랜드 단건 조회
  - 응답 항목: `id`, `name`, `description`
- 어드민 — `/api-admin/v1/brands`
  - 목록 조회(페이징) / 상세 조회 / 등록 / 수정 / 삭제
  - **브랜드 삭제 시 해당 브랜드의 상품도 함께 삭제된다.**

### 4.2 상품

- 고객 — `GET /api/v1/products` : 상품 목록 조회
  - 페이징 기본값 `page=0`, `size=20`
  - 필터: `brandId`
  - 정렬 `sort`: `latest`(기본) / `price_asc` / `likes_desc`
  - 응답 항목: `id`, `brandId`, `name`, `price`, `stock`, `likeCount`
- 고객 — `GET /api/v1/products/{productId}` : 상품 상세 조회 (위 항목 + 브랜드 정보)
- 재고 0(품절) 상품도 목록·상세에 노출하며 품절 상태를 표현한다.
- 어드민 — `/api-admin/v1/products`
  - 목록 조회(페이징, `brandId` 필터) / 상세 조회 / 등록 / 수정 / 삭제
  - **상품 등록 시 브랜드는 이미 등록된 브랜드여야 한다.**
  - **상품 수정 시 브랜드는 변경할 수 없다.**

### 4.3 좋아요 (멱등)

- 등록 — `POST /api/v1/products/{productId}/likes`
  - 이미 좋아요한 상태에서 다시 호출해도 좋아요는 1건으로 유지된다 (멱등).
- 취소 — `DELETE /api/v1/products/{productId}/likes`
  - 좋아요하지 않은 상태에서 호출해도 에러 없이 0건을 유지한다 (멱등).
- 내 좋아요 목록 — `GET /api/v1/users/{userId}/likes`
  - 경로의 `userId`는 인증 헤더의 유저와 일치해야 하며, 타인의 목록은 조회할 수 없다.
- 좋아요 수는 `likes` 테이블 집계(COUNT)로 산출한다 — 별도 카운터 컬럼을 두지 않는다.
- `UNIQUE(user_id, product_id)` 제약으로 중복 좋아요를 차단한다.

### 4.4 주문 생성

- 요청 — `POST /api/v1/orders`, 본문 `items[].productId`, `items[].quantity`
- 주문 시점에 각 상품의 재고를 확인하고 차감한다. **재고가 부족하면 주문이 실패한다.**
- 주문 항목에 당시 상품 정보(상품명·단가)를 **스냅샷**으로 저장한다.
- 주문 생성 직후 상태는 `PENDING`이며, 결제 결과에 따라 `PAID` 또는 `FAILED`로 전이한다.

### 4.5 결제 흐름

- 결제 수단은 고객의 보유 **포인트**이며, 외부 **PG**가 결제 승인/취소를 처리한다.
- 결제 금액은 주문 총액이다. **보유 포인트가 결제 금액 미만이면 결제가 실패한다** (`400 Bad Request`).
- 처리 순서: 재고 차감 → 포인트 차감 → 외부 PG 결제 요청.
- PG 결제 성공 시: `Payment = SUCCESS`, `Order = PAID`.
- PG 결제 실패/타임아웃 시: **보상 트랜잭션**으로 포인트와 재고를 복원하고 `Payment = FAILED`, `Order = FAILED`로 둔다.

### 4.6 주문 조회

- 고객 — `GET /api/v1/orders?startAt&endAt` : 본인 주문 목록 (기간 필터)
- 고객 — `GET /api/v1/orders/{orderId}` : 본인 주문 단건 상세
  - **타인의 주문은 조회할 수 없다** (존재 노출을 막기 위해 `404`로 응답).
- 어드민 — `GET /api-admin/v1/orders` : 전체 주문 목록(페이징)
- 어드민 — `GET /api-admin/v1/orders/{orderId}` : 단건 상세
- 주문 조회 응답 항목:
  - 목록: `orderId`, `status`, `totalAmount`, `orderedAt`(주문 일시)
  - 단건 상세: 목록 항목 + 주문 항목 배열(`productId`, `productNameSnapshot`, `priceSnapshot`, `quantity`, `subtotal`) + 결제 상태(`paymentStatus`)

## 5. 비기능 요구사항 / 제약

- 인증/인가는 헤더 식별만 수행하며 토큰 발급은 하지 않는다 (시나리오 명시).
  - **[알려진 제약]** `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더로 비밀번호를 평문 전송하는 방식은 시나리오가 정한 제약이다. 프록시·로드밸런서·접근 로그에 자격 증명이 노출될 수 있고 HTTPS 미사용 시 스니핑에 취약하다. 따라서 실서비스 운영 시 HTTPS 적용을 전제하고 로그에서 해당 헤더값을 마스킹한다. 토큰(JWT 등) 기반 인증은 시나리오 범위 외이므로 본 설계에서는 도입하지 않는다.
- 고객은 타 유저의 정보·좋아요 목록·주문에 직접 접근할 수 없다.
- 응답 포맷은 `ApiResponse<T>`로 일관 유지하고, 예외는 `ApiControllerAdvice`에서 변환한다 (프로젝트 컨벤션).
- 모든 엔티티는 `BaseEntity`(id, created_at, updated_at, deleted_at)를 상속한다.
- 대고객 API는 `/api/v1`, 어드민 API는 `/api-admin/v1` prefix 를 사용한다.

## 6. 정책 결정 / 가정

> 시나리오에 명시되지 않아 설계자가 결정·가정한 항목.

| 항목 | 결정 | 근거 |
| --- | --- | --- |
| 결제 수단 | 보유 포인트로 결제하고, 외부 PG가 결제를 처리 | 과제 명세 "포인트 차감 + 외부 시스템 연동"을 단일 흐름에 모두 포함 |
| 포인트 충전·적립 | 설계 범위 외, 초기 잔액 보유를 가정 | 시나리오·과제 명세에 충전 기능이 없음 |
| 좋아요 수 | `likes` 테이블 COUNT 집계로 산출 | 카운터 컬럼의 정합성 관리 부담 회피, 데이터 단일 출처 유지 |
| 좋아요 취소 | hard delete (행 삭제) | 멱등성과 `UNIQUE(user_id, product_id)` 양립 — soft delete는 unique 키와 충돌 |
| 재고 차감 시점 | 주문 생성 시 차감, 결제 실패 시 보상 복원 | 과제 명세가 "주문 시 재고 확인·차감"을 보장 요구 |
| 재고 0 상품 | 목록·상세에 노출(품절 표시), 주문 시점에 검증 | 품절 정보도 고객에게 유용 |
| 주문 취소 | API 미설계 (`OrderStatus`에 `CANCELED` 제외) | 시나리오에 취소 기능이 없음 (YAGNI) |
| 타인 주문 접근 | `404`로 응답 (`403` 대신) | 주문의 존재 여부 자체를 노출하지 않기 위함 |
| 동시성 제어 | 정책 수준만 기술하고 구체 구현은 제외 | 시나리오 "나아가며" — 후속 주차 주제 |

## 7. 제외 범위 (Out of Scope)

- 회원가입 / 내 정보 / 비밀번호 변경 (1주차)
- 쿠폰 발급·사용
- 포인트 충전·적립
- 동시성·일관성·랭킹·추천의 구체 구현 (후속 주차)

## 8. 용어집

| 용어 | 정의 |
| --- | --- |
| 포인트 | 고객이 보유한 결제 수단 잔액. 주문 결제 시 차감되고, 결제 실패 시 복원된다 |
| 스냅샷 | 주문 시점의 상품 정보(상품명·단가)를 주문 항목에 복사 저장한 값 |
| 멱등 | 동일 요청을 N번 보내도 시스템 상태가 1번 보낸 것과 같음 |
| 보상 트랜잭션 | 결제 실패 시, 앞서 수행한 재고·포인트 차감을 되돌리는 처리 |
| PG | 외부 결제 게이트웨이. 결제 승인/취소를 처리하는 외부 시스템 |

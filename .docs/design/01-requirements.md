#  이커머스 백엔드 요구사항 명세서

## 1. 목적 및 범위

회원이 브랜드별 상품을 탐색하고 좋아요를 표시하며, 여러 상품을 한 번에 주문할 수 있는 이커머스 백엔드의 기능 요구사항을 정의한다. 어드민은 브랜드·상품을 관리하고 전체 주문을 조회한다.

### 1.1 범위 (In-Scope)

| 도메인 | 한 줄 책임 | 이번 문서가 다루는 범위 |
|---|---|---|
| **브랜드** | 상품을 묶는 상위 분류 단위 | 고객 조회 + 어드민 등록·수정·삭제·조회 |
| **상품** | 탐색·주문의 대상. 가격·재고를 가진다 | 고객 조회(브랜드 필터·정렬) + 어드민 등록·수정·삭제·조회 |
| **좋아요** | 회원이 상품에 관심을 표현하는 행위 | 등록·취소·내 목록 조회 |
| **주문** | 회원의 구매 확정 | 여러 상품 동시 주문, 기간별 내역 조회, 어드민 전체 조회 |

### 1.2 범위 밖 (Out-of-Scope)

- 결제 — 추후 확장
- 쿠폰 — 추후 확장
- 랭킹/추천 — 추후 확장
- 리뷰·평점 — 이번 범위 밖
- 문의/CS — 이번 범위 밖

## 2. 액터

이 서비스를 이용하는 주체는 다음 셋이다.

| 액터 | 설명 | 로그인 |
|---------|---|---|
| **회원**  | 상품을 탐색하고, 좋아요를 누르고, 주문하는 사용자 | 필요 |
| **비회원** | 상품·브랜드를 둘러보기만 하는 사용자 | 불필요 |
| **어드민** | 브랜드·상품을 관리하고 전체 주문을 조회하는 운영자 | 운영자 권한 필요 |

상품과 브랜드를 둘러보는 것은 로그인 없이 누구나 할 수 있다.
좋아요와 주문은 회원으로 로그인해야 한다. 브랜드·상품 관리와 전체 주문 조회는 어드민 권한이 있어야 한다.
회원은 다른 회원의 정보(좋아요 목록·주문 내역 등)에 접근할 수 없다.
로그인이 필요한 기능을 로그인 없이 요청하거나 어드민 기능을 권한 없이 요청하면, 그 요청은 거부된다.

> 회원 가입·인증·내 정보 관리 기능은 이미 구현되어 있으며, 본 설계 문서는 이를 전제로 한다 — 따라서 회원/인증 자체는 설계 범위 밖이다. 회원의 로그인 여부 판정은 위 액터 정의를 따른다.

## 3. 유비쿼터스 언어

이 문서에서 쓰는 핵심 용어를 아래 의미로 고정한다.

| 용어 | 정의 |
|---|---|
| 회원 | 가입을 마친 사용자. 좋아요와 주문을 하는 주체 |
| 어드민 | 브랜드·상품을 관리하고 전체 주문을 조회하는 운영자 |
| 비회원 | 가입하지 않고 둘러보기만 하는 사용자 |
| 브랜드 | 상품을 묶는 그룹 단위. 하나의 브랜드는 여러 상품을 가진다 |
| 상품 | 탐색·주문의 대상. 반드시 하나의 브랜드에 속하며, 가격과 재고를 가진다 |
| 재고 | 상품의 판매 가능 수량. 0보다 작아질 수 없다 |
| 좋아요 | 회원이 상품에 표현하는 관심. 한 회원은 한 상품에 한 번만 누를 수 있다 |
| 주문 | 회원의 구매 확정 |
| 주문 항목 | 주문에 포함된 개별 상품 한 건. 주문 시점의 정보를 보존한다 |
| 스냅샷 | 주문 시점의 상품명·가격·브랜드명을 그대로 복사해 둔 것. 이후 원본 상품이 바뀌거나 삭제되어도 영향받지 않는다 |

## 4. 도메인별 요구사항

### 4.1 브랜드

### 시나리오
29m 운영 어드민은 플랫폼에서 판매할 신발 브랜드 라인업을 관리해야 한다.
신규 입점이 확정된 브랜드를 등록하고, 로고나 소개가 바뀌면 정보를 갱신하며, 계약이 종료된 브랜드는 더 이상 노출되지 않도록 내린다.
브랜드를 내릴 때는 그 브랜드의 신발 상품도 함께 사라져야 한다. 브랜드는 없는데 상품만 남는 상태가 생기면 안 되기 때문이다.
운영 화면에서는 이미 내린 브랜드까지 들여다봐야 하는데, 입점 이력을 확인하거나 잘못 내린 브랜드를 되살릴 일이 있어서다.

한편 29m을 방문한 고객은 마음에 드는 신발의 브랜드가 어떤 곳인지 확인하고 싶어 한다. 브랜드 정보 조회는 로그인 없이 누구나 할 수 있다.
단, 고객에게는 브랜드 소개에 필요한 정보만 보이고, 입점일·수정 이력 같은 운영용 정보는 어드민에게만 보인다.

### 기능 목록

- 고객·비회원은 관심 있는 신발의 브랜드가 어떤 곳인지 확인하기 위해, 특정 브랜드의 정보를 로그인 없이 조회할 수 있다.
- 어드민은 신규 입점 브랜드를 플랫폼에 노출시키기 위해, 브랜드를 등록할 수 있다.
- 어드민은 현재 입점 현황을 파악하기 위해, 브랜드 목록을 조회할 수 있다.
- 어드민은 특정 브랜드의 운영 정보와 이력을 확인하기 위해, 브랜드 상세를 조회할 수 있다.
- 어드민은 변경된 브랜드 정보를 최신 상태로 유지하기 위해, 브랜드 정보를 수정할 수 있다.
- 어드민은 계약이 종료된 브랜드를 노출에서 제거하기 위해, 브랜드를 삭제할 수 있다. 이때 소속 상품도 함께 삭제된다.

#### 핵심 규칙

- 브랜드 이름은 중복될 수 없다. 같은 이름의 브랜드를 등록하려는 시도는 거절된다.
- 브랜드를 삭제하면 소속 상품이 빠짐없이 함께 삭제된다 — "브랜드 없는 상품"은 존재할 수 없다.
- 삭제된 브랜드는 고객에게 더 이상 보이지 않는다. 단, 어드민은 관리 목적으로 조회할 수 있다.
- 고객에게 보이는 정보와 어드민에게 보이는 정보는 다르다. 입점일·수정 이력 같은 운영용 정보는 어드민에게만 보인다.

#### 유스케이스

**UC-B01 — 브랜드 정보 조회 (고객)**

- **정상 흐름**
    1. 사용자가 특정 브랜드의 정보를 요청한다.
    2. 브랜드 정보를 보여준다.
- **예외 흐름**
    - E1. 브랜드가 존재하지 않거나 이미 삭제됨 → 고객은 해당 브랜드를 찾을 수 없다.

**UC-B02 — 브랜드 목록 조회 (어드민)**
    
- **정상 흐름**
    1. 어드민이 브랜드 목록을 요청한다.
    2. 등록된 브랜드 목록을 보여준다.
- **대안 흐름**
    - A1. 삭제된 브랜드도 목록에 포함한다 — 어드민은 입점 이력을 확인해야 하므로, 삭제 여부를 함께 표시한다.
    - A2. 조회 결과가 없으면 빈 목록을 보여준다.
    - A3. 목록은 페이지 단위로 나누어 보여준다 — 페이지 번호·페이지당 개수를 지정할 수 있고, 지정하지 않으면 기본값(0번 페이지, 20개)을 적용한다.

**UC-B03 — 브랜드 상세 조회 (어드민)**

- **정상 흐름**
    1. 어드민이 특정 브랜드의 상세 정보를 요청한다.
    2. 브랜드 상세 정보를 보여준다 — 입점일·수정일 등 운영용 정보를 포함한다.
- **대안 흐름**
    - A1. 삭제된 브랜드도 조회할 수 있다.
- **예외 흐름**
    - E1. 브랜드가 존재하지 않음 → 어드민은 해당 브랜드를 찾을 수 없다.

**UC-B04 — 브랜드 등록 (어드민)**

- **정상 흐름**
    1. 어드민이 브랜드 정보를 입력해 등록을 요청한다.
    2. 입력값이 올바르고 같은 이름의 브랜드가 없으면 브랜드를 등록한다.
- **예외 흐름**
    - E1. 필수 정보가 빠짐 → 등록되지 않는다.
    - E2. 같은 이름의 브랜드가 이미 있음 → 어드민은 다른 이름으로 다시 등록해야 한다.

**UC-B05 — 브랜드 수정 (어드민)**

- **정상 흐름**
    1. 어드민이 브랜드 정보 수정을 요청한다.
    2. 브랜드 정보를 수정한다.
- **예외 흐름**
    - E1. 브랜드가 존재하지 않음 → 수정할 수 없다.
    - E2. 이미 삭제된 브랜드를 수정하려는 경우 → 삭제된 브랜드는 "없는 것"으로 취급되어 수정할 수 없다.
    - E3. 수정하려는 이름이 다른 브랜드와 겹침 → 수정되지 않는다.

**UC-B06 — 브랜드 삭제 (어드민)**

- **정상 흐름**
    1. 어드민이 브랜드 삭제를 요청한다.
    2. 해당 브랜드에 속한 상품을 모두 삭제하고, 브랜드를 삭제한다.
- **대안 흐름**
    - A1. 이미 삭제된 브랜드를 다시 삭제 요청하면, 그대로 성공으로 처리한다(결과가 같으므로 오류로 보지 않는다).
- **후속 동작**
    - 함께 삭제된 상품에 눌려 있던 좋아요는 사용자의 좋아요 목록에서 더 이상 보이지 않는다.
    - 이미 만들어진 주문 내역은 브랜드·상품 삭제와 무관하게 그대로 보존된다.

### 4.2 상품

#### 시나리오

`29m` 어드민은 입점 브랜드의 신발들을 상품으로 등록한다. 
각 신발에 가격과 재고를 설정하고, 가격이 바뀌거나 재입고가 되면 정보를 수정하며, 단종된 신발은 더 이상 노출되지 않게 내린다.
상품을 등록할 때는 반드시 어떤 브랜드의 신발인지 지정해야 하고, 한 번 정해진 소속 브랜드는 바꿀 수 없다 — 잘못 지정했다면 내리고 다시 등록한다.

한편 `29m` 고객은 수많은 신발 중에서 원하는 것을 빠르게 찾고 싶어 한다. 그래서 특정 브랜드의 신발만 모아 보거나, 최신 입고순·가격순·인기순으로 정렬해 탐색한다.
마음에 드는 신발은 상세 페이지에서 가격과 브랜드를 확인한다. 정확한 재고 수량까지 알 필요는 없고 '지금 살 수 있는지'만 알면 되므로, 고객에게는 구매 가능 여부만 보인다.

#### 기능 목록

- 고객·비회원은 원하는 신발을 빠르게 찾기 위해, 상품 목록을 브랜드 필터·정렬을 적용해 조회할 수 있다.
- 고객·비회원은 구매를 결정하기 위해, 특정 상품의 상세 정보를 조회할 수 있다.
- 어드민은 입점 브랜드의 신발을 판매에 올리기 위해, 상품을 등록할 수 있다.
- 어드민은 현재 판매 중인 상품 현황을 파악하기 위해, 상품 목록을 브랜드 필터로 조회할 수 있다.
- 어드민은 특정 상품의 재고와 운영 정보를 확인하기 위해, 상품 상세를 조회할 수 있다.
- 어드민은 가격 변동·재입고를 반영하기 위해, 상품 정보를 수정할 수 있다.
- 어드민은 단종된 신발을 노출에서 제거하기 위해, 상품을 삭제할 수 있다.

#### 핵심 규칙

- 모든 상품은 반드시 하나의 브랜드에 속한다 — "브랜드 없는 상품"은 존재할 수 없다.
- 상품의 소속 브랜드는 등록 시 정해지며, 이후 변경할 수 없다.
- 상품의 가격과 재고는 0보다 작을 수 없다.
- 삭제된 상품은 고객에게 더 이상 보이지 않는다. 단, 어드민은 관리 목적으로 조회할 수 있다.
- 고객에게는 재고의 정확한 수량을 보여주지 않는다 — '지금 구매할 수 있는지'만 알려준다.

#### 유스케이스

**UC-P01 — 상품 목록 조회 (고객)**

- **정상 흐름**
    1. 사용자가 브랜드 필터와 정렬 기준을 선택해 상품 목록을 요청한다.
    2. 조건에 맞는 상품 목록을 보여준다.
- **대안 흐름**
    - A1. 브랜드를 선택하지 않으면 전체 상품에서 조회한다.
    - A2. 정렬 기준을 선택하지 않으면 최신순으로 보여준다.
    - A3. 조건에 맞는 상품이 없으면 빈 목록을 보여준다.
    - A4. 목록은 페이지 단위로 나누어 보여준다 — 페이지 번호·페이지당 개수를 지정할 수 있고, 지정하지 않으면 기본값(0번 페이지, 20개)을 적용한다.
- **예외 흐름**
    - E1. 존재하지 않는 브랜드로 필터링하면, 그 브랜드엔 상품이 없는 것과 같으므로 빈 목록을 보여준다.
    - E2. 지원하지 않는 정렬 기준을 요청했을 때의 동작 → 처리 방식 미정. → DEC-01
> 정렬 기준으로 최신순(기본)·가격 낮은 순·좋아요 많은 순을 제공한다.

**UC-P02 — 상품 상세 조회 (고객)**

- **정상 흐름**
    1. 사용자가 특정 상품의 상세 정보를 요청한다.
    2. 상품명·가격·브랜드·구매 가능 여부를 보여준다.
- **예외 흐름**
    - E1. 상품이 존재하지 않거나 이미 삭제됨 → 고객은 해당 상품을 찾을 수 없다.

**UC-P03 — 상품 목록 조회 (어드민)**

- **정상 흐름**
    1. 어드민이 상품 목록을 요청한다. 특정 브랜드로 범위를 좁힐 수 있다.
    2. 등록된 상품 목록을 보여준다 — 재고 등 운영용 정보를 포함한다.
- **대안 흐름**
    - A1. 삭제된 상품도 목록에 포함한다 — 어드민은 단종 이력을 확인해야 하므로, 삭제 여부를 함께 표시한다.
    - A2. 조회 결과가 없으면 빈 목록을 보여준다.
    - A3. 목록은 페이지 단위로 나누어 보여준다 — 페이지 번호·페이지당 개수를 지정할 수 있고, 지정하지 않으면 기본값(0번 페이지, 20개)을 적용한다.

**UC-P04 — 상품 상세 조회 (어드민)**

- **정상 흐름**
    1. 어드민이 특정 상품의 상세 정보를 요청한다.
    2. 상품 상세 정보를 보여준다 — 재고, 등록일·수정일 등 운영용 정보를 포함한다.
- **대안 흐름**
    - A1. 삭제된 상품도 조회할 수 있다.
- **예외 흐름**
    - E1. 상품이 존재하지 않음 → 어드민은 해당 상품을 찾을 수 없다.

**UC-P05 — 상품 등록 (어드민)**

- **정상 흐름**
    1. 어드민이 소속 브랜드·상품명·가격·재고를 입력해 등록을 요청한다.
    2. 입력값이 올바르고 지정한 브랜드가 유효하면 상품을 등록한다.
- **예외 흐름**
    - E1. 필수 정보가 빠졌거나 가격·재고가 0보다 작음 → 등록되지 않는다.
    - E2. 지정한 브랜드가 존재하지 않거나 이미 삭제됨 → 그 브랜드에는 상품을 등록할 수 없다.

**UC-P06 — 상품 수정 (어드민)**

- **정상 흐름**
    1. 어드민이 상품 정보 수정을 요청한다.
    2. 상품 정보를 수정한다.
- **예외 흐름**
    - E1. 상품이 존재하지 않음 → 수정할 수 없다.
    - E2. 이미 삭제된 상품을 수정하려는 경우 → 삭제된 상품은 "없는 것"으로 취급되어 수정할 수 없다.
    - E3. 가격·재고를 0보다 작게 수정하려는 경우 → 수정되지 않는다.
    - E4. 소속 브랜드를 바꾸려는 경우 → 상품의 브랜드는 등록 후 바꿀 수 없다.

**UC-P07 — 상품 삭제 (어드민)**

- **정상 흐름**
    1. 어드민이 상품 삭제를 요청한다.
    2. 상품을 삭제한다.
- **대안 흐름**
    - A1. 이미 삭제된 상품을 다시 삭제 요청하면, 그대로 성공으로 처리한다.
- **예외 흐름**
    - E1. 상품이 존재하지 않음 → 삭제할 수 없다.
- **후속 동작**
    - 삭제된 상품에 눌려 있던 좋아요는 사용자의 좋아요 목록에서 더 이상 보이지 않는다.
    - 이미 만들어진 주문 내역은 상품 삭제와 무관하게 그대로 보존된다.

### 4.3 좋아요

#### 시나리오

`29m` 회원은 마음에 드는 신발을 발견하면 좋아요를 눌러둔다 — 당장 사지 않더라도 나중에 다시 찾아보기 위해서다. 
좋아요 버튼은 토글이라, 이미 누른 신발을 다시 누르면 해제된다. 
마음이 바뀌거나 손이 미끄러져 같은 버튼을 연속으로 여러 번 눌러도, 시스템은 회원이 마지막에 의도한 상태만 정확히 한 번 반영한다.
따닥 클릭에 좋아요가 두 번 적용되거나 좋아요 수가 잘못 오르면 안 된다. 회원은 마이페이지에서 자신이 좋아요한 신발을 최신순으로 모아 보며, 이 목록은 본인만 볼 수 있다.

좋아요 수는 상품 인기순 정렬의 근거가 된다. 따라서 좋아요 수가 실제 좋아요와 어긋나지 않는 것이 중요하다.

#### 기능 목록

- 회원은 관심 있는 신발을 기록해두기 위해, 상품에 좋아요를 등록할 수 있다.
- 회원은 더 이상 관심 없는 신발을 정리하기 위해, 좋아요를 취소할 수 있다.
- 회원은 찜해둔 신발을 다시 찾아보기 위해, 자신이 좋아요한 상품 목록을 조회할 수 있다.

#### 핵심 규칙

- 한 회원은 하나의 상품에 좋아요를 최대 1개만 가진다. 같은 상품에 좋아요를 다시 눌러도 추가로 쌓이지 않는다.
- 좋아요 수는 여러 명이 동시에 누르거나 취소해도 정확해야 한다.
- 좋아요 목록은 본인의 것만 볼 수 있다.
- 삭제된 상품에는 좋아요할 수 없으며, 좋아요 목록에서도 보이지 않는다.

#### 유스케이스

**UC-L01 — 좋아요 등록**

- **정상 흐름**
    1. 회원이 특정 상품에 좋아요를 누른다.
    2. 아직 좋아요하지 않은 상품이면 좋아요를 등록한다.
- **대안 흐름**
    - A1. 이미 좋아요한 상품에 다시 누르면, 좋아요 수는 그대로 두고 성공으로 처리한다(같은 상품에 좋아요가 중복으로 쌓이지 않는다).
- **예외 흐름**
    - E1. 상품이 존재하지 않거나 이미 삭제됨 → 회원은 해당 상품에 좋아요할 수 없다.

**UC-L02 — 좋아요 취소**

- **정상 흐름**
    1. 회원이 좋아요한 상품의 좋아요를 취소한다.
    2. 좋아요를 해제한다.
- **대안 흐름**
    - A1. 좋아요하지 않은 상품에 취소를 요청하면, 좋아요 수는 그대로 두고 성공으로 처리한다.

**UC-L03 — 내 좋아요 목록 조회**

- **정상 흐름**
    1. 회원이 자신이 좋아요한 상품 목록을 요청한다.
    2. 본인이 좋아요한 상품을 최신순으로 보여준다.
- **대안 흐름**
    - A1. 좋아요한 상품이 없으면 빈 목록을 보여준다.
    - A2. 좋아요한 상품이 그사이 삭제되었으면 목록에서 제외한다.
- **예외 흐름**
    - E1. 다른 회원의 좋아요 목록을 보려고 했을 때의 응답 → 처리 방식 미정. → DEC-04

### 4.4 주문

#### 시나리오

`29m` 회원은 마음에 든 신발 여러 켤레를 한 번에 주문한다. 장바구니를 거치지 않고, 사고 싶은 신발과 수량을 골라 곧장 주문한다. 
주문하려는 신발 중 하나라도 재고가 모자라면 주문 전체가 실패한다 — 일부만 사지는 어중간한 주문은 없다. 
주문이 성사되면 그 시점의 상품명·가격·브랜드명이 주문에 그대로 보존된다. 나중에 그 신발의 가격이 바뀌거나 단종되어도, 회원이 보는 주문 내역은 살 때 그대로다.
회원은 마이페이지에서 기간을 정해 자신의 주문 내역을 찾아본다.

한편 `29m` 어드민은 전체 회원의 주문이 어떻게 들어오는지 파악해야 하므로, 전체 주문 목록과 개별 주문의 상세를 들여다본다.

#### 기능 목록

- 회원은 원하는 신발 여러 켤레를 한 번에 사기 위해, 상품과 수량을 지정해 주문을 생성할 수 있다.
- 회원은 과거 구매 이력을 확인하기 위해, 기간을 지정해 자신의 주문 목록을 조회할 수 있다.
- 회원은 특정 주문에서 무엇을 얼마에 샀는지 확인하기 위해, 주문 상세를 조회할 수 있다.
- 어드민은 전체 주문 현황을 파악하기 위해, 전체 주문 목록을 조회할 수 있다.
- 어드민은 개별 주문의 내용과 주문자를 확인하기 위해, 주문 상세를 조회할 수 있다.

#### 핵심 규칙

- 주문에는 최소 하나 이상의 상품이 포함되어야 한다.
- 주문 상품 중 하나라도 재고가 부족하면 주문 전체가 실패한다 — 일부만 주문되는 경우는 없다.
- 재고는 여러 명이 동시에 주문해도 0보다 작아지지 않는다.
- 주문 시점의 상품명·가격·브랜드명은 그대로 보존되며, 이후 원본 상품이 바뀌거나 삭제되어도 영향받지 않는다.
- 주문 총액은 주문한 각 상품의 (주문 시점 가격 × 수량)을 모두 더한 값이다.
- 회원은 본인의 주문만 조회할 수 있다. 어드민은 전체 주문을 조회할 수 있다.

#### 유스케이스

**UC-O01 — 주문 생성 (회원)**

- **정상 흐름**
    1. 회원이 사고 싶은 상품과 각 수량을 골라 주문을 요청한다.
    2. 모든 상품이 주문 가능하고 재고가 충분하면, 각 상품의 재고를 줄이고 주문을 생성한다.
    3. 주문에는 그 시점의 상품명·가격·브랜드명이 함께 보존되고, 주문 총액이 계산된다.
- **예외 흐름** — 아래 경우 주문 전체가 실패하며, 재고는 줄지 않는다.
    - E1. 상품을 하나도 고르지 않음 → 주문되지 않는다.
    - E2. 주문한 상품 중 존재하지 않거나 삭제된 것이 있음 → 회원은 그 상품을 살 수 없다.
    - E3. 재고가 부족한 상품이 있음 → 회원에게 어떤 상품이 품절인지 알려준다.
    - E4. 수량을 1개 미만으로 요청함 → 주문되지 않는다.
    - E5. 한 주문에 같은 상품을 중복해서 담았을 때의 처리 → 처리 방식 미정. → DEC-07
    - E6. 여러 명이 동시에 주문해 마지막 재고가 빠진 경우 → 늦게 도착한 주문은 품절로 실패한다.

**UC-O02 — 내 주문 목록 조회 (회원)**

- **정상 흐름**
    1. 회원이 조회 기간(시작일·종료일)을 정해 자신의 주문 목록을 요청한다.
    2. 해당 기간의 주문 목록을 보여준다 — 주문 번호·상태·총액·주문 일시 등 요약 정보를 담는다.
- **대안 흐름**
    - A1. 해당 기간에 주문이 없으면 빈 목록을 보여준다.
- **예외 흐름**
    - E1. 시작일이나 종료일을 지정하지 않음 → 주문 목록 조회는 기간을 반드시 지정해야 한다.
    - E2. 시작일이 종료일보다 늦음 → 조회되지 않는다.

**UC-O03 — 내 주문 상세 조회 (회원)**

- **정상 흐름**
    1. 회원이 특정 주문의 상세 내역을 요청한다.
    2. 주문 총액·주문 상품 내역(주문 시점 정보)·주문 일시를 보여준다.
- **예외 흐름**
    - E1. 주문이 존재하지 않음 → 회원은 해당 주문을 찾을 수 없다.
    - E2. 다른 회원의 주문을 보려고 했을 때의 응답 → 처리 방식 미정. → DEC-04

**UC-O04 — 전체 주문 목록 조회 (어드민)**

- **정상 흐름**
    1. 어드민이 전체 주문 목록을 요청한다.
    2. 전체 회원의 주문 목록을 보여준다 — 주문자 정보를 포함한 요약 정보를 담는다.
- **대안 흐름**
    - A1. 주문이 없으면 빈 목록을 보여준다.
    - A2. 목록은 페이지 단위로 나누어 보여준다 — 페이지 번호·페이지당 개수를 지정할 수 있고, 지정하지 않으면 기본값(0번 페이지, 20개)을 적용한다.

**UC-O05 — 주문 상세 조회 (어드민)**

- **정상 흐름**
    1. 어드민이 특정 주문의 상세 내역을 요청한다.
    2. 주문 내역·주문 상품 내역·주문자 정보를 보여준다.
- **예외 흐름**
    - E1. 주문이 존재하지 않음 → 어드민은 해당 주문을 찾을 수 없다.
> 주문은 만들어진 직후 상태로 남는다. 결제·취소 등 이후의 상태 변화는 이 문서의 범위 밖이며, 추후 확장한다.

아직 확정하지 않은 제품 판단 사항이다. 각 항목은 이후 기획 검토를 통해 확정한다.

|---|---|---|---|

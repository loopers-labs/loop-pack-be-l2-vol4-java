# 01. 요구사항 명세

> 본 문서는 이커머스 서비스를 **유저 시나리오 → 행위 중심 기능 목록 → 유스케이스 흐름 → 예외/조건/후속 동작** 순서로 정리한다.
> DDD 기반 구현을 위해 문서 내 용어는 유비쿼터스 언어를 기준으로 통일한다.

## 1. 서비스 개요

- **서비스 성격**: 회원, 브랜드, 상품, 좋아요, 쿠폰, 주문, 결제를 제공하는 이커머스 서비스
- **핵심 흐름**: 사용자는 회원가입 후 상품을 탐색하고, 좋아요와 쿠폰을 활용해 여러 상품을 한 번에 주문 및 결제한다.

## 2. 유비쿼터스 언어

| 용어                       | 정의                                                              |
|--------------------------|-----------------------------------------------------------------|
| 회원 (Member)              | 서비스를 이용하기 위해 가입한 사용자                                            |
| 비회원 (Guest)              | 가입하지 않은 사용자. 공개 브랜드 / 상품 조회만 가능하다.                              |
| 관리자 (Admin)              | 브랜드, 상품, 주문을 운영 목적으로 관리하는 사용자                                   |
| 브랜드 (Brand)              | 상품을 소유하거나 판매 주체로 노출되는 단위                                        |
| 상품 (Product)             | 사용자가 조회, 좋아요, 주문할 수 있는 판매 단위                                    |
| 재고 (Inventory)           | 상품과 1:1로 연결되어 독립된 생명주기를 가지는 주문 가능한 상품 재고 수량. 주문 생성 시 확인 및 차감된다. |
| 좋아요 (ProductLike)        | 회원이 관심 상품을 표시한 상태                                               |
| 쿠폰 (Coupon)              | 주문 금액에 할인을 적용할 수 있는 혜택                                          |
| 주문 (Order)               | 회원이 하나 이상의 상품과 수량을 확정한 구매 요청                                    |
| 주문 상품 (OrderItem)        | 주문에 포함된 개별 상품 라인                                                |
| 주문 상품 스냅샷 (OrderItemSnapshot) | 주문 당시 상품명, 브랜드명, 가격 등 주문 상품 이력 보존용 정보                           |
| 결제 (Payment)             | 주문 금액을 지불하고 주문 상태를 확정하는 행위                                      |
| 행동 기록 (UserActionLog)    | 회원가입, 조회, 좋아요, 주문, 결제 등 사용자의 주요 행동 데이터                          |

## 3. 사용자 및 권한

| 역할 | 가능 행위 | 식별 방식 |
| --- | --- | --- |
| 비회원 | 브랜드/상품 조회 | 없음 |
| 회원 | 내 정보 관리, 좋아요, 쿠폰, 주문, 결제, 내 주문 조회 | 로그인 정보 |
| 관리자 | 브랜드/상품 관리, 전체 주문 조회 | 관리자 식별 정보 |

- 인증/인가는 주요 스코프가 아니므로 별도 보안 체계로 구현하지 않습니다.
- 유저는 타 유저의 정보에 직접 접근할 수 없습니다.


## 4. 유저 시나리오 기반 기능 정의

### S1. 회원가입과 내 정보 관리

비회원은 회원가입 후 회원이 되며, 회원은 자신의 정보를 조회하고 비밀번호를 변경한다.

| 기능 ID | 행위 중심 기능 |
| --- | --- |
| F-Member-1 | 회원가입한다 |
| F-Member-2 | 내 정보를 조회한다 |
| F-Member-3 | 비밀번호를 변경한다 |

### S2. 브랜드 및 상품 탐색

사용자는 여러 브랜드와 상품을 둘러보고 구매 후보를 찾는다.

| 기능 ID | 행위 중심 기능 |
| --- | --- |
| F-Brand-1 | 브랜드 정보를 조회한다 |
| F-Product-1 | 상품 목록을 조회한다 |
| F-Product-2 | 상품 상세 정보를 조회한다 |

### S3. 상품 좋아요

회원은 관심 상품에 좋아요를 등록/취소하고, 좋아요 한 상품 목록을 확인한다.

| 기능 ID | 행위 중심 기능 |
| --- | --- |
| F-ProductLike-1 | 상품 좋아요를 등록한다 |
| F-ProductLike-2 | 상품 좋아요를 취소한다 |
| F-ProductLike-3 | 내가 좋아요 한 상품 목록을 조회한다 |

### S4. 쿠폰, 주문, 결제

회원은 쿠폰을 발급받고, 여러 상품을 한 번에 주문한 뒤 결제한다.

| 기능 ID | 행위 중심 기능 |
| --- | --- |
| F-Coupon-1 | 쿠폰을 발급받는다 |
| F-Order-1 | 여러 상품을 한 번에 주문한다 |
| F-Order-2 | 내 주문 목록을 조회한다 |
| F-Order-3 | 내 주문 상세를 조회한다 |
| F-Payment-1 | 주문을 결제한다 |

### S5. 관리자 브랜드/상품 관리

관리자는 브랜드와 상품을 등록, 조회, 수정, 삭제한다.

| 기능 ID | 행위 중심 기능 |
| --- | --- |
| F-Brand-2 | 브랜드 목록을 조회한다 |
| F-Brand-3 | 브랜드 상세를 조회한다 |
| F-Brand-4 | 브랜드를 등록한다 |
| F-Brand-5 | 브랜드 정보를 수정한다 |
| F-Brand-6 | 브랜드를 삭제한다 |
| F-Product-3 | 상품 목록을 조회한다 |
| F-Product-4 | 상품 상세를 조회한다 |
| F-Product-5 | 상품을 등록한다 |
| F-Product-6 | 상품 정보를 수정한다 |
| F-Product-7 | 상품을 삭제한다 |

### S6. 관리자 주문 조회

관리자는 전체 주문 목록과 단일 주문 상세를 조회한다.

| 기능 ID | 행위 중심 기능 |
| --- | --- |
| F-Order-4 | 주문 목록을 조회한다 |
| F-Order-5 | 주문 상세를 조회한다 |

## 5. 유스케이스 흐름

### UC1. 회원가입 및 내 정보 관리

| 구분 | 내용                                                                                    |
| --- |---------------------------------------------------------------------------------------|
| 기본 흐름 | 비회원이 회원가입 요청을 보낸다 → 시스템이 회원 정보를 저장한다 → 회원은 내 정보를 조회한다 → 회원은 현재 비밀번호 확인 후 새 비밀번호로 변경한다 |
| 예외/조건 분기 | 중복 식별자이면 가입 실패,<br/> 현재 비밀번호가 일치하지 않으면 변경 실패,<br/> 로그인 정보가 없으면 내 정보 조회 실패<br/>       |
| 후속 동작 | 회원가입 행동을 기록한다                                                                         |

### UC2. 브랜드 및 상품 탐색

| 구분 | 내용 |
| --- | --- |
| 기본 흐름 | 사용자가 브랜드 정보를 조회한다 → 상품 목록을 페이지 단위로 조회한다 → 상품 상세를 조회한다 |
| 예외/조건 분기 | 존재하지 않는 브랜드/상품이면 조회 실패, 판매 불가 상품은 사용자 목록에서 제외할 수 있다 |
| 후속 동작 | 브랜드/상품 조회 행동을 기록한다 |

### UC3. 상품 좋아요

| 구분 | 내용 |
| --- | --- |
| 기본 흐름 | 회원이 상품 좋아요를 등록한다 → 같은 상품에 대해 좋아요를 취소한다 → 내가 좋아요 한 상품 목록을 조회한다 |
| 예외/조건 분기 | 로그인하지 않으면 실패, 존재하지 않는 상품이면 실패, 중복 등록/취소는 멱등 처리한다 |
| 후속 동작 | 좋아요 등록/취소 행동을 기록한다 |

### UC4. 쿠폰 발급

| 구분 | 내용 |
| --- | --- |
| 기본 흐름 | 회원이 쿠폰 발급을 요청한다 → 시스템이 발급 가능 여부를 확인한다 → 회원에게 쿠폰을 발급한다 |
| 예외/조건 분기 | 발급 기간이 아니거나 중복 발급 불가 정책에 위배되면 실패한다 |
| 후속 동작 | 쿠폰 발급 행동을 기록한다 |

### UC5. 주문 생성

| 구분 | 내용 |
| --- | --- |
| 기본 흐름 | 회원이 상품 ID/수량 목록으로 주문한다 → 시스템이 상품 존재 여부와 판매 가능 상태를 확인한다 → 상품별 재고를 확인한다 → 재고를 차감한다 → 주문 당시 상품 정보를 스냅샷으로 저장한다 → 주문을 생성한다 |
| 예외/조건 분기 | 상품이 없거나 판매 불가이면 실패, 주문 수량이 유효하지 않으면 실패, 재고 데이터가 없거나 부족하면 실패, 쿠폰 사용 조건을 만족하지 않으면 실패 |
| 후속 동작 | 주문 생성 행동을 기록하고 결제 가능한 주문 상태로 전환한다 |

### UC6. 주문 조회

| 구분 | 내용 |
| --- | --- |

| 예외/조건 분기 | 타인의 주문은 조회할 수 없다, 존재하지 않는 주문이면 실패한다 |
| 후속 동작 | 주문 상세는 상품 원본이 아니라 주문 상품 스냅샷 기준으로 응답한다 |

| 구분 | 내용 |

| 예외/조건 분기 | 주문이 결제 가능한 상태가 아니면 실패, 결제 금액이 불일치하면 실패, 결제 실패 시 주문 상태를 결제 실패로 변경한다 |
| 후속 동작 | 결제 성공/실패 행동을 기록한다 |

### UC8. 관리자 브랜드/상품 관리

| --- | --- |

| 후속 동작 | 브랜드 삭제 시 해당 브랜드의 상품도 삭제한다. 상품에 연결된 재고는 주문 불가 상태로 전환한다. 이미 생성된 주문 상품 스냅샷은 변경하지 않는다 |

| 예외/조건 분기 | LDAP 인증이 없으면 실패, 존재하지 않는 주문이면 실패 |

## 6. API 요구사항

### 6.1 공통

API Prefix:
- 대고객 : `/api/v1`
  - 유저 로그인이 필요한 기능은 아래 헤더를 통해 유저를 식별해 제공합니다. 
  - 유저 식별 헤더: `X-Loopers-LoginId`, `X-Loopers-LoginPw`
- 관리자 : `/api-admin/v1`
  - 관리자 식별 헤더: `X-Loopers-Ldap: loopers.admin`

### 6.2 회원

| Method | URI | 인증 | 기능 |
| --- | --- | :---: | --- |
| POST | `/api/v1/users` | X | F-Member-1 |
| GET | `/api/v1/users/me` | O | F-Member-2 |
| PUT | `/api/v1/users/password` | O | F-Member-3 |

### 6.3 브랜드 및 상품

| Method | URI | 인증 | 기능 |
| --- | --- | :---: | --- |
| GET | `/api/v1/brands/{brandId}` | X | F-Brand-1 |
| GET | `/api/v1/products?page=0&size=20&brandId={brandId}` | X | F-Product-1 |
| GET | `/api/v1/products/{productId}` | X | F-Product-2 |

**상품 목록 조회 쿼리 파라미터**

| 파라미터 | 예시 | 설명 |
| --- | --- | --- |
| `brandId` | `1` | 특정 브랜드의 상품만 필터링 |
| `sort` | `latest` / `price_asc` / `likes_desc` | 정렬 기준 |
| `page` | `0` | 페이지 번호 (기본값 0) |
| `size` | `20` | 페이지당 상품 수 (기본값 20) |

- 정렬 기준은 선택 구현입니다.
- 필수 정렬 기준은 `latest`, 그 외는 `price_asc`, `likes_desc` 정도로 제한해도 충분합니다.

## 7. 핵심 정책

- 주문 생성 시 상품과 독립된 재고의 확인과 차감이 보장되어야 한다.
- 재고는 상품과 1:1로 연결되지만 상품 정보와 독립된 생명주기를 가진다.
- 재고는 음수가 될 수 없으며, 동시 주문에서도 초과 판매가 발생하지 않아야 한다.
- 주문 상품에는 당시 상품 정보가 스냅샷으로 저장되어야 한다.
- 삭제가 필요한 도메인 데이터는 논리 삭제 방식으로 처리한다.
- 브랜드 삭제 시 해당 브랜드의 상품도 함께 논리 삭제되어야 한다.
- 상품 또는 브랜드가 삭제되면 관련 재고도 주문 불가 상태로 전환한다.
- 상품 등록 시 브랜드는 이미 등록되어 있어야 한다.
- 상품 수정 시 브랜드는 변경할 수 없다.
- 좋아요는 회원과 상품의 조합으로 유일해야 한다.
- 유저 행동은 이후 확장 기능을 위해 기록 가능해야 한다.

## 8. 확인 필요 항목

- 쿠폰 발급 조건, 중복 발급, 만료, 할인 계산 정책
- 결제 연동 대상과 실패 시 재고/쿠폰 복구 정책
- 주문 재고 차감 동시성 제어 방식
- 행동 기록 저장소와 이벤트 스키마

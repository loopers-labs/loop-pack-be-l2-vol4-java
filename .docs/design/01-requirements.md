# 요구사항 명세

## 확정된 설계 결정

| 항목 | 결정 |
|---|---|
| 재고 예약 | `reserved_quantity` 컬럼. 주문 생성 시 예약 / 결제 확정 시 차감 |
| 결제 실패 | 즉시 실패 처리, 재고 예약 즉시 해제, 재시도 없음 |
| 결제 수단 | PG 결제 단일 (포인트 기능 제거) — `orders`는 `pg_amount`만 보유 |
| 복수 상품 주문 | 한 주문에 여러 상품 포함 가능. 일부 재고 부족 시 **전체 주문 거부**(원자적) |
| 옵션 | 없음. 재고는 **상품 단위**(상품당 재고 1건) |
| 배송지 | 주문 시 입력하며 `orders`에 **스냅샷으로 보존** |
| 좋아요 | 로그인 필수. `POST`(등록)/`DELETE`(취소) **분리**, 각각 멱등. `(user_id, product_id)` 유니크 |
| 브랜드 | 독립 엔티티. `name` UNIQUE(삭제 후에도 재등록 불가) |
| 트랜잭션 | 결제 확정 시 단일 트랜잭션 (재고 확정 + 주문 확정) |
| 만료 처리 | 스케줄러가 PENDING 주문 주기적 스캔 후 재고 예약 해제 |
| 고객 인증 | `X-Loopers-LoginId/LoginPw` 헤더 식별 + `userService.authenticate()` 검증. 본인 자원만 접근 |
| 인가(권한 정책) | 별도 권한 체계 없음. 본인 자원(주문/좋아요)에만 접근 가능하도록 소유권 검증 |
| 관리자 식별 | `X-Loopers-Ldap: loopers.admin` 헤더 일치 여부만 검증. 실제 LDAP 미연동 |
| 삭제 정책 | 브랜드/상품 = 소프트 딜리트(복구 가능). 재고는 상품에 종속돼 자동 처리. 좋아요 = 비동기 청크 물리 삭제(복구 제외) |
| 쿠폰 모델 | **쿠폰 템플릿**(`coupon_templates`, 관리자 관리)과 **발급 쿠폰**(`user_coupons`, 유저 소유) 분리. 템플릿 = 할인 규칙, 발급 쿠폰 = 유저별 사용 상태 |
| 쿠폰 타입 | 정액(`FIXED`) / 정률(`RATE`). 정률 할인액은 **버림(floor)**, 할인액은 **주문 금액을 초과할 수 없음**(cap) |
| 쿠폰명 | `coupon_templates.name` UNIQUE. 활성 중복 금지 + 삭제된 쿠폰명도 재등록 불가 (브랜드 정책 ⑧ 동일) |
| 쿠폰 발급 | 유저당 동일 템플릿 **1회만** 발급. `(user_id, template_id)` 유니크. 중복 발급 시 409 |
| 쿠폰 사용 시점 | **주문 생성 시 선점** — 주문 생성(PENDING) 시 즉시 `USED` + 할인 반영. 결제 실패/만료 시 `AVAILABLE`로 복구 (재고 reserve/release와 대칭) |
| 쿠폰 동시성 | 재사용 방지는 조건부 UPDATE(`SET status=USED WHERE id=? AND status=AVAILABLE`, affected=0이면 실패)로 Lost Update 방어 |
| 쿠폰 만료 | `EXPIRED`는 별도 저장 없이 조회/사용 시 `expired_at < now`로 **동적 판정**. 저장 status는 `AVAILABLE`/`USED`만 |

## API 공통 규약

### 대고객 API
- prefix: `/api/v1`
- 로그인이 필요한 기능은 아래 헤더로 유저를 식별하고 비밀번호를 검증한다. (`userService.authenticate()`)
  - `X-Loopers-LoginId` : 로그인 ID
  - `X-Loopers-LoginPw` : 비밀번호
- 별도의 권한(role) 체계는 두지 않되, 유저는 타 유저의 정보에 직접 접근할 수 없다. (본인 주문/좋아요 등만 접근 — 소유권 검증)
- 헤더 누락/검증 실패 시 **401**을 반환한다.

### 어드민 API
- prefix: `/api-admin/v1`
- 아래 헤더 값이 `loopers.admin`과 일치하는지만 검증한다. (실제 LDAP 인증은 미연동)
  - `X-Loopers-Ldap` : `loopers.admin`
- 헤더가 없거나 값이 `loopers.admin`이 아니면 **401**을 반환한다.

### 공통 에러 코드

`ErrorType` enum 기준. 도메인 서비스는 `CoreException(ErrorType)`을 던지고 `ApiControllerAdvice`가 HTTP 응답으로 변환한다.

| HTTP | ErrorType | 발생 상황 |
|---|---|---|
| 400 | BAD_REQUEST | 잘못된 요청 형식, 상품 수정 시 브랜드 변경 시도, 미등록 브랜드로 상품 등록 |
| 401 | UNAUTHORIZED | 고객 헤더 누락/검증 실패, 관리자 헤더 불일치, 비로그인 좋아요 |
| 404 | NOT_FOUND | 존재하지 않거나 삭제된 브랜드/상품/주문/쿠폰 템플릿, 타 유저 주문·좋아요 목록·쿠폰 접근 |
| 409 | CONFLICT | 재고 부족(주문 전체 거부), 중복 브랜드명 등록, 쿠폰 중복 발급, 이미 사용/만료된 쿠폰으로 주문, 최소 주문 금액 미달 쿠폰 적용 |
| 500 | INTERNAL_ERROR | 예기치 못한 서버 오류 |

### 응답 정보 정책 (고객 vs 어드민)

URL이 분리(`/api/v1` vs `/api-admin/v1`)되어 있으므로, **엔드포인트별로 응답에 담는 정보를 다르게** 정의한다. (스펙: "고객과 어드민에게 제공되어야 할 정보를 고민")

- **대고객 응답(`/api/v1`)**: 구매 판단에 필요한 정보만 노출한다.
  - 상품: 상품명, 가격, 설명, 브랜드명, 좋아요 수, **품절 여부(stockStatus)**
  - 브랜드: 브랜드명, 설명
- **어드민 응답(`/api-admin/v1`)**: 관리에 필요한 정보까지 노출한다.
  - 상품: 위 정보 + **정확한 재고 수량(stockQuantity)**, 삭제 상태(deleted), 생성/수정 시각
  - 브랜드: 위 정보 + 삭제 상태, 생성/수정 시각
- **대고객에 노출하지 않는 정보**: 정확한 재고 수량, 삭제 상태, 내부 타임스탬프 (경쟁 노출·재고 심리 등 고려)
- 세밀한 DTO 필드 스키마는 구현 단계에서 이 정책을 근거로 정의한다.

## 도메인별 유저 시나리오 및 기능 정의

### 1. 유저

**액터**: 비로그인/로그인 회원

**시나리오**
1. 비로그인 사용자가 회원가입한다.
2. 로그인한 사용자가 자신의 정보를 조회한다.
3. 로그인한 사용자가 비밀번호를 변경한다.

**기능 요구사항**

| 기능 | 엔드포인트 | 인증 | 설명 |
|---|---|---|---|
| 회원가입 | `POST /api/v1/users` | X | 신규 회원 등록 |
| 내 정보 조회 | `GET /api/v1/users/me` | O | 인증된 본인 정보 조회 |
| 비밀번호 변경 | `PUT /api/v1/users/password` | O | 인증된 본인 비밀번호 변경 |

**정책**
- `login_id`는 영문/숫자 1~10자, 유니크. 중복 시 거부.
- 비밀번호는 BCrypt로 암호화 저장.
- `내 정보 조회`/`비밀번호 변경`은 인증된 본인에 대해서만 동작한다.

---

### 2. 상품 탐색 (브랜드 / 상품)

**액터**: 비로그인/로그인 회원

**시나리오**
1. 사용자가 상품 목록을 조회한다.
2. 관심 상품을 클릭해 상세 정보를 확인한다.
3. 상세 페이지에서 브랜드명을 클릭해 브랜드 정보를 조회한다.

**기능 요구사항**

| 기능 | 엔드포인트 | 인증 | 설명 |
|---|---|---|---|
| 상품 목록 조회 | `GET /api/v1/products` | X | 상품 목록 (필터·정렬·페이지네이션) |
| 상품 상세 조회 | `GET /api/v1/products/{productId}` | X | 상품 정보, 품절 여부, 브랜드 포함 |
| 브랜드 조회 | `GET /api/v1/brands/{brandId}` | X | 브랜드 정보 반환 |

**상품 목록 조회 쿼리 파라미터**

| 파라미터 | 예시 | 설명 |
|---|---|---|
| brandId | 1 | 특정 브랜드의 상품만 필터링 |
| sort | latest / price_asc / likes_desc | 정렬 기준 (기본 latest) |
| page | 0 | 페이지 번호 (기본값 0) |
| size | 20 | 페이지당 상품 수 (기본값 20) |

- 정렬 기준 중 `latest`만 필수 구현, `price_asc`·`likes_desc`는 선택 구현.

**정책**
- 재고가 0인 상품도 목록/상세에 노출한다. (품절 표시)
- 가용 재고 = `total_quantity - reserved_quantity`
- 판매 가능 = 가용 재고 > 0, **품절 = 가용 재고 == 0** (응답의 `stockStatus`가 이 값을 표현)
- 브랜드는 독립 엔티티로 상품과 별도 조회 가능
- 소프트 딜리트된 브랜드/상품은 **대고객 조회 결과에서 제외**한다.

### 3. 상품 좋아요

**액터**: 로그인 회원

**시나리오**
1. 사용자가 상품 상세 페이지에서 좋아요를 등록한다.
2. 다시 누르면 좋아요를 취소한다.
3. 내가 좋아요한 상품 목록을 조회한다.

**기능 요구사항**

| 기능 | 엔드포인트 | 인증 | 설명 |
|---|---|---|---|
| 좋아요 등록 | `POST /api/v1/products/{productId}/likes` | O | 좋아요 등록 (멱등) |
| 좋아요 취소 | `DELETE /api/v1/products/{productId}/likes` | O | 좋아요 취소 (멱등) |
| 좋아요 목록 조회 | `GET /api/v1/users/{userId}/likes` | O | 본인이 좋아요한 상품 목록 |

**정책**
- 비로그인 사용자는 좋아요 불가 → 401 반환
- `(user_id, product_id)` 유니크 제약으로 DB 레벨 중복 방어
- 멱등: 이미 등록된 상태에서 `POST` 반복해도 등록 상태 유지, 이미 취소된 상태에서 `DELETE` 반복해도 취소 상태 유지
- 소프트 딜리트된 상품에는 좋아요를 등록할 수 없다.
- 좋아요 목록 조회는 **본인 것만** 가능하다. 경로의 `{userId}`가 인증된 유저와 다르면 **404로 거부**한다(주문 소유권과 동일하게 존재 여부 비노출).

### 4. 주문 및 결제

**액터**: 로그인 회원, 외부 PG

**시나리오 — STEP 1: 주문 생성**
1. 사용자가 장바구니에 담은 **여러 상품**과 배송지를 담아 주문한다.
2. 시스템이 주문에 포함된 모든 상품의 재고를 예약하고 주문을 PENDING 상태로 생성한다.
3. PG 결제창 URL을 반환받아 결제창으로 이동한다.

**시나리오 — STEP 2: 결제 확정**
1. 사용자가 결제창에서 카드 정보를 입력하고 결제를 완료한다.
2. PG가 우리 서버로 결제 결과를 콜백으로 전송한다.
3. 결제 성공 + PG 승인 금액이 주문 금액과 일치 시: 재고 확정 + 주문 CONFIRMED
4. 결제 실패 또는 금액 불일치 시: 재고 예약 즉시 해제 + 주문 FAILED

**기능 요구사항**

| 기능 | 엔드포인트 | 인증 | 설명 |
|---|---|---|---|
| 주문 요청 | `POST /api/v1/orders` | O | 여러 상품 재고 예약, 주문 PENDING 생성, PG 초기화 |
| 주문 목록 조회 | `GET /api/v1/orders?startAt=2026-01-31&endAt=2026-02-10&page=0&size=20` | O | 본인 주문 목록 (기간 필터 + 페이지네이션) |
| 주문 상세 조회 | `GET /api/v1/orders/{orderId}` | O | 본인 단일 주문 상세 |
| 결제 확정 | `POST /api/v1/payments/confirm` | (PG 콜백) | PG 콜백 수신, 결제 검증, 주문 확정 |
| 주문 취소 | `DELETE /api/v1/orders/{orderId}` | O | CONFIRMED 주문 취소, PG 취소 + 재고 복구 |

**주문 목록 조회 쿼리 파라미터**

| 파라미터 | 예시 | 설명 |
|---|---|---|
| startAt | 2026-01-31 | 조회 시작일 (기간 필터) |
| endAt | 2026-02-10 | 조회 종료일 (기간 필터) |
| page | 0 | 페이지 번호 (기본값 0) |
| size | 20 | 페이지당 주문 수 (기본값 20) |

**주문 요청 예시**
```json
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ],
  "couponId": 42,
  "receiverName": "홍길동",
  "receiverPhone": "010-0000-0000",
  "zipCode": "12345",
  "address": "서울시 ...",
  "detailAddress": "101동 1001호"
}
```
- `couponId`는 발급 쿠폰(`user_coupons`)의 ID이며 **미적용 시 생략 가능(NULLABLE)**. 주문 1건당 1장만 적용한다.

**정책**

| 상황 | 처리 |
|---|---|
| 일부 상품 재고 부족 | **주문 전체 거부** → 409 반환 (예약된 재고도 모두 롤백) |
| 결제 성공 + 금액 일치 | 모든 상품 재고 확정(total -qty, reserved -qty) + 주문 CONFIRMED |
| 결제 성공 + 금액 불일치 | **결제 실패와 동일 처리** (재고 예약 해제 + 주문 FAILED) |
| 결제 실패 | 모든 상품 재고 예약 해제(reserved -qty) + 주문 FAILED, 재시도 없음 |
| PG 콜백 미수신 | 스케줄러가 생성 후 **15분 초과** PENDING 주문 스캔 후 예약 해제 |

**쿠폰 적용 정책**
- 쿠폰은 주문 1건당 **1장만** 적용 가능하다. (`couponId` 미지정 시 미적용)
- 다음 중 하나라도 해당하면 **주문 전체를 실패**시킨다. (부분 적용 없음)
  - 존재하지 않는 쿠폰 / 타 유저 소유 쿠폰 → 404
  - 이미 사용된(`USED`) 쿠폰 / 만료된(`expired_at < now`) 쿠폰 → 409
  - 쿠폰의 최소 주문 금액(`min_order_amount`) 미달 → 409
- 할인액 계산
  - `FIXED`: 할인액 = `value`(원).
  - `RATE`: 할인액 = `floor(원금 × value / 100)` — 소수점 버림.
  - 공통: 할인액은 **원금(`original_amount`)을 초과할 수 없다.** 초과 시 원금까지만 할인(최종 금액 0원 하한).
- 주문 성공(생성) 시 해당 쿠폰을 **즉시 `USED`로 전이**하고 `order_id`를 기록한다. 재사용 불가.
- 결제 실패 / 금액 불일치 / PENDING 만료 시 쿠폰을 **`AVAILABLE`로 복구**한다. (재고 예약 해제와 동일 시점)

**금액 정합성 정책**
- 쿠폰 미적용 시: `orders.original_amount = orders.pg_amount = payments.amount = Σ(order_items.price × quantity)`, `discount_amount = 0`.
- 쿠폰 적용 시: `orders.original_amount = Σ(order_items.price × quantity)`, `orders.pg_amount = original_amount - discount_amount = payments.amount`.
- 주문 스냅샷은 **쿠폰 적용 전 금액(`original_amount`), 할인 금액(`discount_amount`), 최종 결제 금액(`pg_amount`)** 을 모두 보존한다.
- 결제 확정 시 PG 승인 금액과 `orders.pg_amount`(최종 결제 금액)를 비교해, 불일치면 결제를 성공으로 인정하지 않고 실패 처리한다. (금액 위·변조/계산 버그 방어)

**결제 콜백 보안·멱등성**
- **콜백 서명 검증**: `/payments/confirm`은 외부에서 호출되는 웹훅이므로, 공유 시크릿 기반 서명(HMAC 등)으로 진짜 PG가 보낸 콜백인지 먼저 검증한다. 위조 콜백은 401로 차단한다. (PG 재조회가 2차 방어)
- **멱등 처리**: PG는 콜백을 중복 전송할 수 있다. 결제 확정은 주문이 `PENDING`일 때만 처리하고, 이미 `CONFIRMED`/`FAILED`면 재고·상태 변경 없이 `200 OK`로 수렴한다. (재고 이중 차감 방지)

**콜백 진위 검증**
- `/payments/confirm`은 외부 PG가 호출하는 웹훅이다. 공유 시크릿 기반 서명(HMAC 등)으로 콜백 진위를 먼저 검증하고, 불일치 시 401로 거부한다. (위조 콜백으로 인한 임의 주문 확정 방지)
- PG 재조회(`verifyPayment`)로 결제 실재·금액을 2차 확인한다.

**복수 상품 주문 정책**
- 한 주문은 1개 이상의 주문 라인(order_items)을 가진다.
- 재고 예약은 주문에 포함된 모든 상품에 대해 수행한다.
- 하나라도 재고가 부족하면 **이미 예약한 항목까지 모두 해제하고 주문 전체를 거부**한다. (부분 주문 없음)
- 각 주문 라인은 주문 시점의 상품명/브랜드명/단가/수량을 스냅샷으로 보관한다.
- 배송지도 주문 시점 스냅샷으로 보존한다. (이후 사용자가 주소를 바꿔도 과거 주문은 당시 주소 유지)

**소유권 정책**
- 주문 조회/취소는 **본인 주문에만** 가능하다. 인증 유저의 `userId`와 주문의 `userId`가 다르면 404로 응답해 존재 여부를 노출하지 않는다.

**주문 상태 전이**

```
PENDING → CONFIRMED   (결제 성공 + 금액 일치)
PENDING → FAILED      (결제 실패 / 금액 불일치 / PG 콜백 미수신 후 만료)
CONFIRMED → CANCELLED (사용자 주문 취소)
```

### 5. 관리자 — 브랜드 관리

**액터**: 관리자 (`X-Loopers-Ldap: loopers.admin` 헤더 보유자)

**시나리오**
1. 관리자가 `X-Loopers-Ldap` 헤더를 담아 관리자 API에 접근한다.
2. 브랜드를 등록/조회/수정/삭제한다.
3. 브랜드 삭제 시 해당 브랜드의 상품도 함께 삭제(소프트 딜리트)된다.

**기능 요구사항**

| 기능 | 엔드포인트 | LDAP | 설명 |
|---|---|---|---|
| 브랜드 목록 조회 | `GET /api-admin/v1/brands?page=0&size=20` | O | 등록된 브랜드 목록 (페이지네이션) |
| 브랜드 상세 조회 | `GET /api-admin/v1/brands/{brandId}` | O | 브랜드 상세 조회 |
| 브랜드 등록 | `POST /api-admin/v1/brands` | O | 브랜드 등록 |
| 브랜드 수정 | `PUT /api-admin/v1/brands/{brandId}` | O | 브랜드 정보 수정 |
| 브랜드 삭제 | `DELETE /api-admin/v1/brands/{brandId}` | O | 브랜드 삭제 시 소속 상품도 함께 삭제 |

**정책**
- 모든 관리자 API는 `X-Loopers-Ldap: loopers.admin` 헤더로 식별한다. 헤더 누락/불일치 시 401 반환.
- 브랜드명(`name`)은 UNIQUE. 활성 중복 금지 + 삭제된 브랜드명도 재등록 불가(브랜드명은 신뢰 자산).
- 브랜드/상품 삭제는 소프트 딜리트(`deleted_at` 기록)로 즉시 노출을 차단한다. (복구 가능)
- 브랜드 삭제 시 소속 상품을 함께 소프트 딜리트한다. (동기 트랜잭션, 저용량)
- 이미 주문에 포함된 상품도 삭제되지만, 과거 주문 라인(order_items)은 스냅샷이므로 영향받지 않는다.
- **종속 데이터 처리 원칙**
  - 재고(stocks): 별도 처리 안 함. 상품을 통해서만 도달하므로 상품 삭제 시 자동 숨김, 복구 시 자동 복구.
  - 좋아요(likes): 고용량이고 복구 가치가 없으므로 **비동기 배치가 청크 단위로 물리 삭제**한다. (`products.likes_purged`로 정리 완료 마킹)
- 관리자 목록/상세 조회는 **소프트 딜리트된 브랜드도 포함**해 노출한다. (대고객은 제외)

### 6. 관리자 — 상품 관리

**액터**: 관리자 (`X-Loopers-Ldap: loopers.admin` 헤더 보유자)

**시나리오**
1. 관리자가 사전 등록된 브랜드를 선택해 상품을 등록한다. (재고 수량 포함)
2. 상품을 조회/수정/삭제한다.
3. 상품 수정 시 브랜드는 변경할 수 없다.

**기능 요구사항**

| 기능 | 엔드포인트 | LDAP | 설명 |
|---|---|---|---|
| 상품 목록 조회 | `GET /api-admin/v1/products?page=0&size=20&brandId={brandId}` | O | 상품 목록 (브랜드 필터, 페이지네이션) |
| 상품 상세 조회 | `GET /api-admin/v1/products/{productId}` | O | 상품 상세 (재고 수량 포함) |
| 상품 등록 | `POST /api-admin/v1/products` | O | 상품 + 재고 등록. 브랜드 사전 등록 필수 |
| 상품 수정 | `PUT /api-admin/v1/products/{productId}` | O | 상품 정보·재고 수정. 브랜드 변경 불가 |
| 상품 삭제 | `DELETE /api-admin/v1/products/{productId}` | O | 상품 삭제 (소프트 딜리트) |

**정책**
- 상품 등록 시 브랜드는 **반드시 사전 등록된 상태**여야 한다. 미등록 브랜드면 거부.
- 상품 등록/수정은 **상품 기본정보(상품명·가격·설명) + 재고 수량(stocks)** 을 함께 받는다. (옵션 개념 없음, 상품당 재고 1건)
- 상품 수정 시 **브랜드(brand_id)는 변경할 수 없다.** 변경 요청 시 거부. (데이터 정합성 문제 유발)
- **재고 수정 정책**
  - 관리자는 `total_quantity`(실물 보유량)만 수정한다. `reserved_quantity`는 주문이 만들어내는 시스템 값이라 직접 수정 불가.
  - 새 `total_quantity`는 **현재 `reserved_quantity` 이상**일 때만 반영한다. 미만이면 거부(409).
    - 불변식 `total_quantity >= reserved_quantity`(가용 ≥ 0)를 보장하기 위함.
    - 조건부 UPDATE(`WHERE :newTotal >= reserved_quantity`)로 원자적 검증.
- 삭제는 소프트 딜리트(`deleted_at` 기록).
- 관리자 목록/상세 조회는 **소프트 딜리트된 상품도 포함**해 노출한다. (대고객은 제외 — 관리 목적상 삭제 이력 확인 필요)

### 8. 쿠폰 (발급 / 조회)

**액터**: 로그인 회원

**시나리오**
1. 사용자가 발급 가능한 쿠폰을 발급받는다.
2. 내가 보유한 쿠폰 목록을 사용 가능/사용 완료/만료 상태와 함께 조회한다.
3. 주문 시 보유 쿠폰을 적용해 할인받는다. (주문 흐름은 섹션 4 참조)

**기능 요구사항**

| 기능 | 엔드포인트 | 인증 | 설명 |
|---|---|---|---|
| 쿠폰 발급 요청 | `POST /api/v1/coupons/{couponId}/issue` | O | 쿠폰 템플릿(`{couponId}`)을 본인에게 발급 |
| 내 쿠폰 목록 조회 | `GET /api/v1/users/me/coupons` | O | 본인 발급 쿠폰 목록 (상태 포함) |

**정책**
- `POST /coupons/{couponId}/issue`의 `{couponId}`는 **쿠폰 템플릿 ID**다. 존재하지 않거나 삭제된 템플릿이면 404.
- 유저당 동일 템플릿은 **1회만** 발급한다. 이미 발급받았으면 409. (`(user_id, template_id)` 유니크)
- 목록 조회는 본인 발급 쿠폰만 반환하며, 각 쿠폰의 상태(`AVAILABLE` / `USED` / `EXPIRED`)를 함께 반환한다.
  - `EXPIRED`는 저장 값이 아니라 조회 시점 `expired_at < now`로 동적 판정한다. (저장 status는 `AVAILABLE`/`USED`)
- 발급 시점에 템플릿이 이미 만료(`expired_at < now`)되었으면 발급을 거부한다. (409)

### 9. 관리자 — 쿠폰 관리

**액터**: 관리자 (`X-Loopers-Ldap: loopers.admin` 헤더 보유자)

**시나리오**
1. 관리자가 쿠폰 템플릿을 등록한다. (정액/정률, 할인값, 최소 주문 금액, 만료일시)
2. 템플릿을 조회/수정/삭제한다.
3. 특정 템플릿의 발급 내역을 조회한다.

**기능 요구사항**

| 기능 | 엔드포인트 | LDAP | 설명 |
|---|---|---|---|
| 쿠폰 템플릿 목록 조회 | `GET /api-admin/v1/coupons?page=0&size=20` | O | 등록된 쿠폰 템플릿 목록 (페이지네이션) |
| 쿠폰 템플릿 상세 조회 | `GET /api-admin/v1/coupons/{couponId}` | O | 템플릿 상세 |
| 쿠폰 템플릿 등록 | `POST /api-admin/v1/coupons` | O | 정액(`FIXED`)/정률(`RATE`) 타입 지정 등록 |
| 쿠폰 템플릿 수정 | `PUT /api-admin/v1/coupons/{couponId}` | O | 템플릿 정보 수정 |
| 쿠폰 템플릿 삭제 | `DELETE /api-admin/v1/coupons/{couponId}` | O | 템플릿 삭제 (소프트 딜리트) |
| 발급 내역 조회 | `GET /api-admin/v1/coupons/{couponId}/issues?page=0&size=20` | O | 특정 템플릿의 발급 내역 |

**쿠폰 템플릿 등록 요청 예시**
```json
{
  "name": "신규가입 10% 할인",
  "type": "RATE",
  "value": 10,
  "minOrderAmount": 10000,
  "expiredAt": "2026-12-31T23:59:59"
}
```

**정책**
- 모든 관리자 API는 `X-Loopers-Ldap: loopers.admin` 헤더로 식별한다. 누락/불일치 시 401.
- `type`은 `FIXED`(정액, `value`=할인 금액 원) / `RATE`(정률, `value`=할인율 %) 중 하나.
  - `RATE`의 `value`는 1~100 범위. `FIXED`의 `value`는 0보다 큰 금액.
- `minOrderAmount`(최소 주문 금액)는 선택값(NULL 허용). 적용 시 주문 원금이 이 값 이상이어야 쿠폰 사용 가능.
- `expiredAt`(만료일시)은 필수. 이 시각 이후 발급 쿠폰은 만료 처리된다.
- 템플릿 삭제는 소프트 딜리트(`deleted_at` 기록). 이미 발급된 쿠폰(`user_coupons`)은 영향받지 않는다(발급 시점 스냅샷).
- 발급 내역 조회는 해당 템플릿으로 발급된 `user_coupons` 목록(유저, 상태, 발급/사용 시각)을 반환한다.

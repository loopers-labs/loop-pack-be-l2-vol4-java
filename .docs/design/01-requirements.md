# 요구사항 명세

## 확정된 설계 결정

| 항목 | 결정 |
|---|---|
| 재고 예약 | reserved_quantity 컬럼, 주문 생성 시 예약 / 결제 확정 시 차감 |
| 결제 실패 | 즉시 실패 처리, 재고 예약 즉시 해제, 재시도 없음 |
| 포인트 | users.point 잔액 컬럼 + point_histories 이력, atomic UPDATE |
| 좋아요 | 로그인 필수, (user_id, product_id) 유니크 제약 |
| 브랜드 | 독립 엔티티로 분리 |
| 트랜잭션 | 결제 확정 시 단일 트랜잭션 (재고 확정 + 포인트 차감 + 주문 확정) |
| 만료 처리 | 스케줄러가 PENDING 주문 주기적 스캔 후 재고 예약 해제 |

---

## 도메인별 유저 시나리오 및 기능 정의

### 1. 상품 탐색

**액터**: 비로그인/로그인 회원

**시나리오**
1. 사용자가 상품 목록을 조회한다.
2. 관심 상품을 클릭해 상세 정보를 확인한다.
3. 상세 페이지에서 브랜드명을 클릭해 브랜드 정보를 조회한다.

**기능 요구사항**

| 기능 | 엔드포인트 | 설명 |
|---|---|---|
| 상품 목록 조회 | `GET /api/v1/products` | 전체 상품 목록 반환 |
| 상품 상세 조회 | `GET /api/v1/products/{id}` | 상품 정보, 재고 여부, 브랜드 포함 |
| 브랜드 조회 | `GET /api/v1/brands/{id}` | 브랜드 정보 반환 |

**정책**
- 재고가 0인 상품도 목록/상세에 노출한다. (품절 표시)
- 가용 재고 = `total_quantity - reserved_quantity`
- 브랜드는 독립 엔티티로 상품과 별도 조회 가능

---

### 2. 상품 좋아요

**액터**: 로그인 회원

**시나리오**
1. 사용자가 상품 상세 페이지에서 좋아요 버튼을 누른다.
2. 좋아요가 없으면 등록, 이미 있으면 취소된다.
3. 같은 요청을 여러 번 보내도 결과가 수렴한다. (멱등)

**기능 요구사항**

| 기능 | 엔드포인트 | 설명 |
|---|---|---|
| 좋아요 등록/취소 | `POST /api/v1/products/{id}/likes` | 토글 방식, 200 OK + liked: true/false |

**정책**
- 비로그인 사용자는 좋아요 불가 → 401 반환
- `(user_id, product_id)` 유니크 제약으로 DB 레벨 중복 방어
- 동일 요청 반복 시 결과가 동일하게 수렴 (멱등 보장)

---

### 3. 주문 및 결제

**액터**: 로그인 회원, 외부 PG

**시나리오 — STEP 1: 주문 생성**
1. 사용자가 상품 상세에서 주문하기 버튼을 누른다.
2. 시스템이 재고를 예약하고 주문을 PENDING 상태로 생성한다.
3. PG 결제창 URL을 반환받아 결제창으로 이동한다.

**시나리오 — STEP 2: 결제 확정**
1. 사용자가 결제창에서 카드 정보를 입력하고 결제를 완료한다.
2. PG가 우리 서버로 결제 결과를 콜백으로 전송한다.
3. 결제 성공 시: 재고 확정 + 포인트 차감 + 주문 CONFIRMED
4. 결제 실패 시: 재고 예약 즉시 해제 + 주문 FAILED

**기능 요구사항**

| 기능 | 엔드포인트 | 설명 |
|---|---|---|
| 주문 생성 | `POST /api/v1/orders` | 재고 예약, 주문 PENDING 생성, PG 초기화 |
| 결제 확정 | `POST /api/v1/payments/confirm` | PG 콜백 수신, 결제 검증, 주문 확정 |

**정책**

| 상황 | 처리 |
|---|---|
| 재고 부족 | 주문 생성 거부 → 409 반환 |
| 결제 성공 | 재고 확정(total -1, reserved -1) + 포인트 차감 + 주문 CONFIRMED |
| 결제 실패 | 재고 예약 해제(reserved -1) + 주문 FAILED, 재시도 없음 |
| PG 콜백 미수신 | 스케줄러가 만료된 PENDING 주문 스캔 후 예약 해제 |
| 포인트 부족 | 포인트 차감 거부 → 결제 실패로 처리 |

**포인트 정책**
- 포인트 단독 또는 포인트 + 외부 결제 혼합 사용 가능
- 포인트 차감: `UPDATE users SET point = point - ? WHERE user_id = ? AND point >= ?`
- affected rows = 0 이면 포인트 부족 예외
- 차감 이력은 `point_histories` 테이블에 기록

**주문 상태 전이**

```
PENDING → CONFIRMED  (결제 성공)
PENDING → FAILED     (결제 실패 / PG 콜백 미수신 후 만료)
```

# 01. Requirements — 유저 시나리오 기반 기능 정의 및 요구사항 명세

---

## 범위

회원가입 / 내 정보 조회 / 비밀번호 변경은 이전에 완료. 아래부터 다룬다.

---

## 유저 시나리오

> 시나리오 1~3(상품·브랜드 탐색)은 비회원도 접근 가능하다.
> 시나리오 4~6(좋아요·주문)은 로그인 완료 상태를 전제로 한다.

---

### 👤 고객

#### 시나리오 1. 고객이 초기 화면에서 상품을 탐색한다

고객이 서비스에 처음 진입하면 상품 목록이 바로 보인다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/products` | X | 상품 목록 조회 |

**정렬**
- 기본 정렬은 최신순(`latest`)
- 고객은 정렬 기준을 변경할 수 있다: 최신순(`latest`) / 좋아요 많은 순(`likes_desc`) / 가격 낮은 순(`price_asc`)
- `likes_desc` 정렬에서 좋아요 수가 같은 경우 최신순으로 정렬한다
- 정렬 기준을 변경하면 목록이 즉시 갱신된다

**필터**
- 특정 브랜드를 선택하면 해당 브랜드의 상품만 표시된다
- 브랜드 필터와 정렬 기준은 함께 적용될 수 있다

**목록 항목**
- 각 상품 카드에는 상품명, 가격, 브랜드명, 좋아요 수가 표시된다
- 삭제된 상품은 목록에 노출되지 않는다
- 재고가 0인 상품은 목록에 노출된다

**페이지네이션**
- 한 페이지에 기본 20개 노출
- 더 많은 상품을 보려면 다음 페이지로 이동할 수 있다

---

#### 시나리오 2. 고객이 상품 상세를 조회한다

목록에서 상품을 선택하면 상세 정보를 볼 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/products/{productId}` | X | 상품 상세 조회 |

**표시 정보**
- 상품명, 가격, 재고, 설명, 브랜드명, 좋아요 수

**예외**
- 존재하지 않거나 삭제된 상품 접근 → 404

---

#### 시나리오 3. 고객이 브랜드 정보를 조회한다

상품 상세에서 브랜드명을 클릭하면 해당 브랜드 정보를 볼 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/brands/{brandId}` | X | 브랜드 정보 조회 |

**표시 정보**
- 브랜드명, 브랜드 설명 등 브랜드가 가진 기본 정보

**예외**
- 존재하지 않거나 삭제된 브랜드 접근 → 404

---

#### 시나리오 4. 고객이 상품에 좋아요를 누른다

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| POST | `/api/v1/products/{productId}/likes` | O | 좋아요 등록 |
| DELETE | `/api/v1/products/{productId}/likes` | O | 좋아요 취소 |
| GET | `/api/v1/users/{userId}/likes` | O | 내 좋아요 목록 조회 |

**등록**
- 상품 상세에서 좋아요를 누르면 즉각 반영된다
- 같은 상품에 좋아요를 두 번 누를 수 없다 (중복 불가)

**취소**
- 이미 누른 좋아요를 취소할 수 있다
- 좋아요하지 않은 상품을 취소하려 하면 허용하지 않는다

**내 좋아요 목록 조회**
- 내가 좋아요한 상품 목록을 볼 수 있다

**예외**
- 존재하지 않는 상품에 좋아요 시도 → 404
- 이미 좋아요한 상품에 재등록 시도 → 409
- 좋아요하지 않은 상품 취소 시도 → 404

---

#### 시나리오 5. 고객이 주문을 요청한다

로그인한 고객이 여러 상품을 한 번에 주문할 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| POST | `/api/v1/orders` | O | 주문 요청 |

**주문 요청**
- 상품과 수량을 지정해 주문한다
- 주문 시점에 재고가 충분해야 주문이 완료된다
- 재고가 충분하면 해당 수량만큼 즉시 차감된다
- 주문 정보에는 당시의 상품명·가격이 그대로 기록된다
  (이후 상품 정보가 바뀌어도 주문 기록은 바뀌지 않는다)

**예외**
- 주문 항목이 비어있으면 주문할 수 없다 → 400
- 재고가 부족한 상품이 포함된 경우 주문 전체가 실패한다 → 400
- 존재하지 않는 상품이 포함된 경우 주문 전체가 실패한다 → 404

---

#### 시나리오 6. 고객이 주문 내역을 조회한다

로그인한 고객이 자신의 주문 이력을 확인할 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/orders?startAt=2026-01-31&endAt=2026-02-10` | O | 주문 목록 조회 |
| GET | `/api/v1/orders/{orderId}` | O | 주문 상세 조회 |

**목록 조회**
- 조회 기간(시작일 ~ 종료일)을 지정해 주문 목록을 볼 수 있다
- 본인의 주문만 조회된다

**상세 조회**
- 단일 주문의 상세 내역(주문 항목, 수량, 당시 가격 등)을 볼 수 있다
- 본인의 주문만 상세 조회할 수 있다

**예외**
- 타인의 주문 상세 접근 → 403
- 존재하지 않는 주문 접근 → 404

### 🔧 어드민

### 역할 정의

| 역할 | 설명 | 브랜드 귀속 |
|------|------|------------|
| `CUSTOMER` | 일반 고객 | — |
| `BRAND_ADMIN` | 특정 브랜드 담당 어드민 | 1개 브랜드에 귀속 |
| `SUPER_ADMIN` | 전체 관리 어드민 | — |

---

#### 시나리오 7. 어드민이 브랜드를 등록·관리한다

브랜드를 생성·수정·삭제할 수 있다. 일반 유저는 이 기능에 접근할 수 없다.

| METHOD | URI | 인증 | 역할 | 설명 |
|--------|-----|:----:|------|------|
| POST | `/api-admin/v1/brands` | O | SUPER_ADMIN | 브랜드 등록 |
| GET | `/api-admin/v1/brands?page=0&size=20` | O | SUPER_ADMIN | 브랜드 목록 조회 |
| GET | `/api-admin/v1/brands/{brandId}` | O | SUPER_ADMIN, BRAND_ADMIN(자신의 브랜드) | 브랜드 상세 조회 |
| PUT | `/api-admin/v1/brands/{brandId}` | O | SUPER_ADMIN, BRAND_ADMIN(자신의 브랜드) | 브랜드 수정 |
| DELETE | `/api-admin/v1/brands/{brandId}` | O | SUPER_ADMIN | 브랜드 삭제 |

**BRAND_ADMIN 제한**
- 자신의 브랜드 상세 조회·수정만 허용된다
- 브랜드 등록·삭제·목록 조회는 SUPER_ADMIN만 가능하다
- 다른 브랜드 접근 시도 → 403

**등록**
- 브랜드명은 필수 입력값이다

**수정**
- 이미 등록된 브랜드의 정보를 수정할 수 있다

**삭제**
- 브랜드를 삭제하면 해당 브랜드에 속한 상품도 함께 삭제된다
- 브랜드를 삭제하면 해당 브랜드에 귀속된 BRAND_ADMIN 계정도 함께 삭제된다
- 삭제된 브랜드와 상품은 고객 화면에서 더 이상 노출되지 않는다

**예외**
- 존재하지 않는 브랜드 접근 → 404
- 다른 브랜드 접근 시도 (BRAND_ADMIN) → 403

---

#### 시나리오 8. 어드민이 상품을 등록·관리한다

상품을 생성·수정·삭제할 수 있다.

| METHOD | URI | 인증 | 역할 | 설명 |
|--------|-----|:----:|------|------|
| POST | `/api-admin/v1/products` | O | SUPER_ADMIN, BRAND_ADMIN(자신의 브랜드) | 상품 등록 |
| GET | `/api-admin/v1/products?page=0&size=20&brandId={brandId}` | O | SUPER_ADMIN, BRAND_ADMIN(자동 필터) | 상품 목록 조회 |
| GET | `/api-admin/v1/products/{productId}` | O | SUPER_ADMIN, BRAND_ADMIN(자신의 브랜드) | 상품 상세 조회 |
| PUT | `/api-admin/v1/products/{productId}` | O | SUPER_ADMIN, BRAND_ADMIN(자신의 브랜드) | 상품 수정 |
| DELETE | `/api-admin/v1/products/{productId}` | O | SUPER_ADMIN, BRAND_ADMIN(자신의 브랜드) | 상품 삭제 |

**BRAND_ADMIN 제한**
- 자신의 브랜드 상품만 등록·조회·수정·삭제할 수 있다
- 상품 목록 조회 시 자신의 brandId로 자동 필터링된다 (쿼리 파라미터 무시)
- 다른 브랜드 상품 접근 시도 → 403
- 상품 등록 시 다른 브랜드 지정 시도 → 403

**등록**
- 상품은 반드시 기존에 등록된 브랜드에 속해야 한다
- 상품명, 가격, 재고, 설명은 필수 입력값이다
- 가격과 재고는 0 이상이어야 한다

**수정**
- 상품명, 가격, 재고, 설명은 수정 가능하다
- 상품의 브랜드는 변경할 수 없다 (브랜드 이동 불가)

**삭제**
- 단건 삭제 가능
- 브랜드 삭제 시 해당 브랜드의 상품은 자동으로 삭제된다

**예외**
- 등록되지 않은 브랜드로 상품 등록 시도 → 404
- 상품 수정 시 브랜드 변경 시도 → 400
- 존재하지 않는 상품 접근 → 404
- 다른 브랜드 상품 접근 시도 (BRAND_ADMIN) → 403

---

#### 시나리오 9. 어드민이 전체 주문을 조회한다

SUPER_ADMIN은 모든 유저의 주문을 조회할 수 있다.

| METHOD | URI | 인증 | 역할 | 설명 |
|--------|-----|:----:|------|------|
| GET | `/api-admin/v1/orders?page=0&size=20` | O | SUPER_ADMIN | 전체 주문 목록 조회 |
| GET | `/api-admin/v1/orders/{orderId}` | O | SUPER_ADMIN | 주문 상세 조회 |

**목록 조회**
- 전체 주문을 페이지 단위로 조회한다

**상세 조회**
- 특정 주문의 상세 내역을 조회한다

**예외**
- 존재하지 않는 주문 접근 → 404

## 미결 사항 (비즈니스 규칙)

아래는 유저 시나리오를 쓰면서 비즈니스 규칙이 명확하지 않아 결정이 필요한 항목들이다.
기술 방식 선택이 아닌, **서비스가 어떻게 동작해야 하는가**에 대한 질문이다.

| # | 질문 | 결정 |
|---|------|------|
| 1 | 좋아요 수가 같은 상품이 여럿일 때 `likes_desc` 정렬의 2차 기준은? | 최신순 |
| 2 | 상품 목록의 각 카드에 브랜드명이 바로 보여야 하는가? | 바로 표시 |
| 3 | 재고가 0인 상품도 목록에 노출되는가? | 노출 |
| 4 | 한 상품이라도 재고 부족이면 주문 전체가 실패하는가? | 전체 실패 |

## 기능적 요구사항 (Functional Requirements)

시스템이 **무엇을** 해야 하는가.

**고객**
- **상품 목록 조회**: 최신순 / 좋아요순 / 가격 낮은 순 정렬, 브랜드 필터, 페이지네이션
- **상품 상세 조회**: 상품 정보 조회
- **브랜드 정보 조회**
- **좋아요 등록 / 취소**
- **내 좋아요 목록 조회**
- **주문 요청**: 여러 상품을 한 번에, 재고 확인 및 차감, 상품 정보 스냅샷 저장
- **주문 목록 조회**: 날짜 범위 필터
- **주문 상세 조회**

**어드민 (SUPER_ADMIN)**
- **브랜드 관리**: 등록 / 목록 조회 / 상세 조회 / 수정 / 삭제 (삭제 시 소속 상품 함께 삭제)
- **상품 관리**: 전체 브랜드 상품 등록 / 목록 조회 / 상세 조회 / 수정 / 삭제
- **주문 조회**: 전체 주문 목록 조회 / 주문 상세 조회

**어드민 (BRAND_ADMIN)**
- **브랜드 관리**: 자신의 브랜드 상세 조회 / 수정
- **상품 관리**: 자신의 브랜드 상품 등록 / 목록 조회 / 상세 조회 / 수정 / 삭제

---

## 비기능적 요구사항 (Non-Functional Requirements)

시스템이 **어떻게** 동작해야 하는가.

- **응답 속도**: 상품 목록 조회, 좋아요 등록 등 고객 인터랙션은 체감상 즉각 반영되어야 한다
- **데이터 불변성**: 주문 완료 후 상품 정보가 변경되어도 주문 기록의 상품명·가격은 변하지 않아야 한다
- **데이터 일관성**: 주문 시 재고 확인과 차감은 원자적으로 처리되어야 한다
- **접근 제어**: 고객은 본인의 데이터(주문)만 접근할 수 있다. SUPER_ADMIN은 전체 브랜드·상품·주문을 관리할 수 있고, BRAND_ADMIN은 자신의 브랜드 상품 관리에만 접근할 수 있다
- **동시성**: 동일 상품에 동시 주문이 발생해도 재고가 초과 차감되지 않아야 한다. 재고가 부족해진 시점 이후의 주문은 실패 처리된다. 비관적 락(Pessimistic Lock)으로 구현한다.

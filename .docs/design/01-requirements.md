## 1. 서비스 개요

IT 직군 기술자가 자신의 기술 스택·수준에 맞는 도서를 찾고, 관심 도서를 모아두고, 한 번에 주문·결제하는 도서 이커머스.

| 구분          | 내용                                                                  |
|---------------|-----------------------------------------------------------------------|
| 주요 사용자   | IT 직군 기술자 (개발자, 엔지니어, DevOps, AI/ML 실무자 등)            |
| 해결하는 문제 | 기술 스택에 맞는 도서를 찾기 어렵고, 구매 전 난이도 판단이 불명확하다 |
| 핵심 차별점   | 기술 카테고리·난이도 기반 탐색 + 관심 도서 통합 관리                  |
| 타겟 제외     | 일반 교양서·소설·어린이 도서                                          |

---

## 2. 액터 (Actor)

| 액터       | 정의                         | 권한 범위                               |
|------------|------------------------------|-----------------------------------------|
| **방문자** | 인증되지 않은 사용자         | 도서·브랜드 조회만 가능                 |
| **유저**   | 가입을 완료한 IT 직군 기술자 | 좋아요, 주문, 결제, 본인 정보 조회/수정 |
| **관리자** | 서비스 운영팀 구성원         | 브랜드·상품 CRUD, 전체 주문 조회        |

---

## 3. 핵심 도메인

```
방문자 → 도서 탐색 (브랜드/상품)
유저  → 관심 도서 (좋아요) → 주문 → 결제
관리자 → 브랜드/상품 관리, 주문 모니터링
```

다루는 도메인: **브랜드 · 상품 · 좋아요 · 주문 · 결제**

## 4. 기능 요구사항

> "유저는 ~할 수 있어야 한다" 관점으로 작성한다.

### FR-01. 회원 관리

| ID      | 기능                                                                           | 액터   |
|---------|--------------------------------------------------------------------------------|--------|
| FR-01.1 | 유저는 아이디·비밀번호·이름·생년월일·이메일로 **회원가입을 할 수 있어야 한다** | 방문자 |
| FR-01.2 | 유저는 **자신의 정보(아이디·이름)를 조회할 수 있어야 한다**                    | 유저   |
| FR-01.3 | 유저는 현재 비밀번호 확인 후 **새 비밀번호로 변경할 수 있어야 한다**           | 유저   |
| FR-01.4 | 유저는 **타인의 정보에 접근할 수 없어야 한다**                                 | 유저   |

### FR-02. 브랜드·상품 탐색

| ID      | 기능                                                                                                                        | 액터        |
|---------|-----------------------------------------------------------------------------------------------------------------------------|-------------|
| FR-02.1 | 누구나 **브랜드 정보를 조회할 수 있어야 한다**                                                                              | 방문자/유저 |
| FR-02.2 | 누구나 **도서 목록을 조회할 수 있어야 한다**                                                                                | 방문자/유저 |
| FR-02.3 | 누구나 도서 목록을 **기술 카테고리**(BACKEND/FRONTEND/DEVOPS/AI_ML/DATABASE/SECURITY/NETWORK/ETC)로 필터링할 수 있어야 한다 | 방문자/유저 |
| FR-02.4 | 누구나 도서 목록을 **난이도**(BEGINNER/INTERMEDIATE/ADVANCED)로 필터링할 수 있어야 한다                                     | 방문자/유저 |
| FR-02.5 | 누구나 도서 목록을 **브랜드(출판사)**로 필터링할 수 있어야 한다                                                             | 방문자/유저 |
| FR-02.6 | 누구나 도서 목록을 **최신순 / 가격 낮은 순 / 좋아요 많은 순**으로 정렬할 수 있어야 한다                                     | 방문자/유저 |
| FR-02.7 | 누구나 **도서 상세 정보를 조회할 수 있어야 한다**                                                                           | 방문자/유저 |

### FR-03. 관심 도서 (좋아요)

| ID      | 기능                                                                  | 액터 |
|---------|-----------------------------------------------------------------------|------|
| FR-03.1 | 유저는 도서에 **좋아요를 등록할 수 있어야 한다**                      | 유저 |
| FR-03.2 | 유저는 등록한 **좋아요를 취소할 수 있어야 한다**                      | 유저 |
| FR-03.3 | 유저는 **자신이 좋아요한 도서 목록을 조회할 수 있어야 한다**          | 유저 |
| FR-03.4 | **동일 도서에 중복 좋아요 등록 시도해도 결과가 동일해야 한다** (멱등) | 유저 |
| FR-03.5 | **존재하지 않는 좋아요를 취소 시도해도 결과가 동일해야 한다** (멱등)  | 유저 |
| FR-03.6 | 유저는 **타인의 좋아요 목록에 접근할 수 없어야 한다**                 | 유저 |

### FR-04. 주문

| ID      | 기능                                                                                       | 액터   |
|---------|--------------------------------------------------------------------------------------------|--------|
| FR-04.1 | 유저는 **1개 이상의 도서를 한 번에 주문할 수 있어야 한다**                                 | 유저   |
| FR-04.2 | 주문 시점의 **도서명·가격이 영구 보존되어** 이후 도서 정보 변경에 무관하게 유지되어야 한다 | System |
| FR-04.3 | 재고가 부족한 도서가 포함되면 **주문 전체가 실패해야 한다**                                | System |
| FR-04.4 | 유저는 **자신의 주문 목록을 날짜 범위로 조회할 수 있어야 한다**                            | 유저   |
| FR-04.5 | 유저는 **자신의 주문 상세를 조회할 수 있어야 한다**                                        | 유저   |
| FR-04.6 | 유저는 **타인의 주문에 접근할 수 없어야 한다**                                             | 유저   |

### FR-05. 결제

| ID      | 기능                                                                      | 액터   |
|---------|---------------------------------------------------------------------------|--------|
| FR-05.1 | 유저는 자신의 **PENDING 상태 주문에 대해 결제를 요청할 수 있어야 한다**   | 유저   |
| FR-05.2 | **결제 금액은 주문 총액과 일치해야 한다**                                 | System |
| FR-05.3 | 결제는 **외부 PG(Payment Gateway) 시스템을 통해 처리되어야 한다**         | System |
| FR-05.4 | 결제 성공 시 **주문이 확정(CONFIRMED)되어야 한다**                        | System |
| FR-05.5 | 결제 실패 시 **주문이 취소(CANCELLED)되고 차감된 재고가 복구되어야 한다** | System |

### FR-06. 관리자 운영

| ID      | 기능                                                       | 액터   |
|---------|------------------------------------------------------------|--------|
| FR-06.1 | 관리자는 **브랜드를 등록·수정·삭제할 수 있어야 한다**      | 관리자 |
| FR-06.2 | 관리자는 **도서를 등록·수정·삭제할 수 있어야 한다**        | 관리자 |
| FR-06.3 | **브랜드 삭제 시 소속 도서가 모두 함께 삭제되어야 한다**   | System |
| FR-06.4 | **도서 등록 시 브랜드는 활성(ACTIVE) 상태여야 한다**       | System |
| FR-06.5 | **도서의 브랜드는 등록 후 수정할 수 없어야 한다**          | System |
| FR-06.6 | 관리자는 **전체 주문 목록과 상세를 조회할 수 있어야 한다** | 관리자 |

## 5. 유저 시나리오

### 시나리오 1. 도서 탐색 → 관심 등록

```
방문자가 메인 페이지에 접속한다
  ↓
관심 카테고리(예: BACKEND)와 난이도(예: INTERMEDIATE) 필터를 적용한다
  ↓
결과를 좋아요 많은 순으로 정렬한다
  ↓
관심 도서를 선택해 상세 페이지로 이동한다
  ↓
[로그인 필요]
좋아요 버튼을 눌러 관심 표시한다
  ↓
나중에 "내가 좋아요한 도서" 페이지에서 다시 확인한다
```

### 시나리오 2. 주문 → 결제

```
유저가 도서 상세 페이지에서 주문을 시작한다
  ↓
여러 도서를 한 번에 담아 주문 요청한다
  ↓
[주문 생성] 재고가 즉시 차감되고 주문 상태는 PENDING이 된다
주문 시점의 도서명·가격이 스냅샷으로 저장된다
  ↓
별도의 결제 요청을 진행한다
  ↓
외부 PG 시스템을 통해 결제가 처리된다
  ↓
┌─ 결제 성공 → 주문 CONFIRMED 로 확정
└─ 결제 실패 → 주문 CANCELLED + 재고 자동 복구
```

### 시나리오 3. 관리자 — 도서 등록

```
관리자가 어드민 페이지에 접속한다 (X-Loopers-Ldap 헤더)
  ↓
신규 브랜드(출판사)를 등록한다
  ↓
해당 브랜드 하위에 신규 도서를 등록한다
  - ISBN(13자리, 유일)
  - 이름, 저자
  - 기술 카테고리, 난이도
  - 가격, 재고
  ↓
필요 시 도서 정보를 수정한다 (브랜드는 수정 불가)
  ↓
브랜드를 삭제할 경우 소속 도서가 모두 함께 삭제된다
```

### 시나리오 4. 좋아요 멱등 동작

```
유저가 도서 A에 좋아요를 등록한다 → 201 Created
  ↓
[네트워크 재시도 또는 중복 탭]
같은 도서에 다시 좋아요 등록을 요청한다 → 200 OK (이미 좋아요, no-op)
  ↓
유저가 좋아요를 취소한다 → 204 No Content
  ↓
[실수로 다시 취소]
존재하지 않는 좋아요를 다시 취소 요청한다 → 204 No Content (no-op)
```

## 6. 비즈니스 규칙

### 6-1. 회원

| #   | 규칙                                                | 액터   |
|-----|-----------------------------------------------------|--------|
| R1  | `LoginId`는 서비스 전체에서 중복 불가               | System |
| R2  | 비밀번호는 8자 이상이어야 한다                      | System |
| R3  | 유저는 타 유저의 정보에 직접 접근할 수 없다         | User   |
| R4  | 비밀번호 변경 시 현재 비밀번호 일치 확인이 필요하다 | User   |

### 6-2. 브랜드·상품

| #   | 규칙                                                                       | 액터   |
|-----|----------------------------------------------------------------------------|--------|
| R1  | 브랜드·도서 등록·수정·삭제는 관리자만 가능하다                             | Admin  |
| R2  | 브랜드 삭제 시 소속 도서도 모두 삭제된다 (소프트 삭제)                     | System |
| R3  | 도서 등록 시 브랜드는 활성(ACTIVE) 상태여야 한다                           | System |
| R4  | 도서의 브랜드는 등록 후 수정할 수 없다                                     | System |
| R5  | ISBN은 서비스 내 유일해야 한다                                             | System |
| R6  | 도서 목록·상세 조회는 누구나 가능하다                                      | 모두   |
| R7  | 정렬 옵션: `latest`(기본) / `price_asc` / `likes_desc`                     | 모두   |
| R8  | 대고객에게 재고 수량·삭제 상태·ISBN을 노출하지 않는다                      | System |
| R9  | 재고 0인 도서에 대한 주문은 재고 부족과 동일하게 주문 전체 실패로 처리한다 | System |

### 6-3. 좋아요 (멱등)

| #   | 규칙                                                                       | 액터   |
|-----|----------------------------------------------------------------------------|--------|
| R1  | 같은 유저가 같은 도서에 중복 좋아요 등록을 시도해도 결과는 동일 (멱등 200) | User   |
| R2  | 존재하지 않는 좋아요 취소를 시도해도 결과는 동일 (멱등 204)                | User   |
| R3  | 좋아요 등록 시 도서의 좋아요 수가 +1 된다 (중복 시도는 no-op)              | System |
| R4  | 좋아요 취소 시 도서의 좋아요 수가 -1 된다 (미존재 시도는 no-op, 최솟값 0)  | System |
| R5  | 좋아요 목록 조회는 본인 것만 가능하다                                      | User   |

> **멱등 정책 근거**: 좋아요는 상태 표현(Binary State Toggle) — REST PUT 시맨틱.  
> 모바일 재시도·네트워크 재전송에 강건한 UX 우선 (Twitter/Instagram/GitHub Star 동일 패턴).

### 6-4. 주문

| #   | 규칙                                                                 | 액터   |
|-----|----------------------------------------------------------------------|--------|
| R1  | 주문 항목은 1개 이상이어야 한다                                      | User   |
| R2  | 재고 부족 시 해당 도서를 포함한 주문 전체가 실패한다                 | System |
| R3  | 주문 생성 시 재고를 즉시 차감하고 주문 상태를 `PENDING`으로 저장한다 | System |
| R4  | 주문 정보에 주문 시점의 도서명·가격 스냅샷을 저장한다                | System |
| R5  | 유저는 자신의 주문만 조회할 수 있다                                  | User   |
| R6  | 주문 목록 조회는 날짜 범위(startAt ~ endAt) 필터를 지원한다          | User   |
| R7  | 주문 항목당 수량은 1 이상이어야 한다                                 | System |
| R8  | 전체 주문 목록·상세 조회는 관리자만 가능하다                         | Admin  |

### 6-5. 결제

| #   | 규칙                                                                   | 액터   |
|-----|------------------------------------------------------------------------|--------|
| R1  | 결제 대상 주문은 본인의 `PENDING` 상태 주문이어야 한다                 | User   |
| R2  | 요청 금액(`amount`)이 주문 총액(`totalAmount`)과 일치해야 한다         | System |
| R3  | 외부 PG(Payment Gateway) API를 호출하여 실제 결제를 처리한다           | System |
| R4  | 결제 성공 시 주문 상태가 `CONFIRMED`로 전환된다                        | System |
| R5  | 결제 실패 시 주문 상태가 `CANCELLED`로 전환되고 차감된 재고가 복구된다 | System |

#### 주문 상태(`OrderStatus`) 생명주기

```
주문 요청 → PENDING (재고 차감 + 스냅샷 저장)
               ↓
          결제 요청
         ↙        ↘
   결제 성공      결제 실패
  CONFIRMED      CANCELLED
                (재고 복구)
```

## 7. 확정된 설계 결정사항

| 항목               | 결정                                     | 근거                                |
|--------------------|------------------------------------------|-------------------------------------|
| 기술 카테고리 필터 | `TechCategory` 기반 도서 필터            | H1 ✅ 62.0% (리서치, 전 직군 채택)   |
| 난이도 표시        | `Level` (BEGINNER/INTERMEDIATE/ADVANCED) | H2 ✅ 79.1% (수준 불일치 경험자 79%) |
| 좋아요 기능        | 단순 토글 + **완전 멱등 동작**           | H4 ✅ 55.0% + REST PUT 시맨틱        |
| 주문 구조          | 브랜드 무관 단일 주문 1건 / 배송 1번     | H5 ⚠️ 40.1% (추가 검증 권고)        |
| 결제 흐름          | 주문 생성(PENDING) → 별도 결제 요청      | 외부 PG 연동·보상 트랜잭션 분리     |
| 인증 방식          | HTTP 헤더 기반 (`X-Loopers-LoginId/Pw`)  | 단순 인증 — 운영 시 JWT/mTLS 교체   |

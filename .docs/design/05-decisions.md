> 01~04 문서가 "확정된 설계(무엇을·어떻게)"를 담는다면, 이 문서는 **"왜 그렇게 결정했는가"** 를 남긴다.
> 나중에 "이걸 왜 이렇게 했지?"를 다시 떠올릴 수 있도록 고려한 대안·트레이드오프·막혔던 지점을 함께 기록한다.

## 🧭 Context & Decision

### 문제 정의
- **현재 동작/제약**: 옵션·포인트·배송 추적이 없는 단순 커머스의 설계 문서(01 요구사항 / 02 시퀀스 / 03 클래스 / 04 ERD)를 도출. 구현이 아닌 **설계 정합성 확보**가 목표.
- **문제(또는 리스크)**:
  - 동시 주문 시 재고 오버셀(초과 판매) 위험
  - 외부 PG 연동 시 위조 콜백·중복 콜백·금액 위변조로 인한 잘못된 주문 확정 위험
  - 복수 상품 주문에서 일부만 재고가 있을 때의 일관성
  - 브랜드/상품 삭제 시 종속 데이터(재고·좋아요) 처리 — 특히 좋아요가 수천만~수억 건일 때 유령 데이터(dead row) 누적
  - 삭제된 브랜드명 재등록을 통한 브랜드 사칭/이미지 편승
  - 타 사용자의 주문/좋아요 자원에 대한 무단 접근
- **성공 기준(완료 정의)**: 4개 문서가 서로 모순 없이 일관되고, 모든 정책이 시퀀스·클래스·ERD 제약으로 일관되게 표현될 것. 호출처 없는 메서드/엔드포인트가 없을 것.

## 선택지와 결정

### ① 결제·재고 차감 방식
- 고려한 대안:
  - A: 주문 즉시 재고 차감 + 결제 흐름 생략(단순)
  - B: 재고 예약(`reserved_quantity`) → 결제 확정 시 차감하는 PG 연동 흐름
- 최종 결정: **B**. 재고를 바꾸는 **4가지 동작(reserve / confirm / release / restore)** 을 `Stock` 도메인이 단독 책임.
  - `reserve`: **주문하기(주문 생성, 결제 직전)** 시점에 `reserved_quantity`를 늘려 결제가 끝날 때까지 해당 수량을 선점(다른 사용자가 못 사게 함). 실물(`total_quantity`)은 건드리지 않음.
  - `confirm`: 결제 성공 시 선점분을 실제 판매로 확정 (`total -= qty`, `reserved -= qty`).
  - `release`: 결제 실패/만료 시 선점만 해제 (`reserved -= qty`, `total` 유지).
  - `restore`: 이미 확정(confirm)된 주문을 취소할 때 실물을 되돌림 (`total += qty`).
  - 오버셀은 원자적 조건부 UPDATE(`SET reserved += qty WHERE (total - reserved) >= qty`)로 방어. 불변식 `total >= reserved` 유지.
- 트레이드오프: 흐름이 복잡해지지만 결제 중 재고 선점/실패 롤백을 정확히 표현. 외부 시스템 연동의 현실성 확보.
- 메모: `Stock`에는 상태 enum이 없다. 위 4가지는 "상태값"이 아니라 `total`/`reserved` 두 수량 컬럼을 바꾸는 **동작**이다. `release`(확정 전 실패)와 `restore`(확정 후 취소)의 갈림길은 "이미 confirm 됐는가".

### ② 복수 상품 주문 시 일부 품절 처리
- 고려한 대안:
  - A: 재고 있는 상품만 부분 주문 생성
  - B: 하나라도 부족하면 주문 전체 거부(원자적)
- 최종 결정: **B**. 하나라도 재고가 부족하면 이미 예약한 항목까지 모두 롤백하고 **409**로 전체 거부. (부분 주문 없음)
- 트레이드오프: 사용자 편의(부분 구매 가능)를 포기하는 대신 데이터 일관성과 흐름 단순성을 확보.

### ③ 결제 수단 / 금액 모델
- 고려한 대안: 포인트 + PG 혼합 결제 / PG 단일
- 최종 결정: **포인트 기능 제거, PG 단일**. `orders`는 `pg_amount` 단일 컬럼만 보유.
- 트레이드오프: 포인트 차감/환불 단계가 사라져 결제·취소 트랜잭션이 단순해짐. 금액 정합성은 `pg_amount = payments.amount = Σ(order_items.price × quantity)` 한 줄로 정의.

### ④ 옵션 / 재고 구조
- 고려한 대안: 상품-옵션-옵션별 재고(`product_options`) / 옵션 없이 상품당 재고 1건
- 최종 결정: **옵션 제거**. `products : stocks = 1:1`(별도 테이블, `stocks.product_id` UNIQUE).
- 트레이드오프: 옵션별 재고 표현력을 포기하는 대신 모델이 크게 단순해짐. 재고는 항상 "상품 단위".

### ⑤ PG 콜백 신뢰성 (위조·중복·위변조 방어)
- 문제: `/payments/confirm`은 외부 PG가 호출하는 웹훅이라 위조·중복 전송·금액 위변조에 노출된다.
- 최종 결정: **3중 방어**
  - **서명 검증(HMAC 등)**: 공유 시크릿 기반 서명으로 진짜 PG 콜백인지 먼저 검증. 위조면 401.
  - **멱등 처리**: 주문이 `PENDING`일 때만 처리하고, 이미 `CONFIRMED`/`FAILED`면 재고·상태 변경 없이 200으로 수렴(중복 콜백 → 재고 이중 차감 방지).
  - **금액 검증**: PG 승인 금액 == `orders.pg_amount` 일치 시에만 확정. 불일치는 결제 실패와 동일 처리.
- 트레이드오프: 검증 단계가 늘지만, "5만원 주문이 5천원에 확정" 류의 사고와 이중 차감을 원천 차단. PG 재조회(`verifyPayment`)가 2차 방어.

### ⑥ PENDING 주문 만료 처리
- 고려한 대안: 만료 없음(영구 PENDING) / 스케줄러 만료
- 최종 결정: **스케줄러가 생성 후 15분 초과 PENDING 주문을 스캔해 재고 예약 해제 + FAILED**. 인덱스 `(status, created_at)`로 조회 풀스캔 방지. 재시도는 없음.
- 트레이드오프: 콜백 미수신 시에도 선점된 재고가 영구히 묶이지 않음. 15분은 임의 기준(추후 조정 가능).

### ⑦ 삭제·복구 정책
- 고려한 대안: 스냅샷 테이블 이관 / 하드 딜리트 / `deleted_seq` 트릭 / 소프트 딜리트
- 핵심 기준: **데이터 용량**과 **복구 가치**에 따라 처리 방식을 달리한다.
- 최종 결정: **대상별 3갈래**
  - **브랜드·상품** (저용량, 복구 가치 O) → **소프트 딜리트**(`deleted_at` 기록). 즉시 노출만 차단하고 데이터는 보존해 향후 복구 가능.
  - **재고**(`stocks`, 상품과 1:1 별도 테이블) → **삭제 처리 안 함(방치)**. 삭제 플래그도, 물리 삭제도 없음. 재고는 상품을 통해서만 도달하므로 상품이 소프트 딜리트되면 조회 경로가 끊겨 자동으로 숨겨지고, 상품 복구 시 그대로 다시 딸려온다. 저용량(상품당 1건)이라 dead row 부담도 없음.
  - **좋아요**(고용량 수천만~수억, 복구 가치 X) → **비동기 배치 청크 물리 삭제(실제 DELETE)**. 상품 삭제 시 `products.likes_purged=false`로 마킹 → 배치가 `deleted_at IS NOT NULL AND likes_purged=false` 상품을 찾아 `likes`를 청크 단위로 DELETE → 완료 후 `likes_purged=true`로 마킹(반복 스캔 방지).
- 트레이드오프: 대상별로 정책이 갈려 복잡해지지만, 삭제 시 동기 트랜잭션은 `brands`/`products`의 `deleted_at` 기록만으로 가볍게 유지(즉시 노출 차단)하고, 무거운 대량 삭제는 비동기 배치로 분리해 응답 지연·락 경합을 피한다.
- 메모: 브랜드 삭제 시 소속 상품도 함께 소프트 딜리트(cascade, 동기 트랜잭션). 과거 주문의 `order_items`는 스냅샷이라 상품 삭제와 무관하게 보존된다.

### ⑧ 브랜드명 유니크 정책
- 고려한 대안:
  - A: `name` UNIQUE + 삭제된 행도 슬롯 유지 → 삭제 후에도 재등록 불가
  - B: 활성 행만 유니크(삭제 시 `deleted_seq` 같은 값을 섞어 유니크 회피) → 삭제된 이름 재등록 허용
  - C: 삭제 브랜드를 별도 스냅샷/아카이브 테이블로 이관 후 원본 삭제
- 판단 기준: **브랜드명을 신뢰 자산으로 볼 것인가** + 추가 장치 없이 단순하게 보장할 수 있는가.
- 최종 결정: **A** — `brands.name` **UNIQUE**, 소프트 딜리트된 행이 유니크 슬롯을 계속 점유하는 것을 **의도된 동작**으로 수용해 **삭제된 이름도 재등록 불가**.
- 트레이드오프: 삭제 행이 슬롯을 점유하므로 동일 브랜드명을 영영 못 쓰지만, 사칭/이미지 편승을 원천 차단. `deleted_seq` 트릭(B)이나 아카이브 테이블(C) 같은 추가 장치가 불필요해 구조가 단순해진다. (브랜드는 저용량이라 슬롯 점유 부담도 없음)
- 메모(B 트릭이란): MySQL 8.0은 부분/조건부 유니크 인덱스를 지원하지 않으므로, "활성 행만 유니크"를 흉내 내려면 `UNIQUE(name, 삭제구분)` 복합키에서 활성은 고정값(예 0), 삭제 행은 행마다 다른 값(id 등)을 넣어 슬롯을 비워줘야 한다. 우리는 재등록을 막는 게 목표라 이 트릭이 필요 없어 단순 `UNIQUE(name)`로 충분했다.

### ⑨ 관리자 재고 수정 동시성
- 고려한 대안:
  - A: 델타(+N/-N) 방식으로 증감
  - B: 절대값(`total_quantity = :newTotal`) 설정 + 조건부 가드
- 최종 결정: **B (델타 미사용)**. 관리자는 `total_quantity`(실물)만 수정하고 `reserved_quantity`(시스템 값)는 못 건드림. `UPDATE ... SET total = :newTotal WHERE :newTotal >= reserved_quantity`로 원자적 검증, affected rows=0이면 **409 거부**.
- 트레이드오프: 절대값 설정 시 발생할 수 있는 lost update는 무해한 수준으로 수용. 불변식 `total >= reserved`(가용 ≥ 0)를 DB 레벨에서 보장하는 게 핵심.

### ⑩ 인증 / 인가
- 고려한 대안: 실제 인증/인가 체계 구현 / 헤더 기반 식별 + 소유권 검증
- 최종 결정:
  - **대고객**(`/api/v1`): `X-Loopers-LoginId/LoginPw` 헤더로 식별 + `userService.authenticate()` 검증. 별도 role 체계 없이 **본인 자원(주문/좋아요)만** 접근 — 소유권 검증. 타인 자원 접근은 존재 여부 비노출 위해 **404**.
  - **관리자**(`/api-admin/v1`): `X-Loopers-Ldap: loopers.admin` 헤더 값 일치만 검증(실제 LDAP 미연동). 인터페이스 계층(Interceptor) 게이트로 처리, 도메인은 인증을 모름.
  - 모든 인증 실패 → **401**.
- 트레이드오프: 실제 인증/인가는 과제 범위 밖이라 단순화. 다만 소유권 검증은 반드시 적용해 데이터 격리를 보장.

### ⑪ 응답 정보 분리 (고객 vs 어드민)
- 최종 결정: URL prefix 분리(`/api/v1` vs `/api-admin/v1`)를 근거로 응답 정보를 다르게 노출.
  - 고객: 구매 판단에 필요한 정보만 — 상품명·가격·설명·브랜드명·좋아요 수·**품절 여부(stockStatus)**. (정확한 재고 수량·삭제 상태·타임스탬프 비노출)
  - 관리자: 위 + **정확한 재고 수량**·삭제 상태·생성/수정 시각.
- 트레이드오프: 같은 자원이라도 DTO를 분리해야 하지만, 경쟁사 노출·재고 심리 등을 고려하면 정보 최소 노출이 합리적.

### ⑫ 좋아요 등록/취소 API 구조
- 고려한 대안: 토글 단일 엔드포인트 / POST·DELETE 분리
- 최종 결정: **POST(등록)·DELETE(취소) 분리, 각각 멱등**. `(user_id, product_id)` 복합 UNIQUE로 DB 레벨 중복 방어. 목록 조회는 본인 것만(타인 `userId` 경로 접근 시 404).
- 트레이드오프: 엔드포인트가 둘로 늘지만 멱등성이 명확해지고 토글의 경합 모호함이 사라짐.

### ⑬ `restore()` 메서드 포함 여부
- 고려한 대안: (가) 03 클래스에 `restore()` 유지(복구 API는 추후) / (나) 제거
- 최종 결정: **(나) 제거**. 복구 API가 요구사항에 없어 호출처가 없으므로 `Brand`/`Product`에서 `restore()` 제거. 소프트 딜리트로 데이터는 보존되니 복구 가능성은 유지 → 향후 복구 API를 설계할 때 메서드와 함께 도입한다.
- 트레이드오프: 미래 확장 의도를 다이어그램에 미리 못 박기보다 "현재 동작하는 것만 그린다"는 문서 정직성을 우선.

### ⑭ 값 타입(Value Object) 추출 범위

- **문제**: 검증 로직이 엔티티 생성자에 직접 산재. 동일 개념(금액, 배송지)이 여러 엔티티에 흩어져 중복.
- **최종 결정**: 아래 8개 VO를 `@Embeddable`로 추출. DB 컬럼 구조(ERD)는 변경 없음.

| VO | 위치 | 검증 책임 |
|---|---|---|
| `LoginId` | `User` | 영문/숫자 1~10자 |
| `Email` | `User` | 이메일 형식 |
| `Birth` | `User` | YYYY-MM-DD 형식, 실존 날짜 |
| `Money` | `Product`, `Order`, `OrderItem`, `Payment` | 음수 불가 |
| `Receiver` | `Order` (ShippingInfo 내) | 수령인명·전화번호 blank 불가 |
| `Address` | `Order` (ShippingInfo 내) | 우편번호·기본주소 blank 불가 |
| `ShippingInfo` | `Order` | Receiver + Address 조합 |
| `ProductSnapshot` | `OrderItem` | 주문 시점 상품명·브랜드명·단가 불변 스냅샷 |

- **트레이드오프**: JPA `existsByLoginId` → `existsByLoginIdValue` 경로 변경 필요(RepositoryImpl에서 브릿징). DB 구조 무변경으로 마이그레이션 없음.
- **제외**: `Password`(PasswordEncoder 분리로 충분), `PgTransactionId`(형식 규칙 없음), 재고 수량(Stock 엔티티가 불변식 직접 관리).

## 추후 개선 여지
- 복구 API(`POST /.../restore`) 도입 시 `restore()` 동시 추가
- `like_count` 카운터 캐시 정합성 보정 배치 (좋아요 등록/취소 시 함께 갱신하는 캐시가 동시성·부분 실패로 틀어질 수 있어, 주기적으로 `COUNT(*)`와 재동기화)
- 정렬 옵션 `price_asc`·`likes_desc` 선택 구현 (현재 `latest`만 필수)
- Service 간 직접 참조 금지 원칙 유지 → 추후 이벤트 기반으로 전환 시 영향 범위를 Facade 한 곳으로 제한

## 🤔 고민한 점 / 막혔던 부분
- **소프트 딜리트 vs 유니크 제약의 충돌**: 삭제된 브랜드명을 막으려면 유니크가 필요한데, 소프트 딜리트 행이 슬롯을 점유한다. 처음엔 스냅샷 이관/`deleted_seq`를 고민했으나 "삭제 행이 슬롯을 점유하는 것 자체가 의도된 동작"으로 받아들이며 해소.
- **유령 데이터 처리**: "상품·재고·좋아요에 플래그가 없으면 나중에 배치로 지울 수도 없다"는 지적에서 막혔다가, `products.likes_purged` 마킹 + 청크 DELETE 배치로 해결.
- **재고 예약 모델 흐름**: `total`/`reserved`/가용 재고의 관계와 동시성하 오버셀 방지(조건부 UPDATE)를 다시 정리. `release`와 `restore`의 차이("확정 전/후")가 특히 헷갈렸음.
- **관리자 재고 수정 동시성**: 델타(+/-) 방식 대신 절대값 설정 + `WHERE :newTotal >= reserved` 가드로 불변식을 원자적으로 보장하는 방식 채택.
- **결제 흐름의 정교화**: 한때 단순화(즉시 차감)했다가, 주문 상태 정교화 필요성을 깨닫고 PG/예약 흐름을 되살린 과정이 있었음.

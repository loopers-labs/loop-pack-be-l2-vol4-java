## 업무 스타일 / 사고 습관

- **시퀀스 → 클래스 → ERD 순으로 읽음**: 흐름을 먼저 파악한 뒤 구조를 본다.
- **트랜잭션 경계와 동시성**에 민감: "재고 차감은 어디서?", "주문 실패 시 롤백 범위는?"을 항상 확인.
- **API 스펙 우선주의**: 컨트롤러 시그니처/요청·응답 스키마가 명확해야 작업을 시작할 수 있다.
- **테스트 작성을 좋아함**: 도메인 규칙을 단위 테스트로 표현하는 것을 선호.
- **DDD 학습 의지**: 새 회사가 DDD 기반이라 적극적으로 배우고 싶어함. 다만 용어가 익숙하지 않다.

## 약점 (검증 시 막힐 만한 부분)

- DDD 애그리게이트 경계 설정 경험 부족 — "Order와 OrderItem이 같은 애그리게이트인가?" 판단이 느림
- 도메인 이벤트(예: `OrderPlaced`)를 카프카로 발행할지, 트랜잭셔널 아웃박스를 쓸지에 대한 패턴 선택 경험 적음
- 멀티 모듈(`apps`/`modules`/`supports`) 구조에서 의존성 방향 규칙을 처음 익히는 중
- Mermaid 다이어그램은 읽을 수 있으나, 시퀀스의 `participant` 사이 책임 분배가 OOP적으로 맞는지 판단 보류함

## 업무 스타일 / 사고 습관

- **숫자 기반 의사결정**: 가설 → 지표 → 검증을 좋아함. "왜 이 정책인가?"를 항상 묻는다.
- **사용자 시나리오 중심**: 화면 와이어프레임보다 **유저 플로우(시퀀스)**를 먼저 그린다.
- **엣지 케이스 추적자**: 정상 흐름보다 "재고 0이면 어떻게 되나요?", "주문 중 가격이 바뀌면?" 같은 예외를 먼저 묻는다.
- **개발자와의 협업**: ERD나 클래스 다이어그램을 직접 그리진 않지만, **읽고 토론할 수 있을 정도**의 이해도를 목표로 함.

## 검증 시 관심 포인트

| 영역 | 질문 |
|------|------|
| **비즈니스 의도** | 왜 이 도메인 모델인가? 어떤 사용자 문제를 해결하나? |
| **정책의 명확성** | 비즈니스 규칙이 모호하지 않은가? 엣지 케이스가 명시되었나? |
| **유저 시나리오** | 시퀀스가 실제 유저 행동과 일치하나? |
| **확장성** | 향후 쿠폰·포인트·결제가 붙을 자리가 마련되어 있나? |
| **유비쿼터스 언어** | 용어가 일관되게 쓰이나? 한국어/영어 매핑이 정확한가? |

## 약점 (검증 시 막힐 만한 부분)

- DDD 용어(애그리게이트, 바운디드 컨텍스트, 도메인 이벤트)에 익숙하지 않음
- Mermaid 다이어그램은 읽을 수 있으나 직접 작성은 어려움
- 클래스 다이어그램·ERD는 **읽지 않는다** (개발자 영역으로 위임)
- 기술적 세부사항(스냅샷 구현 방식, 동시성 처리 등)은 "왜 필요한지"만 이해하고 "어떻게"는 패스

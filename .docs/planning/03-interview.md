# 유저 인터뷰

> **목적:** 설문으로 드러나지 않는 실제 구매 행동과 감정의 맥락을 파악한다.
> **대상:** IT 직군 기술자 / 직군별 1명, 총 5명 (각 30분)
> **방식:** 과거 행동 기반 스토리텔링 — 추측이 아닌 실제 경험만 묻는다.
> **페르소나 출처:** nvidia/Nemotron-Personas-Korea 데이터셋에서 IT 직군 추출

---

## 인터뷰 진행 원칙

```
✓ 서비스·기능 소개 없이 시작한다 — 설명하는 순간 인터뷰이의 답변이 오염된다
✓ "~하면 쓸 것 같나요?"는 절대 묻지 않는다 — 과거 경험만 물어본다
✓ "왜요?"를 최소 2번 파고든다 — 첫 번째 답은 표면이다
✓ 침묵을 채우지 않는다 — 10초는 생각할 시간으로 기다린다
✓ 칭찬하지 않는다 — "좋은 답변이네요"는 답변 방향을 유도한다
```

---

## 도입 (5분, 라포 형성)

> 직군과 현재 사용하는 기술 스택, 최근 공부하고 있는 것만 자연스럽게 묻는다.
> 이 시간에 인터뷰이가 얼마나 기술서를 읽는 사람인지 가늠한다.

---

## IQ1. 가장 최근 기술서 구매 경험 (H1·H2·H3 동시 검증)

**"가장 최근에 기술서를 구매하셨을 때 이야기를 처음부터 말씀해주세요.
어떻게 그 책을 알게 됐고, 어떻게 구매하셨나요?"**

파고들기:
- "그 책을 처음 알게 된 계기가 뭐였나요?"
- "그 책을 선택한 이유가 뭐였나요? 다른 책들과 어떻게 비교하셨나요?"

> **검증 포인트:**
> - 탐색 시작이 기술 키워드인가, 출판사인가, 추천인가 → H1·H3 자연 노출 여부
> - 수준 판단을 어떻게 했는가 → H2 행동 패턴 포착
> **왜 열린 질문으로 시작하는가:** 첫 질문이 좁으면 탐색 경로의 전체 맥락을 잃는다.

---

## IQ2. 원하는 책을 못 찾았던 경험 (H1 검증)

**"찾고 싶은 기술서가 있었는데 못 찾거나, 찾는 데 오래 걸렸던 경험이 있나요?"**

파고들기:
- "어떤 방법으로 찾으려고 하셨나요?"
- "결국 어떻게 해결하셨나요?"

> **검증 포인트:**
> - 어떤 검색 방식이 실패했는가 → 카테고리 필터의 실제 필요성 (H1)
> - 어떤 대안을 썼는가 → 현재 서비스의 공백
> **왜 이렇게 물어보는가:** "카테고리 필터가 있으면 도움이 됐을까요?"가 아니라
> 실패 경험을 통해 실제 니즈를 간접적으로 확인한다.

---

## IQ3. 수준이 안 맞는 책을 산 경험 (H2 검증)

**"사고 나서 '이 책 나한테 너무 어렵다' 또는 '너무 쉽다'고 느꼈던 경험이 있나요?"**

파고들기:
- "구매 전에 어떻게 수준을 판단하려고 하셨나요?"
- "그 방법이 왜 실패했다고 생각하세요?"

> **검증 포인트:**
> - 수준 판단 실패 경험 여부 → H2(Level) 기능의 실질적 필요성
> - 목차 확인, 리뷰 확인 등 현재 행동 → Level 표시로 대체 가능한지
> **왜 실패 경험을 묻는가:** 성공 경험은 기능 필요성을 낮게 만든다.
> 실패 경험이 있어야 개선 니즈가 명확해진다.

---

## IQ4. 출판사가 선택에 영향을 준 경험 (H3 검증)

**"어떤 기술서를 살지 고민할 때, 어느 출판사에서 낸 책인지를 보신 적 있나요?"**

파고들기:
- "그 출판사를 신뢰하게 된 계기가 있었나요?"
- "반대로 출판사를 몰라도 구매한 경우는 어떤 상황이었나요?"

> **검증 포인트:**
> - 출판사가 실제 결정 요소로 작동하는지 → H3(Publisher 컨텍스트) 중요도
> - 어떤 출판사들이 신뢰받는가 → Publisher 데이터 설계에 반영
> **왜 "영향을 주나요?" 대신 "경험이 있나요?"로 묻는가:**
> 영향 여부는 이미 yes-bias를 포함한다. 경험 자체를 물어 진짜 행동을 파악한다.

---

## IQ5. 나중에 사려고 기억해둔 책 (H4 검증)

**"'이 책 나중에 사야지'라고 생각했는데 잊어버리거나, 반대로 잘 관리하고 계신 경험이 있나요?"**

파고들기:
- "지금은 어떻게 기억해두시나요?"
- "그 방법이 불편하거나 아쉬운 점이 있나요?"

> **검증 포인트:**
> - 현재 찜/위시리스트 대체 수단 → H4(Like) 기능의 실질 대체 가능성
> - 잊어버린 경험 → 기능의 실제 필요 강도 판단
> **왜 "좋아요 기능 쓸 것 같나요?"를 묻지 않는가:**
> 기능 소개 전에 현재 행동을 파악해야 기능의 포지셔닝을 결정할 수 있다.

---

## IQ6. 여러 권 구매 시 불편했던 경험 (H5 검증)

**"기술서를 2권 이상 한 번에 사신 경험이 있나요? 그때 어땠나요?"**

파고들기:
- "배송이나 결제에서 불편했던 점이 있었나요?"
- "여러 권을 한 번에 사고 싶었는데 포기한 적도 있나요?"

> **검증 포인트:**
> - 복수 구매 경험의 실제 불편함 → H5(단일 주문) 설계 타당성
> - 포기 경험 → 현재 서비스의 전환 실패 포인트
> **왜 "단일 주문 vs 분리 주문"을 언급하지 않는가:**
> 용어 자체가 낯설어 답변이 왜곡된다. 경험 기반으로 물어야 한다.

---

## IQ7. 기술서 구매에서 가장 큰 불편함 (종합 — 설계 공백 발견)

**"기술서를 사는 게 번거롭거나 불편하다고 느끼신 게 있다면 솔직하게 말씀해주실 수 있나요?
별것 아닌 것 같아도 괜찮습니다."**

파고들기:
- "그 불편함이 실제로 구매를 포기하게 만든 적이 있나요?"
- "그 문제를 지금 어떻게 해결하고 계세요?"

> **검증 포인트:**
> - 도메인 스팩에 없는 미발견 니즈 포착 → 시퀀스 흐름 추가 여부 결정
> - 구매 포기를 유발하는 가장 큰 장벽 → 우선순위 재조정
> **왜 마지막에 묻는가:** 라포가 형성된 마지막 질문이어야 솔직한 불편함을 말한다.
> 첫 질문에는 "잘 모르겠다"는 답변이 많다.

## 인터뷰 결과 기록 양식

인터뷰 직후 30분 이내 기록한다. 기억이 흐릿해지기 전에.

```
페르소나 ID: [Nemotron-Personas-Korea 데이터셋 ID]
직군:
경력:
인터뷰 일시:
인터뷰어:

── 행동 패턴 ──────────────────────────────
[IQ1] 최근 구매 경로 (기술명 검색 / 출판사 / 추천 / 베스트셀러):
[IQ2] 탐색 실패 경험 여부 및 원인:
[IQ3] 수준 판단 실패 경험 여부:
[IQ4] 출판사 영향 여부 (있음 / 없음 / 조건부):
[IQ5] 관심 도서 관리 방법 (서점 찜 / 외부 도구 / 없음):
[IQ6] 복수 구매 불편 경험:
[IQ7] 발견된 미탐지 불편함:

── 가설 판단 ──────────────────────────────
H1 카테고리:  □ 강한 니즈  □ 보통  □ 니즈 없음
H2 난이도:    □ 강한 니즈  □ 보통  □ 니즈 없음
H3 출판사:    □ 강한 니즈  □ 보통  □ 니즈 없음
H4 좋아요:    □ 강한 니즈  □ 보통  □ 니즈 없음
H5 단일 주문: □ 강한 니즈  □ 보통  □ 니즈 없음
H6 쿠폰:      □ 강한 니즈  □ 보통  □ 니즈 없음

── 시퀀스 반영 메모 ────────────────────────
(인터뷰에서 발견된 흐름 중 시퀀스에 추가하거나 제거할 것)
```

## 리서치 완료 후 → 시퀀스 다이어그램 반영 체계

```
설문 결과 분석 + 인터뷰 5건 완료
          ↓
가설별 채택 / 기각 / 보류 판단 (다수결 기준)
          ↓
기각된 가설 → 해당 기능 우선순위 하향 또는 설계 수정
채택된 가설 → 해당 흐름을 시퀀스에 포함
          ↓
02-sequence-diagrams.md 우선순위 결정:
  1순위: 주문 흐름 (재고 확인 · 차감 · 스냅샷) — 핵심 비즈니스
  2순위: 도서 탐색 (카테고리 · 난이도 필터)   — H1·H2 채택 시
  3순위: 좋아요 토글                           — H4 채택 시
  4순위: 회원가입 + 쿠폰 자동 발급             — H6 채택 시
```

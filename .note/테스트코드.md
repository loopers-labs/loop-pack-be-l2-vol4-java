# 테스트 코드 작성 가이드

이 문서는 앞으로 테스트 코드를 작성할 때 따를 기준이다.
테스트는 구현을 보조하는 코드가 아니라, 요구사항과 설계 의도를 검증하는 실행 가능한 문서로 작성한다.

## 핵심 원칙

- 모든 테스트는 **3A 원칙**을 따른다.
  - Arrange: 테스트에 필요한 상태와 의존성을 준비한다.
  - Act: 검증할 동작을 한 번 수행한다.
  - Assert: 결과, 상태 변화, 예외, 호출 여부를 검증한다.
- 테스트 이름과 `@DisplayName`은 요구사항을 설명해야 한다.
- 실제 동작하지 않는 코드나 의미 없는 Mock 데이터로 테스트를 통과시키지 않는다.
- 테스트 대상의 책임만 검증한다.
- `println`은 테스트 코드에 남기지 않는다.
- Java에서는 null 반환보다 `Optional`을 우선 고려한다.
- 테스트는 서로 독립적으로 실행되어야 한다.
- 테스트는 실행 순서에 의존하지 않아야 한다.

## TDD 흐름

기본 개발 흐름은 **Red > Green > Refactor**를 따른다.

| 단계 | 목적 | 작업 |
| --- | --- | --- |
| Red | 실패하는 테스트 작성 | 요구사항을 가장 작은 단위의 테스트로 표현한다. |
| Green | 테스트 통과 | 테스트를 통과시키는 최소 구현을 작성한다. |
| Refactor | 품질 개선 | 중복 제거, 이름 개선, 구조 개선 후 모든 테스트를 다시 실행한다. |

TDD의 핵심은 테스트를 먼저 쓰는 행위 자체보다, 요구사항을 작게 나누고 검증 가능한 구조로 설계하는 것이다.

## 테스트 종류

### 1. 단위 테스트(Unit Test)

| 항목 | 기준 |
| --- | --- |
| 대상 | Entity, VO, Domain Service 등 순수 도메인 로직 |
| 목적 | 비즈니스 규칙과 순수 로직의 정합성 검증 |
| 환경 | Spring 없이 JVM에서 빠르게 실행 |
| 의존성 | Fake, Stub 등 테스트 더블로 대체 |
| 예시 | 포인트 충전 시 최대 한도 초과 여부 검증 |

단위 테스트는 가장 빠르고 많이 작성되어야 한다.
Spring Context가 필요하지 않다면 `@SpringBootTest`를 사용하지 않는다.

### 2. 통합 테스트(Integration Test)

| 항목 | 기준 |
| --- | --- |
| 대상 | Service, Facade, Repository 등 여러 컴포넌트가 연결된 흐름 |
| 목적 | 실제 Bean, DB, 트랜잭션, 저장 결과 검증 |
| 환경 | `@SpringBootTest`, Test DB, Testcontainers 등 |
| 의존성 | 외부 API는 Stub 또는 테스트용 대역으로 통제 |
| 예시 | 포인트 충전 요청 후 DB 반영과 이벤트 발행 검증 |

통합 테스트는 단위 테스트보다 느리다.
따라서 실제 연결이 중요한 시나리오에만 사용한다.

### 3. E2E 테스트(End-to-End Test)

| 항목 | 기준 |
| --- | --- |
| 대상 | Controller > Service > DB까지 이어지는 전체 API 흐름 |
| 목적 | 실제 HTTP 요청 기준의 사용자 시나리오 검증 |
| 환경 | `@SpringBootTest(webEnvironment = RANDOM_PORT)`, `TestRestTemplate`, `MockMvc` 등 |
| 예시 | 회원가입 > 포인트 충전 > 주문 흐름을 HTTP 요청으로 검증 |

E2E 테스트는 실제 API 계약을 검증하는 용도로 작성한다.
개발 완료된 API는 가능하면 `.http/**.http`에도 요청 예시를 분류해 작성한다.

## 테스트 작성 구조

테스트 클래스는 기능 단위로 `@Nested`를 사용해 그룹화한다.

```java
class UserServiceTest {

    @DisplayName("사용자를 조회할 때")
    @Nested
    class GetUser {

        @DisplayName("존재하는 사용자 ID를 주면 사용자를 반환한다.")
        @Test
        void returnsUser_whenUserIdExists() {
            // arrange
            UserRepository userRepository = new FakeUserRepository();
            UserService userService = new UserService(userRepository);
            UserModel user = userRepository.save(createUser("user1"));

            // act
            UserModel result = userService.getUser("user1", "Password1!");

            // assert
            assertThat(result).isSameAs(user);
        }
    }
}
```

## 테스트 이름 규칙

메서드 이름은 다음 형식을 권장한다.

```text
expectedResult_whenCondition
```

예시:

```java
void returnsUser_whenUserIdExists()
void throwsUserNotFound_whenUserIdDoesNotExist()
void chargesPoint_whenAmountIsValid()
void throwsBadRequest_whenAmountIsNegative()
```

`@DisplayName`은 한국어로 요구사항을 자연스럽게 설명한다.

```java
@DisplayName("존재하지 않는 사용자 ID를 주면 USER_NOT_FOUND 예외가 발생한다.")
```

## Assertion 기준

기본 검증은 AssertJ를 우선 사용한다.

```java
assertThat(result.getUserId()).isEqualTo("user1");
assertThat(result.getPoint()).isEqualByComparingTo(BigDecimal.valueOf(1000));
```

여러 속성을 함께 검증할 때는 `assertAll`을 사용할 수 있다.

```java
assertAll(
    () -> assertThat(result.getUserId()).isEqualTo("user1"),
    () -> assertThat(result.getEmail()).isEqualTo("user1@example.com")
);
```

예외는 `assertThrows`로 검증하고, 예외 타입뿐 아니라 에러 코드도 함께 확인한다.

```java
CoreException exception = assertThrows(CoreException.class, () -> {
    userService.getUser("unknown", "Password1!");
});

assertThat(exception.getErrorType()).isEqualTo(ErrorType.USER_NOT_FOUND);
```

## 테스트 더블(Test Double)

테스트 더블은 테스트 대상이 의존하는 외부 객체의 동작을 빠르고 안전하게 대체하는 객체다.

`Stub`, `Mock`, `Spy`, `Fake`는 목적이고, `mock()`, `spy()`는 객체를 만드는 도구다.

```java
UserRepository repository = mock(UserRepository.class); // 도구: mock()
when(repository.findByUserId("user1")).thenReturn(Optional.of(user)); // 역할: Stub
verify(repository).findByUserId("user1"); // 역할: Mock
```

| 역할 | 목적 | 사용 방식 | 예시 |
| --- | --- | --- | --- |
| Dummy | 자리만 채우는 객체 | 생성자나 메서드 인자로 전달 | 사용되지 않는 값 객체 |
| Stub | 고정된 응답 제공 | `when().thenReturn()` | 조회 시 특정 사용자 반환 |
| Mock | 호출 여부와 횟수 검증 | `verify()` | 이벤트 발행 호출 여부 검증 |
| Spy | 실제 객체를 감싸 일부만 조작 | `spy()`, `doReturn()` | 실제 로직 사용 중 일부 메서드만 대체 |
| Fake | 실제처럼 동작하는 단순 구현체 | 직접 클래스 구현 | `InMemoryUserRepository` |

### Stub 예시

```java
UserRepository userRepository = mock(UserRepository.class);
when(userRepository.findByUserId("user1")).thenReturn(Optional.of(user));
```

흐름을 통제하고 싶을 때 사용한다.
"이렇게 호출하면 이렇게 응답한다"를 표현한다.

### Mock 예시

```java
EventPublisher eventPublisher = mock(EventPublisher.class);

pointService.charge("user1", BigDecimal.valueOf(1000));

verify(eventPublisher, times(1)).publish(any(PointChargedEvent.class));
```

호출 자체가 중요한 요구사항일 때만 사용한다.
상태 검증으로 충분하다면 `verify()`를 남용하지 않는다.

### Spy 예시

```java
PointCalculator calculator = spy(new PointCalculator());

doReturn(BigDecimal.ZERO).when(calculator).calculateBonus(any());

BigDecimal result = calculator.calculateTotal(BigDecimal.valueOf(1000));
```

실제 객체의 대부분 동작은 유지하고 일부 동작만 바꾸고 싶을 때 사용한다.
가능하면 Spy보다 명확한 Fake나 Stub을 우선 고려한다.

### Fake 예시

```java
private static class FakeUserRepository implements UserRepository {
    private final Map<String, UserModel> users = new HashMap<>();

    @Override
    public UserModel save(UserModel user) {
        users.put(user.getUserId(), user);
        return user;
    }

    @Override
    public Optional<UserModel> findByUserId(String userId) {
        return Optional.ofNullable(users.get(userId));
    }
}
```

DB 없이 독립적으로 테스트 가능한 저장소가 필요할 때 사용한다.

## 계층별 작성 기준

| 계층 | 권장 테스트 | 검증 내용 |
| --- | --- | --- |
| Domain Model | 단위 테스트 | 생성 규칙, 상태 변경, 불변식, 예외 |
| Domain Service | 단위 테스트 | 비즈니스 규칙, 정책 분기 |
| Application Service | 통합 테스트 | Repository 연동, 트랜잭션, 전체 유스케이스 |
| Repository | 통합 테스트 | 실제 쿼리, 저장, 조회, 제약 조건 |
| Controller API | E2E 테스트 | HTTP 상태 코드, 응답 본문, 요청 검증 |

## 테스트 데이터 기준

- 테스트마다 필요한 데이터만 직접 만든다.
- 의미 없는 임의 값보다 요구사항을 드러내는 값을 사용한다.
- 반복되는 생성 코드는 `private` 헬퍼 메서드나 Fixture로 분리할 수 있다.
- Fixture는 테스트 가독성을 높일 때만 사용한다.
- 테스트 데이터는 테스트 간 공유하지 않는다.

좋은 예:

```java
UserModel user = createUser("user1", "Password1!", "user1@example.com");
```

피해야 할 예:

```java
UserModel user = new UserModel(null, null, null, null, null);
```

## 테스트 작성 체크리스트

- [ ] 실패하는 테스트를 먼저 작성했는가?
- [ ] 테스트가 3A 구조로 읽히는가?
- [ ] 테스트 이름이 기대 결과와 조건을 설명하는가?
- [ ] `@DisplayName`이 요구사항을 설명하는가?
- [ ] Spring Context가 필요 없는 테스트에 `@SpringBootTest`를 사용하지 않았는가?
- [ ] Mock보다 상태 검증, Fake, Stub을 우선 고려했는가?
- [ ] 예외 테스트에서 에러 타입까지 검증했는가?
- [ ] 테스트 간 실행 순서나 공유 상태에 의존하지 않는가?
- [ ] `println`, unused import, 불필요한 private 메서드를 제거했는가?
- [ ] 전체 테스트가 통과하는가?

## 우선순위

1. 실제 동작하는 해결책만 테스트한다.
2. null-safety와 thread-safety를 고려한다.
3. 테스트 가능한 구조로 설계한다.
4. 기존 코드 패턴을 분석하고 일관성을 유지한다.

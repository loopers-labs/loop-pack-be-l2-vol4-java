# 프로젝트 구조 정리

이 문서는 처음 프로젝트를 볼 때 "어디에 무엇을 만들면 되는지"를 빠르게 확인하기 위한 메모입니다.

## 전체 구조

```text
root
|-- apps
|   |-- commerce-api       # 사용자의 HTTP 요청을 받는 API 서버
|   |-- commerce-batch     # 배치 작업용 애플리케이션
|   `-- commerce-streamer  # 스트리밍/메시지 처리용 애플리케이션
|-- modules
|   |-- jpa                # JPA 설정, BaseEntity, 테스트 컨테이너
|   |-- redis              # Redis 설정
|   `-- kafka              # Kafka 설정
|-- supports
|   |-- jackson            # JSON 직렬화/역직렬화 보조 설정
|   |-- logging            # 로깅 설정
|   `-- monitoring         # 모니터링 설정
|-- http                   # API 요청 예시 파일
`-- docker                 # 로컬 인프라 실행용 docker-compose 파일
```

평소 기능 개발은 대부분 `apps/commerce-api` 안에서 진행합니다.

## commerce-api 패키지 역할

```text
apps/commerce-api/src/main/java/com/loopers
|-- interfaces      # 외부 요청/응답 계층
|-- application     # 화면/API 요구사항에 맞게 도메인 기능을 조합
|-- domain          # 핵심 비즈니스 규칙
|-- infrastructure  # DB, 외부 시스템 같은 실제 구현체
`-- support         # 예외, 공통 유틸 등 API 내부 지원 코드
```

## 요청 흐름

보통 API 하나는 아래 방향으로 흐릅니다.

```text
Controller
  -> Facade 또는 Service
  -> Domain Model / Domain Service
  -> Domain Repository interface
  -> Infrastructure Repository 구현체
  -> JpaRepository
  -> DB
```

현재 상품 예시는 이 구조를 따릅니다.

```text
interfaces/api/product/ProductV1Controller
application/product/ProductFacade
domain/product/ProductService
domain/product/ProductRepository
infrastructure/product/ProductRepositoryImpl
infrastructure/product/ProductJpaRepository
domain/product/ProductModel
```

## 각 계층에서 하는 일

### interfaces

`Controller`, `Dto`를 둡니다.

- HTTP 요청을 받습니다.
- 요청 DTO를 검증합니다.
- 응답 DTO로 변환합니다.
- 비즈니스 규칙을 직접 처리하지 않습니다.

### application

`Facade`, `Info`를 둡니다.

- 여러 도메인 기능을 조합합니다.
- API 응답에 필요한 형태로 결과를 정리합니다.
- 단순 CRUD만 있으면 생략될 수도 있습니다.

### domain

`Model`, `Service`, `Repository interface`를 둡니다.

- 핵심 규칙은 여기에 둡니다.
- 엔티티 검증, 상태 변경 메서드는 `Model`에 둡니다.
- 트랜잭션 단위의 기능은 `Service`에 둡니다.
- DB 기술에 직접 의존하지 않는 Repository 인터페이스를 둡니다.

### infrastructure

`JpaRepository`, `RepositoryImpl`을 둡니다.

- 실제 DB 저장/조회 코드를 둡니다.
- Spring Data JPA는 여기에 둡니다.
- domain의 Repository 인터페이스를 구현합니다.

### modules

공통 인프라 설정을 제공합니다.

- `modules/jpa`: JPA 설정, `BaseEntity`
- `modules/redis`: Redis 설정
- `modules/kafka`: Kafka 설정

`UserModel`, `ProductModel`처럼 DB에 저장되는 엔티티는 `BaseEntity`를 상속하고, `BaseEntity`의 `Long id`를 기본 PK로 사용합니다.

## 헷갈리기 쉬운 부분: Service와 RepositoryImpl

처음에는 `Service`와 `RepositoryImpl`이 둘 다 중간에 있는 클래스처럼 보여서 헷갈릴 수 있습니다.
하지만 둘은 책임이 완전히 다릅니다.

```text
Service
= 기능을 완성하기 위한 순서와 흐름을 담당

RepositoryImpl
= Repository 인터페이스를 실제 DB/JPA 코드로 구현
```

현재 프로젝트 기준으로 보면 `Impl`이 붙은 것은 Service가 아니라 Repository입니다.

```text
domain/product/ProductService
domain/example/ExampleService
domain/user/UserService

infrastructure/product/ProductRepositoryImpl
infrastructure/example/ExampleRepositoryImpl
```

즉, 이 프로젝트는 보통 아래처럼 이해하면 됩니다.

```text
Model
= 객체 하나가 스스로 지켜야 하는 규칙

Service
= 하나의 기능을 완성하기 위한 작업 순서

Repository interface
= domain이 저장소에 바라는 기능 목록

RepositoryImpl
= 그 저장소 기능을 JPA로 실제 구현한 클래스

JpaRepository
= Spring Data JPA가 제공하는 DB 접근 도구
```

예를 들어 유저 비밀번호 변경 기능은 아래처럼 나뉩니다.

```text
UserService
1. userId로 유저를 찾는다.
2. 없으면 예외를 던진다.
3. 찾은 유저에게 비밀번호 변경을 요청한다.
4. 트랜잭션 안에서 변경 내용이 저장되게 한다.

UserModel
1. 현재 비밀번호가 맞는지 확인한다.
2. 새 비밀번호가 기존 비밀번호와 같은지 확인한다.
3. 새 비밀번호 형식이 올바른지 확인한다.
4. 자신의 password 값을 바꾼다.

UserRepository
1. 유저를 저장할 수 있어야 한다.
2. userId로 유저를 찾을 수 있어야 한다.

UserRepositoryImpl
1. UserRepository가 약속한 기능을 JPA로 실행한다.
```

코드로 보면 `UserService`는 기능의 흐름을 표현합니다.

```java
@Transactional
public UserModel changePassword(String userId, String currentPassword, String newPassword) {
    UserModel user = userRepository.findByUserId(userId)
        .orElseThrow(() -> new CoreException(ErrorType.NOT_FOUND));

    user.changePassword(currentPassword, newPassword);
    return user;
}
```

반면 `UserRepositoryImpl`은 DB 접근을 JPA에 위임합니다.

```java
@Override
public Optional<UserModel> findByUserId(String userId) {
    return userJpaRepository.findByUserId(userId);
}
```

쉽게 비유하면 아래와 같습니다.

```text
Controller
= 요청을 받는 곳

Service
= 기능의 순서를 정하고 실행하는 곳

Repository
= 저장소에 요청하는 창구

RepositoryImpl
= 실제 DB/JPA로 처리하는 곳
```

따라서 `Service`는 "DB를 어떻게 조회할지"를 자세히 알 필요가 없습니다.
그저 `UserRepository`에게 "userId로 찾아줘", "저장해줘"라고 요청합니다.

반대로 `RepositoryImpl`은 "비밀번호 변경 기능이 어떤 순서로 진행되는지"를 알 필요가 없습니다.
그저 JPA를 사용해서 저장하고 조회하는 역할만 합니다.

정리하면 아래 한 줄로 볼 수 있습니다.

```text
Service는 기능의 흐름, RepositoryImpl은 DB 접근 구현입니다.
```

## Repository 작성 기준

이 프로젝트에서는 domain repository와 JPA repository를 분리하는 방식이 자연스럽습니다.

domain:

```java
public interface UserRepository {
    UserModel save(UserModel user);
    Optional<UserModel> findByUserId(String userId);
}
```

infrastructure:

```java
public interface UserJpaRepository extends JpaRepository<UserModel, Long> {
    Optional<UserModel> findByUserId(String userId);
    boolean existsByUserId(String userId);
}
```

```java
@Component
@RequiredArgsConstructor
public class UserRepositoryImpl implements UserRepository {
    private final UserJpaRepository userJpaRepository;

    @Override
    public UserModel save(UserModel user) {
        return userJpaRepository.save(user);
    }

    @Override
    public Optional<UserModel> findByUserId(String userId) {
        return userJpaRepository.findByUserId(userId);
    }
}
```

주의할 점:

- `JpaRepository<엔티티, ID타입>` 순서입니다.
- 이 프로젝트의 엔티티 ID 타입은 보통 `Long`입니다.
- 조회 결과가 없을 수 있으면 `Optional`을 사용합니다.
- `find(String userId)`보다 `findByUserId(String userId)`처럼 필드명을 명확히 씁니다.
- `save`는 저장된 엔티티를 반환하도록 둡니다.
- `update()` 같은 빈 메서드는 만들지 않습니다.

## User 기능을 만들 때 위치

가입, 조회, 비밀번호 변경을 만든다면 보통 아래처럼 나눕니다.

```text
domain/user/UserModel
domain/user/UserService
domain/user/UserRepository
infrastructure/user/UserJpaRepository
infrastructure/user/UserRepositoryImpl
interfaces/api/user/UserV1Controller
interfaces/api/user/UserV1Dto
```

기능별 책임:

- 가입: `new UserModel(...)`로 도메인 객체 생성 후 `userRepository.save(user)`
- 조회: `userRepository.findByUserId(userId)` 후 없으면 예외 처리
- 비밀번호 변경: 조회한 `UserModel`의 `changePassword(currentPassword, newPassword)` 호출

비밀번호 변경은 별도 `update()` 쿼리를 만들기보다, 트랜잭션 안에서 엔티티 상태를 바꾸면 JPA dirty checking으로 반영됩니다.

```java
@Transactional
public UserModel changePassword(String userId, String currentPassword, String newPassword) {
    UserModel user = userRepository.findByUserId(userId)
        .orElseThrow(() -> new CoreException(ErrorType.NOT_FOUND));

    user.changePassword(currentPassword, newPassword);
    return user;
}
```

## 테스트 위치

```text
apps/commerce-api/src/test/java/com/loopers
|-- domain       # 도메인 모델/서비스 테스트
|-- application  # facade 테스트
|-- interfaces   # API E2E 테스트
`-- support      # 공통 지원 코드 테스트
```

테스트는 3A 원칙으로 작성합니다.

```text
Arrange: 테스트 준비
Act: 실행
Assert: 결과 검증
```

## 기능 추가 순서

새 기능을 만들 때는 아래 순서로 보면 됩니다.

1. `domain`에 핵심 규칙을 먼저 작성합니다.
2. 실패하는 테스트를 먼저 작성합니다.
3. `Repository interface`를 domain에 둡니다.
4. 실제 JPA 구현은 `infrastructure`에 둡니다.
5. API가 필요하면 `interfaces/api`에 Controller와 DTO를 둡니다.
6. 여러 기능을 조합해야 하면 `application`에 Facade를 둡니다.
7. 개발 완료 후 필요하면 `http` 폴더에 API 요청 예시를 추가합니다.

## 자주 하는 실수

- domain에 `JpaRepository`를 바로 상속해서 기술 구현을 섞는 것
- `JpaRepository<String, UserModel>`처럼 제네릭 순서를 반대로 쓰는 것
- 조회 결과가 없을 수 있는데 `UserModel`을 바로 반환하는 것
- 비밀번호 변경을 repository의 `update()`로 처리하려는 것
- 엔티티에 DB 제약 조건을 빼먹는 것
- 테스트 없이 구조부터 크게 만드는 것

처음에는 상품 코드(`product`)를 기준으로 복사해서 `user`에 맞게 바꾸면 구조를 이해하기 쉽습니다.

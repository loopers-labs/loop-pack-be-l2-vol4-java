## 도메인 & 객체 설계 전략
- 도메인 객체는 비즈니스 규칙을 캡슐화해야 합니다.
- 애플리케이션 서비스는 서로 다른 도메인을 조립해, 도메인 로직을 조정하여 기능을 제공해야 합니다.
- 규칙이 여러 서비스에 나타나면 도메인 객체에 속할 가능성이 높습니다.
- 각 기능에 대한 책임과 결합도에 대해 개발자의 의도를 확인하고 개발을 진행합니다.

### 세부 도메인 원칙
1. **재고(Stock) 관리**: 상품(Product)과 재고(Stock)는 1:1 관계이나 독립된 엔티티로 관리한다. 재고 차감 및 동시성 제어 로직은 `Stock` 엔티티와 `StockService`에서 전담한다.
2. **좋아요(Like) 정렬**: 상품 목록의 정렬 성능(likes_desc)을 위해 `Product` 엔티티 내에 `likeCount` 필드를 반정규화하여 관리한다. 좋아요 등록/취소 시 도메인 서비스를 통해 실시간으로 업데이트한다.
3. **주문 불변성(Snapshot)**: 주문 시점의 정보를 보존하기 위해 `OrderItem` 내에 `@Embeddable`로 정의된 `ProductSnapshot` Value Object(VO)를 사용한다.
4. **재고 검증**: 재고의 음수 방지 처리는 `Stock` 도메인 객체 내부에서 원자적으로 처리하며, 부족 시 도메인 예외를 발생시킨다.

## 아키텍처, 패키지 구성 전략
- 본 프로젝트는 레이어드 아키텍처를 따르며, DIP (의존성 역전 원칙) 을 준수합니다. (Application → Domain ← Infrastructure)
- API request, response DTO와 응용 레이어의 DTO는 분리해 작성하도록 합니다.
- 패키징 전략은 4개 레이어 패키지를 두고, 하위에 도메인 별로 패키징하는 형태로 작성합니다.

### 레이어별 책임
1. **Interfaces Layer (API)**: 요청 수신, 응답 반환, 단순한 입력 유효성 검사.
2. **Application Layer (Facade)**: 여러 도메인 서비스를 조합(Orchestration)하여 유스케이스 구현. 트랜잭션의 경계가 되며 DTO 변환을 수행한다.
3. **Domain Layer**: 핵심 비즈니스 로직(Entity, VO)과 도메인 서비스 위치. Repository 인터페이스를 정의한다.
4. **Infrastructure Layer**: JPA, Redis 등 기술 구현체 제공. 외부 연동 처리.

### 협력 흐름 원칙
- **애그리거트 및 연관관계 매핑**: 
  - `Order` ↔ `OrderItem` (1:N), `Product` ↔ `Stock` (1:1)과 같이 도메인 응집도가 높고 생명주기를 함께하는 경우에는 양방향 매핑을 사용하여 객체 탐색을 용이하게 한다.
  - 애그리거트 루트(부모)가 자식의 생명주기를 전적으로 관리하는 경우, 자식 엔티티(`Stock`, `OrderItem`)는 별도의 타임스탬프(`BaseTimeEntity`)를 가지지 않고 순수 필드 위주로 구성하여 경량화한다.
  - `Brand` ↔ `Product`와 같이 데이터 양이 많거나 성능 이슈가 예상되는 관계는 `Product`에서 `brandId`를 참조하는 등 단방향 또는 ID 참조를 지향한다.
- **영속성 전이(Cascade) 전략**:
  - **주문 애그리거트**: `Order` → `OrderItem`은 생성 시의 편의성을 위해 `CascadeType.PERSIST`만 적용한다. 주문 상품은 임의로 삭제되어서는 안 되므로 `orphanRemoval`은 사용하지 않는다.
  - **상품 애그리거트**: `Product` → `Stock`은 강력한 1:1 결합이므로 `CascadeType.ALL`을 적용한다.
  - **브랜드-상품 관계**: 대규모 데이터 보호를 위해 영속성 전이를 사용하지 않으며, 필요 시 도메인 서비스에서 벌크 연산으로 처리한다.
- **물리 제약조건 방지**: 모든 연관관계 매핑 시 `foreignKey = @ForeignKey(ConstraintMode.NO_CONSTRAINT)`를 사용하여 DB 레벨의 물리적 외래키 생성을 방지하고 논리적 매핑만 유지한다.
- **Facade 패턴**: 복합 도메인 정보가 필요한 경우(예: 상품 상세 조회) Facade에서 각 도메인 서비스를 호출하여 결과를 조립한다.
- **상태 없음**: 도메인 서비스는 상태를 가지지 않으며, 도메인 객체 간의 협력을 조율하는 역할에 집중한다.

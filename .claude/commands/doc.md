구현 전 명세부터 구현 중 분석, 구현 결과 정리까지 전 단계를 마크다운 문서로 작성하는 범용 문서화 스킬입니다.

## 입력
`$ARGUMENTS` — 문서화 종류 키워드:
- `spec <기능명>` : 구현 전 — 요구사항 명세
- `impl <파일/클래스>` : 구현 중 — 핵심 로직·설계 분석
- `review` : 구현 중/후 — 현재 브랜치 코드 리뷰
- `result <기능명>` : 구현 후 — 구현 결과 정리 및 변경 이력
- (없음) : 현재 컨텍스트 기반으로 가장 적합한 종류 추천 후 작성

## 문서 템플릿

### `spec` — 요구사항 명세 (구현 전)
```markdown
# [기능명] 요구사항 명세

## 배경 및 목적
## 기능 요구사항
| ID | 요구사항 | 우선순위 | 비고 |
|----|----------|----------|------|
## 비기능 요구사항 (성능, 보안, 가용성)
## 제약 조건
## 수용 기준 (Acceptance Criteria)
## 미결 사항 (Open Questions)
```

### `impl` — 구현 내용 분석 (구현 중)
```markdown
# [클래스/기능] 구현 분석

## 개요
## 모듈 위치 및 의존성
## 핵심 로직 흐름 (순서도 또는 단계 설명)
## 주요 설계 결정 및 이유
## 예외 처리 전략
## 성능 고려사항 (쿼리, 캐싱, 트랜잭션)
## 개선 가능 영역
```

### `review` — 코드 리뷰 (구현 중/후)
```markdown
# 코드 리뷰: [브랜치/기능명]

## 변경 요약
## 잘된 점
## 개선 제안
| 파일:줄 | 심각도 | 내용 | 제안 |
|---------|--------|------|------|
## 보안/성능 위험
## 테스트 커버리지 평가
## 종합 의견
```

### `result` — 구현 결과 정리 (구현 후)
```markdown
# [기능명] 구현 결과

## 구현 범위 요약
## 변경된 파일 목록
| 파일 | 변경 유형 | 주요 내용 |
|------|-----------|-----------|
## 주요 설계 결정 및 이유
## 알려진 한계 및 미구현 항목
## 테스트 현황
## 후속 작업 (있을 경우)
```

## 절차
1. `$ARGUMENTS` 파싱 → 문서 종류 결정
2. 관련 소스 파일 / `git diff` / CLAUDE.md 읽기
3. 해당 템플릿으로 마크다운 초안 작성
4. 저장 경로 제안: `docs/<type>/<이름>.md`
5. 실제 파일 저장은 개발자 승인 후 실행

## 제약
- 문서에 불확실한 내용은 `[확인 필요]`로 명시
- 구현 추정은 실제 코드 기반으로만 서술 (가정 금지)
- 프로덕션 코드 수정 금지 — 문서 작성에만 집중

구현 전 단계에서 요구사항을 분석하고, to-do list와 실행 계획을 수립합니다.

## 입력
`$ARGUMENTS` — 구현하려는 기능이나 목표를 자연어로 기술. 상세할수록 좋지만 한 줄도 가능.

## 절차

### 1. 요구사항 분석
- 기술된 내용에서 **기능 요구사항**과 **비기능 요구사항** 분리
- 모호하거나 미결인 부분은 `[확인 필요]`로 표시하고 질문 목록 제시
- 기존 코드베이스 및 모듈 구조(CLAUDE.md)와의 연관성 파악

### 2. 영향 범위 분석
- 변경/추가될 모듈, 클래스, 테이블, API 엔드포인트 목록
- 기존 코드와의 의존성 및 충돌 가능성
- 성능·보안·트랜잭션 고려사항

### 3. To-do list 생성
우선순위와 의존 관계를 반영한 체크리스트:
```
[ ] 1. 도메인 모델 설계 (Entity, VO)
[ ] 2. Repository 구현 (QueryDSL 포함)
[ ] 3. Service 로직 구현
[ ] 4. Controller / API 설계
[ ] 5. 테스트 작성 (단위 → 통합 → E2E)
[ ] 6. http/*.http 작성
```
각 항목에 **예상 난이도** (하/중/상) 와 **선행 조건** 명시.

### 4. 실행 계획 요약
```markdown
## [기능명] 구현 계획

### 목표
### 영향 모듈
### To-do list
### 기술 결정 사항 및 근거
### 위험 요소 및 대응
### 완료 기준
```

## 출력 방식
- 분석 결과를 먼저 채팅으로 요약 제시
- `[확인 필요]` 항목이 있으면 **개발자 답변 대기 후** 계획 확정
- 최종 계획 문서는 개발자 승인 후 `docs/plan/<기능명>.md`에 저장

## 제약
- 계획 단계에서 프로덕션 코드 작성 금지
- 방향성·주요 결정은 제안만 하고 개발자 승인 후 수행
- 과도한 추상화나 미래 요구사항 예측 금지 — 현재 요청 범위에 집중

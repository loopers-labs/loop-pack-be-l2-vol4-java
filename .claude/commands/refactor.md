대상 클래스/메서드를 성능 및 설계(OOP·함수형) 관점에서 분석하고 리팩터링합니다.

## 환경
- Java 21 (corretto-21), Spring Boot 3.4.4
- QueryDSL 5.1.0, Spring Data JPA, Redis (Lettuce)

## 입력
`$ARGUMENTS` — 리팩터링 대상 파일 경로 또는 클래스명. 없으면 현재 컨텍스트에서 추론.

## 절차

### 1. 분석 (수행 전 개발자에게 결과 보고 후 승인 대기)

**성능 측면**
- N+1 쿼리, 불필요한 DB 왕복 → QueryDSL fetch join / projection 제안
- Redis 캐싱 미적용 구간 식별
- 불필요한 객체 생성·복사 (stream 중간 연산, DTO 변환 체인)
- 트랜잭션 범위 과잉/과소 여부

**설계 측면 (OOP)**
- SRP 위반: 책임이 2개 이상인 클래스/메서드
- 상속보다 조합이 적합한 구간
- 불변 객체 적용 가능 여부 (Java 21 `record`, `sealed` 활용)
- 패키지 의존성 방향 위반

**설계 측면 (함수형)**
- 명령형 루프 → Stream API 전환 가능 구간
- `Optional` 체이닝으로 null-safe하게 개선 가능한 구간
- 순수 함수(side-effect 없음)로 추출 가능한 로직

### 2. 리팩터링 실행 (승인된 항목만)
- 변경 범위를 최소화하고, 기존 코드 패턴과 일관성 유지
- `System.out.println` 제거
- 각 변경마다 한 줄 이유 주석 추가 금지 — WHY가 자명하지 않을 때만 최소한으로

### 3. 검증
- 기존 테스트가 통과하는지 확인: `./gradlew test`
- 테스트 없으면 `/test` 커맨드로 핵심 케이스 먼저 작성 제안

## 제약
- 방향성·주요 의사결정은 제안만 하고, 개발자 승인 후 수행
- 요청하지 않은 기능 추가 또는 테스트 삭제 금지

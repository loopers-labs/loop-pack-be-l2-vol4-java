## 실행 순서

1. `git status`로 변경된 파일 목록을 파악한다
2. `git diff` 및 `git diff --staged`로 실제 변경 내용을 확인한다
3. 변경 내용을 분석해 커밋 메시지 초안을 작성하고 사용자에게 보여준다
4. 사용자가 승인하면 스테이징 및 커밋을 진행한다
5. `git push origin {현재브랜치}`로 원격에 푸쉬한다

## 커밋 메시지 규칙

### 형식
```
{type}: {한 줄 요약}

{본문 — 필요한 경우에만}
```

### type 목록
| type | 사용 기준 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 코드 개선 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서 변경 |
| `chore` | 빌드 설정, 의존성 등 |
| `style` | 포맷, 공백 등 코드 스타일만 변경 |

### 메시지 작성 원칙
- **한국어**로 작성한다
- 제목은 50자 이내, 명령형으로 (예: "유저 로그인 API 추가")
- 본문은 "무엇을"이 아닌 **"왜"** 변경했는지 설명한다
- 관련 없는 변경이 섞인 경우 분리해서 커밋할 것을 제안한다

## 절대 금지 (cautions.md 준수)

- 커밋 메시지에 `Co-Authored-By: Claude` 등 **AI 서명 포함 금지**
- `Co-Authored-By:` 형태의 모든 AI 흔적을 커밋 메시지에 남기지 않는다
- `git add -A` 또는 `git add .` 사용 금지 — 파일을 명시적으로 지정한다
- `.env`, 인증 정보 등 민감한 파일 커밋 금지

## 스테이징 규칙

- 변경된 파일 목록을 보여주고 어떤 파일을 포함할지 사용자에게 확인한다
- 테스트 파일과 구현 파일이 함께 변경된 경우 하나의 커밋으로 묶는다
- 무관한 파일이 섞인 경우 분리 커밋을 제안한다

## 사용자 확인 흐름

```
[변경 파일 목록]

📝 커밋 메시지 초안:
  feat: 유저 회원가입 API 추가

포함할 파일:
  - src/main/java/...UserController.java
  - src/main/java/...UserService.java
  - src/test/java/...UserServiceTest.java

→ 진행할까요? (수정사항 있으면 말씀해주세요)
```

  트랜잭션 범위, 영속성 컨텍스트, 쿼리 실행 시점 관점에서 분석한다.

  - 트랜잭션이 불필요하게 크게 잡혀 있지는 않은지
  - 조회/쓰기 로직이 하나의 트랜잭션에 혼합되어 있지는 않은지

  - 락(비관적/낙관적) 보유 구간이 필요 이상으로 길지는 않은지

  개선 가능 지점을 선택적으로 판단할 수 있도록 돕는다.

### 📌 Analysis Scope

이 스킬은 아래 대상에 대해 분석한다.

- `@Transactional` 이 선언된 클래스 / 메서드
- Application Layer (`application.*` 의 Facade) 코드
- JPA Entity, Repository(IF/Impl), QueryDSL/JPQL/네이티브 쿼리 사용 코드
- 하나의 유스케이스(요청 흐름) 단위

> 컨트롤러 → Facade → 도메인 서비스 → 레포지토리 전체 흐름을 기준으로 분석하며
> 특정 메서드만 떼어내어 판단하지 않는다.

**이 프로젝트의 전제** (com.loopers / commerce-api):
- 트랜잭션 경계는 Facade 메서드 단위 (`@Transactional`). 도메인 서비스는 순수 POJO 로 레포 의존이 없다.
- 자원별 동시성 전략: 재고 = 비관적 락(`findAllForUpdate`), 쿠폰 = 낙관적 락(`@Version`),
  좋아요 집계 = 조건부 UPDATE (`product_like_count`).
- 발급 쿠폰은 `CouponSnapshot` 으로 자기완결 — 주문 트랜잭션에서 쿠폰 템플릿을 조회하면 안 된다.

### 🔍 Analysis Checklist

#### 1. Transaction Boundary 분석

다음을 순서대로 확인한다.

- 트랜잭션 시작 지점은 어디인가? (Facade / 그 외 계층 — Controller 의 @Transactional 은 금지 패턴)
- 트랜잭션이 실제로 필요한 작업은 무엇인가? (상태 변경 vs 단순 조회)
- 트랜잭션 내부에서 수행되는 작업을 나열한다.
  - 외부 API 호출 (있으면 즉시 문제 후보 — 트랜잭션 밖 또는 REQUIRES_NEW 분리 검토)
  - 복잡한 조회 / 대량 조회
  - 반복문 기반 처리
- **락 보유 구간**: 비관적 락 획득 시점부터 커밋까지의 구간에 락이 필요 없는 작업
  (인증/검증용 조회, 외부 호출, 무관한 조회)이 끼어 있지 않은가?

**출력 예시**

```markdown
- 현재 트랜잭션 범위: OrderFacade.createOrder()
  ├─ 유저 검증                  (락 없음)
  ├─ 상품 락 조회 (FOR UPDATE)  ← 락 보유 시작
  ├─ 쿠폰 검증/사용
  ├─ 주문 조립 (도메인 서비스)
  └─ 저장 → 커밋               ← 락 해제

- 트랜잭션이 필요한 핵심 작업: 재고 차감, 쿠폰 사용, 주문 저장
```

#### 2. 불필요하게 큰 트랜잭션 식별

아래 패턴이 존재하는지 점검한다.

- Controller 에서 `@Transactional` 이 사용되고 있음
- 읽기 전용 로직이 쓰기 트랜잭션에 포함됨
- 외부 시스템 호출이 트랜잭션 내부에 포함됨
- 트랜잭션 내부에서 대량 조회 / 복잡한 쿼리 실행
- 상태 변경 이후에도 트랜잭션이 길게 유지됨
- 락 획득 **이후**에 락과 무관한 조회가 수행됨 (락 보유 시간 증가)

#### 3. JPA / 영속성 컨텍스트 관점 분석

다음을 중심으로 분석한다.

- Entity 변경이 언제 flush 되는지 (명시 save vs dirty checking — 이 프로젝트는 레포지토리 패턴상 명시 save 컨벤션)
- 조회용 Entity 가 변경 감지 대상이 되는지 (`@Transactional(readOnly = true)` 적용 여부)
- 지연/즉시 로딩 설정으로 인해 트랜잭션 후반에 의도치 않은 쿼리가 발생할 가능성
- `@Modifying` 네이티브/벌크 쿼리가 영속성 컨텍스트와 어긋날 가능성
  (1차 캐시에 로드된 엔티티가 stale 해지는 경로가 있는지)
- N+1: 목록 조회 시 연관 데이터를 루프 단건 조회하는지, 배치(IN) 조회하는지

#### 4. Improvement Proposal (선택적 제안)

개선안은 강제하지 않고 선택지로 제시한다. 각 제안에 trade-off 를 명시한다.

- 트랜잭션 분리 (조회 → 쓰기 분리, Facade orchestration 유지)
- `@Transactional(readOnly = true)` 적용
- 락 획득 시점 조정 (락 무관 작업을 락 획득 전으로 이동)
- DTO Projection (읽기 전용 모델) 도입
- 외부 호출 / 이벤트 발행을 트랜잭션 외부로 이동
- Application / Domain Service 책임 재조정

**개선안 예시**

```markdown
[개선안 1]
- 쿠폰 조회를 상품 락 획득 이전으로 이동
- 효과: FOR UPDATE 락 보유 시간 단축, 무효 쿠폰이면 락 없이 조기 실패

[고려 사항]
- 쿠폰 검증(use)은 여전히 트랜잭션 내에서 수행되어야 함
- 조회 순서 변경이 실패 응답 순서에 영향을 주는지 E2E 로 확인
```

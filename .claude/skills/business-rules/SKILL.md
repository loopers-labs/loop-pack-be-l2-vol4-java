## 목적

코드 작성 전에 비즈니스 규칙을 명확히 정리한다.
레이어별 책임과 트랜잭션 경계를 미리 확정하여 설계 혼란을 방지한다.

## 실행 순서

1. 사용자에게 **도메인명**과 **기능명**을 확인한다 (예: brand / 브랜드 등록)
2. `docs/domain/{domain}.md`가 존재하면 읽어 기존 규칙을 파악한다
3. 아래 항목들을 사용자와 질문/답변으로 확정한다
4. 확정된 내용을 `docs/domain/{domain}.md`에 기록한다

## 확정해야 할 항목

### 1. 입력 필드 및 형식 검증
- 어떤 필드가 필요한가?
- 각 필드의 제약조건은? (null 허용 여부, 길이 제한, 형식 등)
- → **Entity 내부 검증** 대상

### 2. 비즈니스 규칙
- 단순 형식 외에 도메인 규칙이 있는가? (중복 불가, 상태 조건 등)
- 이 규칙이 모든 유스케이스에서 항상 지켜져야 하는가?
- → **DomainService 검증** 대상

### 3. 유스케이스 흐름
- 어떤 순서로 작업이 진행되는가?
- 특정 유스케이스에서만 필요한 조건이 있는가?
- → **ApplicationService** 책임

### 4. 트랜잭션 경계
- 어떤 작업들이 하나의 원자적 단위로 묶여야 하는가?
- 실패 시 롤백 범위는?
- → **@Transactional** 위치 결정

### 5. 접근 제어
- 누가 이 기능을 호출할 수 있는가? (유저/어드민)
- 인증 방식은? (X-Loopers-LoginId / X-Loopers-Ldap)

## 문서 작성 형식

`docs/domain/{domain}.md`에 기능별로 아래 형식으로 추가한다:

```markdown
## {기능명}

### 입력 필드
- {필드명}: {제약조건}

### 비즈니스 규칙 (Entity)
- {Entity 내부에서 검증할 규칙}

### 비즈니스 규칙 (DomainService)
- {외부 의존이 필요한 규칙}

### 유스케이스 흐름 (ApplicationService)
1. {순서대로 작업 나열}

### 트랜잭션 경계
- {어떤 작업들이 하나의 트랜잭션으로 묶이는지}
- {이유}

### 접근 제어
- {호출 가능한 주체 및 인증 방식}
```

## 주의사항

- 규칙이 불명확하면 추측하지 말고 사용자에게 질문한다
- 확정되지 않은 항목은 문서에 포함하지 않는다
- 문서화 완료 후 TDD 순서(Red → Green → Refactor)로 구현을 시작한다

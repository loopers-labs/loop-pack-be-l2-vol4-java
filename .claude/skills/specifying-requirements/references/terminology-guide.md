# 용어 정의 가이드 (단계 2)

용어 정의는 이후 모든 산출물의 일관성을 좌우한다.

## 출력 형식

3-컬럼 표. 영어 컬럼은 **실제 코드/DB/API에서 사용할 명칭**이다.

| 한글 | 영어 | 의미 |
|---|---|---|
| 회원 | `User` | 서비스에 가입하여 식별 가능한 사용자. 본 명세의 도메인 객체 명칭. |
| 회원가입 | `SignUp` | 신규 회원을 생성하는 행위. |
| 로그인 ID | `loginId` | 회원이 본인을 식별하기 위해 사용하는 비즈니스 키. 유니크 제약. |

## 범위 — 무엇을 표에 넣을지

**포함**:
- 도메인 핵심 명사 (`회원`, `상품`, `주문`)
- 도메인 핵심 동사 (`회원가입`, `재고차감`, `결제승인`)
- 도메인 식별자·값 (`loginId`, `productId`, `birthDate`)
- 상태값 / 정책 명칭 (`pending`, `paid`, `passwordPolicy`)
- 도메인 객체 명칭과 짝이 되는 비즈니스 키 (`userId` 내부 PK ↔ `loginId` 비즈니스 키)

**제외**:
- 일반 개념어: **"검증", "해싱", "요청", "응답", "처리", "저장"** — 도메인 특화 의미가 없으면 표에 넣지 마라.
- 자명한 일반 기술 용어: "트랜잭션", "캐시", "타임아웃" 등.
- 인프라 용어: "HTTPS", "JSON" 등.

## 작성 원칙

### 1. 동의어·유사어 정리

자주 헷갈리는 쌍:
- **회원 / 사용자 / 계정** → user / account 중 어느 것이 도메인 객체이고 어느 것이 인증 단위인가?
- **상품 / 품목 / 제품** → product / item / sku 중 카탈로그 단위와 재고 단위 구별
- **주문 / 구매 / 결제** → order / purchase / payment 중 어느 단계?
- **ID / 식별자 / 코드** → 내부 PK인지 사용자 지정 비즈니스 키인지? (`userId` PK ↔ `loginId` 비즈니스 키 처럼 명확히)

### 2. 영어 명칭 케이스

- 자바 변수/필드: `camelCase` (예: `productId`)
- DB 컬럼: `snake_case` (예: `product_id`) — ERD에서 자동 변환
- 에러 코드: `SCREAMING_SNAKE_CASE` (예: `PRODUCT_OUT_OF_STOCK`)
- URL path: `kebab-case` 또는 `snake_case`

표의 영어 컬럼에는 **자바 필드 기준 camelCase**.

### 3. 의미 컬럼은 짧고 명확하게

경계 명시(무엇이 아닌지)는 도움이 되지만 한 문장으로 충분. 길게 풀어쓰지 마라.

좋은 정의:
> "회원이 본인을 식별하기 위해 사용하는 비즈니스 키. 유니크 제약."

나쁜 정의:
> "회원이 시스템에 자신을 식별할 수 있게 해 주는 매우 중요한 정보로서, 본인 인증의 핵심이 되며 보안상 다른 사람과 절대로 겹쳐서는 안 되는 값."

## 흔한 함정

- **모호한 약어** ("RULE", "정책") → 구체 명사 + `Policy` 접미사 (`passwordPolicy`)
- **띄어쓰기 차이** ("비밀 번호" / "비밀번호") → 하나로 통일
- **외래어 혼용** ("유저", "오더") → 한글 일관 ("회원", "주문")
- **상태값 누락** → `pending`, `paid` 같은 상태값도 정의 대상
- **일반 개념어 포함** → "검증", "해싱", "요청", "응답" 같은 일반어는 표에서 빼고, 필요하면 본문에서 자연어로 사용

## 합의 후

표가 합의되면 산출물의 `## 1. 용어 정의` 섹션에 그대로 옮긴다. 같은 volume 안에서 다른 feature의 용어가 추가되면 그 feature의 요구사항 명세에 추가하고, 기존 용어와 충돌 시 사용자에게 보고.

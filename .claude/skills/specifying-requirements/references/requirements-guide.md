# 요구사항 작성 가이드 (단계 3·4)

## 단계 3: 기능 / 비기능 요구사항

### 출력 형식 (엄격)

**기능 요구사항**: 2-컬럼 표 — `| 번호 | 요구사항 |`. 각 행은 검증 가능한 단언문.

```markdown
## 3. 기능 요구사항

| 번호 | 요구사항 |
|---|---|
| 1 | 클라이언트는 `loginId`, `password`, `name`, `birthDate`, `email` 다섯 필드를 모두 포함한 회원가입 요청을 전송한다. 어느 하나라도 누락되면 가입은 거부된다. |
| 2 | `loginId`는 영문 대소문자와 숫자만 허용하며 길이는 **4~20자**다. 길이 범위는 일반적 ID 컨벤션을 따른 디폴트이며, 대소문자는 원문 그대로 저장한다. |
```

**비기능 요구사항**: 3-컬럼 표 — `| 번호 | 카테고리 | 요구사항 |`.

```markdown
## 4. 비기능 요구사항

| 번호 | 카테고리 | 요구사항 |
|---|---|---|
| 1 | 보안 | 평문 비밀번호는 로그·응답·스택트레이스·에러 메시지 어디에도 노출하지 않는다. |
| 2 | 가용성 | 가입 처리는 단일 트랜잭션 안에서 실행한다. |
| 3 | 관측성 | 가입 성공/실패에 대해서는 구조화된 로그로 기록한다. `password`는 절대 로그에 포함하지 않는다. |
```

ID prefix (`FR-1`, `NFR-1`)는 쓰지 마라. **순번만**.

### 작성 원칙

- 각 문장은 한 가지 행위·제약만 표현한다. "그리고"를 두 번 이상 쓰고 있다면 항목을 분리한다.
- 모호한 표현 ("빨라야 한다", "안전해야 한다") → 측정 가능한 수치/조건으로.
- 구현 디테일이 새어 들어옴 ("JPA의 `@Transactional` 사용") → 요구사항이 아니라 구현 결정. 단계 6 도메인 모델링 또는 단계 7로 미룸.
- 원문 복붙 → **검증 가능한 단언문**으로 다시 쓴다.

### 비기능 5대 카테고리 강제 점검

다음 5개를 모두 훑고, 정해진 게 없으면 사용자에게 묻거나 "현 단계 결정 보류"로 명시.

1. **성능** — 응답 시간, 처리량, 동시 처리
2. **보안** — 인증, 인가, 암호화, 평문 노출 금지
3. **가용성** — 에러 처리, 재시도, 장애 영향 범위
4. **관측성** — 로깅, 메트릭, 트레이싱
5. **호환성** — API 헤더 규약, 버전, 클라이언트 종류

회원가입에서 성능이 빠진 것은 SLA 결정이 미뤄진 예외 케이스다. 다른 기능에서는 항상 합의 대상.

### 디폴트 결정의 본문 통합 (중요)

원문이 명시하지 않아 본 명세에서 임의로 결정한 디폴트값(예: 로그인 ID 길이 4~20)은 **별도 표로 분리하지 말고** 해당 기능 요구사항 본문에 사유와 함께 자연스럽게 녹인다.

다음 표는 **만들지 마라**:

- **"명세에서 임의 결정한 디폴트" 표** — 본문에 통합. 사유는 "(이유: ...)" 또는 자연어로.
- **"검증 책임 분포" 표** — 단계 6 도메인 모델링의 "행위" 컬럼이 자연스럽게 표현한다.
- **"에러 코드 매핑" 표** — 단계 7-a 에러 매트릭스로 일원화.

본문 통합 예시:

> | 2 | `loginId`는 영문 대소문자와 숫자만 허용하며 길이는 **4~20자**다. **길이 범위는 일반적 ID 컨벤션을 따른 디폴트이며, 대소문자는 원문 그대로 저장한다.** |

굵게 표시된 부분이 디폴트 결정 사유다. 별도 표가 아니라 본문 한 줄로.

## 단계 4: 목표가 아닌 것

### 왜 명시하는가

범위를 좁히지 않은 명세는 끝없이 커진다. "이번 라운드에서는 안 한다"고 명시된 항목은 나중 라운드의 입력이 되므로 잃어버리지 않는다.

### 출력 형식

자유 서술 bullet. 각 항목 끝에 사유 한 줄 첨부.

```markdown
## N. 목표가 아닌 것

본 라운드에서 다루지 않을 항목.

- **회원 탈퇴 및 동일 `loginId`/`email` 재사용 정책** — 탈퇴 라운드에서 soft/hard delete 정책과 함께 합의한다.
- **소셜 로그인** — 인증 방식 확장 라운드에서 다룬다.
- **회원 등급 / 권한 / 역할** — 관리자 권한이 필요한 기능이 도입될 때 다룬다.
```

### 헤더 명칭 (엄격)

`## N. 목표가 아닌 것` 한글. **"Out of Scope" 영문 X**.

### 작성 원칙

- 사용자가 자연스럽게 떠올릴 만한 인접 기능을 **선제적으로 나열**해 "이건 빼는 게 맞나?"라고 묻는다.
- 사유는 짧게. "다음 라운드", "장기 과제" 같은 짧은 태그면 충분.
- 너무 멀리 있는 것은 적지 마라. 본 라운드에 헷갈릴 만한 것만.
- 일반 인프라·보안 항목(CAPTCHA, WAF, HTTPS 강제, 다국어 등)이 자명히 본 명세 책임이 아니라면 생략하거나 한 줄로만.

### 도메인별 자주 빠지는 후보

- 알림, 이력, 통계, 검색, 정렬
- 권한/역할, 본인/타인 구분
- 본인 인증 절차, 재인증
- 분실/재발급, 취소/반품
- 일괄 처리, 어드민 화면, 모니터링
- 다국어, 다중 통화, 멀티 테넌트

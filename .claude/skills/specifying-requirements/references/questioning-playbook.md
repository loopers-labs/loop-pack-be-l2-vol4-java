# 질문 플레이북

스킬의 핵심 자산. 단계 3, 5, 6에서 모호함을 끌어낼 때 호출한다. 도메인 무관, 작업 유형(CRUD + 인증 + 외부의존성) 기준으로 정리되어 있다.

## 사용법

SKILL.md의 각 단계에는 시드 질문 3~5개만 인라인으로 두었다. 더 깊이 파야 할 때(특히 단계 5의 예외 시나리오, 단계 3의 누락 요구사항 발굴) 이 문서의 해당 작업 유형 섹션을 읽고 **최소 5개 이상의 질문을 사용자에게 일괄 제시**한다.

질문은 흩뿌리지 말고 다음 형식으로 정리해야 사용자가 답하기 쉽다.

```
다음 항목들에 대해 결정이 필요합니다:

1. [질문] → 선택지 A / B / 직접 답변
2. [질문] → 기본값으로 X를 제안합니다. 다르게 가실 건가요?
3. ...
```

## § 식별자와 명명

거의 모든 도메인에서 공통으로 결정해야 하는 항목.

### 식별자
- **시스템 내부 식별자**(DB PK)와 **외부 노출 식별자**(사용자에게 보이는 ID, slug, code)를 구분할지?
- 외부 노출 식별자의 **포맷**: 자동 증가 정수 / UUID / 영숫자 짧은 코드 / 사람이 읽을 수 있는 슬러그 — 어느 쪽?
- 식별자에 허용되는 **문자셋과 길이** 범위?
- 식별자의 **대소문자 구분** 여부?
- 이미 삭제된 리소스의 **식별자 재사용** 허용?
- 동시 생성 요청에서 식별자 충돌은 어떻게 보장? (DB 유니크 제약 / 애플리케이션 락 / 분산 ID 생성기)

### 명명
- 같은 개념을 가리키는 동의어가 있을 때 표준 명칭을 무엇으로 정할지?
- API/DB/코드에서 영어 명칭이 일관되는지?

## § 생성/등록 작업

새 리소스를 만드는 작업 (POST 계열).

### 입력 검증
- 각 필드의 **형식·길이·범위·필수 여부**가 모두 명시되어 있나?
- 검증 실패 시 **첫 에러만** 반환 vs **모든 에러 일괄** 반환?
- 검증 에러의 **응답 스키마** (필드별 에러 배열 vs 단일 메시지)?
- 트리밍(앞뒤 공백 제거), 정규화(유니코드 NFC), 대소문자 변환 등 **입력 전처리** 정책?

### 중복·유니크 제약
- 비즈니스 키(예: 외부 식별자, 이름) 중복 검사 정책 — 대소문자 구분? 공백 무시?
- 중복 발견 시 응답 코드 (보통 409) 및 에러 코드 명?
- 동시 생성 요청에서 중복은 어떻게 보장? (DB 유니크 제약 vs 애플리케이션 락 vs 둘 다)

### 멱등성
- 같은 요청을 두 번 보내면? 자연 중복 차단 vs **멱등 키**(idempotency key) 도입?
- 클라이언트 재시도 시 중복 생성 방지 방법?

### 부수 효과
- 생성 직후 수행되는 **부수 작업**(알림 발송, 캐시 갱신, 외부 시스템 호출, 통계 업데이트)?
- 부수 작업이 실패하면 **생성 자체를 롤백**할지, **별도 비동기 처리**(메시지 큐, 이벤트 발행)할지?
- 부수 작업의 결과를 응답에 포함시킬지?

### 응답
- 생성된 리소스 전체를 응답으로 반환 vs 식별자만 반환?
- Location 헤더에 새 리소스 URI 포함?
- 생성 시각 같은 서버 결정 필드를 응답에 포함?

## § 조회 작업 — 단건

특정 리소스 하나를 가져오는 작업 (GET /resource/{id}).

### 식별·범위
- **시스템 PK** vs **비즈니스 키** 중 무엇으로 조회?
- 본인 데이터만 가능 vs 다른 사용자 데이터도 조회 가능?
- 권한 검증 위치 (인터셉터 vs 서비스 레이어)?

### 응답 표현
- 응답에 포함될 **필드 목록**과 마스킹/변환이 필요한 필드?
- 연관 리소스를 **함께 포함**(eager) vs 별도 API로 분리?
- 응답 시간 필드의 **timezone과 포맷** (`ISO 8601` UTC / 로컬 시각)?
- `null` 표현 정책 (null 명시 vs 필드 자체 생략)?

### 존재하지 않음
- 리소스 부재 시 404? 또는 본인이 아닌 경우 권한 노출 방지를 위해 항상 404?
- soft delete된 리소스도 조회 대상?

---

## § 조회 작업 — 리스트/검색

여러 리소스를 가져오는 작업 (GET /resources).

### 페이징
- **offset 기반** vs **cursor 기반**? (대용량/실시간 데이터면 cursor 권장)
- 페이지 크기의 **기본값과 최대값**?
- 응답에 **전체 개수**를 포함할지? (성능 영향 — 매번 count 쿼리)

### 정렬
- 정렬 가능한 컬럼 **화이트리스트**?
- 기본 정렬 키와 방향?
- 복수 정렬 기준 허용?

### 필터·검색
- 검색은 정확 일치 / 부분 일치 / 전문(full-text) 중 어느 것?
- 필터 조합 시 AND / OR 의미는 명확한가?
- 검색어 빈 값/와일드카드 처리?

### 빈 결과·경계
- **빈 결과** 표현: 빈 배열 + 200 / 204 / 404 중?
- 페이지 초과 요청 (예: 10번째 페이지인데 8페이지뿐) 응답?

## § 수정/변경 작업

기존 리소스를 변경하는 작업 (PUT/PATCH).

### 변경 범위
- **부분 수정** (PATCH) vs **전체 교체** (PUT)?
- 변경 가능 필드와 **변경 불가 필드**(생성 시점 고정 필드, 식별자) 구분?
- 비어있는 필드 전송이 "값 비우기"인지 "변경 안 함"인지 명확한가?

### 사전 조건
- 본인 확인이나 권한 검증?
- 변경 직전 상태와의 **충돌 검사** (낙관적 락 = 버전 컬럼 / `If-Match` 헤더)?
- 변경 전 상태가 특정 조건일 때만 허용? (예: 상태가 "임시"일 때만 수정 가능)

### 이력
- 변경 **이력 보관**이 필요한 도메인인가? 어디까지 (필드별, 변경자, 시각, 사유)?
- 이력 저장은 동기 vs 비동기?

### 응답
- 수정 후 갱신된 리소스 전체 반환 vs 204 No Content?

---

## § 삭제 작업

리소스를 제거하는 작업 (DELETE).

- **즉시 삭제(hard delete)** vs **soft delete** (`deleted_at` 컬럼)?
- 삭제된 데이터의 **재조회 가능 여부** (특정 권한자만, 또는 영구 불가)?
- 삭제 후 식별자/비즈니스 키 **재사용 정책**?
- 본인 확인 절차 필요?
- 연관 리소스의 **연쇄 삭제** 정책 (CASCADE / 막기 / soft delete만)?
- 멱등성 — 이미 삭제된 리소스에 또 DELETE 요청이 오면? (204 / 404)

## § 인증과 인가

### 인증 방식
- 세션 기반 / 토큰(JWT 등) / 매 요청 자격증명 — 어느 방식?
- 자격증명의 **저장 위치** (쿠키, 헤더, 바디)?
- 자격증명의 **만료와 갱신** 정책?
- 평문 노출 위험은 어떻게 차단(HTTPS 강제, 평문 로깅 금지)?

### 인가
- 어느 API가 **인증 필수**, 어느 API가 **공개**?
- **본인 데이터만** 접근 가능 vs 명시적 권한 부여 필요?
- 역할(role) 기반 인가 필요한가? 역할 종류와 권한 매트릭스?
- 권한 검증 위치 (인터셉터, 어노테이션, 서비스 레이어)?

### 실패 흐름
- 잘못된 자격증명 — 구체적 원인(없는 ID vs 비밀번호 불일치)을 알려줄지, 보안을 위해 통일된 메시지로?
- 실패 횟수 카운트 → 계정 잠금 / 차단 정책?
- 권한 부족과 미인증을 다른 응답 코드로 구분(401 vs 403)?

## § 외부 의존성

DB 외에 의존하는 시스템(캐시, 메시지 큐, 외부 API, 결제 PG, 메일 발송 등).

- 외부 호출 **타임아웃** 기본값?
- **재시도** 정책 (회수, 백오프 전략)?
- **서킷브레이커**가 필요한 수준의 호출인가?
- 외부 호출 실패 시 **사용자에게 보이는 응답** (5xx로 그대로 노출 / fallback 값 / 비동기 처리로 우회)?
- **트랜잭션 경계**가 외부 호출을 포함하는가? (가능하면 분리 — 외부 호출 후 DB 커밋, 또는 별도 보상 로직)
- 외부 시스템과의 **데이터 정합성** (이중 쓰기 문제, eventual consistency 허용 여부)?

## § 민감·제약 데이터 처리

비밀번호, 결제 정보, 개인정보, 토큰처럼 **저장·전송·로깅에 추가 제약이 따르는 데이터**의 처리.

### "포함 불가" 같은 규칙의 정확한 정의

요구사항에 "X에 Y가 포함될 수 없다" 같은 한 줄 규칙이 나오면 다음을 모두 확정해야 한다.

- **형식 패턴** — 검사 대상의 정확한 표현 범위 (예: 날짜라면 `YYYYMMDD`만? `YYMMDD`도? `MMDD`만? 한 자리 월/일 허용? 구분자 있는 형태 모두?)
- **부분 매치** — 정확히 같을 때만 거부 vs 부분 포함도 거부?
- **검사 시점** — 등록 시만? 변경 시도? 매 사용 시도?
- **검사 대상의 변환** — 대소문자 무시? 공백 제거? 한글 자모 분리 케이스?

같은 원칙이 다른 "포함 불가", "유사 금지" 규칙에도 적용된다.

### "이전과 다름" 같은 규칙

- 정확히 동일한 값만 거부 vs 일정 거리(예: 1글자 차이) 이내도 거부?
- 과거 N개의 값을 기억해 재사용 금지할지?
- 비교 시 전처리 (대소문자 무시, 공백 무시) 어디까지?

### 저장 형태

- 원본 그대로 저장 vs 단방향 해시 vs 양방향 암호화?
- 해시라면 알고리즘과 파라미터(salt, cost factor)?
- 암호화라면 키 관리 정책?

### 노출 형태

- 응답에 포함될지? 포함된다면 **마스킹** (어느 위치를 어떤 문자로)?
- 로깅에 노출 금지 — 로깅 위치 식별과 차단 방법?
- 어드민/운영 화면에서의 노출 정책?

### 입력 전처리

- 앞뒤 공백 trim 여부?
- 유니코드 정규화 (NFC) 수행?
- 인코딩 검증 (예: 허용 외 문자 거부)?

## § 예외 카테고리 강제 체크리스트

단계 5(시나리오 다이어그램)에서 **반드시** 이 7개 카테고리를 모두 훑는다. 해당 없으면 "해당 없음"이라고 명시한다.

| 카테고리 | 점검 질문 | 흔한 응답 코드 |
|---|---|---|
| 1. 입력 검증 실패 | 모든 필드의 형식·길이·범위·필수 여부가 명시되었나? 한 요청에 다중 에러 발생 시 어떻게 반환? | 400 |
| 2. 인증·인가 실패 | 인증 헤더/토큰 누락, 잘못된 자격증명, 권한 부족을 각각 어떻게 구분? | 401 / 403 |
| 3. 리소스 부재 | 조회·수정·삭제 대상이 없을 때 응답? soft delete된 리소스 취급? | 404 |
| 4. 충돌 | 중복 등록, 유니크 제약 위반, 동시 수정으로 인한 버전 충돌? | 409 |
| 5. 외부 의존성 실패 | DB/캐시/외부 API 장애 시 사용자에게 보이는 응답? | 502 / 503 |
| 6. 동시성 | 같은 자원에 대한 동시 요청에서 일관성 보장 방법? | 409 / 423 |
| 7. 타임아웃 | 처리가 길어졌을 때 사용자 응답? 서버 측 타임아웃 기준? | 504 |

각 카테고리마다 별도 다이어그램을 그릴 필요는 없지만, **응답 코드와 에러 코드 명**은 단계 7-a의 API 명세에 반드시 들어가야 한다.

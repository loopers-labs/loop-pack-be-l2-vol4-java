# 요구사항 명세화 스킬

## 개요

이 스킬은 **모호한 한 줄짜리 기능 요구사항**을 받아 **구현 직전까지 필요한 명세 산출물 3종**(요구사항_명세.md / API_명세.md / ERD.md)으로 구체화한다.

**본질적 가치는 산출물 자체가 아니라 "사용자가 미처 생각하지 못한 결정사항과 예외 케이스를 끌어내는 좋은 질문"이다.** 단순 템플릿 채우기로 끝내지 마라. 7단계를 한 번에 산출하지도 마라. 한 단계씩 진행하면서 사용자에게 모호함을 짚어 묻고, 확정된 결정만 산출물에 반영한다.

## 사용 시점

다음 신호 중 하나라도 있으면 이 스킬을 가동한다.

- 사용자가 `docs/volume-N/` 같은 경로를 가리키거나 짧은 요구사항 문서를 첨부함
- "이 요구사항 정리해줘", "기능 명세 작성", "기능 기획해보자", "API 명세 만들어줘", "ERD 그려줘" 같은 발화
- 한 줄 또는 한 단락짜리 느슨한 기획 문서를 첨부
- 새로운 **API 단위 기능**(회원/주문/상품/리뷰/결제/조회/수정 등)을 시작하려 함

## 사용하지 않는 경우

- 이미 산출물이 완성된 상태에서 **코드 구현**을 시작하려 함 → 일반 코딩 작업
- 기존 코드의 **버그 수정 / 리뷰 / 리팩터링** → 다른 스킬 또는 일반 작업
- 단순한 **개념 질문** ("JPA가 뭐야?") → 일반 답변

## 먼저 수집할 입력

스킬 시작 시 사용자에게 다음 3가지를 확인한다. 모두 확정될 때까지 다음 단계로 가지 마라.

1. **Volume 번호** (예: `volume-1`) — 산출물 경로의 일부
2. **Feature 이름** (한글, 예: `회원가입`, `상품_등록`, `주문_생성`) — 폴더명이 되므로 공백 대신 `_` 사용
3. **원문 요구사항** — 파일 경로 또는 본문 텍스트. 파일이면 Read하여 이번 라운드에서 다룰 단일 feature를 확정한다.

세 가지가 확정되면 산출물 경로는 다음과 같이 결정된다.

- `docs/volume-N/{feature}/요구사항_명세.md`
- `docs/volume-N/{feature}/API_명세.md`
- `docs/volume-N/ERD.md` (**volume 단위 공통 파일**, 기존 파일이 있으면 갱신)

## 산출물 형식 원칙 (전역, 엄격)

모든 산출물은 다음 원칙을 일관되게 따른다.

1. **섹션 헤더는 한글 번호** — `## 1. 용어 정의`, `## 2. 비밀번호 보호 방식`, `## 3. 기능 요구사항` 등. "단계 N" 명칭은 SKILL 워크플로우의 내부 단계 표현이며, **산출물 헤더로는 쓰지 않는다**.
2. **간결성 우선** — 자세한 정의보다 핵심 한 문장. 자명한 인프라 항목(HTTPS 강제 등)은 본 명세 책임이 아니므로 생략하거나 한 줄로만.
3. **본문 통합 원칙** — "임의 결정한 디폴트", "검증 책임 분포", "에러 매핑", "예외 카테고리 점검" 같은 메타 표는 별도 섹션으로 분리하지 마라. 해당 요구사항·시나리오 본문에 자연스럽게 녹인다.
4. **표 형식 우선** — 요구사항·에러 매트릭스·도메인 모델링 등 반복 항목은 markdown 표. ID prefix(`FR-1`, `NFR-1`) X, **순번만**.
5. **한국어 작성** — 코드/식별자/SQL만 영어.
6. **본 프로젝트 컨벤션 따름** — 클래스명(`{X}Model`, `{X}V1Controller`, `{X}Facade`, `{X}Service`, `{X}Repository`), 응답 envelope (`ApiResponse<T>`), 에러 처리 (`CoreException` + `ErrorType`), 시간 컬럼 (`DATETIME` + UTC) 등.

## 워크플로우 체크리스트

진행 시작 시 다음 체크리스트를 응답에 복사하고 단계가 끝날 때마다 체크한다.

```
명세 진행 상태:
- [ ] 단계 1: 입력 수집 (volume / feature / 원문)
- [ ] 단계 2: 용어 정의 (한글 / 영어 / 의미)
- [ ] 단계 3: 기능 요구사항 + 비기능 요구사항
- [ ] 단계 4: 목표가 아닌 것
- [ ] 단계 5: 정상 시나리오 + 예외 시나리오 다이어그램
- [ ] 단계 6: 도메인 모델링 (역할 / 책임 / 속성 / 행위)
- [ ] 단계 7: API 명세 + ERD
```

각 단계 종료 시점에는 항상 **"이 결과로 단계 N 확정해도 될까요?"** 라고 명시적으로 묻고, 사용자의 명시적 승인(예: "좋아", "이대로 가자", "확정")이 떨어진 후에만 다음 단계로 넘어간다.

---

## 단계 1: 입력 수집

위 "먼저 수집할 입력"을 수행한다. 원문 요구사항을 Read한 뒤, **이 문서에서 다룰 단일 feature를 사용자에게 재확인**한다. 한 요구사항 문서에 여러 기능이 있으면 **하나씩 별도 라운드로 처리**한다.

---

## 단계 2: 용어 정의

산출물 헤더: `## 1. 용어 정의`

**산출 형식 (엄격)**: 3-컬럼 표 — `한글 | 영어 | 의미`. 영어 컬럼은 코드/API/DB에서 실제로 사용할 명칭.

**범위**: 도메인 명사·값에 집중. 도메인 핵심 명사·동사·식별자·값·상태값·정책 명칭을 포함하되, **일반 개념어("검증", "해싱", "요청", "응답", "처리", "저장")는 제외**한다.

핵심 질문 시드:
- "원문의 표현 X와 Y가 같은 개념을 가리키는지, 영어로는 어떤 명칭을 쓸지?"
- "원문에는 없지만 명세를 위해 필요한 보조 용어가 있나요?"

더 깊이: `references/terminology-guide.md`

---

## 단계 3: 기능 요구사항 + 비기능 요구사항

산출물 헤더: `## 3. 기능 요구사항`, `## 4. 비기능 요구사항`. (도메인 특화 보충 섹션이 단계 2 다음에 들어가면 번호가 시프트될 수 있다. 그래도 한글 번호 형식은 유지.)

**산출 형식 (엄격)**:
- 기능: 2-컬럼 표 — `| 번호 | 요구사항 |`
- 비기능: 3-컬럼 표 — `| 번호 | 카테고리 | 요구사항 |`

각 행은 **검증 가능한 단언문**으로. 원문에 명시되지 않은 항목을 능동적으로 끌어내라. `references/questioning-playbook.md`의 도메인 카테고리에서 **최소 5개 이상의 질문**을 사용자에게 일괄 제시한다.

### 비기능 5대 카테고리 강제 점검

다음 5개를 모두 훑고, 정해진 게 없으면 사용자에게 묻거나 "현 단계 결정 보류"로 명시.

1. **성능** — 응답 시간, 처리량, 동시 처리
2. **보안** — 인증, 인가, 암호화, 평문 노출 금지
3. **가용성** — 에러 처리, 재시도, 장애 영향 범위
4. **관측성** — 로깅, 메트릭, 트레이싱
5. **호환성** — API 헤더 규약, 버전, 클라이언트 종류

회원가입에서 성능 카테고리가 빠진 것은 SLA 결정이 미뤄진 예외 케이스다. 다른 기능에서는 항상 합의 대상.

### 디폴트 결정의 본문 통합 (중요)

원문이 명시하지 않아 본 명세에서 임의로 결정한 디폴트값(예: 로그인 ID 길이 4~20)은 **별도 표로 분리하지 말고** 해당 기능 요구사항 본문에 사유와 함께 자연스럽게 녹인다. 예:

> | 2 | `loginId`는 영문 대소문자와 숫자만 허용하며 길이는 **4~20자**다. 길이 범위는 일반적 ID 컨벤션을 따른 디폴트이며, 대소문자는 원문 그대로 저장한다. |

다음 표는 **만들지 마라**:
- "명세에서 임의 결정한 디폴트" 표
- "검증 책임 분포" 표 (단계 6의 도메인 모델링 행위 컬럼이 자연스럽게 표현)
- "에러 코드 매핑" 표 (단계 7-a 에러 매트릭스로 일원화)

더 깊이: `references/requirements-guide.md`

## 단계 4: 목표가 아닌 것

산출물 헤더: `## N. 목표가 아닌 것` (영문 "Out of Scope" 쓰지 마라)

**산출 형식**: 자유 서술 bullet. 각 항목 끝에 사유 한 줄 첨부.

이 단계는 자유도가 높다. 사용자가 "그건 일단 빼자"고 하는 것 위주로 채우면 된다. 일반 인프라 보안 항목(CAPTCHA, WAF, HTTPS 강제 등)이나 영구히 안 할 항목은 길게 적지 마라.

## 단계 5: 시나리오 다이어그램 (정상 + 예외)

산출물 헤더: `## N. 시나리오`

**산출 형식 (엄격)**: Mermaid `sequenceDiagram`. 본 프로젝트의 레이어드 컨벤션 액터를 사용한다.

| 액터 | 표기 |
|---|---|
| 호출자 | `사용자` |
| 컨트롤러 | `{X}V1Controller` |
| 애플리케이션 | `{X}Facade` |
| 도메인 서비스 | `{X}Service` |
| 도메인 객체·값 객체 | `{X}Model / 값 객체` (또는 `Domain`) |
| 영속화 | `{X}Repository` |
| 인증 인자 해석기 | `AuthenticatedUserArgumentResolver` (인증이 필요한 엔드포인트에서 Controller 앞에 액터로 등장) |

**추상 액터(`Client`, `API`, `DB`)는 쓰지 마라.** 외부 의존(캐시·결제 PG)이 있을 때만 `Cache`, `External as {이름}` 추가.

**인증이 필요한 시나리오 패턴**: `사용자 → AuthenticatedUserArgumentResolver → Repository.findByLoginId → UserModel.matchesPassword`로 인증된 후, `Resolver → Controller: @LoginUser AuthenticatedUser(userId)`를 주입하고, Controller가 Facade·Service·Repository로 후속 조회를 진행. 인증 실패는 모두 `CoreException(UNAUTHENTICATED)` → 401 단일 응답으로 통합 (사용자 열거 방지).

### 예외 카테고리 강제 점검

다음 7개 카테고리를 모두 훑고 사용자에게 적용 여부를 묻는다. **점검 결과를 별도 표로 산출하지 마라**. 적용 가능한 케이스에만 다이어그램을 그린다.

1. 입력 검증 실패 / 2. 인증·인가 실패 / 3. 리소스 부재 / 4. 충돌 / 5. 외부 의존성 실패 / 6. 동시성 / 7. 타임아웃

### 폐기된 보조 산출물

- **예외 카테고리 점검 표** (`| # | 카테고리 | 본 라운드 적용 |`) — 만들지 마라.
- **응답 코드·에러 코드 정책 표** (`| 상황 | HTTP | code |`) — 단계 7-a의 에러 매트릭스로 통합. 시나리오 단계에서 별도 표 만들지 마라.

다이어그램의 응답 코드는 단계 7-a 에러 매트릭스와 직접 동기화된다.

더 깊이: `references/scenario-diagrams.md`

## 단계 6: 도메인 모델링

산출물 헤더: `## N. 도메인 모델링`

**철학**:
- DDD 전술 패턴 중 애그리거트·도메인 이벤트는 도입하지 마라.
- **값 객체(Value Object)는 도입한다** — 각 필드의 검증 책임을 자기 자신에게 캡슐화하기 위해.
- **JPA Entity = Domain Entity**: 별도 도메인 모델 레이어를 만들지 않는다.

**산출 범위 (엄격)**:
- **포함**: JPA Entity (`{X}Model`), 값 객체 (예: `LoginId`, `Email`), 도메인 포트 (예: `PasswordEncoder`), 어댑터 (예: `BcryptPasswordEncoder`).
- **제외**: Service / Facade / Repository / Controller / DTO. 이들은 협력자·인프라 어댑터이지 도메인 모델이 아니다. 단계 5 시나리오에서 액터로 표현하면 충분. **도메인 모델링 표는 클래스 다이어그램이 아니다**.

**산출 형식 (엄격)**: 4-컬럼 표 — `| 역할 (객체명) | 책임 | 속성 | 행위 |`.

핵심 질문 시드:
- "단일 필드 값에 대한 검증을 값 객체로 캡슐화할 수 있나요?"
- "여러 필드를 함께 검증해야 하는 경우(예: 비밀번호+생년월일) 어느 값 객체가 둘 다 받는 게 자연스러운가요?"
- "민감 정보의 변환은 도메인의 책임인가요, 인프라 어댑터의 책임인가요?"
- "이 메서드 이름의 어휘 강도가 도메인 모델의 책임과 맞나요?" — 행위 강도가 강한 동사(`authenticate`, `mask`)는 도메인 모델 메서드보다 인증 서비스(Resolver 등)의 책임처럼 읽힌다. boolean 반환은 `matches*`/`is*`/`has*` 같은 명사·상태 접두사, 값 반환은 `*Value` 같은 명사형 접미사가 도메인 모델에 자연스럽다.

더 깊이: `references/domain-modeling-guide.md`

## 단계 7: API 명세 + ERD

두 산출물을 동시에 만든다.

### 7-a) API 명세 (`docs/volume-N/{feature}/API_명세.md`)

**산출물 헤더 구조 (엄격)**:
- `## 1. Overview`
- `## 2. Endpoint` (Method · Path 표)
- `## 3. Request` (Headers / Body 표 + 예시)
- `## 4. Response — Success` (스키마 표 + 예시)
- `## 5. Response — Error` (응답 매트릭스 + 비고)

**응답 envelope 컨벤션 (엄격)**: 본 프로젝트의 `ApiResponse<T>` 스키마(`meta.result/errorCode/message + data`)를 단일 표준으로 사용. 선택 A/B 논의 X.

**에러 매트릭스 (엄격)**: 4-컬럼 표 — `| 상황 | HTTP | errorCode | message 예 |`.

각 행은 단계 5 시나리오 다이어그램의 응답과 일관되어야 한다.

더 깊이: `references/api-spec-guide.md`

### 7-b) ERD (`docs/volume-N/ERD.md`) — volume 단위 공통

**작업 순서**:
1. 기존 파일 Read 시도. 없으면 신규 생성.
2. 있으면 사용자에게 "현재 ERD에 N개 테이블이 있습니다." 요약 보고.
3. 이번 feature의 도메인 모델에서 신규 테이블/컬럼/관계 추출.
4. 충돌 검사 (동일 테이블명·FK 대상 부재·타입 불일치 등).
5. Diff 사용자 승인.
6. `erDiagram` 통합 재작성, DDL 섹션 append.

**산출물 헤더 구조 (엄격)**:
- `# volume-N ERD`
- `## 공통 컨벤션` (엔진·문자셋·PK·시간 컬럼·Soft delete)
- `## 다이어그램` (mermaid `erDiagram`)
- `## 테이블 정의` (테이블별 DDL + 비고)

**본 프로젝트 컨벤션**:
- `BaseEntity`(`modules/jpa`) 상속으로 `id`(`BIGINT AUTO_INCREMENT`), `created_at`, `updated_at`, `deleted_at` 자동.
- 시간 컬럼: `DATETIME` (timezone-naive). Hibernate `NORMALIZE_UTC` + `jdbc.time_zone=UTC`로 UTC wall-clock 저장.
- MySQL 예약어 회피 (예: `user` → `users` 복수형).
- DDL: `InnoDB` / `utf8mb4` / `utf8mb4_unicode_ci`. 모든 컬럼·테이블에 한국어 `COMMENT '...'`.

더 깊이: `references/erd-guide.md`

## 도메인 특화 보충 섹션

특정 도메인에서 결정해야 할 큰 주제(예: **비밀번호 보호 방식**, **멱등성 키 전략**, **결제 게이트웨이 선택**)는 단계 2~7 사이 어느 곳에든 자유 섹션으로 추가할 수 있다. 헤더는 그 위치에 맞춰 한글로 (`## 2. 비밀번호 보호 방식` 등). 사용자와 합의 후 작성한다.

이는 **강제가 아니라 허용**이다. 회원가입의 `## 2. 비밀번호 보호 방식`이 이 패턴의 사용 예다 — 도메인에 특별히 결정해야 할 큰 주제가 있을 때만 등장하며, 다른 기능에서는 보통 등장하지 않는다.

### 파일 생성/갱신 절차 (모든 단계 공통)

각 단계의 산출을 파일로 옮길 때:

1. **Read** 시도: 기존 파일 있으면 읽어서 사용자에게 보여준다.
2. **Diff 제안**: 추가/수정될 내용을 명확히 보여준다.
3. **사용자 승인** 대기.
4. 승인 후 **Write/Edit**.

`요구사항_명세.md`는 단계 2~6의 결과가 누적되는 단일 파일이다. 단계마다 해당 섹션을 추가/갱신한다 — 매번 파일을 통째로 다시 쓰지 마라.

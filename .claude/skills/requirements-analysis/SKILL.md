# 요구사항 분석 스킬

개발자가 아닌 사람도 이해할 수 있는 요구사항 명세서와,
설계 의도가 드러나는 Mermaid 다이어그램을 일관된 순서로 생성한다.

---

## 0. 전체 진행 순서 (항상 이 순서를 지킨다)

[1단계] 질문 → [2단계] 요구사항 명세서 → [3단계] 다이어그램 3종 → [4단계] 리스크 및 대안

단계를 건너뛰거나 순서를 바꾸지 않는다.
의문점이 하나라도 남아 있으면 다음 단계로 넘어가지 않는다.

---

## 1단계: 질문 (반드시 먼저 수행)

요구사항을 받으면 바로 작성하지 않는다.
아래 3가지 카테고리로 의문점을 정리하여 사용자에게 먼저 질문한다.

### 질문 카테고리

**정책 질문** — 기준과 규칙
- 기준 시점이 있는가? (예: "최근 7일", "오늘 기준")
- 성공/실패 조건은 무엇인가?
- 예외 상황은 어떻게 처리하는가?

**경계 질문** — 범위와 책임
- 이 기능의 시작과 끝은 어디인가?
- 어디까지가 한 화면/한 기능인가?
- 외부 시스템과의 연동이 있는가?

**확장 질문** — 미래 변경 가능성
- 나중에 바뀔 가능성이 있는 부분은 어디인가?
- 사용자 유형이 늘어날 수 있는가?
- 데이터 규모가 커질 것을 고려해야 하는가?

### 질문 규칙
- 사용자가 답변하기 전까지 명세서 작성을 시작하지 않는다.
- 사용자가 "그냥 진행해"라고 해도 가정 사항을 명시적으로 나열하고 확인을 받은 뒤 진행한다.
- 답변이 불명확하면 재질문한다.

---

## 2단계: 요구사항 명세서

질문이 모두 해소된 뒤 아래 6개 섹션을 순서대로 작성한다.

### 2-1. 제품 개요
- 이 제품/기능이 왜 필요한가 (배경)
- 누가 사용하는가 (대상 사용자)
- 무엇을 해결하는가 (핵심 문제)
- 개발자가 아닌 사람도 읽고 바로 이해할 수 있는 언어로 작성한다

### 2-2. 유저 스토리
형식: 나는 [사용자 유형]으로서, [목적]을 위해 [기능]을 원한다.

- 각 유저 스토리마다 왜 이 스토리가 도출되었는지 한 줄로 설명한다
- 우선순위(높음/중간/낮음)를 함께 표기한다

### 2-3. 유저 시나리오
- 유저 스토리를 실제 사용 흐름으로 구체화한다
- 정상 흐름(Happy Path)과 예외 흐름(Exception Path)을 모두 작성한다
- 각 단계에서 사용자가 보는 화면/메시지를 함께 기술한다

### 2-4. 기능 요구사항
표 형식으로 작성:

| ID | 기능명 | 설명 | 우선순위 | 관련 유저스토리 |
|----|--------|------|----------|----------------|

- 각 기능에 도출 이유를 한 줄로 붙인다
- 모호한 표현("빠르게", "편리하게") 금지 — 구체적인 조건으로 대체한다

### 2-5. 비기능 요구사항
아래 항목 중 해당하는 것을 작성한다:
- 성능: 응답 시간, 처리량
- 보안: 인증 방식, 데이터 암호화
- 가용성: 장애 허용 범위
- 확장성: 사용자 수 증가 대응

### 2-6. 인수 조건 (Acceptance Criteria)
형식: given [상황] / when [행동] / then [결과]

- 기능 요구사항 ID와 1:1로 연결한다
- 테스트 가능한 형태로 작성한다 (모호한 표현 금지)

## 3단계: 다이어그램 3종 (항상 이유 → 다이어그램 → 해석 순서)

시퀀스 다이어그램, 클래스 다이어그램, ERD를 항상 3개 전부 작성한다.
각 다이어그램은 아래 구조를 반드시 따른다:

[다이어그램 이름]

왜 이 다이어그램이 필요한가
- 이 다이어그램으로 무엇을 검증하려는지 2~3문장으로 설명

다이어그램
(Mermaid 코드 블록)

이 구조에서 특히 봐야 할 포인트
- 포인트 1: 설계 의도 해석
- 포인트 2: 주의해야 할 부분
- 포인트 3: (있을 경우) 확장 고려 포인트

### 3-1. 시퀀스 다이어그램

목적: 책임 분리, 호출 순서, 트랜잭션 경계 확인

작성 기준:
- Actor(사용자/외부 시스템)를 가장 왼쪽에 배치한다
- 각 화살표에 메시지 이름을 명시한다 (예: POST /login)
- 성공 흐름과 실패 흐름을 alt/else로 분기하여 표현한다
- 트랜잭션 범위는 rect 또는 Note로 표시한다

### 3-2. 클래스 다이어그램

목적: 도메인 책임, 의존 방향, 응집도 확인

작성 기준:
- 각 클래스에 주요 속성과 메서드를 함께 표기한다
- 의존 방향을 명시한다 (--> 의존, --|> 상속, --o 집합)
- 한 클래스가 너무 많은 책임을 가지면 분리를 검토하고 주석으로 표시한다

### 3-3. ERD

목적: 영속성 구조, 관계의 주인, 정규화 여부 확인

작성 기준:
- PK는 PK, FK는 FK로 명시한다
- 관계의 카디널리티를 정확히 표현한다 (||--o{ 등)
- 1:N 관계에서 FK가 어느 테이블에 있는지 확인한다
- 정규화가 필요한 부분은 다이어그램 아래 주석으로 언급한다

---

## 4단계: 리스크 및 대안

다이어그램 작성 후 반드시 아래 구조로 리스크를 작성한다.

[리스크 이름]
- 현재 설계의 문제점: 무엇이 문제인가
- 대안: 다른 방법은 무엇인가
- 현재 방법을 선택한 이유: 왜 현재 방법이 더 나은가 (트레이드오프 명시)

리스크는 최소 2개 이상 작성한다.
"문제가 없다"고 판단해도 잠재적 리스크를 찾아 작성한다.

---

## 작성 원칙

1. 추측 금지: 불명확한 부분은 반드시 질문한다. 가정할 경우 명시적으로 표시한다.
2. 쉬운 언어: 개발자가 아닌 사람도 읽을 수 있게 작성한다. 전문 용어는 괄호 안에 설명을 붙인다.
3. 이유 병기: 모든 결정(기능 포함/제외, 구조 선택)에 이유를 한 줄 이상 붙인다.
4. 일관성 유지: 유저 스토리 ID, 기능 요구사항 ID, 인수 조건 ID, 다이어그램이 서로 연결되어야 한다.
5. Mermaid 전용: 다이어그램은 반드시 Mermaid 문법만 사용한다. 다른 형식 사용 금지.

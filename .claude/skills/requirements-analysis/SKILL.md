# 요구사항 분석 스킬

## 개요

유저 스토리 한 문장씩을 받아 네 가지 설계 산출물로 구체화한다. 모두 같은 `docs/volume-N/` 폴더에 누적되며 도메인 사전·결정 번호·어휘를 공유한다.

| Phase | 산출물 | 필수 | 역할 |
| --- | --- | --- | --- |
| 1 | `01-requirements.md` | 필수 | 도메인별 명세(전제 조건·기능 요구사항·비즈니스 규칙·비기능 요구사항·관련 API)와 핵심 의사결정 분기 |
| 2 | `02-sequence-diagrams.md` | 선택 | 도메인 협력·합의 가짓수가 본질인 시나리오를 Mermaid 시퀀스로 |
| 3 | `03-class-diagram.md` | 선택 | 시스템 전체 정적 구조를 도메인 모델(Entity·VO·Enum·관계) 중심의 Mermaid 클래스 다이어그램으로. UML 세 층위 중 명세/구현 지향 |
| 4 | `04-erd.md` | 선택 | 03 도메인 모델을 MySQL 물리 스키마(Mermaid `erDiagram` + DDL)로. 다음은 구현 |

본 스킬의 가치는 템플릿 채우기가 아니라, 한 줄 스토리에서 빠진 결정사항을 끌어내는 질문, 도메인 사전을 자라게 만드는 일관성 관리, 결정 분기와 그 이유의 보존, 큰 정책 전환 시의 정합성 전파에 있다.

## 사용 시점

- 한 문장짜리 유저 스토리 (예: "회원은 좋아요한 상품 목록을 본다") → Phase 1
- `docs/volume-N/` 경로를 가리키고 비어 있거나 누적 작성이 필요한 상태
- 각 phase 진입 신호: "시퀀스 그리자/02"(Phase 2), "클래스 다이어그램/도메인 모델/03"(Phase 3), "ERD/RDB 스키마/테이블 정의/04"(Phase 4)
- 직전 phase가 완성된 상태에서 사용자가 다음 단계로 진행 의사를 보임 → 진입 제안

## 사용하지 않는 경우

- 모호한 한 줄 기획에서 명세 + API + ERD 3종 산출물이 필요 → `specifying-requirements`
- 명세 완성, 코드 구현 시작 → `tdd-helper`
- 단순 개념 질문이나 코드 리뷰 → 일반 작업

## 먼저 수집할 입력

스킬 시작 시 다음 2가지를 확정한 뒤 본 루프로 들어간다.

1. **산출물 폴더 경로** — 예: `docs/volume-3/`. 비어 있으면 새로, 기존 파일 있으면 누적 모드.
2. **프로젝트명** — 예: `감성 이커머스`. `# '<프로젝트명>' 요구사항 명세` 제목에 사용.

기존 파일이 있으면 Read해 도메인 사전·기존 스토리·결정 번호·기존 시나리오를 메모리에 적재한다.

## 전역 작성 원칙

모든 산출물 공통:

- **한국어 작성**. 코드/식별자/SQL/API 경로만 영어.
- **헤더 명칭 고정**. `output-templates/`의 골격을 임의 변형하지 않는다.
- **도메인 어휘 우선**. 명세 본문은 자연어만. 구현 어휘(SQL·잠금·트랜잭션·인덱스·캐시 라이브러리)는 본문 노출 금지, 결정 분기의 트레이드오프 열에서만 허용.
- **결정 분기는 표**. `| 안 | 접근 | 트레이드오프 |` 3열. 트레이드오프 열 필수.
- **어휘는 "기록"으로 통일**. "박제" 같이 색깔 진한 어휘 회피.
- **참고 자료 출처·메타 정보(작성일·작성자) 산출물에 노출 금지**. git이 추적한다.

자세한 양식 규약은 phase별 가이드에.

## Phase 1 워크플로우 (필수)

```
- [ ] 1. 도메인 추론 + 식별자 부여 (예: BRD-3)
- [ ] 2. 문제 재해석 (내사고)
- [ ] 3. 5개 섹션 가안 작성
- [ ] 4. 사용자 검토 사이클
- [ ] 5. 결정 분기 감지 + 기록 질의
- [ ] 6. 결정 폐기·통합·분리 (필요 시)
- [ ] 7. 도메인·용어 추출
- [ ] 8. 01-requirements.md 즉시 누적 저장
- [ ] 9. 정합성·모순 점검 게이트 (종료 직전)
- [ ] 10. 종료 또는 Phase 2 전환
```

각 단계 종료 시점에 **"이대로 가도 될까요?"**라고 명시적으로 묻고 사용자의 명시적 승인 후에만 다음 단계로.

**상세**: `references/요구사항_명세_작성_가이드.md` — 가안 작성 전·매 저장 직전에 참조.
**골격**: `references/output-templates/요구사항_명세_템플릿.md`.

## Phase 2 워크플로우 (선택)

```
- [ ] 1. 시나리오 후보 분류 + 우선순위 합의
- [ ] 2. 양식 합의 (첫 진입 시 1회)
- [ ] 3. 시나리오 가안 작성 (4섹션)
- [ ] 4. 사용자 검토 사이클
- [ ] 5. 02-sequence-diagrams.md 즉시 누적 저장
- [ ] 6. 서문 작성·다듬기
- [ ] 7. 02 특화 점검 게이트
- [ ] 8. 종료 또는 Phase 3 전환
```

시퀀스 다이어그램은 모든 스토리에 그리지 않는다. 도메인 협력·동시성·합의 가짓수가 본질인 시나리오만.

**상세**: `references/시퀀스_다이어그램_작성_가이드.md` — 첫 진입 시 반드시 한 번 Read.
**골격**: `references/output-templates/시퀀스_다이어그램_템플릿.md`.

## Phase 3 워크플로우 (선택)

```
- [ ] 1. 도메인 진행 순서 합의 (01 도메인 순서와 일치 권장)
- [ ] 2. 도메인별 묶음 진행 — 튜터 모드 사이클 (메시지 → 메서드 → 필드)
- [ ] 3. 통합 다이어그램 + 도메인별 작은 다이어그램 + 도메인 모델 표 작성
- [ ] 4. 종합 점검 게이트
- [ ] 5. 종료 또는 Phase 4 전환
```

클래스 다이어그램은 UML 세 층위 중 **명세/구현 지향**. 개념적 has-a 관계는 04 ERD의 책임. 본 03은 객체 간 메시지·책임·Aggregate 경계가 본질.

**튜터 모드**: 학습성 도출 작업. 합리적 default를 만들고 검토만 받지 말고, 한 단계씩 옵션 A/B/C + 추천을 제시해 사용자 답을 매개로 진행. 사용자가 위임 모드("나머지 너가 다 작성해봐")로 전환하면 묶음별 도출 정리 → 통합 작성 → 자가 점검 → 사용자 피드백.

**상세**: `references/클래스_다이어그램_작성_가이드.md` — 첫 진입 시 반드시 한 번 Read.
**골격**: `references/output-templates/클래스_다이어그램_템플릿.md`.

## Phase 4 워크플로우 (선택)

```
- [ ] 1. Plan 단계 — 사전 결정 묶음 합의 (한 번에)
- [ ] 2. ERD 첫 작성 (Mermaid erDiagram + DDL)
- [ ] 3. 사용자 정정 사이클 (단순화 방향)
- [ ] 4. 점검 게이트 (DDL ↔ Mermaid 정합·COMMENT 일관성)
- [ ] 5. 종료
```

ERD는 도메인 결정의 잔물결이다. 결정 근거는 01·03에 있고, 04는 RDB 어휘 자체에 집중한다. "본 단계에 정말 필요한가" 관점으로 단순화하며, 사용자가 직접 정리한 부분은 외과적으로 보존한다.

**상세**: `references/ERD_작성_가이드.md` — 첫 진입 시 반드시 한 번 Read.
**골격**: `references/output-templates/ERD_템플릿.md`.

## 톤 & 스타일 (공통)

본 스킬은 **설계 리뷰 톤**을 유지한다. 강의 톤도, 정답 단정 톤도 아니다.

- **선택지로 제시**: 자명하지 않은 결정은 "A안과 B안 중 어느 쪽으로? 각각 ...의 트레이드오프가 있습니다"로. 정답을 단정하지 않는다.
- **의도·책임·경계 우선**: 구현 방식보다 "왜 이 책임을 누가 지는가", "어디가 경계인가"를 먼저 짚는다.
- **숨기지 않음**: 모호·트레이드오프가 있는 부분을 명시적으로 드러낸다.
- **동료 리뷰어처럼**: "이거 빠진 것 같은데요?" 식으로. 위에서 가르치는 톤 회피.
- **도메인 의미 우선**: "더 이상 운영하지 않는 X"와 "존재하지 않는 X"의 차이를 합치고 있지 않은지 자문.
- **YAGNI**: 본 라운드에 호출 흐름이 없는 모델(enum·상태 컬럼)을 미리 만들지 않는다.
- **단건과 목록은 갈릴 수 있다**: 정책이 같으리라 가정하지 않는다.
- **결과물에 애착 부리지 않기**: 더 나은 기준이 보이면 폐기·전면 재작성을 두려워하지 않는다.
- **외부 가이드는 본인 맥락으로 번역**: 멘토·동료가 준 가이드를 베끼지 말고 본인 프로젝트의 독자·규모로 번역해 적용.
- **외과적 정리**: 사용자가 직접 손본 부분은 그대로 두고, 명시 요청한 변경만 처리. 의도를 임의로 복원하지 않는다.

Phase 특화 톤(객관적 진술형 — Phase 2 / 튜터 모드 — Phase 3 / 단순화 정신 — Phase 4)은 해당 가이드 참조.

## 공통 Anti-patterns

- 사용자의 명시적 일괄 작성 요청 없이 N개를 묶어 처리. (기본은 한 항목씩)
- 가안을 사용자에게 보여주지 않고 곧장 파일에 기록.
- 헤더 명칭 임의 변형 (예: "전제 조건" → "사전 조건").
- 사용자가 외부 자료를 들고 와 재논의를 시작했는데 기존 결과물을 변호하느라 폐기 거부.
- "박제" 같이 색깔 진한 어휘 사용. ("기록"으로 통일)
- 산출물에 참고 자료 출처·메타 정보(작성일·작성자) 노출.

Phase 특화 Anti-patterns는 각 가이드 끝에.

## 산출물 파일 규약

매 단계의 산출을 파일로 옮길 때:

1. **Read**로 기존 파일을 읽어 사용자에게 보여준다.
2. **Diff 제안** — 추가·수정될 내용을 명확히.
3. **사용자 승인** 대기.
4. 승인 후 **Write/Edit**. 매번 전체를 다시 쓰지 말고 해당 섹션만.

- Mermaid 블록은 ```` ```mermaid ```` 펜스.
- 일괄 작성 모드에서 임시 staging 파일을 작성 폴더 안에 잠시 만드는 것은 허용. 최종 저장 직후 삭제.

## 중단 조건

다음 중 하나라도 발생하면 **즉시 멈추고 사용자에게 묻는다**.

- 사용자 답이 모호함 ("알아서 해줘") → 구체 선택지 제시.
- 도메인 약어 충돌 → 어느 쪽인지 확인.
- 결정 폐기 시 영향 범위가 큼 → 점검 자리를 같이 정한 뒤 진행.
- 한 응답에 다 못 담을 정도로 단계가 길어짐 → 더 잘게 쪼개기.

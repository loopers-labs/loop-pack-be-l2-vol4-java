## 클래스 다이어그램의 본질 — UML 세 층위

Fowler "UML Distilled"의 세 층위 중 본 03은 **명세/구현 지향**이다.

| 층위 | 표현 대상 | 산출물 |
| --- | --- | --- |
| **개념 (Conceptual)** | 도메인 개념·관계만 | **04 ERD 책임** (Brand has Products 같은 개념적 has-a) |
| **명세 (Specification)** | 인터페이스·계약·메시지·책임 분담 | **본 03의 책임** |
| **구현 (Implementation)** | 코드 구조·Aggregate 경계·ID 참조·DDD 전술 패턴 | **본 03의 책임** |

객체 간 메시지·책임·Aggregate 경계가 본질이지 개념적 소유 관계가 본질이 아니다. 개념적 has-a는 04 ERD에 맡긴다.

## 핵심 규율 — 메시지 우선, 필드는 메시지가 요구할 때

ERD와의 결정적 차이. **메시지(메서드 시그니처)를 먼저 도출**하고, 그 메시지가 요구하는 필드만 추가한다. 필드를 먼저 정하면 모델이 딱딱해져 수정 불가가 된다.

본 단계의 흐름:
1. 외부가 어떤 **메시지**를 보내는가? 수신자는 누구인가?
2. 그 메시지 처리에 필요한 **메서드 시그니처**는?
3. 메서드가 요구하는 **필드**는?

## 표현 대상

**포함**: Entity (Aggregate Root + 종속 Entity) / Value Object (박스로 표기, Entity 속성 타입으로 등장하면 다이어그램에) / Enum (`<<enumeration>>` 표기) / 관계 (composition `*--`, dependency `..>`, 연관)

**제외**:
- `Service`, `Facade`, `Repository`, `Controller`, `DTO` — 협력자·인프라 어댑터, 도메인 모델 아님.
- `BaseEntity` — 인프라성 부모 클래스.
- **단순 검사용 도메인 서비스** — "이름 중복 검사" 같은 단순 제약은 응용 계층이 Repository와 협력. 도메인 서비스는 환율·가격 책정 같은 본질 정책이 풍부할 때만.

## 튜터 모드

학습성 도출 작업이므로 합리적 default를 만들어 검토만 받지 않고 **한 단계씩 질문**한다.

- "외부가 Brand에 어떤 메시지를 보내야 할까요? 옵션 (a) `create(name, description)` / (b) `BrandFactory.create(...)` / (c) `new Brand(...)`. 추천은 (a) — 자기 생성 책임은 자신이."
- 사용자 답 후 다음 단계 질문 (메서드 → 필드 순)

단순 결정(에러 픽스·이미 합의된 양식·문법 정정)은 그대로 처리.

사용자가 **위임 모드**("나머지 너가 다 작성해봐")로 전환하면 묶음별 도출 정리 → 통합 작성 → 자가 점검 → 사용자 피드백.

## 설계 산출물의 도메인 의미 어휘

설계 산출물(01·02·03)의 메서드명·메시지 라벨은 **도메인 의미가 즉시 보이는 어휘**로.

- 다이어그램: `create`, `update`, `delete`, `decreaseStock`, `isAvailable`, `matches`
- 코드 컨벤션(`from`, `of`, `build`): 구현 단계에서 강제. 다이어그램 `create` ↔ 코드 `of` 매핑은 구현 책임.

설계 산출물의 독자에 PM·디자이너가 포함된다는 점이 이유.

## 워크플로우

```
1. 도메인 진행 순서 합의 (01 도메인 순서와 일치 권장: User → Brand → Product → Like → Order)
2. 도메인별 묶음 진행
   2-1. 튜터 모드 사이클 (메시지 → 메서드 → 필드)
   2-2. 응용 계층 책임은 도메인 모델 표에 명시 (다이어그램 X)
3. 통합 다이어그램 + 도메인별 작은 다이어그램 + 도메인 모델 표
4. 종합 점검 게이트
5. 종료 또는 Phase 4 전환
```

## 단계 1. 도메인 진행 순서 합의

01-requirements.md의 도메인 순서를 그대로 따르는 것이 일반적. 각 도메인 묶음 안에서는 **Aggregate Root → 종속 Entity → VO → Enum** 순서로 도출. 공통 VO(`Money` 등)는 별도 도메인 헤더 없이 namespace 밖 다이어그램에만.

## 단계 2. 도메인별 묶음 진행 — 튜터 모드 사이클

### 2-1. 메시지 도출

해당 도메인의 01 유저 스토리들에서 외부가 도메인에 보내는 메시지를 도출.

예 — BRD-4 (브랜드 등록):
- 외부: 관리자 (응용 계층 경유)
- 메시지: "새 브랜드를 만들어라"
- 수신자: `Brand` 객체 자신 (생성 책임은 자기 자신)
- 메서드 시그니처: `create(name, description) Brand`

### 2-2. 응용 계층 책임 분리

도메인 객체에 자연스럽지 않은 책임은 응용 계층으로. **다이어그램에 그리지 않고** 도메인 모델 표의 "책임" 컬럼에 "응용 계층 책임"으로 명시.

응용 계층 책임:
- **중복 검사** (이름·로그인 ID 중복) — Repository 협력 필요
- **다른 Aggregate 변경** — 재고 차감, cascade 삭제, 좋아요 토글 시 카운트 갱신
- **트랜잭션 단위** — 거래 원자성 보장
- **동시성 보장** — 잠금·원자적 갱신
- **검증** — VO의 `create()` 팩토리에 단일화되어 있으면 별도 메서드 불요

### 2-3. 메서드 → 필드 도출

메서드 시그니처가 정해지면 그 메서드가 요구하는 필드만 추가. 필드 우선 도출 금지.

### Aggregate 경계

- **같은 Aggregate 내부**: 객체 참조 가능 (Order ↔ OrderItem)
- **다른 Aggregate**: `+Long userId` 같은 **ID 참조** (정통 DDD)
- **다른 Aggregate 정보가 필요하면 Snapshot 패턴**: `OrderItem`이 `productName`·`productBrandName`·`unitPrice`를 평탄한 필드로 보존 — Product 변경에 영향받지 않음

**판단 기준**: 함께 변하는가, 따로 변하는가. 같은 Aggregate면 한 트랜잭션 안에서 함께 변한다. Brand·Product는 별도 Aggregate (PRD-1~7이 Brand 무관하게 Product 자체 시나리오).

## 단계 3. 산출물 작성

### 통합 클래스 다이어그램 (전체 한 장)

- 공통 VO (`Money` 등)는 namespace 밖
- 도메인은 `namespace <도메인 영어 복수형>` (예: `Users`, `Brands`, `Products`, `Likes`, `Orders`)
- 관계 화살표는 namespace 안에 둘 수도 밖으로 모아둘 수도 있음 (가독성 우선)

### 도메인별 작은 다이어그램 + 도메인 모델 표

각 도메인 섹션은:
1. **다이어그램** — 해당 도메인의 클래스 + 외부 도메인 `<<external>>` placeholder + 관계
2. **도메인 모델 표** — `| 객체 | 종류 | 책임 |` 3열. 종류는 `Entity`/`Entity (Aggregate Root)`/`VO`/`Enum`. 책임에는 자기 메서드 책임 + 응용 계층 책임 명시.

도메인 진행 순서는 01과 일치.

### 다이어그램 표기 규칙

- **Mermaid `classDiagram`**
- **Static factory 메서드**: `+create(...)$ <ReturnType>` — `$` 표기로 static 명시
- **Composition (보유 관계)**: `*--`. 보유자 → 보유 객체 방향. 예: `Product *-- Stock : create, isAvailable, decrease`
- **Dependency (참조만)**: `..>`. 같은 다이어그램에 외부 도메인을 표기할 때
- **다중성**: `"1" *-- "1..*"` (1개 → 1개 이상)
- **메시지 라벨**: composition 화살표 라벨에 호출하는 메서드명들.
- **namespace로 도메인 격리**: 통합 다이어그램에서 도메인별로 묶기
- **외부 도메인**: 도메인별 작은 다이어그램에서 다른 도메인을 가리킬 때 `<<external>>` placeholder
- **공통 VO** (`Money` 등): 어느 namespace에도 속하지 않음. 통합 다이어그램 최상단. 별도 도메인 헤더 만들지 않음.

## 단계 4. 종합 점검 게이트

| 점검 항목 | 무엇을 보는가 |
| --- | --- |
| **도메인 의미 어휘** | 메서드명이 `create`·`update`·`delete` 같은 도메인 의미인가. `from`·`of`·`build` 같은 코드 컨벤션이 노출되어 있지 않은가 |
| **VO 박스** | Entity 속성 타입으로 등장하는 VO가 박스로 표기되어 있는가 |
| **FK 표기** | 다른 Aggregate를 참조할 때 ID 참조(`+Long userId`)인가. 객체 참조로 되어 있지 않은가 |
| **공통 VO 위치** | `Money` 같은 공통 VO가 namespace 밖에 있는가. 별도 도메인 헤더가 만들어져 있지 않은가 |
| **외부 도메인 표기** | 도메인별 작은 다이어그램에서 다른 도메인이 `<<external>>` placeholder로 등장하는가 |
| **응용 계층 책임 누출** | 중복 검사·cascade·재고 차감 같은 응용 계층 책임이 도메인 객체 메서드로 잘못 그려져 있지 않은가 |
| **인프라 어휘 제거** | Service·Repository·Facade·Controller·BaseEntity가 다이어그램에 등장하지 않는가 |
| **composition 라벨** | `*--` 화살표에 호출 메서드 라벨이 붙어 있는가 |
| **다중성 표기** | `"1" *-- "1..*"` 같은 다중성이 의미 있는 곳에 표기되어 있는가 |
| **개념적 has-a 누출** | 04 ERD가 책임질 개념적 has-a 관계를 본 03에 그려놓지 않았는가 |
| **Snapshot 패턴** | 다른 Aggregate 정보가 필요한 경우 평탄한 스냅샷 필드로 보존되어 있는가 |
| **Enum 임의 추가 금지** | 01 기획에 없는 Enum이 임의로 추가되어 있지 않은가 |

점검 결과를 보고하고 항목별 반영 여부 선택.

## Phase 3 특화 Anti-patterns

본문 룰을 어기는 흔한 함정:

- VO를 단순 속성으로만 표현하고 박스에서 빼기. (Entity 속성 타입으로 등장하면 박스로)
- 다른 Aggregate인데 객체 참조로 그리기. (ID 참조 + 필요 시 Snapshot)
- 다른 Aggregate 변경 메서드(cascade·재고 차감 등)를 도메인 객체에 그리기. (응용 계층 책임)
- 단순 검사용 도메인 서비스를 다이어그램에 그리기.
- 코드 컨벤션 메서드명(`from`·`of`·`build`)을 다이어그램에 노출. (`create`로)
- `Service`·`Repository`·`Facade`·`Controller`·`BaseEntity`가 다이어그램에 등장.
- 시나리오 단위로 클래스 다이어그램 작성. (시나리오 단위는 02 책임)
- 튜터 모드 위반 — default 다이어그램 만들고 사용자에게 검토만 받기.
- 메시지 도출 없이 필드 먼저 박제 (ERD화).
- 개념적 has-a 관계를 본 03에 그리기. (04 책임)
- 01 기획에 없는 Enum·필드 임의 추가.
- composition 다이아몬드 방향·메시지 라벨·다중성 누락.

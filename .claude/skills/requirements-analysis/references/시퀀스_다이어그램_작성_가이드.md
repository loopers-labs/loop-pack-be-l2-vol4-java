# Phase 2 가이드 — `02-sequence-diagrams.md` 작성 (선택)

본 문서는 Phase 2의 단계별 워크플로우와 양식 가드레일을 담는다. 산출물 골격은 `output-templates/시퀀스_다이어그램_템플릿.md`에 있다.

## 진입 조건

다음 중 하나에서 가동.

- 사용자가 명시적으로 "시퀀스 다이어그램 그리자" / "02 작성하자" / "이 시나리오 다이어그램으로" 같은 발화
- Phase 1 종료 시 진입 의사 확인 단계에서 사용자 동의
- 기존 `02-sequence-diagrams.md`에 새 시나리오를 누적하려 함

진입 시 기존 02가 있으면 Read해 기존 시나리오·서문·양식 일관성을 메모리에 적재.

## 시퀀스 다이어그램의 본질

**구현 청사진이 아니라 도메인 협력의 설명 도구**다. 코드로의 변환은 일반 코딩 도구가 더 잘하며, 시퀀스의 가치는 다음에 있다.

- 글만으로 흐름의 형상이 잘 떠오르지 않는 시나리오를 **그림 한 장으로 압축**해 합의를 빠르게.
- 도메인 간 메시지·책임 경계·거래 단위를 모아 **PM·디자이너 등 비개발자와도 같이 읽는** 도구.
- 1년 뒤에도 살아남도록 **"하우"가 아닌 "와이"**를 본문에 남긴다. 한 땀 한 땀 그린 절차는 요구사항 변경에 깨지지만 결정의 이유는 살아남는다.

이 본질이 모든 양식 규약의 근거다.

## 워크플로우

```
1. 시나리오 후보 분류 + 우선순위 합의
2. 양식 합의 (첫 진입 시 1회)
3. 루프
   3-1. 시나리오 한 개 가안 작성 (또는 합의된 N개 일괄)
   3-2. 사용자 검토 사이클
   3-3. 즉시 누적 저장
4. 서문 작성·다듬기
5. 02 특화 점검 게이트
6. 종료 또는 Phase 3 전환
```

## 단계 1. 시나리오 후보 분류와 우선순위 합의

01-requirements.md의 모든 유저 스토리에 다음 기준 적용.

### 후보 선정 (둘 이상 해당)

- **다중 도메인 협력**: 두 개 이상 도메인이 한 거래 안에서 메시지를 주고받는다.
- **동시성·일관성 결정 다수**: 동시 요청·교착·원자성 등 합의해야 할 정책이 여럿.
- **거래 단위 분기 다수**: 전부 성공·부분 실패·원복·멱등 같은 결정이 한 시나리오에 모인다.
- **결정 참조 다수**: 01의 핵심 의사결정 분기를 두 개 이상 동시에 참조.

### 제외

- 단순 CRUD (단건 조회·등록·수정·삭제)
- 인증 분기처럼 모든 API 공통인 분기만 있는 시나리오
- 한 도메인 내부에서 끝나는 시나리오

후보 분류를 짧게 보여주고 진행 순서를 제안. 자연스러운 순서는 **단건 → 일반화 → 다른 도메인** (예: ORD-1 → ORD-2 → BRD-6).

## 단계 2. 양식 합의 (첫 진입 시 1회)

`output-templates/시퀀스_다이어그램_템플릿.md`의 양식을 한 시나리오 분량으로 미리 보여준다. 다음 분기에 답한다.

1. **양식 OK?** 4섹션(`### 그린 이유` / `### 한 줄 요약` / `### 다이어그램` / `### 추가 고려사항`) 구조 동의
2. **참여자 명명 컨벤션** — `<영어> as <한글 도메인>` 형식 동의 (예: `Order as 주문 도메인`)
3. **언해피 패스 범위** — 비즈니스 결정이 갈리는 핵심 분기만, 인증·입력 검증 같은 공통 분기는 제외

합의 후 매 시나리오마다 다시 묻지 않는다.

## 단계 3. 시나리오 가안 작성

시나리오 한 개씩 (또는 양식 합의 후 사용자 명시 요청 시 N개 일괄) 가안 작성. 매 가안은 4섹션을 채운다.

### `### 그린 이유` — 객관적 진술형

- **첫 문장**: 시나리오 정의 — "~ 시나리오다" 명사형으로 마무리. 합의 가짓수의 양으로 본질을 적는다.
  - 예: "단일 상품 주문 한 건이지만 차감 시점·동시성 약속·스냅샷 기록·거래 단위·실패 분기가 한 흐름으로 모이는 시나리오다."
- **두 번째 문장**: 기록 대상 — "~ 흐름을 시퀀스 다이어그램으로 기록한다".

**금지 표현** (주관적 풀이): "머릿속에", "글로 적으면", "직관이 안 잡혀". 1년 뒤 다시 봐도 "왜 이 시나리오를 시퀀스로 그렸나"가 합의의 양으로 설명되어야 살아남는다.

### `### 한 줄 요약`

비개발자가 다이어그램 진입 전에 한 번 읽고 흐름이 잡히도록. 한 문단 안에서 마무리.

- 흐름 전체를 평이한 도메인 어휘로 풀어쓴다.
- 분기는 핵심 한두 개만 압축 (예: "성공하면 ~, 실패하면 ~ 처리").
- 백엔드 어휘(트랜잭션·SQL·잠금) 금지.

예:

> "회원이 상품 한 종을 N개 주문하면, 주문 도메인은 상품 도메인에게 차감을 요청한다. 차감이 성공하면 응답으로 받은 상품 정보(이름·브랜드명·단가)를 주문 항목으로 기록해 그 시점의 거래를 영수증처럼 남기고, 주문 식별자·상태·주문 시각·총 결제 금액과 함께 회원에게 돌려준다."

### `### 다이어그램` — Mermaid `sequenceDiagram`

- **참여자는 액터 + 도메인 단위**. `Controller`/`Facade`/`Service`/`Repository`/`Resolver` 같은 레이어 명 금지.
- **참여자 명명**: 액터는 한국어 풀네임(`회원`/`관리자`/`비회원`), 도메인은 `<영어 도메인> as <한글 도메인>` (예: `participant Order as 주문 도메인`).
- **메시지 어휘는 도메인 어휘**. "차감 요청", "차감 완료 + 상품 정보" 같은 표현. 백엔드 구현 어휘(`@Transactional`, `affectedRows`, `findByIdAndDeletedAtIsNull`, SQL, 잠금 종류, `UPDATE ... WHERE`)는 노출 금지.
- **HTTP 상태 코드 노출 금지**. 응답은 도메인 어휘로(`주문 완료`, `주문 실패 (재고 부족)`).
- **언해피 패스는 비즈니스 결정이 갈리는 핵심 분기만**. 인증 실패·입력 형식 검증 같은 공통 분기는 뺀다 (01 Exception Flow에 이미 있음).
- **거래 단위 표현은 `Note over A,B`**. `rect rgb(...)` 블록은 강조가 강해 비개발자 가독성을 깬다. `Note over Order,Product: 차감·기록·저장은 같은 거래 단위로 묶여<br/>모두 성공하거나 모두 실패` 같은 텍스트 노트로.
- **Note 위치는 적용되는 분기 안**. `alt` 바깥에 두면 모든 분기에 시각적으로 적용되어 모순이 생긴다.
- **결정 번호는 다이어그램 본문이 아닌 추가 고려사항에서 참조**. `(결정 N)` 표기는 추가 고려사항에만.

### `### 추가 고려사항`

다이어그램에 시각적으로 표현하지 않은 와이(트랜잭션 경계·동시성 처리·추후 변경 자리)는 모두 텍스트로 보강.

- **항목별 헤더**: 굵게 + 결정 참조. `- **<주제> (결정 N)**: <왜·의미>`.
- **추후 변화 자리**: 결제 도입·옵션 추가 등 명세에 잠재해 있는 변경점은 별도 항목으로.
- 다이어그램이 빈약해질수록 이 섹션이 커진다 — 자연스럽다, 와이가 본질이므로.

## 단계 4. 사용자 검토 사이클

가안 한 시나리오를 보여주고 의견을 받는다. 수정은 즉시 반영. 명시적 승인 후 다음 단계로.

**선택적 반영**: 사용자가 점검 결과를 받고 "1·2·4번만 반영하고 3번은 의도적으로 뺐다"고 답하는 경우가 잦다. 모든 점검 항목을 강요하지 말고 사용자의 선택을 그대로 따른다.

## 단계 5. 즉시 누적 저장

매 시나리오 검토가 끝나면 `{folder}/02-sequence-diagrams.md`를 갱신.

1. `# '<프로젝트명>' 시퀀스 다이어그램` 제목 + 서문 (없으면 추가)
2. 시나리오 본문 (수평선 `---` 구분, 01의 스토리 헤딩과 동일 형식 `## <스토리 ID>. <스토리 문장>`)

스토리 헤딩은 `01-requirements.md`와 **동일한 형식**. 별도 `SEQ-` 번호를 새로 부여하지 않는다.

## 단계 6. 서문 작성·다듬기

모든 시나리오가 마무리되면 서문을 두 문장 안팎으로.

- **본 문서가 무엇을 기록하는지** (정의형, "기록한다"로 마무리)
- **어떤 시나리오를 뽑았는지·무엇은 뺐는지**의 기준 한 줄
- **표기 규칙**: 도메인 어휘 사용, 백엔드 구현 어휘는 추가 고려사항에 텍스트로

예:

> "본 문서는 [01-requirements.md](01-requirements.md)의 유저 스토리 중 도메인 간 협력이 본질이거나 합의해야 할 정책이 많아 글만으로 흐름의 형상이 잘 떠오르지 않는 시나리오를 그림으로 기록한다. 도메인 어휘로 적어 PM·디자이너도 따라갈 수 있게 하고, 백엔드 구현 어휘(트랜잭션·SQL·잠금)는 다이어그램에서 빼고 추가 고려사항에 텍스트로 보강한다."

## 단계 7. 02 특화 점검 게이트

종료 전 자동 점검 (Phase 1 점검 게이트에 더하여).

| 점검 항목 | 무엇을 보는가 |
| --- | --- |
| **참여자 단위** | 액터 + 도메인만 등장하는가. 레이어 명(Controller/Service/Repository)이 누출되어 있지 않은가 |
| **메시지 어휘** | 도메인 어휘로만 표기되는가. SQL·잠금·트랜잭션 어휘·HTTP 상태 코드가 노출되어 있지 않은가 |
| **공통 분기 제외** | 인증 실패·입력 검증 같은 모든 API 공통 분기가 빠져 있는가 |
| **Note 위치** | 거래 단위·정책 설명 Note가 자신이 적용되는 분기 안에 놓여 있는가 |
| **그린 이유 톤** | 객관적 진술형(`~ 시나리오다`·`~를 기록한다`)으로 통일되었는가 |
| **결정 참조 유효성** | `(결정 N)` 참조가 01의 실제 번호와 일치하는가. 폐기된 결정을 가리키지 않는가 |
| **단순 CRUD 누출** | 시나리오 선정 기준을 어긴 단건 CRUD가 그려져 있지 않은가 |
| **서문 존재·길이** | 서문이 있고 두 문장 안팎으로 간결한가 |

## Phase 2 특화 Anti-patterns

본문 룰을 어기는 흔한 함정:

- 모든 유저 스토리에 시퀀스 다이어그램 작성. (선정 기준 적용)
- 참여자에 레이어 명(`Controller`/`Service`/`Repository`/`Facade`/`Resolver`) 등장.
- 메시지·응답에 SQL·잠금·트랜잭션 어휘·HTTP 상태 코드 노출.
- 인증 분기·입력 검증 같은 모든 API 공통 분기를 다이어그램에 포함.
- 거래 단위 Note를 `alt` 바깥에 두어 모든 분기에 시각적으로 적용.
- "그린 이유"에 주관적 풀이("머릿속에", "글로 적으면", "직관이 안 잡혀") 사용.
- 시퀀스 본문에 결정 번호 노출 (참조는 추가 고려사항에서만).
- 한 땀 한 땀 모든 구현 단계를 그려 1년 뒤 깨질 청사진 만들기.
- 사용자가 직접 수정한 부분을 점검하면서 의도를 무시하고 강제 반영.

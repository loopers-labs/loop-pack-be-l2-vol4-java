# Phase 1 가이드 — `01-requirements.md` 작성

본 문서는 Phase 1의 단계별 워크플로우와 양식 가드레일을 담는다. 산출물 골격은 `output-templates/요구사항_명세_템플릿.md`에 있다.

## 워크플로우

```
1. 입력 수집 (폴더 + 프로젝트명)
2. 루프
   2-1. 유저 스토리 한 문장 제시 (또는 명시 요청 시 N개 일괄)
   2-2. 도메인 추론 + 식별자 부여
   2-3. 문제 재해석 (내사고)
   2-4. 5개 섹션 가안 작성
   2-5. 사용자 검토 사이클
   2-6. 결정 분기 감지 시 기록 질의
   2-7. 결정 폐기·통합·분리 (필요 시)
   2-8. 도메인·용어 추출
   2-9. 즉시 누적 저장
3. 사용자 "더 없다" → 정합성 점검 게이트 → Phase 2 의사 확인 → 종료/전환
```

각 단계 종료 시점에 **"이대로 가도 될까요?"** 명시적 확인 후 다음 단계로.

---

## 단계 1. 도메인 추론과 식별자 부여

1. **주 도메인 추론**: 핵심 명사(브랜드/상품/좋아요/주문/회원 등)에서 도메인 식별. 새 도메인이면 약어(BRD/PRD/LIK/ORD/USR/ADM 등) 정함.
2. **액터 추론**: 회원/비회원/관리자 중 누가 수행하는지.
3. **식별자 부여**: 같은 도메인 약어의 마지막 번호 + 1. 첫 스토리는 `-1`.
4. **확인 요청**: `<약어>-<번호>. <원문 또는 정제된 한 문장>`로 짧게.

- **관리자 도메인 분리 vs 통합**: 첫 관리자 스토리 등장 시 한 번 묻는다. 통합이 보통 더 응집도 높음.
- **약어 충돌 시**: 한 글자 확장(예: 회원 `USR` vs 관리자용 회원 `UAD`).
- **스토리 ID는 한 번 부여하면 재정렬하지 않는다** — 외부 산출물·계획서·이슈 참조의 안정성 때문.

---

## 단계 2. 문제 재해석 (내사고)

가안 작성 **전에**, 유저 스토리를 "무엇을 만드는가"가 아닌 **"어떤 문제를 왜 해결하는가"**로 한 줄 재진술해 내사고로 갖는다. 본문에는 적지 않는다.

> 스토리: "회원 또는 비회원은 특정 브랜드의 정보를 확인할 수 있다."
> 재해석: "단순 조회가 아니라, 삭제된 브랜드의 노출을 차단하고 관리자 전용 정보 노출 범위를 검증하는 게이트 정책이 본질."

재해석을 거치면 전제 조건의 운영 상태 점검, Exception Flow의 "존재하지 않음" 케이스, 비즈니스 규칙의 노출 범위 정책이 자연스럽게 드러난다. 재해석이 약하면 가안이 단순 동작 나열로 흐른다 — 이 단계를 생략하지 않는다.

---

## 단계 3. 5개 섹션 가안 작성

다음 5개 섹션을 작성한다. **모두 적을 수 있는 만큼 적되, 자명한 항목까지 억지로 채우지 않는다.** 비즈니스 규칙·비기능 요구사항이 특별히 없으면 섹션 자체를 생략.

| 섹션 | 작성 기준 |
| --- | --- |
| 전제 조건 | 인증 상태, 대상 자원의 운영 상태, 요청 형식 등 호출 전에 만족해야 하는 조건 |
| 기능 요구사항 | Main Flow (번호 단계) / Alternate Flow / Exception Flow |
| 비즈니스 규칙 | 도메인 정책(중복·정렬·페이지 크기·노출 범위 등). 자명하면 생략 |
| 비기능 요구사항 | 동시성·응답 시간·보안·일관성 등. 해당 없으면 생략 |
| 관련 API | `<HTTP METHOD> <URL>` 형태로 1줄씩. **항상 마지막** |

### 가안 작성 원칙

- Main Flow는 시스템 관점 단계별 동작. 사용자 행동과 시스템 응답이 교차되며 마지막 응답까지 끝맺기.
- Exception Flow는 "어느 입력 / 어느 상태 → 어떤 응답" 매핑. 응답 어휘는 `해당 자원을 찾을 수 없다` / `자원 충돌` / `입력 검증 실패` / `인증 실패`로 통일.
- 명세 본문은 **자연어로만**.
- **계산이 들어가는 응답 필드**(총 결제 금액·총 항목 수 등)는 산식을 비즈니스 규칙에 한 줄.
- **단건과 목록의 정책은 갈릴 수 있다**. 응답 필드와 포함 항목이 다를 수 있음을 미리 짚는다.

### 양식 가드레일

- **섹션 헤더 순서**: 전제 조건 → 기능 요구사항 → (비즈니스 규칙) → (비기능 요구사항) → 관련 API. 관련 API는 항상 마지막.
- **단계 번호 들여쓰기**: Main Flow는 `    1.`(4-space + 숫자), Alternate/Exception Flow는 `    - `(4-space + 대시).
- **백틱 코드 표기**: 도메인 영어 이름(`Like`, `OrderItem`)과 API 경로(`GET /api/v1/...`)는 백틱.
- **표 형식**: 도메인 사전·액터 정의는 `| 한글 | 영어 | 의미 |` 3열.
- **도메인 그룹 구분**: 도메인 섹션 사이에 수평선 `---`.
- **API URL 컨벤션**: 회원/비회원 `/api/v1/...`, 관리자 `/api-admin/v1/...`.
- **API URL 쿼리 파라미터**: 비즈니스 규칙에 페이지네이션·정렬·필터가 있으면 URL에 함께 표기. 예: `GET /api/v1/products?brandId={brandId}&sort=latest&page=0&size=20`.

### 구현 디테일 제외 룰

명세 본문(전제 조건·기능 요구사항·비즈니스 규칙·비기능 요구사항)은 자연어로만.

- **노출 금지**: DB Unique 제약, 비관/낙관 잠금 어휘, `SELECT ... FOR UPDATE` 같은 SQL, 인덱스 종류, ORM 기능명, 트랜잭션 격리 수준 enum, 캐시 라이브러리 이름.
- **노출 가능**: 도메인 의미 표현 ("재고가 음수가 되지 않아야 한다", "동시 등록 요청이 한 건만 남는다", "같은 트랜잭션 안에서 처리한다").
- **예외 — 결정 분기의 트레이드오프 열에서는 구현 어휘 허용**. 옵션 비교가 본질이므로 "원자적 조건부 갱신", "비관 잠금", "캐시 미스 폴백" 같은 어휘로 트레이드오프를 짚어야 한다.

## 단계 4. 사용자 검토 사이클

가안 5개 섹션을 모두 보여주고 의견을 받는다. 수정 요청은 즉시 반영하고 다시 보여준다. 명시적 승인이 있을 때만 다음 단계로.

**일괄 작성 모드**: 사용자가 명시적으로 "N개 한 번에"라고 요청하면 N개 가안을 한 번에(예: 동질 패턴인 관리자 CRUD 12개). 항목별 피드백 사이클은 동일. 임시 staging 파일(예: `temp-admin.md`)을 작성 폴더 안에 잠시 만드는 것은 허용하며 최종 저장 직후 삭제.

---

## 단계 5. 결정 분기 감지와 기록 질의

다음 신호 **중 하나라도** 발생하면 결정 분기다.

1. **스킬이 A안/B안/C안을 명시적으로 제시**한 경우
2. **사용자가 재논의**를 시작한 경우 ("이게 맞나?", "다른 방식은?")
3. **트레이드오프가 있는 정책 선택**이 논의된 경우 (예: 멱등 방식, 차감 시점, 삭제 정책, 스냅샷 범위)
4. 가안에 포함된 결정이 다음 **세 카테고리** 중 하나:
   - **정책 결정** — 기준 시점, 성공/실패 조건, 예외 처리 규칙
   - **경계 결정** — 어디까지가 한 책임인지, 어디서 분리되는지
   - **확장 결정** — 나중에 바뀔 가능성이 있는 자리

가안 승인 직전에 묻는다.

> "이 항목에 결정 분기가 있었습니다(요약: <한 줄 요지>). 핵심 결정사항으로 기록할까요?"

사용자 동의 시 `핵심 의사결정 분기` 섹션에 **표 기반 양식**으로 추가.

### 기록 양식

```markdown
### 결정 <번호>. <분기 제목>

**문제**: <무엇이 결정 대상이었는지 한 문단>

| 안 | 접근 | 트레이드오프 |
|---|---|---|
| **A (채택)** | <접근 한 줄> | <영향도: 구현 부담·확장성·동시성·운영 비용 중 핵심 한 줄> |
| B | <접근> | <영향도> |
| C | <접근> | <영향도> |

**사유**: <왜 채택안인지>

**추후 개선**: (선택)
```

**트레이드오프 열은 필수**. 영향도 없이 단순 나열만 하면 후속 추적 불가. 채택안은 `**A (채택)**` 형식으로 굵게 + 라벨.

단순 표현 교정·자명한 선택은 결정 분기로 보지 않음. 사용자가 "기록 안 해도 된다"고 하면 묻고 넘어간다.

## 단계 6. 결정 폐기·통합·분리

작업 도중 큰 정책 전환이 발생할 수 있다 (예: "비활성화" 개념 자체를 폐지하고 "삭제"로 통일).

### 영향 점검 자동 가동

사용자가 이미 기록된 정책을 뒤집는 발언을 하면 스킬이 먼저 짚는다.

> "이 변경은 결정 N과 스토리 X·Y의 전제를 무너뜨립니다. 영향 받는 자리를 같이 점검할까요?"

### 폐기 결정 처리

- 본문에서 해당 결정 섹션을 **완전히 제거** ("폐기됨" 표시만 남기지 않음)
- 결정 번호 **재정렬** (빈자리 없이 연속)
- 다른 스토리의 `(결정 N 참조)`가 폐기 결정을 가리키지 않는지 점검 후 갱신

### 어휘 일괄 교체

"운영 중단" → "삭제" 같은 어휘 전환이 동반되면 사전·액터·모든 스토리 본문에서 일관 교체. **02-sequence-diagrams.md가 작성되어 있으면 02 본문·다이어그램 메시지·추가 고려사항까지 함께 교체**한다.

### 결정 통합/분리

한 결정 블록에 여러 의사결정이 묶여 있으면 (예: "재고 차감 시점·보관 위치·동시성 메커니즘"이 한 결정에 들어감) 의미 단위로 분리. 분리 후 각 결정의 트레이드오프 표는 자기 결정의 안만.

**스토리 ID는 재정렬하지 않음** (결정 번호만 재정렬).

## 단계 7. 도메인 및 용어 추출

승인된 스토리에서 새 명사·값 객체·상태 어휘가 등장하면 `도메인 및 용어 정의`에 추가.

### 도메인 사전 풍부화 룰

- **본질 속성을 함께 적는다**. 길이·범위·중복 정책 같은 도메인 제약이 함께 들어가 검증 룰의 정의소가 된다.
  - 예: `브랜드 이름 / Name / 1자 이상 50자 이하 문자열, 삭제되지 않은 브랜드 사이에서 중복될 수 없다.`
- **메타 필드는 제외**. `isActive`, `createdAt`, `updatedAt`, `deletedAt`, 운영 메모 등은 도메인 본질이 아니라 운영·기술 메타.
- **영어명 컨벤션**. 도메인 자체와 본질 속성 모두 PascalCase: `Brand`, `Product`, `Name`, `Price`, `Stock`, `Quantity`.
- **값 객체도 도메인 표 안에**. `Price`·`Stock`·`Quantity`·`UnitPrice` 같은 값 객체는 별도 섹션 없이 해당 도메인 표에.
- **이미 정의된 도메인·용어는 절대 중복 정의하지 않고 재사용**.
- 액터(`회원`/`비회원`/`관리자`)는 첫 등장 시 `액터 정의`에 추가. 인증 헤더 정보도 함께.

---

## 단계 8. 즉시 누적 저장

매 스토리 검토가 끝나면 `{folder}/01-requirements.md`를 다음 순서로 갱신한다.

1. `# '<프로젝트명>' 요구사항 명세` 제목 + 서문 (이미 있으면 유지)
2. `## 도메인 및 용어 정의` (도메인별 표 병합, 본질 속성 포함)
3. `## 액터 정의` (병합)
4. 도메인별 유저 스토리 섹션 (`## <도메인 한글> (<영어>)` 아래에 `### <약어>-<번호>. ...`)
5. `## 핵심 의사결정 분기` (있을 때만, 결정 번호 순, 표 기반)

도메인 그룹은 `## <도메인 한글> (<영어>)` 단위로 묶고 그 사이는 수평선 `---`로 구분. 매번 전체를 다시 쓰지 말고 해당 섹션만 추가·갱신.

서문(선택): "본 문서는 '<프로젝트명>'의 기능에 대한 합의 사항을 기록한다. 한 줄 요청만으로는 드러나지 않는 전제·예외·정책을 풀어 적고, 트레이드오프가 있는 결정은 핵심 의사결정 분기에 그 이유와 함께 남긴다."

## 단계 9. 정합성·모순 점검 게이트 (종료 직전)

사용자가 "더 없다", "끝", "마무리"라고 말하면 종료 전에 다음을 **자동 가동**한다.

| 점검 항목 | 무엇을 보는가 |
| --- | --- |
| **어휘 일관성** | 같은 개념을 다른 어휘로 쓰고 있지 않은가 (예: "운영 중단" vs "삭제" 혼용) |
| **상호 참조 유효성** | `(결정 N 참조)`가 실제 N과 맞는가, 폐기된 결정을 참조하고 있지 않은가 |
| **단건·일반화 정합성** | ORD-1(단건)과 ORD-2(일반화) 같은 케이스에서 동일 계산 규칙·정책이 양쪽에 명시되어 있는가 |
| **계산 규칙 누락** | 총 금액·총 항목 수 같은 계산 응답 필드의 산식이 비즈니스 규칙에 한 줄 있는가 |
| **URL ↔ 비즈니스 규칙 정합성** | 비즈니스 규칙의 페이지네이션·정렬·필터가 관련 API URL에 표기되어 있는가 |
| **API URL 컨벤션** | 회원/비회원 `/api/v1/...`, 관리자 `/api-admin/v1/...` 분리 일관 |
| **메타 필드 누출** | 도메인 사전에 `isActive`·`createdAt` 등 메타 필드가 잘못 들어가 있지 않은가 |
| **구현 디테일 누출** | 명세 본문에 DB Unique·잠금 명·SQL·인덱스 종류가 노출되어 있지 않은가 |
| **결정 양식 완정성** | 모든 결정에 트레이드오프 열이 채워져 있는가 |

결과 보고 형식:

```
점검 게이트 결과:
- 어휘 일관성: 통과
- 상호 참조: 결정 4 참조 1건이 폐기된 결정 가리킴 → 갱신 필요
- 계산 규칙 누락: ORD-1 총 결제 금액 산식 누락
...
```

사용자가 항목별 반영 여부를 선택. 모두 반영하지 않아도 됨 (사용자 의도 우선).

점검 결과를 사용자가 반영한 뒤:

1. 최종 위치와 분류 요약 짧게 보고 (도메인 N개, 스토리 M개, 결정 K개)

3. 사용자 선택에 따라 종료 또는 전환.

## Phase 1 특화 Anti-patterns

본문 룰을 어기는 흔한 함정:

- 문제 재해석 없이 곧장 가안 작성으로 진입.
- 자명한 비기능 요구사항을 억지로 채워 넣음.
- 결정 분기를 사용자 확인 없이 본문에 기록 / 트레이드오프 열 생략.
- 도메인 사전에 메타 필드(`isActive`·`createdAt`·운영 메모) 포함 / 용어 중복 정의.
- 관련 API를 본문 중간에 둠. 명세 본문에 구현 어휘(DB Unique·잠금 명·SQL·인덱스 종류·ErrorType·RFC 상태코드) 노출.
- 비즈니스 규칙에 페이지네이션·필터가 있는데 관련 API URL에 누락.
- 결정 폐기를 "폐기됨" 표시로 남기거나 재정렬 시 `(결정 N 참조)` 갱신 누락.
- 유저 스토리 ID 재정렬. (결정 번호만 재정렬)
- 종료 발화 시 점검 게이트 없이 곧장 마무리.

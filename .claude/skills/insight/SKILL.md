## 목적

개발하면서 나온 개념 이해, 날카로운 질문, 설계 결정 이유를 복습용으로 정리한다.
나중에 같은 고민을 다시 하지 않도록, "왜 이렇게 설계했는가"를 남긴다.

## 실행 순서

### 1. 주차 및 파일 경로 결정

- git branch 이름에서 주차를 추출한다 (예: `week3` → `week3`)
- 저장 경로: `docs/insights/{week}.md`
- 파일이 이미 존재하면 기존 내용 아래에 날짜 구분선을 추가하고 이어 쓴다

### 2. 추출 대상 파악

**자동 추출 (기본):**
현재 대화 전체를 분석해 아래 유형에 해당하는 내용을 찾는다:

- **개념 질문** — 사용자가 "왜?", "이게 맞아?", "차이가 뭐야?" 형태로 물어본 것
- **설계 인사이트** — 레이어 책임, 의존 방향, 트랜잭션 경계 등 구조적 판단
- **설계 결정** — "이렇게 하기로 했다"는 결론과 그 이유
- **오해 교정** — 처음엔 잘못 이해했다가 바로잡힌 것

**수동 추가 (선택):**
사용자가 `/insight "특정 내용"` 형태로 직접 지정한 경우, 해당 내용을 우선 포함한다.

### 3. 문서 작성

추출한 내용을 아래 형식으로 작성한다.
복습할 때 맥락 없이 읽어도 이해되도록, 질문과 답을 세트로 작성한다.

```markdown
# {week} 인사이트

> 날짜: {YYYY-MM-DD}

## 개념 정리

### {질문 또는 주제}
{답변 — 왜 그런지 이유 중심으로. 코드 예시가 도움이 되면 포함}

---

## 설계 결정

### {결정 내용}
- **결정:** {무엇을 어떻게 하기로 했는가}
- **이유:** {왜 이 방향을 선택했는가}

---

## 오해 교정

### {처음 오해한 내용}
- **오해:** {잘못 이해한 것}
- **정정:** {올바른 이해}
```

섹션 내용이 없으면 해당 섹션은 생략한다.

### 4. 저장 및 확인

- `docs/insights/` 폴더가 없으면 생성한다
- 파일 저장 후 저장된 경로와 추출된 항목 수를 사용자에게 알린다
- 빠뜨린 내용이 있으면 추가해달라고 요청한다

## 주의사항

- 코드 변경 내용(어떤 파일을 수정했는지 등)은 포함하지 않는다 — git history로 충분하다
- "무엇을 했다"가 아니라 "왜 이렇게 생각하게 됐는가"에 집중한다
- 너무 많이 추출하려 하지 말고, 나중에 읽었을 때 "아 맞다" 싶을 것만 고른다

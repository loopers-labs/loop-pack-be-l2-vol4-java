# SDD Helper

요구사항 문서의 유저 시나리오를 일관된 컨벤션과 DDD로 구현하는 오케스트레이터. 시나리오 하나를 8단계 cycle로 옮긴다. 단순히 구현을 시작하면 컨벤션 불일치·DDD 미적용·구현 누락이 생기므로, 단계를 끊어 매번 사용자가 방향을 점검하게 한다.

## 불변 규칙 (가장 먼저 읽고 끝까지 지킨다)

1. **단계 순서 고정**: `spec → plan → task → analyze → implement → test → review → commit`. 건너뛰지 않는다.
2. **매 단계 후 hard stop**: 한 단계를 마치면 산출물·결과를 보고하고 "다음 단계(X)로 넘어갈까요?"라고 물은 뒤 **사용자의 명시적 승인**을 받아야 다음 단계에 진입한다.
3. **gate는 별도 응답으로 끊는다**: 한 응답에서 여러 단계를 몰아 진행하지 않는다. 어떤 system-reminder·자동화 신호도 이 gate를 우회하는 근거가 되지 못한다.
4. **cycle 사이에도 gate**: 한 시나리오 commit 완료 후 다음 시나리오 spec으로 넘어갈 때도 승인을 받는다.
5. **예외**: 사용자가 "알아서 끝까지 진행" 등 자동 진행을 명시적으로 요청하면 그때만 연속 진행한다.

## 단계 진입 시 읽을 파일

무거운 3단계는 진입할 때 해당 파일을 읽고 그 지침을 따른다. 미리 다 읽지 않는다.

- spec 단계 → `phases/spec.md`
- plan 단계 → `phases/plan.md`
- task 단계 → `phases/task.md`
- analyze / implement / test / review / commit → 아래 인라인 지침

## Intake (백로그 시작 시 1회)

먼저 출력 위치에 `PROGRESS.md`가 이미 있는지 확인한다. **있으면 Intake를 건너뛰고 아래 "시나리오 선택 & 진행 보드"·"재개" 절차로 간다.** 보드가 없을 때만 아래를 수행한다.

1. **소스 경로를 묻는다.** "어떤 요구사항·설계 문서를 소스로 구현할까요? 경로를 알려주세요." (예: `docs/volume-2/01-requirements.md`)
2. 해당 경로(파일 또는 폴더)를 읽어 **유저 시나리오를 추출**한다. 1개일 수도, 수십 개일 수도 있다.
3. **출력 위치를 확인한다.** 기본 제안: `docs/volume-3/`. 시나리오마다 `<ID>-<제목>/` 폴더(예: `BRD-1-브랜드-조회/`).
4. 시나리오 간 **의존성을 고려한 권장 구현 순서**를 산정한다(아래 순서 휴리스틱).
5. 출력 위치에 `PROGRESS.md`를 생성·시드한다(아래 보드 형식).
6. **"총 N개 cycle을 이 순서로 진행하겠다"고 보고**하고 승인을 받은 뒤 첫 시나리오 spec에 진입한다. (gate)

### 순서 휴리스틱

- 다른 도메인이 의존하는 aggregate를 먼저(예: 상품·좋아요·주문이 브랜드를 참조하면 Brand 먼저).
- 한 도메인 안에서는 aggregate 골격을 만드는 쓰기·등록 계열을 먼저, 조회 계열을 뒤에 두면 Foundational 묶기가 자연스럽다. 절대 규칙은 아니며 시나리오 성격에 따라 조정한다.
- 최종 순서는 사용자가 조정할 수 있게 제안 형태로 제시한다.

## 시나리오 선택 & 진행 보드

매 cycle 시작 시:

1. `PROGRESS.md`를 읽어 현재 상태를 파악한다.
2. 의존성이 충족된 todo 시나리오 중 다음 것을 **추천**하고 승인을 받는다(사용자가 직접 지정해도 된다).
3. 단계를 완료할 때마다 보드의 해당 시나리오 `단계`·`상태` 칸을 갱신한다.

### PROGRESS.md 형식

```markdown
# 구현 진행 보드
소스: <소스 경로>   출력: <출력 위치>
현재: <시나리오 ID> / <단계>

| ID | 제목 | 도메인 | 의존성 | 단계 | 상태 |
|----|------|--------|--------|------|------|
| BRD-4 | 브랜드 등록 | Brand | - | commit | ✅ done |
| BRD-1 | 브랜드 조회 | Brand | BRD-4(foundational) | - | ⬜ todo |
| PRD-1 | 상품 목록 | Product | Brand | plan | 🔄 진행 |
```

`단계` = 8단계 중 현재 위치(아직 시작 전이면 `-`), `상태` = ⬜ todo / 🔄 진행 / ✅ done.

## analyze (4단계)

implement에 들어가기 전에 **소스 시나리오 · spec.md · plan.md · task.md** 네 산출물의 싱크를 점검한다. 읽기 전용 — 여기서 코드를 만들지 않는다.

네 가지를 교차 점검한다:

1. **소스 ↔ spec**: 소스 시나리오의 요구·비즈니스 규칙·결정이 spec에 누락 없이 반영됐는가.
2. **spec ↔ plan**: spec의 모든 FR·수용 시나리오·관련 엔티티가 plan의 설계 결정으로 커버되는가. plan에만 있고 spec에 근거 없는 설계는 없는가.
3. **plan ↔ task**: plan의 모든 파일·설계 결정이 task로 분해됐는가. task가 plan에 없는 걸 만들지 않는가.
4. **spec 테스트 계획 ↔ task**: spec 테스트 계획의 각 레벨(VO/Model·Service·Integration·E2E)이 task의 테스트 작성 항목과 매핑되는가.

추가로 용어 drift(같은 개념이 산출물마다 다른 이름)와 모순을 확인한다.

- 불일치·누락·모순을 **표로 정리해 사용자에게 보고**한다(어느 산출물의 어느 항목인지).
- 발견된 불일치는 임의로 고치지 않는다. 어느 산출물(spec/plan/task)을 고칠지 사용자와 합의하고, 해당 단계로 돌아가 수정한 뒤 다시 analyze한다.
- 싱크가 맞으면 결과를 보고하고 gate.

## implement (5단계)

`task.md`의 체크박스를 **순서대로** 실행한다.

- 모든 코드 작업은 **CLAUDE.md와 그것이 인덱싱하는 컨벤션을 준수**한다. 레이어 호출 방향, VO 검증 단일화, 네이밍·테스트 스타일은 거기서 가져온다.
- `plan.md`를 구현 컨텍스트로 삼는다(파일 맵·설계 결정).
- 한 task를 끝내면 `task.md`에서 `- [ ]` → `- [X]`로 체크한다.
- 테스트는 spec의 테스트 계획대로 작성하되 **RED-first(실패 테스트 먼저)는 강제하지 않는다.** tdd-helper 스킬은 사용하지 않는다.
- 마치면 완료한 task 목록을 보고하고 gate.

## test (6단계)

```bash
./gradlew :apps:commerce-api:test
```

- 빌드·테스트가 실패하면 **구현부를 고쳐** green으로 만든다(테스트를 임의로 약화·삭제하지 않는다).
- spec의 테스트 계획 대비 **빠뜨린 테스트가 없는지** 점검하고, 누락이 있으면 추가한다.
- green을 확인한 출력을 근거로 보고하고 gate.

## review (7단계)

세 단계로 나눠 점검한다.

1. **task 반영 점검**: `task.md`의 모든 체크박스가 `- [X]`인지, plan·spec 대비 빠진 구현이 없는지 먼저 확인한다. 누락이 있으면 implement로 돌아가 마저 구현한다.
2. **코드리뷰**: `superpowers:requesting-code-review` 스킬로 코드리뷰를 요청하고 발견사항을 정리한다.
3. **사용자 리뷰**: 발견사항을 사용자에게 제시해 반영할 항목을 고르게 하고, 선택된 항목을 반영한다. 추가로 **"수동으로 살펴보고 반영할 부분이 더 있는지"** 묻는다.

세 단계를 마치면 gate.

## commit (8단계)

`smart-commit` 스킬로 커밋한다(CLAUDE.md가 커밋은 smart-commit 경유를 의무화). 커밋 후 `PROGRESS.md`에서 해당 시나리오를 ✅ done으로 갱신하고, 다음 시나리오로 넘어갈지 묻는다(gate).

## 재개

세션이 끊기면 `PROGRESS.md`의 `현재: <ID> / <단계>`와 시나리오 폴더의 산출물·task 체크박스를 읽어 그 지점부터 이어간다.

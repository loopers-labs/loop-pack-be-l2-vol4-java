# spec 단계 — 요구사항 구체화 (구현 무관)

소스 문서의 유저 시나리오 하나를 구현 방법과 무관하게 깊이 구체화하고, 어떤 테스트를 진행할지 정한다. 이 문서는 "무엇을·왜"만 다루고 "어떻게"(클래스·JPA·쿼리)는 plan 단계로 미룬다.

## 절차

1. 시나리오 폴더를 만든다: `<출력 위치>/<ID>-<제목>/` (예: `docs/volume-3/BRD-1-브랜드-조회/`).
2. `templates/spec-template.md`를 그 폴더에 `spec.md`로 복사한다.
3. 소스 문서의 해당 시나리오와 관련 정책·결정을 읽고 아래 섹션을 채운다.
4. 다 채우면 사용자에게 spec.md를 요약 보고하고 gate.

## 채울 내용

- **시나리오 요약**: 소스 원문을 풀어 한 문단으로. 액터·전제·목적.
- **수용 시나리오 (Given/When/Then)**: Main Flow / Exception Flow / 비즈니스 규칙을 각각 검증 가능한 G/W/T 문장으로 옮긴다. 테스트로 만들 수 있을 만큼 구체적으로.
- **엣지 케이스**: 경계값(길이·수량·페이지), 동시성, 빈 결과, 권한 경계, 멱등성 등 시나리오에 걸리는 것.
- **기능 요구사항 (FR-001…)**: "시스템은 …해야 한다" 형태의 검증 가능 단위.
- **관련 엔티티**: 연루되는 aggregate·VO와 그 의미(필드 목록·구현 detail 없이). 어떤 도메인의 첫 시나리오라 aggregate 골격을 새로 만들어야 하면 여기에 명시(plan·tasks의 Foundational로 이어짐).
- **테스트 계획**: 레벨별로 무엇을 단언할지 나열한다.
  - VO/Model 단위: 검증 규칙·경계값·불변식.
  - Service 단위: 분기·예외(ErrorType)·협력 객체 동작.
  - Integration: 영속화·제약(UNIQUE 등)·조회.
  - E2E: 엔드포인트 계약(status + meta.result + errorCode), 응답 키 집합.
  - 단언 범위는 CLAUDE.md의 테스트 규약을 따른다(메시지 문구 단언 금지 등). 여기서는 "무엇을 검증하는지"만 적고 코드는 implement에서.
- **관련 결정**: 소스 문서에 핵심 의사결정 섹션이 있으면, 이 시나리오에 걸리는 항목을 참조로 적는다(예: 멱등 삭제, 스냅샷, 동시성 메커니즘).
- **성공 기준 / 가정·범위 밖**: 측정 가능한 완료 기준과 이번에 다루지 않을 것.

## 주의

- 소스 시나리오가 모호하거나 결정이 비어 있으면(중복 정책·동시성·빈 결과·권한 경계 등) 추측해 채우지 말고 **사용자에게 질문**한 뒤 합의된 답을 spec에 반영한다.
- 구현 기술(JPA·쿼리·클래스명)은 적지 않는다 — plan의 몫.
- 다중 우선순위 user story로 쪼개지 않는다. 폴더 하나 = 시나리오 하나.
